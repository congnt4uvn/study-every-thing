## 1. Why AWS Security Matters

- Security is a **shared responsibility** between AWS and the customer.
- AWS secures the **infrastructure** (hardware, networking, physical facilities).
- Customers secure **everything they put on the cloud** (data, IAM, encryption, network config).
- Almost every AWS service has security integration points.

## 2. AWS KMS – Key Management Service

### What is KMS?
AWS Key Management Service (KMS) makes it easy to **create and manage cryptographic keys** and control their use across AWS services.

### Key Concepts

| Term | Description |
|---|---|
| **CMK** (Customer Master Key) | The primary resource in KMS; used to encrypt/decrypt data |
| **Symmetric Key** | Single key used for both encryption and decryption (AES-256) |
| **Asymmetric Key** | Key pair (public/private) used for encrypt/decrypt or sign/verify |
| **Data Key** | Generated by KMS to encrypt large amounts of data (envelope encryption) |
| **Key Policy** | Resource-based policy that controls access to a CMK |

### Types of CMKs

- **AWS Managed Keys** – Created and managed by AWS on your behalf (e.g., `aws/s3`)
- **Customer Managed Keys** – You create, own, and manage them
- **AWS Owned Keys** – Used internally by AWS services; you have no visibility

### Envelope Encryption

1. KMS generates a **Data Encryption Key (DEK)**
2. Your data is encrypted with the DEK locally
3. The DEK itself is encrypted by the CMK and stored alongside the data
4. To decrypt: KMS decrypts the DEK → DEK decrypts data

### KMS Integration with AWS Services

- **S3** – Server-side encryption with KMS (SSE-KMS)
- **EBS** – Encrypted volumes using CMKs
- **RDS** – Encrypted databases
- **Lambda** – Encrypt environment variables
- **Secrets Manager / Parameter Store** – Encrypted secrets

### Key Exam Points

- KMS keys are **regional** — cannot be used across regions directly
- KMS API calls are **audited via CloudTrail**
- You can **rotate keys** automatically (annual rotation for customer-managed CMKs)
- `GenerateDataKey` vs `Encrypt` API call difference

## 3. Encryption in AWS

### Encryption at Rest

Protects data stored on disk:
- S3, EBS, RDS, DynamoDB, Glacier all support encryption at rest
- Uses KMS or AES-256

### Encryption in Transit

Protects data moving over the network:
- Use **TLS/SSL** (HTTPS)
- AWS Certificate Manager (ACM) manages TLS certificates
- Enforced via bucket policies (`aws:SecureTransport`) or HTTPS endpoints

### S3 Encryption Options

| Type | Description |
|---|---|
| **SSE-S3** | AWS manages keys using AES-256 |
| **SSE-KMS** | Uses KMS CMK; audit trail, key rotation |
| **SSE-C** | Customer provides encryption keys |
| **Client-Side Encryption** | Encrypt before uploading to S3 |

## 4. AWS Systems Manager – Parameter Store

### What is it?
A secure, hierarchical storage for **configuration data and secrets** (API keys, passwords, connection strings).

### Tiers

| Tier | Description |
|---|---|
| **Standard** | Up to 10,000 parameters, no additional cost |
| **Advanced** | Up to 100,000 parameters, supports policies, larger value size |

### Parameter Types

- `String` – Plain text
- `StringList` – Comma-separated list
- `SecureString` – Encrypted with KMS

### Hierarchy Example

```
/myapp/
  /dev/
    db_password      → SecureString
    api_key          → SecureString
  /prod/
    db_password      → SecureString
```

### Integration
- Used with **Lambda**, **EC2**, **ECS** to inject config at runtime
- Access controlled via **IAM policies**
- All access is logged in **CloudTrail**

### Parameter Store vs Secrets Manager

| Feature | Parameter Store | Secrets Manager |
|---|---|---|
| Cost | Free (Standard) | Paid per secret |
| Auto rotation | Manual | Automatic (native for RDS) |
| Cross-account | Limited | Supported |
| Use case | Config + simple secrets | Sensitive credentials |

## 5. AWS Lambda & Security Integration

### Environment Variable Encryption
- Lambda encrypts env vars **at rest** using KMS by default
- You can use your own CMK for additional control

### IAM Execution Role
- Every Lambda function has an **execution role**
- Grants permissions to access AWS services (S3, DynamoDB, KMS, etc.)
- Principle of **least privilege** always applies

### Accessing Secrets from Lambda

```python
import boto3

ssm = boto3.client('ssm')
secret = ssm.get_parameter(Name='/myapp/prod/db_password', WithDecryption=True)
print(secret['Parameter']['Value'])
```

### VPC Integration
- Lambda can run inside a **VPC** to access private resources
- Requires VPC, subnets, and security groups configuration

### Lambda & KMS
```python
import boto3, base64

kms = boto3.client('kms')
encrypted = kms.encrypt(KeyId='alias/my-key', Plaintext=b'my-secret')
ciphertext = base64.b64encode(encrypted['CiphertextBlob'])
```

## 6. Other Key Security Services

### AWS IAM
- **Users, Groups, Roles, Policies**
- Use **roles** instead of long-term credentials
- Enable **MFA** for all privileged users
- Use **IAM Access Analyzer** to find unintended public/cross-account access

### AWS Secrets Manager
- Stores and **auto-rotates** secrets (DB passwords, API keys)
- Native integration with RDS, Redshift, DocumentDB
- Charges per secret per month

### AWS CloudTrail
- Logs **all API calls** across your AWS account
- Used for security auditing and compliance
- Integrate with CloudWatch alarms for suspicious activity

### AWS Shield
- **Shield Standard** – Free DDoS protection for all AWS customers
- **Shield Advanced** – Paid; enhanced DDoS protection + 24/7 DRT support

### AWS WAF (Web Application Firewall)
- Protects web apps from common exploits (SQLi, XSS)
- Attach to **CloudFront**, **ALB**, **API Gateway**
- Create **Web ACLs** with rules and rule groups

### Amazon GuardDuty
- Intelligent **threat detection** service
- Analyzes CloudTrail, VPC Flow Logs, DNS logs
- Detects: unusual API calls, cryptocurrency mining, compromised EC2 instances

### Amazon Macie
- Uses ML to **discover and protect sensitive data** in S3
- Identifies PII, financial data, credentials

## 7. Exam Tips

- **KMS is regional** — cross-region replication requires re-encryption
- **CloudTrail** logs every KMS API call — important for compliance
- Prefer **Secrets Manager** for auto-rotating credentials; **Parameter Store** for config
- Always use **IAM roles** over access keys where possible
- `SSE-KMS` gives you audit capability; `SSE-S3` is simpler but less control
- Lambda env vars should use **KMS** for sensitive values
- Know the difference: **encryption at rest** vs **in transit**
- **GuardDuty** = threat detection; **Macie** = sensitive data discovery; **Inspector** = vulnerability assessment
