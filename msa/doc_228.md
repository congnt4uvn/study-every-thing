
Dưới đây là tóm tắt nội dung:

1. Mục tiêu  
   - Chuẩn bị triển khai các microservice lên Kubernetes.  
   - Cần tạo các biến môi trường trong cụm Kubernetes để các container (microservice) sử dụng, tương tự như khi dùng Docker Compose.

2. Lý do cần biến môi trường  
   - Các container phụ thuộc vào nhiều biến môi trường:  
     - profile (cấu hình Spring profile)  
     - URL Spring Config Server  
     - URL Eureka Server  
     - Thông tin Keycloak (admin user, admin password, URL)  
   - Nếu không có các biến này, microservice không thể khởi động đúng.

3. Sử dụng ConfigMap  
   - ConfigMap là đối tượng Kubernetes dùng để lưu dữ liệu không bí mật dưới dạng key–value.  
   - Pod/container có thể dùng ConfigMap dưới dạng:  
     - biến môi trường  
     - tham số dòng lệnh  
     - file cấu hình mount vào volume  
   - Tác giả tạo file `configmaps.yaml` trong thư mục `kubernetes` để định nghĩa ConfigMap.

4. Nội dung file ConfigMap  
   - `apiVersion: v1`  
   - `kind: ConfigMap`  
   - `metadata.name: easybank-config-map`  
   - `data:` chứa các biến môi trường, ví dụ:  
     - profile cần dùng khi deploy microservice  
     - URL Config Server (tên host là tên service trong Kubernetes, ví dụ `configserver`)  
     - URL Eureka Server (tên host là `eurekaserver`)  
     - Tên ứng dụng: configserver, eurekaserver, accounts, loans, cards, gateway…  
     - Thông tin Keycloak:  
       - admin username  
       - admin password  
       - URL Keycloak (host là `keycloak`, port 7080)  
   - Không cấu hình Kafka, RabbitMQ, OpenTelemetry, Grafana trong file này; các thành phần đó sẽ được thiết lập sau bằng Helm chart.

5. Keycloak là bắt buộc  
   - Gateway đã được bảo mật bằng Keycloak.  
   - Nếu không có Keycloak, không thể gọi API để tạo dữ liệu trong các microservice.  
   - Vì vậy, trong giai đoạn này sẽ triển khai microservice + Keycloak.

6. Áp dụng ConfigMap vào cụm  
   - Dùng lệnh:  
     - `kubectl apply -f configmaps.yaml`  
   - Sau khi chạy, ConfigMap được tạo thành công.  
   - Kiểm tra trong Kubernetes Dashboard:  
     - Chọn namespace `default`  
     - Vào mục ConfigMaps  
     - Thấy `easybank-config-map` và xem được toàn bộ key–value.

7. So sánh ConfigMap và Secret  
   - ConfigMap:  
     - Dữ liệu hiển thị rõ trong Dashboard.  
   - Secret:  
     - Dữ liệu được mã hóa base64, mặc định bị ẩn trong Dashboard.  
     - Có thể bấm “view” để xem giá trị (nên không phải là cách bảo mật hoàn hảo).  
   - Có câu “đùa” trong cộng đồng: “Secrets trong Kubernetes thực ra không bí mật”.  
   - Trong thực tế, nên dùng giải pháp bảo mật tốt hơn (các dịch vụ secret manager trên cloud, do team DevOps/Platform thiết lập).

8. Kết luận  
   - Đã hiểu cách tạo ConfigMap trong Kubernetes để cung cấp biến môi trường cho microservice.  
   - Bước tiếp theo: dùng ConfigMap này trong các manifest deployment để triển khai toàn bộ microservice lên cụm Kubernetes.
