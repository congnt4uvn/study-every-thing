
Dưới đây là tóm tắt nội dung:

1. Mục tiêu  
   - Thực hành **Authorization Code Grant** (luồng cấp mã ủy quyền) để truy cập các API bảo vệ phía sau API Gateway, sử dụng **Postman** mô phỏng ứng dụng web/mobile.

2. Cấu hình OAuth2 trong Postman  
   - Chọn tab **Authorization** → Type: **OAuth 2.0**.  
   - Tạo cấu hình token mới:
     - Đặt tên token (ví dụ: `authorization_code_access_token`).
     - Grant Type: **Authorization Code**.
     - Điền:
       - **Callback URL**: dùng URL mặc định của Postman (tương đương redirect URI).
       - **Auth URL** (authorization endpoint) và **Access Token URL** (token endpoint) lấy từ danh sách URL của Keycloak.
       - **Client ID**, **Client Secret**.
       - **Scope**: `openid profile email`.
       - **State**: chuỗi số ngẫu nhiên để chống CSRF.
     - Client Authentication: chọn **Send client credentials in body**.
   - Bật tùy chọn **Authorize using browser** để Postman mở trình duyệt và chuyển hướng đến trang login của Keycloak.

3. Quy trình lấy Access Token với Authorization Code  
   - Đóng tất cả trình duyệt đang mở (tránh dùng nhầm session admin).  
   - Nhấn **Get New Access Token** trong Postman:
     - Trình duyệt mở trang login Keycloak.
     - Đăng nhập với user cuối: username `Madan`, password `12345`.
     - Sau khi login và consent, Keycloak trả về **authorization code**, Postman dùng code này gọi token endpoint để lấy **access token**.
   - Chọn **Use Token** để gắn token vào request.

4. Gọi API Account và lỗi 403  
   - Gọi API Account với token vừa lấy:
     - Ban đầu nhận **401** nếu chưa cấu hình OAuth.
     - Sau khi có token, nhận **403** vì user `Madan` chưa có role phù hợp trong Keycloak.
   - Lý do: Resource server (microservice Account) yêu cầu role liên quan đến account; user chưa có role nên bị từ chối.

5. Gán role cho user trong Keycloak  
   - Đăng nhập lại vào **Keycloak Admin Console** với `admin/admin`.  
   - Vào mục **Users** → chọn user `Madan`.  
   - Vào tab **Role Mappings**:
     - Nhấn **Assign Role**.
     - Chọn các role: `accounts`, `cards`, `loans`.
     - Nhấn **Assign** để gán role cho user.
   - Đóng session admin trên trình duyệt.

6. Lấy lại token và gọi API thành công  
   - Trong Postman, nhấn **Get New Access Token** lần nữa:
     - Đăng nhập lại với user `Madan`.
     - Nhận token mới (lúc này token chứa các role vừa gán).
   - Chọn **Use Token** và gửi request:
     - API Account trả về **200** với dữ liệu thành công.
   - Lặp lại tương tự cho:
     - API **Cards**: cấu hình OAuth giống, lấy token, gửi request → thành công.
     - API **Loans**: tương tự → thành công.

7. Kiểm tra tổng hợp bằng API fetchCustomerDetails  
   - Gọi API `fetchCustomerDetails` (GET) với số điện thoại:
     - API này không yêu cầu auth.
     - Nhận được dữ liệu chứa accounts, loans, cards → xác nhận các bản ghi đã được tạo “ẩn” (behind the scenes).

8. Vai trò của API Gateway  
   - Dù dùng **Client Credentials Grant** hay **Authorization Code Grant**, **API Gateway** không cần thay đổi:
     - Luôn đóng vai trò **Resource Server**.
   - Authorization Code Grant là luồng được dùng phổ biến nhất trong thực tế khi bảo mật hệ thống microservices.

9. Bước tiếp theo (chuẩn bị phần 12)  
   - Tác giả sẽ:
     - Tạo **Docker images** cho các microservice ở background.
     - Cập nhật file **docker-compose**.
   - Người học cần:
     - Dừng tất cả ứng dụng đang chạy.
     - Mở `pom.xml` của tất cả microservice.
     - Đổi tag version từ `S11` sang `S12` (vì sang phần 12).
     - Sau đó build lại để tạo image Docker mới.

Tóm lại, bài giảng minh họa chi tiết cách dùng Postman để thực hiện OAuth2 Authorization Code với Keycloak, gán role cho user, kiểm tra truy cập các API (accounts, cards, loans) qua API Gateway, và chuẩn bị chuyển sang phần tiếp theo với Docker.
