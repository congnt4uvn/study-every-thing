
Dưới đây là tóm tắt ngắn gọn nội dung bài giảng:

1. **Nguyên tắc 1: Một ứng dụng – một codebase – một artifact**
   - Mỗi microservice phải có codebase/kho Git riêng.
   - Mã dùng chung phải tách riêng (thành thư viện hoặc một service hỗ trợ).
   - Code chỉ được build/đóng gói (Docker image, artifact…) **một lần**, sau đó dùng chung cho mọi môi trường (dev, test, prod).
   - Cấu hình theo môi trường không được “nhúng” vào lúc build mà phải đưa từ bên ngoài khi triển khai.

2. **Nguyên tắc 2: API-first (luôn nghĩ API trước)**
   - Mọi chức năng nên được thiết kế và triển khai dưới dạng API (REST).
   - Giúp các microservice khác dễ dàng gọi, tái sử dụng logic.
   - Cho phép nhiều team làm việc song song trên các API khác nhau.
   - Tách biệt nội bộ service với bên ngoài: có thể thay đổi bên trong mà không ảnh hưởng client nếu giữ ổn định API.
   - Hỗ trợ tốt cho pipeline CI/CD và các bài test tích hợp.

3. **Nguyên tắc 3: Quản lý dependency rõ ràng**
   - Tất cả dependency phải được khai báo trong một file manifest (ví dụ: `pom.xml`, `build.gradle`).
   - Build tool (Maven/Gradle) tự động tải dependency từ kho trung tâm về kho cục bộ.
   - Khi build (tạo jar, Docker image), toàn bộ dependency được đóng gói thành một artifact duy nhất (fat jar, image).
   - Tránh tải thư viện thủ công và thêm vào classpath, vì sẽ rất phức tạp với hàng trăm microservice.

4. **Nguyên tắc 4: Design – Build – Release – Run tách bạch**
   - **Design**: xác định công nghệ, dependency, công cụ; bao gồm phát triển và unit test.
   - **Build**: biên dịch, đóng gói code + dependency thành một artifact **bất biến**, có mã định danh/version riêng.
   - **Release**: kết hợp artifact build với cấu hình theo môi trường (DB, credential, cấu trúc thư mục…) để tạo “bản phát hành” bất biến, có ID riêng.
   - **Run**: chạy ứng dụng trong môi trường cụ thể bằng đúng bản release đó.
   - Không được sửa code hay nội dung artifact ở giai đoạn release/run; nếu dùng cùng artifact và release, hành vi phải luôn giống nhau (dễ tái hiện, rollback).

5. **Nguyên tắc 5: Tách cấu hình khỏi mã nguồn**
   - “Configuration” là mọi thứ thay đổi giữa các môi trường (DB URL/credential, message system config, API key bên thứ ba, feature flag…).
   - Không nhúng các cấu hình này vào codebase (trừ cấu hình mặc định).
   - Không phải build lại ứng dụng cho từng môi trường; chỉ cần cung cấp cấu hình tương ứng khi triển khai.
   - Cấu hình nên được quản lý ở nơi riêng, bảo mật (không lộ thông tin nhạy cảm).
   - Trong hệ sinh thái Spring, có thể dùng **Spring Cloud Config Server** để hiện thực nguyên tắc này.

Bài giảng kết thúc bằng việc nhấn mạnh: 5 nguyên tắc đầu tiên này là nền tảng cho kiến trúc cloud-native/microservice, và các nguyên tắc tiếp theo sẽ được trình bày trong bài giảng sau.
