
Dưới đây là tóm tắt ngắn gọn nội dung:

1. **Trạng thái các container Docker**
   - Chạy `docker compose up` và tất cả container (gateway, accounts, loans, cards, keycloak…) đều khởi động thành công.
   - Chỉ Keycloak được expose ra ngoài (cổng 7080); các service accounts, loans, cards không map port ra host, nên chỉ truy cập được trong Docker network.

2. **Kiểm tra không thể gọi trực tiếp microservice**
   - Dùng Postman gọi trực tiếp các API của accounts (8080), cards (9000), loans (8090) trên localhost đều bị lỗi “connection refused”.
   - Điều này chứng minh các service chỉ chạy trong Docker network, không public ra ngoài.

3. **Kiểm tra gateway và cấu hình bảo mật**
   - Gọi các API GET qua Gateway Security (accounts, cards, loans) đều trả về thành công → xác nhận gateway và cấu hình resource server hoạt động đúng.
   - Gọi API tạo tài khoản (POST) mà không gửi credentials → nhận 401 → xác nhận security đang chặn request không có auth.

4. **Vấn đề với Keycloak container mới**
   - Khi dựng Keycloak bằng Docker compose, một container Keycloak mới được tạo với DB nội bộ.
   - Tất cả client/user đã tạo trước đó bị mất → khi dùng client credentials cũ để lấy token thì bị “authentication failed”.
   - Trong môi trường production sẽ dùng DB riêng nên dữ liệu client/user không bị mất khi container thay đổi.

5. **Đăng ký lại client cho luồng Client Credentials**
   - Mở Keycloak tại `localhost:7080`, đăng nhập admin/admin.
   - Tạo lại client `easybank-callcenter-cc`:
     - Bật “Client authentication”.
     - Tắt “Standard flow”, “Direct access grants”.
     - Bật “Service accounts roles”.
   - Lưu client, sang tab Credentials, copy “client secret” và cấu hình lại trong Postman.
   - Lúc này lấy được access token, nhưng khi gọi API vẫn bị 403 vì chưa gán role.

6. **Tạo và gán các role trong Keycloak**
   - Vào “Realm Roles”, tạo 3 role: `ACCOUNTS`, `LOANS`, `CARDS`.
   - Vào client `easybank-callcenter-cc` → Service account roles → Assign role:
     - Gán các role `ACCOUNTS`, `CARDS`, `LOANS`.
   - Quay lại Postman:
     - Lấy access token mới bằng client credentials.
     - Dùng token này gọi API create account/cards/loans → nhận response thành công.

7. **Kiểm tra luồng Authorization Code (end-user login)**
   - Tạo client mới cho Authorization Code:
     - Dùng cùng clientId như trong request Authorization Code của Postman.
     - Bật “Standard flow”, bật “Client authentication”, tắt “Direct access grants”.
     - Cấu hình Redirect URI và Web origins là `*`.
   - Copy client secret mới vào Postman (phần client secret của Authorization Code flow).

   - Tạo user end-user:
     - Vào Users → tạo user `madan`, email `tutor@eazybank.com`.
     - Đặt password `12345`, tắt “Temporary”.
     - Vào Role Mappings, gán các role `ACCOUNTS`, `CARDS`, `LOANS`.

   - Trong Postman:
     - Lấy access token bằng Authorization Code:
       - Trình duyệt mở, đăng nhập với user `madan` / `12345`.
       - Sau khi login, Postman nhận token.
     - Dùng token này gọi API accounts/cards/loans → đều trả về thành công.

8. **Kết luận về bảo mật**
   - Các microservice accounts, loans, cards:
     - Không expose port ra ngoài, chỉ gateway mới truy cập được.
     - Được bảo vệ bởi Keycloak với 2 luồng:
       - Client Credentials (machine-to-machine).
       - Authorization Code (end-user login).
   - Tất cả thay đổi đã được commit lên GitHub trong thư mục `section12`.
   - Đây là cột mốc quan trọng: hệ thống microservices đã được bảo mật đúng chuẩn, chỉ gateway là entry point ra bên ngoài.
