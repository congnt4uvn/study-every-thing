
Dưới đây là tóm tắt ngắn gọn nội dung bạn cung cấp:

1. Tác giả đã:
   - Xây dựng lại toàn bộ Docker images cho các microservice với tag `s13` (thay cho `s12`), bao gồm thêm một microservice mới tên `message`.
   - Đẩy tất cả images này lên Docker Hub.

2. Cập nhật file `docker-compose` (profile `prod`):
   - Thêm service RabbitMQ (service name: `rabbit`) với:
     - Image, hostname, port mapping, healthcheck.
     - Gắn vào cùng network với các microservice khác.
   - Với từng microservice (config server, account, eureka, loan, card, gateway):
     - Đổi tag image từ `s12` sang `s13`.
   - Với service `account`:
     - Khai báo `depends_on` để phụ thuộc vào `rabbit`.
     - Thêm biến môi trường `SPRING_RABBITMQ_HOST` (hoặc tương đương) trỏ tới tên service `rabbit`.
     - Các thông số khác (user, password, port) giữ nguyên giá trị mặc định trong `application.yml`.
   - Thêm service mới `message`:
     - Cũng `depends_on` `rabbit`.
     - Thêm biến môi trường `SPRING_RABBITMQ_HOST=rabbit`.
     - Tham gia cùng network với các microservice khác.

3. Khởi động môi trường Docker:
   - Dừng các container đang chạy (Keycloak, RabbitMQ cũ, Grafana nếu còn).
   - Chạy `docker compose up -d` trong thư mục `prod`.
   - Đợi khoảng 2 phút để tất cả container khởi động.

4. Kiểm tra RabbitMQ:
   - Truy cập RabbitMQ Management UI với user/password `guest/guest`.
   - Thấy:
     - Hai exchange: `communication-send` và `send-communication`.
     - Hai queue tương ứng như đã cấu hình trong ứng dụng.
   - Đây là RabbitMQ container mới, nên không còn các exchange/queue “mồ côi”.

5. Cấu hình Keycloak:
   - Vì là container mới nên chưa có client/role.
   - Tạo client mới:
     - Client ID: `EazyBankCallCenter-cc`.
     - Bật “Client authentication”.
     - Tắt “Standard flow”, bật “Service accounts roles”.
   - Lưu client, lấy thông tin credentials để dùng trong Postman.
   - Tạo role mới: `accounts`.
   - Gán role `accounts` cho service account của client vừa tạo.

6. Kiểm thử với Postman:
   - Lấy access token từ Keycloak bằng client vừa tạo.
   - Dùng token này gọi API tạo account:
     - Ví dụ số điện thoại kết thúc bằng 687.
   - Nhận response thành công.

7. Kiểm tra log các microservice:
   - Trong container `message`:
     - Thấy log của email sender và SMS sender, chứng tỏ message service nhận được sự kiện và xử lý.
   - Trong container `account`:
     - Thấy log:
       - Gửi yêu cầu communication.
       - “Communication request triggered successfully”.
       - Cập nhật trạng thái communication cho account tương ứng.
   - Không truy cập được H2 console vì account service không expose ra ngoài Docker network.

8. Kết luận:
   - Các thay đổi tích hợp RabbitMQ đã biến hệ thống thành microservice theo sự kiện (event-driven) và hoạt động tốt trong môi trường Docker.
   - Toàn bộ mã nguồn phần này được commit lên GitHub trong thư mục `section_13` với ghi chú: “Event-driven microservices using Spring Cloud Function & Stream with RabbitMQ”.
   - Phần tiếp theo sẽ trình bày cách dùng Kafka để triển khai event-driven microservices.
