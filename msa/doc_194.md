
Nội dung trên là phần giảng giải và demo về **xác thực – ủy quyền** với **OAuth2** và **Keycloak** trong bối cảnh các API của hệ thống eazybank (Accounts, Cards, Loans):

1. **Tạo tài khoản (Accounts API)**  
   - Giảng viên dùng Postman gọi endpoint `accounts/api/create` để tạo tài khoản mới.  
   - Thay đổi số điện thoại (từ 687 sang 688) để tránh trùng dữ liệu.  
   - Vào tab Authorization, lấy **Access Token mới** từ Keycloak.  
   - Keycloak gắn thông tin **role** vào access token.  
   - Dùng token này gọi API tạo tài khoản → nhận mã **201** (tạo thành công).

2. **Gọi Cards API – bị 403 Forbidden**  
   - Chuyển sang API Thẻ (Cards), dùng cùng số điện thoại và access token mới.  
   - Kết quả: **403 Forbidden**  
     - Nghĩa là: đã **được xác thực** (nên không phải 401) nhưng **không đủ quyền** (thiếu role phù hợp).

3. **Thêm role CARDS trong Keycloak**  
   - Vào **Keycloak Admin Console** → mục **Realm Roles**.  
   - Tạo role mới: **CARDS** (mô tả: Cards role).  
   - Vào phần **Clients** → mở client `eazybank-callcenter`.  
   - Trong **Service Account Roles**, gán thêm role **CARDS** cho client.  
   - Lấy **Access Token mới** trong Postman.  
   - Giải mã token (JWT) sẽ thấy có 2 role: **ACCOUNTS** và **CARDS**.  
   - Gọi lại Cards API → lần này **thành công**.

4. **Gọi Loans API – lại bị 403**  
   - Chuyển sang phần Loans, gọi API với số điện thoại 688.  
   - Lấy access token mới, nhưng token **chưa có role LOANS**.  
   - Kết quả: **403 Forbidden** vì thiếu quyền.

5. **Thêm role LOANS và gán cho client**  
   - Vào **Realm Roles** → tạo role **LOANS** (chú ý phân biệt hoa/thường).  
   - Mô tả: Loans description.  
   - Vào **Clients** → `eazybank-callcenter` → **Service Account Roles**.  
   - Gán role **LOANS** cho client.  
   - Quay lại Postman, lấy access token mới, dùng token này gọi Loans API.  
   - Kết quả: **phản hồi thành công**.

6. **Kết luận**  
   - Demo minh họa cách:  
     - Xác thực (authentication) bằng OAuth2 + Keycloak.  
     - Ủy quyền (authorization) dựa trên **roles** được nhúng trong access token.  
   - Trong thực tế, yêu cầu bảo mật có thể phức tạp hơn, nhưng cách tiếp cận vẫn tương tự:  
     - Định nghĩa roles trong Keycloak.  
     - Gán roles cho client/service account.  
     - Kiểm tra roles trong token ở phía API.  
   - Giảng viên gợi ý nếu muốn tìm hiểu sâu hơn thì nên học khóa Spring Security của anh ấy, và kết thúc bài giảng về **Client Credentials Grant Flow**.
