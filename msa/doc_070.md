
Bài giảng nói về “cách tiếp cận thứ hai” để đọc cấu hình trong Spring Boot: dùng interface `Environment` để lấy các biến môi trường (environment variables).

1. **Vì sao dùng biến môi trường?**  
   - Các thông tin nhạy cảm (mật khẩu, thông tin bí mật…) không nên đặt trong `application.yaml` hay file cấu hình dễ bị lộ.  
   - Thay vào đó, nên cấu hình chúng dưới dạng biến môi trường trên server.  
   - Chỉ admin server mới có quyền truy cập, giúp tăng bảo mật.

2. **Sử dụng `Environment` trong Spring**  
   - Trong controller (ví dụ `AccountsController`), khai báo:
     - Trường `private Environment environment;`
     - Thêm `@Autowired` để Spring tự inject.  
   - Lưu ý: phải import đúng `org.springframework.core.env.Environment`, không phải Environment của Hibernate.

3. **Tạo REST API đọc biến môi trường**  
   - Tạo một REST API mới, ví dụ:  
     - Method: `getJavaVersion`  
     - Path: `/api/java-version` (HTTP GET)  
   - Trong thân method, dùng:
     ```java
     String javaHome = environment.getProperty("JAVA_HOME");
     ```
   - Trả về giá trị này cho client.  
   - Cập nhật phần mô tả (summary/description) của API cho đúng với chức năng (lấy thông tin phiên bản/đường dẫn Java).

4. **Demo với Postman**  
   - Build và (nếu cần) restart ứng dụng.  
   - Gọi `/api/java-version` bằng Postman:  
     - Server trả về giá trị biến `JAVA_HOME` (thường là đường dẫn cài đặt Java).  
   - Tương tự, có thể thử với `MAVEN_HOME` để lấy đường dẫn cài đặt Maven.

5. **Nhận xét về cách tiếp cận này**  
   - Ưu điểm:  
     - Đọc trực tiếp các biến môi trường trong code.  
     - Phù hợp khi chỉ cần vài thuộc tính môi trường.  
   - Nhược điểm:  
     - Mỗi lần chỉ đọc một thuộc tính.  
     - Phải “hard-code” tên key (ví dụ `"JAVA_HOME"`) trong mã Java.  
     - Không phù hợp khi có nhiều thuộc tính môi trường cần đọc.  

Giảng viên kết luận: cách thứ hai (dùng `Environment`) đã được trình bày và minh họa; ở các bài sau sẽ giới thiệu những cách nâng cao, phù hợp hơn khi có nhiều cấu hình.
