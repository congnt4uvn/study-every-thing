
Bài giảng trên tóm tắt cách xây dựng và kiểm thử microservice “thẻ” (cards) trong hệ thống:

1. Cấu hình dự án  
- Kiểm tra `pom.xml`:  
  - `artifactId` và `name` phải là `cards`.  
  - Đảm bảo đầy đủ dependency, đặc biệt là H2 database.  
- Trong `application.yaml`:  
  - Đặt port cho service thẻ là `9000`.  
  - Service account: `8080`, loan: `8090`.  
  - Các cấu hình còn lại chủ yếu liên quan H2.

2. Cấu trúc database  
- File `schema.sql`: tạo một bảng `cards` với:  
  - `card_id` (PK, auto-increment, do Spring Data JPA quản lý).  
  - `card_number`, `card_type`, `total_limit`, `amount_used`, `available_amount` và các cột metadata.  
- Thiết kế đơn giản để tập trung vào chuẩn và quy trình microservices, không sa đà vào logic nghiệp vụ phức tạp.

3. Entity, DTO và validation  
- Có `BaseEntity` và `Card` entity:  
  - `cardId` là khóa chính, có annotation để auto-generate.  
- DTO: `CardDto`, `ErrorResponseDto`, `ResponseDto`:  
  - Chứa thông tin schema, annotation validation, ví dụ mẫu.  
- Mục tiêu: áp dụng mô hình DTO, validation rõ ràng.

4. Repository và service  
- `CardRepository` với các method custom:  
  - `findByMobileNumber(...)`  
  - `findByCardNumber(...)`  
- Service `CardService` (implements interface):  
  - Tự động inject `CardRepository`.  
  - `createCard(mobileNumber)`  
    - Kiểm tra đã có thẻ với mobile đó chưa; nếu có, ném custom exception.  
    - Nếu chưa, tạo thẻ mới:  
      - Sinh số thẻ ngẫu nhiên 12 chữ số.  
      - `card_type = CREDIT`, `total_limit = 100_000`.  
      - `amount_used = 0`, `available_amount = total_limit - amount_used`.  
      - Lưu vào DB.  
  - `fetchCard(mobileNumber)`  
    - Tìm theo mobile; nếu không có, ném `ResourceNotFoundException`.  
    - Nếu có, convert entity → DTO trả về controller.  
  - `updateCard(cardDto)`  
    - Lấy thẻ theo `cardNumber` (card number không đổi).  
    - Nếu không có, ném `ResourceNotFoundException`.  
    - Map dữ liệu từ DTO sang entity, gọi `save` để update.  
  - `deleteCard(mobileNumber)`  
    - Tìm thẻ theo mobile; nếu không có, ném exception.  
    - Nếu có, xóa bằng `deleteById(cardId)`.

5. Controller và API  
- `CardController` với 4 API, base path `api/` (ví dụ):  
  1. `POST /api/create` – tạo thẻ mới từ `mobileNumber`.  
  2. `GET /api/fetch` – lấy thông tin thẻ theo `mobileNumber`.  
  3. `PUT /api/update` – cập nhật thẻ, nhận `CardDto`.  
     - Thành công: HTTP 200.  
     - Thất bại (validation/logic): HTTP 417.  
  4. `DELETE /api/delete` – xóa thẻ theo `mobileNumber`.  
     - Thành công: HTTP 200.  
     - Thất bại: HTTP 417.  
- Có global exception handler, `CardException`, `ResourceNotFoundException`.  
- Có `CardMapper` để map entity ↔ DTO, file `Constants`, và file `application.properties`/`yaml` cho config.

6. Kiểm thử với Postman  
- Chạy ứng dụng `CardsApplication` ở chế độ debug, port 9000.  
- Test lần lượt:  
  - `POST /api/create` với `mobileNumber` → nhận 201, thẻ được tạo.  
  - `GET /api/fetch` với cùng `mobileNumber` → xem chi tiết thẻ (card number, type, limit, used, available).  
  - Copy response body → dùng cho `PUT /api/update`:  
    - Đổi `card_type` từ CREDIT sang DEBIT.  
    - Đổi `amount_used` = 10_000, `available_amount` = 90_000.  
    - Gửi → nhận 200, sau đó `GET` lại để xác nhận dữ liệu đã cập nhật.  
  - Test case âm:  
    - Dùng số âm, số thẻ 13 chữ số → nhận lỗi validation.  
  - `DELETE /api/delete` với mobile sai → 404.  
  - Sửa mobile đúng → 200.  
  - `GET` lại → `ResourceNotFoundException`.

7. Swagger UI  
- Truy cập `http://localhost:9000/swagger-ui/index.html`.  
- Xem tài liệu 4 API của service thẻ, schema `Card`, `ErrorResponse`, `ResponseDto`.  
- Xác nhận swagger hoạt động đúng.

8. Kết luận và lời khuyên  
- Tất cả microservices (account, loan, card) đã hoàn thiện ở mức cần thiết cho khóa học.  
- Mục tiêu chính: hiểu chuẩn, quy trình, khái niệm microservices, không tập trung vào nghiệp vụ phức tạp.  
- Khuyến nghị:  
  - Thiết lập các service trong IDE cục bộ.  
  - Dùng Postman collection để test toàn bộ API với H2.  
  - Dành thời gian đọc và hiểu từng dòng code.  
- Kết thúc phần này, giảng viên khuyên nghỉ ngơi rồi tiếp tục phần tiếp theo.
