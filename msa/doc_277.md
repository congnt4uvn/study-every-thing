
Nội dung trên nói về một “best practice” khi phát triển microservice: dùng BOM để quản lý phụ thuộc.

Tác giả mô tả bối cảnh:

- Dự án có nhiều microservice: accounts, cards, config server, Eureka server, gateway server, loans, messages…
- Mỗi service có pom.xml (Maven) riêng, trong đó khai báo:
  - Version Spring Boot
  - Version Java
  - Version Spring Cloud
  - Version OpenTelemetry
  - Các dependency khác
  - Plugin (ví dụ Google Jib để build image Docker)

Vấn đề:

- Các version (Spring Boot, Java, Spring Cloud, thư viện bên thứ ba…) đang bị “hard-code” trong từng pom.xml.
- Khi tổ chức có nhiều (ví dụ >30) microservice, nếu muốn nâng version Spring Boot (hoặc bất kỳ thư viện nào), phải:
  - Mở từng pom.xml của từng service
  - Sửa thủ công số version
- Cách làm này không hiệu quả, dễ sai, không phải là best practice.

Giải pháp: sử dụng BOM (Bill Of Materials)

- BOM là một dạng đặc biệt của pom (Project Object Model) dùng để:
  - Quản lý tập trung version của một nhóm dependency liên quan.
- Tác giả so sánh với “interface” trong Java:
  - Interface định nghĩa các “hợp đồng” chung cho các lớp con.
  - BOM định nghĩa các thuộc tính chung / dependency chung mà các microservice cần.
- Khi dùng BOM:
  - Tất cả version chung (Spring Boot, Spring Cloud, Java, thư viện bên thứ ba…) được khai báo ở một nơi.
  - Các microservice chỉ “import” BOM, không cần lặp lại version.
  - Khi cần nâng version, chỉ sửa ở BOM, toàn bộ service sẽ dùng version mới.

Kết luận:

- Vấn đề: khó bảo trì version phụ thuộc khi có nhiều microservice.
- Giải pháp: áp dụng BOM để tập trung quản lý version, tránh sửa từng pom.xml.
- Phần tiếp theo sẽ tiếp tục giải thích chi tiết cách áp dụng BOM trong phát triển microservice.
