
Dưới đây là tóm tắt ngắn gọn nội dung bài giảng:

1. **API Gateway Pattern (Mô hình Cổng API)**  
   - API Gateway là một **điểm vào duy nhất (edge server)** cho toàn bộ hệ thống microservices.  
   - Nó đứng giữa client (web, mobile, tablet…) và các microservice, chịu trách nhiệm:  
     - Bảo mật  
     - Định tuyến  
     - Điều phối giao tiếp  
   - Trong bài giảng, thành phần này được xây dựng bằng **Spring Cloud Gateway**.

2. **Gateway Routing Pattern (Mô hình định tuyến Gateway)**  
   - Nếu edge server có khả năng **định tuyến request** đến đúng microservice dựa trên:  
     - URL  
     - Header  
     - Request parameters  
   - Thì nó đang triển khai **mẫu định tuyến Gateway**.  
   - Spring Cloud Gateway hỗ trợ sẵn việc định tuyến theo URL, nên hệ thống trong bài đã áp dụng mẫu này.

3. **Gateway Offloading Pattern (Mô hình dỡ tải Gateway)**  
   - Gateway sẽ **“dỡ tải” (offload) các mối quan tâm xuyên suốt** khỏi từng microservice, như:  
     - Bảo mật  
     - Caching  
     - Rate limiting  
     - Monitoring / auditing  
     - Load balancing  
     - SSL termination  
   - Nhờ đó, các microservice tập trung vào **logic nghiệp vụ**, còn các concern chung được xử lý tại Gateway.  
   - Không chuyển logic nghiệp vụ sang Gateway, chỉ chuyển các cross-cutting concerns.

4. **Backend for Frontend (BFF) Pattern**  
   - Có **nhiều Gateway khác nhau**, mỗi cái phục vụ **một loại client cụ thể**:  
     - Gateway cho web  
     - Gateway cho mobile  
     - Gateway cho tablet  
   - Lý do: mỗi loại client có yêu cầu khác nhau về:  
     - Định dạng/khối lượng dữ liệu (ví dụ mobile cần dữ liệu gọn, có thể nén, ít hình ảnh hơn)  
     - Cách trình bày, nội dung chi tiết hay tóm tắt  
   - Các microservice trả về dữ liệu “chuẩn”, còn từng BFF sẽ **lọc, biến đổi, tùy biến** response cho từng loại client.

5. **Gateway Aggregation / Composition Pattern (Mô hình tổng hợp Gateway)**  
   - Khi client cần dữ liệu **tổng hợp từ nhiều microservice** (ví dụ: trang tóm tắt tài khoản ngân hàng gồm: account, loan, card):  
     - Cách 1: Client gọi trực tiếp từng service (3 request) → không khuyến khích vì tăng traffic.  
     - Cách 2: Một microservice (ví dụ account) gọi tiếp các service khác, rồi tổng hợp.  
     - Cách 3: **API Gateway** gọi nhiều microservice, **tổng hợp dữ liệu** và trả về **một response duy nhất** cho client.  
   - Khi Gateway làm nhiệm vụ này, ta gọi đó là **Gateway Aggregation/Composition Pattern**.  
   - Từ góc nhìn client: chỉ cần gọi **một API** (ví dụ `/fetchSummary`), còn Gateway sẽ gọi các service phía sau và ghép dữ liệu.

Kết luận:  
Bài giảng giới thiệu 5 mẫu thiết kế liên quan đến API Gateway trong kiến trúc microservices:

1. API Gateway Pattern  
2. Gateway Routing Pattern  
3. Gateway Offloading Pattern  
4. Backend for Frontend (BFF) Pattern  
5. Gateway Aggregation/Composition Pattern
