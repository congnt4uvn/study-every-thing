
Nội dung trên mô tả cách khởi động và kiểm thử **Gateway Server** trong kiến trúc microservices sử dụng **Spring Cloud Gateway** và **Eureka**:

1. **Thứ tự khởi động các dịch vụ**
   - Phải khởi động theo thứ tự:
     1) Config Server  
     2) Eureka Server  
     3) Các microservice: Accounts, Cards, Loans (thứ tự nào cũng được)  
     4) Cuối cùng mới khởi động Gateway Server  
   - Lý do: Gateway cần lấy thông tin các dịch vụ từ Eureka. Nếu Gateway khởi động trước, Eureka chưa có thông tin dịch vụ nên Gateway không thể định tuyến đúng.

2. **Kiểm tra trên Eureka Dashboard**
   - Trên giao diện Eureka, ta thấy:
     - 3 microservice: accounts, cards, loans
     - Gateway Server cũng được đăng ký như một service.
   - Nhấp vào Gateway trong Eureka sẽ thấy các thông tin cấu hình đã khai báo trong `application.yml`.

3. **Kiểm tra actuator của Gateway**
   - Mở: `http://localhost:8072/actuator`
   - Trong JSON trả về có đường dẫn `actuator/gateway`.
   - Gọi tiếp: `http://localhost:8072/actuator/gateway/routes`
   - Kết quả trả về danh sách các route, mỗi route chứa:
     - Thông tin điều kiện khớp (path, predicate)
     - Thông tin URI đích dạng `lb://LOANS`, `lb://ACCOUNTS` (lb = load balancer)
     - Các filter được áp dụng.

4. **Cơ chế định tuyến và load balancing**
   - Ví dụ với dịch vụ Loans:
     - Nếu client gửi request đến Gateway với path bắt đầu bằng `/loans`, Gateway:
       - Kiểm tra predicate (điều kiện khớp path).
       - Dùng URI `lb://LOANS` để chuyển tiếp request đến một instance của service Loans.
       - Việc chọn instance dựa trên cơ chế cân bằng tải (Spring Cloud LoadBalancer) và thông tin từ Eureka.
   - Tên ứng dụng (LOANS, ACCOUNTS, CARDS) phải trùng với tên đăng ký trong Eureka (thường là chữ in hoa).

5. **Filter rewrite path (ghi đè đường dẫn)**
   - Gateway dùng filter để **viết lại path**:
     - Client gửi: `/loans/...` hoặc `/accounts/...`
     - Gateway sẽ **xóa tiền tố** (`/loans`, `/accounts`) rồi chỉ chuyển phần path phía sau đến microservice.
   - Nhờ đó:
     - Bên ngoài: client chỉ biết `/loans/...`, `/accounts/...`
     - Bên trong: microservice nhận đúng path nội bộ mà nó mong đợi (ví dụ `/api/create`, `/api/fetch`).

6. **Demo với Postman – dịch vụ Accounts**
   - Có thư mục “Gateway Server” trong Postman:
     - Request `gatewayserver_1`:
       - Gửi POST đến Gateway: `http://localhost:8072/ACCOUNTS/api/create`
       - Body: name, email, mobileNumber
       - Gateway:
         - Nhận request trên cổng 8072
         - Nhận diện route với path `accounts`
         - Xóa tiền tố `accounts`
         - Chuyển tiếp `/api/create` đến service Accounts (thông qua `lb://ACCOUNTS`)
       - Nhận phản hồi thành công → chứng minh định tuyến hoạt động.
     - Request `gatewayserver_2`:
       - Gửi GET đến Gateway: `http://localhost:8072/ACCOUNTS/api/fetch?mobileNumber=...`
       - Lần đầu dùng sai số điện thoại → “Customer not found”.
       - Dùng đúng số đã tạo ở request 1 → nhận phản hồi thành công.

7. **Demo với dịch vụ Loans**
   - Client gửi request với tiền tố `/loans`.
   - Gateway xử lý tương tự:
     - Xóa tiền tố `loans`
     - Chuyển tiếp đến `lb://LOANS`.
   - Ví dụ: chưa có khoản vay cho số điện thoại đó → trả về “not found”.

8. **Gateway như edge server (máy chủ biên)**
   - Tất cả traffic từ client bên ngoài được yêu cầu đi qua Gateway.
   - Gateway:
     - Định tuyến đến các microservice nội bộ.
     - Giúp **không phải lộ trực tiếp** URL của từng microservice ra bên ngoài.
   - Hiện tại, client vẫn có thể gọi trực tiếp microservice (vì chưa bật bảo mật).
   - Trong các phần sau, tác giả sẽ:
     - Thêm cơ chế bảo mật để client **chỉ có thể** đi qua Gateway.
     - Tối ưu thêm Gateway.

Tóm lại: nội dung giải thích cách cấu hình và vận hành Spring Cloud Gateway với Eureka, cách Gateway tự động lấy route, dùng load balancer, rewrite path, và cách kiểm thử việc định tuyến thông qua Postman để gọi các microservice Accounts và Loans thông qua Gateway thay vì gọi trực tiếp.
