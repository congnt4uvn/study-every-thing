
Dưới đây là tóm tắt nội dung:

1. Bối cảnh  
- Microservice Tài khoản hiện đã có REST API để tạo và truy xuất tài khoản.  
- Bước tiếp theo là xây dựng thêm REST API để cập nhật thông tin chi tiết tài khoản trong DB.  
- Ứng dụng khách sẽ:
  - Gọi API GET theo `mobileNumber` để lấy toàn bộ thông tin khách hàng + tài khoản.
  - Gửi lại toàn bộ dữ liệu đó (có thể chỉnh sửa) vào API cập nhật (PUT/PATCH).

2. Ràng buộc nghiệp vụ  
- Người dùng có thể cập nhật: tên, email, số điện thoại, loại tài khoản, địa chỉ chi nhánh, v.v.  
- Không được phép thay đổi số tài khoản (accountNumber) sau khi đã tạo.

3. Thay đổi ở tầng Service  
- Trong `IAccountService`:
  - Thêm phương thức trừu tượng `updateAccount(CustomerDto customerDto)` trả về `boolean` để cho biết cập nhật thành công hay không.
- Trong `AccountService`:
  - Override phương thức `updateAccount`.
  - Logic chính:
    1. Lấy `AccountsDto` từ `CustomerDto` đầu vào.
    2. Lấy `accountNumber` từ `AccountsDto`.
    3. Dùng `accountNumber` gọi `accountsRepository.findById()`:
       - Nếu không tìm thấy: ném `ResourceNotFoundException`.
       - Nếu tìm thấy: lấy thực thể `Accounts` từ DB.
    4. Dùng mapper (`mapper()` hoặc tương tự) để copy dữ liệu từ `AccountsDto` sang thực thể `Accounts`.
       - Dù `accountNumber` có trong DTO, việc cập nhật chỉ diễn ra nếu DB có bản ghi với số tài khoản đó, nên không lo người dùng đổi số tài khoản sang giá trị khác mà vẫn cập nhật được.
    5. Gọi `accountsRepository.save(accountsEntity)` để lưu dữ liệu tài khoản đã cập nhật.
    6. Lấy `customerId` từ thực thể `Accounts` vừa lưu.
    7. Dùng `customerId` gọi `customerRepository.findById()`:
       - Nếu không có: ném `ResourceNotFoundException`.
       - Nếu có: dùng mapper để copy dữ liệu từ `CustomerDto` sang thực thể `Customer`.
    8. Gọi `customerRepository.save(customerEntity)`:
       - Spring Data JPA tự quyết định insert hay update dựa trên khóa chính.
    9. Nếu mọi thứ thành công, đặt `updated = true` và trả về `true`.

4. Thay đổi ở tầng Controller  
- Trong `AccountsController`:
  - Thêm phương thức mới, ví dụ `updateAccountDetails`, map với endpoint `/api/update` và HTTP method `PUT`.
  - Phương thức:
    - Nhận `CustomerDto` qua `@RequestBody`.
    - Gọi `accountService.updateAccount(customerDto)`.
    - Nếu trả về `true`:
      - Trả `ResponseEntity<ResponseDto>` với HTTP 200 (OK), message kiểu “Request processed successfully”.
    - Nếu trả về `false`:
      - Trả HTTP 500 (Internal Server Error) với message “An error occurred, please try again or contact the development team”.

5. Kiểm thử chức năng  
- Sau khi build lại ứng dụng (DB H2 bị reset):
  1. Gọi API tạo tài khoản để tạo dữ liệu mẫu.
  2. Gọi API GET theo `mobileNumber` để lấy thông tin khách hàng + tài khoản.
  3. Copy response đó, chỉnh sửa:
     - Đổi tên (vd: từ “Madan Reddy” thành “Madan Mohan”).
     - Đổi email, số điện thoại, loại tài khoản, địa chỉ chi nhánh.
     - Giữ nguyên `accountNumber`.
  4. Gửi request PUT tới `/api/update` với payload đã chỉnh sửa.
  5. Nhận response 200, xác nhận cập nhật thành công.
  6. Gọi lại API GET với `mobileNumber` mới để kiểm tra:
     - Thấy các giá trị tên, email, mobile, loại tài khoản, địa chỉ chi nhánh đã được cập nhật.

- Kiểm thử kịch bản lỗi:
  - Gửi request cập nhật với `accountNumber` không tồn tại.
  - Nhận lỗi `ResourceNotFoundException` với thông báo “No accounts found with the given account number” (hoặc tương tự).

6. Vấn đề nhỏ còn tồn tại  
- Trong DB H2:
  - Khi cập nhật, các trường `updatedAt`, `updatedBy` không được cập nhật.
  - Lý do: hiện chưa set các giá trị này thủ công trước khi `save`.
- Tác giả sẽ sửa trong các bài giảng sau bằng cách cấu hình Spring Data JPA để tự động xử lý `createdAt`, `createdBy`, `updatedAt`, `updatedBy` (auditing).
- Tạm thời chấp nhận lỗi nhỏ này.

7. Kết luận  
- API cập nhật tài khoản trong microservice Tài khoản đã được hoàn thiện:
  - Cho phép cập nhật hầu hết thông tin tài khoản/khách hàng.
  - Không cho phép thay đổi số tài khoản.
- Bài tiếp theo sẽ tập trung vào xây dựng API xóa (delete API) và sau đó xử lý vấn đề auditing (created/updated fields).
