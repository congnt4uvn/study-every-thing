
Dưới đây là tóm tắt nội dung:

1. **Mục tiêu**  
   Bài giảng giới thiệu cách triển khai mô hình giới hạn tốc độ (Rate Limiting) trong Spring Cloud Gateway, sử dụng `RequestRateLimiterGatewayFilterFactory` và Redis.

2. **RequestRateLimiter & KeyResolver**  
   - `RequestRateLimiterGatewayFilterFactory` quyết định xem một request có được phép tiếp tục hay không. Nếu vượt quá giới hạn, Gateway trả về HTTP 429 (Too Many Requests).  
   - Cần một `KeyResolver` để xác định “khóa” dùng cho việc giới hạn: theo user, session, IP, host,…  
   - Spring cung cấp sẵn `PrincipalNameKeyResolver` (dùng với Spring Security) để lấy username hiện tại làm key.  
   - Nếu không tìm được key, mặc định request sẽ bị từ chối (có thể cấu hình lại hành vi này).

3. **Redis RateLimiter**  
   - Spring Cloud Gateway có sẵn một triển khai RateLimiter dựa trên Redis, lấy cảm hứng từ giải pháp của Stripe.  
   - Redis là hệ thống lưu trữ in-memory, dùng để lưu trạng thái giới hạn tốc độ phân tán.  
   - Stripe đưa ra nhiều tình huống cần Rate Limiting:
     - Một user gây bùng nổ lưu lượng, ảnh hưởng người dùng khác.  
     - Script lỗi hoặc cố ý spam request.  
     - Request ưu tiên thấp làm ảnh hưởng request ưu tiên cao.  
     - Request phân tích dữ liệu khối lượng lớn ảnh hưởng giao dịch quan trọng.  
     - Hệ thống gặp sự cố, cần cắt bớt lưu lượng ưu tiên thấp.

4. **Thuật toán Token Bucket & 3 thuộc tính chính**  
   Redis RateLimiter dùng thuật toán “token bucket” với 3 tham số:

   - **replenishRate**  
     - Số token (tức số request) được thêm vào mỗi giây.  
     - Ví dụ: `replenishRate = 100` → mỗi giây thêm 100 token vào bucket.

   - **burstCapacity**  
     - Số token tối đa bucket có thể chứa (giới hạn “dung lượng bùng nổ”).  
     - Ngăn việc tích lũy vô hạn token nếu người dùng không sử dụng.  
     - Ví dụ: `burstCapacity = 200`, `replenishRate = 100` → sau 2 giây bucket đầy 200 token; từ giây thứ 3 trở đi nếu không dùng, token mới không được thêm nữa.

   - **requestedTokens**  
     - Số token tiêu tốn cho mỗi request (mặc định là 1).  
     - Có thể tăng lên (10, 100,…) nếu muốn một request “tốn” nhiều token hơn.

5. **Các mô hình cấu hình điển hình**  
   - **Tốc độ ổn định**:  
     - `replenishRate = burstCapacity` → mỗi giây thêm token, nhưng nếu không dùng thì token cũ bị “thay thế” bởi token mới, không tích lũy.  
   - **Cho phép bùng nổ tạm thời**:  
     - `burstCapacity > replenishRate` → cho phép người dùng “dồn” token chưa dùng để bùng nổ trong một khoảng ngắn.  
     - Ví dụ: `replenishRate = 100`, `burstCapacity = 200` → nếu không dùng trong giây đầu, sang giây thứ hai có thể dùng 200 token cùng lúc.

   - **Ví dụ 1 request/phút**:  
     - Mục tiêu: chỉ cho phép 1 request mỗi phút.  
     - Cấu hình:
       - `replenishRate = 1` (mỗi giây thêm 1 token → 60 giây được 60 token)  
       - `burstCapacity = 60` (tối đa 60 token trong bucket)  
       - `requestedTokens = 60` (mỗi request tiêu 60 token)  
     → Kết quả: mỗi phút chỉ thực hiện được 1 request.

6. **Cấu hình trong Spring Cloud Gateway**  
   - Có thể cấu hình RateLimiter và 3 thuộc tính trên trong file YAML hoặc bằng Java config.  
   - Cần thêm dependency Redis RateLimiter vào `pom.xml`.  
   - Có thể định nghĩa `KeyResolver` dưới dạng bean, ví dụ lấy giá trị từ query param `user` làm key.  
   - Logic trong `KeyResolver` hoàn toàn tùy thuộc yêu cầu nghiệp vụ (user, IP, header,…).

7. **Lưu ý quan trọng**  
   - Nếu `burstCapacity = 0` → tất cả request sẽ bị chặn.  
   - Cần cẩn thận khi cấu hình để tránh vô tình chặn toàn bộ lưu lượng.

8. **Kết luận**  
   Bài giảng này dừng ở phần giải thích khái niệm và cách hoạt động của RateLimiter trong Spring Cloud Gateway (dựa trên Redis, token bucket, KeyResolver). Bài giảng tiếp theo sẽ hướng dẫn triển khai thực tế trong Gateway Server bằng Java config.
