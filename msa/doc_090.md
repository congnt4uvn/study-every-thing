
Nội dung trên mô tả kế hoạch và các bước chuẩn bị để đưa toàn bộ hệ thống microservices (tài khoản, khoản vay, thẻ) cùng với Config Server vào môi trường Docker, rồi sau đó có thể triển khai lên môi trường production/Kubernetes. Tóm tắt các ý chính:

1. **Mục tiêu tổng thể**  
   - Hiện tại đã quản lý cấu hình trong môi trường cục bộ với Spring Cloud Config Server.  
   - Bước tiếp theo: container hóa tất cả microservices và Config Server, khởi động chúng cùng nhau bằng Docker Compose.  
   - Đây là bước chuẩn bị để sau này triển khai lên production (ví dụ Kubernetes).

2. **Tổ chức cấu trúc Docker Compose theo môi trường**  
   - Tạo thư mục `docker-compose` trong project `cloud-config-v2-spring`.  
   - Bên trong tạo 3 thư mục con: `default`, `qa`, `prod`.  
   - Mỗi thư mục sẽ chứa file `docker-compose.yml` riêng cho từng môi trường, giúp linh hoạt cấu hình.

3. **Tái sử dụng và chỉnh sửa file Docker Compose hiện có**  
   - Đang có một file `docker-compose.yml` trong service `accounts`.  
   - Sao chép file này vào thư mục `docker-compose/default` rồi xóa file cũ trong service `accounts`.  
   - Trong file mới:
     - Cập nhật tên image của các service `accounts`, `loans`, `cards` từ `:S4` sang `:S6` (tương ứng với phần/bản phát hành thứ 6).  

4. **Thêm Config Server vào Docker Compose**  
   - Thêm một service mới trong `docker-compose.yml` cho Config Server:
     - `image: eazybytes/configserver:S6`
     - `container_name: configserver-ms`
     - `ports: "8071:8071"`
     - Giới hạn bộ nhớ: 700MB (giống các service khác).
     - Tham gia cùng network `eazybank` để các container giao tiếp với nhau qua tên service.

5. **Vấn đề với `spring.config.import` và localhost trong Docker**  
   - Trong `application.yml` của mỗi microservice, đang dùng:
     - `spring.config.import: configserver:http://localhost:8071`  
   - Khi chạy trong Docker:
     - `localhost` bên trong container chỉ trỏ tới chính container đó, không phải Config Server.  
     - Do đó, microservice không thể dùng trực tiếp cấu hình này khi chạy trong Docker.

6. **Đưa cấu hình Config Server ra biến môi trường**  
   - Cần đưa `spring.config.import` ra ngoài, cấu hình qua biến môi trường trong Docker Compose, và ghi đè giá trị trong `application.yml`.  
   - Trong `docker-compose.yml` của từng microservice, thêm phần `environment:`:
     1. `SPRING_CONFIG_IMPORT`  
        - Tên biến môi trường phải viết hoa và thay dấu chấm bằng dấu gạch dưới:  
          `spring.config.import` → `SPRING_CONFIG_IMPORT`.  
        - Giá trị: `configserver:http://configserver:8071`  
          - Thay `localhost` bằng tên service của Config Server trong Docker Compose (`configserver`).  
          - Docker sẽ cho phép các container trong cùng network gọi nhau bằng tên service.
     2. `SPRING_PROFILES_ACTIVE`  
        - Chỉ định profile sẽ dùng, ví dụ `default` cho file trong thư mục `default`.  
     3. `SPRING_APPLICATION_NAME`  
        - Phải trùng với tên ứng dụng dùng trong Config Server (và trong `application.yml`), ví dụ:
          - `accounts` cho service accounts  
          - `loans` cho service loans  
          - `cards` cho service cards  
        - Việc này là workaround cho một bug hiện tại của Spring Cloud Config: cần khai báo lại `spring.application.name` qua biến môi trường, dù trùng với cấu hình trong `application.yml`.

   - Đồng thời, trong `application.yml` nên bỏ tùy chọn `optional:` trong `spring.config.import` khi triển khai lên môi trường cao hơn (Docker, Kubernetes), để service sẽ fail nếu không kết nối được Config Server (tránh chạy trong trạng thái cấu hình không đầy đủ).

7. **Áp dụng cho tất cả microservices**  
   - Các biến môi trường trên (`SPRING_CONFIG_IMPORT`, `SPRING_PROFILES_ACTIVE`, `SPRING_APPLICATION_NAME`) được cấu hình tương tự cho:
     - `accounts` (application name: `accounts`)
     - `loans` (application name: `loans`)
     - `cards` (application name: `cards`)

8. **Vấn đề khởi động thứ tự và readiness của Config Server**  
   - Nếu chỉ chạy `docker compose up`, Docker sẽ tạo container theo thứ tự khai báo (configserver, accounts, loans, cards), nhưng:
     - Docker không chờ Config Server khởi động hoàn toàn trước khi khởi động các microservices.  
     - Nếu microservice khởi động trước khi Config Server sẵn sàng, nó sẽ fail vì không lấy được cấu hình.
   - Cần:
     - Cung cấp cho Docker cách kiểm tra xem Config Server đã “sống” và “sẵn sàng” (liveness & readiness).  
     - Khai báo dependencies để Docker chỉ khởi động `accounts`, `loans`, `cards` sau khi Config Server sẵn sàng.

9. **Chuẩn bị cho bước tiếp theo**  
   - Bài giảng dừng ở việc chuẩn bị gần xong `docker-compose.yml` cho môi trường default.  
   - Bước tiếp theo: tìm hiểu khái niệm liveness và readiness, rồi áp dụng vào Docker Compose để đảm bảo Config Server khởi động hoàn toàn trước các microservices.
