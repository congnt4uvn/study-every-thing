
Bài giảng trên nói về cách dùng Helm để rollback (khôi phục) một phiên bản release trước đó trong Kubernetes, cụ thể với release tên “easy-bank”.

Nội dung chính:

1. So sánh Helm và kubectl  
   - Helm rollback cho phép khôi phục toàn bộ các thành phần (nhiều microservice) về trạng thái trước đó chỉ với **một lệnh**.  
   - Nếu dùng kubectl, phải tự khôi phục **từng thành phần** một.

2. Lệnh `helm history`  
   - Dùng `helm history easy-bank` để xem lịch sử cài đặt/nâng cấp của chart.  
   - Ví dụ có 3 revision:
     - Revision 1: cài đặt ban đầu (ổn định).  
     - Revision 2: nâng cấp với tag `s12` (bị lỗi).  
     - Revision 3: nâng cấp tiếp với tag `s11` (hiện đang được deploy).

3. Thực hiện rollback  
   - Mục tiêu: quay lại phiên bản hoạt động tốt trước đó (revision 1), nơi gateway server được bảo mật đúng.  
   - Lệnh dùng:  
     `helm rollback easy-bank 1`  
   - Chỉ với lệnh này, toàn bộ release được khôi phục về revision 1.

4. Xác nhận kết quả rollback  
   - Vào Kubernetes Dashboard → Pods → mở pod mới của gateway.  
   - Trong Events thấy image mới (ví dụ `s14`) đang được deploy.  
   - Xem logs: gateway server khởi động thành công.  
   - Kiểm tra bằng Postman: gọi API nhận lỗi 401 → chứng tỏ bảo mật đã hoạt động như mong muốn.

5. Lịch sử sau rollback  
   - Chạy lại `helm history easy-bank`:
     - Xuất hiện revision 4 với comment kiểu “rollback to 1”.  
     - Điều này giúp dễ hiểu lịch sử triển khai và các lần rollback trong tương lai.

6. Kết thúc  
   - Giảng viên quay lại file `values.yaml`, chỉnh lại tag về `s14` để tiếp tục kiểm tra trên GitHub repo.  
   - Nhấn mạnh: rollback với Helm rất đơn giản, rõ ràng và mạnh mẽ.
