
Bài giảng mô tả cách cải thiện tài liệu REST API trong Spring Boot với Swagger/OpenAPI:

1. **Tài liệu cho lớp Controller**
   - Mặc định Swagger chỉ hiển thị tên kỹ thuật như `AccountsController`.
   - Dùng annotation `@Tag` (từ swagger / springdoc-openapi) trên lớp controller:
     - `name`: tóm tắt ngắn gọn về nhóm API, ví dụ: “REST APIs thô sơ dành cho tài khoản trong EasyBank”.
     - `description`: mô tả chi tiết hơn, ví dụ: “REST API để tạo, cập nhật, tìm nạp và xóa thông tin chi tiết tài khoản”.
   - Sau khi build và refresh Swagger UI, phần controller hiển thị tên và mô tả thân thiện, giúp khách hàng hiểu mục đích các API trong controller.

2. **Tài liệu cho từng API (method)**
   - Dùng annotation `@Operation` trên từng method:
     - `summary`: tiêu đề ngắn, ví dụ: “Create Account REST API”.
     - `description`: mô tả chi tiết chức năng, ví dụ: “REST API để tạo Khách hàng và Tài khoản mới bên trong EasyBank”.
   - Sau khi build, Swagger UI hiển thị summary và description cho từng endpoint.

3. **Mô tả mã phản hồi (HTTP status code)**
   - Mặc định Swagger hiển thị response code 200 với schema mặc định.
   - Nếu API trả về mã khác (ví dụ POST trả 201):
     - Dùng `@ApiResponse`:
       - `responseCode = "201"`
       - `description = "HttpStatus.CREATED"`
   - Với các API có nhiều loại phản hồi (ví dụ update trả 200 nếu thành công, 500 nếu lỗi):
     - Dùng `@ApiResponses` (dạng số nhiều) chứa nhiều `@ApiResponse`:
       - Một `@ApiResponse` cho `responseCode = "200"`.
       - Một `@ApiResponse` cho `responseCode = "500"`.
   - Tương tự áp dụng cho API delete (ví dụ 200 và 500).
   - Sau khi build, Swagger UI hiển thị đầy đủ các response code và mô tả, giúp dev/tester biết trước các phản hồi có thể nhận.

4. **Kết quả**
   - Mỗi controller có tên và mô tả rõ ràng.
   - Mỗi endpoint có summary, description, và danh sách response code chi tiết.
   - Khách hàng, developer, tester dễ hiểu mục đích API và các tình huống phản hồi để xây dựng logic phù hợp.

5. **Bước tiếp theo (nhá trước bài sau)**
   - Hiện tại các schema (DTO) hiển thị tên kỹ thuật và ví dụ mặc định (ví dụ “String”).
   - Bài tiếp theo sẽ cải thiện tài liệu cho các đối tượng schema: đặt tên thân thiện, thêm dữ liệu mẫu / ví dụ để Swagger UI hiển thị rõ ràng hơn cho người dùng.
