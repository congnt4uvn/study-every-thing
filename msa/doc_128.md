
Dưới đây là tóm tắt ngắn gọn nội dung bạn cung cấp:

1. **Trạng thái ban đầu của Gateway**  
   - Gateway server nhận thông tin định tuyến từ Eureka (discovery locator bật).  
   - Hành vi mặc định: đường dẫn bên ngoài trùng với tên service (accounts, loans, cards…) và được tự động route đến các microservice tương ứng.

2. **Nhu cầu định tuyến tùy chỉnh**  
   - Muốn các client gọi API theo dạng:  
     - `easybank/accounts/**`  
     - `easybank/loans/**`  
     - `easybank/cards/**`  
   - Lý do: trông chuyên nghiệp hơn, thể hiện rõ tổ chức (easybank) + tên microservice.

3. **Cấu hình RouteLocator bằng Java**  
   - Trong lớp main của Gateway (GatewayServerApplication), tạo một bean:
     ```java
     @Bean
     public RouteLocator easybankRouteConfig(RouteLocatorBuilder builder) {
         return builder.routes()
             .route(p -> p.path("/easybank/accounts/**")
                 .filters(f -> f.rewritePath("/easybank/accounts/(?<segment>.*)", "/${segment}"))
                 .uri("lb://ACCOUNTS"))
             .route(p -> p.path("/easybank/loans/**")
                 .filters(f -> f.rewritePath("/easybank/loans/(?<segment>.*)", "/${segment}"))
                 .uri("lb://LOANS"))
             .route(p -> p.path("/easybank/cards/**")
                 .filters(f -> f.rewritePath("/easybank/cards/(?<segment>.*)", "/${segment}"))
                 .uri("lb://CARDS"))
             .build();
     }
     ```
   - `RouteLocatorBuilder` dùng để định nghĩa các route tùy chỉnh.  
   - `path(...)`: định nghĩa URL mà client gọi.  
   - `filters(...)` + `rewritePath(...)`:  
     - Lấy phần sau `easybank/accounts` (gọi là `segment`)  
     - Chuyển tiếp sang microservice với đúng path tương ứng (`/${segment}`).  
   - `uri("lb://SERVICE_NAME")`:  
     - `lb` là load balancer client-side.  
     - Tên service phải trùng với tên đăng ký trên Eureka (thường là IN HOA: ACCOUNTS, LOANS, CARDS).

4. **Lỗi và cách sửa**  
   - Lỗi “invalid hostname” do cấu hình `uri` thiếu `//` sau dấu `:` (phải là `lb://ACCOUNTS`).  
   - Sửa lại, build lại, sau đó gọi lại API bằng Postman:  
     - Ví dụ: `POST http://localhost:8007/easybank/accounts/api/create`  
   - Sau khi sửa số điện thoại để tránh trùng dữ liệu, API trả về thành công.

5. **Quan sát trên actuator routes**  
   - Khi còn bật discovery locator, actuator hiển thị cả:  
     - Route mặc định (accounts, loans, cards…)  
     - Route tùy chỉnh (easybank/accounts, easybank/loans, easybank/cards).  
   - Điều này dễ gây nhầm lẫn cho client.

6. **Tắt hành vi mặc định của discovery locator**  
   - Trong `application.yml` của Gateway:
     ```yaml
     spring:
       cloud:
         gateway:
           discovery:
             locator:
               enabled: false
     ```
   - Sau khi tắt, chỉ còn 3 route tùy chỉnh (easybank/…) trong actuator.  
   - Khi thử gọi lại các route cũ (không có tiền tố easybank), sẽ nhận 404 vì route mặc định đã bị vô hiệu hóa.

7. **Khuyến nghị cấu hình**  
   - Spring Cloud Gateway cho phép cấu hình route cả trong `application.yml` lẫn bằng Java.  
   - Tác giả khuyến nghị dùng **Java-based configuration** vì:  
     - Dễ định nghĩa các cấu hình route phức tạp.  
     - Dễ thêm nhiều filter tùy chỉnh.  
   - Cấu hình trong `application.yml` có nhiều hạn chế hơn.

Tóm lại:  
- Ban đầu dùng route mặc định từ Eureka.  
- Chuyển sang định tuyến tùy chỉnh với tiền tố `easybank/` cho tất cả microservice.  
- Định nghĩa route bằng Java (`RouteLocator` + `RouteLocatorBuilder`), dùng `rewritePath` và `lb://SERVICE_NAME`.  
- Tắt discovery locator mặc định để chỉ còn một kiểu route thống nhất, chuyên nghiệp hơn cho client.
