
Dưới đây là tóm tắt nội dung:

1. Triển khai microservices bằng Helm  
   - Đứng trong thư mục `environments`, chọn chart tương ứng với môi trường (`dev`, `prod`, `qa`).  
   - Ví dụ triển khai môi trường production: dùng lệnh `helm install` với release name là `easybank` và chọn chart `prod-env`.  
   - Sau khi chạy lệnh, Helm triển khai toàn bộ microservices lên Kubernetes.

2. Quan sát trên Kubernetes Dashboard  
   - Vào mục Pods để xem các microservice khởi động.  
   - Ban đầu nhiều pod sẽ lỗi và bị restart liên tục vì Config Server và Eureka Server chưa sẵn sàng.  
   - Kubernetes tự động restart pod với backoff theo cấp số nhân (2s, 4–5s, …) cho đến khi thành công hoặc đạt số lần tối đa.

3. Kiểm tra log và thứ tự khởi động  
   - Bắt đầu với Config Server: mở log, bật auto-refresh 5 giây, chờ đến khi thấy log báo “started successfully”.  
   - Sau đó kiểm tra Eureka Server (phụ thuộc Config Server): ban đầu restart nhiều lần, nhưng cuối cùng cũng khởi động thành công (log báo ứng dụng Eureka khởi động trong ~58 giây).  
   - Tiếp theo kiểm tra các dịch vụ:  
     - Account Service: khởi động thành công và kết nối Eureka.  
     - Loans Service, Cards Service: kiểm tra log, đều khởi động thành công.  
     - Gateway Server: log báo ứng dụng gateway khởi động thành công.  
     - Message Service: khởi động ngay từ đầu (không phụ thuộc Config/Eureka), log cho thấy kết nối thành công tới Kafka.

4. Kiểm thử các API microservices  
   - Gọi API trong Account Service (ví dụ API “contact details”) và nhận được dữ liệu với cấu hình profile `prod` → xác nhận ứng dụng chạy với profile production.  
   - Gọi các API thông tin như `java-version` (Cards) và `build-info` (Loans) → hoạt động bình thường.

5. Luồng xác thực với Keycloak và tạo dữ liệu  
   - Đảm bảo dùng đúng `client secret` và đúng cổng Keycloak (80).  
   - Lấy access token từ Keycloak, sau đó:  
     - Gọi API tạo tài khoản mới → nhận HTTP 201, tài khoản được tạo.  
     - Với Cards: dùng cùng `client secret`, chỉnh cổng token URL sang 80, lấy token, gọi API tạo thẻ → thành công.  
     - Với Loans: tương tự, lấy token, gọi API tạo khoản vay → thành công.  
   - Cuối cùng gọi API `fetchCustomerDetails` với số điện thoại đã dùng:  
     - Lần đầu chỉ thấy thông tin tài khoản (có timeout với loans/cards).  
     - Gửi lại lần nữa thì nhận đủ thông tin account, loans, cards → xác nhận toàn bộ chuỗi microservices hoạt động.

6. Kiểm tra hệ thống quan sát (Observability) với Grafana  
   - Mở Grafana trên cổng 3000 (đã port-forward).  
   - Trong mục “Explore”:  
     - Chọn nguồn dữ liệu Loki, lọc theo nhãn `container=gateway-server`, chạy query → xem được log của Gateway.  
     - Từ log, nhấp vào link trace ID → mở Tempo và xem trace phân tán đầy đủ → xác nhận Loki, Tempo và tích hợp giữa chúng hoạt động.

7. Kiểm tra metrics với Prometheus + Grafana  
   - Trong “Explore”, chọn nguồn dữ liệu Prometheus.  
   - Chọn một metric có giá trị (ví dụ metric về container), thêm filter theo `container`, chạy query.  
   - Hiển thị biểu đồ trong khoảng thời gian (ví dụ 15 phút), đổi kiểu sang line chart → xem được biểu đồ uptime/metrics cho các container trong Kubernetes.  
   - Điều này xác nhận Prometheus đã tích hợp thành công với Grafana.

8. Kết luận  
   - Toàn bộ hệ thống microservices (Config Server, Eureka, Account, Loans, Cards, Gateway, Message) cùng các thành phần hỗ trợ (Kafka, Grafana, Prometheus, Loki, Tempo) đã được triển khai và vận hành thành công trên Kubernetes bằng Helm.  
   - Giảng viên nhấn mạnh đây là một quy trình hoàn chỉnh, phức tạp; người học nên tự hào vì đã nắm được toàn bộ luồng triển khai và quan sát hệ thống.  
   - Các phần tiếp theo sẽ hướng dẫn cách thiết lập các môi trường này trên cloud.
