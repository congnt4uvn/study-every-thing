
Nội dung trên mô tả quy trình cập nhật và triển khai lại các microservice trên Kubernetes bằng Helm, cụ thể với ví dụ Gateway Server:

1. **Bối cảnh**  
   - Các microservice được triển khai bằng Helm chart.  
   - Cần cập nhật cấu hình (values) hoặc image (tag), hoặc thay đổi số replica.  
   - Trước đây dùng `kubectl`, giờ dùng lệnh `helm`.

2. **Thay đổi cấu hình Gateway Server**  
   - Vào thư mục `helm/environments/prod-env`.  
   - Mở file `values.yaml` của Gateway Server.  
   - Ban đầu image tag là `s14`.  
   - Tác giả đổi thành `s12` với mục đích bỏ yêu cầu xác thực/ủy quyền, để có thể gọi API không cần security.

3. **Build lại dependency và upgrade Helm**  
   - Vì `prod-env` phụ thuộc vào chart Gateway Server, cần:  
     - Vào thư mục `prod-env`  
     - Chạy `helm dependency build`  
   - Sau đó quay lại thư mục gốc môi trường và chạy:  
     - `helm upgrade easybank prod-env`  
   - Helm tự phát hiện thay đổi và triển khai lên cluster, tạo revision mới (revision 2).

4. **Kiểm tra trên Kubernetes và Postman**  
   - Vào Kubernetes Dashboard, kiểm tra pod Gateway Server, xem log cho đến khi pod khởi động xong.  
   - Dùng Postman gọi API `Accounts_Post_Noauth` (API không yêu cầu auth).  
   - Kết quả vẫn nhận lỗi 401 Unauthorized → chứng tỏ thay đổi chưa đúng.

5. **Phát hiện và sửa lỗi tag**  
   - Nhận ra đã chọn sai tag:  
     - `s12` có thay đổi liên quan đến OAuth2 (có security).  
     - `s11` mới là phiên bản không có bảo mật.  
   - Sửa lại `values.yaml` → đổi tag thành `s11`.  
   - Lặp lại quy trình:  
     - Vào `prod-env` → `helm dependency build`  
     - Quay lại thư mục gốc → `helm upgrade easybank prod-env`  
   - Lần này tạo revision 3.

6. **Triển khai lại và xác nhận**  
   - Quan sát pod Gateway Server mới trong Kubernetes, chờ pod khởi động.  
   - Gọi lại API `Accounts_Post_Noauth` trong Postman.  
   - Lần này nhận được response thành công dù không gửi thông tin xác thực → xác nhận thay đổi đã được triển khai đúng.  
   - Điều này chứng minh `helm upgrade` hoạt động đúng, chỉ triển khai phần thay đổi.

7. **Gợi ý các thay đổi khác**  
   - Có thể dùng cùng cách để thay đổi số replica (ví dụ từ 1 lên 3 hoặc 5).  
   - Tác giả không tăng replica vì máy tính cá nhân không đủ tài nguyên (kể vui về laptop “than phiền”).

8. **Kết luận**  
   - Đã minh họa cách triển khai thay đổi mới bằng `helm upgrade`.  
   - Bài tiếp theo sẽ hướng dẫn cách rollback (khôi phục) các thay đổi bằng Helm.
