
Dưới đây là tóm tắt ngắn gọn nội dung:

1. Tác giả đã:
   - Xây dựng lại Docker image cho 6 microservice với tag mới `s10` (trước là `s9`).
   - Cập nhật `pom.xml` của tất cả microservice để đổi tag từ `s9` sang `s10`.
   - Đẩy các image mới này lên Docker Hub, có thể thấy tag `s10` trong từng service.

2. Lệnh build/push:
   - Sử dụng Google Jib để tạo Docker image (lệnh được ghi trong README trên GitHub).
   - Dùng `docker image push` để đẩy image lên Docker Hub.

3. Cập nhật Docker Compose:
   - Thêm một service mới là `redis`:
     - Image: `redis`
     - Mapping port, healthcheck (lệnh lấy từ tài liệu chính thức Redis).
     - Cấu hình timeout, retry, và gắn vào cùng network với các service khác.
   - Trong Gateway Server:
     - Thêm dependency Redis (Spring Cloud Gateway + RedisRateLimiter).
     - Thêm biến môi trường cấu hình Redis:
       - Host là tên service `redis` (không dùng `localhost`).
       - Port, timeout, connection timeout, v.v.
     - Cập nhật tag image từ `s9` sang `s10` giống các service khác.
   - Sao chép file `docker-compose` đã chỉnh sửa sang profile `qa` và `default`.
   - Không thay đổi gì trong `common-config.yml`.

4. Khởi chạy hệ thống bằng Docker:
   - Ở profile `prod`, chạy `docker compose up -d`.
   - Chờ 1–2 phút, tất cả container khởi động thành công.
   - Kiểm tra log của Gateway Server để xác nhận ứng dụng đã start.

5. Kiểm thử RateLimiter:
   - Gọi API: `easy-bank-accounts Java` (API tài khoản, nơi đã cấu hình RateLimiter).
   - Gửi nhiều request trong 1 giây:
     - Khi vượt ngưỡng, nhận response từ fallback của RateLimiter (ví dụ trả về `Java 17`).
   - Dùng Apache Benchmark để test:
     - Gửi 10 request, 8 bị từ chối.
     - Nhiều response trả về HTTP 429 (Too Many Requests), chứng tỏ RedisRateLimiter hoạt động đúng trong môi trường Docker.

6. Các mẫu resilience khác:
   - Không test lại Retry, Circuit Breaker trong Docker vì:
     - Đã test kỹ trên môi trường local.
     - Muốn test thì phải cố tình thêm RuntimeException vào code rồi build lại image, điều này không cần thiết.

7. Mã nguồn:
   - Toàn bộ thay đổi được commit trong GitHub repository, nằm ở phần `section_10` (hoặc `_10`), có thể xem lại toàn bộ cấu hình và code.

8. Lời khuyên cuối:
   - Tác giả nhấn mạnh: đã hoàn thành phần về khả năng chịu lỗi và phục hồi của microservice.
   - Khuyến khích người học nghỉ ngơi, không cố học dồn vì nội dung nhiều, dễ kiệt sức.
   - Hẹn gặp lại ở phần tiếp theo.
