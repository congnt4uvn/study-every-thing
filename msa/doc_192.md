
Dưới đây là tóm tắt nội dung:

1. Khởi động hệ thống  
   - Chạy lần lượt: Config Server → Eureka Server → Account, Loan, CardsApplication → GatewayserverApplication.  
   - Mục tiêu: kiểm tra các thay đổi liên quan đến bảo mật (security) ở chế độ nền.

2. Kiểm tra các API GET (không yêu cầu bảo mật)  
   - Dùng Postman, trong collection “Gateway Server”.  
   - Gọi các API GET:
     - `/contact-info` của Account microservice  
     - API GET trong Cards (Java version)  
     - `build-info` trong Loan microservice  
   - Tất cả đều trả về phản hồi thành công mà không cần bất kỳ thông tin ủy quyền nào → xác nhận các API GET không bị bảo vệ.

3. Kiểm tra API POST bị bảo vệ  
   - Gọi API POST `create` (tạo tài khoản mới) trong Account microservice qua Gateway.  
   - Không gửi bất kỳ thông tin ủy quyền / token nào.  
   - Nhận lỗi 401 Unauthorized → chứng tỏ Gateway đã được bảo mật, API POST yêu cầu access token.

4. Lấy access token bằng OAuth2 trong Postman  
   - Trong tab Authorization, chọn:
     - Type: OAuth 2.0  
     - Grant Type: Client Credentials  
   - Cấu hình:
     - Token Name  
     - Access Token URL (giống URL token OpenID đã dùng trước đó)  
     - Client ID, Client Secret  
     - Scope  
     - Client Authentication: gửi thông tin đăng nhập trong body.  
   - Nhấn “Get New Access Token” → server trả về access token.  
   - Nhấn “Use Token” → Postman tự động thêm header:
     - `Authorization: Bearer <access_token>`  
   - Nhờ vậy không cần copy–paste token thủ công mỗi lần.

5. Gọi lại API POST với token hợp lệ  
   - Gửi lại request `create` với access token hợp lệ → nhận phản hồi thành công.  
   - Gửi lại lần nữa với cùng dữ liệu → nhận lỗi “invalid request” vì khách hàng đã tồn tại (đã đăng ký).  
   - Thử sửa token (xóa vài ký tự) → gửi lại → nhận 401 Unauthorized → chứng minh Gateway/Resource Server đang kiểm tra và xác thực token đúng cách.

6. Lặp lại với các dịch vụ Cards và Loans  
   - Với API POST tạo thẻ mới (Cards) và API POST trong Loans:
     - Cấu hình OAuth 2.0 tương tự (token name, token URL, grant type, client id/secret, scope).  
     - Lấy access token mới, nhấn “Use Token”, gửi request → đều nhận phản hồi thành công.

7. Kiểm tra API GET tổng hợp (fetchCustomerDetails)  
   - Gọi API `fetchCustomerDetails()` để lấy thông tin tổng hợp Account, Loan, Card.  
   - Đây là API GET nên không yêu cầu xác thực.  
   - Gửi request → nhận phản hồi thành công.

Kết luận:  
- Các API GET vẫn mở (không yêu cầu bảo mật).  
- Các API POST qua Gateway đã được bảo vệ bằng OAuth2 (client credentials).  
- Gateway/Resource Server xác thực access token đúng, trả 401 khi token thiếu hoặc sai.  
- Postman được cấu hình để tự động lấy và gắn access token vào header Authorization (Bearer).
