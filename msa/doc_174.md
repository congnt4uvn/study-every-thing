
Dưới đây là tóm tắt nội dung bài giảng:

1. Dừng môi trường hiện tại  
   - Chạy `docker compose down` để dừng toàn bộ các container đang chạy.

2. Thêm OpenTelemetry vào microservice (ví dụ: account-service)  
   - Mở `pom.xml` của dịch vụ tài khoản.  
   - Khai báo biến phiên bản `otelVersion` (ví dụ: `1.27.0`).  
   - Thêm dependency cho Java agent của OpenTelemetry:
     - `groupId`: `io.opentelemetry.javaagent`  
     - `artifactId`: `opentelemetry-javaagent`  
     - `version`: dùng biến `otelVersion`  
     - `scope`: `runtime` (chỉ cần khi chạy, không cần lúc compile).  
   - Làm mới Maven để tải dependency.  
   - Lưu ý: jar của OpenTelemetry sẽ nằm trong classpath và được đóng gói vào Docker image.

3. Cấu hình logging để gắn trace/span vào log  
   - Mở `application.yml` của microservice.  
   - Thêm cấu hình `logging.pattern.level` (hoặc tương tự) để định nghĩa log pattern tùy chỉnh.  
   - Pattern chứa:
     - `%5p` để hiển thị mức độ log (INFO, DEBUG, WARN, ERROR) với độ dài 5 ký tự.  
     - Trong dấu ngoặc vuông `[...]` thêm:
       - `spring.application.name` (metadata về tên ứng dụng).  
       - `traceId` (ID trace do OpenTelemetry tạo).  
       - `spanId` (ID span).  
   - Kết quả: mọi log được sinh ra sẽ tự động kèm thông tin trace/span theo pattern này.

4. Áp dụng cấu hình cho tất cả microservices  
   - Các thay đổi về OpenTelemetry (pom + logging pattern) được áp dụng cho tất cả dịch vụ vi mô.

5. Cập nhật các controller để dùng log mới (thay vì correlationId thủ công)  
   - Trong `CustomerController`:
     - Thêm log: “fetchCustomerDetails() method start” ở đầu method.  
     - Thêm log: “fetchCustomerDetails() method end” trước `return`.  
     - Xóa việc dùng `relativeId`/correlationId thủ công, vì traceId/spanId đã được OTel gắn tự động.  
   - Trong `LoansController`:
     - Tương tự, thêm log “fetchLoanDetails() method start/end”.  
   - Trong `CardsController`:
     - Tương tự, thêm log “fetchCardDetails() method start/end”.  
   - Kết quả: mọi log ở controller/service/DAO đều có traceId/spanId theo pattern đã cấu hình.

6. Giải thích vai trò của OpenTelemetry và hệ sinh thái Grafana  
   - OpenTelemetry (OTel):
     - Framework quan sát (observability) mã nguồn mở, không phụ thuộc nhà cung cấp.  
     - Tự động gắn bytecode vào ứng dụng để thu thập trace, metrics, logs.  
     - Sinh ra traceId, spanId và metadata khác trong quá trình chạy.  
   - Tempo:
     - Thành phần trong hệ sinh thái Grafana, chuyên lưu trữ và lập chỉ mục trace.  
     - Tương tự như:
       - Loki cho logs  
       - Prometheus cho metrics  
     - Là giải pháp lưu trữ trace có khả năng mở rộng, chi phí thấp.  
   - Grafana:
     - UI chung kết nối tới nhiều nguồn dữ liệu (Loki, Prometheus, Tempo, …).  
     - Khi kết nối với Tempo, Grafana hiển thị trace phân tán dưới dạng giao diện trực quan.  
     - Quan trọng cho developer/platform/ops để xem toàn bộ luồng request, trace, logs, metrics.

7. Hướng đi tiếp theo  
   - Đã hoàn tất phần tích hợp OpenTelemetry trong code và logging.  
   - Bước tiếp theo:
     - Build lại Docker image cho tất cả microservices với thay đổi mới (OTel).  
     - Cập nhật `docker-compose` để:
       - Thêm dịch vụ Tempo.  
       - Tích hợp Tempo với Grafana.  
   - Hướng dẫn cho người học:
     - Xóa các Docker image cũ.  
     - Build lại image với tag mới, ví dụ: `S11`.  
   - Bài giảng tiếp theo sẽ demo việc tạo image và cấu hình Tempo + Grafana để xem trace phân tán.
