
Nội dung trên nói về lý do và cách tài liệu hóa REST API trong AccountsMicroservice bằng OpenAPI và Springdoc:

1. **Vì sao phải tài liệu hóa REST API?**  
   - Khi API được dùng bởi team UI, mobile, bên thứ ba, QA…, họ sẽ cần biết:
     - Định dạng request, response.
     - Các validation áp dụng.
   - Nếu không có tài liệu chuẩn, bạn phải liên tục giải thích qua meeting, email… rất mất thời gian và dễ sai sót.
   - Tài liệu chuẩn giúp người khác:
     - Hiểu nhanh cách hoạt động API.
     - Cấu hình hạ tầng, tạo client code, server stub, test case tự động.

2. **OpenAPI là gì?**  
   - Là chuẩn/đặc tả mở để mô tả API HTTP (REST API).
   - Cho phép mô tả API theo cách chuẩn hóa, để bất kỳ ai cũng có thể đọc và sử dụng.

3. **Springdoc OpenAPI**  
   - Thư viện hỗ trợ Spring Boot tự động sinh tài liệu OpenAPI và giao diện Swagger UI.
   - Website: `springdoc.org` hướng dẫn chi tiết cách tích hợp.

4. **Thêm dependency vào Spring Boot**  
   - Với Spring Boot ≥ 3.0: dùng `springdoc-openapi-starter-webmvc-ui` (phiên bản phù hợp với Spring Boot 3.x).  
   - Với Spring Boot 2.x hoặc 1.x: dùng `springdoc-openapi v1.7.0`.
   - Thêm dependency vào `pom.xml`, load Maven changes, build lại và restart ứng dụng.

5. **Kết quả sau khi tích hợp**  
   - Truy cập `http://localhost:8080/swagger-ui/index.html`:
     - Tự động hiển thị danh sách tất cả REST API trong `AccountsController` (PUT, POST, GET, DELETE).
     - Cho biết:
       - Đường dẫn API.
       - Kiểu request body (JSON), các field cần gửi (name, email, mobileNumber, accountNumber, accountType, branchAddress…).
       - Các validation (bắt buộc, min/max length, pattern…) lấy từ các DTO như `CustomerDto`, `AccountsDto`.
       - Cấu trúc response (ví dụ `ResponseDto` với `statusCode`, `statusMessage`).
       - Tham số query (ví dụ `mobileNumber` cho API fetch và delete).
     - Phần “Schemas” liệt kê toàn bộ cấu trúc DTO request/response.
     - Có nút “Try it out” để test trực tiếp API từ Swagger UI.

6. **Lợi ích**  
   - Người dùng API (dev UI, mobile, QA, bên thứ ba) chỉ cần URL Swagger UI để:
     - Hiểu rõ API, field nào bắt buộc, format ra sao.
     - Thử gọi API trực tiếp.
   - Giảm rất nhiều trao đổi thủ công, tránh hiểu nhầm.

7. **Hạn chế hiện tại & bước nâng cao**  
   - Tài liệu tự sinh vẫn còn “thô”:
     - Hiển thị tên kỹ thuật (AccountsController, CustomerDto…).
     - Không có ví dụ dữ liệu (sample values).
   - Để tài liệu chuyên nghiệp, production-ready:
     - Cần dùng thêm các annotation của Springdoc OpenAPI để:
       - Đặt mô tả rõ ràng.
       - Thêm ví dụ request/response.
       - Ẩn bớt chi tiết kỹ thuật không cần thiết.
   - Đây sẽ là nội dung của bài giảng tiếp theo.
