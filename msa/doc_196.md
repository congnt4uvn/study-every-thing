
Dưới đây là tóm tắt ngắn gọn nội dung trên về **luồng Authorization Code** trong kiến trúc vi dịch vụ ngân hàng:

1. **Thành phần tham gia**  
   - Ứng dụng khách: Web UI hoặc mobile app.  
   - Người dùng cuối: sử dụng ứng dụng khách.  
   - Gateway (Spring Cloud Gateway): đóng vai trò **Resource Server** / API Gateway.  
   - Keycloak: **Authorization Server** (Identity Provider).  
   - Các microservice phía sau Gateway.

2. **Luồng tổng quát**  
   - Người dùng thao tác trên ứng dụng khách → cần dữ liệu từ hệ thống vi dịch vụ.  
   - Ứng dụng khách gọi API Gateway nhưng chưa có token → Gateway từ chối, yêu cầu có **access token**.  
   - Ứng dụng khách chuyển hướng người dùng đến trang đăng nhập của Keycloak (chỉ gửi **client_id**, không gửi client_secret).  
   - Người dùng đăng nhập, được xác thực và cấp quyền (consent) cho ứng dụng khách thay mặt mình truy cập tài nguyên.  
   - Keycloak trả về **authorization code** cho ứng dụng khách.  
   - Ứng dụng khách dùng **authorization code + client_id + client_secret** gọi Keycloak (ở hậu trường) để đổi lấy **access token**.  
   - Nếu thông tin hợp lệ, Keycloak cấp **access token** (thuộc về người dùng, không phải ứng dụng khách; ứng dụng khách được nhắc không lạm dụng dữ liệu người dùng).

3. **Gọi API qua Gateway**  
   - Ứng dụng khách dùng access token gọi các API bảo mật trên Spring Cloud Gateway.  
   - Gateway xác thực token bằng cách kết nối với Keycloak.  
   - Nếu token hợp lệ, Gateway chuyển tiếp request đến các microservice tương ứng.  
   - Microservice xử lý, trả kết quả về Gateway → Gateway trả về ứng dụng khách → ứng dụng khách hiển thị dữ liệu cho người dùng.

4. **Ý nghĩa**  
   - Luồng **Authorization Code** được dùng khi có **người dùng cuối** tham gia (không phải machine-to-machine).  
   - Bài giảng đã giải thích luồng này nhiều lần dưới các góc nhìn khác nhau và demo trên OAuth Playground.  
   - Bước tiếp theo là triển khai thực tế luồng này trong mạng vi dịch vụ của hệ thống ngân hàng.
