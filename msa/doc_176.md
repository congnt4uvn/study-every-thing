
Dưới đây là tóm tắt nội dung:

1. Người nói chuẩn bị demo về distributed tracing với OpenTelemetry, sử dụng Docker Compose profile prod để khởi động toàn bộ các container microservices.

2. Lúc đầu, một số dịch vụ (cards, accounts, loans) bị lỗi vì:
   - Ứng dụng trong container thực ra vẫn khởi động được, nhưng mất nhiều thời gian.
   - Ngay khi khởi động, ứng dụng phải load Java Agent (file jar 20–30MB) vào bộ nhớ, nên startup chậm.
   - Health check trong docker-compose chỉ cho phép 10 lần thử, mỗi lần cách nhau 10 giây, không đủ thời gian để dịch vụ chứng minh “healthy”.

3. Cách khắc phục:
   - Mở file docker-compose.
   - Tăng thời gian giữa các lần health check lên 20 giây.
   - Tăng số lần retry lên 20 cho tất cả microservices (accounts, loans, cards), cũng như config server và Eureka.
   - Gateway server không có health check nên không cần chỉnh.

4. Sau khi chỉnh:
   - Xóa toàn bộ container cũ.
   - Chạy lại `docker compose up -d`.
   - Chờ hơn 4 phút, tất cả container khởi động thành công (do thời gian health check dài hơn).

5. Kiểm tra trong container:
   - Vào container của account service, xem log:
     - Dòng đầu cho thấy Java Agent được load từ đường dẫn trong container.
   - Duyệt file trong container:
     - Thư mục `application/libs` chứa tất cả jar, trong đó có jar OpenTelemetry đúng như đã khai báo trong `pom.xml` và cấu hình Docker Compose.

6. Kiểm tra Gateway:
   - Gateway server khởi động thành công trên port 8072.

7. Kiểm thử API bằng Postman:
   - Gọi API tạo account → trả về 201, “account created successfully”.
   - Gọi API tạo card → card tạo thành công.
   - Gọi API tạo loan → loan tạo thành công.
   - Gọi API `fetchCustomerDetails` → trả về đầy đủ thông tin account, loan, card.

8. Bước tiếp theo:
   - Chuyển sang Grafana để xem dữ liệu tracing (sẽ được trình bày trong bài giảng tiếp theo).
