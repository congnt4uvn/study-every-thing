
Bài giảng mô tả một demo về **cân bằng tải phía máy khách (client‑side load balancing)** trong kiến trúc microservice dùng **Eureka**.

Nội dung chính:

1. **Tạo 2 instance cho microservice Loans**
   - Trong `docker-compose.yml`, sao chép cấu hình dịch vụ loans.
   - Đổi:
     - `service name` thành `loans1` (khác với loans cũ).
     - `container_name` thành `loans-ms1` (tên container phải duy nhất).
     - Cổng ánh xạ ra ngoài (host port) để tránh xung đột.
   - Chạy `docker compose up -d`.  
     Việc khởi động nhiều container làm CPU tăng cao, nên mất vài phút.

2. **Kiểm tra đăng ký với Eureka**
   - Mở Eureka Dashboard: ban đầu trống vì chưa có service nào chạy.
   - Sau khi các container khởi động:
     - Thấy các service: accounts, cards, loans.
     - Riêng loans có **2 instance** (thể hiện bằng số 2 trong ngoặc).
   - Truy cập `localhost:8070/eureka/apps`:
     - Thấy phần loans có 2 instance với thông tin chi tiết riêng.
     - Các service khác (accounts, cards) chỉ có 1 instance.
   → Xác nhận nhiều instance loans đã đăng ký với Eureka.

3. **Chuẩn bị dữ liệu để test cân bằng tải**
   - Tạo dữ liệu:
     - Gọi API tạo account.
     - Gọi API tạo card.
     - Gọi API tạo loan.
   - Lưu ý:
     - Mỗi instance loans dùng **database H2 riêng**.
     - Tác giả chỉ tạo loan qua **cổng 8090** (instance thứ nhất).
     - Không tạo loan qua **cổng 8091** (instance thứ hai).
   → Vì vậy, dữ liệu loan chỉ tồn tại ở một instance.

   - Trong môi trường production thực tế:
     - Thường dùng MySQL hoặc DB chung cho mọi instance.
     - Khi đó dữ liệu loan sẽ dùng chung, không bị lệch giữa các instance.

4. **Demo cân bằng tải phía máy khách**
   - Gọi API `fetchCustomerDetails` (từ service accounts).
   - Accounts service dùng thông tin từ Eureka và thư viện OpenFeign (OpenFeign Client Library) để:
     - Tự chọn instance loans (client‑side load balancing).
   - Do chỉ có loan ở instance 8090:
     - Lần gọi thứ nhất:
       - Request đến instance loans **không có dữ liệu** → trả về lỗi “không tìm thấy khoản vay”.
     - Lần gọi thứ hai:
       - Request được chuyển sang instance loans **có dữ liệu** → trả về đầy đủ thông tin loan + các thông tin khác.
     - Gọi nhiều lần:
       - Khi request đến instance có dữ liệu → thành công.
       - Khi đến instance không có dữ liệu → lỗi.
   → Điều này chứng minh **cân bằng tải đang diễn ra ở phía client** (service accounts), dựa trên thông tin đăng ký từ Eureka.

5. **Kết luận**
   - Bài giảng minh họa:
     - Service discovery (Eureka).
     - Service registration (các instance loans, accounts, cards).
     - Client‑side load balancing (qua OpenFeign).
   - Tác giả nhấn mạnh:
     - Hãy dành thời gian tiêu hóa các khái niệm này.
     - Sau phần này sẽ chuyển sang chủ đề **service discovery và load balancing phía server** khi triển khai lên **Kubernetes**.
