
Dưới đây là tóm tắt ngắn gọn nội dung trên:

1. Chuẩn bị mã nguồn  
- Sao chép mã của section_9 sang thư mục mới section_10 trong workspace microservices.  
- Mở section_10 bằng IntelliJ, load tất cả project Maven, chạy `clean build` và bật annotation processing cho Lombok.  
- Từ đây, mọi thay đổi về khả năng phục hồi (resiliency) sẽ thực hiện trong section_10.

2. Thêm Resilience4j Circuit Breaker vào Gateway  
- Mở `pom.xml` của Gateway Server.  
- Thêm dependency:  
  `spring-cloud-starter-circuitbreaker-reactor-resilience4j`  
  (vì Gateway dùng Spring WebFlux/Reactive).  
- Reload Maven.

3. Cấu hình Circuit Breaker trong Gateway  
- Mở class main của Gateway (GatewayApplication).  
- Trong cấu hình route, thêm filter `circuitBreaker` cho route trỏ đến account microservice.  
- Trong lambda config của filter:  
  - Gọi `setName("accountCircuitBreaker")` để đặt tên circuit breaker cho các route liên quan account service.

4. Cấu hình chi tiết trong `application.yml`  
Thêm cấu hình Resilience4j Circuit Breaker, ví dụ:

- `slidingWindowSize: 10`  
  → Circuit breaker sẽ theo dõi 10 request đầu tiên trước khi quyết định chuyển trạng thái (closed → open).

- `permittedNumberOfCallsInHalfOpenState: 2`  
  → Khi ở trạng thái half-open, chỉ cho phép 2 request thử vào account service để “test” lại.  
  → Dựa trên kết quả 2 request này, circuit breaker quyết định quay lại closed hay mở lại (open).

- `failureRateThreshold: 50` (50%)  
  → Nếu ≥ 50% request trong cửa sổ theo dõi bị lỗi, circuit breaker chuyển từ closed sang open.

- `waitDurationInOpenState: 10000` (10.000 ms = 10s)  
  → Khi đã open, nó sẽ chờ 10 giây trước khi chuyển sang half-open để cho phép một phần lưu lượng thử lại.

- Cấu hình `default` áp dụng cho mọi circuit breaker.  
  Nếu muốn cấu hình riêng cho từng circuit breaker, dùng tên cụ thể, ví dụ `accountCircuitBreaker` thay cho `default`.

5. Khởi động hệ thống  
- Khởi động lần lượt:  
  1) Config Server  
  2) Eureka Server  
  3) Accounts microservice (AccountsApplication)  
  4) Gateway Server  
- Không khởi động card và loan service vì demo chỉ dùng account service.

6. Kiểm tra bằng Eureka và Actuator  
- Mở Eureka Dashboard: xác nhận Accounts và Gateway đã đăng ký.  
- Mở `http://localhost:8072/actuator` để xem các endpoint.  
- Mở `http://localhost:8072/actuator/circuitbreakers`  
  → Lúc đầu danh sách circuit breaker trống.  
- Gọi API account qua Gateway, sau đó quay lại `circuitbreakers`  
  → Thấy `accountCircuitBreaker` với trạng thái `CLOSED`, số lỗi = 0.

- Mở `http://localhost:8072/actuator/circuitbreakerevents?name=accountCircuitBreaker`  
  → Xem các event (success/failure, state transition, …).  
  → Mỗi lần gọi API, sẽ có thêm event.

7. Demo hành vi Circuit Breaker (Closed → Open → Half-open → Closed)  
- Vào `AccountsController`, tìm API `contact-info`.  
- Đặt breakpoint trong method xử lý `contact-info` và không release breakpoint:  
  → Request đến account service sẽ bị “treo”, không trả về.

- Từ Postman, gọi API `easybank/accounts/api/contact-info` qua Gateway:  
  - Gateway chờ một thời gian rồi trả về lỗi `504 Gateway Timeout`.  
  - Circuit breaker ghi nhận các request thất bại.

- Gửi nhiều request liên tiếp:  
  - Ban đầu: lỗi 504 (gateway timeout).  
  - Khi tỷ lệ lỗi vượt `failureRateThreshold` (≥ 50% trong 10 request):  
    → Circuit breaker chuyển sang trạng thái `OPEN`.  
    → Từ lúc này, Gateway không gửi request xuống account service nữa.  
    → Gateway trả về lỗi `503 Service Unavailable` với thông báo “Upstream service is temporarily unavailable”.  
    → Điều này giúp Gateway không lãng phí tài nguyên/luồng chờ phản hồi từ service đang lỗi.

- Sau `waitDurationInOpenState` (10s):  
  - Circuit breaker chuyển sang `HALF_OPEN`.  
  - Cho phép số request giới hạn (2) đi xuống account service:  
    - Nếu các request này thành công → chuyển về `CLOSED`.  
    - Nếu tiếp tục thất bại → quay lại `OPEN`.

- Quan sát trên Actuator:  
  - `circuitbreakers`: trạng thái lần lượt `CLOSED` → `OPEN` → `HALF_OPEN` → `OPEN` hoặc `CLOSED`.  
  - `circuitbreakerevents`: các event state transition (closed_to_open, open_to_half_open, half_open_to_closed, …) và success/failure.

8. Đưa circuit breaker về trạng thái Closed  
- Xóa breakpoint trong `AccountsController`.  
- Gửi lại request từ Postman:  
  → Request được xử lý thành công.  
- Sau một số request thành công, circuit breaker chuyển từ `HALF_OPEN` về `CLOSED`.  
- Trên Actuator:  
  - `state: CLOSED`,  
  - Event log cho thấy transition từ half-open → closed.

9. Ý nghĩa  
- Circuit breaker giúp:  
  - Bảo vệ Gateway và các service khác khỏi bị “kéo chết” bởi một service chậm hoặc lỗi.  
  - Giải phóng tài nguyên (thread, connection) vì không phải chờ các request chắc chắn sẽ thất bại.  
  - Tự động thử lại sau một khoảng thời gian (half-open) để phục hồi khi service ổn định trở lại.  

Bài giảng này tập trung vào việc triển khai và quan sát Circuit Breaker Pattern trên Gateway Server với Resilience4j; bài tiếp theo sẽ tiếp tục mở rộng chủ đề resiliency.
