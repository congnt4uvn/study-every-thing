
Dưới đây là phần tóm tắt nội dung bài giảng:

1. **Mục tiêu nghiệp vụ (Business Scenario)**  
   - Khi Gateway Server nhận **một request từ bên ngoài**, nó phải:
     - Tạo **Correlation ID / Tracking ID** (một UUID ngẫu nhiên) nếu request chưa có.
     - Gửi **cùng một Correlation ID** đó đến tất cả các microservice mà request đi qua (Accounts, Loans, Cards, …).
   - Các microservice sẽ dùng Correlation ID này để:
     - Ghi log (logging) kèm theo Correlation ID.
     - Hỗ trợ việc truy vết (trace) khi có lỗi hoặc dữ liệu sai.
   - Khi trả response về cho client:
     - Gateway phải **thêm cùng Correlation ID** vào **response header** để client biết ID gắn với request đó.
     - Nhờ đó, khi khách hàng báo lỗi, developer có thể dùng Correlation ID để lần theo log và xác định request đã đi qua microservice nào, lỗi ở đâu.

2. **Cấu trúc gói và các lớp sẽ tạo**  
   - Tạo package: `com.easybytes.gatewayserver.filters`.
   - Trong package này tạo:
     1. `RequestTraceFilter`  
        - Nhiệm vụ: tạo Correlation ID / Tracking ID cho **request vào Gateway**.
     2. `ResponseTraceFilter`  
        - Nhiệm vụ: thêm Correlation ID vào **response gửi ra cho client**.
     3. `FilterUtility`  
        - Chứa **logic dùng chung** cho cả Request và Response filter:
          - Lấy Correlation ID từ request.
          - Set Correlation ID vào request/response.
          - Định nghĩa tên header (ví dụ: `easybank-correlation-id`).

3. **RequestTraceFilter – chi tiết hoạt động**  
   - Được đánh dấu bằng:
     - `@Component` để Spring nhận diện như một bean filter.
     - `@Order(1)` để đảm bảo filter này chạy **đầu tiên** trong chuỗi filter của Gateway.
   - Lớp này **implements `GlobalFilter`**:
     - `GlobalFilter` đảm bảo filter được áp dụng cho **mọi request** đi qua Gateway.
     - Phải override method `filter(ServerWebExchange exchange, GatewayFilterChain chain)`.
   - Về reactive:
     - Spring Cloud Gateway dựa trên **Spring WebFlux** (reactive), không dùng servlet truyền thống.
     - Do đó dùng `ServerWebExchange`, `Mono`, `Flux`, …
     - `filter` trả về `Mono<Void>` vì:
       - Không trả về dữ liệu cụ thể, chỉ tiếp tục chuỗi filter (`chain.filter(exchange)`).
   - Logic trong `filter()`:
     1. Lấy request từ `exchange`.
     2. Lấy headers của request.
     3. Dùng `FilterUtility` để **kiểm tra xem đã có header Correlation ID** (ví dụ `easybank-correlation-id`) hay chưa:
        - Nếu **đã có**:
          - Không tạo lại ID mới (tránh ghi đè ID đã có, hữu ích khi request bị redirect và quay lại Gateway).
          - Ghi log: “easybank-correlation-id found in RequestTraceFilter”.
        - Nếu **chưa có**:
          - Tạo Correlation ID mới bằng `UUID.randomUUID()`.
          - Gọi `FilterUtility.setCorrelationId(exchange, newId)` để set vào request header.
          - Ghi log: “easybank-correlation-id generated in RequestTraceFilter”.
     4. Cuối cùng, gọi `return chain.filter(exchange);` để tiếp tục chuỗi filter.

4. **FilterUtility – logic dùng chung**  
   - Chứa các phương thức tiện ích, ví dụ:
     - `getCorrelationId(ServerWebExchange exchange)`:
       - Đọc từ request headers.
       - Nếu có header `easybank-correlation-id` → trả về giá trị.
       - Nếu không → trả về `null`.
     - `setCorrelationId(ServerWebExchange exchange, String correlationId)`:
       - Tạo request mới với header `easybank-correlation-id` và giá trị là `correlationId`.
       - Gắn lại vào `exchange`.
   - Nhờ đó, cả RequestTraceFilter và ResponseTraceFilter dùng chung logic này.

5. **ResponseTraceFilter – chi tiết hoạt động**  
   - Được định nghĩa theo cách khác so với RequestTraceFilter:
     - Dùng `@Configuration` trên class.
     - Bên trong class, tạo một method trả về `GlobalFilter` và đánh dấu `@Bean`.
       - Ví dụ: `@Bean public GlobalFilter postGlobalFilter() { ... }`
   - Inject `FilterUtility` vào ResponseTraceFilter.
   - Trong method `postGlobalFilter()`:
     - Trả về một `GlobalFilter` bằng biểu thức lambda:
       - Nhận `exchange` và `chain`.
       - Gọi `chain.filter(exchange)` rồi `.then(...)` để chạy logic **sau khi** microservice xử lý xong và trả response (post filter).
   - Logic chính:
     1. Sau khi nhận response từ microservice:
        - Lấy request headers từ `exchange`.
        - Dùng `FilterUtility.getCorrelationId(exchange)` để lấy Correlation ID đã được set trước đó.
     2. Set cùng Correlation ID đó vào **response header**:
        - Tên header: `easybank-correlation-id` (giống với request).
        - Giá trị: Correlation ID lấy từ request.
     3. Ghi log: “Updated the correlation id to the outbound headers”.

   - Như vậy, ResponseTraceFilter hoạt động như **post filter**:
     - Chặn response trước khi gửi ra client.
     - Thêm header Correlation ID vào response.

6. **Logging (ghi log) cho các filter**  
   - Tổng cộng có **3 loggers**:
     - 2 logger trong `RequestTraceFilter` (khi tìm thấy hoặc tạo Correlation ID).
     - 1 logger trong `ResponseTraceFilter` (khi thêm Correlation ID vào response).
   - Tất cả log này ở **mức DEBUG**.
   - Để in được log DEBUG, cấu hình trong `application.yml`:
     ```yaml
     logging:
       level:
         com.easybytes.gatewayserver: DEBUG
     ```
   - Ý nghĩa:
     - Bất kỳ log nào thuộc package `com.easybytes.gatewayserver` ở mức DEBUG sẽ được in ra.

7. **Bước tiếp theo (trong bài giảng sau)**  
   - Gateway đã:
     - Tạo Correlation ID cho request.
     - Truyền Correlation ID đến các microservice.
     - Thêm Correlation ID vào response trả về client.
   - Việc cần làm tiếp:
     - Chỉnh sửa từng microservice (Accounts, Loans, Cards, …):
       - Đọc header `easybank-correlation-id` từ request.
       - Dùng giá trị này để ghi log trong logic nghiệp vụ của từng service.
   - Người học được khuyến nghị:
     - Hiểu kỹ toàn bộ code trong các filter.
     - Nếu chưa rõ, xem lại video.
     - Có thể tham khảo và copy code từ GitHub repository của giảng viên.

Nếu bạn muốn, tôi có thể tóm tắt lại ngắn hơn nữa hoặc vẽ sơ đồ luồng request/response với Correlation ID.
