
Dưới đây là tóm tắt nội dung bài giảng:

1. Giảng viên demo về **mạng Docker**:
   - Trước giờ tất cả container và microservice đều chạy trong cùng một mạng Docker tên **EazyBank**.
   - Các dịch vụ DB microservice đang `extends` một service `network-deploy-service`, nên mặc định chúng cùng chung mạng.

2. Demo tách DB khỏi mạng:
   - Giảng viên chỉnh file `common-config.yml` (profile `qa`) để **bỏ DB container ra khỏi mạng EazyBank**.
   - Đồng thời xóa biến môi trường liên quan đến **Spring RabbitMQ** vì không còn dùng.
   - Dừng toàn bộ container (`docker compose down`), chuyển sang thư mục `qa` và chạy `docker compose up -d`.

3. Kết quả khi DB không cùng mạng:
   - Các container DB (accounts DB, loans DB, cards DB) và config server **khởi động thành công**.
   - Các microservice (accounts, loans, cards) ban đầu ở trạng thái running, sau đó **bị exit** vì:
     - Không kết nối được tới DB.
     - Giao tiếp giữa microservice và DB **thất bại** do khác mạng.

4. Kiểm tra mạng Docker:
   - Dùng `docker network ls` để xem các mạng:
     - Có mạng `EazyBank` (profile `qa`) và các mạng khác, trong đó có `qa_default`.
   - Dùng `docker ps` lấy container ID của loans DB, rồi `docker inspect <containerId>`:
     - Thấy container DB đang gắn với network ID tương ứng `qa_default`.
     - Trong khi các microservice khác gắn với mạng EazyBank.
   - Kết luận: vì **hai mạng khác nhau**, nên container không thể giao tiếp → minh họa tính **cô lập mạng** của Docker.

5. Khôi phục cấu hình:
   - Dừng container (`docker compose down`).
   - Mở lại file cấu hình, **khôi phục** phần `extends network-deploy-service` cho DB microservice trong profile `qa`.
   - Sau khi khôi phục và chạy lại, tất cả container khởi động bình thường, giao tiếp thành công.

6. Lý do chuyển sang dùng H2 thay vì MySQL:
   - Hiện tại để chạy đầy đủ:
     - 3 container DB (accounts, loans, cards),
     - config server,
     - 3 microservice (accounts, loans, cards),
     - tổng cộng khoảng **7 container**.
   - Sau này sẽ thêm:
     - Eureka Server, Gateway Server, Grafana, v.v.
   - Chạy quá nhiều container sẽ làm **máy chậm**, đặc biệt với laptop RAM 8GB, gây khó debug và mất động lực.
   - Vì vậy, từ phần tiếp theo giảng viên sẽ:
     - Chuyển sang dùng **H2 in-memory** để giảm số container.
   - Tuy nhiên:
     - Nếu máy bạn đủ mạnh, bạn **vẫn có thể tiếp tục dùng MySQL**.
     - Học viên có thể tự chọn: dùng H2 hoặc MySQL.

Tóm lại, bài giảng minh họa:
- Cách Docker network cô lập các container và ảnh hưởng đến giao tiếp microservice–database.
- Cách kiểm tra mạng và container bằng `docker network ls`, `docker ps`, `docker inspect`.
- Lý do thực tế (hiệu năng máy) để tạm thời quay lại dùng H2 thay vì MySQL trong các bài học tiếp theo.
