
Dưới đây là tóm tắt nội dung:

1. Thiết lập REST API trong AccountsController  
   - Xóa API `sayHello` cũ, bắt đầu xây dựng API thực tế.  
   - Dùng `@RequestMapping(path = "/api", produces = MediaType.APPLICATION_JSON_VALUE)` để:
     - Đặt tiền tố chung `/api` cho tất cả endpoint trong controller.
     - Quy định kiểu dữ liệu trả về là JSON.  
   - Giải thích thêm: trong dự án thực tế có thể thêm version như `/api/v1`, nhưng ở đây không dùng.

2. Tạo API tạo tài khoản mới  
   - Tạo method `createAccount()` trong controller:
     - Kiểu trả về: `ResponseEntity<ResponseDto>`.
     - Tham số: `@RequestBody CustomerDto customerDto` – dữ liệu client gửi trong body sẽ được map vào `CustomerDto`.
   - Gắn `@PostMapping("/create")`:
     - Endpoint đầy đủ: `POST /api/create`.
     - Client phải gửi dữ liệu theo cấu trúc `CustomerDto` (tên, email, số điện thoại, …).

3. Sử dụng ResponseEntity và ResponseDto  
   - Dùng `ResponseEntity` để:
     - Gửi HTTP status (ví dụ `HttpStatus.CREATED` – 201).
     - Gửi body là `ResponseDto` (chứa `statusCode`, `statusMsg`).
     - Có thể thêm header, contentType, metadata khác.  
   - Nếu chỉ trả về `ResponseDto` mà không dùng `ResponseEntity`, client sẽ không nhận được HTTP status rõ ràng.

4. Tạo lớp hằng số AccountsConstants  
   - Tạo package `constants`, lớp `AccountsConstants`.  
   - Khai báo các hằng số `public static final` như:
     - `STATUS_201`, `MESSAGE_201`, …  
   - Có constructor `private` để:
     - Không cho tạo object của lớp này.
     - Lớp chỉ dùng để chứa hằng số.  
   - Quy ước đặt tên hằng số: toàn chữ in hoa, nhiều từ thì nối bằng `_`.

5. Phân tách logic controller và service  
   - Controller:
     - Nhận request, trả response, thực hiện validation cơ bản.
   - Service:
     - Chứa toàn bộ logic nghiệp vụ (business logic).  
   - Tạo package `service`, trong đó:
     - Tạo interface `IAccountsService`.
       - Phương thức: `void createAccount(CustomerDto customerDto);`
       - Viết JavaDoc mô tả mục đích, tham số, giúp người khác dễ hiểu khi bảo trì.

6. Tạo lớp triển khai service  
   - Trong `service.impl`, tạo `AccountServiceImpl`:
     - Gắn `@Service` để Spring tạo bean service.  
     - Implement `IAccountsService` và override `createAccount`.  
   - Dùng Lombok:
     - Gắn `@AllArgsConstructor` để Lombok tạo constructor với tất cả field:
       - `AccountsRepository accountsRepository;`
       - `CustomerRepository customerRepository;`
     - Nhờ có constructor duy nhất, Spring tự động autowire mà không cần `@Autowired` trên field.

7. Sử dụng repository và mapper  
   - Để lưu dữ liệu vào H2 DB:
     - Repository cần entity (Customer, Accounts), không phải DTO.  
   - Cần logic chuyển đổi giữa DTO và entity:
     - Tạo package `mapper`.
     - Tạo `AccountsMapper`:
       - Phương thức tĩnh `mapToAccountDto(Accounts accounts, AccountsDto accountsDto)`:
         - Copy dữ liệu từ entity `Accounts` sang `AccountsDto`.
       - Phương thức tĩnh `mapToAccounts(AccountsDto accountsDto, Accounts accounts)`:
         - Copy dữ liệu từ DTO sang entity.  
     - Tạo `CustomerMapper` với hai phương thức tương tự cho `Customer` và `CustomerDto`.

8. Thảo luận về thư viện mapping tự động  
   - Có các thư viện như:
     - `ModelMapper`
     - `MapStruct`  
   - Ưu điểm: tự động map DTO ↔ entity, chỉ cần thêm dependency và vài dòng code.  
   - Tuy nhiên:
     - Không phải dependency “chính thức” của Spring.
     - Có thể bị từ chối trong dự án thực tế vì lo ngại bảo mật, kiểm soát, hoặc chính sách khách hàng.  
   - Tác giả chọn cách viết mapper thủ công:
     - Kiểm soát hoàn toàn logic mapping.
     - Ví dụ: muốn che số điện thoại, chỉ hiển thị 4 số cuối, có thể viết logic này trong `CustomerMapper`.
     - Một số thư viện mapping tự động có thể không hỗ trợ linh hoạt kiểu này.

9. Kết  
   - Bài giảng này dừng ở:
     - Thiết kế API tạo tài khoản.
     - Tạo cấu trúc controller, service, repository, mapper, constants.
   - Bài tiếp theo sẽ tiếp tục viết chi tiết logic nghiệp vụ tạo tài khoản trong H2 DB.
