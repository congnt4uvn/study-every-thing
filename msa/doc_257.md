
Dưới đây là tóm tắt nội dung:

1. Mục tiêu  
   - Thiết lập **Discovery Server** và **server-side load balancing** bên trong cụm Kubernetes.  
   - Sử dụng dự án **Spring Cloud Kubernetes** thay cho việc tự xây Eureka Server như trước.

2. Nguồn tham khảo  
   - Dựa trên một blog chính thức của **Spring Cloud Kubernetes** (năm 2021) hướng dẫn cách triển khai Discovery Server trong Kubernetes.  
   - Blog cung cấp sẵn **file manifest Kubernetes** để triển khai.

3. Chuẩn bị cấu trúc thư mục & file  
   - Trong workspace, tạo thư mục `section_17`.  
   - Bên trong tạo thư mục `kubernetes`.  
   - Tạo file manifest: `kubernetes-discoveryserver.yaml`.  
   - Dán nội dung manifest từ blog vào file này.

4. Phân tích nội dung manifest  

   a. Phần đầu  
   - `apiVersion: v1`, `kind: List` → trong `items` có thể khai báo nhiều đối tượng Kubernetes.

   b. Đối tượng Service  
   - `kind: Service`, tên: `spring-cloud-kubernetes-discoveryserver`.  
   - `spec.ports`:  
     - Port bên ngoài: `80`.  
     - Target port (trong container): `8761`.  
   - `type: ClusterIP` → chỉ dùng nội bộ cluster, không xung đột với Keycloak đang dùng port 80 với LoadBalancer.

   c. ServiceAccount + RoleBinding  
   - Tạo **ServiceAccount**: `spring-cloud-kubernetes-discoveryserver`.  
   - Tạo **RoleBinding** để gán một **Role** cho ServiceAccount này.  
   - Trong `subjects` chỉ rõ `kind: ServiceAccount` và tên ServiceAccount.  
   - `roleRef` trỏ tới Role có tên `namespace-reader`.

   d. Role (namespace-reader)  
   - `kind: Role`, tên: `namespace-reader`.  
   - Chỉ định các quyền đọc (verbs: `get`, `list`, `watch`) trên một số tài nguyên.  
   - Bản gốc blog chỉ cấp quyền cho `services` và `endpoints`.  
   - Cập nhật thêm: cần cấp quyền đọc cho **pods** (do thay đổi thực tế hiện nay).

   e. Deployment  
   - `kind: Deployment`.  
   - Metadata + labels cho app.  
   - `spec.template.spec.serviceAccountName` trỏ tới ServiceAccount đã tạo.  
   - Container:  
     - `name`: discovery server container.  
     - `image`: image Discovery Server do Spring Cloud Kubernetes cung cấp.  
       - Blog dùng tag cũ `2.1.0-M3`.  
       - Tác giả lên Docker Hub, tìm tag ổn định mới hơn: **`3.0.4`** và thay vào.  
     - `imagePullPolicy: IfNotPresent`.  
   - Health checks:  
     - Định nghĩa **readinessProbe** và **livenessProbe** dùng các endpoint actuator:  
       - readiness: `/actuator/health/readiness`  
       - liveness: `/actuator/health/liveness`  
     - Đây là mẫu chuẩn để khai báo probe trong Kubernetes.

5. Điều chỉnh về readiness/liveness probe  
   - Nếu giữ cấu hình mặc định (interval ngắn, ví dụ 10–20 giây), Discovery Server có thể chưa kịp khởi động → Kubernetes liên tục restart pod → vòng lặp restart.  
   - Để tránh: thêm cấu hình:  
     - `initialDelaySeconds: 100`  
       - Kubernetes đợi 100 giây trước khi thực hiện probe lần đầu.  
     - `periodSeconds: 30`  
       - Sau khi sẵn sàng, cứ 30 giây kiểm tra lại.  
   - Nếu 100 giây vẫn chưa đủ, có thể tăng lên 150–200 giây.  
   - Cần cấu hình tương tự cho cả readiness và liveness probe.

6. Lý do không cấu hình probe cho mọi microservice  
   - Kubernetes có khả năng tự động restart pod nhiều lần nếu gặp lỗi.  
   - Trong Docker Compose, container lỗi sẽ không tự restart như vậy, nên trước đây cần healthcheck kỹ hơn.  
   - Với Kubernetes, tác giả chấp nhận không định nghĩa readiness/liveness cho từng microservice, nhưng với Discovery Server thì cần cấu hình cẩn thận để tránh vòng lặp restart.

7. Kết luận  
   - Manifest này (sau khi chỉnh sửa: thêm quyền đọc pods, cập nhật image tag, cấu hình `initialDelaySeconds` và `periodSeconds`) sẽ được dùng để triển khai Discovery Server trong cụm Kubernetes.  
   - Bài giảng tiếp theo sẽ thực hiện triển khai thực tế Discovery Server bằng manifest này.
