
Dưới đây là tóm tắt ngắn gọn nội dung:

1. Triển khai lần lượt các microservice lên Kubernetes bằng các file manifest:
   - 1_keycloak.yaml (Keycloak)
   - 2_configmaps.yaml (ConfigMap)
   - 3_configserver.yaml (Config Server – đổi image từ S14 → S12)
   - Eureka Server
   - 5_accounts.yaml (Accounts)
   - 6_loans.yaml (Loans)
   - 7_cards.yaml (Cards)
   - Gateway Server

2. Giải thích hành vi `kubectl apply`:
   - Nếu manifest không đổi, Kubernetes báo “no changes” và không làm gì thêm.
   - Nếu có thay đổi (ví dụ đổi image), Deployment sẽ được cập nhật, Service không đổi nếu manifest Service không đổi.

3. Quy trình kiểm tra:
   - Dùng Kubernetes Dashboard để xem Deployment, Pod, log.
   - Đảm bảo từng dịch vụ (Config Server, Eureka, các microservice) khởi động thành công trước khi triển khai dịch vụ phụ thuộc (đặc biệt là Gateway).

4. Lưu ý về dependency trong Kubernetes:
   - Khác với Docker Compose (có healthcheck và dependency), Kubernetes không có cơ chế đơn giản để định nghĩa thứ tự phụ thuộc giữa Service.
   - Nếu dịch vụ phụ thuộc chưa sẵn sàng, container có thể restart nhiều lần rồi mới ổn định.
   - Thực tế production: DevOps/Ops dùng các kỹ thuật nâng cao để triển khai theo thứ tự; trong khóa học chỉ tập trung vào xây dựng microservice.

5. Cấu hình Keycloak:
   - Truy cập Keycloak (port 7080), đăng nhập admin/admin.
   - Tạo client `easybank-callcenter-cc`:
     - Bật “Client authentication”.
     - Tắt “Direct access grants”.
     - Bật “Service account roles”.
   - Lấy client secret, dùng trong Postman để lấy access token.
   - Tạo các role: `accounts`, `cards`, `loans`.
   - Gán các role này cho service account của client.

6. Kiểm thử bằng Postman:
   - Lấy access token từ Keycloak bằng client secret.
   - Gọi API tạo account, card, loan với token tương ứng → nhận phản hồi thành công.
   - Gọi API `fetchCustomerDetails` của Accounts:
     - Lần đầu sai số điện thoại → “customer not found”.
     - Điều chỉnh đúng số điện thoại, tạo lại dữ liệu (accounts/loans/cards) với token mới.
     - Gọi lại `fetchCustomerDetails` → nhận đủ thông tin account, loan, card → xác nhận toàn bộ hệ thống hoạt động đúng trong Kubernetes.

7. Hướng dẫn về file manifest Kubernetes:
   - Slides cung cấp mẫu:
     - Tạo ConfigMap.
     - Tạo Deployment (triển khai container).
     - Tạo Service (expose container).
   - Giải thích chi tiết từng trường trong manifest.
   - Cách liên kết Deployment và Service:
     - Dùng nhãn (label) và selector:
       - Ví dụ: Deployment có `metadata.labels.app: accounts`.
       - Service có `spec.selector.app: accounts`.
   - Hiện tại dùng Service type `ClusterIP` (không expose ra ngoài cluster).
   - Các bài sau sẽ giải thích thêm về `LoadBalancer`, `ClusterIP`, khi nào dùng loại nào.

8. Kết luận:
   - Mục tiêu: nắm vững cách viết và hiểu file manifest Kubernetes cơ bản để triển khai microservice.
   - Không cần trở thành chuyên gia viết manifest, nhưng phải hiểu được file do team DevOps cung cấp để có thể debug sự cố trong hệ thống microservice.
   - Các bài tiếp theo sẽ đi sâu hơn vào “sự kỳ diệu” của Kubernetes.
