
Dưới đây là tóm tắt ngắn gọn về cách **partition** trong Kafka ảnh hưởng đến **thứ tự (ordering)** và **key affinity**, kèm ví dụ code Spring Boot.

---

## 1. Partition ảnh hưởng gì đến Ordering?

- **Trong 1 partition**: Kafka **đảm bảo thứ tự** theo offset.  
  → Các message được đọc theo đúng thứ tự đã ghi.

- **Giữa nhiều partition**: **Không có đảm bảo thứ tự tổng thể**.  
  → Nếu một topic có nhiều partition, các message cùng topic nhưng khác partition có thể đến consumer **không theo thứ tự thời gian gửi**.

- **Ý nghĩa**:
  - Nếu bạn cần **thứ tự tuyệt đối** cho một luồng dữ liệu → phải đảm bảo **tất cả message liên quan nằm trong cùng một partition**.
  - Tăng số partition giúp **tăng throughput**, nhưng **giảm khả năng đảm bảo thứ tự toàn cục**.

---

## 2. Key Affinity là gì?

- Khi producer gửi message với **key**, Kafka dùng key để **hash → chọn partition**.
- **Cùng một key** → luôn đi vào **cùng một partition** (miễn là số partition không đổi).
- Điều này tạo ra **key affinity**:
  - Tất cả message của cùng một **userId**, **orderId**, v.v. sẽ được xử lý **tuần tự** trong cùng một partition.
  - Giúp đảm bảo **ordering theo key**, đồng thời vẫn có thể **scale** bằng nhiều partition (mỗi key là một “dòng” riêng).

---

## 3. Thiết kế thực tế

- Nếu cần:
  - **Ordering theo user** → dùng `userId` làm key.
  - **Ordering theo đơn hàng** → dùng `orderId` làm key.
- Không nên dùng key ngẫu nhiên nếu bạn cần **thứ tự theo một thực thể** (user, order, account…).

## 4. Ví dụ Spring Boot Kafka Producer

### 4.1. Cấu hình Producer (application.yml)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

### 4.2. Producer gửi message với key (đảm bảo key affinity)

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private static final String TOPIC = "orders";

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        // orderId là key → tất cả message của cùng orderId vào cùng 1 partition
        kafkaTemplate.send(TOPIC, orderId, payload);
    }
}
```

**Input ví dụ:**

```java
orderProducer.sendOrder("ORDER-123", "created");
orderProducer.sendOrder("ORDER-123", "paid");
orderProducer.sendOrder("ORDER-123", "shipped");
```

**Kết quả (trong Kafka):**

- Cả 3 message có key `"ORDER-123"` → cùng 1 partition.
- Consumer đọc theo thứ tự:
  1. `created`
  2. `paid`
  3. `shipped`

→ **Ordering theo `orderId` được đảm bảo**.

## 5. Ví dụ Consumer Spring Boot

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void consume(ConsumerRecord<String, String> record) {
        String key = record.key();      // orderId
        String value = record.value();  // trạng thái đơn hàng
        int partition = record.partition();
        long offset = record.offset();

        System.out.printf(
            "Received: key=%s, value=%s, partition=%d, offset=%d%n",
            key, value, partition, offset
        );
    }
}
```

**Output console (ví dụ):**

```text
Received: key=ORDER-123, value=created, partition=1, offset=10
Received: key=ORDER-123, value=paid, partition=1, offset=11
Received: key=ORDER-123, value=shipped, partition=1, offset=12
```

---

## 6. Tóm tắt ngắn

- **Ordering**:
  - Được đảm bảo **trong 1 partition**.
  - **Không đảm bảo** giữa các partition khác nhau.
- **Key affinity**:
  - Cùng key → cùng partition → **ordering theo key**.
- Thiết kế:
  - Chọn key theo thực thể cần đảm bảo thứ tự (userId, orderId, accountId…).
  - Tăng partition để scale, nhưng chấp nhận chỉ có **ordering cục bộ theo key**, không phải toàn topic.
