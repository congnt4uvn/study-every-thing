
Dưới đây là tóm tắt ngắn gọn về **Retention** và **Compaction** trong Kafka, tập trung vào:  
ảnh hưởng đến lưu trữ, hiệu năng, hành vi consumer, kèm ví dụ code Spring Boot.

---

## 1. Khái niệm chính

### Retention (giữ dữ liệu theo thời gian/kích thước)
- Kafka **xóa message cũ** dựa trên:
  - Thời gian: `retention.ms`
  - Kích thước: `retention.bytes`
- Không phụ thuộc vào việc consumer đã đọc hay chưa.
- Dùng cho:
  - Log sự kiện, tracking, audit, dữ liệu chỉ cần trong X ngày/giờ.

### Log Compaction (nén log theo key)
- Kafka **giữ lại bản ghi mới nhất cho mỗi key**, xóa bản ghi cũ hơn của cùng key.
- Không xóa toàn bộ dữ liệu cũ theo thời gian, mà theo **key**.
- Dùng cho:
  - Bảng trạng thái (state), cache, snapshot dữ liệu (ví dụ: trạng thái user, số dư tài khoản).
- Có thể kết hợp với retention theo thời gian.

---

## 2. Ảnh hưởng đến lưu trữ

### Retention
- **Retention lớn** → tốn nhiều disk hơn, nhưng có lịch sử dài hơn.
- **Retention nhỏ** → tiết kiệm disk, nhưng mất dữ liệu cũ nhanh.

### Compaction
- Giảm dung lượng bằng cách:
  - Chỉ giữ bản ghi mới nhất cho mỗi key.
- Tuy nhiên:
  - File log vẫn có thể lớn nếu có nhiều key khác nhau.
  - Compaction chạy nền, không phải lúc nào cũng “gọn” ngay lập tức.

---

## 3. Ảnh hưởng đến hiệu năng

### Retention
- Xóa theo segment (file) → tương đối hiệu quả.
- Retention quá dài:
  - Nhiều segment → tốn I/O, thời gian startup consumer (nếu đọc từ đầu).

### Compaction
- Compaction là quá trình nền:
  - Tốn CPU, I/O khi chạy.
  - Có thể ảnh hưởng throughput nếu cấu hình không hợp lý.
- Đổi lại:
  - Consumer mới có thể đọc ít dữ liệu hơn (chỉ bản ghi mới nhất cho mỗi key).

---

## 4. Ảnh hưởng đến hành vi consumer

### Với Retention
- Consumer:
  - Có thể đọc lại dữ liệu trong khoảng thời gian retention.
  - Nếu offset trỏ đến message đã bị xóa → consumer sẽ bị lỗi `OffsetOutOfRange` và phải reset offset (ví dụ: về earliest/latest).

### Với Compaction
- Consumer:
  - Khi đọc từ đầu topic compacted:
    - Thấy **chuỗi update** cho mỗi key, nhưng các bản ghi rất cũ có thể đã bị xóa.
  - Thường dùng cho:
    - Xây dựng lại state (materialized view) từ log compacted.
- “Tombstone” record (value = null):
  - Được dùng để đánh dấu xóa key.
  - Sau compaction, key đó sẽ biến mất khỏi log.

## 5. Ví dụ cấu hình topic Kafka

### Topic chỉ dùng retention theo thời gian

```bash
kafka-topics.sh --create \
  --topic events-log \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=604800000   # 7 ngày
  --config retention.bytes=10737418240  # 10 GB
```

### Topic dùng log compaction

```bash
kafka-topics.sh --create \
  --topic user-state \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.5 \
  --config segment.ms=3600000
```

### Kết hợp retention + compaction

```bash
kafka-topics.sh --create \
  --topic user-state-history \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact,delete \
  --config retention.ms=2592000000  # 30 ngày
```

## 6. Ví dụ Spring Boot (Producer/Consumer)

### 6.1. Cấu hình Kafka trong Spring Boot (`application.yml`)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
    consumer:
      group-id: user-state-consumer
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

---

### 6.2. Producer gửi trạng thái user (phù hợp topic compacted)

```java
@Service
public class UserStateProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private static final String TOPIC = "user-state";

    public UserStateProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    // Input: userId, state JSON
    public void sendUserState(String userId, String stateJson) {
        kafkaTemplate.send(TOPIC, userId, stateJson);
    }

    // Xóa user bằng tombstone (value = null)
    public void deleteUser(String userId) {
        kafkaTemplate.send(TOPIC, userId, null);
    }
}
```

**Ví dụ Input/Output:**

- Gọi:

```java
sendUserState("user-1", "{\"name\":\"An\",\"status\":\"ACTIVE\"}");
sendUserState("user-1", "{\"name\":\"An\",\"status\":\"INACTIVE\"}");
deleteUser("user-2");
```

- Trước compaction, log có thể là:

| offset | key    | value                                      |
|--------|--------|--------------------------------------------|
| 0      | user-1 | {"name":"An","status":"ACTIVE"}           |
| 1      | user-1 | {"name":"An","status":"INACTIVE"}         |
| 2      | user-2 | null (tombstone)                          |

- Sau compaction (đơn giản hóa):

| offset | key    | value                                      |
|--------|--------|--------------------------------------------|
| 1      | user-1 | {"name":"An","status":"INACTIVE"}         |
| (user-2 bị xóa) |

---

### 6.3. Consumer xây dựng lại state từ topic compacted

```java
@Service
public class UserStateConsumer {

    private final Map<String, String> userStateStore = new ConcurrentHashMap<>();

    @KafkaListener(topics = "user-state", groupId = "user-state-consumer")
    public void consume(ConsumerRecord<String, String> record) {
        String userId = record.key();
        String value = record.value();

        if (value == null) {
            // tombstone -> xóa khỏi state
            userStateStore.remove(userId);
        } else {
            userStateStore.put(userId, value);
        }
    }

    // Lấy state hiện tại của user
    public String getUserState(String userId) {
        return userStateStore.get(userId);
    }
}
```

**Hành vi:**

- Consumer đọc từ đầu topic:
  - Sau khi đọc hết, `userStateStore` chứa **trạng thái mới nhất** cho mỗi user.
- Nếu compaction đã chạy:
  - Consumer vẫn có thể reconstruct state từ log compacted.

---

## 7. Khi nào dùng cái gì?

- **Chỉ retention (delete)**:
  - Log sự kiện, analytics, audit, clickstream.
  - Không cần giữ trạng thái cuối cùng cho mỗi key.

- **Chỉ compaction**:
  - Bảng trạng thái, cấu hình, cache phân tán.
  - Cần luôn có “snapshot” mới nhất cho mỗi key.

- **Kết hợp retention + compaction**:
  - Vừa cần trạng thái cuối cùng, vừa muốn giữ lịch sử trong một khoảng thời gian.
