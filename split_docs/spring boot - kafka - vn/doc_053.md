## 1. Tối ưu `fetch` (fetch sizes)

Mục tiêu: Giảm độ trễ và tăng throughput bằng cách điều chỉnh lượng dữ liệu consumer lấy mỗi lần từ broker.

Các tham số quan trọng (Spring Boot / Spring Kafka):

```yaml
spring:
  kafka:
    consumer:
      properties:
        fetch.min.bytes: 50000        # tối thiểu 50KB mới trả về
        fetch.max.bytes: 52428800     # tối đa 50MB mỗi lần fetch
        fetch.max.wait.ms: 100        # chờ tối đa 100ms để đủ fetch.min.bytes
        max.partition.fetch.bytes: 1048576  # tối đa 1MB mỗi partition
```

- `fetch.min.bytes`:  
  - Thấp → độ trễ thấp, nhưng nhiều request nhỏ → tốn overhead.  
  - Cao → throughput tốt hơn, nhưng tăng độ trễ.
- `fetch.max.wait.ms`:  
  - Thời gian chờ để gom đủ `fetch.min.bytes`.  
  - Tăng giá trị → gom batch lớn hơn, throughput cao hơn, nhưng tăng latency.
- `max.partition.fetch.bytes`:  
  - Giới hạn dữ liệu mỗi partition trong một lần fetch → tránh consumer bị “ngộp” nếu message quá lớn.

**Ví dụ input/output (logic):**

- Input: 10.000 message/giây, mỗi message ~1KB  
- Cấu hình:
  - `fetch.min.bytes = 50000` (50KB)  
  - `fetch.max.wait.ms = 100` ms  

→ Consumer sẽ cố gắng gom khoảng 50 message (50KB) trước khi trả về, hoặc chờ tối đa 100ms.  
Kết quả: ít request hơn, throughput cao hơn, nhưng mỗi message có thể chậm thêm tối đa ~100ms.

## 2. Tối ưu concurrency (số thread consumer)

Mục tiêu: Tận dụng nhiều CPU core và xử lý song song nhiều partition.

Trong Spring Kafka, concurrency thường được cấu hình ở `ConcurrentKafkaListenerContainerFactory` hoặc trực tiếp trên `@KafkaListener`.

### 2.1. Cấu hình bằng `ConcurrentKafkaListenerContainerFactory`

```java
@Configuration
@EnableKafka
public class KafkaConsumerConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);
        factory.setConcurrency(3); // 3 thread consumer
        factory.getContainerProperties().setPollTimeout(3000);

        return factory;
    }
}
```

### 2.2. Cấu hình concurrency trên `@KafkaListener`

```java
@Component
public class MyKafkaConsumer {

    @KafkaListener(
        topics = "orders",
        groupId = "order-consumer-group",
        concurrency = "3"   // 3 thread cho listener này
    )
    public void consume(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                + " | Received: " + message);
    }
}
```

**Input/output minh họa:**

- Topic `orders` có 6 partition: `orders-0` … `orders-5`
- `concurrency = 3`

→ Spring Kafka tạo 3 thread consumer trong cùng group:

- Thread-1: đọc từ partition 0,1  
- Thread-2: đọc từ partition 2,3  
- Thread-3: đọc từ partition 4,5  

Khi gửi 6 message (mỗi partition 1 message), log có thể như:

```text
Thread: consumer-0-C-1 | Received: order-0
Thread: consumer-0-C-2 | Received: order-1
Thread: consumer-0-C-3 | Received: order-2
Thread: consumer-0-C-1 | Received: order-3
Thread: consumer-0-C-2 | Received: order-4
Thread: consumer-0-C-3 | Received: order-5
```

## 3. Partition assignment (cách phân chia partition cho consumer)

Mục tiêu: Đảm bảo phân phối đều partition giữa các consumer, giảm rebalancing không cần thiết, và tối ưu locality.

### 3.1. Chiến lược assignment

Kafka hỗ trợ nhiều chiến lược, ví dụ:

- `RangeAssignor` (mặc định): phân chia theo dải (range).
- `RoundRobinAssignor`: phân chia vòng tròn, thường cân bằng hơn.
- `StickyAssignor`: cố gắng giữ assignment ổn định, giảm rebalancing.

Cấu hình trong Spring Boot:

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor
```

Hoặc nhiều chiến lược (Kafka sẽ dùng theo thứ tự):

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor,
          org.apache.kafka.clients.consumer.RoundRobinAssignor
```

### 3.2. Ảnh hưởng đến performance

- `RoundRobinAssignor`:  
  - Phù hợp khi có nhiều topic/partition, muốn phân phối đều giữa consumer.  
- `StickyAssignor`:  
  - Giảm số lần rebalancing → ít gián đoạn, tốt cho hệ thống cần ổn định.

**Ví dụ logic:**

- 1 topic `payments` với 4 partition: 0,1,2,3  
- 2 consumer trong cùng group: C1, C2  

- Với `RangeAssignor`:
  - C1: partition 0,1  
  - C2: partition 2,3  

- Với `RoundRobinAssignor`:
  - C1: partition 0,2  
  - C2: partition 1,3  

Nếu thêm consumer C3:

- `StickyAssignor` sẽ cố gắng giữ phần lớn assignment cũ, chỉ chuyển một số partition sang C3, giảm gián đoạn.

## 4. Kết hợp 3 yếu tố để tối ưu

1. **Fetch sizes**  
   - Tăng `fetch.min.bytes` và `fetch.max.wait.ms` để tăng throughput (batch lớn hơn).  
   - Giảm nếu cần latency thấp.

2. **Concurrency**  
   - `concurrency` ≈ số partition / số instance service.  
   - Không nên vượt quá số partition (vì mỗi partition chỉ có 1 consumer trong 1 group).

3. **Partition assignment**  
   - Dùng `StickyAssignor` hoặc `RoundRobinAssignor` để phân phối đều và ổn định.  
   - Thiết kế số partition đủ lớn để scale về sau (ví dụ: gấp 2–3 lần số consumer tối đa dự kiến).

## Ví dụ cấu hình tổng hợp (application.yml)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: order-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        fetch.min.bytes: 50000
        fetch.max.wait.ms: 100
        max.partition.fetch.bytes: 1048576
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor
```

Và listener:

```java
@Component
public class OrderConsumer {

    @KafkaListener(
        topics = "orders",
        groupId = "order-consumer-group",
        concurrency = "4"
    )
    public void consume(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                + " | Received order: " + message);
    }
}
```

---

Tóm lại:

- Điều chỉnh **fetch sizes** để cân bằng giữa latency và throughput.  
- Tăng **concurrency** (số thread consumer) để tận dụng nhiều partition và CPU.  
- Chọn **partition assignment strategy** phù hợp (thường là `StickyAssignor` hoặc `RoundRobinAssignor`) để phân phối đều và giảm rebalancing.
