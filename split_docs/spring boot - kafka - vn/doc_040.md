
Dưới đây là tóm tắt ngắn gọn về **Consumer Group Model** trong Kafka, cơ chế **Rebalancing** và các **chiến lược phân phối partition** (range, round-robin, sticky), kèm ví dụ Spring Boot.

---

## 1. Consumer Group Model

- **Consumer Group**: Tập các consumer cùng chia sẻ việc đọc dữ liệu từ một hoặc nhiều topic.
- Mỗi **partition** của topic chỉ được **một consumer trong group** đọc tại một thời điểm.
- Nếu có nhiều consumer hơn số partition, một số consumer sẽ **nhàn rỗi** (không nhận message).
- Nếu một consumer trong group chết hoặc join/leave group → Kafka sẽ **rebalance** để phân phối lại partition.

---

## 2. Rebalancing

**Rebalancing** là quá trình Kafka phân phối lại các partition cho các consumer trong cùng một group khi có thay đổi:

- Consumer mới join group.
- Consumer rời group (crash, shutdown).
- Thay đổi số partition của topic.
- Thay đổi cấu hình group.

Hệ quả:

- Trong lúc rebalance, consumer **tạm thời không nhận message**.
- Sau khi rebalance, mỗi consumer có thể được gán **tập partition mới**.

---

## 3. Chiến lược phân phối partition

Kafka hỗ trợ nhiều **Partition Assignment Strategy**. Ba loại chính:

### 3.1 RangeAssignor (Range)

- Mặc định trong nhiều trường hợp.
- Partition được chia theo **dải liên tục** cho từng consumer.
- Ví dụ:
  - Topic `t1` có 6 partition: `0..5`
  - Group có 2 consumer: `C1`, `C2`
  - Range:
    - `C1`: partition `0,1,2`
    - `C2`: partition `3,4,5`
- Nếu có nhiều topic, mỗi topic được chia range riêng, có thể dẫn đến **mất cân bằng** giữa các consumer.

### 3.2 RoundRobinAssignor (Round-robin)

- Phân phối partition theo kiểu **xoay vòng** giữa các consumer.
- Ví dụ:
  - Topic `t1` có 6 partition: `0..5`
  - Group có 2 consumer: `C1`, `C2`
  - Round-robin:
    - `C1`: `0,2,4`
    - `C2`: `1,3,5`
- Thường **cân bằng hơn** khi có nhiều topic/partition.

### 3.3 StickyAssignor (Sticky)

- Mục tiêu:
  - **Cân bằng** số partition giữa các consumer.
  - **Giữ ổn định** phân phối cũ tối đa (hạn chế thay đổi không cần thiết).
- Khi rebalance:
  - Cố gắng giữ consumer tiếp tục xử lý các partition cũ.
  - Chỉ di chuyển partition khi cần để cân bằng.
- Giảm **dao động** và **gián đoạn** trong hệ thống.

## 4. Cấu hình trong Spring Boot (Spring Kafka)

### 4.1 Cấu hình `application.yml`

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: my-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        # Chọn chiến lược phân phối partition
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor
        # Các lựa chọn khác:
        # org.apache.kafka.clients.consumer.RangeAssignor
        # org.apache.kafka.clients.consumer.RoundRobinAssignor
```

### 4.2 Consumer đơn giản với @KafkaListener

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class MyKafkaConsumer {

    @KafkaListener(topics = "demo-topic", groupId = "my-consumer-group")
    public void listen(String message) {
        System.out.println("Received message: " + message);
    }
}
```

## 5. Ví dụ luồng Input/Output

### 5.1 Setup

- Topic: `demo-topic`
  - 3 partition: `0, 1, 2`
- Consumer group: `my-consumer-group`
- 2 instance service (2 consumer cùng group).

### 5.2 Với StickyAssignor

**Input (Producer gửi):**

```bash
# Partition 0
key=user1, value="A"
key=user1, value="B"

# Partition 1
key=user2, value="C"

# Partition 2
key=user3, value="D"
```

**Lần đầu khởi động:**

- `Consumer-1`: partition `0,1`
- `Consumer-2`: partition `2`

**Output (log):**

- Consumer-1:
  - `Received message: A`
  - `Received message: B`
  - `Received message: C`
- Consumer-2:
  - `Received message: D`

**Khi thêm Consumer-3 vào group:**

- StickyAssignor cố gắng:
  - Giữ `Consumer-1` vẫn có partition `0`
  - Chia lại `1` và `2` cho `Consumer-2` và `Consumer-3` sao cho cân bằng.

Ví dụ sau rebalance:

- `Consumer-1`: partition `0`
- `Consumer-2`: partition `1`
- `Consumer-3`: partition `2`

Phân phối thay đổi **ít nhất có thể** nhưng vẫn cân bằng.

---

## 6. Khi nào dùng chiến lược nào?

- **RangeAssignor**:
  - Đơn giản, mặc định.
  - Phù hợp khi số topic ít, số partition không quá lệch.
- **RoundRobinAssignor**:
  - Cần **cân bằng** tốt giữa các consumer, nhiều topic/partition.
- **StickyAssignor** (thường khuyến nghị):
  - Cần vừa **cân bằng**, vừa **ổn định** phân phối.
  - Hạn chế gián đoạn khi rebalance (ít thay đổi partition giữa các consumer).

---
