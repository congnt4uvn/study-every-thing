
Dưới đây là tóm tắt ngắn gọn về “Multi-environment patterns: Backward-compatible deployments and rolling upgrades” trong bối cảnh Kafka + Spring Boot, kèm ví dụ code và input/output minh họa.

---

## 1. Ý tưởng chính

1. **Nhiều môi trường**  
   - Thường có: `dev`, `staging`, `prod`.  
   - Mỗi môi trường có cụm Kafka riêng hoặc topic riêng (ví dụ: `orders-dev`, `orders-prod`).

2. **Triển khai tương thích ngược (backward-compatible)**  
   - Khi nâng cấp service (producer/consumer), **phiên bản mới phải đọc được dữ liệu cũ**.  
   - Tránh thay đổi schema phá vỡ consumer cũ (breaking change).  
   - Dùng schema registry (Avro/JSON Schema/Protobuf) và quy tắc tương thích (compatibility rules).

3. **Rolling upgrade**  
   - Triển khai dần dần: không dừng toàn bộ hệ thống.  
   - Ví dụ: nâng cấp từng instance của Spring Boot service, một phần producer/consumer mới chạy song song với phiên bản cũ.  
   - Kafka hỗ trợ tốt vì:  
     - Consumer group có thể có nhiều phiên bản cùng đọc.  
     - Producer mới vẫn ghi vào cùng topic với format tương thích.

4. **Chiến lược schema**  
   - Thêm field mới với default → **backward compatible**.  
   - Không xóa/đổi kiểu field đang dùng.  
   - Nếu cần thay đổi lớn → tạo **topic mới** hoặc **version mới của message** (ví dụ: `OrderCreatedV2`).

## 2. Ví dụ Spring Boot + Kafka

### 2.1. Cấu hình Kafka trong Spring Boot

```yaml
# application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: order-service
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "*"
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
```

---

### 2.2. Message cũ (v1) – tương thích ngược

```java
// v1
public class OrderEventV1 {
    private String orderId;
    private String status; // NEW, PAID, SHIPPED
    // getters/setters
}
```

Producer v1:

```java
@Service
public class OrderProducerV1 {

    private final KafkaTemplate<String, OrderEventV1> kafkaTemplate;

    public OrderProducerV1(KafkaTemplate<String, OrderEventV1> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderCreated(String orderId) {
        OrderEventV1 event = new OrderEventV1();
        event.setOrderId(orderId);
        event.setStatus("NEW");
        kafkaTemplate.send("orders", orderId, event);
    }
}
```

Consumer v1:

```java
@Service
public class OrderConsumerV1 {

    @KafkaListener(topics = "orders", groupId = "order-service-v1")
    public void handleOrder(OrderEventV1 event) {
        System.out.println("V1 received: " + event.getOrderId() + " - " + event.getStatus());
    }
}
```

**Input (message trên topic `orders`):**

```json
{
  "orderId": "123",
  "status": "NEW"
}
```

**Output (log consumer v1):**

```text
V1 received: 123 - NEW
```

### 2.3. Message mới (v2) – thêm field, vẫn đọc được dữ liệu cũ

```java
// v2 - thêm field mới có default
public class OrderEventV2 {
    private String orderId;
    private String status;
    private String customerEmail; // field mới, có thể null
    // getters/setters
}
```

Producer v2:

```java
@Service
public class OrderProducerV2 {

    private final KafkaTemplate<String, OrderEventV2> kafkaTemplate;

    public OrderProducerV2(KafkaTemplate<String, OrderEventV2> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderCreated(String orderId, String email) {
        OrderEventV2 event = new OrderEventV2();
        event.setOrderId(orderId);
        event.setStatus("NEW");
        event.setCustomerEmail(email);
        kafkaTemplate.send("orders", orderId, event);
    }
}
```

Consumer v2 (rolling upgrade: chạy song song với v1, group-id khác):

```java
@Service
public class OrderConsumerV2 {

    @KafkaListener(topics = "orders", groupId = "order-service-v2")
    public void handleOrder(OrderEventV2 event) {
        System.out.println("V2 received: " + event.getOrderId()
                + " - " + event.getStatus()
                + " - " + event.getCustomerEmail());
    }
}
```

**Input 1 (message cũ, không có `customerEmail`):**

```json
{
  "orderId": "123",
  "status": "NEW"
}
```

**Output consumer v2:**

```text
V2 received: 123 - NEW - null
```

**Input 2 (message mới, có `customerEmail`):**

```json
{
  "orderId": "124",
  "status": "NEW",
  "customerEmail": "user@example.com"
}
```

**Output consumer v2:**

```text
V2 received: 124 - NEW - user@example.com
```

→ V2 vẫn đọc được dữ liệu cũ (backward-compatible).

---

## 3. Quy trình rolling upgrade (tóm tắt)

1. Thiết kế schema mới **tương thích ngược** (chỉ thêm field, không phá vỡ field cũ).  
2. Deploy **consumer v2** (group-id mới) song song với v1.  
3. Deploy **producer v2** bắt đầu gửi message với schema mới.  
4. Khi v2 ổn định, dừng consumer v1 (hoặc giữ nếu cần).  
5. Lặp lại cho các môi trường: `dev` → `staging` → `prod`.
