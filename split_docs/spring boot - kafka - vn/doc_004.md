
Dưới đây là tóm tắt ngắn gọn về “Kafka Architecture: Cluster metadata – cách metadata được lưu trữ, truyền bá và cache trên client”, kèm ví dụ code (Java/Spring Boot) và input/output minh họa.

---

## 1. Khái niệm Cluster Metadata trong Kafka

**Cluster metadata** là tập thông tin mô tả trạng thái của Kafka cluster, ví dụ:

- Danh sách broker (ID, host, port)
- Danh sách topic, partition
- Leader/replica của từng partition
- Thông tin controller
- Cấu hình topic (số partition, replication factor, v.v.)

Client (producer/consumer) cần metadata để:
- Biết gửi message đến broker/partition nào
- Biết đọc message từ broker/partition nào
- Xử lý khi có thay đổi (broker down, leader thay đổi, topic mới, v.v.)

---

## 2. Metadata được lưu trữ ở đâu?

1. **Trên broker (cluster)**  
   - Metadata được lưu trong **Kafka controller** (một broker được bầu làm controller).  
   - Từ Kafka 2.x trở lên, metadata được lưu trong **Kafka’s internal metadata log** (trước đây dùng ZooKeeper, các phiên bản mới dùng KRaft mode – Kafka Raft).  
   - Controller quản lý:
     - Tạo/xóa topic
     - Gán partition cho broker
     - Leader election cho partition
     - Thay đổi cấu hình

2. **Trên ZooKeeper (các phiên bản cũ)**  
   - Lưu thông tin broker, topic, ACL, v.v.  
   - Controller đọc/ghi metadata vào ZooKeeper.

3. **Trên client (cache)**  
   - Client không lưu metadata vĩnh viễn, mà **cache tạm thời** trong bộ nhớ:
     - Map topic → partition → leader broker
     - Danh sách broker
   - Cache này được cập nhật định kỳ hoặc khi có lỗi.

---

## 3. Metadata được truyền bá (propagate) như thế nào?

1. **Từ controller đến các broker**  
   - Khi có thay đổi (topic mới, leader thay đổi, broker join/leave), controller gửi **UpdateMetadataRequest** đến các broker.  
   - Broker cập nhật metadata nội bộ để xử lý request từ client.

2. **Từ broker đến client**  
   - Client gửi **MetadataRequest** đến broker (thường là bootstrap broker hoặc broker bất kỳ).  
   - Broker trả về **MetadataResponse** chứa:
     - Danh sách broker
     - Thông tin topic/partition/leader
   - Client cập nhật cache metadata của mình.

3. **Khi nào client yêu cầu metadata mới?**
   - Lúc khởi tạo (bootstrap)
   - Khi gặp lỗi như:
     - `LEADER_NOT_AVAILABLE`
     - `UNKNOWN_TOPIC_OR_PARTITION`
     - `NOT_LEADER_OR_FOLLOWER`
   - Định kỳ theo cấu hình (metadata refresh interval)

## 4. Caching metadata trên client

- Client (producer/consumer) giữ một **cache metadata** trong bộ nhớ:
  - Giảm số lần gọi metadata đến broker
  - Tăng hiệu năng gửi/nhận message
- Cache được làm mới khi:
  - Hết hạn (timeout)
  - Gặp lỗi liên quan đến leader/partition
  - Topic mới được sử dụng lần đầu

### Một số config quan trọng (Java client)

```properties
# Thời gian tối đa giữ metadata trước khi bắt buộc refresh
metadata.max.age.ms=300000   # 5 phút

# Thời gian chờ response metadata
request.timeout.ms=30000

# Thời gian chờ kết nối bootstrap
bootstrap.servers=broker1:9092,broker2:9092
```

## 5. Ví dụ với Spring Boot + Spring Kafka

### 5.1. Cấu hình Producer/Consumer (application.yml)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      properties:
        metadata.max.age.ms: 300000   # 5 phút
    consumer:
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        metadata.max.age.ms: 300000
```

### 5.2. Producer đơn giản

```java
@Service
public class SimpleProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public SimpleProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    // Nếu leader thay đổi hoặc partition lỗi,
                    // client sẽ tự refresh metadata và retry (nếu cấu hình)
                    ex.printStackTrace();
                } else {
                    System.out.println("Sent to partition: " +
                        result.getRecordMetadata().partition() +
                        ", offset: " + result.getRecordMetadata().offset());
                }
            });
    }
}
```

**Input (gọi service):**

```java
simpleProducer.sendMessage("demo-topic", "user-1", "Hello Kafka");
```

**Output (log):**

```text
Sent to partition: 0, offset: 15
```

Ở đây, client đã dùng **metadata cache** để biết `demo-topic` có partition 0, leader là broker nào, rồi gửi message đến đúng broker.

### 5.3. Lấy metadata bằng AdminClient (Java thuần)

```java
Properties props = new Properties();
props.put(AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");

try (AdminClient admin = AdminClient.create(props)) {
    DescribeClusterResult cluster = admin.describeCluster();
    System.out.println("Cluster id: " + cluster.clusterId().get());
    System.out.println("Controller: " + cluster.controller().get());
    System.out.println("Nodes: " + cluster.nodes().get());

    DescribeTopicsResult topics = admin.describeTopics(List.of("demo-topic"));
    Map<String, TopicDescription> desc = topics.all().get();
    System.out.println(desc.get("demo-topic"));
}
```

**Output (ví dụ):**

```text
Cluster id: 7xk3JH2xQ2e3
Controller: localhost:9092 (id: 0 rack: null)
Nodes: [localhost:9092 (id: 0 rack: null)]
(demo-topic partitions: 3, replication-factor: 1
  partition: 0 leader: 0 replicas: [0] isr: [0]
  partition: 1 leader: 0 replicas: [0] isr: [0]
  partition: 2 leader: 0 replicas: [0] isr: [0])
```

Đây chính là **cluster metadata** mà client dùng để cache.

## 6. Tóm tắt ý chính

- **Cluster metadata** mô tả toàn bộ cấu trúc và trạng thái Kafka cluster (broker, topic, partition, leader, v.v.).
- Metadata được:
  - Lưu trên **controller** (và ZooKeeper/KRaft tùy phiên bản).
  - Truyền từ controller → broker → client thông qua các request/response metadata.
- Client (producer/consumer) **cache metadata** trong bộ nhớ để:
  - Giảm số lần gọi metadata
  - Tăng hiệu năng
  - Tự động refresh khi có lỗi hoặc hết hạn
- Trong Spring Boot/Spring Kafka:
  - Cấu hình `bootstrap-servers` để client có thể lấy metadata ban đầu.
  - Có thể điều chỉnh `metadata.max.age.ms` để kiểm soát tần suất refresh metadata.
  - AdminClient có thể dùng để xem metadata cluster và topic.
