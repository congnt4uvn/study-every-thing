
Dưới đây là tóm tắt ngắn gọn về **Retention & Compaction** trong Kafka, tập trung vào **retention theo thời gian** và **theo kích thước**, kèm ví dụ cấu hình (Kafka + Spring Boot).

---

## 1. Khái niệm chính

### Retention (giữ dữ liệu)
Kafka không xóa message ngay sau khi consumer đọc, mà giữ lại trong một khoảng:
- **Theo thời gian (time-based)**: giữ message trong X giờ/ngày.
- **Theo kích thước (size-based)**: giữ tối đa Y GB dữ liệu cho mỗi partition.

Khi vượt quá ngưỡng (thời gian hoặc kích thước), Kafka sẽ xóa các segment log cũ hơn.

### Compaction (log compaction – chỉ khi `cleanup.policy=compact` hoặc `compact,delete`)
- Kafka chỉ giữ **bản ghi mới nhất cho mỗi key**.
- Phù hợp cho:
  - Bảng trạng thái (state table)
  - Cấu hình, profile user, cache…

---

## 2. Retention theo thời gian

### Cấu hình topic (Kafka)

```bash
# Tạo topic với retention theo thời gian: 7 ngày
kafka-topics.sh --create \
  --topic my-time-retention-topic \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=604800000 \
  --config cleanup.policy=delete
```

- `retention.ms=604800000` → 7 ngày (7 * 24 * 60 * 60 * 1000 ms)
- Sau 7 ngày, các segment cũ hơn sẽ bị xóa (nếu không bị ràng buộc bởi size).

---

## 3. Retention theo kích thước

### Cấu hình topic (Kafka)

```bash
# Tạo topic với retention theo kích thước: 1 GB mỗi partition
kafka-topics.sh --create \
  --topic my-size-retention-topic \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.bytes=1073741824 \
  --config cleanup.policy=delete
```

- `retention.bytes=1073741824` → 1 GB
- Khi log của partition > 1 GB, Kafka xóa segment cũ nhất.

---

## 4. Kết hợp time-based & size-based

Kafka áp dụng **cả hai**:
- Xóa segment nếu:
  - Quá **retention.ms** *hoặc*
  - Vượt **retention.bytes**

```bash
kafka-configs.sh --alter \
  --bootstrap-server localhost:9092 \
  --entity-type topics \
  --entity-name my-topic \
  --add-config retention.ms=86400000,retention.bytes=536870912
```

- `retention.ms=86400000` → 1 ngày
- `retention.bytes=536870912` → 512 MB

---

## 5. Log compaction (tóm tắt)

```bash
# Topic với log compaction
kafka-topics.sh --create \
  --topic my-compacted-topic \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.5
```

- Kafka giữ **bản ghi mới nhất cho mỗi key**.
- Các bản ghi cũ hơn với cùng key sẽ bị compact (xóa dần).

## 6. Ví dụ Spring Boot (Producer/Consumer)

### 6.1. Cấu hình application.yml

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
    consumer:
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

---

### 6.2. Producer gửi message

```java
@Service
public class DemoProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public DemoProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value);
    }
}
```

**Input ví dụ:**

```java
demoProducer.sendMessage("my-time-retention-topic", "user-1", "first login");
demoProducer.sendMessage("my-time-retention-topic", "user-2", "second login");
```

**Output (trên Kafka):**
- Topic `my-time-retention-topic` nhận 2 record:
  - key=`user-1`, value=`first login`
  - key=`user-2`, value=`second login`
- Sau khi vượt thời gian/kích thước retention, các record cũ sẽ bị xóa.

---

### 6.3. Consumer đọc message

```java
@Service
public class DemoConsumer {

    @KafkaListener(topics = "my-time-retention-topic", groupId = "demo-group")
    public void listen(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Output console (khi có message):**

```text
Received: first login
Received: second login
```

## 7. Ví dụ log compaction với key

Giả sử topic `my-compacted-topic` có `cleanup.policy=compact`.

**Producer:**

```java
demoProducer.sendMessage("my-compacted-topic", "user-1", "name=A");
demoProducer.sendMessage("my-compacted-topic", "user-1", "name=B");
demoProducer.sendMessage("my-compacted-topic", "user-2", "name=C");
```

**Trước compaction (log đầy đủ):**
1. key=`user-1`, value=`name=A`
2. key=`user-1`, value=`name=B`
3. key=`user-2`, value=`name=C`

**Sau compaction (ảnh logical):**
- `user-1` → `name=B` (bản ghi mới nhất)
- `user-2` → `name=C`

Consumer mới join sau này có thể chỉ thấy bản ghi mới nhất cho mỗi key (tùy offset và trạng thái compaction).

---

## Tóm tắt ngắn

- **Retention theo thời gian** (`retention.ms`): giữ message tối đa X ms.
- **Retention theo kích thước** (`retention.bytes`): giữ tối đa Y bytes mỗi partition.
- Kafka xóa segment khi **quá thời gian** hoặc **vượt kích thước**.
- **Compaction** (`cleanup.policy=compact`): giữ bản ghi **mới nhất cho mỗi key**, phù hợp cho dữ liệu dạng “bảng trạng thái”.
- Có thể kết hợp `compact,delete` để vừa compaction vừa xóa theo retention.
