
Dưới đây là tóm tắt ngắn gọn về **Topics, Partitions, Ordering** trong Kafka, tập trung vào ảnh hưởng của **số lượng partition** đến **throughput, latency, và khả năng scale consumer**.

---

## 1. Topic & Partition

- **Topic**: “Tên” của luồng dữ liệu (ví dụ: `orders`, `payments`).
- **Partition**: Mỗi topic được chia thành nhiều partition.  
  - Mỗi partition là một log tuần tự, bất biến.
  - Dữ liệu trong **một partition** có **thứ tự** (ordering) đảm bảo.
  - **Giữa các partition khác nhau** thì **không có đảm bảo thứ tự**.

---

## 2. Ảnh hưởng của số lượng partition

### a) Throughput (tốc độ xử lý)

- **Nhiều partition hơn → throughput cao hơn**:
  - Producer có thể ghi song song vào nhiều partition.
  - Nhiều broker có thể chia tải.
- Nhưng:
  - Quá nhiều partition → tăng overhead (metadata, memory, network) → có thể giảm hiệu năng tổng thể.

### b) Latency (độ trễ)

- **Tăng partition** thường giúp:
  - Nhiều consumer xử lý song song → giảm thời gian chờ tổng thể.
- Tuy nhiên:
  - Quá nhiều partition → tăng thời gian rebalance, tăng chi phí I/O → có thể làm **tăng latency** trong một số trường hợp.

### c) Consumer scaling (khả năng scale consumer)

- **Số consumer tối đa trong một consumer group ≈ số partition**:
  - Mỗi partition chỉ được xử lý bởi **tối đa 1 consumer** trong cùng một group.
  - Nếu có 3 partition, bạn chỉ có thể tận dụng tối đa 3 consumer trong cùng group.
- Muốn scale thêm consumer → thường phải **tăng số partition**.

---

## 3. Ordering (thứ tự message)

- Kafka **chỉ đảm bảo thứ tự trong phạm vi 1 partition**.
- Nếu bạn cần **thứ tự theo key** (ví dụ: theo `userId`, `orderId`):
  - Hãy dùng **partition key** để đảm bảo cùng key luôn vào **cùng một partition**.
- Nếu bạn tăng số partition sau này:
  - Cách phân phối key → partition có thể thay đổi (trừ khi bạn kiểm soát hashing) → có thể ảnh hưởng đến ordering theo key.

---

## 4. Ví dụ code với Spring Boot + Kafka

### 4.1. Cấu hình topic với nhiều partition

```java
// Maven dependency (pom.xml)
// <dependency>
//   <groupId>org.springframework.kafka</groupId>
//   <artifactId>spring-kafka</artifactId>
// </dependency>

import org.apache.kafka.clients.admin.NewTopic;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic ordersTopic() {
        // Tạo topic "orders" với 6 partition, replication factor = 3
        return new NewTopic("orders", 6, (short) 3);
    }
}
```

**Ý nghĩa:**
- `orders` có **6 partition** → có thể chạy **tối đa 6 consumer** trong cùng một consumer group để xử lý song song.

---

### 4.2. Producer: gửi message với key để giữ thứ tự

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        // orderId là key → tất cả message cùng orderId sẽ vào cùng 1 partition
        kafkaTemplate.send("orders", orderId, payload);
    }
}
```

**Input ví dụ:**

```java
orderProducer.sendOrder("ORDER-123", "created");
orderProducer.sendOrder("ORDER-123", "paid");
orderProducer.sendOrder("ORDER-123", "shipped");
```

**Kết quả (trong 1 partition):**

- Partition P2 (ví dụ) sẽ chứa:
  1. `("ORDER-123", "created")`
  2. `("ORDER-123", "paid")`
  3. `("ORDER-123", "shipped")`

→ Consumer đọc từ P2 sẽ luôn thấy đúng thứ tự: `created → paid → shipped`.

---

### 4.3. Consumer: scale theo partition

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class OrderConsumer {

    @KafkaListener(
        topics = "orders",
        groupId = "order-service-group"
    )
    public void consume(String message) {
        System.out.println("Received: " + message);
        // Xử lý business logic...
    }
}
```

**Scale consumer:**

- Nếu `orders` có 6 partition:
  - Bạn có thể chạy **6 instance** của service `order-service` (cùng `groupId = "order-service-group"`).
  - Kafka sẽ phân chia mỗi partition cho 1 instance → xử lý song song.

**Ví dụ:**

- Partition 0 → Consumer instance #1  
- Partition 1 → Consumer instance #2  
- …  
- Partition 5 → Consumer instance #6  

Nếu bạn chạy **8 instance**:
- 2 instance dư sẽ **không nhận message** (vì chỉ có 6 partition).

---

## 5. Tóm tắt nhanh

- **Nhiều partition**:
  - + Tăng throughput (ghi/đọc song song).
  - + Cho phép nhiều consumer hơn trong cùng group.
  - – Tăng overhead, rebalance lâu hơn, có thể tăng latency nếu quá nhiều.
- **Ordering**:
  - Được đảm bảo **trong 1 partition**.
  - Dùng **key** (ví dụ `orderId`) để giữ thứ tự cho cùng một thực thể.
- **Consumer scaling**:
  - Số consumer hiệu quả trong 1 group ≤ số partition.
  - Muốn scale thêm → tăng partition (cân nhắc chi phí).
