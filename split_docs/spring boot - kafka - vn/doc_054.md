
Dưới đây là tóm tắt ngắn gọn về chủ đề: **Hiệu năng Consumer: Giám sát Consumer Lag trong Kafka (metrics, tools)**, kèm ví dụ code Spring Boot.

---

## 1. Consumer Lag là gì?

- **Consumer Lag** = `Last Produced Offset` − `Last Committed Offset` của consumer trong một partition.  
- Lag lớn nghĩa là:
  - Consumer xử lý chậm hơn tốc độ producer gửi dữ liệu.
  - Có nguy cơ trễ xử lý, backlog tăng, có thể mất SLA.

---

## 2. Tại sao cần giám sát Consumer Lag?

- Phát hiện sớm:
  - Consumer bị “đuối” (không theo kịp).
  - Lỗi cấu hình (max.poll.interval.ms, max.poll.records, …).
  - Thiếu tài nguyên (CPU, RAM, network).
- Quyết định:
  - Scale out thêm instance consumer.
  - Tối ưu logic xử lý, batch size, commit strategy.
  - Điều chỉnh partition, cấu hình Kafka.

---

## 3. Các metrics quan trọng

### a. Từ Kafka / Broker / Consumer Group

- `records-lag-max` (per partition / per consumer):
  - Số bản ghi tối đa mà consumer đang bị trễ so với log end.
- `records-lag`:
  - Lag hiện tại.
- `records-consumed-rate`:
  - Tốc độ đọc (records/second).
- `records-lag-avg`:
  - Lag trung bình.

### b. Từ Spring Kafka (Micrometer / Actuator)

Khi dùng Spring Boot + Spring Kafka + Micrometer, bạn có thể có metrics như:

- `kafka_consumer_records_lag_max`
- `kafka_consumer_records_lag`
- `kafka_consumer_records_consumed_total`
- `kafka_consumer_bytes_consumed_total`

---

## 4. Công cụ giám sát Consumer Lag

- **Kafka CLI**:
  - `kafka-consumer-groups.sh --bootstrap-server ... --describe --group <group-id>`
- **Kafka UI / Monitoring tools**:
  - Conduktor, Kafka UI, AKHQ, Kafdrop.
- **Observability stack**:
  - Prometheus + Grafana (scrape metrics từ Spring Boot / JMX).
  - Datadog, New Relic, Dynatrace, v.v.

## 5. Ví dụ: Spring Boot + Spring Kafka + Micrometer

### 5.1. `build.gradle` (hoặc `pom.xml` tương đương)

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.kafka:spring-kafka'
    implementation 'io.micrometer:micrometer-registry-prometheus'
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
}
```

### 5.2. Cấu hình Kafka Consumer

```yaml
# application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: demo-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: latest
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
```

### 5.3. Consumer đơn giản

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class DemoConsumer {

    @KafkaListener(topics = "demo-topic", groupId = "demo-consumer-group")
    public void consume(String message) {
        // Xử lý message
        System.out.println("Received: " + message);
        // Giả lập xử lý chậm để thấy lag tăng
        try {
            Thread.sleep(500); // 0.5s / message
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

**Input (Producer gửi):**

```bash
# Gửi 1000 message rất nhanh
kafka-console-producer.sh --broker-list localhost:9092 --topic demo-topic
>msg-1
>msg-2
...
>msg-1000
```

**Output (Consumer):**

- Log:
  - `Received: msg-1`
  - `Received: msg-2`
  - ...
- Lag sẽ tăng vì consumer xử lý 0.5s/message trong khi producer gửi rất nhanh.

## 6. Lấy metrics lag qua Prometheus

Sau khi chạy app, truy cập:

- `http://localhost:8080/actuator/prometheus`

Bạn sẽ thấy các dòng tương tự:

```text
# HELP kafka_consumer_records_lag_max The maximum lag in terms of number of records for any partition in this window.
# TYPE kafka_consumer_records_lag_max gauge
kafka_consumer_records_lag_max{client_id="consumer-demo-consumer-group-1",topic="demo-topic",partition="0",} 850.0

# HELP kafka_consumer_records_lag Current lag of the consumer
# TYPE kafka_consumer_records_lag gauge
kafka_consumer_records_lag{client_id="consumer-demo-consumer-group-1",topic="demo-topic",partition="0",} 850.0
```

- `records_lag_max` ~ 850 nghĩa là consumer đang chậm 850 message so với log end.

---

## 7. Ví dụ dùng CLI để xem lag

```bash
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group demo-consumer-group
```

**Output mẫu:**

```text
TOPIC       PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG   CONSUMER-ID           HOST            CLIENT-ID
demo-topic  0          150             1000            850   consumer-1-...        /192.168.1.10   consumer-demo-consumer-group-1
```

- `LAG = 850` → consumer đang bị trễ 850 bản ghi.

---

## 8. Cách xử lý khi lag cao

- Tăng số instance consumer (scale out).
- Tăng số partition (để tăng parallelism).
- Tối ưu code xử lý (batch, async, giảm I/O blocking).
- Điều chỉnh config:
  - `max.poll.records`, `fetch.min.bytes`, `fetch.max.wait.ms`, `max.poll.interval.ms`.
- Đảm bảo đủ tài nguyên (CPU, RAM, network, disk).

---

- Ví dụ dashboard Grafana query Prometheus để vẽ biểu đồ consumer lag.
- Ví dụ cấu hình alert khi lag vượt ngưỡng (Prometheus alerting rule).
