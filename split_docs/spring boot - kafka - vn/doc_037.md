
Dưới đây là tóm tắt ngắn gọn về chủ đề **“Hiệu năng Producer: Ảnh hưởng của số partition đến thông lượng (throughput)”** trong Kafka, kèm ví dụ Spring Boot.

---

## 1. Ý chính

1. **Số partition càng nhiều → khả năng song song càng cao → thông lượng producer có thể tăng**  
   - Mỗi partition là một “đường ống” ghi riêng biệt.  
   - Producer có thể gửi dữ liệu song song tới nhiều partition, tận dụng nhiều CPU core, nhiều broker.

2. **Nhưng tăng partition không phải lúc nào cũng tốt**  
   - Quá nhiều partition → overhead quản lý metadata, network, file handle, replication tăng.  
   - Có điểm “tối ưu” (optimal point): sau đó tăng partition nữa thì throughput tăng rất ít hoặc giảm.

3. **Yếu tố ảnh hưởng đến throughput producer**  
   - `acks` (0, 1, all):  
     - `acks=0`: nhanh nhất, nhưng dễ mất dữ liệu.  
     - `acks=1`: cân bằng giữa tốc độ và an toàn.  
     - `acks=all`: chậm hơn, an toàn hơn.  
   - `batch.size`, `linger.ms`:  
     - Batch lớn + linger cao → gộp nhiều record trong 1 request → throughput cao hơn, latency tăng.  
   - `compression.type` (snappy, lz4, zstd):  
     - Nén giúp giảm băng thông, tăng throughput nếu CPU đủ.  
   - `max.in.flight.requests.per.connection`:  
     - Cho phép nhiều request song song trên 1 connection.

4. **Mối quan hệ giữa partition và throughput**  
   - Ít partition: producer bị “nghẽn” vì không đủ đường song song.  
   - Tăng partition: throughput tăng đến khi:
     - CPU producer/broker bão hòa, hoặc  
     - Network bão hòa, hoặc  
     - Overhead quản lý partition quá lớn.  

5. **Cân bằng**  
   - Chọn số partition dựa trên:
     - Throughput mong muốn (MB/s, record/s).  
     - Số consumer (để scale đọc).  
     - Tài nguyên cluster (CPU, RAM, disk, network).

## 2. Ví dụ cấu hình Producer (Java / Spring Boot)

### 2.1. application.yml (Spring Boot Kafka)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      acks: all
      batch-size: 32768        # 32 KB
      linger-ms: 10            # chờ tối đa 10ms để gom batch
      compression-type: lz4
      properties:
        max.in.flight.requests.per.connection: 5
        enable.idempotence: true
```

### 2.2. Tạo topic với số partition khác nhau

```bash
# Topic với 3 partition
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic demo-topic-3 \
  --partitions 3 \
  --replication-factor 1

# Topic với 12 partition
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic demo-topic-12 \
  --partitions 12 \
  --replication-factor 1
```

## 3. Ví dụ Producer Spring Boot gửi nhiều message để đo throughput

### 3.1. Producer Service

```java
@Service
public class BenchmarkProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public BenchmarkProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessages(String topic, int count) {
        long start = System.currentTimeMillis();

        for (int i = 0; i < count; i++) {
            String key = "key-" + (i % 10);
            String value = "message-" + i;
            kafkaTemplate.send(topic, key, value);
        }

        kafkaTemplate.flush();
        long end = System.currentTimeMillis();

        double seconds = (end - start) / 1000.0;
        double throughput = count / seconds;

        System.out.printf("Topic=%s, Sent=%d messages in %.2f s, Throughput=%.2f msg/s%n",
                topic, count, seconds, throughput);
    }
}
```

### 3.2. Chạy benchmark với số partition khác nhau

```java
@Component
public class Runner implements CommandLineRunner {

    private final BenchmarkProducer producer;

    public Runner(BenchmarkProducer producer) {
        this.producer = producer;
    }

    @Override
    public void run(String... args) {
        int count = 500_000;

        producer.sendMessages("demo-topic-3", count);
        producer.sendMessages("demo-topic-12", count);
    }
}
```

### 3.3. Ví dụ kết quả (output giả định)

```text
Topic=demo-topic-3, Sent=500000 messages in 10.00 s, Throughput=50000.00 msg/s
Topic=demo-topic-12, Sent=500000 messages in 4.00 s, Throughput=125000.00 msg/s
```

- Khi tăng từ 3 → 12 partition, throughput tăng đáng kể.  
- Nếu tiếp tục tăng lên 100 partition, có thể throughput chỉ tăng nhẹ hoặc giảm do overhead.

---

## 4. Tóm tắt ngắn

- **Số partition nhiều hơn → tăng khả năng song song → thường tăng throughput producer.**  
- **Nhưng quá nhiều partition → overhead lớn → hiệu năng có thể giảm.**  
- Cần **tuning kết hợp**: `acks`, `batch.size`, `linger.ms`, `compression`, `max.in.flight` và **chọn số partition phù hợp** với mục tiêu throughput và tài nguyên cluster.
