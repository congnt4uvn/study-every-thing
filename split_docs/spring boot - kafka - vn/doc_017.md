
Dưới đây là tóm tắt ngắn gọn về chủ đề: **Topics, Partitions, Ordering & Rebalancing partitions across brokers** trong Kafka, kèm ví dụ Spring Boot.

---

## 1. Topic & Partition

- **Topic**: “Tên” của luồng dữ liệu (ví dụ: `orders`, `payments`).
- **Partition**: Mỗi topic được chia thành nhiều partition (P0, P1, P2…).  
  - Mỗi partition là một log **append-only**, có offset tăng dần.
  - Dữ liệu trong **một partition** được đảm bảo **thứ tự** (ordering).
  - Giữa **các partition khác nhau** thì **không có** đảm bảo thứ tự.

**Ý nghĩa**:
- Tăng khả năng **song song** (parallelism) khi có nhiều partition.
- Cho phép **scale**: nhiều broker, nhiều consumer.

---

## 2. Ordering (Thứ tự thông điệp)

- Kafka chỉ đảm bảo **ordering trong phạm vi 1 partition**.
- Nếu bạn cần tất cả message của cùng một “key” (ví dụ: `userId`) có thứ tự:
  - Dùng **message key** → Kafka sẽ hash key để quyết định partition.
  - Tất cả message cùng key sẽ vào **cùng một partition** → giữ được thứ tự.

**Ví dụ**:  
- `userId = 123` → luôn vào partition 0  
- `userId = 456` → luôn vào partition 1  

---

## 3. Rebalancing partitions across brokers

- **Rebalancing**: phân phối lại các partition giữa các broker.
- Xảy ra khi:
  - Thêm broker mới.
  - Broker cũ bị down hoặc bị remove.
  - Thay đổi số partition hoặc replication factor.
- Mục tiêu:
  - Cân bằng tải (load) giữa các broker.
  - Đảm bảo tính sẵn sàng (availability) và fault-tolerance.

**Ảnh hưởng**:
- Trong lúc rebalance, leader của một số partition có thể chuyển sang broker khác.
- Consumer group cũng có thể bị **rebalance** (phân lại partition cho các consumer).
- Có thể gây **tạm thời gián đoạn** nhỏ (short pause) trong việc consume.

## 4. Ví dụ Spring Boot + Kafka

### 4.1. Cấu hình Maven

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>
</dependencies>
```

### 4.2. application.yml

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
    consumer:
      group-id: order-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

### 4.3. Producer – gửi message với key để giữ ordering

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String userId, String orderJson) {
        // key = userId → tất cả order của 1 user vào cùng 1 partition
        kafkaTemplate.send("orders-topic", userId, orderJson);
    }
}
```

**Input** (ví dụ gọi từ REST):

```http
POST /orders
Content-Type: application/json

{
  "userId": "123",
  "orderId": "ORD-1",
  "amount": 100000
}
```

**Output** (log producer):

```text
Sent order for user 123 to topic orders-topic
```

### 4.4. Consumer – đọc theo group, chịu ảnh hưởng khi rebalance

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders-topic", groupId = "order-group")
    public void consume(String message,
                        @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
                        @Header(KafkaHeaders.OFFSET) long offset) {
        System.out.printf("Received message: %s, partition: %d, offset: %d%n",
                message, partition, offset);
    }
}
```

**Output** (khi chạy nhiều instance consumer):

```text
Instance 1: Received message: {...}, partition: 0, offset: 10
Instance 2: Received message: {...}, partition: 1, offset: 5
```

Khi thêm/bớt instance consumer, Kafka sẽ **rebalance consumer group**:
- Partition được phân lại cho các instance.
- Có thể thấy log “Revoking assignments” / “Assigned partitions” trong consumer.

---

## 5. Tóm tắt ý chính

1. **Topic** được chia thành nhiều **partition** để scale và xử lý song song.  
2. Kafka chỉ đảm bảo **thứ tự trong 1 partition**. Dùng **key** để đảm bảo message cùng key vào cùng partition.  
3. **Rebalancing partitions across brokers** giúp cân bằng tải và tăng độ sẵn sàng, nhưng có thể gây pause ngắn khi leader/assignment thay đổi.  
4. Trong Spring Boot, dùng `KafkaTemplate` để gửi message có key, `@KafkaListener` để consume; việc rebalance diễn ra tự động ở phía Kafka và consumer group.
