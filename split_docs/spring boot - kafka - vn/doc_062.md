
Dưới đây là tóm tắt ngắn gọn về các kịch bản lỗi trong Kafka, kèm ví dụ Spring Boot (Java) và mô phỏng input/output.

---

## 1. Broker Failure (Broker bị lỗi)

**Ý chính:**
- Một hoặc nhiều broker trong cluster bị down (crash, mất điện, OOM, v.v.).
- Kafka vẫn hoạt động nếu:
  - Có nhiều broker.
  - Topic được cấu hình `replication.factor > 1`.
  - Có đủ ISR (in-sync replicas).

**Hậu quả:**
- Partition leader có thể bị mất → controller sẽ bầu leader mới từ các replica còn sống.
- Nếu không còn replica nào đủ điều kiện → partition tạm thời không khả dụng (producer/consumer lỗi).

**Cách giảm thiểu:**
- Dùng replication factor ≥ 3 cho các topic quan trọng.
- Cấu hình `min.insync.replicas` + `acks=all` để đảm bảo dữ liệu không mất.

### Ví dụ Producer Spring Boot (Java)

```java
// application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092,localhost:9093,localhost:9094
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      properties:
        acks: all
        retries: 5
        delivery.timeout.ms: 120000
```

```java
@Service
public class DemoProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public DemoProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String key, String value) {
        kafkaTemplate.send("demo-topic", key, value)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    System.out.println("Send failed: " + ex.getMessage());
                } else {
                    System.out.println("Send success, offset=" +
                        result.getRecordMetadata().offset());
                }
            });
    }
}
```

**Input (gửi message):**
```java
demoProducer.sendMessage("user-1", "Hello Kafka");
```

**Output (khi 1 broker bị down nhưng còn leader mới):**
```text
Send success, offset=15
```

**Output (khi không còn leader khả dụng):**
```text
Send failed: org.apache.kafka.common.errors.NotLeaderOrFollowerException: ...
```

## 2. Controller Failure (Controller bị lỗi)

**Ý chính:**
- Controller là broker đặc biệt, quản lý:
  - Thay đổi metadata (topic, partition).
  - Bầu leader cho partition.
- Nếu controller bị down:
  - Các broker còn lại bầu một controller mới.
  - Trong thời gian bầu lại, có thể bị gián đoạn ngắn (leader election, update metadata).

**Hậu quả:**
- Tạm thời không thể tạo topic mới, thay đổi partition.
- Producer/consumer có thể gặp lỗi ngắn hạn (metadata error) rồi tự recover.

**Cách giảm thiểu:**
- Đảm bảo nhiều broker trong cluster.
- Giám sát controller và thời gian bầu lại (controller election time).

### Ví dụ Consumer Spring Boot (Java)

```java
// application.yml
spring:
  kafka:
    consumer:
      bootstrap-servers: localhost:9092,localhost:9093,localhost:9094
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        max.poll.interval.ms: 300000
        session.timeout.ms: 10000
```

```java
@Service
public class DemoConsumer {

    @KafkaListener(topics = "demo-topic", groupId = "demo-group")
    public void listen(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Input (message đã được gửi trước đó):**
```text
"Hello Kafka"
```

**Output bình thường:**
```text
Received: Hello Kafka
```

**Khi controller bị down rồi bầu lại:**
- Có thể tạm thời không nhận message trong vài giây.
- Sau đó consumer tự động tiếp tục:
```text
Received: Hello Kafka
```

## 3. Network Partitions (Phân hoạch mạng / mất kết nối mạng)

**Ý chính:**
- Một phần cluster bị tách khỏi phần còn lại (network split).
- Có thể xảy ra:
  - Producer/consumer không kết nối được tới broker.
  - Broker không replicate được dữ liệu cho nhau → replica rơi khỏi ISR.

**Hậu quả:**
- Producer có thể gặp lỗi timeout, `TimeoutException`, `NotEnoughReplicasException`.
- Consumer có thể không nhận được message, hoặc bị rebalancing liên tục.
- Nguy cơ mất dữ liệu hoặc ghi không nhất quán nếu cấu hình không chặt.

**Cách giảm thiểu:**
- Dùng `acks=all` + `min.insync.replicas` để tránh ghi khi không đủ replica.
- Cấu hình timeout hợp lý (`request.timeout.ms`, `retries`, `delivery.timeout.ms`).
- Thiết kế retry + DLQ (dead-letter topic) ở ứng dụng.

### Ví dụ Producer với xử lý lỗi mạng

```java
@Service
public class ReliableProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public ReliableProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendWithRetry(String key, String value) {
        kafkaTemplate.send("reliable-topic", key, value)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    // Log và có thể lưu vào DB / queue để retry sau
                    System.err.println("Network or broker error: " + ex.getClass().getSimpleName());
                } else {
                    System.out.println("Sent to partition " +
                        result.getRecordMetadata().partition() +
                        " offset " + result.getRecordMetadata().offset());
                }
            });
    }
}
```

**Input (khi mạng bị phân hoạch):**
```java
reliableProducer.sendWithRetry("user-2", "Order Created");
```

**Output (lỗi do network partition):**
```text
Network or broker error: TimeoutException
```

**Khi mạng ổn định lại và gửi lại:**
```text
Sent to partition 1 offset 42
```

## Tóm tắt ngắn gọn (Tiếng Việt)

- **Broker failure**: broker bị down → controller bầu leader mới; nếu không còn replica leader → partition không khả dụng. Giải pháp: replication factor cao, `acks=all`, `min.insync.replicas`.
- **Controller failure**: controller bị down → broker khác lên làm controller; có gián đoạn ngắn trong leader election và cập nhật metadata.
- **Network partitions**: mất kết nối giữa các phần của cluster → timeout, mất ISR, nguy cơ mất dữ liệu. Giải pháp: cấu hình an toàn (`acks=all`, `min.insync.replicas`), retry, DLQ, giám sát mạng.
