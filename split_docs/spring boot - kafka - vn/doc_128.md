## 1. Kích thước message (Message size)

**Ý chính:**
- Message càng lớn → tốn nhiều:
  - **Disk I/O** (ghi/đọc nhiều hơn)
  - **Network bandwidth** (truyền nhiều dữ liệu hơn)
  - **Memory** (buffer, page cache)
- Message quá lớn có thể:
  - Tăng độ trễ (latency)
  - Gây lỗi `RecordTooLargeException`
  - Làm giảm throughput tổng thể

**Hướng dẫn:**
- Ưu tiên **nhiều message nhỏ** thay vì ít message rất lớn.
- Điều chỉnh:
  - `message.max.bytes` (broker)
  - `max.request.size` (producer)
  - `fetch.max.bytes` (consumer)

**Ví dụ cấu hình Spring Boot (application.yml):**
```yaml
spring:
  kafka:
    producer:
      properties:
        max.request.size: 1048576   # 1 MB
    consumer:
      properties:
        fetch.max.bytes: 5242880    # 5 MB
```

**Ví dụ input/output:**

- Input: gửi message 5 MB, `max.request.size = 1 MB`  
- Output: Producer lỗi:
  ```text
  org.apache.kafka.common.errors.RecordTooLargeException: 
  The request included a message larger than the max message size the server will accept.
  ```

## 2. Nén dữ liệu (Compression)

**Ý chính:**
- Nén giúp:
  - Giảm **disk usage**
  - Giảm **network bandwidth**
  - Có thể tăng **throughput** (ít dữ liệu phải gửi)
- Nhưng:
  - Tốn **CPU** để nén/giải nén
- Các codec phổ biến: `gzip`, `snappy`, `lz4`, `zstd`
  - `gzip`: nén tốt, CPU cao
  - `snappy`, `lz4`: nhanh, nén vừa
  - `zstd`: cân bằng tốt (Kafka mới)

**Hướng dẫn:**
- Bật nén ở **producer** (thường hiệu quả nhất).
- Chọn codec phù hợp với trade-off CPU vs disk/network.

**Ví dụ cấu hình Spring Boot (producer compression):**
```yaml
spring:
  kafka:
    producer:
      properties:
        compression.type: lz4
```

**Ví dụ input/output:**

- Input: gửi 1 triệu message, mỗi message 1 KB
- Không nén:
  - Dữ liệu trên disk ~ 1 GB
- Có nén `lz4`:
  - Dữ liệu trên disk ~ 300–500 MB (tùy nội dung)
  - CPU sử dụng tăng, nhưng network/disk giảm

## 3. Retention (thời gian lưu trữ dữ liệu)

**Ý chính:**
- Retention quyết định **dung lượng disk cần thiết**.
- Hai kiểu chính:
  - Theo **thời gian**: `retention.ms` (ví dụ giữ 7 ngày)
  - Theo **dung lượng**: `retention.bytes` (ví dụ tối đa 500 GB)
- Retention càng dài → cần **nhiều disk** hơn.
- Cần tính toán:
  - `throughput (MB/s) * 3600 * 24 * số ngày * replication factor`

**Ví dụ tính nhanh:**
- Throughput: 10 MB/s
- Retention: 7 ngày
- Replication factor: 3

Dung lượng 1 replica:
- 10 MB/s * 3600 * 24 * 7 ≈ 6,048,000 MB ≈ 6 TB

Tổng cho 3 replica:
- ~ 18 TB

**Ví dụ cấu hình topic (CLI):**
```bash
kafka-topics.sh --create \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 6 \
  --replication-factor 3 \
  --config retention.ms=604800000 \
  --config retention.bytes=107374182400
```

**Ví dụ cấu hình qua Spring Kafka Admin:**
```java
@Bean
public NewTopic ordersTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("retention.ms", "604800000");        // 7 ngày
    configs.put("retention.bytes", "107374182400");  // 100 GB

    return TopicBuilder.name("orders")
            .partitions(6)
            .replicas(3)
            .configs(configs)
            .build();
}
```

## 4. Tác động lên phần cứng (hardware)

### CPU
- Bị ảnh hưởng bởi:
  - Nén/giải nén (compression)
  - Serialization/deserialization (JSON, Avro, Protobuf)
- Nhiều nén + message lớn → cần CPU mạnh hơn.

### Disk
- Bị ảnh hưởng bởi:
  - Kích thước message
  - Throughput
  - Retention
  - Replication factor
- Cần disk đủ lớn + IOPS tốt (thường SSD).

### Network
- Bị ảnh hưởng bởi:
  - Kích thước message
  - Throughput
  - Replication (mỗi message gửi đến nhiều broker)
- Nén giúp giảm băng thông.

### Memory
- Bị ảnh hưởng bởi:
  - Page cache (Kafka dùng OS cache)
  - Buffer của producer/consumer
  - Message lớn → cần buffer lớn hơn

## 5. Ví dụ Spring Boot Producer/Consumer đơn giản

**Producer gửi message JSON nhỏ, có nén:**

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payloadJson) {
        kafkaTemplate.send("orders", orderId, payloadJson);
    }
}
```

`application.yml`:
```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      properties:
        compression.type: lz4
        max.request.size: 1048576
```

**Consumer:**

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.println("Received orderId=" + record.key()
                + ", payload=" + record.value());
    }
}
```

**Input (producer gọi):**
```java
orderProducer.sendOrder("order-123", "{\"amount\": 100000, \"currency\": \"VND\"}");
```

**Output (consumer log):**
```text
Received orderId=order-123, payload={"amount": 100000, "currency": "VND"}
```

## 6. Tóm tắt ngắn

- **Message size**: giữ message nhỏ, cấu hình giới hạn kích thước để tránh lỗi và giảm latency.
- **Compression**: bật nén ở producer để giảm disk/network, chấp nhận tăng CPU.
- **Retention**: tính toán dung lượng disk dựa trên throughput × thời gian × replication; điều chỉnh `retention.ms` và `retention.bytes`.
- **Hardware**:
  - CPU: chịu tải nén/giải nén.
  - Disk: chịu tải retention + replication.
  - Network: chịu tải kích thước message + throughput.
  - Memory: chịu tải buffer và page cache.

Nếu bạn cho thêm thông tin về throughput dự kiến, kích thước message trung bình và thời gian retention mong muốn, tôi có thể giúp bạn ước lượng dung lượng disk và cấu hình cụ thể hơn.
