
Dưới đây là tóm tắt ngắn gọn về các pattern “Reliability” trong Kafka/Spring Boot: **Idempotent Consumer** và **Deduplication Store**, kèm ví dụ code và input/output.

---

## 1. Idempotent Consumer (Consumer bất biến)

**Ý tưởng chính**

- Consumer được thiết kế sao cho **xử lý cùng một message nhiều lần vẫn cho ra cùng một kết quả** (không gây tác dụng phụ thêm).
- Phù hợp khi Kafka có thể gửi lại message (retries, rebalancing, offset commit lỗi, v.v.).
- Thường dùng khi:
  - Ghi vào DB với `UPSERT` (insert nếu chưa có, update nếu đã có).
  - Kiểm tra trạng thái trước khi cập nhật (ví dụ: chỉ cập nhật nếu trạng thái hiện tại phù hợp).

**Ví dụ: Thanh toán chỉ được xử lý 1 lần**

- Mỗi message có `transactionId` duy nhất.
- Consumer kiểm tra DB xem `transactionId` đã xử lý chưa.

### Code ví dụ (Spring Boot + Kafka + JPA)

**Entity lưu trạng thái thanh toán**

```java
@Entity
@Table(name = "payments")
public class Payment {
    @Id
    private String transactionId;

    private BigDecimal amount;
    private String status; // e.g. "COMPLETED", "FAILED"

    // getters/setters
}
```

**Repository**

```java
public interface PaymentRepository extends JpaRepository<Payment, String> {
}
```

**Consumer idempotent**

```java
@Service
public class PaymentConsumer {

    private final PaymentRepository paymentRepository;

    public PaymentConsumer(PaymentRepository paymentRepository) {
        this.paymentRepository = paymentRepository;
    }

    @KafkaListener(topics = "payments", groupId = "payment-service")
    @Transactional
    public void consume(PaymentEvent event) {
        String txId = event.getTransactionId();

        // Nếu đã tồn tại => bỏ qua (idempotent)
        if (paymentRepository.existsById(txId)) {
            return;
        }

        Payment payment = new Payment();
        payment.setTransactionId(txId);
        payment.setAmount(event.getAmount());
        payment.setStatus("COMPLETED");

        paymentRepository.save(payment);
    }
}
```

**Input (message Kafka)**

```json
{
  "transactionId": "TX123",
  "amount": 100000
}
```

**Output (trong DB)**

Bản ghi trong bảng `payments`:

| transaction_id | amount  | status     |
|----------------|---------|-----------|
| TX123          | 100000  | COMPLETED |

Nếu message `TX123` được gửi lại lần nữa, consumer sẽ kiểm tra `existsById("TX123")` và **không tạo bản ghi mới** → kết quả không đổi.

## 2. Deduplication Store (Kho chống trùng lặp)

**Ý tưởng chính**

- Lưu lại danh sách các message (hoặc key) đã xử lý để **bỏ qua các message trùng lặp**.
- Có thể dùng:
  - DB (SQL/NoSQL)
  - Redis
  - In-memory cache (Caffeine, Guava) – chỉ phù hợp khi chấp nhận mất dữ liệu cache.
- Thường lưu: `messageId` hoặc `businessKey` + thời gian hết hạn (TTL).

**Luồng xử lý**

1. Consumer nhận message.
2. Kiểm tra trong dedup store:
   - Nếu **đã tồn tại** `messageId` → bỏ qua.
   - Nếu **chưa tồn tại** → lưu `messageId` vào store, sau đó xử lý business logic.

### Ví dụ: Dùng Redis làm dedup store

**Message**

```json
{
  "messageId": "MSG-001",
  "userId": 10,
  "points": 50
}
```

**Consumer**

```java
@Service
public class RewardConsumer {

    private final StringRedisTemplate redisTemplate;
    private final UserRewardService userRewardService;

    public RewardConsumer(StringRedisTemplate redisTemplate,
                          UserRewardService userRewardService) {
        this.redisTemplate = redisTemplate;
        this.userRewardService = userRewardService;
    }

    @KafkaListener(topics = "user-rewards", groupId = "reward-service")
    @Transactional
    public void consume(RewardEvent event) {
        String messageId = event.getMessageId();
        String key = "dedup:reward:" + messageId;

        // SETNX: chỉ set nếu chưa tồn tại
        Boolean success = redisTemplate.opsForValue().setIfAbsent(key, "1", Duration.ofHours(24));

        // Nếu key đã tồn tại => message trùng => bỏ qua
        if (Boolean.FALSE.equals(success)) {
            return;
        }

        // Xử lý business logic
        userRewardService.addPoints(event.getUserId(), event.getPoints());
    }
}
```

**Input**

Kafka topic `user-rewards` nhận 2 message giống nhau:

```json
{"messageId": "MSG-001", "userId": 10, "points": 50}
{"messageId": "MSG-001", "userId": 10, "points": 50}
```

**Output**

- Redis có key: `dedup:reward:MSG-001`.
- `userRewardService.addPoints(10, 50)` chỉ được gọi **1 lần**.
- Điểm của user 10 chỉ tăng 50, không phải 100.

## So sánh nhanh

| Pattern              | Mục tiêu chính                                  | Cách làm chính                                  |
|----------------------|--------------------------------------------------|-------------------------------------------------|
| Idempotent Consumer  | Xử lý lại nhiều lần nhưng kết quả không đổi     | Thiết kế logic/DB theo kiểu idempotent         |
| Deduplication Store  | Bỏ qua message trùng lặp trước khi xử lý        | Lưu `messageId` đã xử lý (DB/Redis/cache)      |

Trong thực tế, thường **kết hợp cả hai**:
- Dedup store để giảm xử lý thừa.
- Logic idempotent để an toàn khi dedup store không hoàn hảo (mất cache, race condition, v.v.).
