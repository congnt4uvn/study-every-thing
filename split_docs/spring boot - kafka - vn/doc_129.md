
Dưới đây là tóm tắt ngắn gọn về các cấu hình Kafka Broker liên quan đến tuning, kèm ví dụ cấu hình và giải thích:

---

## 1. `num.network.threads`
- **Ý nghĩa**: Số thread xử lý I/O mạng (nhận/gửi request từ client: producer, consumer).
- **Tác dụng**: Tăng khi có nhiều kết nối client, throughput cao.
- **Ví dụ cấu hình** (`server.properties`):
  ```properties
  num.network.threads=8
  ```
- **Khi nào tăng?**
  - Nhiều producer/consumer kết nối.
  - Thấy độ trễ cao ở network, CPU còn dư.

---

## 2. `num.io.threads`
- **Ý nghĩa**: Số thread xử lý I/O đĩa (đọc/ghi log, replication).
- **Tác dụng**: Tăng khi broker phải xử lý nhiều partition, nhiều topic, nhiều replica.
- **Ví dụ**:
  ```properties
  num.io.threads=16
  ```
- **Khi nào tăng?**
  - Nhiều partition trên broker.
  - Disk I/O là bottleneck, CPU còn dư.

---

## 3. `log.segment.bytes`
- **Ý nghĩa**: Kích thước tối đa của một segment log (file) trước khi Kafka tạo file mới.
- **Tác dụng**:
  - Segment nhỏ → nhiều file hơn, dễ xóa theo retention, nhưng overhead metadata cao.
  - Segment lớn → ít file hơn, nhưng xóa chậm hơn (phải chờ segment đầy).
- **Ví dụ**:
  ```properties
  log.segment.bytes=1073741824  # 1 GB
  ```
- **Khi nào chỉnh?**
  - Lưu lượng message lớn → tăng lên (512MB–1GB).
  - Cần xóa dữ liệu nhanh hơn → giảm xuống (128MB–256MB).

---

## 4. `log.retention.hours`
- **Ý nghĩa**: Thời gian giữ dữ liệu log trước khi xóa (tính theo giờ).
- **Tác dụng**:
  - Quyết định “window” dữ liệu có thể được consumer đọc lại.
  - Ảnh hưởng trực tiếp đến dung lượng disk cần thiết.
- **Ví dụ**:
  ```properties
  log.retention.hours=168  # 7 ngày
  ```
- **Khi nào chỉnh?**
  - Cần giữ dữ liệu lâu hơn cho analytics, replay → tăng.
  - Disk hạn chế → giảm, hoặc dùng `log.retention.bytes`.

---

## Ví dụ cấu hình broker tổng hợp

`server.properties`:
```properties
# Tuning network & IO
num.network.threads=8
num.io.threads=16

# Log segment & retention
log.segment.bytes=1073741824   # 1GB
log.retention.hours=168        # 7 ngày
```

## Ví dụ Spring Boot (Producer/Consumer) để test hiệu năng

**application.yml**:
```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      acks: all
      batch-size: 16384
      linger-ms: 5
      buffer-memory: 33554432
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
    consumer:
      group-id: test-group
      auto-offset-reset: earliest
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

**Producer (Spring Boot)**:
```java
@RestController
@RequestMapping("/kafka")
public class KafkaTestController {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public KafkaTestController(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    @PostMapping("/send")
    public String send(@RequestParam String topic,
                       @RequestParam String message) {
        kafkaTemplate.send(topic, message);
        return "Sent: " + message;
    }
}
```

**Input HTTP**:
```http
POST /kafka/send?topic=test-topic&message=hello-kafka
```

**Output HTTP**:
```text
Sent: hello-kafka
```

**Consumer (Spring Boot)**:
```java
@Component
public class TestConsumer {

    @KafkaListener(topics = "test-topic", groupId = "test-group")
    public void listen(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Output console**:
```text
Received: hello-kafka
```

---

Tóm lại, 4 config này dùng để:
- `num.network.threads`: tối ưu xử lý kết nối mạng.
- `num.io.threads`: tối ưu đọc/ghi đĩa và replication.
- `log.segment.bytes`: điều khiển kích thước file log.
- `log.retention.hours`: điều khiển thời gian giữ dữ liệu trên broker.
