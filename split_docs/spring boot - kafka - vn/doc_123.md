
Dưới đây là tóm tắt ngắn gọn về **repartitioning trong Kafka Streams** và ảnh hưởng đến hiệu năng, kèm ví dụ code (Spring Boot + Kafka Streams).

---

## 1. Repartitioning là gì?

Trong Kafka Streams, **repartitioning** xảy ra khi dữ liệu cần được **phân phối lại (shuffle)** sang các partition khác dựa trên một key mới, để các bản ghi có cùng key được xử lý trên cùng một instance/partition.

Điều này thường xảy ra khi:

- Dùng `selectKey()` để đổi key.
- Dùng `groupBy()` / `groupByKey()` trước các phép toán như `aggregate()`, `count()`, `reduce()`.
- Join giữa các stream/table mà key không trùng khớp.

Khi đó, Kafka Streams sẽ tạo **repartition topic** trung gian.

---

## 2. Ảnh hưởng đến hiệu năng

Repartitioning có chi phí:

1. **Ghi/đọc thêm vào Kafka**  
   - Mỗi bản ghi phải được ghi vào **repartition topic** rồi đọc lại.
   - Tăng độ trễ (latency) và tải I/O.

2. **Tăng network traffic**  
   - Dữ liệu được “shuffle” giữa các broker/instance.

3. **Tăng số topic/partition**  
   - Nhiều topic trung gian → tốn tài nguyên cluster, khó quản lý.

4. **State store lớn hơn**  
   - Các phép toán `groupBy` + `aggregate` cần state store → tốn RAM/disk.

**Kết luận:** Repartitioning là cần thiết cho một số phép toán, nhưng nên **hạn chế số lần** và **thiết kế key hợp lý** để giảm chi phí.

---

## 3. Khi nào cần / không cần repartition?

### Không cần (hoặc tránh được) repartition

- Dữ liệu đã có key đúng ngay từ đầu.
- Dùng `groupByKey()` trên stream đã được partition theo key đó.
- Join trên cùng key và cùng số partition.

### Cần repartition

- Đổi key bằng `selectKey()` rồi `groupByKey()`.
- Dùng `groupBy()` với một key mới (khác key ban đầu).
- Join hai stream/table có key khác nhau → phải map/transform key trước.

## 4. Ví dụ với Spring Boot + Kafka Streams

### 4.1. Cấu hình cơ bản

```java
@Configuration
@EnableKafkaStreams
public class KafkaStreamsConfig {

    @Bean(name = KafkaStreamsDefaultConfiguration.DEFAULT_STREAMS_CONFIG_BEAN_NAME)
    public KafkaStreamsConfiguration kStreamsConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "order-app");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        return new KafkaStreamsConfiguration(props);
    }
}
```

---

### 4.2. Ví dụ 1 – Repartitioning do đổi key

Giả sử topic `orders` có key là `orderId`, nhưng ta muốn thống kê theo `customerId`.

```java
@Configuration
public class OrderTopology {

    @Bean
    public KStream<String, String> orderStream(StreamsBuilder builder) {
        KStream<String, String> orders = builder.stream("orders");

        // JSON: {"orderId":"o1","customerId":"c1","amount":100}
        KStream<String, String> byCustomer = orders
            .selectKey((oldKey, value) -> extractCustomerId(value)); // đổi key → cần repartition

        KTable<String, Long> orderCountByCustomer = byCustomer
            .groupByKey()          // groupByKey trên key mới → tạo repartition topic
            .count(Materialized.as("order-count-by-customer"));

        orderCountByCustomer
            .toStream()
            .to("order-counts", Produced.with(Serdes.String(), Serdes.Long()));

        return orders;
    }

    private String extractCustomerId(String json) {
        // ví dụ đơn giản, thực tế dùng Jackson/Gson
        // input: {"orderId":"o1","customerId":"c1","amount":100}
        int start = json.indexOf("customerId") + 13;
        int end = json.indexOf("\"", start);
        return json.substring(start, end);
    }
}
```

**Input (topic `orders`):**

Key: `o1`  
Value: `{"orderId":"o1","customerId":"c1","amount":100}`  

Key: `o2`  
Value: `{"orderId":"o2","customerId":"c1","amount":50}`  

**Sau `selectKey` + `groupByKey` (repartition):**

- Repartition topic sẽ có key là `c1`.
- `orderCountByCustomer` sẽ cho kết quả:

Topic `order-counts`:

- Key: `c1`, Value: `1` (sau bản ghi đầu)
- Key: `c1`, Value: `2` (sau bản ghi thứ hai)

### 4.3. Ví dụ 2 – Tránh repartition bằng cách thiết kế key đúng

Nếu ngay từ đầu producer gửi dữ liệu với key là `customerId`, ta không cần `selectKey`.

Producer (ví dụ):

```java
// key = customerId
producer.send(new ProducerRecord<>("orders", "c1",
    "{\"orderId\":\"o1\",\"customerId\":\"c1\",\"amount\":100}"));
```

Topology:

```java
@Bean
public KStream<String, String> orderStreamNoRepartition(StreamsBuilder builder) {
    KStream<String, String> orders = builder.stream("orders"); // key đã là customerId

    KTable<String, Long> orderCountByCustomer = orders
        .groupByKey()   // không cần repartition, vì key đã đúng
        .count(Materialized.as("order-count-by-customer-no-repartition"));

    orderCountByCustomer
        .toStream()
        .to("order-counts-no-repartition", Produced.with(Serdes.String(), Serdes.Long()));

    return orders;
}
```

**Kết quả logic giống ví dụ 1**, nhưng **không tạo repartition topic**, hiệu năng tốt hơn.

## 5. Thực hành tốt (Best practices)

1. **Thiết kế key ngay từ producer**  
   - Gửi dữ liệu với key phù hợp với cách bạn sẽ group/join sau này.

2. **Giảm số lần `selectKey()` / `groupBy()`**  
   - Gom các thao tác đổi key nếu có thể.
   - Tránh đổi key nhiều lần trong cùng một pipeline.

3. **Kiểm tra topology**  
   - Dùng `Topology#describe()` hoặc log của Kafka Streams để xem có bao nhiêu repartition topic được tạo.

4. **Cân nhắc số partition**  
   - Repartition topic nên có số partition phù hợp với throughput và số instance.
