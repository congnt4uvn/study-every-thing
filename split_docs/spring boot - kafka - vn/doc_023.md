
Dưới đây là tóm tắt ngắn gọn về “Partitioning strategies: Key-based partitioning and its impact on data locality and hot partitions” kèm ví dụ code (Kafka + Spring Boot).

---

## 1. Khái niệm chính

### a. Partitioning trong Kafka
- Mỗi topic được chia thành nhiều partition.
- Partition giúp:
  - Tăng khả năng song song (parallelism).
  - Tăng throughput.
  - Cho phép scale consumer theo nhóm.

### b. Key-based partitioning (Partition theo key)
- Khi producer gửi message, nó có thể gán `key` cho record.
- Kafka dùng `key` để quyết định message sẽ vào partition nào (thường là `hash(key) % numPartitions`).
- Kết quả:
  - Cùng một key → luôn vào cùng một partition → giữ được **ordering** theo key.
  - Dữ liệu liên quan (cùng key) được “gom” về một partition → **data locality** tốt.

---

## 2. Data locality (Tính cục bộ dữ liệu)

- Khi cùng một key luôn vào cùng một partition:
  - Consumer xử lý dữ liệu liên quan ở cùng một nơi.
  - Dễ làm các thao tác stateful: aggregation, join, session, v.v.
  - Giảm việc phải chia sẻ state giữa nhiều consumer.

Ví dụ:  
Key = `userId` → tất cả event của cùng một user vào cùng một partition → một consumer xử lý toàn bộ lịch sử của user đó.

---

## 3. Hot partitions (Partition “nóng”)

- Vấn đề xảy ra khi:
  - Một số key xuất hiện rất nhiều (skewed keys).
  - Hoặc một số key được truy cập nhiều hơn hẳn.
- Hệ quả:
  - Partition chứa key “hot” nhận nhiều message hơn → **hot partition**.
  - Consumer xử lý partition đó bị quá tải, trong khi các consumer khác rảnh.
  - Mất cân bằng load, giảm throughput tổng thể.

Cách giảm hot partition (ý tưởng):
- Dùng key “phân tán” hơn (ví dụ: thêm random suffix).
- Dùng custom partitioner để phân phối đều hơn.
- Tăng số partition (nhưng phải cân nhắc chi phí).

## 4. Ví dụ code với Spring Boot + Kafka

### 4.1. Producer gửi message với key

```java
@Service
public class UserEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public UserEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(String userId, String eventJson) {
        // userId là key → tất cả event của user này vào cùng partition
        kafkaTemplate.send("user-events", userId, eventJson)
                     .addCallback(
                         result -> {
                             if (result != null) {
                                 System.out.println("Sent to partition: " +
                                     result.getRecordMetadata().partition());
                             }
                         },
                         ex -> System.err.println("Send failed: " + ex.getMessage())
                     );
    }
}
```

**Input ví dụ:**

```java
sendUserEvent("user-123", "{\"action\":\"LOGIN\"}");
sendUserEvent("user-123", "{\"action\":\"VIEW_PAGE\",\"page\":\"/home\"}");
sendUserEvent("user-999", "{\"action\":\"LOGOUT\"}");
```

**Output (log ví dụ):**

```text
Sent to partition: 1   // user-123
Sent to partition: 1   // user-123 (cùng partition → giữ thứ tự)
Sent to partition: 3   // user-999
```

→ Tất cả event của `user-123` vào cùng partition 1 → data locality tốt.

### 4.2. Consumer đọc theo partition

```java
@Service
public class UserEventConsumer {

    @KafkaListener(topics = "user-events", groupId = "user-event-group")
    public void consume(ConsumerRecord<String, String> record) {
        String key = record.key();
        String value = record.value();
        int partition = record.partition();
        long offset = record.offset();

        System.out.printf(
            "Received message: key=%s, value=%s, partition=%d, offset=%d%n",
            key, value, partition, offset
        );

        // Xử lý stateful theo userId (key)
        // Ví dụ: cập nhật session, thống kê hành vi user, v.v.
    }
}
```

**Output (log ví dụ):**

```text
Received message: key=user-123, value={"action":"LOGIN"}, partition=1, offset=10
Received message: key=user-123, value={"action":"VIEW_PAGE","page":"/home"}, partition=1, offset=11
Received message: key=user-999, value={"action":"LOGOUT"}, partition=3, offset=5
```

### 4.3. Custom partitioner (giảm hot partition – ví dụ đơn giản)

```java
public class UserIdPartitioner implements Partitioner {

    @Override
    public int partition(String topic, Object keyObj, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        String key = (String) keyObj;
        int numPartitions = cluster.partitionsForTopic(topic).size();

        // Ví dụ: thêm một chút random để tránh hot partition
        int salt = ThreadLocalRandom.current().nextInt(0, 3); // 0..2
        int hash = (key + "-" + salt).hashCode();

        return Math.abs(hash) % numPartitions;
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}
```

`application.yml`:

```yaml
spring:
  kafka:
    producer:
      properties:
        partitioner.class: com.example.kafka.UserIdPartitioner
```

Lưu ý:  
- Cách này làm giảm tính “cứng” của data locality (cùng user có thể vào nhiều partition) nhưng giúp phân tán load tốt hơn nếu có key “hot”.

---

## 5. Tóm tắt ngắn

- **Key-based partitioning**:
  - Dùng key để quyết định partition.
  - Giữ thứ tự theo key, tăng data locality.
- **Data locality**:
  - Dữ liệu liên quan (cùng key) nằm cùng partition → dễ xử lý stateful, aggregation.
- **Hot partitions**:
  - Khi một số key quá “nóng” → partition chứa chúng bị quá tải.
  - Gây mất cân bằng load, giảm hiệu năng.
- Giải pháp:
  - Thiết kế key hợp lý, tránh skew.
  - Dùng custom partitioner.
  - Cân nhắc tăng số partition và cách scale consumer.
