
Dưới đây là tóm tắt ngắn gọn về chủ đề: **Spring + Schema Registry: Xử lý schema evolution trong các service Spring**.

---

## 1. Mục tiêu chính

- Sử dụng **Kafka + Schema Registry** (thường là Confluent Schema Registry) trong ứng dụng **Spring Boot**.
- Đảm bảo các service có thể:
  - Gửi/nhận message Avro/JSON có schema.
  - **Tiến hóa schema (schema evolution)** mà không làm hỏng các service cũ (backward/forward compatibility).

---

## 2. Khái niệm chính

1. **Schema Registry**  
   - Lưu trữ và quản lý các schema (Avro/JSON Schema/Protobuf) cho các topic Kafka.
   - Mỗi schema có **version**; khi thay đổi schema, ta đăng ký version mới.
   - Đảm bảo **compatibility** (BACKWARD, FORWARD, FULL, v.v.).

2. **Schema Evolution**  
   Các thay đổi thường gặp:
   - Thêm field mới với **default value** → thường **backward compatible**.
   - Đổi tên field, đổi type → có thể **breaking** nếu không cẩn thận.
   - Xóa field → có thể gây lỗi cho consumer cũ.

3. **Spring Kafka + Schema Registry**  
   - Producer/Consumer dùng serializer/deserializer tích hợp với Schema Registry.
   - Spring Boot cấu hình thông qua `application.yml`/`application.properties`.

---

## 3. Cấu hình cơ bản trong Spring Boot

Ví dụ dùng **Avro** với Confluent Schema Registry.

### `build.gradle` (hoặc `pom.xml`)

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.kafka:spring-kafka'
    implementation 'io.confluent:kafka-avro-serializer:7.6.0' // phiên bản ví dụ
    implementation 'org.apache.avro:avro:1.11.3'
}
```

### `application.yml`

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
      specific.avro.reader: true

    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer

    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      group-id: my-service-group
      auto-offset-reset: earliest
```

## 4. Ví dụ schema và evolution

### Version 1 – `User` (Avro)

```avro
{
  "type": "record",
  "name": "User",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "name", "type": "string" }
  ]
}
```

### Version 2 – Thêm field mới có default

```avro
{
  "type": "record",
  "name": "User",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "name", "type": "string" },
    { "name": "email", "type": ["null", "string"], "default": null }
  ]
}
```

- Đây là **backward compatible**:  
  - Consumer cũ (chỉ biết V1) vẫn đọc được message V2 (bỏ qua `email`).  
  - Consumer mới (V2) đọc message cũ → `email = null`.

## 5. Producer Spring Boot

### Service gửi message

```java
@Service
public class UserProducer {

    private final KafkaTemplate<String, User> kafkaTemplate;

    @Value("${app.kafka.user-topic}")
    private String userTopic;

    public UserProducer(KafkaTemplate<String, User> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUser(User user) {
        kafkaTemplate.send(userTopic, user.getId(), user);
    }
}
```

### Input/Output ví dụ

- **Input (Java object)**

```java
User user = User.newBuilder()
        .setId("u-123")
        .setName("Nguyen Van A")
        .setEmail("a@example.com") // field mới ở V2
        .build();

userProducer.sendUser(user);
```

- **Output (Kafka message)**  
  - Topic: `user-topic`  
  - Key: `"u-123"`  
  - Value (Avro binary) + schema id được ghi trong Schema Registry (ví dụ: subject `user-topic-value`, version 2).

## 6. Consumer Spring Boot

### Listener nhận message

```java
@Service
public class UserConsumer {

    @KafkaListener(topics = "${app.kafka.user-topic}", groupId = "user-service")
    public void listen(User user) {
        // Có thể là V1 hoặc V2, tùy schema đã đăng ký
        System.out.println("Received user: " + user.getId()
                + ", name=" + user.getName()
                + ", email=" + user.getEmail()); // có thể null nếu message cũ
    }
}
```

### Input/Output ví dụ

- **Input (Kafka)**: message với schema V1 (không có `email`).
- **Output (log)**:

```text
Received user: u-001, name=Tran Thi B, email=null
```

## 7. Chiến lược xử lý schema evolution

1. **Chọn compatibility mode** trong Schema Registry:
   - `BACKWARD` (phổ biến cho consumer-driven).
   - `FORWARD` hoặc `FULL` tùy kiến trúc.

2. **Nguyên tắc an toàn**:
   - Chỉ **thêm field** với `default` → an toàn.
   - Tránh **đổi type** hoặc **đổi tên field**; nếu cần, hãy:
     - Tạo field mới, giữ field cũ một thời gian.
     - Cập nhật tất cả consumer, sau đó mới xóa field cũ.

3. **Versioning trong code**:
   - Dùng **generated classes** từ Avro (specific record).
   - Deploy dần dần: producer/consumer mới phải tương thích với schema cũ.

---

## 8. Tóm tắt

- Spring Boot + Spring Kafka + Schema Registry giúp:
  - Quản lý schema tập trung.
  - Hỗ trợ schema evolution an toàn.
- Cấu hình serializer/deserializer trỏ tới Schema Registry.
- Thiết kế schema với **compatibility** trong đầu (thêm field có default, hạn chế thay đổi phá vỡ).
- Producer/Consumer Spring chỉ cần làm việc với class Avro đã generate, Schema Registry xử lý version và compatibility.

Nếu bạn muốn, tôi có thể viết thêm:
- Ví dụ dùng **JSON Schema** thay vì Avro.
- Cấu hình chi tiết cho **compatibility mode** trong Schema Registry.
