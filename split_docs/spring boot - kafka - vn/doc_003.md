
Dưới đây là tóm tắt ngắn gọn về kiến trúc Kafka, tập trung vào Zookeeper vs KRaft, kèm ví dụ code Spring Boot (Java) và input/output minh họa.

---

## 1. Kiến trúc Kafka – Thành phần chính

- **Broker**: Node Kafka lưu trữ dữ liệu (messages) trong các topic/partition.
- **Topic**: Kênh logic để gửi/nhận message (ví dụ: `orders`, `payments`).
- **Partition**: Chia nhỏ topic để tăng song song và khả năng mở rộng.
- **Producer**: Ứng dụng gửi message vào topic.
- **Consumer**: Ứng dụng đọc message từ topic.
- **Consumer Group**: Nhiều consumer cùng chia sẻ việc đọc các partition của một topic.

## 2. Zookeeper vs KRaft (Metadata Quorum mới)

### 2.1. Zookeeper (cách cũ, truyền thống)

- Dùng **Zookeeper cluster** để:
  - Lưu metadata: topic, partition, leader, ACL, config…
  - Quản lý **leader election** cho Kafka controller.
  - Theo dõi trạng thái broker (online/offline).
- Nhược điểm:
  - Thêm một hệ thống phụ (Zookeeper) → phức tạp vận hành.
  - Khó mở rộng, khó quản lý khi cluster lớn.
  - Hai hệ thống (Kafka + Zookeeper) phải đồng bộ.

### 2.2. KRaft (Kafka Raft – Metadata Quorum mới)

- Thay thế Zookeeper bằng **KRaft mode**:
  - Metadata được lưu trực tiếp trong **Kafka controller quorum** (dựa trên Raft).
  - Không cần Zookeeper nữa.
- Ưu điểm:
  - Đơn giản hóa kiến trúc: chỉ còn **Kafka cluster**.
  - Tăng độ tin cậy và khả năng mở rộng metadata.
  - Quản lý, deploy, upgrade dễ hơn.
- Trạng thái hiện tại (theo lộ trình Kafka):
  - KRaft dần trở thành **mặc định** trong các phiên bản Kafka mới.
  - Zookeeper sẽ bị loại bỏ trong tương lai.

---

## 3. So sánh nhanh

| Tiêu chí              | Zookeeper mode                      | KRaft mode (Metadata Quorum)              |
|-----------------------|--------------------------------------|-------------------------------------------|
| Thành phần phụ        | Cần Zookeeper cluster               | Không cần Zookeeper                       |
| Lưu metadata          | Trong Zookeeper                     | Trong Kafka controller quorum             |
| Cơ chế đồng thuận     | ZAB (Zookeeper Atomic Broadcast)    | Raft-based quorum                         |
| Độ phức tạp vận hành  | Cao hơn (2 hệ thống)                | Thấp hơn (1 hệ thống)                     |
| Tương lai             | Bị loại bỏ dần                      | Hướng đi chính thức của Kafka             |

## 4. Ví dụ cấu hình Kafka (Zookeeper vs KRaft)

### 4.1. Kafka với Zookeeper (cũ)

`server.properties` (rút gọn):

```properties
broker.id=1
listeners=PLAINTEXT://localhost:9092
zookeeper.connect=localhost:2181
log.dirs=/tmp/kafka-logs
```

Chạy:
```bash
# Start Zookeeper
bin/zookeeper-server-start.sh config/zookeeper.properties

# Start Kafka broker
bin/kafka-server-start.sh config/server.properties
```

### 4.2. Kafka với KRaft (mới)

`kraft-server.properties` (rút gọn):

```properties
process.roles=broker,controller
node.id=1
controller.quorum.voters=1@localhost:9093
listeners=PLAINTEXT://localhost:9092,CONTROLLER://localhost:9093
listener.security.protocol.map=PLAINTEXT:PLAINTEXT,CONTROLLER:PLAINTEXT
log.dirs=/tmp/kraft-combined-logs
```

Khởi tạo metadata và chạy:

```bash
# Tạo cluster ID
KAFKA_CLUSTER_ID="$(bin/kafka-storage.sh random-uuid)"

# Format storage
bin/kafka-storage.sh format \
  -t $KAFKA_CLUSTER_ID \
  -c config/kraft-server.properties

# Start Kafka (KRaft mode)
bin/kafka-server-start.sh config/kraft-server.properties
```

## 5. Ví dụ Spring Boot + Kafka (áp dụng cho cả Zookeeper/KRaft)

Từ góc nhìn ứng dụng Spring Boot, **cách dùng Kafka không đổi** dù backend là Zookeeper hay KRaft. Bạn chỉ cần biết `bootstrap.servers`.

### 5.1. `pom.xml`

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter</artifactId>
    </dependency>

    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>
</dependencies>
```

### 5.2. Cấu hình `application.yml`

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
    consumer:
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

### 5.3. Producer (gửi message)

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class SimpleProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public SimpleProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId) {
        String topic = "orders";
        String message = "Order created: " + orderId;
        kafkaTemplate.send(topic, orderId, message);
    }
}
```

**Input (gọi từ controller):**

```java
@RestController
@RequestMapping("/orders")
public class OrderController {

    private final SimpleProducer producer;

    public OrderController(SimpleProducer producer) {
        this.producer = producer;
    }

    @PostMapping("/{id}")
    public String createOrder(@PathVariable String id) {
        producer.sendOrder(id);
        return "Order " + id + " sent to Kafka";
    }
}
```

**Ví dụ request:**

```http
POST /orders/123
```

**Output HTTP:**

```text
Order 123 sent to Kafka
```

**Message thực tế trong Kafka topic `orders`:**

- Key: `"123"`
- Value: `"Order created: 123"`

---

### 5.4. Consumer (nhận message)

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class SimpleConsumer {

    @KafkaListener(topics = "orders", groupId = "demo-group")
    public void listenOrders(String message) {
        System.out.println("Received message: " + message);
    }
}
```

**Output console khi có message:**

```text
Received message: Order created: 123
```

---

## 6. Tóm tắt ý chính

- Kafka gồm các thành phần: broker, topic, partition, producer, consumer, consumer group.
- Trước đây Kafka dùng **Zookeeper** để quản lý metadata và leader election.
- **KRaft (Kafka Raft)** là cơ chế metadata quorum mới:
  - Loại bỏ Zookeeper.
  - Metadata được quản lý trực tiếp bởi Kafka controller quorum.
  - Đơn giản hóa vận hành, tăng khả năng mở rộng.
- Ứng dụng Spring Boot sử dụng Kafka **không cần quan tâm** phía sau là Zookeeper hay KRaft; chỉ cần cấu hình `bootstrap-servers` và dùng `KafkaTemplate` (producer) + `@KafkaListener` (consumer).
