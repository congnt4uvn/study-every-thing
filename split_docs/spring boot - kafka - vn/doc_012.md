
Dưới đây là tóm tắt ngắn gọn về **Retention & Log Compaction trong Kafka**, tập trung vào ý chính, kèm ví dụ Spring Boot (Java) và input/output.

---

## 1. Khái niệm chính

### a. Retention (giữ log theo thời gian/kích thước)
- Kafka giữ message trong topic theo:
  - **thời gian**: `retention.ms`
  - **kích thước**: `retention.bytes`
- Sau khi vượt ngưỡng, **segment cũ bị xóa** (không quan tâm key, chỉ dựa vào thời gian/kích thước).
- Phù hợp cho:
  - Log sự kiện, tracking, analytics
  - Dữ liệu không cần giữ mãi mãi

---

### b. Log Compaction (nén log theo key)
- Mục tiêu: **giữ lại bản ghi mới nhất cho mỗi key**.
- Kafka sẽ:
  - Duyệt log theo segment
  - Với mỗi key, chỉ giữ **record mới nhất**
  - Các record cũ hơn cùng key có thể bị xóa
- Kết quả: topic luôn có thể được dùng để **rebuild trạng thái hiện tại** (latest state) của hệ thống.

---

## 2. Use cases chính

### a. CDC (Change Data Capture)
- Mỗi thay đổi trong DB (INSERT/UPDATE/DELETE) được publish vào Kafka.
- Topic được **compaction** để:
  - Giữ lại trạng thái mới nhất của mỗi row (key = id)
  - Consumer mới có thể đọc toàn bộ topic và **xây lại snapshot** hiện tại.

### b. Latest state (trạng thái mới nhất)
- Lưu trạng thái của:
  - User profile
  - Cấu hình hệ thống
  - Số dư tài khoản, v.v.
- Mỗi key = entity id, value = trạng thái mới nhất.
- Consumer có thể:
  - Đọc toàn bộ topic từ đầu → build map<key, value> hiện tại
  - Sau đó chỉ cần consume tiếp các update mới.

---

## 3. Tombstones (bản ghi xóa)

- **Tombstone** = record có:
  - `key` = id
  - `value = null`
- Ý nghĩa: “xóa key này khỏi trạng thái hiện tại”.
- Trong log compaction:
  - Tombstone được giữ lại một thời gian (để các consumer thấy event xóa).
  - Sau đó có thể bị xóa hoàn toàn (tùy config).
- Dùng trong CDC:
  - DELETE row trong DB → gửi tombstone vào Kafka.

---

## 4. Cấu hình topic compaction

Ví dụ tạo topic với log compaction:

```bash
kafka-topics.sh \
  --create \
  --topic user-profile \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.5 \
  --config segment.ms=600000
```

- `cleanup.policy=compact` → bật log compaction.
- Có thể kết hợp: `cleanup.policy=compact,delete` (vừa compaction vừa retention theo thời gian).

## 5. Ví dụ Spring Boot + Kafka

### 5.1. Producer: gửi trạng thái user (upsert + delete)

**Entity trạng thái user (JSON)**

```json
// Input: cập nhật user
{
  "userId": "u123",
  "name": "Nguyen Van A",
  "email": "a@example.com"
}
```

**Producer config (application.yml)**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
```

**Producer service**

```java
@Service
public class UserProfileProducer {

    private final KafkaTemplate<String, UserProfile> kafkaTemplate;
    private static final String TOPIC = "user-profile";

    public UserProfileProducer(KafkaTemplate<String, UserProfile> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    // Upsert user (create/update)
    public void upsertUser(UserProfile user) {
        kafkaTemplate.send(TOPIC, user.getUserId(), user);
    }

    // Delete user -> gửi tombstone (value = null)
    public void deleteUser(String userId) {
        kafkaTemplate.send(TOPIC, userId, null);
    }
}
```

**Class UserProfile**

```java
public class UserProfile {
    private String userId;
    private String name;
    private String email;
    // getters/setters
}
```

**Output trong topic (trước compaction)**

Partition chứa các record:

| offset | key   | value (JSON)                                  |
|--------|-------|-----------------------------------------------|
| 0      | u123  | {"userId":"u123","name":"A","email":"a@x"}    |
| 1      | u123  | {"userId":"u123","name":"A1","email":"a1@x"}  |
| 2      | u456  | {"userId":"u456","name":"B","email":"b@x"}    |
| 3      | u123  | null (tombstone – delete)                     |

**Sau compaction (logic)**

| key   | value cuối cùng                         |
|-------|-----------------------------------------|
| u123  | null (có thể bị xóa hẳn sau 1 thời gian)|
| u456  | {"userId":"u456","name":"B","email":"b@x"} |

Consumer rebuild state sẽ hiểu:
- `u123` đã bị xóa
- `u456` còn tồn tại

---

### 5.2. Consumer: build trạng thái hiện tại

```java
@Service
public class UserProfileConsumer {

    private final Map<String, UserProfile> stateStore = new ConcurrentHashMap<>();

    @KafkaListener(topics = "user-profile", groupId = "user-profile-consumer")
    public void consume(ConsumerRecord<String, UserProfile> record) {
        String key = record.key();
        UserProfile value = record.value();

        if (value == null) {
            // Tombstone -> xóa khỏi state
            stateStore.remove(key);
        } else {
            // Upsert
            stateStore.put(key, value);
        }
    }

    // API để xem trạng thái hiện tại
    public UserProfile getUser(String userId) {
        return stateStore.get(userId);
    }
}
```

**Kết quả (output logic)**

- Sau khi consumer đọc hết topic:
  - `stateStore` chứa trạng thái mới nhất của tất cả user chưa bị xóa.
- Khi có message mới:
  - `stateStore` được cập nhật incremental.

---

## 6. Tóm tắt ngắn

- **Retention**: xóa message theo thời gian/kích thước, không quan tâm key.
- **Log compaction**: giữ lại **record mới nhất cho mỗi key**, dùng để rebuild **latest state**.
- **Use cases**:
  - CDC: đồng bộ DB → Kafka → các service khác.
  - Lưu trạng thái hiện tại (user, config, balance, …).
- **Tombstone (value = null)**:
  - Biểu diễn xóa key trong mô hình state.
  - Được compaction xử lý để loại bỏ key khỏi snapshot cuối cùng.
