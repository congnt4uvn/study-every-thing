
Dưới đây là tóm tắt ngắn gọn về chủ đề “Data contracts: Handling breaking changes and deprecation” trong bối cảnh Kafka + Spring Boot, kèm ví dụ code minh họa.

---

## 1. Khái niệm chính

**Data contract** (hợp đồng dữ liệu) là thỏa thuận giữa producer và consumer về:
- Cấu trúc message (schema: field, kiểu dữ liệu, bắt buộc/tùy chọn…)
- Cách tiến hóa (evolution) schema mà không làm hỏng hệ thống.

Mục tiêu:
- Tránh **breaking changes** (thay đổi làm consumer cũ bị lỗi).
- Quản lý **deprecation** (khai tử field/phiên bản) có kiểm soát.

---

## 2. Breaking changes là gì?

Breaking changes là những thay đổi khiến consumer hiện tại không còn xử lý được message mới, ví dụ:

- Đổi kiểu dữ liệu field (String → Integer).
- Xóa field bắt buộc.
- Đổi ý nghĩa field mà không cập nhật consumer.
- Thay đổi format JSON không tương thích.

Trong Kafka, điều này nguy hiểm vì:
- Consumer có thể chạy nhiều version song song.
- Message cũ vẫn còn trong topic (không thể “sửa lại” lịch sử).

---

## 3. Nguyên tắc tiến hóa schema (schema evolution)

Một số nguyên tắc để tránh breaking changes:

1. **Chỉ thêm field mới với giá trị mặc định**  
   - Field mới nên là optional hoặc có default.
   - Consumer cũ bỏ qua field mới; consumer mới có thể dùng.

2. **Không đổi kiểu dữ liệu field hiện có**  
   - Nếu cần, tạo field mới (vd: `age` → `ageInYears`) và deprecate field cũ.

3. **Không xóa field đang được sử dụng**  
   - Đầu tiên đánh dấu deprecated, sau đó đảm bảo không còn consumer dùng, rồi mới xóa.

4. **Giữ backward compatibility**  
   - Producer mới phải tạo message mà consumer cũ vẫn đọc được.

---

## 4. Deprecation (khai tử field/phiên bản)

Quy trình gợi ý:

1. **Đánh dấu deprecated**  
   - Trong schema / DTO / tài liệu: `@Deprecated`, comment, doc.
2. **Triển khai consumer mới**  
   - Consumer mới không còn phụ thuộc field cũ.
3. **Theo dõi & đảm bảo không còn consumer cũ**  
   - Dùng logging, metrics, feature flags.
4. **Xóa field / version cũ**  
   - Sau khi chắc chắn an toàn.

## 5. Ví dụ với Spring Boot + Kafka (JSON)

### 5.1. Version 1 – DTO ban đầu

```java
// V1: Hợp đồng dữ liệu ban đầu
public class OrderEventV1 {
    private String orderId;
    private String customerName; // sẽ bị deprecate sau này
    private BigDecimal amount;

    // getters/setters
}
```

Producer gửi message:

```java
@Service
public class OrderProducerV1 {

    private final KafkaTemplate<String, OrderEventV1> kafkaTemplate;

    public OrderProducerV1(KafkaTemplate<String, OrderEventV1> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder() {
        OrderEventV1 event = new OrderEventV1();
        event.setOrderId("ORD-001");
        event.setCustomerName("Nguyen Van A");
        event.setAmount(new BigDecimal("100000"));

        kafkaTemplate.send("orders-topic", event.getOrderId(), event);
    }
}
```

**Message JSON (output trên topic):**

```json
{
  "orderId": "ORD-001",
  "customerName": "Nguyen Van A",
  "amount": 100000
}
```

Consumer:

```java
@KafkaListener(topics = "orders-topic", groupId = "order-consumer-v1")
public void handleOrderV1(OrderEventV1 event) {
    // Input: JSON như trên
    System.out.println("Received V1: " + event.getOrderId() + " - " + event.getCustomerName());
}
```

---

### 5.2. Thay đổi yêu cầu – thêm field mới, deprecate field cũ

Giả sử yêu cầu mới:
- Không dùng `customerName` nữa.
- Dùng `customerId` và `customerEmail`.
- Vẫn phải hỗ trợ consumer cũ trong một thời gian.

**V2 DTO (backward compatible):**

```java
public class OrderEventV2 {

    private String orderId;

    @Deprecated // đánh dấu deprecate, nhưng vẫn giữ để consumer cũ không lỗi
    private String customerName;

    private String customerId;    // field mới
    private String customerEmail; // field mới

    private BigDecimal amount;

    // getters/setters
}
```

Producer V2:

```java
@Service
public class OrderProducerV2 {

    private final KafkaTemplate<String, OrderEventV2> kafkaTemplate;

    public OrderProducerV2(KafkaTemplate<String, OrderEventV2> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderV2() {
        OrderEventV2 event = new OrderEventV2();
        event.setOrderId("ORD-002");
        event.setCustomerName("Nguyen Van B"); // vẫn set để consumer cũ không bị null
        event.setCustomerId("CUST-123");
        event.setCustomerEmail("cust123@example.com");
        event.setAmount(new BigDecimal("200000"));

        kafkaTemplate.send("orders-topic", event.getOrderId(), event);
    }
}
```

**Message JSON (output trên topic):**

```json
{
  "orderId": "ORD-002",
  "customerName": "Nguyen Van B",
  "customerId": "CUST-123",
  "customerEmail": "cust123@example.com",
  "amount": 200000
}
```

Consumer cũ (V1) vẫn hoạt động:
- Nó chỉ map các field mà nó biết (`orderId`, `customerName`, `amount`).
- Các field mới (`customerId`, `customerEmail`) bị bỏ qua.

Consumer mới (V2):

```java
@KafkaListener(topics = "orders-topic", groupId = "order-consumer-v2")
public void handleOrderV2(OrderEventV2 event) {
    // Input: JSON có thể là V1 hoặc V2
    System.out.println("Received V2: " + event.getOrderId()
            + " - " + event.getCustomerId()
            + " - " + event.getCustomerEmail());
}
```

- Nếu message cũ (V1) không có `customerId`, `customerEmail` → các field này sẽ là `null`.
- Consumer mới phải xử lý trường hợp `null` (ví dụ fallback sang `customerName` nếu cần).

## 6. Ví dụ với Avro + Schema Registry (tùy chọn)

Nếu dùng Avro + Confluent Schema Registry, bạn có thể:

- Định nghĩa schema V1:

```avro
{
  "type": "record",
  "name": "OrderEvent",
  "namespace": "com.example",
  "fields": [
    {"name": "orderId", "type": "string"},
    {"name": "customerName", "type": "string"},
    {"name": "amount", "type": "double"}
  ]
}
```

- V2 (backward compatible): chỉ **thêm field mới với default**:

```avro
{
  "type": "record",
  "name": "OrderEvent",
  "namespace": "com.example",
  "fields": [
    {"name": "orderId", "type": "string"},
    {"name": "customerName", "type": "string"},
    {"name": "amount", "type": "double"},
    {"name": "customerId", "type": ["null", "string"], "default": null},
    {"name": "customerEmail", "type": ["null", "string"], "default": null}
  ]
}
```

Schema Registry sẽ kiểm tra compatibility (BACKWARD, FORWARD, FULL…) để tránh breaking changes.

---

## 7. Tóm tắt ý chính

- Data contract là thỏa thuận về cấu trúc message giữa producer và consumer.
- Tránh breaking changes bằng cách:
  - Không đổi kiểu/xóa field đang dùng.
  - Chỉ thêm field mới với default/optional.
  - Giữ backward compatibility.
- Deprecation:
  - Đánh dấu field/phiên bản cũ là deprecated.
  - Chạy song song version mới và cũ.
  - Khi chắc chắn không còn dùng, mới xóa.
- Trong Spring Boot + Kafka:
  - Dùng DTO versioned (V1, V2…) hoặc schema (Avro/JSON Schema).
  - Consumer phải xử lý được cả message cũ và mới trong giai đoạn chuyển đổi.

Nếu bạn gửi thêm nội dung chi tiết hơn (ví dụ đoạn tài liệu gốc), tôi có thể tóm tắt và viết lại code ví dụ sát hơn với nội dung đó.
