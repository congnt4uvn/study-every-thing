
Dưới đây là tóm tắt ngắn gọn về **Schema Registry – Compatibility modes** trong Kafka (thường dùng với Avro/Protobuf/JSON Schema).

---

## 1. Khái niệm chung

**Schema Registry** lưu trữ schema cho các message Kafka.  
**Compatibility mode** quy định: *schema mới* có được phép đăng ký hay không, dựa trên việc nó có tương thích với *schema cũ* hay không.

Các mode chính:

- `BACKWARD`
- `FORWARD`
- `FULL`
- `NONE`

---

## 2. BACKWARD Compatibility

- Schema mới phải **đọc được** dữ liệu đã được ghi bằng schema cũ.
- Tức là: consumer dùng schema mới vẫn đọc được message cũ.

Ví dụ Avro:

```avro
// Schema v1
{
  "type": "record",
  "name": "User",
  "fields": [
    {"name": "id", "type": "int"}
  ]
}

// Schema v2 (BACKWARD compatible với v1)
{
  "type": "record",
  "name": "User",
  "fields": [
    {"name": "id", "type": "int"},
    {"name": "name", "type": ["null", "string"], "default": null}
  ]
}
```

- Thêm field có default → BACKWARD compatible.

---

## 3. FORWARD Compatibility

- Schema mới phải **ghi được** dữ liệu mà consumer cũ (dùng schema cũ) vẫn đọc được.
- Tức là: producer dùng schema mới, consumer dùng schema cũ vẫn OK.

Ví dụ:

```avro
// Schema v1
{
  "type": "record",
  "name": "User",
  "fields": [
    {"name": "id", "type": "int"},
    {"name": "name", "type": "string"}
  ]
}

// Schema v2 (FORWARD compatible với v1)
{
  "type": "record",
  "name": "User",
  "fields": [
    {"name": "id", "type": "int"}
  ]
}
```

- Xóa field mà consumer cũ không cần → FORWARD compatible (consumer cũ chỉ đọc field `id`).

---

## 4. FULL Compatibility

- Kết hợp cả **BACKWARD** và **FORWARD**:
  - Schema mới đọc được dữ liệu cũ.
  - Consumer cũ đọc được dữ liệu mới.
- Đảm bảo tương thích 2 chiều.

Ví dụ (đơn giản):

```avro
// v1
{
  "type": "record",
  "name": "User",
  "fields": [
    {"name": "id", "type": "int"}
  ]
}

// v2 (FULL compatible với v1)
{
  "type": "record",
  "name": "User",
  "fields": [
    {"name": "id", "type": "int"},
    {"name": "name", "type": ["null", "string"], "default": null}
  ]
}
```

- Thêm field optional với default → thường FULL compatible.

---

## 5. NONE Compatibility

- Không kiểm tra tương thích.
- Bất kỳ schema mới nào cũng được đăng ký.
- Nguy hiểm trong môi trường production vì dễ gây lỗi deserialize.

---

## 6. Ví dụ cấu hình với Spring Boot + Kafka + Confluent Schema Registry

**application.yml:**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
      specific.avro.reader: true
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      properties:
        specific.avro.reader: true
```

**Đặt compatibility mode qua REST API:**

```bash
# Đặt mode BACKWARD cho subject "user-value"
curl -X PUT \
  -H "Content-Type: application/json" \
  --data '{"compatibility": "BACKWARD"}' \
  http://localhost:8081/config/user-value
```

**Kiểm tra mode hiện tại:**

```bash
curl http://localhost:8081/config/user-value
# Output ví dụ:
# {"compatibilityLevel":"BACKWARD"}
```

---

## 7. Ví dụ luồng dữ liệu (Input/Output)

1. **Producer** gửi message với schema v1:

```json
// Payload (User v1)
{"id": 1}
```

2. Sau đó bạn đăng ký schema v2 (BACKWARD compatible) và producer bắt đầu gửi:

```json
// Payload (User v2)
{"id": 2, "name": "Alice"}
```

3. **Consumer cũ** (dùng schema v1) khi đọc message v2:
- Với BACKWARD mode: consumer cũ vẫn đọc được `id`, field `name` bị bỏ qua.

4. **Consumer mới** (dùng schema v2) khi đọc message v1:
- Field `name` sẽ nhận giá trị default `null`.

---

## 8. Tóm tắt nhanh

- `BACKWARD`: ưu tiên **consumer mới** đọc được dữ liệu cũ.  
- `FORWARD`: ưu tiên **consumer cũ** đọc được dữ liệu mới.  
- `FULL`: đảm bảo tương thích 2 chiều (an toàn nhất, nhưng hạn chế thay đổi).  
- `NONE`: không kiểm tra, chỉ nên dùng cho dev/test.

Nếu bạn cho biết thêm ngữ cảnh (microservice mới, thay đổi field, v.v.), tôi có thể gợi ý mode phù hợp và ví dụ schema cụ thể hơn.
