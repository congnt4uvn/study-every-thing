
Dưới đây là tóm tắt ngắn gọn về **Monitoring & Observability trong Kafka với Spring Boot**, tập trung vào: **metrics producer/consumer** và **lag monitoring**.

---

## 1. Metrics cho Producer/Consumer

### Ý chính
- Kafka client (producer/consumer) tự động expose nhiều metrics (thông qua JMX).
- Spring Boot + Micrometer có thể thu thập và gửi metrics này tới Prometheus, Grafana, v.v.
- Các metrics quan trọng:
  - Producer: `record-send-rate`, `record-error-rate`, `request-latency-avg`, `buffer-available-bytes`, …
  - Consumer: `records-consumed-rate`, `bytes-consumed-rate`, `fetch-latency-avg`, `commit-latency-avg`, …

### Cấu hình Spring Boot để expose metrics (Actuator + Micrometer)

**pom.xml (Maven)**

```xml
<dependencies>
    <!-- Actuator -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-actuator</artifactId>
    </dependency>

    <!-- Micrometer Prometheus -->
    <dependency>
        <groupId>io.micrometer</groupId>
        <artifactId>micrometer-registry-prometheus</artifactId>
    </dependency>

    <!-- Kafka -->
    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>
</dependencies>
```

**application.yml**

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
  endpoint:
    prometheus:
      enabled: true
  metrics:
    tags:
      application: kafka-demo
```

**Kết quả (output)**  
- Truy cập: `GET /actuator/prometheus`  
- Bạn sẽ thấy các metrics như:

```text
kafka_producer_record_send_total{client_id="producer-1",topic="orders",} 1234
kafka_producer_record_error_total{client_id="producer-1",topic="orders",} 2
kafka_consumer_records_consumed_total{client_id="consumer-1",topic="orders",} 5678
kafka_consumer_fetch_latency_avg{client_id="consumer-1",} 10.5
```

## 2. Lag Monitoring (Theo dõi độ trễ tiêu thụ)

**Lag** = `log-end-offset (trên broker)` − `consumer-committed-offset`  
→ Cho biết consumer đang “chậm” bao nhiêu message so với đầu queue.

### Cách phổ biến để monitor lag

1. **Kafka Exporter / Burrow / Cruise Control**  
   - Dùng tool chuyên dụng để đọc offset từ broker và consumer group, sau đó export metrics cho Prometheus.
   - Ví dụ metric:  
     `kafka_consumergroup_lag{group="order-service",topic="orders",partition="0"} 150`

2. **Tự tính lag trong ứng dụng Spring Boot (demo)**  
   - Dùng `AdminClient` để lấy `endOffsets`.
   - Dùng `KafkaConsumer` hoặc `KafkaAdmin` để lấy `committed offsets`.

### Ví dụ code tính lag đơn giản (Java, Spring Boot)

```java
import org.apache.kafka.clients.admin.*;
import org.apache.kafka.common.TopicPartition;
import org.springframework.kafka.core.ConsumerFactory;
import org.springframework.stereotype.Service;

import java.time.Duration;
import java.util.*;
import java.util.concurrent.ExecutionException;

@Service
public class KafkaLagService {

    private final AdminClient adminClient;
    private final ConsumerFactory<String, String> consumerFactory;

    public KafkaLagService(AdminClient adminClient,
                           ConsumerFactory<String, String> consumerFactory) {
        this.adminClient = adminClient;
        this.consumerFactory = consumerFactory;
    }

    public Map<TopicPartition, Long> getLag(String topic, String groupId)
            throws ExecutionException, InterruptedException {

        // 1. Lấy danh sách partition của topic
        DescribeTopicsResult describeTopicsResult = adminClient.describeTopics(Collections.singletonList(topic));
        TopicDescription topicDescription = describeTopicsResult.values().get(topic).get();

        List<TopicPartition> partitions = new ArrayList<>();
        for (TopicPartitionInfo pInfo : topicDescription.partitions()) {
            partitions.add(new TopicPartition(topic, pInfo.partition()));
        }

        // 2. Lấy end offsets (log-end-offset) từ broker
        Map<TopicPartition, ListOffsetsResult.ListOffsetsResultInfo> endOffsets =
                adminClient.listOffsets(
                        partitions.stream().collect(
                                HashMap::new,
                                (m, tp) -> m.put(tp, OffsetSpec.latest()),
                                HashMap::putAll
                        )
                ).all().get();

        // 3. Lấy committed offsets của consumer group
        try (var consumer = consumerFactory.createConsumer(groupId, null)) {
            Map<TopicPartition, Long> lagPerPartition = new HashMap<>();
            for (TopicPartition tp : partitions) {
                var committed = consumer.committed(tp);
                long committedOffset = committed != null ? committed.offset() : 0L;
                long endOffset = endOffsets.get(tp).offset();
                long lag = endOffset - committedOffset;
                lagPerPartition.put(tp, lag);
            }
            return lagPerPartition;
        }
    }
}
```

**Ví dụ sử dụng (input/output)**

```java
Map<TopicPartition, Long> lag = kafkaLagService.getLag("orders", "order-service");
lag.forEach((tp, value) ->
        System.out.println("Partition " + tp.partition() + " lag = " + value));
```

**Output mẫu:**

```text
Partition 0 lag = 120
Partition 1 lag = 0
Partition 2 lag = 45
```

## 3. Những điểm chính cần nhớ

- **Producer/Consumer metrics**:
  - Dùng Spring Boot Actuator + Micrometer để expose metrics.
  - Theo dõi throughput, error rate, latency, buffer, v.v.
- **Lag monitoring**:
  - Lag = end offset − committed offset.
  - Có thể dùng tool ngoài (Kafka Exporter, Burrow) hoặc tự tính trong app.
  - Lag cao kéo dài → consumer chậm, cần scale thêm instance, tối ưu xử lý, hoặc tăng partition.

Nếu bạn cho biết stack cụ thể (Prometheus/Grafana, Datadog, New Relic, …), tôi có thể đưa thêm ví dụ dashboard/alert chi tiết hơn.
