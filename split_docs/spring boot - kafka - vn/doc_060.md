
Dưới đây là tóm tắt ngắn gọn về chủ đề “Độ bền (durability) & tính nhất quán (consistency): đánh đổi giữa độ trễ (latency) và độ bền” trong Kafka + Spring Boot, kèm ví dụ code.

---

## 1. Ý chính

Trong Kafka, ta thường phải **đánh đổi** giữa:

- **Độ bền (durability)**: Dữ liệu được ghi an toàn, không mất khi broker chết.
- **Tính nhất quán (consistency)**: Tất cả replica có cùng dữ liệu.
- **Độ trễ (latency)**: Tốc độ ghi/đọc message.

Cấu hình càng “an toàn” (durable, consistent) thì **latency càng cao**. Cấu hình càng “nhanh” thì **nguy cơ mất dữ liệu** càng lớn.

---

## 2. Các tham số Kafka quan trọng

### 2.1. `acks` (producer)

- `acks=0`: Producer không chờ broker trả lời → rất nhanh, **dễ mất dữ liệu**.
- `acks=1`: Chờ leader ghi xong → cân bằng giữa tốc độ và an toàn.
- `acks=all` (hoặc `-1`): Chờ tất cả replica ghi xong → **an toàn nhất**, latency cao hơn.

### 2.2. `min.insync.replicas` (broker/topic)

- Số replica tối thiểu phải “in-sync” để chấp nhận ghi khi `acks=all`.
- Ví dụ: `replication.factor=3`, `min.insync.replicas=2` → cần ít nhất 2 replica ghi thành công.

### 2.3. `enable.idempotence` (producer)

- Bật idempotent producer để tránh ghi trùng khi retry.
- Kết hợp với `acks=all` giúp đạt **at-least-once** an toàn hơn, gần **exactly-once** trong nhiều trường hợp.

---

## 3. Các “profile” cấu hình điển hình

### 3.1. Tối ưu tốc độ (chấp nhận rủi ro mất dữ liệu)

- `acks=0` hoặc `acks=1`
- `retries` thấp
- `linger.ms` nhỏ
- `min.insync.replicas` nhỏ

→ Latency thấp, throughput cao, nhưng có thể mất message khi broker/producer lỗi.

### 3.2. Cân bằng (thường dùng trong hệ thống business)

- `acks=1`
- `retries` > 0
- `enable.idempotence=true`
- `replication.factor>=3`

→ Độ trễ vừa phải, độ bền tương đối tốt.

### 3.3. Tối ưu độ bền & nhất quán

- `acks=all`
- `enable.idempotence=true`
- `retries` cao
- `min.insync.replicas>=2` (với replication.factor>=3)

→ Latency cao hơn, nhưng giảm tối đa nguy cơ mất dữ liệu.

## 4. Ví dụ Spring Boot Kafka Producer

### 4.1. Cấu hình Producer (Java)

```java
import org.apache.kafka.clients.producer.ProducerConfig;
import org.apache.kafka.common.serialization.StringSerializer;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.DefaultKafkaProducerFactory;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.core.ProducerFactory;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> configProps = new HashMap<>();
        configProps.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");

        // Độ bền cao (durable, consistent)
        configProps.put(ProducerConfig.ACKS_CONFIG, "all");
        configProps.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
        configProps.put(ProducerConfig.RETRIES_CONFIG, 5);
        configProps.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 5);

        // Tối ưu latency (có thể điều chỉnh)
        configProps.put(ProducerConfig.LINGER_MS_CONFIG, 5);      // batch trong 5ms
        configProps.put(ProducerConfig.BATCH_SIZE_CONFIG, 32_768); // 32KB

        configProps.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        configProps.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        return new DefaultKafkaProducerFactory<>(configProps);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

### 4.2. Gửi message và log kết quả (input/output)

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class MyKafkaProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public MyKafkaProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        String topic = "orders";

        // INPUT: orderId, payload
        kafkaTemplate.send(topic, orderId, payload)
                .whenComplete((result, ex) -> {
                    if (ex != null) {
                        // OUTPUT khi lỗi
                        System.err.println("Send failed for orderId=" + orderId + " error=" + ex.getMessage());
                    } else {
                        // OUTPUT khi thành công
                        System.out.println("Sent orderId=" + orderId +
                                " to partition=" + result.getRecordMetadata().partition() +
                                " offset=" + result.getRecordMetadata().offset());
                    }
                });
    }
}
```

**Ví dụ Input:**

```java
myKafkaProducer.sendOrder("ORD-123", "{ \"amount\": 100000, \"currency\": \"VND\" }");
```

**Ví dụ Output (log):**

```text
Sent orderId=ORD-123 to partition=0 offset=42
```

Hoặc nếu lỗi:

```text
Send failed for orderId=ORD-123 error=TimeoutException: Expiring 1 record(s) due to ...
```

## 5. Ví dụ cấu hình broker/topic cho độ bền

`server.properties` (broker):

```properties
# Ít nhất 2 replica phải in-sync để ghi thành công
min.insync.replicas=2
```

Tạo topic với replication factor 3:

```bash
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --replication-factor 3 \
  --partitions 6 \
  --topic orders
```

Kết hợp với producer `acks=all` → ghi chỉ thành công khi **>=2 replica** ghi được, giúp tăng độ bền.

---

## 6. Tóm tắt ngắn

- Độ bền & nhất quán cao → dùng `acks=all`, `min.insync.replicas>=2`, `enable.idempotence=true`, replication factor cao → **latency tăng**.
- Cần tốc độ cao, chấp nhận rủi ro → `acks=0` hoặc `acks=1`, ít replica → **latency thấp**, nhưng có thể mất dữ liệu.
- Spring Boot + Kafka cho phép cấu hình linh hoạt để chọn điểm cân bằng phù hợp với yêu cầu business.
