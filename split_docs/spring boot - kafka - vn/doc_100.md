
Dưới đây là tóm tắt ngắn gọn về chủ đề “Data contracts: Versioning topics vs versioning schemas” trong bối cảnh Kafka + Spring Boot, kèm ví dụ code và input/output minh họa.

---

## 1. Khái niệm chính

**Data contract**:  
Là thỏa thuận giữa producer và consumer về:
- Tên topic
- Cấu trúc dữ liệu (schema)
- Kiểu dữ liệu, field bắt buộc/tùy chọn, ý nghĩa các field

Khi hệ thống phát triển, data contract phải **tiến hóa** (evolve) mà không làm hỏng (break) các consumer cũ. Có 2 chiến lược chính:

1. **Versioning topics** – tạo topic mới khi thay đổi lớn
2. **Versioning schemas** – giữ nguyên topic, chỉ version schema (thường dùng với Schema Registry)

---

## 2. Versioning topics

### Ý tưởng
- Mỗi phiên bản lớn của dữ liệu → topic mới  
  Ví dụ:  
  - `user-events-v1`  
  - `user-events-v2`

### Ưu điểm
- Cô lập rõ ràng giữa các phiên bản
- Dễ rollback: quay lại đọc topic cũ
- Consumer đơn giản: mỗi consumer chỉ cần hiểu 1 schema cho 1 topic

### Nhược điểm
- Nhiều topic → khó quản lý
- Cần pipeline để migrate dữ liệu giữa các topic (nếu cần)
- Consumer có thể phải subscribe nhiều topic nếu cần dữ liệu từ nhiều version

### Ví dụ với Spring Boot + Kafka

**Producer gửi vào topic v1**

```java
// DTO v1
public class UserEventV1 {
    private String userId;
    private String action;
    // getters/setters
}

@Service
public class UserEventV1Producer {

    private final KafkaTemplate<String, UserEventV1> kafkaTemplate;

    public UserEventV1Producer(KafkaTemplate<String, UserEventV1> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(UserEventV1 event) {
        kafkaTemplate.send("user-events-v1", event.getUserId(), event);
    }
}
```

**Consumer đọc topic v1**

```java
@KafkaListener(topics = "user-events-v1", groupId = "user-service-v1")
public void handleUserEventV1(UserEventV1 event) {
    // Xử lý dữ liệu v1
}
```

**Producer gửi vào topic v2 (schema mới)**

```java
// DTO v2: thêm field metadata
public class UserEventV2 {
    private String userId;
    private String action;
    private Map<String, String> metadata;
    // getters/setters
}

@Service
public class UserEventV2Producer {

    private final KafkaTemplate<String, UserEventV2> kafkaTemplate;

    public UserEventV2Producer(KafkaTemplate<String, UserEventV2> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(UserEventV2 event) {
        kafkaTemplate.send("user-events-v2", event.getUserId(), event);
    }
}
```

**Consumer đọc topic v2**

```java
@KafkaListener(topics = "user-events-v2", groupId = "user-service-v2")
public void handleUserEventV2(UserEventV2 event) {
    // Xử lý dữ liệu v2, có metadata
}
```

**Ví dụ input/output (JSON)**

- Topic `user-events-v1`:
  ```json
  {
    "userId": "u123",
    "action": "LOGIN"
  }
  ```

- Topic `user-events-v2`:
  ```json
  {
    "userId": "u123",
    "action": "LOGIN",
    "metadata": {
      "ip": "10.0.0.1",
      "device": "mobile"
    }
  }
  ```

## 3. Versioning schemas (giữ nguyên topic)

### Ý tưởng
- Chỉ dùng **1 topic**, ví dụ: `user-events`
- Schema thay đổi theo version, quản lý bằng **Schema Registry** (Avro/Protobuf/JSON Schema)
- Producer ghi dữ liệu với schema version mới, consumer có thể đọc nhiều version nếu schema tương thích (backward/forward compatible)

### Ưu điểm
- Ít topic hơn, dễ quản lý
- Dễ phân tích dữ liệu vì tất cả event cùng loại nằm trong 1 topic
- Tận dụng được tính năng tương thích schema (schema compatibility)

### Nhược điểm
- Cần Schema Registry
- Consumer phức tạp hơn nếu phải xử lý nhiều version
- Cần tuân thủ chặt chẽ quy tắc tương thích schema (không xóa field bắt buộc, không đổi type…)

### Ví dụ với Spring Boot + Kafka + Avro

**Schema v1 (Avro)**

```avro
{
  "type": "record",
  "name": "UserEvent",
  "namespace": "com.example.avro",
  "fields": [
    {"name": "userId", "type": "string"},
    {"name": "action", "type": "string"}
  ]
}
```

**Schema v2 (thêm field optional)**

```avro
{
  "type": "record",
  "name": "UserEvent",
  "namespace": "com.example.avro",
  "fields": [
    {"name": "userId", "type": "string"},
    {"name": "action", "type": "string"},
    {
      "name": "metadata",
      "type": ["null", {"type": "map", "values": "string"}],
      "default": null
    }
  ]
}
```

> Đây là thay đổi **backward compatible**: consumer cũ (v1) vẫn đọc được vì field mới có default.

**Producer (Spring Boot) – gửi UserEvent v2 vào topic `user-events`**

```java
@Service
public class UserEventProducer {

    private final KafkaTemplate<String, UserEvent> kafkaTemplate;

    public UserEventProducer(KafkaTemplate<String, UserEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(String userId, String action, Map<String, String> metadata) {
        UserEvent event = UserEvent.newBuilder()
                .setUserId(userId)
                .setAction(action)
                .setMetadata(metadata) // có thể null
                .build();

        kafkaTemplate.send("user-events", userId, event);
    }
}
```

**Consumer v1 (chỉ quan tâm field cũ)**

```java
@KafkaListener(topics = "user-events", groupId = "user-service-v1")
public void handleUserEventV1(UserEvent event) {
    String userId = event.getUserId();
    String action = event.getAction();
    // Bỏ qua metadata
}
```

**Consumer v2 (dùng thêm metadata)**

```java
@KafkaListener(topics = "user-events", groupId = "user-service-v2")
public void handleUserEventV2(UserEvent event) {
    String userId = event.getUserId();
    String action = event.getAction();
    Map<String, String> metadata = event.getMetadata();
    // Xử lý thêm metadata
}
```

**Ví dụ input/output (Avro → JSON biểu diễn)**

```json
{
  "userId": "u123",
  "action": "LOGIN",
  "metadata": {
    "ip": "10.0.0.1",
    "device": "mobile"
  }
}
```

Consumer v1 vẫn đọc được, chỉ không dùng `metadata`.

## 4. Khi nào chọn versioning topics vs versioning schemas?

**Nên versioning topics khi:**
- Thay đổi **breaking** (không tương thích ngược)  
  - Ví dụ: đổi type field, đổi meaning field, xóa field bắt buộc
- Muốn tách hẳn luồng dữ liệu cũ/mới
- Không dùng Schema Registry hoặc không muốn phức tạp hóa schema

**Nên versioning schemas (1 topic) khi:**
- Thay đổi **non-breaking** (backward/forward compatible)  
  - Thêm field optional, thêm enum value, v.v.
- Muốn giữ tất cả event cùng loại trong 1 topic
- Đã dùng Schema Registry (Avro/Protobuf/JSON Schema)

---

## 5. Tóm tắt ngắn

- **Data contract** = topic + schema + quy tắc tương thích.
- Có 2 chiến lược chính:
  - **Versioning topics**: topic mới cho version mới → đơn giản cho consumer, nhiều topic.
  - **Versioning schemas**: 1 topic, nhiều version schema → cần Schema Registry, phải tuân thủ compatibility.
- Thay đổi **breaking** → thường nên tạo **topic mới**.  
- Thay đổi **non-breaking** → nên **version schema** trong cùng topic.
