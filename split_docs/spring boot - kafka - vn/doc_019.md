
Dưới đây là tóm tắt ngắn gọn về “Cách partition ảnh hưởng đến: Parallelism và Throughput” trong Kafka + Spring Boot, kèm ví dụ code.

---

## 1. Khái niệm chính

- **Partition**: Mỗi topic trong Kafka được chia thành nhiều partition.  
- **Parallelism (mức độ song song)**:
  - Mỗi partition chỉ được xử lý bởi **tối đa 1 consumer trong cùng một consumer group** tại một thời điểm.
  - Số lượng partition quyết định **số luồng xử lý song song tối đa** trong một consumer group.
- **Throughput (lưu lượng xử lý)**:
  - Nhiều partition hơn → nhiều consumer xử lý song song hơn → **tăng throughput** (số message/giây).
  - Tuy nhiên, quá nhiều partition cũng có chi phí: metadata, open file, network, rebalance lâu hơn.

**Nguyên tắc:**  
- Muốn tăng khả năng xử lý song song → tăng số partition.  
- Số consumer trong một consumer group **không nên vượt quá** số partition (consumer dư sẽ bị “nhàn rỗi”).

## 2. Ví dụ: Tạo topic với nhiều partition

### Tạo topic bằng lệnh Kafka

```bash
kafka-topics.sh \
  --create \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 6 \
  --replication-factor 1
```

- Topic `orders` có **6 partition** → tối đa 6 consumer trong cùng một group có thể xử lý song song.

## 3. Spring Boot: Producer gửi message

```java
// build.gradle (hoặc pom.xml) cần spring-kafka
// implementation 'org.springframework.kafka:spring-kafka'

@Configuration
@EnableKafka
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}

@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        // orderId làm key → quyết định partition
        kafkaTemplate.send("orders", orderId, payload);
    }
}
```

### Input (gửi từ service)

```java
orderProducer.sendOrder("order-1", "{ \"id\": \"order-1\", \"amount\": 100 }");
orderProducer.sendOrder("order-2", "{ \"id\": \"order-2\", \"amount\": 200 }");
orderProducer.sendOrder("order-3", "{ \"id\": \"order-3\", \"amount\": 300 }");
```

### Output (trên Kafka)

- Các message được phân tán vào 6 partition của topic `orders` dựa trên key (`order-1`, `order-2`, …).  
- Các partition khác nhau có thể được xử lý song song bởi nhiều consumer.

## 4. Spring Boot: Consumer với nhiều instance (tăng parallelism)

### Cấu hình consumer

```java
@Configuration
public class KafkaConsumerConfig {

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "order-group");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        return new DefaultKafkaConsumerFactory<>(props);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        // concurrency = số thread consumer trong cùng 1 app instance
        factory.setConcurrency(3); // ví dụ 3
        return factory;
    }
}
```

### Listener

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-group")
    public void listen(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                + " | Received: " + message);
        // xử lý message...
    }
}
```

### Cách parallelism hoạt động

- Topic `orders` có 6 partition.
- `concurrency = 3` → 3 thread consumer trong 1 instance.
- Nếu bạn chạy **2 instance** của ứng dụng (cùng `groupId = order-group`):
  - Tổng cộng có thể có **tối đa 6 consumer** (3 thread x 2 instance).
  - Kafka sẽ phân chia 6 partition cho 6 consumer → xử lý song song tối đa.

## 5. Ví dụ Input/Output về parallelism

### Input

Giả sử gửi 6 message:

```java
for (int i = 1; i <= 6; i++) {
    orderProducer.sendOrder("order-" + i, "payload-" + i);
}
```

### Output (log giả định)

Instance 1:
```text
Thread: consumer-0-C-1 | Received: payload-1
Thread: consumer-0-C-2 | Received: payload-2
Thread: consumer-0-C-3 | Received: payload-3
```

Instance 2:
```text
Thread: consumer-0-C-1 | Received: payload-4
Thread: consumer-0-C-2 | Received: payload-5
Thread: consumer-0-C-3 | Received: payload-6
```

- Mỗi thread xử lý message từ các partition khác nhau → **song song**.
- Throughput tăng vì nhiều message được xử lý cùng lúc.

---

## 6. Tóm tắt ngắn

- **Partition** quyết định:
  - Số consumer tối đa có thể xử lý song song trong một consumer group.
  - Do đó ảnh hưởng trực tiếp đến **parallelism** và **throughput**.
- Để tăng throughput:
  1. Tăng số partition của topic.
  2. Tăng số consumer (hoặc `concurrency`) nhưng **không vượt quá số partition**.
- Spring Boot + Spring Kafka hỗ trợ dễ dàng cấu hình producer/consumer để tận dụng partition cho xử lý song song.
