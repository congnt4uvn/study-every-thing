
Dưới đây là tóm tắt ngắn gọn về “Data contracts: Backward/forward compatibility strategies” trong bối cảnh Kafka + Spring Boot, kèm ví dụ code và input/output minh họa.

---

## 1. Khái niệm chính

**Data contract**:  
Là “hợp đồng” về cấu trúc dữ liệu giữa producer và consumer (ví dụ: schema của message Kafka).  
Mục tiêu: thay đổi được schema mà **không làm hỏng** các service đang chạy.

- **Backward compatibility**:  
  Consumer mới vẫn đọc được dữ liệu cũ (message cũ).
- **Forward compatibility**:  
  Consumer cũ vẫn đọc được dữ liệu mới (message mới).

Trong Kafka, điều này thường được thực hiện bằng:
- Schema (Avro/JSON/Protobuf) + Schema Registry
- Quy tắc thay đổi field (thêm field, bỏ field, đổi kiểu, đổi tên, …)

---

## 2. Chiến lược Backward / Forward compatibility

### 2.1. Thêm field mới

- **Backward compatible**:  
  Thêm field mới với **default value** → consumer cũ bỏ qua field này, consumer mới dùng được.
- **Forward compatible**:  
  Consumer cũ không biết field mới → phải **bỏ qua field không biết**.

**Ví dụ schema JSON (giản lược)**

Version 1:
```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "name", "type": "string" }
  ]
}
```

Version 2 (thêm email, backward compatible):
```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "name", "type": "string" },
    { "name": "email", "type": ["null", "string"], "default": null }
  ]
}
```

- Consumer cũ (v1) nhận message v2:
  ```json
  { "id": "u1", "name": "Alice", "email": "a@example.com" }
  ```
  → Nó chỉ dùng `id`, `name`, bỏ qua `email`.

---

### 2.2. Xóa field

- **Backward compatible**:  
  Xóa field mà consumer cũ **không còn dùng** nữa (hoặc có default).
- **Forward compatible**:  
  Consumer cũ vẫn mong đợi field đó → có thể **không forward compatible** nếu xóa thẳng.

Chiến lược an toàn:
1. Đầu tiên: đánh dấu field là “deprecated”, không dùng nữa.
2. Sau khi tất cả consumer đã nâng cấp: mới xóa.

---

### 2.3. Đổi kiểu dữ liệu (type)

- Thường **không backward/forward compatible** nếu đổi mạnh (vd: `string` → `int`).
- Có thể dùng:
  - Field mới: `age` (int) thay cho `ageStr` (string).
  - Giữ cả hai trong một thời gian, rồi xóa field cũ.

---

### 2.4. Đổi tên field

- Không an toàn nếu đổi trực tiếp.
- Cách an toàn:
  - Thêm field mới `fullName`, giữ `name` cũ.
  - Consumer mới dùng `fullName`, consumer cũ dùng `name`.
  - Sau khi migrate xong, xóa `name`.

## 3. Ví dụ với Spring Boot + Kafka

### 3.1. DTO v1

```java
public class UserV1 {
    private String id;
    private String name;
    // getters/setters
}
```

### 3.2. DTO v2 (thêm email – backward compatible)

```java
public class UserV2 {
    private String id;
    private String name;
    private String email; // field mới, có thể optional
    // getters/setters
}
```

### 3.3. Producer gửi message (Spring Kafka)

```java
@Service
public class UserProducer {

    private final KafkaTemplate<String, UserV2> kafkaTemplate;

    public UserProducer(KafkaTemplate<String, UserV2> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUser(UserV2 user) {
        kafkaTemplate.send("user-topic", user.getId(), user);
    }
}
```

**Input (Java):**
```java
UserV2 user = new UserV2();
user.setId("u1");
user.setName("Alice");
user.setEmail("alice@example.com");
userProducer.sendUser(user);
```

**Message trên Kafka (JSON minh họa):**
```json
{
  "id": "u1",
  "name": "Alice",
  "email": "alice@example.com"
}
```

### 3.4. Consumer cũ (v1) – chỉ biết UserV1

```java
@KafkaListener(topics = "user-topic", groupId = "user-group-v1")
public void consumeUserV1(UserV1 user) {
    System.out.println("Consumer V1 nhận: id=" + user.getId() + ", name=" + user.getName());
}
```

**Output console:**
```text
Consumer V1 nhận: id=u1, name=Alice
```

- Consumer v1 không cần biết `email`, vẫn chạy bình thường → **backward compatible**.

---

## 4. Nguyên tắc thực tế khi thiết kế data contract

1. **Không đổi kiểu field một cách phá vỡ** (breaking change).
2. **Thêm field mới với default/optional** để giữ backward compatibility.
3. **Không xóa field ngay**: deprecate trước, xóa sau khi tất cả consumer đã nâng cấp.
4. **Không đổi tên field trực tiếp**: tạo field mới, giữ field cũ một thời gian.
5. Sử dụng **Schema Registry** (nếu dùng Avro/Protobuf) để:
   - Quản lý version schema.
   - Cấu hình compatibility mode (BACKWARD, FORWARD, FULL, v.v.).

Nếu bạn gửi thêm nội dung chi tiết (ví dụ đoạn text hoặc schema cụ thể), tôi có thể tóm tắt sâu hơn và viết code mẫu sát với trường hợp của bạn.
