
Dưới đây là tóm tắt ngắn gọn về quản lý offset trong Kafka với Spring Boot, tập trung vào 3 pattern: **at-least-once**, **at-most-once**, và **effectively-once**.

---

## 1. Khái niệm chung

- **Offset**: vị trí của message trong partition Kafka.  
- Cách ta **commit offset** quyết định pattern:
  - Commit **sau** khi xử lý: at-least-once
  - Commit **trước** khi xử lý: at-most-once
  - Kết hợp transaction / idempotent: effectively-once

---

## 2. At-least-once

**Đảm bảo không mất message**, nhưng có thể xử lý **trùng** nếu consumer restart sau khi xử lý nhưng trước khi commit offset.

### Ý tưởng
- Nhận message
- Xử lý business logic
- Commit offset (thủ công hoặc auto-commit sau xử lý)

### Ví dụ Spring Kafka (manual commit)

```java
@EnableKafka
@Configuration
public class KafkaConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);

        // Tắt auto-commit, dùng manual
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
        return factory;
    }
}
```

```java
@Component
public class AtLeastOnceConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void consume(String message, Acknowledgment ack) {
        // 1. Xử lý message
        System.out.println("Processing: " + message);

        // 2. Commit offset sau khi xử lý thành công
        ack.acknowledge();
    }
}
```

**Input (message trong topic `orders`):**
```json
{"orderId": 1, "amount": 100}
```

**Output (log):**
```text
Processing: {"orderId": 1, "amount": 100}
```

Nếu service crash sau khi log nhưng trước `ack.acknowledge()`, message sẽ được đọc lại → có thể xử lý 2 lần.

---

## 3. At-most-once

**Không xử lý trùng**, nhưng có thể **mất message** nếu crash sau commit offset nhưng trước khi xử lý.

### Ý tưởng
- Commit offset **ngay khi nhận** message
- Sau đó mới xử lý business logic

Trong Spring Kafka, pattern này thường đạt được bằng:
- Auto-commit offset (mặc định)  
- Hoặc commit sớm trong listener (ít dùng vì rủi ro mất dữ liệu)

### Ví dụ (auto-commit)

```yaml
spring:
  kafka:
    consumer:
      enable-auto-commit: true
      auto-commit-interval: 1000
```

```java
@Component
public class AtMostOnceConsumer {

    @KafkaListener(topics = "logs", groupId = "log-service")
    public void consume(String message) {
        // Offset đã được commit tự động định kỳ
        System.out.println("Received log: " + message);
    }
}
```

Nếu service crash sau khi commit offset nhưng trước khi xử lý, message đó sẽ không được xử lý lại.

---

## 4. Effectively-once (gần như exactly-once)

Mục tiêu: **không mất message** và **không xử lý trùng** (trong thực tế, “effectively-once”).

### Cách đạt được
1. **Idempotent processing**: xử lý nhiều lần nhưng kết quả như một lần.
   - Ví dụ: lưu vào DB với `orderId` unique, nếu insert trùng thì bỏ qua.
2. **Kafka Transactions**: kết hợp transaction giữa producer và consumer (read-process-write) để đảm bảo atomic.

### Ví dụ: Idempotent với DB (Spring Boot + Kafka)

```java
@Entity
@Table(name = "orders", uniqueConstraints = @UniqueConstraint(columnNames = "orderId"))
public class OrderEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private Long orderId;
    private Integer amount;
}
```

```java
public interface OrderRepository extends JpaRepository<OrderEntity, Long> {
    boolean existsByOrderId(Long orderId);
}
```

```java
@Component
public class EffectivelyOnceConsumer {

    private final OrderRepository orderRepository;

    public EffectivelyOnceConsumer(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @KafkaListener(topics = "orders", groupId = "order-service")
    @Transactional
    public void consume(String message, Acknowledgment ack) {
        // Parse JSON (giả sử dùng Jackson)
        ObjectMapper mapper = new ObjectMapper();
        try {
            JsonNode node = mapper.readTree(message);
            Long orderId = node.get("orderId").asLong();
            Integer amount = node.get("amount").asInt();

            // Idempotent: kiểm tra đã xử lý chưa
            if (!orderRepository.existsByOrderId(orderId)) {
                OrderEntity order = new OrderEntity();
                order.setOrderId(orderId);
                order.setAmount(amount);
                orderRepository.save(order);
                System.out.println("Saved order: " + orderId);
            } else {
                System.out.println("Duplicate order, skip: " + orderId);
            }

            // Commit offset sau khi DB transaction ok
            ack.acknowledge();
        } catch (Exception e) {
            // Không commit offset → sẽ đọc lại
            System.err.println("Error processing message: " + e.getMessage());
        }
    }
}
```

**Input (2 lần cùng message):**
```json
{"orderId": 10, "amount": 200}
{"orderId": 10, "amount": 200}
```

**Output (log):**
```text
Saved order: 10
Duplicate order, skip: 10
```

DB chỉ có 1 record → **effectively-once**.

---

## 5. So sánh nhanh

| Pattern            | Mất message? | Trùng lặp? | Cách dùng chính |
|--------------------|-------------|-----------|-----------------|
| At-most-once       | Có thể      | Không     | Log, metrics không quá quan trọng |
| At-least-once      | Không       | Có thể    | Hầu hết use case quan trọng |
| Effectively-once   | Không       | Tránh được| Thanh toán, đơn hàng, tài chính |
