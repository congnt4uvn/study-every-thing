
Dưới đây là tóm tắt ngắn gọn về **capacity planning** cho Kafka trong ứng dụng Spring Boot, tập trung vào: **số partition, throughput, dung lượng lưu trữ (retention)**, kèm ví dụ code.

---

## 1. Partition – ước lượng số lượng partition

**Mục tiêu chính**:  
- Tăng **throughput** (song song xử lý)  
- Đảm bảo **khả năng mở rộng** (scalability)  
- Đảm bảo **tính sẵn sàng** (availability, replication)

**Nguyên tắc cơ bản**:

1. **Số consumer tối đa trong 1 consumer group cho 1 topic = số partition**  
   → Nếu bạn cần 10 consumer xử lý song song, topic đó nên có **≥ 10 partition**.

2. **Throughput mỗi partition** (ví dụ):  
   - Giả sử 1 partition xử lý ~ **5 MB/s** ổn định  
   - Bạn cần tổng **50 MB/s** → cần khoảng **10 partition**.

3. **Replication factor**:  
   - Thường dùng: **3** (1 leader + 2 follower)  
   - Tăng độ an toàn dữ liệu nhưng tốn thêm dung lượng.

---

## 2. Throughput – ước lượng lưu lượng

**Cần xác định**:

- **Message size trung bình** (byte)  
- **Số message/giây**  
- **Tổng throughput** = message_size * messages_per_second

**Ví dụ**:

- Mỗi message ~ **1 KB**  
- 10.000 message/giây  
- Throughput ≈ 10.000 * 1 KB = **10 MB/s**

Nếu mỗi partition xử lý ~5 MB/s → cần ít nhất **2 partition**, nhưng nên dự phòng → **3–4 partition**.

---

## 3. Retention & Storage – ước lượng dung lượng lưu trữ

**Retention**: thời gian Kafka giữ message (ví dụ: 7 ngày, 30 ngày).

**Công thức gần đúng**:

```text
Dung lượng/ngày = throughput (MB/s) * 86.400 (s/ngày)
Dung lượng cần = dung lượng/ngày * số ngày retention * replication_factor
```

**Ví dụ**:

- Throughput: **10 MB/s**  
- Retention: **7 ngày**  
- Replication factor: **3**

```text
Dung lượng/ngày = 10 MB/s * 86.400 ≈ 864.000 MB ≈ 864 GB
Dung lượng cần = 864 GB * 7 * 3 ≈ 18.144 GB ≈ 18 TB
```

→ Cần khoảng **18 TB** cho cluster (tổng tất cả broker).

## 4. Ví dụ cấu hình Kafka topic (Spring Boot + AdminClient)

### application.yml

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    admin:
      properties:
        bootstrap.servers: localhost:9092
```

### Tạo topic với số partition & retention

```java
import org.apache.kafka.clients.admin.NewTopic;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic userEventsTopic() {
        int partitions = 10;          // ước lượng từ throughput & số consumer
        short replicationFactor = 3;  // độ an toàn dữ liệu

        NewTopic topic = new NewTopic("user-events", partitions, replicationFactor);

        // retention 7 ngày (ms)
        long retentionMs = 7L * 24 * 60 * 60 * 1000;
        topic.configs(Map.of(
                "retention.ms", String.valueOf(retentionMs),
                "cleanup.policy", "delete" // hoặc compact, tuỳ use case
        ));

        return topic;
    }
}
```

## 5. Ví dụ Producer/Consumer (Spring Boot)

### Producer – gửi message

```java
@Service
public class UserEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public UserEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(String userId, String action) {
        String topic = "user-events";
        String key = userId;
        String value = "{\"userId\":\"" + userId + "\",\"action\":\"" + action + "\"}";

        kafkaTemplate.send(topic, key, value);
    }
}
```

**Input** (gọi service):

```java
userEventProducer.sendUserEvent("u123", "LOGIN");
```

**Output** (message trong Kafka topic `user-events`):

```json
Key: "u123"
Value: {"userId":"u123","action":"LOGIN"}
```

### Consumer – đọc message

```java
@Service
public class UserEventConsumer {

    @KafkaListener(topics = "user-events", groupId = "user-event-group")
    public void consume(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Output console**:

```text
Received: {"userId":"u123","action":"LOGIN"}
```

---

## 6. Tóm tắt ý chính

- **Partition**: quyết định mức độ song song và số consumer tối đa; ước lượng dựa trên throughput và số consumer mong muốn.  
- **Throughput**: tính từ kích thước message * số message/giây; dùng để quyết định số partition.  
- **Retention & Storage**: retention càng dài → cần nhiều dung lượng; tính dung lượng/ngày * số ngày * replication_factor.  
- Trong Spring Boot, có thể **tạo topic** với số partition, replication, retention bằng `NewTopic` + Kafka Admin, và dùng `KafkaTemplate` + `@KafkaListener` để gửi/nhận dữ liệu.
