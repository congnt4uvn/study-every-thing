
Dưới đây là tóm tắt ngắn gọn về các cấu hình consumer trong Spring Kafka liên quan đến:

- Cơ chế ack (manual vs auto)
- Chiến lược commit offset

---

## 1. Auto Ack (Tự động xác nhận & commit offset)

**Ý tưởng chính**

- Kafka listener tự động đánh dấu message là “đã xử lý” và commit offset.
- Đơn giản, ít code, nhưng khó kiểm soát khi xử lý lỗi hoặc xử lý lâu.

**Cấu hình chính**

```properties
spring.kafka.consumer.enable-auto-commit=true
spring.kafka.consumer.auto-offset-reset=earliest
spring.kafka.listener.ack-mode=record   # hoặc batch, time, count...
```

**Cách hoạt động**

- `enable-auto-commit=true`: Kafka client sẽ commit offset định kỳ (mặc định mỗi 5s).
- `ack-mode=record`: mỗi record được xử lý xong sẽ được commit (theo cơ chế auto-commit của client hoặc container).

**Ví dụ code**

```java
@KafkaListener(topics = "demo-topic", groupId = "auto-ack-group")
public void listen(String message) {
    System.out.println("Received: " + message);
    // Không cần gọi ack, offset sẽ được commit tự động
}
```

**Ưu điểm**

- Cấu hình đơn giản.
- Phù hợp cho các use case không quá khắt khe về “exactly-once” hoặc xử lý lỗi.

**Nhược điểm**

- Có thể commit offset trước khi xử lý thực sự hoàn tất (tùy mode).
- Khó kiểm soát khi cần retry, xử lý bù, hoặc đảm bảo không mất message.

## 2. Manual Ack (Xác nhận thủ công & commit offset chủ động)

**Ý tưởng chính**

- Bạn tự quyết định khi nào một message được coi là “đã xử lý” và commit offset.
- Phù hợp khi cần kiểm soát chặt chẽ, xử lý giao dịch, hoặc logic phức tạp.

**Cấu hình chính**

```properties
spring.kafka.consumer.enable-auto-commit=false
spring.kafka.listener.ack-mode=manual   # hoặc manual_immediate
```

**Cách hoạt động**

- `enable-auto-commit=false`: Kafka client không tự commit offset.
- `ack-mode=manual`: bạn phải gọi `ack.acknowledge()` trong listener.
- `ack-mode=manual_immediate`: commit offset ngay lập tức khi gọi `ack.acknowledge()`.

**Ví dụ code (manual ack)**

```java
@KafkaListener(
    topics = "demo-topic",
    groupId = "manual-ack-group",
    containerFactory = "manualAckContainerFactory"
)
public void listen(ConsumerRecord<String, String> record,
                   Acknowledgment ack) {
    try {
        System.out.println("Received: " + record.value());
        // Xử lý business logic
        // ...
        // Nếu xử lý OK, commit offset
        ack.acknowledge();
    } catch (Exception e) {
        // Không gọi ack => offset không commit
        // Có thể log, gửi sang DLQ, retry...
        System.err.println("Error: " + e.getMessage());
    }
}
```

**Ví dụ cấu hình containerFactory**

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> manualAckContainerFactory(
        ConsumerFactory<String, String> consumerFactory) {
    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);
    factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
    return factory;
}
```

**Ưu điểm**

- Kiểm soát chính xác khi commit offset.
- Dễ kết hợp với transaction (DB + Kafka), retry, DLQ.

**Nhược điểm**

- Code phức tạp hơn.
- Nếu quên `ack.acknowledge()`, offset sẽ không commit → có thể đọc lại message.

## 3. Các chiến lược commit offset (AckMode)

Spring Kafka cung cấp nhiều `AckMode` để điều khiển commit offset:

1. **`RECORD`**  
   - Commit sau mỗi record (khi xử lý xong).
   - Dùng cho auto ack, độ an toàn cao hơn nhưng commit nhiều.

2. **`BATCH`**  
   - Commit sau khi xử lý xong cả batch (poll).
   - Giảm số lần commit, nhưng nếu lỗi 1 record có thể ảnh hưởng cả batch.

3. **`TIME`**  
   - Commit theo chu kỳ thời gian (vd: mỗi 5s), dựa trên `ackTime`.
   - Phù hợp khi muốn giảm commit nhưng vẫn có giới hạn thời gian.

4. **`COUNT`**  
   - Commit sau khi xử lý đủ N record (dựa trên `ackCount`).

5. **`COUNT_TIME`**  
   - Kết hợp COUNT + TIME: commit khi đủ N record hoặc đủ T thời gian.

6. **`MANUAL`**  
   - Bạn gọi `ack.acknowledge()`; commit được thực hiện theo chu kỳ container.

7. **`MANUAL_IMMEDIATE`**  
   - Commit ngay lập tức khi gọi `ack.acknowledge()`.

**Ví dụ cấu hình AckMode khác**

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> batchAckContainerFactory(
        ConsumerFactory<String, String> consumerFactory) {
    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);
    factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.BATCH);
    return factory;
}
```

## 4. Ví dụ Input/Output đơn giản

**Input (message gửi lên Kafka)**

Topic: `demo-topic`  
Messages:
- `"order-1"`
- `"order-2"`

**Consumer với manual ack**

```java
@KafkaListener(topics = "demo-topic", groupId = "manual-ack-group")
public void listen(ConsumerRecord<String, String> record,
                   Acknowledgment ack) {
    System.out.println("Processing: " + record.value());
    // Giả sử xử lý thành công
    ack.acknowledge();
}
```

**Output (log)**

```text
Processing: order-1
Processing: order-2
```

- Offset của `order-1`, `order-2` được commit chỉ sau khi `ack.acknowledge()` được gọi.
- Nếu xảy ra lỗi trước khi gọi `ack.acknowledge()`, message sẽ được đọc lại ở lần poll sau.

## 5. Khi nào dùng auto ack vs manual ack?

- **Auto ack**:
  - Hệ thống đơn giản, không cần logic retry phức tạp.
  - Chấp nhận khả năng mất message trong một số trường hợp hiếm.

- **Manual ack**:
  - Cần đảm bảo xử lý chính xác (gần “exactly-once”).
  - Cần kết hợp với transaction (DB, service khác).
  - Cần kiểm soát rõ ràng khi commit offset, khi retry, khi gửi DLQ.
