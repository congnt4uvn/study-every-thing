## 1. Khái niệm chính

### Partition là gì?
- Mỗi topic trong Kafka được chia thành nhiều partition.
- Mỗi partition là một log tuần tự, được xử lý độc lập.
- Partition quyết định:
  - Khả năng song song (parallelism).
  - Cân bằng tải giữa broker và consumer.

### Hot Partition là gì?
- “Hot partition” là partition nhận quá nhiều message so với các partition khác.
- Hậu quả:
  - Broker chứa partition đó bị quá tải (CPU, disk, network).
  - Consumer xử lý partition đó bị chậm, tăng độ trễ (latency).
  - Giảm throughput tổng thể.

### Skew (lệch phân phối)
- Skew là tình trạng dữ liệu không phân bố đều giữa các partition.
- Nguyên nhân thường gặp:
  - Key phân vùng (partition key) không đa dạng (ví dụ: quá nhiều message có cùng key).
  - Custom partitioner không tốt.
  - Một số key “hot” (ví dụ: user VIP, tenant lớn).

---

## 2. Cách Kafka quyết định partition

Mặc định (DefaultPartitioner):
- Nếu có key:
  - `partition = hash(key) % numPartitions`
- Nếu không có key:
  - Kafka sẽ round-robin giữa các partition (tương đối đều).

Nếu key bị lệch (ví dụ: 80% message có cùng key), partition tương ứng sẽ bị “hot”.

---

## 3. Chiến lược giảm thiểu hot partition / skew

1. **Thiết kế key tốt hơn**
   - Chọn key có độ phân tán cao (high cardinality).
   - Tránh dùng một vài giá trị key lặp lại quá nhiều.
   - Ví dụ: thay vì key = `tenantId`, có thể dùng `tenantId + "-" + randomSuffix`.

2. **Custom Partitioner**
   - Tự viết `Partitioner` để:
     - Phân tán tốt hơn các key “hot”.
     - Áp dụng logic riêng (ví dụ: hash + random, hoặc consistent hashing).

3. **Tăng số partition**
   - Tăng số partition của topic để:
     - Tăng khả năng song song.
     - Giảm tải trên mỗi partition.
   - Lưu ý: tăng partition có thể ảnh hưởng đến ordering (thứ tự) nếu không cẩn thận.

4. **Rebalancing / Repartitioning**
   - Tạo topic mới với partition hợp lý hơn, rồi chuyển dữ liệu.
   - Dùng Kafka Streams / ksqlDB để repartition theo key mới.

5. **Consumer scaling**
   - Tăng số consumer trong consumer group để xử lý nhiều partition song song.
   - Nhưng: số consumer hiệu quả ≤ số partition.

## 4. Ví dụ code Spring Boot + Spring Kafka

### 4.1. Producer với key “xấu” (dễ gây hot partition)

```java
@Service
public class BadKeyProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public BadKeyProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    // Gần như tất cả message dùng cùng một key => hot partition
    public void sendBad(String message) {
        String key = "TENANT_1"; // key bị lặp quá nhiều
        kafkaTemplate.send("orders-topic", key, message);
    }
}
```

**Input** (gửi 5 message):

```java
for (int i = 1; i <= 5; i++) {
    badKeyProducer.sendBad("Order #" + i);
}
```

**Kết quả (conceptual)**:
- Tất cả message có key `"TENANT_1"` → hash giống nhau → rơi vào cùng 1 partition, ví dụ `partition 0`.
- `partition 0`: 5 message  
- `partition 1..N`: 0 message → skew.

---

### 4.2. Producer với key “tốt” hơn (giảm hot partition)

```java
@Service
public class BetterKeyProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private final Random random = new Random();

    public BetterKeyProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendBetter(String tenantId, String message) {
        // Thêm suffix ngẫu nhiên để phân tán
        int suffix = random.nextInt(10); // 0..9
        String key = tenantId + "-" + suffix;
        kafkaTemplate.send("orders-topic", key, message);
    }
}
```

**Input**:

```java
for (int i = 1; i <= 20; i++) {
    betterKeyProducer.sendBetter("TENANT_1", "Order #" + i);
}
```

**Kết quả (conceptual)**:
- Key có dạng: `TENANT_1-0`, `TENANT_1-1`, …, `TENANT_1-9`.
- Hash của các key khác nhau → phân tán lên nhiều partition.
- Giảm khả năng một partition bị “hot”.

### 4.3. Custom Partitioner (Spring Kafka)

**1) Tạo class Partitioner:**

```java
import org.apache.kafka.clients.producer.Partitioner;
import org.apache.kafka.common.Cluster;

import java.util.Map;
import java.util.Random;

public class CustomHotKeyPartitioner implements Partitioner {

    private final Random random = new Random();

    @Override
    public int partition(String topic,
                         Object key,
                         byte[] keyBytes,
                         Object value,
                         byte[] valueBytes,
                         Cluster cluster) {

        int numPartitions = cluster.partitionCountForTopic(topic);

        if (key == null) {
            // Không có key -> round-robin (dùng random đơn giản)
            return random.nextInt(numPartitions);
        }

        String keyStr = key.toString();

        // Ví dụ: nếu key là TENANT_1 (hot key) thì phân tán ngẫu nhiên
        if ("TENANT_1".equals(keyStr)) {
            return random.nextInt(numPartitions);
        }

        // Các key khác dùng hash bình thường
        return (keyStr.hashCode() & Integer.MAX_VALUE) % numPartitions;
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}
```

**2) Cấu hình trong Spring Boot:**

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public Map<String, Object> producerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
                  org.apache.kafka.common.serialization.StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
                  org.apache.kafka.common.serialization.StringSerializer.class);

        // Dùng custom partitioner
        props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG,
                  CustomHotKeyPartitioner.class.getName());

        return props;
    }

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        return new DefaultKafkaProducerFactory<>(producerConfigs());
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

**Input**:

```java
for (int i = 1; i <= 10; i++) {
    kafkaTemplate.send("orders-topic", "TENANT_1", "Order #" + i);
}
```

**Kết quả (conceptual)**:
- Vì key `"TENANT_1"` được xử lý đặc biệt → mỗi message có thể rơi vào partition khác nhau (random).
- Giảm hot partition cho key này.

### 4.4. Consumer đơn giản (Spring Kafka)

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders-topic", groupId = "order-consumer-group")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf(
            "Partition: %d, Offset: %d, Key: %s, Value: %s%n",
            record.partition(),
            record.offset(),
            record.key(),
            record.value()
        );
    }
}
```

**Output (ví dụ)**:

```
Partition: 0, Offset: 0, Key: TENANT_1-3, Value: Order #1
Partition: 2, Offset: 5, Key: TENANT_1-7, Value: Order #2
Partition: 1, Offset: 3, Key: TENANT_1-1, Value: Order #3
...
```

Ta có thể quan sát phân phối message giữa các partition để phát hiện skew/hot partition.

## 5. Tóm tắt ngắn gọn (tiếng Việt)

- Partition quyết định khả năng song song và cân bằng tải trong Kafka.
- Hot partition xảy ra khi một partition nhận quá nhiều message → gây quá tải và tăng độ trễ.
- Skew là phân phối dữ liệu không đều giữa các partition, thường do key không đa dạng hoặc key “hot”.
- Cách giảm thiểu:
  - Thiết kế key phân tán tốt (high cardinality).
  - Dùng custom partitioner để xử lý key “hot”.
  - Tăng số partition, scale consumer.
  - Repartition topic nếu cần.
