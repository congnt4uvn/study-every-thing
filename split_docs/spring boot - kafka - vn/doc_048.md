
Dưới đây là tóm tắt ngắn gọn về quản lý offset trong Kafka với Spring Boot, tập trung vào:  
- Commit thủ công (manual commit)  
- Commit đồng bộ (sync) vs bất đồng bộ (async)  
- Commit theo từng record vs theo batch  

---

## 1. Khái niệm chung

- **Offset**: vị trí của message trong partition.  
- **Commit offset**: lưu lại “đã xử lý đến đâu” để khi consumer restart sẽ đọc tiếp từ đó.  
- **Manual commit**: bạn tự quyết định khi nào commit offset (thay vì để Kafka auto-commit).

---

## 2. Commit đồng bộ (Synchronous Commit)

- **Đặc điểm**:
  - Gọi commit và **chờ** Kafka xác nhận.
  - Đảm bảo offset đã được lưu thành công trước khi xử lý tiếp.
  - An toàn hơn nhưng có thể chậm hơn (tăng độ trễ).

### Ví dụ Spring Kafka – commit sync theo batch

```java
@Configuration
@EnableKafka
public class KafkaConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);

        // Tắt auto-commit
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE);

        return factory;
    }
}
```

```java
@KafkaListener(topics = "demo-topic", groupId = "demo-group")
public void listen(List<String> messages, Acknowledgment ack) {
    // Xử lý cả batch
    for (String msg : messages) {
        System.out.println("Received: " + msg);
        // xử lý nghiệp vụ...
    }

    // Commit offset đồng bộ cho cả batch
    ack.acknowledge();
}
```

**Input (Kafka messages)**:  
- Partition 0: offset 10, 11, 12 (3 message)

**Output (log + offset)**:  
- Log:  
  - `Received: msg-10`  
  - `Received: msg-11`  
  - `Received: msg-12`  
- Offset commit: consumer group lưu offset = 13 (đã xử lý đến 12, commit next offset).

## 3. Commit bất đồng bộ (Asynchronous Commit)

- **Đặc điểm**:
  - Gửi yêu cầu commit và **không chờ** Kafka trả về.
  - Nhanh hơn, giảm độ trễ.
  - Có rủi ro: nếu commit thất bại, bạn có thể không biết ngay; có thể mất đồng bộ giữa xử lý và offset.

Trong Spring Kafka, commit async thường được dùng ở mức **Kafka client thuần** hoặc cấu hình nâng cao. Với `KafkaConsumer` thuần:

```java
KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Collections.singletonList("demo-topic"));

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf("Received: key=%s value=%s offset=%d%n",
                    record.key(), record.value(), record.offset());
            // xử lý nghiệp vụ...
        }

        // Commit offset bất đồng bộ
        consumer.commitAsync((offsets, exception) -> {
            if (exception != null) {
                System.err.println("Commit async failed: " + exception.getMessage());
            } else {
                System.out.println("Commit async success: " + offsets);
            }
        });
    }
} finally {
    consumer.close();
}
```

**Input**: 5 message liên tiếp.  
**Output**:  
- Log xử lý 5 message.  
- Log commit async (có thể thành công hoặc lỗi, nhưng vòng lặp không chờ).

## 4. Commit theo từng record (Per-record commit)

- **Đặc điểm**:
  - Sau khi xử lý **mỗi message**, commit offset ngay.
  - Độ an toàn cao: nếu crash, chỉ lặp lại tối đa 1 message.
  - Tốn chi phí commit nhiều lần → giảm throughput.

### Ví dụ Spring Kafka – commit per record (manual)

```java
@KafkaListener(topics = "demo-topic", groupId = "demo-group")
public void listenPerRecord(String message, Acknowledgment ack) {
    System.out.println("Received: " + message);
    // xử lý nghiệp vụ...

    // Commit offset cho message này
    ack.acknowledge();
}
```

**Input**: 3 message: `A`, `B`, `C`  
**Output**:  
- Log:  
  - `Received: A` → commit offset sau A  
  - `Received: B` → commit offset sau B  
  - `Received: C` → commit offset sau C  

Nếu crash sau khi xử lý B nhưng trước khi xử lý C, khi restart sẽ đọc lại từ C.

---

## 5. Commit theo batch (Per-batch commit)

- **Đặc điểm**:
  - Xử lý nhiều message (batch) rồi commit một lần.
  - Hiệu năng tốt hơn (ít commit hơn).
  - Nếu crash giữa batch, có thể phải xử lý lại nhiều message (từ đầu batch).

### Ví dụ Spring Kafka – batch listener

```java
@Configuration
@EnableKafka
public class KafkaBatchConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> batchFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);
        factory.setBatchListener(true); // bật batch mode
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE);
        return factory;
    }
}
```

```java
@KafkaListener(
        topics = "demo-topic",
        groupId = "demo-group",
        containerFactory = "batchFactory"
)
public void listenBatch(List<String> messages, Acknowledgment ack) {
    for (String msg : messages) {
        System.out.println("Received in batch: " + msg);
        // xử lý nghiệp vụ...
    }
    // Commit offset cho cả batch
    ack.acknowledge();
}
```

**Input**: batch 10 message.  
**Output**:  
- Log 10 dòng `Received in batch: ...`  
- 1 lần commit offset cho cả 10 message.

---

## 6. So sánh nhanh

- **Sync commit**:
  - Ưu: an toàn, biết chắc commit thành công.
  - Nhược: chậm hơn, tăng latency.

- **Async commit**:
  - Ưu: nhanh, throughput cao.
  - Nhược: khó xử lý lỗi commit, có thể mất đồng bộ.

- **Per-record commit**:
  - Ưu: ít mất dữ liệu, dễ reasoning.
  - Nhược: nhiều commit → giảm hiệu năng.

- **Per-batch commit**:
  - Ưu: hiệu năng tốt, ít commit.
  - Nhược: có thể phải xử lý lại nhiều message nếu crash.
