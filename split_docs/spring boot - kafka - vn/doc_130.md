
Dưới đây là tóm tắt ngắn gọn về việc **tuning Producer/Consumer Kafka trong Spring Boot** cho hai mục tiêu khác nhau: **độ trễ thấp (low latency)** và **băng thông cao (high throughput)**, kèm ví dụ code.

---

## 1. Nguyên tắc chung

- **Low latency**: gửi/nhận nhanh nhất có thể, chấp nhận tốn tài nguyên hơn, batch nhỏ.
- **High throughput**: tối đa hóa số lượng message/giây, chấp nhận độ trễ cao hơn, batch lớn.

---

## 2. Tuning Producer

### 2.1. Cấu hình cho Low Latency

Mục tiêu: gửi message gần như ngay lập tức.

Các tham số chính:

- `linger.ms` = 0–5 (gần như không chờ gom batch)
- `batch.size` = nhỏ (ví dụ 16KB)
- `acks` = 1 (hoặc 0 nếu chấp nhận mất dữ liệu)
- `compression.type` = none hoặc lz4 (nếu CPU đủ)
- `max.in.flight.requests.per.connection` = 1–5 (giảm reordering)
- `buffer.memory` = vừa phải

**application.yml (Producer – low latency)**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      acks: 1
      linger-ms: 0
      batch-size: 16384        # 16KB
      buffer-memory: 33554432  # 32MB
      compression-type: none
      max-in-flight-requests-per-connection: 1
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

**Ví dụ Producer (Spring Boot)**

```java
@Service
public class LowLatencyProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public LowLatencyProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void send(String topic, String key, String value) {
        long start = System.currentTimeMillis();
        ListenableFuture<SendResult<String, String>> future =
                kafkaTemplate.send(topic, key, value);

        future.addCallback(result -> {
            long latency = System.currentTimeMillis() - start;
            System.out.println("Sent message with latency = " + latency + " ms");
        }, ex -> {
            System.err.println("Send failed: " + ex.getMessage());
        });
    }
}
```

**Input**: `topic = "orders"`, `key = "order-1"`, `value = "{\"id\":1,\"amount\":100}"`  
**Output (log)**: `Sent message with latency = 3 ms` (ví dụ)

---

### 2.2. Cấu hình cho High Throughput

Mục tiêu: gửi được nhiều message/giây nhất.

Các tham số chính:

- `linger.ms` = 20–100 (chờ gom batch)
- `batch.size` = lớn (32KB–128KB)
- `acks` = all (an toàn + throughput tốt với batch)
- `compression.type` = snappy hoặc lz4
- `buffer.memory` = lớn hơn

**application.yml (Producer – high throughput)**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      acks: all
      linger-ms: 50           # chờ 50ms để gom batch
      batch-size: 65536       # 64KB
      buffer-memory: 67108864 # 64MB
      compression-type: snappy
      max-in-flight-requests-per-connection: 5
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

## 3. Tuning Consumer

### 3.1. Cấu hình cho Low Latency

Mục tiêu: xử lý message càng sớm càng tốt sau khi được publish.

Các tham số chính:

- `fetch.min.bytes` = nhỏ (1–1024)
- `fetch.max.wait.ms` = nhỏ (10–50)
- `max.poll.records` = nhỏ (10–100)
- `enable.auto.commit` = false (tự commit sau khi xử lý xong)
- `session.timeout.ms` = 10–15s (tùy hệ thống)
- `concurrency` = tăng số consumer (nếu cần)

**application.yml (Consumer – low latency)**

```yaml
spring:
  kafka:
    consumer:
      group-id: low-latency-group
      enable-auto-commit: false
      auto-offset-reset: earliest
      fetch-min-size: 1
      fetch-max-wait: 20
      max-poll-records: 50
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
    listener:
      ack-mode: MANUAL_IMMEDIATE
      concurrency: 3
```

**Listener ví dụ**

```java
@Service
public class LowLatencyConsumer {

    @KafkaListener(topics = "orders", groupId = "low-latency-group")
    public void listen(String message, Acknowledgment ack) {
        long start = System.currentTimeMillis();
        // Xử lý message
        System.out.println("Received: " + message);
        ack.acknowledge();
        long latency = System.currentTimeMillis() - start;
        System.out.println("Process latency = " + latency + " ms");
    }
}
```

**Input**: message `"{"id":1,"amount":100}"`  
**Output (log)**:  
- `Received: {"id":1,"amount":100}`  
- `Process latency = 2 ms` (ví dụ)

---

### 3.2. Cấu hình cho High Throughput

Mục tiêu: xử lý nhiều message/giây, chấp nhận độ trễ cao hơn.

Các tham số chính:

- `fetch.min.bytes` = lớn (50KB–1MB)
- `fetch.max.wait.ms` = 50–500
- `max.poll.records` = lớn (500–1000+)
- `enable.auto.commit` = true (hoặc batch commit)
- `concurrency` = cao (tùy số partition)

**application.yml (Consumer – high throughput)**

```yaml
spring:
  kafka:
    consumer:
      group-id: high-throughput-group
      enable-auto-commit: true
      auto-offset-reset: earliest
      fetch-min-size: 524288      # 512KB
      fetch-max-wait: 200
      max-poll-records: 1000
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
    listener:
      concurrency: 6
```

**Listener xử lý batch (nếu dùng)**

```java
@Service
public class HighThroughputConsumer {

    @KafkaListener(topics = "orders", groupId = "high-throughput-group")
    public void listen(List<String> messages) {
        System.out.println("Received batch size = " + messages.size());
        // Xử lý batch
    }
}
```

**Input**: batch 1000 messages  
**Output (log)**: `Received batch size = 1000`

---

## 4. Tóm tắt nhanh

- **Low latency**:
  - Producer: `linger.ms` thấp, `batch.size` nhỏ, có thể `acks=1`.
  - Consumer: `fetch.min.bytes` nhỏ, `fetch.max.wait.ms` thấp, `max.poll.records` nhỏ.
- **High throughput**:
  - Producer: `linger.ms` cao hơn, `batch.size` lớn, `compression` bật, `acks=all`.
  - Consumer: `fetch.min.bytes` lớn, `fetch.max.wait.ms` cao hơn, `max.poll.records` lớn, xử lý theo batch.

Nếu bạn cho biết thêm: kích thước message, QPS mong muốn, và yêu cầu mất dữ liệu có chấp nhận được không, tôi có thể gợi ý bộ thông số cụ thể hơn.
