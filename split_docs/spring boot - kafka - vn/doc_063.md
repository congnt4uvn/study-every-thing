
Dưới đây là tóm tắt ngắn gọn về các kịch bản lỗi trong Kafka liên quan đến **leader election** và **rebalancing**, kèm ví dụ Spring Boot (Spring Kafka) và mô tả input/output.

---

## 1. Khái niệm chính

### Leader election (bầu leader)
- Mỗi partition có:
  - **Leader**: broker xử lý đọc/ghi.
  - **Followers**: sao chép dữ liệu từ leader.
- Khi **leader broker** bị lỗi:
  - Kafka sẽ **chọn leader mới** từ danh sách ISR (in-sync replicas).
  - Nếu không có replica nào trong ISR, có thể:
    - Chờ replica bắt kịp (an toàn dữ liệu, nhưng downtime).
    - Hoặc cho phép chọn replica không đồng bộ (có thể mất dữ liệu).

### Rebalancing (tái phân phối)
- Xảy ra khi:
  - Consumer join/leave group.
  - Topic/partition thay đổi.
  - Lỗi broker/consumer.
- Kết quả:
  - Phân vùng được phân lại cho các consumer.
  - Trong lúc rebalancing, consumer **tạm dừng** nhận message.

---

## 2. Kịch bản lỗi thường gặp

1. **Leader broker chết**
   - Kafka chọn leader mới cho các partition bị ảnh hưởng.
   - Producer/consumer có thể bị lỗi tạm thời:
     - `LEADER_NOT_AVAILABLE`
     - `NOT_LEADER_FOR_PARTITION`
   - Sau khi leader mới ổn định, client tiếp tục hoạt động.

2. **Follower chậm hoặc chết**
   - Replica đó bị loại khỏi ISR.
   - Nếu leader chết và không còn ISR:
     - Có thể mất dữ liệu hoặc downtime, tùy cấu hình:
       - `unclean.leader.election.enable`
       - `min.insync.replicas`

3. **Consumer group rebalancing liên tục**
   - Do:
     - Consumer crash/timeout (session timeout).
     - Thay đổi số lượng consumer.
     - Thay đổi subscription (topic).
   - Hậu quả:
     - Tăng latency.
     - Có thể xử lý trùng message nếu offset commit không hợp lý.

---

## 3. Cấu hình quan trọng (Kafka server)

```properties
# Cho phép bầu leader không sạch (có thể mất dữ liệu)
unclean.leader.election.enable=false

# Số replica tối thiểu phải in-sync để cho phép ghi
min.insync.replicas=2

# Số replica tổng
default.replication.factor=3
```

## 4. Spring Boot + Spring Kafka: ví dụ xử lý lỗi & rebalancing

### 4.1. Consumer config (Java)

```java
@Configuration
@EnableKafka
public class KafkaConsumerConfig {

    @Bean
    public Map<String, Object> consumerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "demo-group");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
                  StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
                  StringDeserializer.class);

        // Giảm rebalancing không cần thiết
        props.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG, "10000");
        props.put(ConsumerConfig.HEARTBEAT_INTERVAL_MS_CONFIG, "3000");
        props.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, "300000");

        // Tự commit offset hoặc dùng manual
        props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, false);

        return props;
    }

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        return new DefaultKafkaConsumerFactory<>(consumerConfigs());
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String>
    kafkaListenerContainerFactory() {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());

        // Xử lý lỗi tạm thời (ví dụ leader election)
        factory.setCommonErrorHandler(new DefaultErrorHandler(
                new FixedBackOff(1000L, 3L) // retry 3 lần, mỗi lần cách 1s
        ));

        // Log rebalancing
        factory.getContainerProperties().setConsumerRebalanceListener(
                new ConsumerRebalanceListener() {
                    @Override
                    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
                        System.out.println("Partitions revoked: " + partitions);
                    }

                    @Override
                    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
                        System.out.println("Partitions assigned: " + partitions);
                    }
                }
        );

        return factory;
    }
}
```

### 4.2. Consumer với @KafkaListener (manual commit)

```java
@Component
public class DemoConsumer {

    @KafkaListener(topics = "demo-topic", groupId = "demo-group")
    public void listen(String message, Acknowledgment ack) {
        try {
            // Input: message từ Kafka
            System.out.println("Received message: " + message);

            // Xử lý business logic
            // ...

            // Commit offset sau khi xử lý thành công
            ack.acknowledge();
        } catch (Exception e) {
            // Không commit offset -> message sẽ được đọc lại sau khi rebalancing hoặc restart
            System.err.println("Error processing message: " + e.getMessage());
            // Có thể ném exception để DefaultErrorHandler xử lý retry
            throw e;
        }
    }
}
```

**Input ví dụ**:  
Producer gửi message `"order-123-created"` vào topic `demo-topic`.

**Output (log consumer)**:
```text
Partitions assigned: [demo-topic-0]
Received message: order-123-created
```

Nếu trong lúc xử lý, broker leader của `demo-topic-0` chết:
- Kafka bầu leader mới.
- Consumer có thể log:
```text
Partitions revoked: [demo-topic-0]
Partitions assigned: [demo-topic-0]
```
- Message có thể được đọc lại (nếu chưa commit offset).

## 5. Producer: xử lý lỗi leader election

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public Map<String, Object> producerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
                  StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
                  StringSerializer.class);

        // Đảm bảo ghi an toàn
        props.put(ProducerConfig.ACKS_CONFIG, "all");
        props.put(ProducerConfig.RETRIES_CONFIG, 5);
        props.put(ProducerConfig.RETRY_BACKOFF_MS_CONFIG, 1000);

        return props;
    }

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        return new DefaultKafkaProducerFactory<>(producerConfigs());
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

Gửi message:

```java
@Service
public class DemoProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public DemoProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderCreated(String orderId) {
        String message = "order-" + orderId + "-created";
        kafkaTemplate.send("demo-topic", orderId, message)
                     .addCallback(
                         result -> System.out.println("Sent: " + message),
                         ex -> System.err.println("Send failed: " + ex.getMessage())
                     );
    }
}
```

**Input**: `sendOrderCreated("123")`  
**Output (log)**:
```text
Sent: order-123-created
```
Nếu đang leader election, có thể log:
```text
Send failed: NOT_LEADER_FOR_PARTITION
```
và producer sẽ retry theo cấu hình.

## 6. Ý chính cần nhớ

- **Leader election** đảm bảo partition vẫn hoạt động khi broker leader chết, nhưng có thể gây:
  - Lỗi tạm thời cho producer/consumer.
  - Nguy cơ mất dữ liệu nếu cấu hình không an toàn.
- **Rebalancing** là quá trình bình thường nhưng:
  - Làm gián đoạn xử lý message tạm thời.
  - Cần cấu hình timeout, commit offset, và xử lý lỗi hợp lý để tránh trùng lặp/mất message.
- Spring Boot + Spring Kafka cung cấp:
  - `DefaultErrorHandler`, `ConsumerRebalanceListener`, manual commit (`Acknowledgment`) để kiểm soát hành vi trong các kịch bản lỗi này.
