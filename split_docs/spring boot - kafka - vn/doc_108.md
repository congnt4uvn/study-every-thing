
Dưới đây là tóm tắt ngắn gọn về **design pattern Multi-tenant topics vs Per-tenant topics** trong Kafka + Spring Boot, kèm ví dụ code và input/output minh họa.

---

## 1. Bối cảnh: Multi-tenant trong Kafka

Multi-tenant = nhiều “khách hàng/tenant” cùng dùng chung hệ thống.  
Vấn đề: tổ chức topic Kafka như thế nào?

Có 2 pattern chính:

1. **Multi-tenant topic (shared topic)**: nhiều tenant dùng chung 1 (hoặc vài) topic.
2. **Per-tenant topic**: mỗi tenant có topic riêng.

---

## 2. Pattern 1: Multi-tenant Topic (Shared Topic)

### Ý tưởng chính

- Tất cả tenant gửi message vào **cùng một topic**, ví dụ: `orders`.
- Phân biệt tenant bằng **header** hoặc **field trong payload** (ví dụ `tenantId`).
- Consumer đọc từ topic chung, sau đó **filter theo tenantId**.

### Ưu điểm

- Ít topic → dễ quản lý, ít gánh nặng cho Kafka cluster.
- Dễ scale theo **partition** thay vì số lượng topic.
- Phù hợp khi có **nhiều tenant nhỏ**, traffic thấp.

### Nhược điểm

- Logic filter/route phức tạp hơn ở consumer.
- Khó tách biệt hoàn toàn dữ liệu giữa các tenant (về mặt hạ tầng).
- Nếu một tenant có traffic rất lớn, có thể ảnh hưởng tenant khác (noise).

### Ví dụ: Producer (Spring Boot)

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, OrderEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String tenantId, OrderEvent event) {
        kafkaTemplate.send(
                MessageBuilder
                        .withPayload(event)
                        .setHeader(KafkaHeaders.TOPIC, "orders") // topic chung
                        .setHeader("tenantId", tenantId)        // phân biệt tenant
                        .build()
        );
    }
}
```

**Input (gửi message)**

```json
{
  "tenantId": "tenantA",
  "orderId": "ORD-1001",
  "amount": 150000
}
```

**Message trên Kafka (payload + header)**

- Topic: `orders`
- Headers: `tenantId=tenantA`
- Payload:

```json
{
  "orderId": "ORD-1001",
  "amount": 150000
}
```

### Ví dụ: Consumer filter theo tenant

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void consume(ConsumerRecord<String, OrderEvent> record) {
        String tenantId = (String) record.headers()
                .lastHeader("tenantId")
                .value();

        OrderEvent event = record.value();

        // Ví dụ: chỉ xử lý tenantA
        if ("tenantA".equals(tenantId)) {
            processTenantAOrder(event);
        } else {
            // route sang luồng xử lý khác hoặc bỏ qua
        }
    }

    private void processTenantAOrder(OrderEvent event) {
        // xử lý đơn hàng cho tenantA
    }
}
```

**Output (log)**

```text
Processing order ORD-1001 for tenant tenantA, amount=150000
```

## 3. Pattern 2: Per-tenant Topic

### Ý tưởng chính

- Mỗi tenant có **topic riêng**, ví dụ:
  - `orders-tenantA`
  - `orders-tenantB`
- Producer gửi vào topic tương ứng với tenant.
- Consumer subscribe đúng topic của tenant.

### Ưu điểm

- Cô lập tốt giữa các tenant (isolation):
  - Có thể cấu hình retention, partition, ACL riêng.
  - Dễ scale/monitor từng tenant.
- Dễ áp dụng chính sách bảo mật, quota riêng.

### Nhược điểm

- Số lượng topic tăng mạnh nếu nhiều tenant.
- Quản lý lifecycle topic phức tạp (tạo/xóa topic khi tenant tạo/suspend).
- Kafka cluster có giới hạn thực tế về số topic/partition.

### Ví dụ: Producer chọn topic theo tenant

```java
@Service
public class OrderProducerPerTenant {

    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public OrderProducerPerTenant(KafkaTemplate<String, OrderEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String tenantId, OrderEvent event) {
        String topic = "orders-" + tenantId; // ví dụ: orders-tenantA
        kafkaTemplate.send(topic, event.getOrderId(), event);
    }
}
```

**Input (gửi message)**

```json
{
  "tenantId": "tenantA",
  "orderId": "ORD-2001",
  "amount": 300000
}
```

**Message trên Kafka**

- Topic: `orders-tenantA`
- Key: `ORD-2001`
- Payload:

```json
{
  "orderId": "ORD-2001",
  "amount": 300000
}
```

### Ví dụ: Consumer cho từng tenant

```java
@Service
public class OrderConsumerTenantA {

    @KafkaListener(topics = "orders-tenantA", groupId = "order-service-tenantA")
    public void consumeTenantA(OrderEvent event) {
        // xử lý đơn hàng cho tenantA
        System.out.println("TenantA order: " + event.getOrderId());
    }
}
```

**Output (log)**

```text
TenantA order: ORD-2001
```

## 4. Khi nào chọn pattern nào?

**Chọn Multi-tenant Topic (shared)** khi:

- Có **rất nhiều tenant**, mỗi tenant traffic nhỏ.
- Muốn giảm số lượng topic, đơn giản hóa quản lý Kafka.
- Chấp nhận xử lý logic phân tách tenant ở tầng ứng dụng.

**Chọn Per-tenant Topic** khi:

- Cần **cô lập mạnh** giữa các tenant (bảo mật, SLA, quota).
- Một số tenant có traffic lớn, cần cấu hình riêng (partition, retention).
- Số lượng tenant **không quá lớn** hoặc có cơ chế quản lý topic tự động.

---

## 5. Tóm tắt ngắn

- **Multi-tenant topic**: 1 topic cho nhiều tenant, phân biệt bằng `tenantId` trong header/payload.  
  → Ít topic, dễ quản lý, nhưng logic consumer phức tạp hơn, isolation kém hơn.

- **Per-tenant topic**: mỗi tenant 1 topic riêng.  
  → Isolation tốt, cấu hình riêng cho từng tenant, nhưng nhiều topic, quản lý khó hơn.

Nếu bạn cho biết thêm: số lượng tenant, mức traffic, yêu cầu bảo mật/SLA, mình có thể gợi ý pattern phù hợp hơn cho trường hợp cụ thể.
