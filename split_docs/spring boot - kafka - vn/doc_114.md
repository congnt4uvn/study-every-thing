
Dưới đây là tóm tắt ngắn gọn về các pattern độ tin cậy trong Kafka + Spring Boot: **DLQ, retry topics, exponential backoff**.

---

## 1. Dead Letter Queue (DLQ)

**Ý tưởng chính**  
- Khi consumer xử lý message bị lỗi (sau khi đã retry đủ số lần), message sẽ được gửi sang một topic đặc biệt gọi là **DLQ**.  
- DLQ giúp không làm “kẹt” consumer và cho phép phân tích, xử lý thủ công hoặc xử lý bằng job riêng.

**Cấu hình Spring Kafka (ví dụ)**

```java
@Bean
public DeadLetterPublishingRecoverer deadLetterPublishingRecoverer(KafkaTemplate<Object, Object> template) {
    // Topic DLQ sẽ có hậu tố ".DLT"
    return new DeadLetterPublishingRecoverer(template,
            (record, ex) -> new TopicPartition(record.topic() + ".DLT", record.partition()));
}

@Bean
public DefaultErrorHandler errorHandler(DeadLetterPublishingRecoverer recoverer) {
    // Không retry, gửi thẳng vào DLQ
    DefaultErrorHandler errorHandler = new DefaultErrorHandler(recoverer, new FixedBackOff(0L, 0));
    return errorHandler;
}
```

**Input/Output ví dụ**

- Input topic: `orders`
- DLQ topic: `orders.DLT`

Khi consumer đọc message từ `orders` và ném exception không recover được:
- **Input**: `{"orderId": 123, "status": "NEW"}`
- **Output**: message này được publish sang `orders.DLT` để xử lý sau.

## 2. Retry Topics

**Ý tưởng chính**  
- Thay vì retry ngay lập tức trong cùng consumer, ta dùng **các topic retry** với delay khác nhau.  
- Luồng: `main-topic` → `retry-topic-1` → `retry-topic-2` → … → `DLQ`  
- Mỗi retry topic có consumer riêng, chạy với delay hoặc backoff khác nhau.

### 2.1. Dùng Spring Kafka RetryTopic (từ 2.7+)

Spring Kafka hỗ trợ annotation `@RetryableTopic` để tự động tạo retry topics.

**Ví dụ code**

```java
@EnableKafka
@Configuration
public class KafkaConfig {
}

@Service
public class OrderConsumer {

    @RetryableTopic(
            attempts = 3,                      // tổng số lần xử lý (1 lần chính + 2 lần retry)
            backoff = @Backoff(delay = 2000),  // delay 2s giữa các lần
            dltTopicSuffix = ".DLT"            // hậu tố cho DLQ
    )
    @KafkaListener(topics = "orders", groupId = "order-group")
    public void consumeOrder(String message) {
        // Giả sử lỗi khi orderId lẻ
        if (message.contains("\"orderId\":") && message.contains("1")) {
            throw new RuntimeException("Lỗi xử lý order");
        }
        System.out.println("Processed: " + message);
    }

    @DltHandler
    public void dltHandler(String message) {
        System.out.println("DLQ received: " + message);
    }
}
```

**Luồng Input/Output**

- Input topic: `orders`
- Retry topics (tự tạo): `orders-retry-0`, `orders-retry-1`, …
- DLQ topic: `orders.DLT`

Ví dụ message:

```json
{"orderId": 1, "status": "NEW"}
```

1. Lần 1: consumer `orders` xử lý → lỗi → gửi sang `orders-retry-0`
2. Lần 2: consumer `orders-retry-0` xử lý sau 2s → lỗi → gửi sang `orders-retry-1`
3. Lần 3: consumer `orders-retry-1` xử lý sau 2s → lỗi → gửi sang `orders.DLT`

## 3. Exponential Backoff (Backoff lũy thừa)

**Ý tưởng chính**  
- Thay vì retry với delay cố định, dùng delay tăng dần:  
  - Ví dụ: 1s → 2s → 4s → 8s → …  
- Giảm tải hệ thống khi lỗi tạm thời (ví dụ service downstream bị down).

### 3.1. Dùng `DefaultErrorHandler` với `ExponentialBackOffWithMaxRetries`

```java
@Bean
public DefaultErrorHandler errorHandler(DeadLetterPublishingRecoverer recoverer) {
    ExponentialBackOffWithMaxRetries backOff = new ExponentialBackOffWithMaxRetries(3);
    backOff.setInitialInterval(1000L);   // 1s
    backOff.setMultiplier(2.0);          // nhân 2 mỗi lần
    backOff.setMaxInterval(8000L);       // tối đa 8s

    DefaultErrorHandler errorHandler = new DefaultErrorHandler(recoverer, backOff);
    return errorHandler;
}
```

**Chuỗi delay**

- Lần 1 lỗi: delay ~1s
- Lần 2 lỗi: delay ~2s
- Lần 3 lỗi: delay ~4s  
Sau đó nếu vẫn lỗi → gửi sang DLQ.

---

## 4. Kết hợp các pattern

Thực tế thường **kết hợp cả 3**:

1. **Retry trong memory** (error handler) với **exponential backoff** cho lỗi tạm thời, số lần ít.
2. Nếu vẫn lỗi → **retry topics** để tách xử lý, giảm áp lực consumer chính.
3. Sau số lần retry tối đa → gửi vào **DLQ** để phân tích/ xử lý thủ công.
