
Dưới đây là tóm tắt ngắn gọn về “Replication: Unclean leader election, data loss scenarios” trong Kafka, kèm ví dụ và một chút Spring Boot.

---

## 1. Khái niệm cơ bản

### Replication trong Kafka
- Mỗi partition có:
  - **Leader**: broker chính nhận/ghi dữ liệu.
  - **Followers**: broker sao chép dữ liệu từ leader.
- **ISR (In-Sync Replicas)**: tập các replica đang bắt kịp leader (không bị trễ quá ngưỡng).

---

## 2. Unclean Leader Election là gì?

- Khi **leader** của một partition bị down, Kafka phải bầu leader mới.
- **Clean leader election**: chỉ chọn leader mới từ **ISR** → an toàn dữ liệu.
- **Unclean leader election**: cho phép chọn leader mới từ **replica không nằm trong ISR** (out-of-sync) → có thể **mất dữ liệu**.

Cấu hình chính:
```properties
unclean.leader.election.enable=true  # cho phép bầu leader không sạch (có thể mất dữ liệu)
unclean.leader.election.enable=false # chỉ bầu leader từ ISR (an toàn hơn)
```

---

## 3. Các kịch bản mất dữ liệu (Data Loss Scenarios)

### 3.1. Kịch bản 1: Unclean leader election bật (true)

1. Producer ghi dữ liệu vào leader (Broker 1).
2. Followers (Broker 2, 3) chưa kịp replicate (chưa vào ISR hoặc bị trễ).
3. Broker 1 bị down.
4. Kafka bầu leader mới là Broker 2 (không nằm trong ISR, thiếu dữ liệu mới nhất).
5. Khi Broker 1 quay lại, dữ liệu mới nhất trên Broker 1 bị **ghi đè** hoặc **bỏ qua** → **mất dữ liệu**.

**Kết luận**: `unclean.leader.election.enable=true` giúp hệ thống tiếp tục hoạt động (tăng availability) nhưng có nguy cơ mất dữ liệu.

---

### 3.2. Kịch bản 2: Unclean leader election tắt (false)

1. Producer ghi dữ liệu vào leader.
2. Leader bị down.
3. Không có replica nào trong ISR đủ điều kiện làm leader.
4. Partition trở nên **unavailable** (không đọc/ghi được) cho đến khi leader cũ quay lại hoặc replica bắt kịp.

**Kết luận**: `unclean.leader.election.enable=false` ưu tiên **an toàn dữ liệu** hơn **tính sẵn sàng**.

## 4. Cấu hình an toàn hơn cho Producer

Để giảm nguy cơ mất dữ liệu, cấu hình Producer như sau:

```java
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());

// Đảm bảo ghi vào tất cả replica trong ISR
props.put(ProducerConfig.ACKS_CONFIG, "all");
// Retry khi lỗi tạm thời
props.put(ProducerConfig.RETRIES_CONFIG, 3);
// Đảm bảo thứ tự trong 1 partition
props.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 1);
// Bật idempotent producer để tránh ghi trùng
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
```

## 5. Ví dụ Spring Boot + Kafka (Producer/Consumer)

### 5.1. application.yml

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      properties:
        acks: all
        enable.idempotence: true
    consumer:
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

---

### 5.2. Producer Service (Spring Boot)

```java
@Service
public class KafkaProducerService {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public KafkaProducerService(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        ListenableFuture<SendResult<String, String>> future =
                kafkaTemplate.send(topic, key, value);

        future.addCallback(
            result -> {
                if (result != null) {
                    RecordMetadata meta = result.getRecordMetadata();
                    System.out.printf("Sent: key=%s, value=%s to %s-%d@%d%n",
                            key, value, meta.topic(), meta.partition(), meta.offset());
                }
            },
            ex -> System.err.println("Send failed: " + ex.getMessage())
        );
    }
}
```

**Input (REST)**:
```http
POST /api/messages
Content-Type: application/json

{
  "key": "user-1",
  "value": "Hello Kafka"
}
```

**Output (log)**:
```text
Sent: key=user-1, value=Hello Kafka to demo-topic-0@15
```

---

### 5.3. Consumer Listener (Spring Boot)

```java
@Service
public class KafkaConsumerService {

    @KafkaListener(topics = "demo-topic", groupId = "demo-group")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf("Received: key=%s, value=%s from %s-%d@%d%n",
                record.key(), record.value(),
                record.topic(), record.partition(), record.offset());
    }
}
```

**Output (log)**:
```text
Received: key=user-1, value=Hello Kafka from demo-topic-0@15
```

## 6. Tóm tắt ý chính (tiếng Việt)

- **Replication** giúp Kafka chịu lỗi bằng cách sao chép dữ liệu giữa các broker.
- **Unclean leader election**:
  - `true`: tăng tính sẵn sàng, nhưng có thể **mất dữ liệu** khi leader mới không có dữ liệu mới nhất.
  - `false`: ưu tiên **an toàn dữ liệu**, nhưng có thể làm partition **tạm thời không sử dụng được**.
- Để giảm rủi ro mất dữ liệu:
  - Tắt `unclean.leader.election.enable` nếu ưu tiên dữ liệu.
  - Cấu hình producer với `acks=all`, `enable.idempotence=true`, `retries>0`.
- Spring Boot + Kafka có thể cấu hình dễ dàng qua `application.yml` và sử dụng `KafkaTemplate` (producer) + `@KafkaListener` (consumer).
