
Dưới đây là tóm tắt ngắn gọn (tiếng Việt) về chủ đề: **Version Control & CI/CD: Xử lý thay đổi schema trong môi trường production (Oracle Database)**, kèm ví dụ code.

---

## 1. Ý tưởng chính

1. **Schema cũng phải được quản lý phiên bản (version control)**  
   - Không chỉ code ứng dụng, mà **các thay đổi DB (DDL, script migration)** cũng phải được lưu trong Git/SVN.
   - Mỗi thay đổi schema là một “migration” có ID, mô tả, và script tương ứng.

2. **Tự động hóa qua CI/CD**  
   - Pipeline CI/CD sẽ:
     - Kiểm tra cú pháp script SQL.
     - Chạy migration trên môi trường dev/test.
     - Nếu pass, mới deploy lên staging/production.
   - Giảm rủi ro thao tác tay, đảm bảo tính lặp lại (repeatable).

3. **Migration theo từng bước nhỏ, có thể rollback**  
   - Thay đổi schema nên:
     - Nhỏ, rõ ràng, có thể test độc lập.
     - Có chiến lược rollback (hoặc script rollback, hoặc chiến lược “roll-forward”).
   - Tránh thay đổi lớn, đột ngột trên production.

4. **Quy trình chuẩn cho thay đổi schema**  
   - Thiết kế → Viết script migration → Code review → Test trên dev/test → Chạy qua CI/CD → Deploy production.
   - Ghi log/metadata các migration đã chạy (bảng lịch sử migration).

5. **Tương thích ngược (backward compatible)**  
   - Khi deploy, schema mới phải **tương thích với version code cũ** trong một khoảng thời gian:
     - Thêm cột mới (nullable) thay vì đổi kiểu dữ liệu đột ngột.
     - Dùng view hoặc synonym để giữ API DB ổn định.
   - Cho phép deploy ứng dụng và DB tách rời, giảm downtime.

## 2. Ví dụ: Quản lý migration schema với bảng lịch sử

### 2.1. Tạo bảng lưu lịch sử migration

```sql
-- Tạo bảng lưu các migration đã chạy
CREATE TABLE SCHEMA_MIGRATIONS (
    VERSION       VARCHAR2(50) PRIMARY KEY,
    DESCRIPTION   VARCHAR2(200),
    APPLIED_AT    DATE DEFAULT SYSDATE
);
```

**Ý nghĩa:**
- `VERSION`: mã phiên bản migration, ví dụ `V001`, `V002`.
- `DESCRIPTION`: mô tả ngắn.
- `APPLIED_AT`: thời điểm áp dụng.

---

### 2.2. Ví dụ migration: Thêm bảng mới

**File: `V001__create_customers_table.sql`**

```sql
-- V001: Tạo bảng CUSTOMERS
CREATE TABLE CUSTOMERS (
    ID           NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    NAME         VARCHAR2(100) NOT NULL,
    EMAIL        VARCHAR2(200),
    CREATED_AT   DATE DEFAULT SYSDATE
);

INSERT INTO SCHEMA_MIGRATIONS (VERSION, DESCRIPTION)
VALUES ('V001', 'Create CUSTOMERS table');
COMMIT;
```

**Kết quả (output) khi chạy:**
- Bảng `CUSTOMERS` được tạo.
- Bảng `SCHEMA_MIGRATIONS` có thêm dòng:

```sql
SELECT * FROM SCHEMA_MIGRATIONS;

-- OUTPUT (ví dụ)
-- VERSION | DESCRIPTION             | APPLIED_AT
-- ------- | ----------------------- | -------------------
-- V001    | Create CUSTOMERS table  | 2026-05-20 10:15:00
```

---

### 2.3. Ví dụ migration: Thêm cột mới (tương thích ngược)

**File: `V002__add_status_to_customers.sql`**

```sql
-- V002: Thêm cột STATUS, mặc định 'ACTIVE'
ALTER TABLE CUSTOMERS
    ADD (STATUS VARCHAR2(20) DEFAULT 'ACTIVE');

UPDATE SCHEMA_MIGRATIONS
SET DESCRIPTION = 'Add STATUS column to CUSTOMERS'
WHERE VERSION = 'V002';

-- Nếu chưa có dòng V002 thì:
INSERT INTO SCHEMA_MIGRATIONS (VERSION, DESCRIPTION)
VALUES ('V002', 'Add STATUS column to CUSTOMERS');

COMMIT;
```

**Kiểm tra:**

```sql
DESC CUSTOMERS;

-- OUTPUT (cột mới)
-- NAME      NULL?   TYPE
-- --------- ------- -------------
-- ID        NOT NULL NUMBER
-- NAME      NOT NULL VARCHAR2(100)
-- EMAIL              VARCHAR2(200)
-- CREATED_AT         DATE
-- STATUS             VARCHAR2(20)
```

## 3. Tích hợp vào CI/CD (mô phỏng)

Ví dụ pipeline (giả sử dùng script shell + SQL*Plus):

```bash
#!/bin/bash
# deploy_migrations.sh

ORACLE_CONN="user/password@//host:1521/ORCLPDB1"

# Danh sách migration theo thứ tự
MIGRATIONS=(
  "V001__create_customers_table.sql"
  "V002__add_status_to_customers.sql"
)

for MIG in "${MIGRATIONS[@]}"; do
  VERSION=$(echo "$MIG" | cut -d'_' -f1)  # V001, V002

  # Kiểm tra đã chạy chưa
  EXISTS=$(sqlplus -s "$ORACLE_CONN" <<EOF
SET HEADING OFF FEEDBACK OFF PAGESIZE 0
SELECT COUNT(*) FROM SCHEMA_MIGRATIONS WHERE VERSION = '$VERSION';
EXIT;
EOF
)

  if [ "$EXISTS" -eq 0 ]; then
    echo "Applying migration: $MIG"
    sqlplus "$ORACLE_CONN" @"$MIG"
  else
    echo "Migration $MIG already applied, skipping."
  fi
done
```

**Ý tưởng:**
- CI/CD gọi script này trên dev/test/prod.
- Chỉ chạy những migration chưa có trong `SCHEMA_MIGRATIONS`.

## 4. Một số nguyên tắc quan trọng

- Mọi thay đổi schema phải:
  - Có script rõ ràng, lưu trong version control.
  - Được review và test trước khi lên production.
- Tránh:
  - Chạy lệnh DDL trực tiếp trên production mà không qua pipeline.
  - Thay đổi phá vỡ tương thích (breaking changes) mà không có kế hoạch nâng cấp nhiều bước.
- Nên:
  - Ghi log đầy đủ (ai, khi nào, migration nào).
  - Có chiến lược rollback hoặc roll-forward (migration sửa lỗi tiếp theo).

---

Nếu bạn muốn, tôi có thể viết thêm ví dụ cụ thể cho:
- Chiến lược rollback (ví dụ: thêm cột rồi rollback).
- Migration phức tạp hơn (đổi tên cột, tách bảng, thêm index, partition, v.v.).
