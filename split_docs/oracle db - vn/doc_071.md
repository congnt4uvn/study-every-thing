
Dưới đây là tóm tắt ngắn gọn về các kiểu refresh trong thiết kế vật lý (Materialized View) của Oracle, kèm ví dụ code và minh họa input/output.

---

## 1. Các kiểu Refresh chính

### 1.1. FAST REFRESH
- Chỉ cập nhật phần dữ liệu thay đổi (delta), không làm lại toàn bộ.
- Yêu cầu:
  - Bảng nguồn có **materialized view log**.
  - Câu query của MV phải thỏa điều kiện cho fast refresh.
- Ưu điểm: Nhanh, ít tốn tài nguyên.
- Nhược điểm: Cấu hình phức tạp hơn, có ràng buộc về cú pháp.

**Ví dụ:**

```sql
-- Bảng nguồn
CREATE TABLE sales (
    sale_id     NUMBER PRIMARY KEY,
    sale_date   DATE,
    amount      NUMBER
);

-- Log cho fast refresh
CREATE MATERIALIZED VIEW LOG ON sales
WITH ROWID, SEQUENCE (sale_date, amount)
INCLUDING NEW VALUES;

-- Materialized View với fast refresh
CREATE MATERIALIZED VIEW mv_sales_daily
BUILD IMMEDIATE
REFRESH FAST
START WITH SYSDATE
NEXT SYSDATE + 1/24  -- mỗi giờ
AS
SELECT TRUNC(sale_date) AS sale_day,
       SUM(amount)      AS total_amount
FROM   sales
GROUP BY TRUNC(sale_date);
```

**Input (dữ liệu mới vào bảng `sales`):**

```sql
INSERT INTO sales (sale_id, sale_date, amount)
VALUES (101, DATE '2024-05-20', 500);

COMMIT;
```

**Refresh:**

```sql
EXEC DBMS_MVIEW.REFRESH('MV_SALES_DAILY', 'F');  -- F = FAST
```

**Output (kết quả trong `mv_sales_daily`):**

```sql
SELECT * FROM mv_sales_daily;

SALE_DAY    TOTAL_AMOUNT
----------  ------------
20-05-2024  500
```

---

### 1.2. COMPLETE REFRESH
- Xóa và xây dựng lại toàn bộ dữ liệu trong MV.
- Không cần materialized view log.
- Ưu điểm: Đơn giản, ít ràng buộc.
- Nhược điểm: Chậm hơn, tốn tài nguyên, khóa nhiều.

**Ví dụ:**

```sql
CREATE MATERIALIZED VIEW mv_sales_complete
BUILD IMMEDIATE
REFRESH COMPLETE
START WITH SYSDATE
NEXT SYSDATE + 1  -- mỗi ngày
AS
SELECT TRUNC(sale_date) AS sale_day,
       SUM(amount)      AS total_amount
FROM   sales
GROUP BY TRUNC(sale_date);
```

**Refresh:**

```sql
EXEC DBMS_MVIEW.REFRESH('MV_SALES_COMPLETE', 'C');  -- C = COMPLETE
```

## 2. Thời điểm Refresh

### 2.1. ON COMMIT
- MV được refresh **tự động khi COMMIT** trên bảng nguồn.
- Thường dùng với FAST REFRESH.
- Phù hợp khi cần dữ liệu gần như real-time, nhưng có thể làm chậm transaction.

**Ví dụ:**

```sql
CREATE MATERIALIZED VIEW mv_sales_on_commit
BUILD IMMEDIATE
REFRESH FAST
ON COMMIT
AS
SELECT TRUNC(sale_date) AS sale_day,
       SUM(amount)      AS total_amount
FROM   sales
GROUP BY TRUNC(sale_date);
```

**Input:**

```sql
INSERT INTO sales (sale_id, sale_date, amount)
VALUES (102, DATE '2024-05-21', 300);

COMMIT;  -- lúc này MV được refresh tự động
```

**Output:**

```sql
SELECT * FROM mv_sales_on_commit;

SALE_DAY    TOTAL_AMOUNT
----------  ------------
20-05-2024  500
21-05-2024  300
```

---

### 2.2. ON DEMAND (ON DEMAND / ON REQUEST)
- MV chỉ refresh khi **gọi lệnh** (thủ công hoặc qua job).
- Thường dùng với `DBMS_MVIEW.REFRESH`.
- Phù hợp khi:
  - Dữ liệu không cần cập nhật liên tục.
  - Muốn chủ động thời điểm refresh (ngoài giờ cao điểm).

**Ví dụ:**

```sql
CREATE MATERIALIZED VIEW mv_sales_on_demand
BUILD IMMEDIATE
REFRESH FAST ON DEMAND
AS
SELECT TRUNC(sale_date) AS sale_day,
       SUM(amount)      AS total_amount
FROM   sales
GROUP BY TRUNC(sale_date);
```

**Refresh thủ công:**

```sql
-- FAST
EXEC DBMS_MVIEW.REFRESH('MV_SALES_ON_DEMAND', 'F');

-- hoặc COMPLETE
EXEC DBMS_MVIEW.REFRESH('MV_SALES_ON_DEMAND', 'C');
```

---

## 3. Tóm tắt nhanh

- **FAST REFRESH**: chỉ cập nhật phần thay đổi, cần MV log, nhanh.
- **COMPLETE REFRESH**: xây lại toàn bộ, không cần log, chậm hơn.
- **ON COMMIT**: refresh ngay khi COMMIT, dữ liệu gần real-time, có thể ảnh hưởng hiệu năng giao dịch.
- **ON DEMAND**: refresh khi được gọi, linh hoạt, thường dùng với job định kỳ.

Nếu bạn cho biết thêm bối cảnh (OLTP/OLAP, kích thước dữ liệu, tần suất cập nhật), tôi có thể gợi ý kiểu refresh phù hợp hơn.
