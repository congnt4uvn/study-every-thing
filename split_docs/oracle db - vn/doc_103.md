---

## 1. Version Control cho Oracle Database

**Ý chính:**
- Lưu trữ script DDL/DML (CREATE TABLE, ALTER TABLE, PACKAGE, VIEW, …) trong Git/SVN.
- Mỗi thay đổi schema phải đi qua commit, review (pull request), và tag version.
- Tách rõ:
  - `schema/` (cấu trúc DB)
  - `data/` (dữ liệu tham chiếu, seed data)
  - `scripts/` (script migration, rollback)

**Ví dụ cấu trúc repo:**
```text
db/
  schema/
    001_create_tables.sql
    002_add_indexes.sql
  data/
    001_seed_lookup_data.sql
  migrations/
    V1__init_schema.sql
    V2__add_customer_table.sql
    V3__alter_order_table.sql
```

## 2. CI/CD cho Oracle Database

**Ý chính:**
- CI: Tự động kiểm tra script DB khi có commit:
  - Kiểm tra syntax SQL/PLSQL
  - Chạy unit test (utPLSQL, SQLcl, …)
  - Chạy migration trên DB test
- CD: Tự động (hoặc bán tự động) triển khai lên:
  - DEV → TEST → UAT → PROD
- Dùng công cụ: Jenkins, GitLab CI, GitHub Actions, Azure DevOps, kết hợp Flyway, Liquibase, SQLcl, SQL*Plus.

**Ví dụ pipeline (GitLab CI, dùng Flyway):**
```yaml
stages:
  - build
  - test
  - deploy

variables:
  FLYWAY_URL: "jdbc:oracle:thin:@//test-db:1521/ORCLPDB1"
  FLYWAY_USER: "app_user"
  FLYWAY_PASSWORD: "secret"

build:
  stage: build
  script:
    - echo "Check SQL format & lint"
    - ./tools/sql_lint.sh db/schema

test:
  stage: test
  script:
    - flyway -locations=filesystem:db/migrations migrate
    - ./run_utplsql_tests.sh
  artifacts:
    when: always
    paths:
      - test-reports/

deploy_prod:
  stage: deploy
  when: manual
  script:
    - export FLYWAY_URL="jdbc:oracle:thin:@//prod-db:1521/ORCLPDB1"
    - flyway -locations=filesystem:db/migrations migrate
```

## 3. Chiến lược triển khai (Deployment Strategies)

### 3.1. Incremental / Migration-based

- Mỗi thay đổi DB là một migration file (V1, V2, V3…).
- Công cụ: Flyway, Liquibase.
- Dễ audit, dễ rollback (nếu viết script rollback).

**Ví dụ migration (Flyway – V2__add_customer_table.sql):**
```sql
-- Input: DB hiện tại chưa có bảng CUSTOMER
-- Output: DB có thêm bảng CUSTOMER

CREATE TABLE CUSTOMER (
    ID          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    NAME        VARCHAR2(100) NOT NULL,
    EMAIL       VARCHAR2(200),
    CREATED_AT  DATE DEFAULT SYSDATE
);

CREATE UNIQUE INDEX UX_CUSTOMER_EMAIL ON CUSTOMER(EMAIL);
```

### 3.2. State-based (so sánh trạng thái)

- Lưu “mô hình chuẩn” (model) của schema.
- Dùng tool so sánh (schema compare) giữa model và DB hiện tại, sinh script diff.
- Phù hợp khi có nhiều thay đổi phức tạp, nhưng cần kiểm soát script sinh ra.

### 3.3. Blue-Green / Shadow Schema (khi có downtime thấp)

- Tạo schema mới (GREEN) song song với schema cũ (BLUE).
- Chạy migration trên GREEN, test, sau đó chuyển ứng dụng sang GREEN.
- Có thể dùng DB link, materialized view, hoặc job để sync dữ liệu.

**Ví dụ ý tưởng:**
```sql
-- Input: Schema hiện tại APP_BLUE, tạo schema mới APP_GREEN
-- Output: APP_GREEN có cấu trúc mới, dữ liệu copy từ APP_BLUE

CREATE USER APP_GREEN IDENTIFIED BY "secret";
GRANT CONNECT, RESOURCE TO APP_GREEN;

-- Copy cấu trúc & dữ liệu (ví dụ đơn giản)
CREATE TABLE APP_GREEN.CUSTOMER AS SELECT * FROM APP_BLUE.CUSTOMER;

-- Sau khi test xong, app đổi connection string sang APP_GREEN
```

### 3.4. Rolling / Zero-downtime

- Thiết kế thay đổi tương thích ngược (backward compatible):
  - Bước 1: Thêm cột mới, view mới, package mới (không phá vỡ code cũ).
  - Bước 2: Deploy app mới dùng cấu trúc mới.
  - Bước 3: Xóa cột/đối tượng cũ sau khi chắc chắn không dùng.
- Thường kết hợp với feature flag ở tầng ứng dụng.

**Ví dụ thay đổi cột (2 bước):**

Bước 1 – Thêm cột mới, giữ cột cũ:
```sql
-- Input: Bảng CUSTOMER có cột FULL_NAME
-- Output: Thêm cột FIRST_NAME, LAST_NAME, vẫn giữ FULL_NAME

ALTER TABLE CUSTOMER ADD (
    FIRST_NAME VARCHAR2(50),
    LAST_NAME  VARCHAR2(50)
);

-- Tạm thời sync dữ liệu
UPDATE CUSTOMER
SET FIRST_NAME = REGEXP_SUBSTR(FULL_NAME, '^[^ ]+'),
    LAST_NAME  = REGEXP_SUBSTR(FULL_NAME, '[^ ]+$')
WHERE FULL_NAME IS NOT NULL;
```

Bước 2 – Sau khi app đã dùng FIRST_NAME/LAST_NAME:
```sql
-- Input: App không còn dùng FULL_NAME
-- Output: Xóa cột FULL_NAME

ALTER TABLE CUSTOMER DROP COLUMN FULL_NAME;
```

## 4. Rollback & Backup

**Ý chính:**
- Mỗi migration nên có kế hoạch rollback (hoặc script `undo`).
- Với thay đổi dữ liệu lớn, cần:
  - Backup (RMAN, Data Pump)
  - Hoặc dùng Flashback (Flashback Table/Database) nếu bật.

**Ví dụ rollback đơn giản:**
```sql
-- Migration: thêm cột STATUS
ALTER TABLE ORDER ADD (STATUS VARCHAR2(20) DEFAULT 'NEW');

-- Rollback: xóa cột STATUS
ALTER TABLE ORDER DROP COLUMN STATUS;
```

## 5. Ví dụ CI đơn giản với SQL*Plus (Jenkins)

**Jenkins pipeline (Groovy):**
```groovy
pipeline {
    agent any
    stages {
        stage('Test DB') {
            steps {
                sh '''
                sqlplus -s app_user/secret@testdb <<EOF
                WHENEVER SQLERROR EXIT 1
                @db/migrations/V1__init_schema.sql
                @db/migrations/V2__add_customer_table.sql
                @tests/run_all_tests.sql
                EXIT
                EOF
                '''
            }
        }
        stage('Deploy to PROD') {
            when { branch 'main' }
            steps {
                input message: 'Deploy to PROD?'
                sh '''
                sqlplus -s app_user/secret@proddb <<EOF
                WHENEVER SQLERROR EXIT 1
                @db/migrations/V1__init_schema.sql
                @db/migrations/V2__add_customer_table.sql
                EXIT
                EOF
                '''
            }
        }
    }
}
```
