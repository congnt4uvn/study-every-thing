
Dưới đây là tóm tắt ngắn gọn về **Deadlock trong Oracle**: nguyên nhân, cách phòng tránh, kèm ví dụ (giả lập) bằng SQL.  

---

## 1. Deadlock là gì?

- Deadlock xảy ra khi **hai (hoặc nhiều) session** cùng chờ nhau giải phóng tài nguyên (lock), dẫn đến **vòng lặp chờ vô hạn**.
- Oracle sẽ tự động phát hiện deadlock và **rollback một trong các transaction** để phá vỡ deadlock.

---

## 2. Nguyên nhân chính gây deadlock

1. **Truy cập tài nguyên theo thứ tự khác nhau**  
   - Session 1: lock bảng A rồi lock bảng B  
   - Session 2: lock bảng B rồi lock bảng A  
   → Hai session chờ nhau, tạo deadlock.

2. **Cập nhật cùng một tập dữ liệu theo thứ tự khác nhau**  
   - Cùng update nhiều dòng nhưng theo thứ tự khác nhau (ví dụ: theo ID khác nhau).

3. **Kết hợp giữa DML và foreign key không có index**  
   - Bảng con có foreign key nhưng **không có index** → khi update/delete bảng cha, Oracle phải lock nhiều dòng bảng con → dễ gây deadlock.

4. **Ứng dụng giữ lock quá lâu**  
   - Transaction mở lâu, không commit/rollback → tăng khả năng va chạm lock.

---

## 3. Cách phòng tránh deadlock

1. **Truy cập tài nguyên theo cùng một thứ tự**  
   - Quy ước trong ứng dụng: luôn update bảng theo thứ tự cố định (ví dụ: A → B → C).

2. **Giảm thời gian giữ lock**
   - Commit/rollback sớm, tránh transaction kéo dài.
   - Tránh thao tác giao diện người dùng trong khi transaction đang mở.

3. **Tạo index cho foreign key**
   - Đảm bảo các cột foreign key có index để giảm lock không cần thiết.

4. **Chia nhỏ transaction**
   - Thay vì update/delete rất nhiều dòng trong một transaction lớn, chia nhỏ thành nhiều batch.

5. **Sử dụng mức isolation phù hợp**
   - Oracle mặc định là **READ COMMITTED**, thường đủ cho đa số ứng dụng.
   - Tránh tự ý tăng lên mức cao hơn (như SERIALIZABLE) nếu không cần thiết, vì có thể tăng xung đột.

---

## 4. Ví dụ minh họa deadlock (giả lập)

### Chuẩn bị dữ liệu

```sql
-- Tạo bảng
CREATE TABLE accounts (
    id      NUMBER PRIMARY KEY,
    balance NUMBER
);

INSERT INTO accounts VALUES (1, 1000);
INSERT INTO accounts VALUES (2, 2000);
COMMIT;
```

### Session 1

```sql
-- Session 1: bắt đầu transaction
UPDATE accounts
SET balance = balance + 100
WHERE id = 1;   -- lock dòng id = 1

-- Sau đó cố gắng lock dòng id = 2
UPDATE accounts
SET balance = balance - 100
WHERE id = 2;
```

### Session 2

```sql
-- Session 2: bắt đầu transaction
UPDATE accounts
SET balance = balance + 200
WHERE id = 2;   -- lock dòng id = 2

-- Sau đó cố gắng lock dòng id = 1
UPDATE accounts
SET balance = balance - 200
WHERE id = 1;
```

**Diễn biến:**

- Session 1 lock `id = 1`, Session 2 lock `id = 2`.
- Session 1 chờ lock `id = 2` (đang bị Session 2 giữ).
- Session 2 chờ lock `id = 1` (đang bị Session 1 giữ).
- Oracle phát hiện deadlock → rollback một session, ví dụ Session 2.

**Kết quả (output) ở Session 2:**

```text
ORA-00060: deadlock detected while waiting for resource
```

---

## 5. Ví dụ về phòng tránh deadlock bằng thứ tự truy cập

Quy ước: **luôn update theo thứ tự id tăng dần**.

### Session 1

```sql
UPDATE accounts
SET balance = balance + 100
WHERE id = 1;

UPDATE accounts
SET balance = balance - 100
WHERE id = 2;

COMMIT;
```

### Session 2

```sql
UPDATE accounts
SET balance = balance + 200
WHERE id = 1;

UPDATE accounts
SET balance = balance - 200
WHERE id = 2;

COMMIT;
```

- Cả hai session đều lock theo thứ tự: `id = 1` → `id = 2`.  
- Không tạo vòng chờ → **không deadlock**.  

---

## 6. Ví dụ về index trên foreign key

```sql
-- Bảng cha
CREATE TABLE parent (
    id NUMBER PRIMARY KEY
);

-- Bảng con
CREATE TABLE child (
    id        NUMBER PRIMARY KEY,
    parent_id NUMBER,
    CONSTRAINT fk_child_parent
        FOREIGN KEY (parent_id)
        REFERENCES parent(id)
);

-- Tạo index trên foreign key để giảm lock
CREATE INDEX idx_child_parent_id ON child(parent_id);
```

- Khi `DELETE` hoặc `UPDATE parent.id`, Oracle dùng index `idx_child_parent_id` để tìm nhanh các dòng liên quan trong `child`, giảm lock diện rộng, giảm nguy cơ deadlock.

---

Tóm lại:  
- Deadlock là vòng chờ lock lẫn nhau giữa các session.  
- Nguyên nhân chính: truy cập tài nguyên không theo cùng thứ tự, thiếu index trên foreign key, transaction dài.  
- Phòng tránh bằng: quy ước thứ tự truy cập, commit sớm, tạo index foreign key, chia nhỏ transaction, dùng isolation level hợp lý.
