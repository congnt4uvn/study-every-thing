
Dưới đây là tóm tắt ngắn gọn bằng tiếng Việt, tập trung vào ý chính và có ví dụ minh họa.

---

## Ý chính của nội dung

Tình huống điển hình:  
- Bạn có một **báo cáo (report)** chạy trên **Oracle Database** mất khoảng **30 phút** để hoàn thành.  
- Câu hỏi: **Làm sao để cải thiện (tối ưu) thời gian chạy của báo cáo này?**

Mục tiêu:  
- Giảm thời gian chạy từ 30 phút xuống mức chấp nhận được (ví dụ vài phút hoặc giây).  
- Tập trung vào tối ưu **câu lệnh SQL**, **chỉ mục (index)**, **kế hoạch thực thi (execution plan)**, và **cấu trúc dữ liệu**.

---

## Các bước tối ưu hóa (tóm tắt)

1. **Xác định câu SQL gây chậm**
   - Tìm câu SELECT/INSERT/UPDATE trong report chạy lâu nhất.
   - Dùng `SQL_ID` trong `V$SQL` hoặc log của ứng dụng/report.

2. **Xem Execution Plan**
   - Dùng `EXPLAIN PLAN` hoặc `DBMS_XPLAN.DISPLAY_CURSOR` để xem Oracle đang thực thi câu SQL như thế nào.
   - Tìm các dấu hiệu:
     - Full table scan không cần thiết
     - Nested loop trên tập dữ liệu rất lớn
     - Sort, Hash Join tốn nhiều tài nguyên

3. **Tối ưu câu SQL**
   - Viết lại câu SQL (rewrite) để:
     - Giảm số lượng join không cần thiết
     - Giảm subquery lồng nhau
     - Dùng `WITH` (subquery factoring) nếu cần
   - Thêm điều kiện lọc (`WHERE`) rõ ràng, tránh lọc ở tầng ứng dụng.

4. **Tối ưu chỉ mục (Index)**
   - Tạo index trên các cột thường dùng trong:
     - `WHERE`
     - `JOIN`
     - `ORDER BY`, `GROUP BY` (khi phù hợp)
   - Xóa hoặc điều chỉnh index không hiệu quả.

5. **Kiểm tra thống kê (Statistics)**
   - Đảm bảo bảng và index có thống kê mới:
     ```sql
     EXEC DBMS_STATS.GATHER_TABLE_STATS(
       ownname => 'SCOTT',
       tabname => 'EMP',
       cascade => TRUE
     );
     ```

6. **Kiểm tra cấu hình & tài nguyên**
   - PGA, TEMP, I/O, CPU có đủ không.
   - Có thể cần partition table nếu dữ liệu rất lớn.

## Ví dụ minh họa

### 1. Câu SQL ban đầu (chạy 30 phút)

Giả sử report lấy dữ liệu bán hàng 1 năm, join nhiều bảng:

```sql
SELECT c.customer_name,
       p.product_name,
       SUM(s.amount) total_amount,
       COUNT(*) total_orders
FROM   sales s,
       customers c,
       products p
WHERE  s.customer_id = c.customer_id
AND    s.product_id  = p.product_id
AND    s.sale_date  >= TO_DATE('2024-01-01', 'YYYY-MM-DD')
AND    s.sale_date  <  TO_DATE('2025-01-01', 'YYYY-MM-DD')
GROUP BY c.customer_name, p.product_name;
```

Vấn đề thường gặp:
- Không có index trên `sales.sale_date`, `sales.customer_id`, `sales.product_id`.
- Bảng `sales` rất lớn (hàng chục triệu dòng).

### 2. Thêm index để cải thiện

```sql
-- Index trên cột lọc theo ngày
CREATE INDEX idx_sales_date ON sales(sale_date);

-- Index trên cột join
CREATE INDEX idx_sales_cust ON sales(customer_id);
CREATE INDEX idx_sales_prod ON sales(product_id);
```

**Kỳ vọng đầu ra (Output):**
- Trước khi tạo index: câu SQL chạy ~30 phút.
- Sau khi tạo index: có thể giảm xuống còn vài phút hoặc vài chục giây (tùy dữ liệu và hệ thống).

### 3. Viết lại câu SQL (sử dụng ANSI JOIN + WITH)

```sql
WITH filtered_sales AS (
  SELECT /*+ INDEX(s idx_sales_date) */
         s.customer_id,
         s.product_id,
         s.amount
  FROM   sales s
  WHERE  s.sale_date >= DATE '2024-01-01'
  AND    s.sale_date <  DATE '2025-01-01'
)
SELECT c.customer_name,
       p.product_name,
       SUM(fs.amount) total_amount,
       COUNT(*) total_orders
FROM   filtered_sales fs
JOIN   customers c ON fs.customer_id = c.customer_id
JOIN   products  p ON fs.product_id  = p.product_id
GROUP BY c.customer_name, p.product_name;
```

**Ý tưởng:**
- Lọc dữ liệu `sales` trước (chỉ lấy 1 năm) rồi mới join.
- Gợi ý index (`/*+ INDEX */`) để Oracle dùng index phù hợp.

## Ví dụ xem Execution Plan

```sql
EXPLAIN PLAN FOR
SELECT c.customer_name,
       p.product_name,
       SUM(s.amount) total_amount
FROM   sales s
JOIN   customers c ON s.customer_id = c.customer_id
JOIN   products  p ON s.product_id  = p.product_id
WHERE  s.sale_date >= DATE '2024-01-01'
AND    s.sale_date <  DATE '2025-01-01'
GROUP BY c.customer_name, p.product_name;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Đầu ra (rút gọn):**

```text
--------------------------------------------------------------------------------
| Id  | Operation                     | Name          | Rows  | Cost | Time    |
--------------------------------------------------------------------------------
|   0 | SELECT STATEMENT              |               |       |      |         |
|   1 |  HASH GROUP BY                |               | 100K  | ...  |         |
|   2 |   HASH JOIN                   |               | 100K  | ...  |         |
|   3 |    HASH JOIN                  |               | 100K  | ...  |         |
|   4 |     TABLE ACCESS BY INDEX ROWID| CUSTOMERS    | 10K   | ...  |         |
|   5 |      INDEX FULL SCAN          | PK_CUSTOMERS  | 10K   | ...  |         |
|   6 |     TABLE ACCESS BY INDEX ROWID| PRODUCTS     | 5K    | ...  |         |
|   7 |      INDEX FULL SCAN          | PK_PRODUCTS   | 5K    | ...  |         |
|   8 |    TABLE ACCESS BY INDEX ROWID| SALES         | 1M    | ...  |         |
|   9 |     INDEX RANGE SCAN          | IDX_SALES_DATE| 1M    | ...  |         |
--------------------------------------------------------------------------------
```

Nếu thấy `TABLE ACCESS FULL` trên `SALES` mà không cần thiết, cần xem lại index và điều kiện lọc.

## Tóm tắt ngắn

- Báo cáo chạy 30 phút thường do:
  - Câu SQL chưa tối ưu
  - Thiếu index phù hợp
  - Thống kê cũ
  - Dữ liệu rất lớn nhưng không partition
- Cách cải thiện:
  1. Xác định câu SQL chậm.
  2. Xem execution plan.
  3. Tối ưu câu SQL (lọc sớm, giảm join, dùng WITH).
  4. Tạo/chỉnh sửa index trên cột lọc và join.
  5. Cập nhật statistics.
  6. Kiểm tra tài nguyên hệ thống nếu vẫn chậm.

Nếu bạn cung cấp thêm đoạn SQL cụ thể của report, tôi có thể giúp tối ưu chi tiết hơn (viết lại câu lệnh, đề xuất index cụ thể, ước lượng tác động).
