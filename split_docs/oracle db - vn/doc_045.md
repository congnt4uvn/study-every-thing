
Dường như nội dung bạn định gửi (“Indexing Strategy: Types of indexes:”) chưa đầy đủ, nên mình chưa có đủ thông tin cụ thể để tóm tắt chính xác theo tài liệu của bạn.

Tuy nhiên, mình có thể tóm tắt ngắn gọn **các loại index phổ biến trong Oracle Database** và minh họa bằng ví dụ code (kèm input/output) để bạn tham khảo. Nếu bạn gửi thêm phần nội dung chi tiết, mình sẽ tóm tắt lại đúng theo tài liệu đó.

---

## 1. Các loại index chính trong Oracle

### 1. B-tree Index (chuẩn, mặc định)

- Dùng cho hầu hết các cột được lọc/joins thường xuyên.
- Tối ưu cho truy vấn tìm kiếm bằng `=`, `>`, `<`, `BETWEEN`, `LIKE 'abc%'`.
- Không phù hợp lắm cho cột có rất ít giá trị khác nhau (low cardinality).

**Tạo B-tree index:**

```sql
CREATE INDEX idx_emp_lastname
ON employees (last_name);
```

**Ví dụ truy vấn (input):**

```sql
SELECT employee_id, last_name
FROM employees
WHERE last_name = 'Nguyen';
```

**Kết quả (output – ví dụ):**

```text
EMPLOYEE_ID | LAST_NAME
----------- | ----------
       101  | Nguyen
       205  | Nguyen
```

Oracle sẽ dùng `idx_emp_lastname` để tìm nhanh các dòng có `last_name = 'Nguyen'`.

---

### 2. Bitmap Index

- Dùng cho cột **ít giá trị khác nhau** (low cardinality), ví dụ: giới tính, trạng thái, cờ Y/N.
- Rất hiệu quả cho các truy vấn phân tích, data warehouse, khi kết hợp nhiều điều kiện.
- Không phù hợp cho OLTP có nhiều thao tác DML (INSERT/UPDATE/DELETE) vì chi phí cập nhật cao.

**Tạo bitmap index:**

```sql
CREATE BITMAP INDEX idx_emp_gender
ON employees (gender);
```

**Ví dụ truy vấn (input):**

```sql
SELECT COUNT(*)
FROM employees
WHERE gender = 'F'
  AND status = 'ACTIVE';
```

Nếu `gender` và `status` đều có bitmap index, Oracle có thể kết hợp bitmap rất nhanh để lọc dữ liệu.

---

### 3. Unique Index

- Đảm bảo **không trùng lặp** giá trị trong cột (hoặc nhóm cột).
- Thường được tạo tự động khi tạo `PRIMARY KEY` hoặc `UNIQUE` constraint.

**Tạo unique index trực tiếp:**

```sql
CREATE UNIQUE INDEX idx_emp_email_uk
ON employees (email);
```

**Ví dụ (input):**

```sql
INSERT INTO employees (employee_id, email)
VALUES (300, 'user@example.com');
```

Nếu đã có một dòng với `email = 'user@example.com'`, output sẽ là lỗi:

```text
ORA-00001: unique constraint (SCHEMA.IDX_EMP_EMAIL_UK) violated
```

---

### 4. Composite (Multi-column) Index

- Index trên **nhiều cột**.
- Hữu ích khi truy vấn thường xuyên lọc theo nhiều cột cùng lúc.
- Thứ tự cột trong index rất quan trọng.

**Tạo composite index:**

```sql
CREATE INDEX idx_emp_dept_job
ON employees (department_id, job_id);
```

**Ví dụ truy vấn (input):**

```sql
SELECT employee_id
FROM employees
WHERE department_id = 10
  AND job_id = 'SA_REP';
```

Truy vấn này có thể dùng `idx_emp_dept_job`.  
Truy vấn chỉ có `WHERE job_id = 'SA_REP'` thì **không** tận dụng tốt index này (vì `department_id` là cột đầu tiên).

---

### 5. Function-based Index

- Index trên **biểu thức** hoặc **hàm** thay vì cột thô.
- Hữu ích khi truy vấn thường xuyên dùng hàm trên cột (UPPER, TRUNC, NVL, …).

**Tạo function-based index:**

```sql
CREATE INDEX idx_emp_upper_lastname
ON employees (UPPER(last_name));
```

**Ví dụ truy vấn (input):**

```sql
SELECT employee_id, last_name
FROM employees
WHERE UPPER(last_name) = 'NGUYEN';
```

Oracle có thể dùng `idx_emp_upper_lastname` để tránh full table scan.

---

### 6. Partitioned Index

- Index được chia thành nhiều partition, phù hợp với bảng partitioned lớn.
- Giúp quản lý, bảo trì, và truy vấn trên dữ liệu lớn hiệu quả hơn.

**Ví dụ (local partitioned index):**

```sql
CREATE INDEX idx_sales_date
ON sales (sale_date)
LOCAL;
```

Mỗi partition của bảng `sales` sẽ có index partition tương ứng.

---

### 7. Reverse Key Index

- Đảo ngược bit của giá trị index (thường dùng cho cột tăng dần như sequence).
- Giảm hiện tượng “hot block” khi nhiều session chèn dữ liệu vào cuối index.

**Tạo reverse key index:**

```sql
CREATE INDEX idx_emp_id_rev
ON employees (employee_id)
REVERSE;
```

Thường dùng trong hệ thống OLTP có rất nhiều insert.

---

### 8. Index trên cột ảo (Virtual Column Index)

- Cột ảo được tính từ biểu thức, không lưu dữ liệu thật.
- Có thể tạo index trên cột ảo để tối ưu truy vấn.

**Ví dụ:**

```sql
ALTER TABLE employees
ADD full_name AS (first_name || ' ' || last_name);

CREATE INDEX idx_emp_fullname
ON employees (full_name);
```

---

## Tóm tắt ngắn gọn (tiếng Việt)

- **B-tree index**: loại chuẩn, dùng cho hầu hết truy vấn lọc/so sánh.
- **Bitmap index**: cho cột ít giá trị, phù hợp data warehouse, phân tích.
- **Unique index**: đảm bảo giá trị không trùng, thường gắn với PK/Unique.
- **Composite index**: index nhiều cột, thứ tự cột rất quan trọng.
- **Function-based index**: index trên biểu thức/hàm, tối ưu truy vấn có dùng hàm.
- **Partitioned index**: index chia partition, phù hợp bảng lớn partitioned.
- **Reverse key index**: giảm hot block cho cột tăng dần.
- **Index trên cột ảo**: dùng cột tính toán, không lưu dữ liệu, vẫn index được.
