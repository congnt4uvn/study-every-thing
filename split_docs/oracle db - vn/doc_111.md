## Ý chính

Khi làm việc với Oracle Database (và thiết kế hệ thống nói chung), người ta thường đánh giá khả năng **tư duy đánh đổi (trade-off)** giữa:

1. **Hiệu năng (Performance)**  
2. **Tính linh hoạt (Flexibility)**  
3. **Độ phức tạp (Complexity)**  

Không có giải pháp “hoàn hảo” cho mọi mặt. Bạn cần biết:
- Khi nào ưu tiên hiệu năng, chấp nhận giảm linh hoạt hoặc tăng phức tạp.
- Khi nào ưu tiên đơn giản, chấp nhận hiệu năng không tối ưu.
- Khi nào cần thiết kế linh hoạt, chấp nhận code phức tạp hơn.

---

## Ví dụ 1: SQL đơn giản vs linh hoạt

### 1) Câu SQL đơn giản, dễ hiểu (ít linh hoạt, dễ bảo trì)

```sql
-- Input: Lấy danh sách khách hàng đang hoạt động
SELECT customer_id, name, status
FROM customers
WHERE status = 'ACTIVE';
```

- Ưu điểm: Đơn giản, dễ tối ưu, dễ đọc.
- Nhược điểm: Nếu sau này cần lọc theo nhiều trạng thái khác nhau, phải sửa code.

**Output (ví dụ):**

| CUSTOMER_ID | NAME        | STATUS  |
|-------------|------------|---------|
| 101         | Nguyen A    | ACTIVE  |
| 102         | Tran B      | ACTIVE  |

---

### 2) Câu SQL linh hoạt hơn (dùng tham số, phức tạp hơn)

```sql
-- Input: :p_status_list là danh sách trạng thái, ví dụ: 'ACTIVE,INACTIVE'
SELECT customer_id, name, status
FROM customers
WHERE status IN (
    SELECT REGEXP_SUBSTR(:p_status_list, '[^,]+', 1, LEVEL)
    FROM dual
    CONNECT BY REGEXP_SUBSTR(:p_status_list, '[^,]+', 1, LEVEL) IS NOT NULL
);
```

- Ưu điểm: Linh hoạt, có thể truyền nhiều trạng thái khác nhau.
- Nhược điểm: Câu lệnh phức tạp hơn, khó tối ưu hơn, khó đọc hơn.

**Ví dụ Input:**

```text
:p_status_list = 'ACTIVE,INACTIVE'
```

**Output (ví dụ):**

| CUSTOMER_ID | NAME        | STATUS    |
|-------------|------------|-----------|
| 101         | Nguyen A    | ACTIVE    |
| 103         | Le C        | INACTIVE  |

**Trade-off:**
- Chọn câu SQL đơn giản nếu yêu cầu ít thay đổi, ưu tiên hiệu năng và dễ bảo trì.
- Chọn câu SQL linh hoạt nếu yêu cầu thay đổi thường xuyên, chấp nhận phức tạp hơn.

## Ví dụ 2: PL/SQL – hiệu năng vs linh hoạt

### 1) Thủ tục đơn nhiệm vụ (đơn giản, nhanh)

```plsql
CREATE OR REPLACE PROCEDURE update_salary (
    p_emp_id   IN employees.employee_id%TYPE,
    p_percent  IN NUMBER
) AS
BEGIN
    UPDATE employees
    SET salary = salary * (1 + p_percent / 100)
    WHERE employee_id = p_emp_id;
    
    COMMIT;
END;
/
```

- Ưu điểm: Rõ ràng, dễ tối ưu, ít lỗi.
- Nhược điểm: Chỉ làm được một việc cụ thể.

---

### 2) Thủ tục đa mục đích (linh hoạt, phức tạp hơn)

```plsql
CREATE OR REPLACE PROCEDURE update_employee (
    p_emp_id      IN employees.employee_id%TYPE,
    p_new_salary  IN employees.salary%TYPE DEFAULT NULL,
    p_new_title   IN employees.job_title%TYPE DEFAULT NULL
) AS
BEGIN
    UPDATE employees
    SET salary    = COALESCE(p_new_salary, salary),
        job_title = COALESCE(p_new_title, job_title)
    WHERE employee_id = p_emp_id;
    
    COMMIT;
END;
/
```

- Ưu điểm: Linh hoạt, có thể cập nhật nhiều loại thông tin.
- Nhược điểm: Logic phức tạp hơn, khó test, khó tối ưu.

---

## Kết luận

Khi thiết kế và viết code cho Oracle Database, điều “họ” tìm kiếm là khả năng bạn:

- Nhận diện được **trade-off** giữa hiệu năng, linh hoạt và độ phức tạp.
- Giải thích được **vì sao** chọn giải pháp A thay vì B.
- Biết **đơn giản hóa** khi có thể, và **chấp nhận phức tạp** khi thật sự cần thiết.
