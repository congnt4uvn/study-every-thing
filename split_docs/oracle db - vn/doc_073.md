
Dưới đây là tóm tắt ngắn gọn về khái niệm Transaction và tính chất ACID trong Oracle Database (và hệ quản trị CSDL quan hệ nói chung).

---

## 1. Transaction là gì?

- Transaction (giao dịch) là một đơn vị công việc logic gồm một hoặc nhiều câu lệnh SQL.
- Mục tiêu: hoặc **thực hiện toàn bộ** (commit) hoặc **hủy toàn bộ** (rollback) để đảm bảo dữ liệu nhất quán.

Ví dụ: Chuyển tiền từ tài khoản A sang B:
1. Trừ tiền từ A  
2. Cộng tiền vào B  
→ Hai bước này phải thành công cùng nhau, hoặc cùng bị hủy.

---

## 2. ACID là gì?

ACID là 4 tính chất quan trọng của transaction:

1. **Atomicity (Tính nguyên tử)**  
   - Giao dịch là “tất cả hoặc không gì cả”.  
   - Nếu một phần thất bại → toàn bộ giao dịch bị rollback.

2. **Consistency (Tính nhất quán)**  
   - Sau khi transaction kết thúc (commit hoặc rollback), dữ liệu phải ở trạng thái nhất quán, tuân thủ mọi ràng buộc (constraints, triggers, rules…).

3. **Isolation (Tính cô lập)**  
   - Các transaction đang chạy song song không được làm sai lệch kết quả của nhau.  
   - Mỗi transaction “nhìn thấy” dữ liệu như thể nó đang chạy một mình (tùy mức độ isolation).

4. **Durability (Tính bền vững)**  
   - Sau khi commit, dữ liệu được ghi bền vững (thường là xuống đĩa).  
   - Hệ thống có thể bị crash nhưng dữ liệu đã commit vẫn được khôi phục.

## 3. Ví dụ code trong Oracle (SQL*Plus / PL/SQL)

### 3.1. Atomicity + Consistency

```sql
-- Giả sử có bảng:
-- ACCOUNTS(id NUMBER PRIMARY KEY, balance NUMBER)

-- Bắt đầu transaction (ngầm định khi bạn chạy lệnh đầu tiên)
UPDATE accounts
SET balance = balance - 1000
WHERE id = 1;  -- Tài khoản A

UPDATE accounts
SET balance = balance + 1000
WHERE id = 2;  -- Tài khoản B

-- Nếu mọi thứ OK:
COMMIT;
-- Nếu có lỗi (ví dụ update thứ 2 thất bại):
ROLLBACK;
```

**Ý nghĩa:**
- Nếu một trong hai câu lệnh UPDATE lỗi → `ROLLBACK` sẽ đưa dữ liệu về trạng thái trước khi bắt đầu transaction.
- Đảm bảo **Atomicity** và **Consistency**.

---

### 3.2. Isolation (ví dụ đơn giản)

Session 1:

```sql
-- Session 1
UPDATE accounts
SET balance = balance - 500
WHERE id = 1;

-- Chưa COMMIT, giữ nguyên transaction mở
```

Session 2:

```sql
-- Session 2
SELECT balance
FROM accounts
WHERE id = 1;
```

**Kết quả (tùy isolation level):**
- Thông thường (READ COMMITTED – mặc định của Oracle):  
  Session 2 **không thấy** số dư đã trừ 500 cho đến khi Session 1 `COMMIT`.  
- Điều này thể hiện **Isolation**: transaction chưa commit không “lộ” dữ liệu ra ngoài.

---

### 3.3. Durability

```sql
UPDATE accounts
SET balance = balance + 200
WHERE id = 3;

COMMIT;
```

- Sau `COMMIT`, Oracle ghi log và dữ liệu xuống storage.  
- Nếu hệ thống bị crash sau đó, khi khởi động lại, dữ liệu đã commit vẫn tồn tại → **Durability**.

---

## 4. Ví dụ input/output đơn giản

Giả sử bảng:

```sql
CREATE TABLE accounts (
  id      NUMBER PRIMARY KEY,
  balance NUMBER
);

INSERT INTO accounts VALUES (1, 1000);
INSERT INTO accounts VALUES (2, 500);
COMMIT;
```

### Transaction chuyển 300 từ 1 sang 2

**Input (các lệnh SQL):**

```sql
UPDATE accounts SET balance = balance - 300 WHERE id = 1;
UPDATE accounts SET balance = balance + 300 WHERE id = 2;
COMMIT;
```

**Output (trạng thái dữ liệu sau commit):**

```sql
SELECT * FROM accounts;

-- Kết quả:
-- ID | BALANCE
--  1 | 700
--  2 | 800
```

Nếu thay `COMMIT;` bằng `ROLLBACK;`:

**Output:**

```sql
SELECT * FROM accounts;

-- Kết quả:
-- ID | BALANCE
--  1 | 1000
--  2 | 500
```

→ Thể hiện rõ **Atomicity**: hoặc cả hai update cùng có hiệu lực, hoặc không update nào có hiệu lực.
