
Dưới đây là tóm tắt ngắn gọn về các khái niệm Transaction trong Oracle: **Read Consistency** và **Undo Segments**, kèm ví dụ.

---

## 1. Read Consistency (Nhất quán khi đọc)

**Ý chính:**

- Oracle đảm bảo mỗi câu SELECT “nhìn thấy” dữ liệu ở một **thời điểm nhất quán** (consistent point in time), dù dữ liệu đang bị các transaction khác cập nhật.
- Khi bạn chạy một câu SELECT dài, nếu trong lúc đó có transaction khác UPDATE/DELETE dữ liệu, Oracle sẽ dùng **undo** để cho bạn thấy **phiên bản cũ** của dữ liệu (trước khi commit), giúp kết quả SELECT nhất quán.

**Cách hoạt động:**

- Khi một transaction thay đổi dữ liệu, Oracle ghi bản sao cũ của dữ liệu vào **undo segment**.
- Nếu một câu SELECT cần đọc dữ liệu tại thời điểm trước đó, Oracle sẽ dùng dữ liệu trong undo để tái tạo lại “ảnh chụp” (snapshot) của dữ liệu.

---

## 2. Undo Segments (Phân đoạn Undo)

**Ý chính:**

- Undo segments lưu trữ **bản ghi trước khi thay đổi** (before image) của dữ liệu.
- Dùng cho:
  1. **ROLLBACK**: hoàn tác các thay đổi chưa commit.
  2. **Read Consistency**: cung cấp dữ liệu cũ cho các câu SELECT.
  3. **Khôi phục (Recovery)**: hỗ trợ phục hồi sau sự cố.

**Đặc điểm:**

- Undo được lưu trong **UNDO tablespace**.
- Oracle tự động quản lý undo (AUTO UNDO MANAGEMENT) trong hầu hết các hệ thống hiện đại.

## 3. Ví dụ minh họa Read Consistency & Undo

### 3.1. Chuẩn bị bảng

```sql
-- Tạo bảng demo
CREATE TABLE employees_demo (
    emp_id   NUMBER PRIMARY KEY,
    name     VARCHAR2(50),
    salary   NUMBER
);

-- Chèn dữ liệu
INSERT INTO employees_demo VALUES (1, 'A', 1000);
INSERT INTO employees_demo VALUES (2, 'B', 2000);
COMMIT;
```

### 3.2. Transaction 1: Đọc dữ liệu (Session 1)

```sql
-- Session 1
-- Bắt đầu một câu SELECT dài (giả sử chạy lâu)
SELECT emp_id, name, salary
FROM employees_demo
WHERE emp_id = 1;
```

**Kết quả (Session 1):**

```text
EMP_ID | NAME | SALARY
----------------------
1      | A    | 1000
```

### 3.3. Transaction 2: Cập nhật dữ liệu (Session 2)

```sql
-- Session 2
UPDATE employees_demo
SET salary = 3000
WHERE emp_id = 1;

COMMIT;
```

**Sau COMMIT, dữ liệu thực tế trong bảng:**

```text
EMP_ID | NAME | SALARY
----------------------
1      | A    | 3000
2      | B    | 2000
```

### 3.4. Read Consistency hoạt động thế nào?

- **Session 1** đã bắt đầu SELECT trước khi Session 2 COMMIT.
- Oracle sẽ dùng **undo** để cung cấp **phiên bản cũ** của dòng `emp_id = 1` (salary = 1000) cho Session 1.
- Vì vậy, Session 1 vẫn thấy:

```text
EMP_ID | NAME | SALARY
----------------------
1      | A    | 1000
```

Dù dữ liệu hiện tại trong bảng đã là 3000.

## 4. Ví dụ ROLLBACK dùng Undo

```sql
-- Session 3
UPDATE employees_demo
SET salary = 5000
WHERE emp_id = 2;

-- Chưa commit, kiểm tra trong Session 3
SELECT emp_id, name, salary
FROM employees_demo
WHERE emp_id = 2;
```

**Kết quả (Session 3, chưa COMMIT):**

```text
EMP_ID | NAME | SALARY
----------------------
2      | B    | 5000
```

Trong khi đó, **Session 4** (session khác):

```sql
SELECT emp_id, name, salary
FROM employees_demo
WHERE emp_id = 2;
```

**Kết quả (Session 4):**

```text
EMP_ID | NAME | SALARY
----------------------
2      | B    | 2000
```

- Session 4 thấy giá trị **cũ** (2000) nhờ **read consistency**.
- Nếu Session 3 chạy:

```sql
ROLLBACK;
```

Oracle dùng **undo** để khôi phục salary của emp_id = 2 về 2000.

---

## 5. Tóm tắt ngắn

- **Read Consistency**:  
  - Mỗi câu SELECT thấy dữ liệu ở một thời điểm nhất quán.  
  - Không bị “nhiễu” bởi các transaction đang cập nhật/commit song song.

- **Undo Segments**:  
  - Lưu bản ghi trước khi thay đổi.  
  - Dùng cho ROLLBACK, read consistency, và recovery.  
  - Được lưu trong UNDO tablespace và thường được Oracle tự động quản lý.

Nếu bạn muốn, tôi có thể thêm ví dụ với `SET TRANSACTION READ ONLY` hoặc minh họa lỗi `ORA-01555: snapshot too old` liên quan đến undo.
