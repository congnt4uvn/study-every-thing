
Dưới đây là tóm tắt ngắn gọn (tiếng Việt) dựa trên nội dung bạn cung cấp:

---

## Ý chính

- Họ muốn kiểm tra **mức độ hiểu biết của bạn về tác động của các quyết định thiết kế** (schema, index, query, kiến trúc) lên:
  - **Hiệu năng** (performance): tốc độ truy vấn, tài nguyên sử dụng, khả năng mở rộng.
  - **Khả năng bảo trì** (maintainability): dễ đọc, dễ sửa, dễ mở rộng, ít lỗi.

- Không chỉ “làm cho chạy được”, mà còn phải:
  - Biết **tại sao** chọn thiết kế đó.
  - Biết **trade-off**: ví dụ, thêm index giúp truy vấn nhanh hơn nhưng làm chậm INSERT/UPDATE và tốn thêm dung lượng.

---

## Ví dụ minh họa (Oracle)

### 1. Thiết kế bảng & index

**Thiết kế 1: Không có index phù hợp**

```sql
-- Bảng đơn giản lưu đơn hàng
CREATE TABLE orders (
    order_id      NUMBER PRIMARY KEY,
    customer_id   NUMBER,
    order_date    DATE,
    total_amount  NUMBER
);

-- Truy vấn thường xuyên: tìm đơn hàng theo customer_id
SELECT *
FROM orders
WHERE customer_id = 101;
```

- **Vấn đề hiệu năng**: Nếu bảng `orders` rất lớn, truy vấn trên `customer_id` sẽ phải full table scan.
- **Bảo trì**: Code vẫn chạy được, nhưng khó mở rộng khi dữ liệu tăng.

**Thiết kế 2: Thêm index phù hợp**

```sql
CREATE INDEX idx_orders_customer
    ON orders (customer_id);
```

- **Hiệu năng**: Truy vấn theo `customer_id` nhanh hơn nhiều.
- **Trade-off**:  
  - INSERT/UPDATE/DELETE trên `orders` sẽ chậm hơn một chút vì phải cập nhật index.  
  - Tốn thêm dung lượng lưu trữ cho index.

**Input/Output minh họa (logic)**

- Input: `customer_id = 101`
- Trước khi có index:
  - Oracle phải đọc hầu hết các block của bảng `orders`.
- Sau khi có index:
  - Oracle dùng `idx_orders_customer` để tìm nhanh các `rowid` tương ứng, rồi đọc đúng các dòng cần.

### 2. Thiết kế câu lệnh SQL (dễ bảo trì vs khó bảo trì)

**Câu lệnh khó bảo trì**

```sql
SELECT *
FROM orders o, customers c
WHERE o.customer_id = c.customer_id
  AND o.order_date >= TO_DATE('2024-01-01', 'YYYY-MM-DD')
  AND o.order_date <  TO_DATE('2024-02-01', 'YYYY-MM-DD')
  AND c.status = 'ACTIVE';
```

- Dùng cú pháp join cũ (comma join), dễ gây nhầm lẫn khi thêm điều kiện.
- `SELECT *` khó kiểm soát cột, dễ gây lỗi khi schema thay đổi.

**Câu lệnh dễ bảo trì, rõ ràng hơn**

```sql
SELECT 
    o.order_id,
    o.order_date,
    o.total_amount,
    c.customer_name
FROM orders   o
JOIN customers c
    ON o.customer_id = c.customer_id
WHERE o.order_date >= DATE '2024-01-01'
  AND o.order_date <  DATE '2024-02-01'
  AND c.status = 'ACTIVE';
```

- **Hiệu năng**: Tương đương hoặc tốt hơn (tùy plan), nhưng:
  - Chỉ chọn cột cần thiết → ít I/O hơn.
- **Bảo trì**:
  - Cú pháp `JOIN` rõ ràng, dễ thêm/bớt điều kiện.
  - Ít rủi ro hơn khi schema thay đổi.

### 3. Thiết kế schema: chuẩn hóa vs phi chuẩn hóa

**Chuẩn hóa (normalized)**

```sql
CREATE TABLE customers (
    customer_id   NUMBER PRIMARY KEY,
    customer_name VARCHAR2(100),
    status        VARCHAR2(20)
);

CREATE TABLE orders (
    order_id      NUMBER PRIMARY KEY,
    customer_id   NUMBER REFERENCES customers(customer_id),
    order_date    DATE,
    total_amount  NUMBER
);
```

- **Ưu điểm**:
  - Dễ bảo trì: thông tin khách hàng chỉ lưu một nơi.
  - Ít trùng lặp dữ liệu, giảm nguy cơ không nhất quán.
- **Nhược điểm**:
  - Một số truy vấn phải JOIN → có thể chậm hơn nếu không tối ưu.

**Phi chuẩn hóa (denormalized)**

```sql
CREATE TABLE orders_denorm (
    order_id        NUMBER PRIMARY KEY,
    customer_id     NUMBER,
    customer_name   VARCHAR2(100),
    customer_status VARCHAR2(20),
    order_date      DATE,
    total_amount    NUMBER
);
```

- **Ưu điểm**:
  - Một số truy vấn đọc nhanh hơn (không cần JOIN).
- **Nhược điểm**:
  - Khó bảo trì: thay đổi tên khách hàng phải cập nhật nhiều dòng.
  - Dễ sai lệch dữ liệu nếu cập nhật không đồng bộ.

## Kết luận

- Điều họ tìm kiếm: **khả năng giải thích**:
  - “Nếu tôi thiết kế bảng/index/câu lệnh SQL theo cách A, hiệu năng sẽ thế nào?”
  - “Nếu tôi chọn cách B, việc bảo trì sau này có dễ không?”
- Bạn cần thể hiện:
  - Biết đánh đổi (trade-off) giữa **performance** và **maintainability**.
  - Biết đưa ra ví dụ cụ thể (như index, join, chuẩn hóa/phi chuẩn hóa) và giải thích tác động của chúng.
