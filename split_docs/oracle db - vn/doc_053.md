
Dưới đây là tóm tắt ngắn gọn về chủ đề “SQL Tuning Techniques: Bind variables vs literals (and bind peeking)” bằng tiếng Việt, kèm ví dụ mã và input/output minh họa.

---

## 1. Bind variables là gì? Literals là gì?

- **Literals**: Giá trị được ghi trực tiếp trong câu SQL.  
  ```sql
  SELECT * FROM orders WHERE customer_id = 101;
  ```
  101 là literal.

- **Bind variables**: Giá trị được truyền vào thông qua biến ràng buộc, không ghi trực tiếp trong câu SQL.  
  ```sql
  SELECT * FROM orders WHERE customer_id = :cust_id;
  ```
  `:cust_id` là bind variable.

---

## 2. Ưu/nhược điểm của bind variables

### Ưu điểm

1. **Giảm hard parse, tăng hiệu năng**
   - Câu SQL với bind variables có cùng cấu trúc sẽ được **tái sử dụng** plan.
   - Giảm CPU, giảm latch contention, tăng throughput.

   Ví dụ:  
   ```sql
   -- Lần 1
   SELECT * FROM orders WHERE customer_id = :cust_id;  -- cust_id = 101
   -- Lần 2
   SELECT * FROM orders WHERE customer_id = :cust_id;  -- cust_id = 202
   ```
   Cả hai lần dùng chung 1 execution plan.

2. **Bảo mật (tránh SQL Injection)**
   - Dùng bind variables giúp tách dữ liệu khỏi câu lệnh SQL, giảm nguy cơ injection.

3. **Giảm kích thước shared pool**
   - Không tạo ra hàng loạt câu SQL khác nhau chỉ vì khác literal.

### Nhược điểm

1. **Mất khả năng tối ưu theo giá trị cụ thể**
   - Optimizer không thấy được giá trị cụ thể (101 hay 999999), nên khó chọn plan tối ưu cho từng giá trị.
   - Dễ dẫn đến **one-size-fits-all plan** (một plan dùng cho mọi giá trị, có thể tốt cho giá trị này nhưng tệ cho giá trị khác).

---

## 3. Literals: khi nào hữu ích?

- Khi dữ liệu **phân bố không đều** (skewed data), ví dụ:
  - `status = 'ACTIVE'` chiếm 99% dữ liệu
  - `status = 'INACTIVE'` chiếm 1% dữ liệu
- Optimizer có thể chọn:
  - Plan A (full scan) cho giá trị phổ biến
  - Plan B (index range scan) cho giá trị hiếm

Ví dụ:

```sql
-- Truy vấn với literal
SELECT * FROM users WHERE status = 'ACTIVE';
SELECT * FROM users WHERE status = 'INACTIVE';
```

- Hai câu trên có thể có **2 execution plan khác nhau** nếu dùng literals.

---

## 4. Bind peeking là gì?

**Bind peeking**: Khi parse lần đầu câu SQL có bind variable, Oracle sẽ “nhìn” (peek) giá trị bind tại thời điểm đó để chọn execution plan.

Ví dụ:

```sql
SELECT * FROM users WHERE status = :status;
```

- Lần đầu chạy:
  ```sql
  :status = 'INACTIVE'  -- rất ít dòng
  ```
  → Optimizer chọn **index range scan** (plan tốt cho giá trị hiếm).

- Lần sau:
  ```sql
  :status = 'ACTIVE'  -- rất nhiều dòng
  ```
  → Oracle **tái sử dụng plan cũ** (index range scan), nhưng lần này lại **không tối ưu** cho giá trị phổ biến.

Kết quả:  
- Một plan được tối ưu cho giá trị hiếm, nhưng bị dùng cho giá trị phổ biến → hiệu năng kém.

---

## 5. Adaptive Cursor Sharing (ACS)

Để khắc phục vấn đề bind peeking, Oracle có **Adaptive Cursor Sharing**:

- Theo dõi hiệu năng của plan với các giá trị bind khác nhau.
- Nếu thấy plan hiện tại không phù hợp cho một nhóm giá trị, Oracle có thể:
  - Tạo **nhiều cursor** (nhiều execution plan) cho cùng một câu SQL, mỗi plan áp dụng cho một “phạm vi” giá trị bind khác nhau.

Ví dụ (mô phỏng logic):

```sql
SELECT * FROM users WHERE status = :status;
```

- Plan 1: dùng cho `status IN ('INACTIVE', 'DELETED')` (ít dòng, dùng index).
- Plan 2: dùng cho `status = 'ACTIVE'` (nhiều dòng, dùng full scan).

---

## 6. Ví dụ cụ thể: bind vs literal

### 6.1. Dùng literals

```sql
-- Câu 1
SELECT * FROM orders WHERE order_date = DATE '2024-01-01';

-- Câu 2
SELECT * FROM orders WHERE order_date = DATE '2024-05-01';
```

- Input:
  - Câu 1: `order_date = 2024-01-01` (giả sử có 10 dòng)
  - Câu 2: `order_date = 2024-05-01` (giả sử có 1,000,000 dòng)
- Output:
  - Câu 1: Optimizer có thể chọn index range scan.
  - Câu 2: Optimizer có thể chọn full table scan.

### 6.2. Dùng bind variable

```sql
SELECT * FROM orders WHERE order_date = :dt;
```

- Lần 1:
  - Input: `:dt = DATE '2024-01-01'` (10 dòng)
  - Plan: index range scan (tối ưu cho ít dòng).
- Lần 2:
  - Input: `:dt = DATE '2024-05-01'` (1,000,000 dòng)
  - Oracle vẫn dùng plan index range scan (do reuse plan) → có thể chậm.

---

## 7. Khi nào nên dùng bind variables, khi nào nên dùng literals?

**Nên dùng bind variables khi:**
- Ứng dụng OLTP, nhiều câu SQL giống nhau chỉ khác giá trị.
- Cần giảm hard parse, tăng throughput.
- Cần bảo mật, tránh SQL injection.

**Có thể dùng literals (hoặc hint/SQL Profile/ACS) khi:**
- Truy vấn phức tạp, dữ liệu phân bố rất lệch.
- Cần tối ưu mạnh cho từng giá trị cụ thể.
- Báo cáo, batch job, ít lặp lại, không lo nhiều hard parse.

## 8. Ví dụ trong PL/SQL: bind vs literal

### 8.1. PL/SQL với bind variable

```plsql
DECLARE
  v_cust_id NUMBER := 101;
  v_name    VARCHAR2(100);
BEGIN
  SELECT customer_name
  INTO   v_name
  FROM   customers
  WHERE  customer_id = v_cust_id;  -- Oracle tự bind
END;
/
```

- Input: `v_cust_id = 101`
- Output: `v_name` nhận tên khách hàng có `customer_id = 101`.
- Câu SQL được parse 1 lần, dùng lại nhiều lần nếu block được gọi nhiều lần.

### 8.2. Dynamic SQL với bind variable

```plsql
DECLARE
  v_sql   VARCHAR2(200);
  v_id    NUMBER := 101;
  v_name  VARCHAR2(100);
BEGIN
  v_sql := 'SELECT customer_name FROM customers WHERE customer_id = :id';

  EXECUTE IMMEDIATE v_sql
    INTO v_name
    USING v_id;

  DBMS_OUTPUT.PUT_LINE('Customer name: ' || v_name);
END;
/
```

- Input: `v_id = 101`
- Output: In ra tên khách hàng tương ứng.
- `:id` là bind variable → tốt cho hiệu năng và bảo mật.

## 9. Tóm tắt ý chính

1. **Bind variables**:
   - Giảm hard parse, tăng hiệu năng, bảo mật tốt hơn.
   - Nhưng có thể làm mất khả năng tối ưu theo từng giá trị cụ thể.

2. **Literals**:
   - Cho phép optimizer chọn plan tối ưu cho từng giá trị.
   - Nhưng gây ra nhiều hard parse, tốn tài nguyên, shared pool bị “bẩn”.

3. **Bind peeking**:
   - Oracle nhìn giá trị bind lần đầu để chọn plan.
   - Plan đó có thể không phù hợp cho các giá trị bind khác → hiệu năng không ổn định.

4. **Adaptive Cursor Sharing**:
   - Cho phép nhiều plan cho cùng một câu SQL với bind variables.
   - Giúp cân bằng giữa hiệu năng (reuse plan) và tối ưu theo giá trị.

5. **Thực tế**:
   - OLTP: ưu tiên bind variables.
   - Reporting/Batch, dữ liệu skew: cân nhắc literals, hint, hoặc cấu hình ACS/SQL Plan Management.

Nếu bạn cho thêm đoạn SQL cụ thể, tôi có thể phân tích execution plan và minh họa rõ hơn cách bind peeking/ACS ảnh hưởng đến hiệu năng.
