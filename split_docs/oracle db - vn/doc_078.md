
Dưới đây là tóm tắt ngắn gọn về wait event `enq: TX - row lock contention` trong Oracle, kèm ví dụ.

---

## 1. Khái niệm chính

- **`enq: TX - row lock contention`** là wait event xảy ra khi một session muốn cập nhật/xóa một dòng (row) nhưng dòng đó đang bị khóa bởi session khác.
- Đây là **transaction enqueue (TX)**, liên quan đến **row-level lock**.
- Thường xuất hiện khi:
  - Nhiều session cập nhật cùng một dòng.
  - Có transaction lâu commit/rollback.
  - Ứng dụng không xử lý tốt việc cập nhật đồng thời (concurrency).

---

## 2. Cơ chế cơ bản

- Khi session A cập nhật một row:
  - Oracle đặt **row lock** (TX lock) trên row đó.
  - Session A giữ lock cho đến khi **COMMIT** hoặc **ROLLBACK**.
- Khi session B cố gắng cập nhật cùng row:
  - Session B phải **chờ** → phát sinh wait event `enq: TX - row lock contention`.

---

## 3. Ví dụ minh họa

### 3.1. Chuẩn bị bảng

```sql
CREATE TABLE employees_demo (
    emp_id   NUMBER PRIMARY KEY,
    salary   NUMBER
);

INSERT INTO employees_demo (emp_id, salary) VALUES (1, 1000);
COMMIT;
```

---

### 3.2. Session 1: Cập nhật và không commit

```sql
-- SESSION 1
UPDATE employees_demo
   SET salary = 2000
 WHERE emp_id = 1;

-- Không COMMIT, giữ nguyên trạng thái này
```

**Trạng thái:**  
- Row có `emp_id = 1` đang bị khóa bởi Session 1.

---

### 3.3. Session 2: Cố gắng cập nhật cùng row

```sql
-- SESSION 2
UPDATE employees_demo
   SET salary = 3000
 WHERE emp_id = 1;
```

**Kết quả (OUTPUT):**

- Lệnh UPDATE ở Session 2 **không hoàn thành ngay**, nó **bị treo (hang)** và chờ.
- Trong view `V$SESSION`, bạn sẽ thấy Session 2 đang wait trên event:

```sql
SELECT sid, event, wait_class, blocking_session
FROM   v$session
WHERE  event LIKE 'enq: TX%';
```

**Ví dụ output:**

| SID | EVENT                         | WAIT_CLASS | BLOCKING_SESSION |
|-----|-------------------------------|------------|------------------|
| 123 | enq: TX - row lock contention | Application| 122              |

- `BLOCKING_SESSION = 122` là SID của Session 1.

---

### 3.4. Khi Session 1 COMMIT

```sql
-- SESSION 1
COMMIT;
```

**Kết quả:**

- Lock được giải phóng.
- Session 2 tiếp tục thực thi, UPDATE thành công.

---

## 4. Cách xử lý / giảm thiểu

- **Commit sớm**: Không giữ transaction mở quá lâu.
- **Giảm xung đột trên cùng row**:
  - Tránh nhiều session cập nhật cùng một bản ghi “hot row”.
  - Thiết kế lại dữ liệu để phân tán cập nhật (ví dụ: sharding theo key).
- **Kiểm tra session gây block**:
  - Dùng `V$SESSION`, `V$LOCK`, `V$SESSION_BLOCKERS`, `V$SESSION_WAIT`.
- **Xử lý logic ứng dụng**:
  - Dùng retry logic khi gặp ORA-00054/ORA-00060 (nếu có).
  - Hạn chế thao tác “SELECT FOR UPDATE” kéo dài.

---

Nếu bạn muốn, tôi có thể thêm ví dụ truy vấn chi tiết để tìm session đang block và script để kill session gây khóa.
