## 1. Các lỗi thường gặp khi thiết kế Index trong Oracle

### 1.1 Tạo quá nhiều index
- Mỗi index làm chậm `INSERT`, `UPDATE`, `DELETE` vì Oracle phải cập nhật thêm index.
- Nhiều index không dùng đến → tốn không gian, tốn tài nguyên khi tối ưu kế hoạch thực thi.

**Ví dụ:**

```sql
-- Bảng đơn giản
CREATE TABLE employees (
    emp_id      NUMBER PRIMARY KEY,
    first_name  VARCHAR2(50),
    last_name   VARCHAR2(50),
    dept_id     NUMBER,
    salary      NUMBER
);

-- Tạo quá nhiều index
CREATE INDEX emp_fname_idx ON employees(first_name);
CREATE INDEX emp_lname_idx ON employees(last_name);
CREATE INDEX emp_dept_idx  ON employees(dept_id);
CREATE INDEX emp_sal_idx   ON employees(salary);
```

**Hệ quả (output / tác động):**
- Truy vấn SELECT có thể nhanh hơn chút, nhưng:
- Mỗi lần `INSERT` 1 dòng, Oracle phải ghi vào 4 index + 1 PK index → chậm.

---

### 1.2 Index không hỗ trợ truy vấn thực tế
- Tạo index trên cột ít khi được dùng trong `WHERE`, `JOIN`, `ORDER BY`.
- Không phân tích workload thực tế (AWR, ASH, SQL Monitor).

**Ví dụ:**

```sql
-- Truy vấn thường dùng:
SELECT * 
FROM employees
WHERE dept_id = :dept
ORDER BY last_name;
```

**Index phù hợp:**

```sql
CREATE INDEX emp_dept_lname_idx ON employees(dept_id, last_name);
```

**Pitfall:**  
Chỉ tạo index trên `last_name` hoặc `first_name` mà không có `dept_id` → Oracle vẫn phải đọc nhiều block.

---

### 1.3 Không hiểu thứ tự cột trong composite index
- Oracle chỉ dùng phần “leading” của index hiệu quả.
- Composite index `(col1, col2, col3)` hữu ích nhất khi điều kiện có `col1`, hoặc `col1 + col2`, …

**Ví dụ:**

```sql
CREATE INDEX emp_dept_sal_idx ON employees(dept_id, salary);
```

- Truy vấn:

```sql
SELECT * FROM employees
WHERE dept_id = 10 AND salary > 5000;
```

→ Có thể dùng index tốt.

- Nhưng:

```sql
SELECT * FROM employees
WHERE salary > 5000;
```

→ Thường **không** dùng được index này hiệu quả (vì thiếu cột dẫn `dept_id`).

---

### 1.4 Dùng hàm trên cột nhưng không có function-based index
- Điều kiện `WHERE UPPER(last_name) = 'SMITH'` sẽ không dùng index thường trên `last_name`.

**Ví dụ:**

```sql
-- Index thường
CREATE INDEX emp_lname_idx ON employees(last_name);

-- Truy vấn
SELECT * 
FROM employees
WHERE UPPER(last_name) = 'SMITH';
```

→ Oracle khó dùng `emp_lname_idx`.

**Giải pháp: function-based index**

```sql
CREATE INDEX emp_lname_upper_idx ON employees(UPPER(last_name));

-- Truy vấn
SELECT * 
FROM employees
WHERE UPPER(last_name) = 'SMITH';
```

→ Oracle có thể dùng `emp_lname_upper_idx` để tìm nhanh.

---

### 1.5 Không phân biệt B-tree index và Bitmap index
- **B-tree**: phù hợp OLTP, cột có độ phân biệt cao (nhiều giá trị khác nhau).
- **Bitmap**: phù hợp OLAP, cột ít giá trị (giới tính, trạng thái), ít cập nhật.

**Pitfall:**  
Dùng bitmap index trên bảng cập nhật thường xuyên → gây khóa (locking) và contention.

**Ví dụ:**

```sql
-- Cột ít giá trị, dùng cho báo cáo
CREATE BITMAP INDEX emp_status_bm_idx ON employees(status);
```

- Tốt cho hệ thống báo cáo (ít DML).
- Không tốt cho hệ thống giao dịch (nhiều INSERT/UPDATE/DELETE).

---

### 1.6 Không kiểm tra index không dùng (unused index)
- Nhiều index tồn tại nhưng không bao giờ được dùng.
- Có thể kiểm tra qua `V$OBJECT_USAGE` hoặc AWR.

**Ví dụ kiểm tra:**

```sql
ALTER INDEX emp_dept_idx MONITORING USAGE;

-- Sau một thời gian chạy hệ thống, kiểm tra:
SELECT index_name, used
FROM   v$object_usage
WHERE  index_name = 'EMP_DEPT_IDX';
```

**Output mẫu:**

| INDEX_NAME    | USED |
|---------------|------|
| EMP_DEPT_IDX  | NO   |

→ Có thể cân nhắc drop index này.

---

### 1.7 Không cập nhật thống kê (statistics)
- Optimizer cần thống kê chính xác để quyết định có dùng index hay không.
- Thiếu thống kê → kế hoạch thực thi kém.

**Ví dụ:**

```sql
EXEC DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES',
    cascade => TRUE
);
```

---

## 2. Tóm tắt ngắn gọn (tiếng Việt)

- Đừng tạo quá nhiều index; mỗi index làm chậm DML.
- Tạo index dựa trên truy vấn thực tế (WHERE, JOIN, ORDER BY).
- Chú ý thứ tự cột trong composite index; cột dẫn (leading column) rất quan trọng.
- Dùng function-based index khi có hàm trong điều kiện WHERE.
- Chọn đúng loại index: B-tree cho OLTP, Bitmap cho OLAP, cột ít giá trị.
- Thường xuyên kiểm tra và loại bỏ index không dùng.
- Luôn cập nhật thống kê để optimizer chọn kế hoạch tốt.
