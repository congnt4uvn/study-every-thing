
Dưới đây là tóm tắt ngắn gọn về chủ đề “Relational Design: Surrogate vs Natural Keys” trong bối cảnh Oracle Database, kèm ví dụ code và input/output minh họa.

---

## 1. Khái niệm chính

### Natural Key (Khóa tự nhiên)
- Là khóa lấy trực tiếp từ dữ liệu thực tế trong đời sống.
- Có ý nghĩa nghiệp vụ (business meaning).
- Ví dụ: số CMND/CCCD, mã sản phẩm, số VIN của xe, mã ISBN sách.
- Đặc điểm:
  - Thường là một hoặc nhiều cột có sẵn trong bảng.
  - Có thể thay đổi nếu quy định nghiệp vụ thay đổi (ví dụ đổi format mã khách hàng).
  - Dễ hiểu với người dùng, nhưng có thể dài, phức tạp, hoặc không ổn định.

### Surrogate Key (Khóa thay thế)
- Là khóa nhân tạo, không có ý nghĩa nghiệp vụ.
- Thường là số tăng tự động (sequence, identity).
- Ví dụ: cột `CUSTOMER_ID` kiểu NUMBER, tự tăng.
- Đặc điểm:
  - Ổn định, không đổi theo nghiệp vụ.
  - Đơn giản, ngắn, tối ưu cho join và index.
  - Cần thêm ràng buộc UNIQUE để bảo vệ tính duy nhất của dữ liệu nghiệp vụ (natural key).

---

## 2. So sánh nhanh

| Tiêu chí              | Natural Key                         | Surrogate Key                          |
|-----------------------|-------------------------------------|----------------------------------------|
| Ý nghĩa nghiệp vụ     | Có                                  | Không                                  |
| Độ ổn định            | Có thể thay đổi                     | Rất ổn định                            |
| Độ dài/đơn giản       | Có thể dài, phức tạp                | Thường là số, ngắn, đơn giản           |
| Hiệu năng join/index  | Có thể kém hơn (chuỗi dài, nhiều cột)| Thường tốt hơn (số, 1 cột)             |
| Phụ thuộc nghiệp vụ   | Cao                                 | Thấp                                   |

Thông lệ phổ biến trong thiết kế quan hệ hiện đại:
- Dùng **surrogate key** làm **primary key**.
- Dùng **natural key** với **UNIQUE constraint** để đảm bảo tính duy nhất nghiệp vụ.

## 3. Ví dụ trong Oracle

### 3.1. Dùng Natural Key làm Primary Key

Giả sử `CUSTOMER` dùng số CCCD làm khóa chính:

```sql
CREATE TABLE customer_natural (
    cccd          VARCHAR2(12)   CONSTRAINT pk_customer_nat PRIMARY KEY,
    full_name     VARCHAR2(100)  NOT NULL,
    date_of_birth DATE,
    email         VARCHAR2(100)
);
```

- Input (INSERT):

```sql
INSERT INTO customer_natural (cccd, full_name, date_of_birth, email)
VALUES ('012345678901', 'Nguyen Van A', DATE '1990-01-01', 'a@example.com');
```

- Output (SELECT):

```sql
SELECT * FROM customer_natural;

-- Kết quả (giả sử 1 dòng):
-- CCCD        FULL_NAME      DATE_OF_BIRTH  EMAIL
-- ----------- -------------  ------------   ----------------
-- 012345678901 Nguyen Van A  1990-01-01     a@example.com
```

Nhược điểm: nếu sau này nhà nước đổi quy tắc mã CCCD, hoặc cần sửa mã, việc thay đổi PK sẽ phức tạp (cascade sang các bảng con).

---

### 3.2. Dùng Surrogate Key + Natural Key UNIQUE

```sql
CREATE TABLE customer_surrogate (
    customer_id   NUMBER         CONSTRAINT pk_customer_sur PRIMARY KEY,
    cccd          VARCHAR2(12)   CONSTRAINT uq_customer_cccd UNIQUE,
    full_name     VARCHAR2(100)  NOT NULL,
    date_of_birth DATE,
    email         VARCHAR2(100)
);
```

Tạo sequence và trigger (Oracle trước 12c) hoặc identity (12c+):

#### Cách 1: Sequence + Trigger

```sql
CREATE SEQUENCE seq_customer_id START WITH 1 INCREMENT BY 1;

CREATE OR REPLACE TRIGGER trg_customer_bi
BEFORE INSERT ON customer_surrogate
FOR EACH ROW
BEGIN
    IF :NEW.customer_id IS NULL THEN
        :NEW.customer_id := seq_customer_id.NEXTVAL;
    END IF;
END;
/
```

- Input (INSERT – không cần truyền `customer_id`):

```sql
INSERT INTO customer_surrogate (cccd, full_name, date_of_birth, email)
VALUES ('012345678901', 'Nguyen Van A', DATE '1990-01-01', 'a@example.com');
```

- Output (SELECT):

```sql
SELECT customer_id, cccd, full_name FROM customer_surrogate;

-- Ví dụ kết quả:
-- CUSTOMER_ID  CCCD         FULL_NAME
-- -----------  -----------  -------------
-- 1            012345678901 Nguyen Van A
```

#### Cách 2: Identity Column (Oracle 12c+)

```sql
CREATE TABLE customer_surrogate_id (
    customer_id   NUMBER GENERATED BY DEFAULT AS IDENTITY
                  CONSTRAINT pk_customer_sur_id PRIMARY KEY,
    cccd          VARCHAR2(12) CONSTRAINT uq_customer_cccd_id UNIQUE,
    full_name     VARCHAR2(100) NOT NULL,
    date_of_birth DATE,
    email         VARCHAR2(100)
);
```

---

### 3.3. Bảng con tham chiếu Surrogate Key

Ví dụ bảng `ORDERS` tham chiếu `CUSTOMER`:

```sql
CREATE TABLE orders (
    order_id     NUMBER GENERATED BY DEFAULT AS IDENTITY
                 CONSTRAINT pk_orders PRIMARY KEY,
    customer_id  NUMBER NOT NULL,
    order_date   DATE   NOT NULL,
    amount       NUMBER(12,2) NOT NULL,
    CONSTRAINT fk_orders_customer
        FOREIGN KEY (customer_id)
        REFERENCES customer_surrogate (customer_id)
);
```

- Input:

```sql
INSERT INTO orders (customer_id, order_date, amount)
VALUES (1, SYSDATE, 1000000);
```

- Output (join):

```sql
SELECT o.order_id, c.full_name, o.amount
FROM orders o
JOIN customer_surrogate c
  ON o.customer_id = c.customer_id;

-- Ví dụ kết quả:
-- ORDER_ID  FULL_NAME      AMOUNT
-- --------  -------------  -----------
-- 1         Nguyen Van A   1000000
```

## 4. Khi nào dùng loại nào?

- Ưu tiên **Surrogate Key** làm PK khi:
  - Hệ thống lớn, nhiều bảng, nhiều quan hệ.
  - Natural key có thể thay đổi, dài, hoặc gồm nhiều cột.
  - Cần tối ưu join, index, partition.

- Có thể dùng **Natural Key** làm PK khi:
  - Natural key ngắn, ổn định, thực sự không đổi (ví dụ mã chuẩn quốc tế rất ổn định).
  - Hệ thống nhỏ, ít quan hệ, yêu cầu đơn giản.

Thực tế, mô hình phổ biến và an toàn:
- **PK = Surrogate Key (NUMBER/IDENTITY)**
- **Natural Key = UNIQUE + NOT NULL** để bảo vệ dữ liệu nghiệp vụ.
