
Dưới đây là tóm tắt ngắn gọn về **Index-Organized Table (IOT) trong Oracle** bằng tiếng Việt, kèm ví dụ code và input/output minh họa.

---

## 1. Khái niệm Index-Organized Table (IOT)

- **IOT** là bảng mà **dữ liệu được lưu trữ trực tiếp trong cấu trúc chỉ mục B-tree** dựa trên **khóa chính (PRIMARY KEY)**.
- Khác với bảng dạng heap (bình thường):
  - Bảng heap: dữ liệu lưu “rải rác”, chỉ mục chỉ trỏ đến vị trí hàng.
  - IOT: chỉ mục **chính là nơi chứa dữ liệu**, không cần thêm bước truy cập bảng.

**Lợi ích chính:**
- Truy vấn theo khóa chính rất nhanh (ít I/O).
- Dữ liệu được lưu **theo thứ tự khóa chính** → tốt cho truy vấn range (BETWEEN, >, <).
- Giảm không gian lưu trữ vì không cần thêm index riêng cho PK (PK index = data).

**Hạn chế:**
- Phù hợp cho workload đọc nhiều, truy vấn theo PK hoặc range.
- Chèn/xóa/cập nhật có thể tốn chi phí hơn do phải duy trì cấu trúc B-tree.
- Không phù hợp nếu truy vấn chủ yếu theo cột không phải khóa chính.

---

## 2. Cấu trúc cơ bản của IOT

- **PRIMARY KEY** là bắt buộc.
- Dữ liệu được lưu trong **index segment** thay vì table segment.
- Có thể dùng **overflow segment** để chứa các cột không thuộc khóa hoặc cột lớn.

---

## 3. Tạo IOT – Ví dụ

### 3.1. Tạo IOT đơn giản

```sql
CREATE TABLE customers_iot (
    customer_id   NUMBER,
    name          VARCHAR2(100),
    email         VARCHAR2(200),
    created_at    DATE,
    CONSTRAINT pk_customers_iot PRIMARY KEY (customer_id)
)
ORGANIZATION INDEX;
```

- `ORGANIZATION INDEX` → khai báo đây là IOT.
- `customer_id` là khóa chính, dữ liệu được lưu theo thứ tự `customer_id`.

**Input (INSERT):**

```sql
INSERT INTO customers_iot (customer_id, name, email, created_at)
VALUES (101, 'Nguyen Van A', 'a@example.com', SYSDATE);

INSERT INTO customers_iot (customer_id, name, email, created_at)
VALUES (50, 'Tran Thi B', 'b@example.com', SYSDATE);

INSERT INTO customers_iot (customer_id, name, email, created_at)
VALUES (200, 'Le Van C', 'c@example.com', SYSDATE);
```

**Cách lưu trữ nội bộ (khái niệm):**

- Dù bạn chèn theo thứ tự 101, 50, 200, IOT sẽ sắp xếp theo khóa:
  - 50 → 101 → 200

**Output (SELECT):**

```sql
SELECT customer_id, name
FROM customers_iot
ORDER BY customer_id;
```

Kết quả:

| CUSTOMER_ID | NAME         |
|------------:|-------------|
| 50          | Tran Thi B  |
| 101         | Nguyen Van A|
| 200         | Le Van C    |

---

### 3.2. IOT với OVERFLOW (các cột lớn)

Khi có nhiều cột hoặc cột lớn, ta có thể tách phần không thuộc khóa ra **overflow segment** để tránh làm index quá to.

```sql
CREATE TABLE orders_iot (
    order_id      NUMBER,
    customer_id   NUMBER,
    order_date    DATE,
    status        VARCHAR2(20),
    notes         VARCHAR2(4000),
    CONSTRAINT pk_orders_iot PRIMARY KEY (order_id)
)
ORGANIZATION INDEX
INCLUDING order_date
OVERFLOW;
```

- `INCLUDING order_date`: các cột từ khóa chính đến `order_date` sẽ nằm trong index chính.
- Các cột sau `order_date` (`status`, `notes`) sẽ nằm trong **overflow segment**.

**Input (INSERT):**

```sql
INSERT INTO orders_iot (order_id, customer_id, order_date, status, notes)
VALUES (1, 101, DATE '2024-01-01', 'NEW', 'Ghi chú đơn hàng 1');

INSERT INTO orders_iot (order_id, customer_id, order_date, status, notes)
VALUES (2, 101, DATE '2024-01-02', 'PAID', 'Ghi chú đơn hàng 2');
```

**Output (SELECT theo PK):**

```sql
SELECT order_id, customer_id, order_date, status
FROM orders_iot
WHERE order_id = 1;
```

Kết quả:

| ORDER_ID | CUSTOMER_ID | ORDER_DATE  | STATUS |
|---------:|------------:|------------|--------|
| 1        | 101         | 2024-01-01 | NEW    |

---

## 4. Khi nào nên dùng IOT?

Nên dùng IOT khi:

- Bảng có **truy vấn chủ yếu theo khóa chính** hoặc theo range của khóa chính.
- Bảng có **nhiều đọc, ít ghi** (OLTP đọc nhiều).
- Cần **giảm I/O** và **tối ưu truy vấn PK** (tránh “table lookup”).

Không nên dùng IOT khi:

- Truy vấn chủ yếu theo cột không phải PK.
- Bảng có nhiều cập nhật/chèn/xóa ngẫu nhiên, không theo khóa.

---

## 5. So sánh nhanh: Heap vs IOT

| Tiêu chí                 | Heap Table                         | IOT                                      |
|--------------------------|-------------------------------------|------------------------------------------|
| Lưu trữ dữ liệu          | Table segment                      | Index segment                            |
| Truy vấn theo PK         | Index scan + table access          | Chỉ index scan (dữ liệu nằm trong index) |
| Thứ tự lưu trữ           | Không đảm bảo                      | Theo thứ tự khóa chính                   |
| Phù hợp                  | Truy vấn đa dạng                   | Truy vấn theo PK / range trên PK         |
