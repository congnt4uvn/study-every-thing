
Dưới đây là tóm tắt ngắn gọn về chiến lược dùng **bitmap index** và **B-tree index** trong Oracle, kèm ví dụ.

---

## 1. B-tree Index

### Khi nên dùng B-tree

- Cột có **độ phân biệt cao** (high cardinality): nhiều giá trị khác nhau  
  (VD: `EMP_ID`, `ORDER_ID`, `EMAIL`).
- Các truy vấn thường:
  - Tìm kiếm theo **=, >, <, BETWEEN, LIKE 'ABC%'**.
  - Truy cập **OLTP** (giao dịch online, nhiều INSERT/UPDATE/DELETE).
- Bảng có **nhiều cập nhật**: B-tree xử lý DML tốt hơn bitmap.

### Đặc điểm chính

- Tốt cho truy vấn **điểm** (point lookup) và **range scan**.
- Ít bị khóa (locking) khi DML.
- Thường dùng cho **primary key**, **unique key**.

### Ví dụ B-tree index

```sql
-- Tạo bảng
CREATE TABLE employees (
    emp_id      NUMBER PRIMARY KEY,
    first_name  VARCHAR2(50),
    last_name   VARCHAR2(50),
    salary      NUMBER,
    dept_id     NUMBER
);

-- Tạo B-tree index (mặc định là B-tree)
CREATE INDEX idx_emp_lastname ON employees(last_name);

-- Truy vấn sử dụng B-tree index
EXPLAIN PLAN FOR
SELECT emp_id, first_name, last_name
FROM employees
WHERE last_name = 'Nguyen';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Kỳ vọng**: Plan sẽ dùng `INDEX RANGE SCAN` trên `IDX_EMP_LASTNAME`.

## 2. Bitmap Index

### Khi nên dùng Bitmap

- Cột có **độ phân biệt thấp** (low cardinality): ít giá trị khác nhau  
  (VD: `GENDER` (M/F), `STATUS` (A/I), `REGION` với vài vùng).
- Môi trường **Data Warehouse / OLAP**:
  - Bảng lớn, chủ yếu **SELECT**, ít DML.
  - Truy vấn phức tạp, nhiều điều kiện `AND/OR` trên nhiều cột.
- Phù hợp cho các cột dùng trong **WHERE** để lọc dữ liệu, đặc biệt khi kết hợp nhiều cột.

### Đặc điểm chính

- Rất hiệu quả khi:
  - Kết hợp nhiều điều kiện trên nhiều cột low-cardinality.
  - Thực hiện các phép toán logic (AND, OR) trên bitmap.
- Không phù hợp cho bảng **cập nhật thường xuyên**:
  - DML gây **khóa ở mức rowid range** → dễ gây contention.

### Ví dụ Bitmap index

```sql
-- Tạo bảng fact trong data warehouse
CREATE TABLE sales_fact (
    sale_id      NUMBER,
    customer_id  NUMBER,
    product_id   NUMBER,
    region       VARCHAR2(10),
    channel      VARCHAR2(10),  -- 'ONLINE', 'STORE', 'PHONE'
    status       CHAR(1),       -- 'A' (active), 'R' (returned)
    amount       NUMBER
);

-- Tạo bitmap index trên các cột low-cardinality
CREATE BITMAP INDEX idx_sales_region  ON sales_fact(region);
CREATE BITMAP INDEX idx_sales_channel ON sales_fact(channel);
CREATE BITMAP INDEX idx_sales_status  ON sales_fact(status);

-- Truy vấn phân tích
EXPLAIN PLAN FOR
SELECT region, channel, SUM(amount)
FROM sales_fact
WHERE region  IN ('NORTH', 'SOUTH')
  AND channel = 'ONLINE'
  AND status  = 'A'
GROUP BY region, channel;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Kỳ vọng**: Optimizer sẽ kết hợp các bitmap index (`BITMAP AND`, `BITMAP MERGE`) để lọc nhanh.

## 3. So sánh nhanh

| Tiêu chí                     | B-tree Index                          | Bitmap Index                               |
|-----------------------------|----------------------------------------|--------------------------------------------|
| Cardinality cột             | Cao (nhiều giá trị khác nhau)         | Thấp (ít giá trị)                          |
| Môi trường                  | OLTP (giao dịch)                      | OLAP / Data Warehouse                      |
| Tần suất DML                | Cao (nhiều INSERT/UPDATE/DELETE)      | Thấp (chủ yếu SELECT)                      |
| Khóa khi DML                | Ít, chi tiết theo row                 | Có thể khóa nhiều row (range)             |
| Truy vấn range, LIKE prefix| Rất tốt                               | Không phải mục tiêu chính                  |
| Kết hợp nhiều điều kiện AND/OR | Bình thường                        | Rất mạnh (bitmap operations)               |

---

## 4. Hướng dẫn chọn nhanh

- **Chọn B-tree** nếu:
  - Cột là **primary key**, **unique**, hoặc có nhiều giá trị khác nhau.
  - Bảng được cập nhật thường xuyên.
  - Truy vấn dạng tìm kiếm theo khóa hoặc range.

- **Chọn Bitmap** nếu:
  - Cột có **ít giá trị** (low cardinality).
  - Bảng chủ yếu dùng cho **báo cáo, phân tích**, ít DML.
  - Thường xuyên có truy vấn với nhiều điều kiện lọc kết hợp (`AND`, `OR`) trên các cột low-cardinality.

Nếu bạn gửi thêm ví dụ truy vấn thực tế, tôi có thể gợi ý cụ thể nên dùng loại index nào cho từng cột.
