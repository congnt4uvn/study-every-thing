
Dưới đây là tóm tắt ngắn gọn về chủ đề **SQL Tuning Techniques: Vai trò của DBMS_STATS** (Oracle Database), kèm ví dụ mã và input/output minh họa.

---

## 1. DBMS_STATS là gì?

- Là package chuẩn của Oracle dùng để:
  - Thu thập (gather) thống kê (statistics) cho **bảng, index, schema, toàn bộ database**.
  - Cập nhật, khóa/mở khóa, export/import thống kê.
- Thống kê này được **Cost-Based Optimizer (CBO)** sử dụng để chọn **execution plan tối ưu** cho câu SQL.

---

## 2. Vì sao DBMS_STATS quan trọng trong SQL Tuning?

- CBO dựa vào thống kê để ước lượng:
  - Số dòng (cardinality)
  - Phân bố dữ liệu (histogram)
  - Kích thước bảng, index
- Thống kê **cũ hoặc thiếu** → CBO chọn plan kém:
  - Dùng full table scan thay vì index
  - Join order không tối ưu
  - Join method không phù hợp (nested loop vs hash join)
- Do đó, **SQL tuning hiệu quả luôn gắn với việc quản lý thống kê đúng cách**.

---

## 3. Các thao tác chính với DBMS_STATS

### 3.1. Thu thập thống kê cho bảng

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',          -- schema
    tabname          => 'EMPLOYEES',   -- tên bảng
    estimate_percent => 10,            -- % mẫu (sample)
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO', -- histogram
    cascade          => TRUE           -- gather luôn index
  );
END;
/
```

**Ý nghĩa:**
- Thu thập thống kê cho bảng HR.EMPLOYEES và các index của nó.
- Cho phép Oracle tự quyết định histogram cần thiết.

---

### 3.2. Thu thập thống kê cho toàn schema

```sql
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(
    ownname          => 'HR',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => DBMS_STATS.AUTO_CASCADE
  );
END;
/
```

**Input:** tên schema (HR)  
**Output:** thống kê mới cho tất cả bảng/index trong schema HR.

---

### 3.3. Thu thập thống kê cho toàn database

```sql
BEGIN
  DBMS_STATS.GATHER_DATABASE_STATS(
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO'
  );
END;
/
```

Thường dùng trong môi trường test/dev hoặc khi mới triển khai.

---

### 3.4. Xem thống kê hiện tại

```sql
-- Thống kê của bảng
SELECT table_name, num_rows, blocks, last_analyzed
FROM   dba_tab_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';

-- Thống kê của cột
SELECT table_name, column_name, num_distinct, density, histogram
FROM   dba_tab_col_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

**Output ví dụ (rút gọn):**

```text
TABLE_NAME  NUM_ROWS  BLOCKS  LAST_ANALYZED
----------- --------- ------- -------------------
EMPLOYEES   107       5       2026-05-20 10:15:00

TABLE_NAME  COLUMN_NAME  NUM_DISTINCT  DENSITY   HISTOGRAM
----------- ------------ ------------  --------- -----------
EMPLOYEES   DEPARTMENT_ID  11         0.0909    HEIGHT BALANCED
```

---

### 3.5. Khóa/Mở khóa thống kê (LOCK/UNLOCK)

Dùng khi bạn **không muốn thống kê bị thay đổi** (ví dụ: đã tuning ổn định).

```sql
-- Khóa thống kê bảng
BEGIN
  DBMS_STATS.LOCK_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES'
  );
END;
/

-- Mở khóa thống kê bảng
BEGIN
  DBMS_STATS.UNLOCK_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES'
  );
END;
/
```

---

### 3.6. Lưu/Phục hồi thống kê (Export/Import)

Hữu ích khi:
- Muốn thử thống kê mới, nhưng vẫn có thể quay lại thống kê cũ.
- Di chuyển thống kê giữa môi trường (dev → test → prod).

```sql
-- Tạo bảng lưu thống kê
BEGIN
  DBMS_STATS.CREATE_STAT_TABLE(
    ownname => 'HR',
    stattab => 'HR_STATS_BACKUP'
  );
END;
/

-- Export thống kê của schema HR
BEGIN
  DBMS_STATS.EXPORT_SCHEMA_STATS(
    ownname => 'HR',
    stattab => 'HR_STATS_BACKUP'
  );
END;
/

-- Import lại thống kê
BEGIN
  DBMS_STATS.IMPORT_SCHEMA_STATS(
    ownname => 'HR',
    stattab => 'HR_STATS_BACKUP'
  );
END;
/
```

## 4. Liên hệ trực tiếp với SQL Tuning

1. **SQL chạy chậm sau khi gather stats**  
   - Có thể do plan thay đổi vì thống kê mới.  
   - Giải pháp:
     - So sánh execution plan trước/sau.
     - Nếu cần, **import lại thống kê cũ** hoặc **lock stats**.

2. **SQL dùng plan không hợp lý (full scan, join sai)**  
   - Kiểm tra thống kê:
     - `NUM_ROWS`, `NUM_DISTINCT`, histogram có hợp lý không?
   - Nếu thống kê sai lệch:
     - Gather lại với `AUTO_SAMPLE_SIZE` hoặc tăng `estimate_percent`.
     - Tạo histogram cho cột có phân bố skewed.

3. **Tự động hóa**  
   - Oracle có **auto stats job** (thường chạy ban đêm).
   - Có thể tùy chỉnh window, disable/enable job, hoặc tự viết job riêng dùng DBMS_STATS.

## 5. Ví dụ: Tác động của thống kê đến execution plan

### Bước 1: Chạy SQL trước khi gather stats

```sql
EXPLAIN PLAN FOR
SELECT * FROM HR.EMPLOYEES WHERE DEPARTMENT_ID = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Output (giả sử):**

```text
Plan hash value: 123456789

---------------------------------------------------
| Id | Operation          | Name       | Cost ... |
---------------------------------------------------
|  0 | SELECT STATEMENT   |            |   15     |
|  1 |  TABLE ACCESS FULL | EMPLOYEES  |   15     |
---------------------------------------------------
```

### Bước 2: Gather stats

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',
    tabname          => 'EMPLOYEES',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR COLUMNS DEPARTMENT_ID SIZE AUTO',
    cascade          => TRUE
  );
END;
/
```

### Bước 3: Chạy lại EXPLAIN PLAN

```sql
EXPLAIN PLAN FOR
SELECT * FROM HR.EMPLOYEES WHERE DEPARTMENT_ID = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Output (giả sử):**

```text
Plan hash value: 987654321

---------------------------------------------------------
| Id | Operation                   | Name        | Cost |
---------------------------------------------------------
|  0 | SELECT STATEMENT            |             |   2  |
|  1 |  TABLE ACCESS BY INDEX ROWID| EMPLOYEES   |   2  |
|  2 |   INDEX RANGE SCAN          | EMP_DEPT_IX |   1  |
---------------------------------------------------------
```

→ Sau khi có thống kê chính xác, CBO chọn **INDEX RANGE SCAN** thay vì **FULL TABLE SCAN**, chi phí (cost) giảm.

---

## 6. Ý chính cần nhớ

- **DBMS_STATS là công cụ chuẩn để quản lý thống kê** cho CBO.
- Thống kê **đúng và cập nhật** là nền tảng của **SQL tuning hiệu quả**.
- Các thao tác quan trọng:
  - `GATHER_*_STATS` (TABLE/SCHEMA/DATABASE)
  - `LOCK/UNLOCK_TABLE_STATS`
  - `EXPORT/IMPORT_*_STATS`
  - Kiểm tra thống kê qua `DBA_TAB_STATISTICS`, `DBA_TAB_COL_STATISTICS`.
- Khi SQL chạy chậm hoặc plan thay đổi bất thường, **luôn kiểm tra thống kê trước**.

Nếu bạn cho biết thêm nội dung chi tiết (ví dụ: đoạn tài liệu cụ thể, version Oracle, môi trường OLTP/OLAP), tôi có thể viết lại bản tóm tắt chi tiết hơn và thêm các script phù hợp để dùng trực tiếp trong hệ thống của bạn.
