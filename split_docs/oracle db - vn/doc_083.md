
Dưới đây là tóm tắt ngắn gọn về cách thiết kế code để **giảm thiểu vấn đề khóa (locking)** trong Oracle Database, kèm ví dụ.

---

## 1. Giữ giao dịch (transaction) càng ngắn càng tốt

- Chỉ bao gồm những câu lệnh **thực sự cần thiết** trong một transaction.
- Tránh:
  - Chờ input người dùng giữa chừng.
  - Gọi dịch vụ bên ngoài lâu (HTTP, file, API) trong cùng transaction.

**Ví dụ xấu (PL/SQL):**

```sql
BEGIN
  UPDATE accounts
  SET balance = balance - 100
  WHERE id = 1;

  -- Chờ người dùng xác nhận (giả sử)
  -- DB session bị giữ khóa trong lúc chờ
  -- ...

  UPDATE accounts
  SET balance = balance + 100
  WHERE id = 2;

  COMMIT;
END;
/
```

**Ví dụ tốt:**

- Tách logic UI / gọi dịch vụ ra khỏi transaction DB.
- Chỉ bắt đầu transaction khi đã có đủ dữ liệu.

```sql
BEGIN
  -- Đã có đủ thông tin, chỉ thực hiện nhanh gọn
  UPDATE accounts
  SET balance = balance - 100
  WHERE id = 1;

  UPDATE accounts
  SET balance = balance + 100
  WHERE id = 2;

  COMMIT;
END;
/
```

## 2. Cập nhật theo thứ tự nhất quán (consistent ordering)

- Khi nhiều session cập nhật nhiều bảng / nhiều dòng, hãy **luôn cập nhật theo cùng một thứ tự** để tránh deadlock.

**Ví dụ:**

Giả sử có 2 tài khoản: `id = 1` và `id = 2`.

- Quy ước: luôn cập nhật theo thứ tự `id nhỏ → id lớn`.

```sql
DECLARE
  v_from_id NUMBER := :from_id; -- input
  v_to_id   NUMBER := :to_id;   -- input
  v_min_id  NUMBER;
  v_max_id  NUMBER;
BEGIN
  IF v_from_id < v_to_id THEN
    v_min_id := v_from_id;
    v_max_id := v_to_id;
  ELSE
    v_min_id := v_to_id;
    v_max_id := v_from_id;
  END IF;

  -- Luôn khóa theo thứ tự id tăng dần
  SELECT balance
  INTO   :dummy
  FROM   accounts
  WHERE  id IN (v_min_id, v_max_id)
  FOR UPDATE;

  -- Sau đó mới update
  UPDATE accounts
  SET balance = balance - 100
  WHERE id = v_from_id;

  UPDATE accounts
  SET balance = balance + 100
  WHERE id = v_to_id;

  COMMIT;
END;
/
```

**Ý tưởng:** mọi session đều khóa theo cùng thứ tự → giảm khả năng deadlock.

---

## 3. Tránh “SELECT … FOR UPDATE” không cần thiết

- Chỉ dùng `SELECT ... FOR UPDATE` khi **thực sự cần khóa dòng** để cập nhật sau đó.
- Nếu chỉ đọc dữ liệu, dùng `SELECT` bình thường (Oracle hỗ trợ read consistency, không cần khóa).

**Ví dụ:**

```sql
-- Chỉ đọc, không cần khóa
SELECT balance
FROM accounts
WHERE id = 1;
```

## 4. Cập nhật theo batch nhỏ

- Khi cần cập nhật nhiều dòng, chia thành **batch nhỏ** để:
  - Giảm thời gian giữ khóa.
  - Giảm contention.

**Ví dụ: cập nhật 1000 dòng mỗi lần**

```sql
DECLARE
  CURSOR c IS
    SELECT id
    FROM   big_table
    WHERE  status = 'NEW'
    FOR UPDATE SKIP LOCKED;  -- tránh chờ khóa

  v_count NUMBER := 0;
BEGIN
  FOR r IN c LOOP
    UPDATE big_table
    SET    status = 'PROCESSED'
    WHERE  CURRENT OF c;

    v_count := v_count + 1;

    IF v_count >= 1000 THEN
      COMMIT;        -- commit batch
      v_count := 0;
    END IF;
  END LOOP;

  COMMIT;
END;
/
```

**Input:** bảng `big_table` có nhiều dòng `status = 'NEW'`  
**Output:** các dòng được cập nhật dần sang `PROCESSED`, commit theo batch 1000 dòng.

## 5. Dùng `SKIP LOCKED` khi xử lý song song

- Khi nhiều session cùng xử lý hàng đợi / job, dùng `FOR UPDATE SKIP LOCKED` để:
  - Bỏ qua các dòng đang bị khóa.
  - Tránh chờ (blocking).

**Ví dụ:**

```sql
SELECT id, payload
FROM   job_queue
WHERE  status = 'NEW'
FOR UPDATE SKIP LOCKED;
```

**Luồng xử lý:**

1. Session A lấy 10 job `FOR UPDATE SKIP LOCKED`, set `status = 'RUNNING'`.
2. Session B chạy cùng câu lệnh, sẽ bỏ qua 10 job đang bị khóa, lấy job khác.
3. Sau khi xử lý xong, mỗi session `COMMIT` → giải phóng khóa.

## 6. Thiết kế index tốt để giảm khóa hàng loạt

- Câu lệnh `UPDATE`/`DELETE` quét nhiều dòng không cần thiết sẽ:
  - Giữ khóa trên nhiều row.
  - Tăng contention.

**Giải pháp:**

- Tạo index phù hợp với điều kiện `WHERE` để chỉ chạm vào những dòng cần thiết.

**Ví dụ:**

```sql
-- Câu lệnh
UPDATE orders
SET    status = 'CLOSED'
WHERE  customer_id = :cust_id
AND    status = 'OPEN';
```

Tạo index:

```sql
CREATE INDEX idx_orders_cust_status
ON orders (customer_id, status);
```

Kết quả: Oracle chỉ khóa những dòng phù hợp, không quét toàn bảng.

## 7. Chọn isolation level phù hợp

- Oracle mặc định `READ COMMITTED` (thường là đủ).
- Tránh dùng `SERIALIZABLE` nếu không cần, vì:
  - Dễ gây lỗi `ORA-08177: can't serialize access for this transaction`.
  - Tăng contention.

**Ví dụ set isolation level:**

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- hoặc
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

Chỉ dùng `SERIALIZABLE` cho các nghiệp vụ cần tính nhất quán rất cao và đã được thiết kế kỹ.

## 8. Commit/Rollback rõ ràng, tránh “treo” session

- Luôn đảm bảo:
  - Có `COMMIT` hoặc `ROLLBACK` ở cuối transaction.
  - Không để session mở transaction lâu mà không làm gì.

**Ví dụ:**

```sql
BEGIN
  UPDATE accounts
  SET balance = balance - 100
  WHERE id = 1;

  -- Nếu có lỗi, rollback
  EXCEPTION
    WHEN OTHERS THEN
      ROLLBACK;
      RAISE;
END;
/
```

---

## 9. Tóm tắt ý chính

- Giữ transaction **ngắn** và **tập trung**.
- Cập nhật dữ liệu theo **thứ tự nhất quán** để tránh deadlock.
- Chỉ khóa khi cần (`SELECT ... FOR UPDATE`).
- Dùng **batch nhỏ** và `SKIP LOCKED` cho xử lý song song.
- Thiết kế **index tốt** để giảm số row bị khóa.
- Chọn **isolation level** phù hợp (thường là `READ COMMITTED`).
- Luôn **commit/rollback rõ ràng**, tránh để transaction mở lâu.

Nếu bạn cho biết thêm: loại ứng dụng (web, batch, job queue, microservice, v.v.), mình có thể đưa ví dụ code cụ thể hơn (PL/SQL, Java JDBC, .NET, v.v.) phù hợp với Oracle.
