
Dưới đây là tóm tắt ngắn gọn về **SQL Tuning – Statistics trong Oracle** (dựa trên chủ đề, vì nội dung chi tiết chưa được cung cấp):

---

## 1. Statistics là gì?

Trong Oracle, **optimizer statistics** là tập thông tin về:
- Số dòng (row count)
- Số khối (blocks)
- Phân bố giá trị cột (histograms)
- Số giá trị khác nhau (NDV – number of distinct values)
- Độ chọn lọc của index, v.v.

Optimizer dùng statistics để chọn **execution plan tốt nhất**. Statistics không chính xác hoặc lỗi thời → plan kém → SQL chạy chậm.

---

## 2. Các loại statistics chính

1. **Table statistics**
   - Số dòng: `NUM_ROWS`
   - Số block: `BLOCKS`
   - Kích thước: `AVG_ROW_LEN`, v.v.

2. **Column statistics**
   - `NUM_DISTINCT`, `DENSITY`
   - Histograms (phân bố giá trị)

3. **Index statistics**
   - `BLEVEL` (độ sâu B-tree)
   - `LEAF_BLOCKS`, `DISTINCT_KEYS`

4. **System statistics**
   - Thông tin về I/O, CPU, tốc độ đọc, v.v.

---

## 3. Thu thập statistics (gather statistics)

### 3.1. Thu thập statistics cho bảng

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',
    tabname          => 'EMPLOYEES',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE  -- gather luôn index stats
  );
END;
/
```

**Ý nghĩa chính:**
- `AUTO_SAMPLE_SIZE`: Oracle tự chọn tỷ lệ mẫu.
- `SIZE AUTO`: Oracle tự quyết định có tạo histogram hay không.
- `cascade => TRUE`: thu thập luôn statistics cho index của bảng.

---

### 3.2. Thu thập statistics cho schema

```sql
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(
    ownname          => 'HR',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    options          => 'GATHER AUTO'
  );
END;
/
```

`GATHER AUTO`: chỉ gather cho các đối tượng **thiếu** hoặc **cần cập nhật** statistics.

---

### 3.3. Thu thập system statistics

```sql
BEGIN
  DBMS_STATS.GATHER_SYSTEM_STATS('START');
  -- chạy workload thực tế trong vài giờ
  -- ...
  DBMS_STATS.GATHER_SYSTEM_STATS('STOP');
END;
/
```

System stats giúp optimizer hiểu rõ hơn về **CPU vs I/O**, từ đó chọn plan phù hợp.

---

## 4. Xem statistics

### 4.1. Xem table statistics

```sql
SELECT table_name, num_rows, blocks, last_analyzed
FROM   dba_tab_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

**Ví dụ output (giả định):**

| TABLE_NAME | NUM_ROWS | BLOCKS | LAST_ANALYZED        |
|-----------|----------|--------|----------------------|
| EMPLOYEES | 107      | 8      | 20-MAY-2026 10:15:00 |

---

### 4.2. Xem column statistics

```sql
SELECT column_name, num_distinct, density, histogram
FROM   dba_tab_col_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

## 5. Ảnh hưởng của statistics đến execution plan

Ví dụ: cùng một câu SQL, nhưng statistics khác nhau → plan khác nhau.

### 5.1. Trước khi gather statistics

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Output (giả định):**

- Optimizer chọn **FULL TABLE SCAN** vì nghĩ bảng nhỏ hoặc không biết phân bố dữ liệu.

---

### 5.2. Sau khi gather statistics

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES',
    cascade => TRUE
  );
END;
/

EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Output (giả định):**

- Optimizer chuyển sang dùng **INDEX RANGE SCAN** trên index `EMP_DEPT_IX` vì biết rằng `department_id = 10` rất chọn lọc.

## 6. Một số kỹ thuật tuning liên quan đến statistics

1. **Lock statistics** (khóa statistics để không bị ghi đè):

```sql
BEGIN
  DBMS_STATS.LOCK_TABLE_STATS('HR', 'EMPLOYEES');
END;
/
```

2. **Unlock statistics**:

```sql
BEGIN
  DBMS_STATS.UNLOCK_TABLE_STATS('HR', 'EMPLOYEES');
END;
/
```

3. **Export/Import statistics** (backup/restore statistics):

```sql
-- Tạo table lưu stats
EXEC DBMS_STATS.CREATE_STAT_TABLE('HR', 'HR_STATS_BACKUP');

-- Export
EXEC DBMS_STATS.EXPORT_SCHEMA_STATS('HR', 'HR_STATS_BACKUP');

-- Import
EXEC DBMS_STATS.IMPORT_SCHEMA_STATS('HR', 'HR_STATS_BACKUP');
```

---

## 7. Ý chính cần nhớ

- Statistics **chính xác và cập nhật** là nền tảng của SQL tuning trong Oracle.
- Dùng `DBMS_STATS` (không dùng `ANALYZE` cũ).
- Tự động hóa việc gather statistics (scheduler job) nhưng cẩn thận với:
  - Bảng rất lớn, thay đổi ít.
  - Bảng nhỏ nhưng thay đổi nhiều.
- Kiểm tra execution plan trước/sau khi thay đổi statistics để đánh giá tác động.
