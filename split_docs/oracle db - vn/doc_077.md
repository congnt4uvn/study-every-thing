
Dưới đây là tóm tắt ngắn gọn về **Table Locks trong Oracle** (khóa bảng), kèm ví dụ code. Vì bạn chỉ đưa tiêu đề, mình sẽ tập trung vào các ý chính thường gặp về “Locking & Isolation: Table locks” trong Oracle.

---

## 1. Khái niệm chung

- **Table lock**: Khóa ở mức bảng, dùng để:
  - Bảo vệ cấu trúc bảng (DDL).
  - Điều khiển mức độ truy cập đồng thời (DML).
- Oracle chủ yếu dùng **row-level lock** cho DML, nhưng **table lock** vẫn được dùng:
  - Tự động (khi DDL).
  - Hoặc thủ công (LOCK TABLE).

---

## 2. Các chế độ khóa bảng phổ biến

Một số mode thường gặp:

- `ROW SHARE` (RS)
- `ROW EXCLUSIVE` (RX)
- `SHARE` (S)
- `SHARE ROW EXCLUSIVE` (SRX)
- `EXCLUSIVE` (X)

Mức độ “càng mạnh” thì càng hạn chế session khác.

---

## 3. Khi nào Oracle khóa bảng?

1. **DML bình thường (INSERT/UPDATE/DELETE)**  
   - Oracle khóa **row-level** cho các dòng bị ảnh hưởng.  
   - Đồng thời đặt **table lock nhẹ** (thường là ROW EXCLUSIVE) để bảo vệ cấu trúc bảng.

2. **DDL (ALTER TABLE, DROP TABLE, TRUNCATE, …)**  
   - Thường cần **EXCLUSIVE table lock**.  
   - Có thể chờ các transaction khác kết thúc hoặc gây lỗi nếu không lấy được lock.

3. **LOCK TABLE**  
   - Lập trình viên có thể chủ động khóa bảng ở mode mong muốn.

---

## 4. Ví dụ: Khóa bảng bằng `LOCK TABLE`

### 4.1. Khóa bảng để chỉ cho phép đọc (SHARE)

```sql
-- Session 1
LOCK TABLE employees IN SHARE MODE;
```

- Ý nghĩa:  
  - Cho phép session khác **SELECT**.  
  - Hạn chế một số DML/DDL xung đột.

**Kiểm tra trong Session 2:**

```sql
-- Session 2
SELECT * FROM employees;  -- OK
UPDATE employees SET salary = salary + 100 WHERE employee_id = 100; -- có thể bị chờ hoặc lỗi tùy mode
```

---

### 4.2. Khóa bảng độc quyền (EXCLUSIVE)

```sql
-- Session 1
LOCK TABLE employees IN EXCLUSIVE MODE;
```

- Ý nghĩa:
  - Session 1 có thể SELECT/INSERT/UPDATE/DELETE.
  - Session khác bị chặn với hầu hết DML/DDL trên bảng này cho đến khi Session 1 COMMIT/ROLLBACK.

**Session 2 (trong lúc Session 1 chưa COMMIT):**

```sql
UPDATE employees SET salary = salary + 100 WHERE employee_id = 100;
-- Bị treo (waiting) cho đến khi Session 1 COMMIT/ROLLBACK
```

---

## 5. Ví dụ: Ảnh hưởng của Isolation (READ COMMITTED)

Giả sử mức isolation mặc định: `READ COMMITTED`.

### Session 1

```sql
UPDATE employees
SET salary = salary + 100
WHERE employee_id = 100;

-- Chưa COMMIT
```

### Session 2

```sql
SELECT salary
FROM employees
WHERE employee_id = 100;
```

**Kết quả (Output):**

- Session 2 **thấy giá trị cũ** (trước UPDATE) nhờ cơ chế **read-consistent** (undo).
- Session 2 **không bị block** khi chỉ SELECT.
- Nếu Session 2 cố UPDATE cùng dòng:

```sql
UPDATE employees
SET salary = salary + 200
WHERE employee_id = 100;
```

- Lệnh này sẽ **bị chờ** cho đến khi Session 1 COMMIT/ROLLBACK, vì row-level lock.

---

## 6. Ví dụ: DDL cần EXCLUSIVE table lock

### Session 1

```sql
UPDATE employees
SET salary = salary + 100
WHERE employee_id = 100;

-- Chưa COMMIT
```

### Session 2

```sql
ALTER TABLE employees ADD (bonus NUMBER);
```

**Kết quả:**

- `ALTER TABLE` cần **EXCLUSIVE table lock**.
- Nó sẽ **chờ** Session 1 COMMIT/ROLLBACK.
- Nếu timeout hoặc bị hủy, sẽ báo lỗi (ví dụ ORA-00054: resource busy and acquire with NOWAIT specified).

---

## 7. Tóm tắt ý chính

- Oracle dùng **row-level lock** cho DML, nhưng vẫn có **table lock**:
  - Tự động (DML/DDL).
  - Thủ công (`LOCK TABLE`).
- **Isolation level** (thường là READ COMMITTED) đảm bảo:
  - SELECT không bị block bởi DML chưa commit (nhờ read-consistency).
  - DML trên cùng dòng sẽ chờ nhau (row lock).
- **DDL** thường cần **EXCLUSIVE table lock**, có thể bị chặn bởi DML đang mở transaction.

Nếu bạn gửi thêm nội dung chi tiết (ví dụ đoạn tài liệu cụ thể), mình có thể tóm tắt lại chính xác hơn theo đúng tài liệu đó.
