
Dưới đây là tóm tắt ngắn gọn về chuẩn hoá quan hệ (Normalization) trong thiết kế CSDL quan hệ Oracle, tập trung vào 1NF, 2NF, 3NF, BCNF, kèm ví dụ (SQL) và dữ liệu minh hoạ.

---

## 1. Mục tiêu của chuẩn hoá (Normalization)

- Giảm dư thừa dữ liệu (data redundancy)  
- Tránh bất thường khi:
  - Thêm dữ liệu (insertion anomaly)
  - Cập nhật dữ liệu (update anomaly)
  - Xoá dữ liệu (deletion anomaly)
- Tổ chức bảng thành các quan hệ “tốt”, dễ bảo trì, dễ mở rộng.

---

## 2. First Normal Form (1NF)

**Ý chính:**
- Mỗi ô (cell) chỉ chứa **một giá trị nguyên tử** (atomic value).
- Không có cột lặp (repeating groups), không lưu danh sách/array trong một cột.

**Ví dụ vi phạm 1NF**

```sql
-- Bảng KHACH_HANG vi phạm 1NF vì cột DIEN_THOAI chứa nhiều số
CREATE TABLE KHACH_HANG_RAW (
    MA_KH      NUMBER PRIMARY KEY,
    TEN_KH     VARCHAR2(100),
    DIEN_THOAI VARCHAR2(200)  -- "0901..., 0902..., 0903..."
);
```

**Dữ liệu (input)**

```sql
INSERT INTO KHACH_HANG_RAW VALUES (1, 'Nguyen Van A', '0901000001, 0902000002');
```

**Chuẩn hoá lên 1NF**

Tách thành bảng KHACH_HANG và bảng DIEN_THOAI_KH:

```sql
CREATE TABLE KHACH_HANG (
    MA_KH  NUMBER PRIMARY KEY,
    TEN_KH VARCHAR2(100)
);

CREATE TABLE DIEN_THOAI_KH (
    MA_KH      NUMBER,
    SO_DT      VARCHAR2(20),
    CONSTRAINT PK_DT PRIMARY KEY (MA_KH, SO_DT),
    CONSTRAINT FK_DT_KH FOREIGN KEY (MA_KH)
        REFERENCES KHACH_HANG(MA_KH)
);
```

**Dữ liệu sau chuẩn hoá (output)**

```sql
INSERT INTO KHACH_HANG VALUES (1, 'Nguyen Van A');
INSERT INTO DIEN_THOAI_KH VALUES (1, '0901000001');
INSERT INTO DIEN_THOAI_KH VALUES (1, '0902000002');
```

## 3. Second Normal Form (2NF)

**Áp dụng khi khóa chính là khóa tổng hợp (composite key).**

**Ý chính:**
- Bảng đã ở 1NF.
- Mọi thuộc tính không khoá phải **phụ thuộc đầy đủ** vào toàn bộ khóa chính,  
  không được phụ thuộc **một phần** vào một phần của khóa tổng hợp.

**Ví dụ vi phạm 2NF**

```sql
-- Khóa chính: (MA_DON, MA_SP)
CREATE TABLE CHI_TIET_DON_HANG_RAW (
    MA_DON      NUMBER,
    MA_SP       NUMBER,
    TEN_SP      VARCHAR2(100),  -- phụ thuộc vào MA_SP, không cần MA_DON
    GIA_SP      NUMBER,         -- phụ thuộc vào MA_SP
    SO_LUONG    NUMBER,
    CONSTRAINT PK_CT_RAW PRIMARY KEY (MA_DON, MA_SP)
);
```

Ở đây:
- TEN_SP, GIA_SP chỉ phụ thuộc vào MA_SP, không phụ thuộc vào MA_DON  
→ phụ thuộc **một phần** vào khóa tổng hợp → vi phạm 2NF.

**Chuẩn hoá lên 2NF**

Tách thông tin sản phẩm ra bảng riêng:

```sql
CREATE TABLE SAN_PHAM (
    MA_SP   NUMBER PRIMARY KEY,
    TEN_SP  VARCHAR2(100),
    GIA_SP  NUMBER
);

CREATE TABLE CHI_TIET_DON_HANG (
    MA_DON   NUMBER,
    MA_SP    NUMBER,
    SO_LUONG NUMBER,
    CONSTRAINT PK_CT PRIMARY KEY (MA_DON, MA_SP),
    CONSTRAINT FK_CT_SP FOREIGN KEY (MA_SP)
        REFERENCES SAN_PHAM(MA_SP)
);
```

## 4. Third Normal Form (3NF)

**Ý chính:**
- Bảng đã ở 2NF.
- Không có **phụ thuộc bắc cầu (transitive dependency)** từ khóa chính đến thuộc tính không khoá thông qua một thuộc tính không khoá khác.

Nói cách khác:  
Thuộc tính không khoá **chỉ** phụ thuộc vào khóa chính, **không** phụ thuộc vào một thuộc tính không khoá khác.

**Ví dụ vi phạm 3NF**

```sql
CREATE TABLE NHAN_VIEN_RAW (
    MA_NV       NUMBER PRIMARY KEY,
    TEN_NV      VARCHAR2(100),
    MA_PHONG    NUMBER,
    TEN_PHONG   VARCHAR2(100)  -- phụ thuộc vào MA_PHONG, không phải MA_NV trực tiếp
);
```

Phụ thuộc:
- MA_NV → MA_PHONG
- MA_PHONG → TEN_PHONG  
→ MA_NV → TEN_PHONG (phụ thuộc bắc cầu) → vi phạm 3NF.

**Chuẩn hoá lên 3NF**

```sql
CREATE TABLE PHONG_BAN (
    MA_PHONG   NUMBER PRIMARY KEY,
    TEN_PHONG  VARCHAR2(100)
);

CREATE TABLE NHAN_VIEN (
    MA_NV      NUMBER PRIMARY KEY,
    TEN_NV     VARCHAR2(100),
    MA_PHONG   NUMBER,
    CONSTRAINT FK_NV_PB FOREIGN KEY (MA_PHONG)
        REFERENCES PHONG_BAN(MA_PHONG)
);
```

## 5. Boyce–Codd Normal Form (BCNF)

**Ý chính (mạnh hơn 3NF):**
- Với mọi phụ thuộc hàm X → Y trong bảng, **X phải là siêu khoá (superkey)**.
- Loại bỏ các trường hợp mà thuộc tính không phải khoá lại quyết định một phần của khoá.

BCNF ⊇ 3NF:  
- Mọi bảng ở BCNF đều ở 3NF, nhưng ngược lại không phải lúc nào cũng đúng.

**Ví dụ vi phạm BCNF (nhưng có thể vẫn ở 3NF)**

Giả sử bảng phân công giảng dạy:

```sql
CREATE TABLE PHAN_CONG_RAW (
    GIANG_VIEN   VARCHAR2(50),
    MON_HOC      VARCHAR2(50),
    PHONG        VARCHAR2(20),
    CONSTRAINT PK_PC_RAW PRIMARY KEY (GIANG_VIEN, MON_HOC)
);
```

Giả sử quy tắc nghiệp vụ:
- Mỗi PHONG chỉ dạy **một MON_HOC** (PHONG → MON_HOC).
- Mỗi GIANG_VIEN dạy nhiều môn, nhưng mỗi (GIANG_VIEN, MON_HOC) là duy nhất.

Phụ thuộc hàm:
- (GIANG_VIEN, MON_HOC) → PHONG  (do là khóa chính)
- PHONG → MON_HOC

Ở đây:
- PHONG không phải là siêu khoá, nhưng lại quyết định MON_HOC  
→ vi phạm BCNF.

**Chuẩn hoá lên BCNF**

Tách thành:

```sql
CREATE TABLE PHONG_MON (
    PHONG    VARCHAR2(20) PRIMARY KEY,
    MON_HOC  VARCHAR2(50)
);

CREATE TABLE PHAN_CONG (
    GIANG_VIEN  VARCHAR2(50),
    PHONG       VARCHAR2(20),
    CONSTRAINT PK_PC PRIMARY KEY (GIANG_VIEN, PHONG),
    CONSTRAINT FK_PC_PM FOREIGN KEY (PHONG)
        REFERENCES PHONG_MON(PHONG)
);
```

## 6. Tóm tắt ngắn

- **1NF**:  
  - Không nhóm lặp, không danh sách trong một cột.  
  - Mỗi ô là một giá trị nguyên tử.

- **2NF**:  
  - Đã 1NF.  
  - Không có phụ thuộc **một phần** vào khóa tổng hợp.

- **3NF**:  
  - Đã 2NF.  
  - Không có phụ thuộc **bắc cầu** từ khóa chính qua thuộc tính không khoá.

- **BCNF**:  
  - Mọi phụ thuộc hàm X → Y đều có X là **siêu khoá**.  
  - Mạnh hơn 3NF, xử lý các trường hợp phức tạp về phụ thuộc hàm.

Nếu bạn muốn, tôi có thể tiếp tục với ví dụ đầy đủ hơn (từ bảng “xấu” đến mô hình chuẩn hoá hoàn chỉnh) kèm các truy vấn SELECT minh hoạ.
