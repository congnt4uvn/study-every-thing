
Dưới đây là tóm tắt ngắn gọn về chủ đề “SQL Tuning Techniques: Stale/missing statistics impact” (Ảnh hưởng của thống kê cũ/thiếu trong tối ưu SQL trên Oracle), kèm ví dụ mã.

---

## 1. Ý chính

1. **Optimizer của Oracle dựa rất nhiều vào statistics**  
   - Statistics (thống kê) trên bảng, index, cột giúp Oracle ước lượng số dòng, độ chọn lọc, phân phối dữ liệu…  
   - Dựa trên đó, Optimizer chọn kế hoạch thực thi (execution plan) tốt nhất.

2. **Statistics bị cũ (stale) hoặc thiếu (missing) → kế hoạch kém**  
   - Khi dữ liệu thay đổi nhiều mà không cập nhật statistics, Oracle vẫn dùng thống kê cũ → ước lượng sai số dòng.  
   - Khi không có statistics, Oracle dùng giá trị mặc định hoặc dynamic sampling → dễ chọn plan không tối ưu.  
   - Hậu quả: full table scan không cần thiết, dùng sai index, join order không hợp lý, tăng I/O, CPU, thời gian chạy.

3. **Cần kiểm tra và cập nhật statistics định kỳ**  
   - Dùng `DBMS_STATS` để thu thập (gather) statistics.  
   - Kiểm tra bảng/index nào đang stale/missing.  
   - Thiết lập job tự động hoặc quy trình thủ công phù hợp với mức độ thay đổi dữ liệu.

---

## 2. Kiểm tra statistics stale/missing

### 2.1. Kiểm tra trạng thái statistics của bảng

```sql
SELECT owner,
       table_name,
       num_rows,
       last_analyzed,
       stale_stats
FROM   dba_tab_statistics
WHERE  owner = 'HR';
```

**Ý nghĩa cột chính:**
- `NUM_ROWS`: số dòng ước lượng theo statistics.
- `LAST_ANALYZED`: thời điểm cuối cùng gather statistics.
- `STALE_STATS`: `YES` nếu Oracle đánh dấu thống kê đã cũ.

**Ví dụ output (rút gọn):**

| OWNER | TABLE_NAME | NUM_ROWS | LAST_ANALYZED        | STALE_STATS |
|-------|------------|----------|----------------------|------------|
| HR    | EMPLOYEES  | 107      | 2026-05-01 10:15:00  | NO         |
| HR    | JOBS       | 19       | 2025-12-10 09:00:00  | YES        |

---

### 2.2. Kiểm tra bảng chưa có statistics

```sql
SELECT owner,
       table_name,
       num_rows,
       last_analyzed
FROM   dba_tab_statistics
WHERE  owner = 'HR'
AND    last_analyzed IS NULL;
```

Bảng có `LAST_ANALYZED` = `NULL` thường là chưa từng gather statistics.

---

## 3. Thu thập (gather) statistics

### 3.1. Gather statistics cho một bảng

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',
    tabname          => 'EMPLOYEES',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE  -- gather luôn index
  );
END;
/
```

- `AUTO_SAMPLE_SIZE`: để Oracle tự chọn tỷ lệ mẫu.
- `CASCADE => TRUE`: gather statistics cho index của bảng.

---

### 3.2. Gather statistics cho cả schema

```sql
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(
    ownname          => 'HR',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    options          => 'GATHER AUTO'  -- chỉ gather cho đối tượng stale/missing
  );
END;
/
```

`GATHER AUTO` giúp chỉ gather nơi cần thiết (stale/missing), tránh tốn tài nguyên.

## 4. Ví dụ: Ảnh hưởng của statistics cũ/thiếu

### 4.1. Trước khi gather statistics

Giả sử bảng `SALES` mới insert thêm rất nhiều dữ liệu nhưng chưa cập nhật statistics.

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   sales
WHERE  sale_date >= DATE '2026-01-01';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Ví dụ output (rút gọn):**

```text
Plan hash value: 123456789

---------------------------------------------------------
| Id  | Operation         | Name  | Rows  | Cost (%CPU) |
---------------------------------------------------------
|   0 | SELECT STATEMENT  |       |  1000 |   10   (10) |
|*  1 | TABLE ACCESS FULL | SALES |  1000 |   10   (10) |
---------------------------------------------------------
```

- `Rows` ước lượng = 1000 (theo statistics cũ), trong khi thực tế có thể là 1,000,000 → Optimizer đánh giá full scan rẻ.

### 4.2. Sau khi gather statistics

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'SH',
    tabname => 'SALES',
    cascade => TRUE
  );
END;
/

EXPLAIN PLAN FOR
SELECT *
FROM   sales
WHERE  sale_date >= DATE '2026-01-01';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Ví dụ output (rút gọn):**

```text
Plan hash value: 987654321

--------------------------------------------------------------
| Id  | Operation                           | Name   | Rows   |
--------------------------------------------------------------
|   0 | SELECT STATEMENT                    |        | 100000 |
|*  1 | INDEX RANGE SCAN                    | IDX_DT | 100000 |
|   2 | TABLE ACCESS BY INDEX ROWID BATCHED | SALES  | 100000 |
--------------------------------------------------------------
```

- Sau khi statistics chính xác, Optimizer chọn `INDEX RANGE SCAN` thay vì full table scan → giảm I/O, tăng tốc độ.

## 5. Một số lưu ý thực tế

1. **Thiết lập job tự động**  
   - Oracle có auto task gather stats (từ 10g trở lên).  
   - Có thể kiểm tra/điều chỉnh qua `DBA_AUTOTASK_CLIENT`, `DBA_SCHEDULER_JOBS`.

2. **Không gather quá thường xuyên**  
   - Gather quá nhiều trên bảng lớn gây tốn tài nguyên, lock metadata.  
   - Nên dựa vào mức độ thay đổi dữ liệu (số DML) hoặc dùng `GATHER AUTO`.

3. **Lưu/restore statistics quan trọng**  
   - Trước khi thay đổi lớn, có thể lưu statistics để rollback nếu plan xấu:
   ```sql
   BEGIN
     DBMS_STATS.CREATE_STAT_TABLE('HR', 'HR_STATS_BACKUP');
     DBMS_STATS.EXPORT_SCHEMA_STATS('HR', 'HR_STATS_BACKUP');
   END;
   /
   ```

---

## 6. Tóm tắt ngắn

- Optimizer cần statistics chính xác để chọn execution plan tốt.  
- Statistics cũ (stale) hoặc thiếu (missing) → ước lượng sai → plan kém → SQL chạy chậm.  
- Dùng `DBMS_STATS` để kiểm tra và gather statistics định kỳ, ưu tiên `GATHER AUTO`.  
- Luôn xem execution plan trước/sau khi cập nhật statistics để đánh giá tác động.
