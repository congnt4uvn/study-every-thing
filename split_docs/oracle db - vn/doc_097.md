
Dưới đây là tóm tắt ngắn gọn (mức cao) về “High Availability & Backup” trong Oracle Database, tập trung vào lý do tại sao lập trình viên nên quan tâm, kèm ví dụ code minh họa.

---

## 1. High Availability (HA) là gì và vì sao dev phải quan tâm?

**High Availability** = Hệ thống luôn sẵn sàng, ít downtime nhất có thể.  
Với ứng dụng:

- Nếu DB ngừng hoạt động → API lỗi, web/app không dùng được.
- Mất kết nối DB giữa chừng → giao dịch dở dang, dữ liệu có thể không nhất quán.
- HA tốt giúp:
  - Giảm lỗi 500/timeout từ backend.
  - Giảm rủi ro mất dữ liệu khi server chết.
  - Cho phép nâng cấp/bảo trì với downtime rất thấp.

Trong Oracle, HA thường liên quan đến:

- **RAC (Real Application Clusters)**: nhiều node cùng truy cập 1 database.
- **Data Guard**: database standby để failover khi primary hỏng.
- **Services & Transparent Application Failover (TAF)**: client tự động chuyển sang node khác.

### Ví dụ: Kết nối JDBC với Oracle RAC (HA)

```java
// Ví dụ URL kết nối RAC với 2 node
String url = "jdbc:oracle:thin:@(DESCRIPTION=" +
             "(ADDRESS_LIST=" +
               "(ADDRESS=(PROTOCOL=TCP)(HOST=rac-node1)(PORT=1521))" +
               "(ADDRESS=(PROTOCOL=TCP)(HOST=rac-node2)(PORT=1521))" +
             ")" +
             "(CONNECT_DATA=(SERVICE_NAME=myservice)))";

Properties props = new Properties();
props.put("user", "app_user");
props.put("password", "secret");
// TAF / Fast Connection Failover có thể cấu hình thêm qua Oracle UCP hoặc ONS

Connection conn = DriverManager.getConnection(url, props);
```

**Ý nghĩa với dev**:  
Nếu một node DB chết, kết nối có thể tự động chuyển sang node khác, giảm lỗi cho người dùng. Dev cần:

- Dùng đúng connection string (có nhiều ADDRESS).
- Thiết kế retry logic hợp lý (retry idempotent, không lặp giao dịch nguy hiểm).
- Xử lý lỗi kết nối (SQLRecoverableException, timeout, v.v.).

## 2. Backup là gì và vì sao dev phải quan tâm?

**Backup** = Bản sao dữ liệu để khôi phục khi:

- Lỗi logic: code bug xóa nhầm dữ liệu.
- Lỗi người dùng: xóa nhầm record, import nhầm file.
- Lỗi hệ thống: hỏng disk, corruption, ransomware, v.v.

Dù DBA chịu trách nhiệm backup, dev vẫn cần hiểu:

- **RPO (Recovery Point Objective)**: chấp nhận mất tối đa bao nhiêu dữ liệu (theo thời gian) khi khôi phục? (ví dụ: 5 phút, 1 giờ).
- **RTO (Recovery Time Objective)**: chấp nhận downtime bao lâu để khôi phục? (ví dụ: 15 phút, 2 giờ).
- Thiết kế tính năng ứng dụng phù hợp với chiến lược backup/restore (soft delete, audit, versioning).

---

## 3. Các cơ chế backup/khôi phục trong Oracle (mức cao)

- **RMAN (Recovery Manager)**: công cụ chuẩn để backup/restore.
- **ARCHIVELOG mode**: lưu redo log để có thể khôi phục đến thời điểm cụ thể.
- **Point-in-Time Recovery (PITR)**: khôi phục database/tablespace đến thời điểm trước khi lỗi xảy ra.
- **Flashback**: khôi phục dữ liệu ở mức row/table/database mà không cần full restore.

### Ví dụ: Flashback Table (khôi phục bảng bị xóa nhầm)

Giả sử dev chạy nhầm:

```sql
-- Lệnh gây lỗi
DROP TABLE customers;
```

Nếu DBA đã bật Flashback cho tablespace, có thể khôi phục:

```sql
-- Khôi phục bảng về trạng thái trước khi drop
FLASHBACK TABLE customers TO BEFORE DROP;
```

**Kết quả (output)**:  
Bảng `customers` xuất hiện lại với dữ liệu như trước khi bị DROP.

---

## 4. Ví dụ: Flashback Query (xem dữ liệu quá khứ)

Dev muốn xem dữ liệu của một row trước khi bị update:

```sql
-- Xem dữ liệu của khách hàng cách đây 10 phút
SELECT *
FROM customers AS OF TIMESTAMP (SYSTIMESTAMP - INTERVAL '10' MINUTE)
WHERE customer_id = 123;
```

**Ý nghĩa**:  
Có thể so sánh dữ liệu hiện tại và dữ liệu cũ để:

- Debug bug logic.
- Xây dựng tính năng “khôi phục phiên bản cũ” ở tầng ứng dụng.

## 5. Dev nên làm gì trong code?

1. **Thiết kế giao dịch an toàn**  
   - Dùng `COMMIT`/`ROLLBACK` rõ ràng.
   - Tránh transaction quá dài (dễ lock, khó khôi phục).

2. **Xử lý lỗi kết nối & retry**  
   - Phân biệt lỗi tạm thời (network, failover) vs lỗi logic (constraint, syntax).
   - Retry có kiểm soát, tránh double-insert/double-charge.

   Ví dụ pseudo-code:

   ```pseudo
   function saveOrder(order):
       for attempt in 1..3:
           try:
               begin transaction
               insert into orders ...
               insert into order_items ...
               commit
               return success
           catch transient_db_error:
               rollback
               if attempt == 3:
                   throw
               wait a bit and retry
           catch logical_error:
               rollback
               throw
   ```

3. **Hỗ trợ khôi phục dữ liệu ở tầng ứng dụng**  
   - Soft delete (cột `deleted_at`) thay vì `DELETE` cứng.
   - Lưu lịch sử thay đổi (audit table, event sourcing).
   - Giảm phụ thuộc vào việc DBA phải restore cả database chỉ vì vài record.

---

## 6. Tóm tắt cho lập trình viên

- **High Availability** giúp ứng dụng ít downtime, cần:
  - Connection string hỗ trợ nhiều node / service.
  - Retry logic và xử lý lỗi kết nối.
- **Backup & Recovery** bảo vệ khỏi mất dữ liệu:
  - Hiểu RPO/RTO để thiết kế tính năng phù hợp.
  - Tận dụng Flashback, PITR khi cần khôi phục.
- Dev không cần cấu hình Oracle ở mức DBA, nhưng cần viết code “thân thiện với HA & backup” để hệ thống ổn định, dễ khôi phục khi có sự cố.
