
Key idea  
Correlated subqueries often run row‑by‑row and can be slow. When possible, rewrite them as joins so Oracle can optimize better (use hash joins, merge joins, etc.), reducing logical I/O and execution time.

---

## 1. What is a correlated subquery?

A correlated subquery references columns from the outer query. It is executed repeatedly (once per row of the outer query).

### Example (correlated subquery)

```sql
-- Find employees whose salary is greater than the average salary in their department
SELECT e.employee_id,
       e.first_name,
       e.salary
FROM   employees e
WHERE  e.salary > (
         SELECT AVG(e2.salary)
         FROM   employees e2
         WHERE  e2.department_id = e.department_id
       );
```

- The inner query uses `e.department_id` from the outer query.
- Oracle may execute the inner query once for each department in the outer result set.

**Conceptual input/output**

- Input: `employees` table with columns `(employee_id, first_name, salary, department_id)`
- Output: rows of employees whose salary is above their own department’s average.

---

## 2. Rewrite as a join

You can often compute the needed value once (e.g., per department) and then join to it.

### Example (join rewrite)

```sql
SELECT e.employee_id,
       e.first_name,
       e.salary
FROM   employees e
JOIN   (
         SELECT   department_id,
                  AVG(salary) AS avg_salary
         FROM     employees
         GROUP BY department_id
       ) d
  ON   e.department_id = d.department_id
WHERE e.salary > d.avg_salary;
```

**What changed**

- The average salary per department is computed once in the inline view `d`.
- The main query joins `employees` to this precomputed result.
- Oracle can optimize this as a standard join, often more efficiently than repeated subquery execution.

**Same conceptual input/output**

- Input: same `employees` table.
- Output: same set of employees as the correlated subquery, but usually faster.

## 3. Another example: EXISTS → JOIN

### Correlated subquery with `EXISTS`

```sql
-- Get customers who have at least one order
SELECT c.customer_id,
       c.customer_name
FROM   customers c
WHERE  EXISTS (
         SELECT 1
         FROM   orders o
         WHERE  o.customer_id = c.customer_id
       );
```

### Equivalent join

```sql
SELECT DISTINCT
       c.customer_id,
       c.customer_name
FROM   customers c
JOIN   orders o
  ON   o.customer_id = c.customer_id;
```

- `EXISTS` is often efficient already, but in some cases a join with proper indexing can be as good or better.
- `DISTINCT` is used to avoid duplicates when a customer has multiple orders.

**Input**

- `customers(customer_id, customer_name, …)`
- `orders(order_id, customer_id, …)`

**Output**

- List of customers who have at least one order.

---

## 4. When to rewrite

Rewrite correlated subqueries as joins when:

- The subquery is executed many times (e.g., in the `SELECT` or `WHERE` clause).
- The subquery aggregates or filters on a key that can be grouped once and joined.
- You see high logical I/O or long execution times in the execution plan.

Be cautious when:

- The correlated subquery has complex conditions or needs `ROWNUM`, `CONNECT BY`, or analytic functions; ensure the join version returns identical results.
- The correlated subquery is using `EXISTS`/`NOT EXISTS` with subtle semantics; test carefully.

---

## 5. Summary

- Correlated subqueries can be less efficient because they may run once per outer row.
- Rewriting them as joins lets Oracle:
  - Compute aggregates once (e.g., per group).
  - Use more efficient join methods.
  - Reduce repeated work and logical I/O.
- Always compare execution plans and test results to confirm performance and correctness.
