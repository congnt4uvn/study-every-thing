
Here’s a concise summary of the main physical design partitioning methods in Oracle Database, with simple examples.

---

## 1. Range Partitioning

**Idea:** Rows are placed into partitions based on a range of values (often dates or numbers).

**Use case:** Time-based data (e.g., monthly sales), numeric ranges.

**Example: Range partition by date**

```sql
CREATE TABLE sales_range (
    sale_id     NUMBER,
    sale_date   DATE,
    amount      NUMBER
)
PARTITION BY RANGE (sale_date) (
    PARTITION p_2024_q1 VALUES LESS THAN (DATE '2024-04-01'),
    PARTITION p_2024_q2 VALUES LESS THAN (DATE '2024-07-01'),
    PARTITION p_2024_q3 VALUES LESS THAN (DATE '2024-10-01'),
    PARTITION p_2024_q4 VALUES LESS THAN (DATE '2025-01-01')
);
```

**Example query and “logical” output:**

```sql
SELECT sale_id, sale_date, amount
FROM   sales_range
WHERE  sale_date BETWEEN DATE '2024-04-01' AND DATE '2024-06-30';
```

Oracle will only scan partition `p_2024_q2` (partition pruning), improving performance.

---

## 2. List Partitioning

**Idea:** Rows are placed into partitions based on a discrete list of values.

**Use case:** Categorical data (e.g., country, region, status).

**Example: List partition by country**

```sql
CREATE TABLE customers_list (
    customer_id NUMBER,
    country     VARCHAR2(2),
    name        VARCHAR2(100)
)
PARTITION BY LIST (country) (
    PARTITION p_us VALUES ('US'),
    PARTITION p_uk VALUES ('UK'),
    PARTITION p_eu VALUES ('FR', 'DE', 'ES', 'IT'),
    PARTITION p_other VALUES (DEFAULT)
);
```

**Example query and “logical” output:**

```sql
SELECT customer_id, country
FROM   customers_list
WHERE  country = 'US';
```

Oracle will only scan partition `p_us`.

---

## 3. Hash Partitioning

**Idea:** Oracle applies a hash function to the partition key and distributes rows evenly across partitions.

**Use case:** Even distribution for high-volume OLTP tables, when there is no natural range or list grouping.

**Example: Hash partition by customer_id**

```sql
CREATE TABLE orders_hash (
    order_id     NUMBER,
    customer_id  NUMBER,
    order_date   DATE,
    amount       NUMBER
)
PARTITION BY HASH (customer_id)
PARTITIONS 4;  -- Oracle creates 4 hash partitions: SYS_P0, SYS_P1, ...
```

**Example query:**

```sql
SELECT *
FROM   orders_hash
WHERE  customer_id = 12345;
```

Oracle uses the hash of `12345` to find the single partition that contains the row.

---

## 4. Composite Partitioning

**Idea:** Combines two partitioning methods:  
- First level: usually RANGE or LIST  
- Second level (subpartition): usually HASH or LIST  

**Use case:** Large tables where you want both logical grouping (by time or category) and even distribution within each group.

### 4.1 Range-Hash Composite

**Example: Range by date, hash by customer_id**

```sql
CREATE TABLE sales_comp_range_hash (
    sale_id     NUMBER,
    sale_date   DATE,
    customer_id NUMBER,
    amount      NUMBER
)
PARTITION BY RANGE (sale_date)
SUBPARTITION BY HASH (customer_id)
SUBPARTITIONS 4
(
    PARTITION p_2024_q1 VALUES LESS THAN (DATE '2024-04-01'),
    PARTITION p_2024_q2 VALUES LESS THAN (DATE '2024-07-01')
);
```

- First, Oracle chooses the partition by `sale_date` (e.g., `p_2024_q2`).
- Then, within that partition, it uses a hash on `customer_id` to choose a subpartition.

**Example query:**

```sql
SELECT *
FROM   sales_comp_range_hash
WHERE  sale_date   = DATE '2024-05-10'
AND    customer_id = 1001;
```

Oracle prunes to partition `p_2024_q2` and then to one of its 4 hash subpartitions.

### 4.2 List-Hash Composite

**Example: List by region, hash by customer_id**

```sql
CREATE TABLE customers_comp_list_hash (
    customer_id NUMBER,
    region      VARCHAR2(10),
    name        VARCHAR2(100)
)
PARTITION BY LIST (region)
SUBPARTITION BY HASH (customer_id)
SUBPARTITIONS 2
(
    PARTITION p_americas VALUES ('NA', 'SA'),
    PARTITION p_europe   VALUES ('EU'),
    PARTITION p_apac     VALUES ('APAC')
);
```

---

## Key Points Summary

- **Range partitioning:** Based on value ranges (e.g., dates, numbers). Good for time-based data and partition pruning by range.
- **List partitioning:** Based on explicit lists of values (e.g., country codes). Good for categorical data.
- **Hash partitioning:** Uses a hash function to distribute rows evenly. Good for load balancing and avoiding hotspots.
- **Composite partitioning:** Combines methods (e.g., range-hash, list-hash) to get both logical grouping and even distribution, useful for very large tables.

These designs help with:
- Performance (partition pruning, parallelism)
- Manageability (archiving, backup, maintenance per partition)
- Scalability (handling very large tables efficiently).
