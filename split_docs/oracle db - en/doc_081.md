
Here’s a concise summary of **locking, isolation, and deadlocks in Oracle**, plus example code.

---

## 1. Locking in Oracle

- Oracle uses **locks** to control concurrent access to data.
- Main types:
  - **Row-level locks**: when you `UPDATE`/`DELETE` rows.
  - **Table-level locks**: when you do DDL (e.g., `ALTER TABLE`) or certain DML with explicit locks.
- Oracle uses **multi-version read consistency**:
  - Readers don’t block writers.
  - Writers don’t block readers.
  - Conflicts mainly occur between **writers** (DML) and **DDL** or other writers.

---

## 2. Isolation Levels (Oracle)

Oracle supports standard isolation levels, but internally behaves mostly like **READ COMMITTED** with MVCC:

- `READ COMMITTED` (default)
  - Each statement sees data committed before it starts.
  - Non-repeatable reads and phantom reads are possible.
- `SERIALIZABLE`
  - Transaction sees a consistent snapshot as of its start.
  - If a conflict is detected, Oracle raises `ORA-08177: can't serialize access for this transaction`.
- `READ ONLY`
  - Like SERIALIZABLE but no DML allowed.

**Example: set isolation level**

```sql
-- Set session isolation level to SERIALIZABLE
ALTER SESSION SET ISOLATION_LEVEL = SERIALIZABLE;
```

**No direct input/output**, but subsequent queries in the session will follow this isolation behavior.

---

## 3. What is a Deadlock?

- A **deadlock** occurs when two (or more) sessions hold locks that the other needs, and none can proceed.
- Oracle automatically:
  - Detects deadlocks.
  - Chooses one session as a victim.
  - Rolls back that statement and raises `ORA-00060: deadlock detected while waiting for resource`.

---

## 4. Typical Causes of Deadlocks

1. **Circular locking order**
   - Session A locks row 1, then tries row 2.
   - Session B locks row 2, then tries row 1.
2. **Mixed DML and foreign keys without indexes**
   - Updates/deletes on parent/child tables can cause full table locks if foreign key columns are not indexed.
3. **Application logic that updates the same set of rows in different orders**
4. **Lock escalation via DDL or explicit locks**
   - `LOCK TABLE ... IN EXCLUSIVE MODE` while others are doing DML.

---

## 5. Example: Deadlock Scenario

Assume a table:

```sql
CREATE TABLE accounts (
    id      NUMBER PRIMARY KEY,
    balance NUMBER
);

INSERT INTO accounts VALUES (1, 100);
INSERT INTO accounts VALUES (2, 200);
COMMIT;
```

### Session 1 (S1) – Input

```sql
-- S1: lock row with id = 1
UPDATE accounts SET balance = balance + 10 WHERE id = 1;

-- S1: now try to lock row with id = 2
UPDATE accounts SET balance = balance + 10 WHERE id = 2;
-- This may block if S2 already holds a lock on id = 2
```

### Session 2 (S2) – Input

```sql
-- S2: lock row with id = 2
UPDATE accounts SET balance = balance + 20 WHERE id = 2;

-- S2: now try to lock row with id = 1
UPDATE accounts SET balance = balance + 20 WHERE id = 1;
-- This may block if S1 already holds a lock on id = 1
```

If both sessions run in this interleaved way, Oracle detects a deadlock.

### Session 2 – Output (example)

```text
ORA-00060: deadlock detected while waiting for resource
```

The failing statement in S2 is rolled back; the transaction can be retried.

## 6. Detecting Deadlocks

- Oracle writes details to the **alert log** and a **trace file**.
- You can inspect locks with:

```sql
-- Current locks
SELECT
    s.sid,
    s.serial#,
    l.type,
    l.lmode,
    l.request,
    o.object_name
FROM v$lock l
JOIN v$session s ON l.sid = s.sid
LEFT JOIN dba_objects o ON l.id1 = o.object_id;
```

**Output (simplified example)**

```text
SID  SERIAL#  TYPE  LMODE  REQUEST  OBJECT_NAME
---  -------  ----  -----  -------  -----------
 10    12345   TX      6       0    (transaction lock)
 11    23456   TX      6       0    (transaction lock)
```

## 7. Preventing Deadlocks

1. **Consistent locking order**
   - Always update rows in the same order (e.g., by primary key).
   - Example: always `UPDATE ... WHERE id IN (1,2) ORDER BY id;`

2. **Index foreign keys**
   - For parent-child relationships, index the foreign key columns to avoid full table locks.
   - Example:

     ```sql
     CREATE TABLE parent (
         id NUMBER PRIMARY KEY
     );

     CREATE TABLE child (
         id        NUMBER PRIMARY KEY,
         parent_id NUMBER REFERENCES parent(id)
     );

     -- Prevention: index the foreign key
     CREATE INDEX idx_child_parent_id ON child(parent_id);
     ```

3. **Keep transactions short**
   - Commit frequently when appropriate.
   - Avoid holding locks while waiting for user input.

4. **Avoid unnecessary explicit locks**
   - Use `SELECT ... FOR UPDATE` only when needed.
   - Avoid `LOCK TABLE ... IN EXCLUSIVE MODE` unless absolutely required.

5. **Handle ORA-00060 in application code**
   - Catch the error and **retry** the transaction or statement.

   Pseudo-code example:

   ```pseudo
   try:
       execute_transaction()
   except ORA_00060:
       rollback
       wait_random_short_time()
       retry_transaction()
   ```

---

## 8. Example: Safer Update Pattern

To avoid deadlocks when updating multiple rows:

```sql
-- Always lock rows in a consistent order
UPDATE accounts
SET balance = balance + 10
WHERE id IN (1, 2)
ORDER BY id;  -- ensures same lock order across sessions
```

This reduces the chance of circular waits.
