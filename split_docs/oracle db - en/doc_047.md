
Here’s a concise summary and some practical examples about **when to use bitmap vs B-tree indexes in Oracle**.

---

## 1. B-tree Indexes

**Best for:**
- OLTP / transactional systems
- Columns with **high cardinality** (many distinct values)
  - e.g., `CUSTOMER_ID`, `ORDER_ID`, `EMAIL`
- Frequent **INSERT/UPDATE/DELETE** operations

**Characteristics:**
- Each index entry points directly to row(s) via ROWID.
- Very efficient for:
  - Equality predicates: `WHERE id = :b1`
  - Range predicates: `WHERE order_date BETWEEN :d1 AND :d2`
- Handles concurrent DML well.

**Example: Create B-tree index**

```sql
-- High-cardinality column, OLTP system
CREATE INDEX idx_orders_customer_id
    ON orders (customer_id);
```

**Example query using B-tree index**

```sql
-- Input
SELECT order_id, order_date, amount
FROM   orders
WHERE  customer_id = 101;

-- Expected behavior
-- Oracle uses IDX_ORDERS_CUSTOMER_ID to quickly find all rows
-- with customer_id = 101 via index range scan.
```

---

## 2. Bitmap Indexes

**Best for:**
- Data warehouse / reporting / OLAP systems
- Columns with **low to medium cardinality** (few distinct values)
  - e.g., `GENDER`, `STATUS`, `REGION_CODE`, `YES/NO` flags
- **Read-mostly** tables (few DML operations)
- Complex ad-hoc queries with many combined conditions

**Characteristics:**
- Store a bitmap for each distinct value; each bit represents a row.
- Very efficient for combining conditions with `AND` / `OR`:
  - `WHERE gender = 'F' AND status = 'ACTIVE'`
- Poor for heavy concurrent DML:
  - Updates/Deletes can cause locking of many rows.

**Example: Create bitmap index**

```sql
-- Low-cardinality column, data warehouse
CREATE BITMAP INDEX idx_customers_gender
    ON customers (gender);

CREATE BITMAP INDEX idx_customers_status
    ON customers (status);
```

**Example query using bitmap indexes**

```sql
-- Input
SELECT customer_id, name
FROM   customers
WHERE  gender = 'F'
AND    status = 'ACTIVE';

-- Expected behavior
-- Oracle can:
-- 1) Use IDX_CUSTOMERS_GENDER to get bitmap for 'F'
-- 2) Use IDX_CUSTOMERS_STATUS to get bitmap for 'ACTIVE'
-- 3) AND the bitmaps together
-- 4) Convert resulting bits to ROWIDs and fetch rows
```

---

## 3. When to Choose Which

**Use B-tree when:**
- OLTP system with many small transactions
- Column has many distinct values
- Queries often use ranges (`>`, `<`, `BETWEEN`)
- Table is heavily updated

**Use Bitmap when:**
- Data warehouse / reporting environment
- Table is large but mostly read-only
- Column has few distinct values
- Queries often combine multiple conditions on such columns
- You need fast ad-hoc analytics with many `AND`/`OR` filters

---

## 4. Mixed Example

```sql
-- Fact table in a data warehouse
CREATE TABLE sales (
    sale_id        NUMBER,
    customer_id    NUMBER,
    product_id     NUMBER,
    sale_date      DATE,
    region_code    VARCHAR2(2),  -- low cardinality
    channel        VARCHAR2(10), -- low cardinality
    amount         NUMBER
);

-- B-tree on high-cardinality, often joined columns
CREATE INDEX idx_sales_customer_id ON sales (customer_id);
CREATE INDEX idx_sales_product_id  ON sales (product_id);

-- Bitmap on low-cardinality, filter columns
CREATE BITMAP INDEX idx_sales_region  ON sales (region_code);
CREATE BITMAP INDEX idx_sales_channel ON sales (channel);

-- Input query
SELECT region_code, channel, SUM(amount) total_amount
FROM   sales
WHERE  region_code IN ('NA', 'EU')
AND    channel = 'ONLINE'
AND    sale_date BETWEEN DATE '2024-01-01' AND DATE '2024-12-31'
GROUP BY region_code, channel;

-- Expected behavior
-- Oracle can:
-- - Use bitmap indexes on REGION_CODE and CHANNEL
-- - Combine bitmaps (IN + AND)
-- - Possibly use B-tree on SALE_DATE if indexed
-- - Aggregate matching rows
```

---

**Key idea:**  
- **B-tree**: high-cardinality, OLTP, frequent DML, range lookups.  
- **Bitmap**: low-cardinality, data warehouse, read-mostly, complex combined filters.
