
Here’s a concise summary of the main Oracle indexing strategies, with simple examples.

---

## 1. B-tree Index
**Use for:**  
- Columns with many distinct values (high cardinality)  
- OLTP systems (frequent inserts/updates)  

**Example:**
```sql
-- Create table
CREATE TABLE employees (
    emp_id      NUMBER PRIMARY KEY,
    last_name   VARCHAR2(50),
    dept_id     NUMBER,
    salary      NUMBER
);

-- B-tree index on last_name
CREATE INDEX idx_emp_lastname ON employees(last_name);
```

**Query using the index:**
```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE last_name = 'SMITH';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Expected behavior:**  
Oracle uses `IDX_EMP_LASTNAME` to quickly find rows where `last_name = 'SMITH'`.

---

## 2. Bitmap Index
**Use for:**  
- Columns with few distinct values (low cardinality), e.g., gender, status  
- Data warehouse / read-mostly environments  
- Not good for heavy concurrent DML (insert/update/delete)

**Example:**
```sql
-- Column with low cardinality
ALTER TABLE employees ADD (status CHAR(1));  -- 'A' = active, 'I' = inactive

-- Bitmap index
CREATE BITMAP INDEX idx_emp_status ON employees(status);
```

**Query using the index:**
```sql
EXPLAIN PLAN FOR
SELECT COUNT(*) FROM employees WHERE status = 'A';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Expected behavior:**  
Oracle uses bitmap operations to quickly count/filter rows by `status`.

---

## 3. Function-Based Index
**Use for:**  
- Queries that use expressions or functions on columns  
- Avoids full scans when predicates use functions

**Example:**
```sql
-- Function-based index on UPPER(last_name)
CREATE INDEX idx_emp_upper_lastname
    ON employees(UPPER(last_name));
```

**Query using the index:**
```sql
EXPLAIN PLAN FOR
SELECT * FROM employees
WHERE UPPER(last_name) = 'SMITH';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Expected behavior:**  
Oracle can use `IDX_EMP_UPPER_LASTNAME` because the query’s predicate matches the indexed expression.

---

## 4. Unique Index
**Use for:**  
- Enforcing uniqueness of column(s)  
- Often created automatically by PRIMARY KEY / UNIQUE constraints  

**Example:**
```sql
-- Unique index on email
ALTER TABLE employees ADD (email VARCHAR2(100));

CREATE UNIQUE INDEX idx_emp_email
    ON employees(email);
```

**Test input/output:**
```sql
INSERT INTO employees (emp_id, last_name, email)
VALUES (1, 'SMITH', 'smith@example.com');  -- succeeds

INSERT INTO employees (emp_id, last_name, email)
VALUES (2, 'JONES', 'smith@example.com');  -- ORA-00001: unique constraint violated
```

**Expected behavior:**  
Second insert fails because `email` must be unique.

---

## 5. Composite (Multi-Column) Index
**Use for:**  
- Queries that filter/sort on multiple columns together  
- Order of columns in the index matters  

**Example:**
```sql
-- Composite B-tree index on (dept_id, last_name)
CREATE INDEX idx_emp_dept_lastname
    ON employees(dept_id, last_name);
```

**Queries that can use this index efficiently:**
```sql
-- Good: leading column dept_id is used
SELECT * FROM employees
WHERE dept_id = 10 AND last_name = 'SMITH';

-- Good: only leading column
SELECT * FROM employees
WHERE dept_id = 10;

-- Not ideal: skips leading column
SELECT * FROM employees
WHERE last_name = 'SMITH';  -- may not use this index
```

**Expected behavior:**  
Oracle can efficiently use the index when the predicate starts with `dept_id`.

---

### Summary of When to Use What

- **B-tree:** High-cardinality columns, OLTP, equality and range searches.  
- **Bitmap:** Low-cardinality columns, data warehouse, complex AND/OR conditions.  
- **Function-based:** When queries use functions/expressions in WHERE/ORDER BY.  
- **Unique:** To enforce uniqueness of a column or set of columns.  
- **Composite:** When queries commonly filter/sort by the same set of columns in a specific order.
