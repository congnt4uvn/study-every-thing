
Here’s a concise summary of the key ideas around `CONNECT BY`, `START WITH`, and `LEVEL` in Oracle, with code examples and sample outputs.

---

## 1. Purpose

These features are used for **hierarchical queries** in Oracle, such as:

- Organization charts (manager → employee)
- Category trees (parent category → subcategory)
- Bill of materials (assembly → parts)

---

## 2. Basic Syntax

```sql
SELECT
    LEVEL,
    column_list
FROM
    table_name
START WITH
    root_condition
CONNECT BY
    prior parent_column = child_column;
```

- `START WITH`: defines the **root rows** of the hierarchy.
- `CONNECT BY`: defines the **parent-child relationship**.
- `PRIOR`: indicates which side is the parent.
- `LEVEL`: pseudo-column indicating the **depth** in the hierarchy (root = 1).

---

## 3. Example Table

```sql
CREATE TABLE employees (
    emp_id      NUMBER PRIMARY KEY,
    emp_name    VARCHAR2(50),
    manager_id  NUMBER
);

INSERT INTO employees VALUES (1, 'CEO',        NULL);
INSERT INTO employees VALUES (2, 'VP Sales',   1);
INSERT INTO employees VALUES (3, 'VP Tech',    1);
INSERT INTO employees VALUES (4, 'Sales Rep',  2);
INSERT INTO employees VALUES (5, 'Engineer',   3);
COMMIT;
```

---

## 4. Simple Hierarchical Query

### Query

```sql
SELECT
    LEVEL          AS lvl,
    emp_id,
    emp_name,
    manager_id
FROM
    employees
START WITH
    manager_id IS NULL          -- root: CEO
CONNECT BY
    PRIOR emp_id = manager_id   -- parent emp_id = child manager_id
ORDER BY
    LEVEL, emp_id;
```

### Sample Output

| LVL | EMP_ID | EMP_NAME   | MANAGER_ID |
|-----|--------|------------|------------|
| 1   | 1      | CEO        | (null)     |
| 2   | 2      | VP Sales   | 1          |
| 2   | 3      | VP Tech    | 1          |
| 3   | 4      | Sales Rep  | 2          |
| 3   | 5      | Engineer   | 3          |

---

## 5. Indented Tree Display

Use `LPAD` with `LEVEL` to show hierarchy visually.

### Query

```sql
SELECT
    LPAD(' ', 2 * (LEVEL - 1)) || emp_name AS emp_tree,
    emp_id,
    manager_id
FROM
    employees
START WITH
    manager_id IS NULL
CONNECT BY
    PRIOR emp_id = manager_id;
```

### Sample Output

| EMP_TREE      | EMP_ID | MANAGER_ID |
|---------------|--------|------------|
| CEO           | 1      | (null)     |
|   VP Sales    | 2      | 1          |
|     Sales Rep | 4      | 2          |
|   VP Tech     | 3      | 1          |
|     Engineer  | 5      | 3          |

---

## 6. Using LEVEL in Conditions

You can filter by depth:

```sql
-- Only first 2 levels (e.g., CEO and direct reports)
SELECT emp_name, LEVEL
FROM employees
START WITH manager_id IS NULL
CONNECT BY PRIOR emp_id = manager_id
AND LEVEL <= 2;
```

---

## 7. Reverse Direction (Bottom-Up)

You can also go from child to parent by reversing `PRIOR`:

```sql
-- Find the chain of managers for employee 4 (Sales Rep)
SELECT
    LEVEL,
    emp_id,
    emp_name,
    manager_id
FROM
    employees
START WITH
    emp_id = 4
CONNECT BY
    PRIOR manager_id = emp_id;  -- now parent is found via manager_id
```

Sample Output:

| LEVEL | EMP_ID | EMP_NAME   | MANAGER_ID |
|-------|--------|------------|------------|
| 1     | 4      | Sales Rep  | 2          |
| 2     | 2      | VP Sales   | 1          |
| 3     | 1      | CEO        | (null)     |

---

## 8. Avoiding Cycles

If data can contain loops, use `NOCYCLE`:

```sql
SELECT
    LEVEL,
    emp_id,
    emp_name
FROM
    employees
START WITH
    manager_id IS NULL
CONNECT BY NOCYCLE
    PRIOR emp_id = manager_id;
```

---

## 9. Key Points

- `START WITH` defines **where the hierarchy begins**.
- `CONNECT BY` defines **how rows are related** (parent-child).
- `PRIOR` marks the **parent side** of the relationship.
- `LEVEL` gives the **depth** and can be used for:
  - Indentation
  - Filtering (e.g., `LEVEL <= 3`)
- Useful for any **tree-like data** stored in a single table.
