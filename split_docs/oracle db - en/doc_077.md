
Here’s a concise summary of “Locking & Isolation: Table locks” in Oracle, with examples.

---

## 1. What are table locks?

- Oracle uses **table locks** to control concurrent access to an entire table.
- They prevent conflicting operations (e.g., two sessions trying to change the structure or data in incompatible ways).
- Table locks are **automatically acquired** by Oracle, but you can also **request them explicitly**.

---

## 2. Types of table locks (common ones)

Oracle uses several lock modes; key ones for application developers:

- **ROW SHARE (RS)**  
  - Allows concurrent DML (INSERT/UPDATE/DELETE) by others.  
  - Prevents exclusive locks that would block your DML.

- **ROW EXCLUSIVE (RX)**  
  - Acquired automatically by DML (INSERT/UPDATE/DELETE).  
  - Allows other sessions to read and also do DML, but blocks some stronger locks.

- **SHARE (S)**  
  - Allows concurrent queries.  
  - Blocks DML that requires stronger locks.

- **SHARE ROW EXCLUSIVE (SRX)**  
  - Allows queries.  
  - Restricts other DML and some lock modes.

- **EXCLUSIVE (X)**  
  - Only the holder can read and write the table.  
  - Blocks all other access that needs locks.

---

## 3. How Oracle uses locks with isolation

- Default isolation level: **READ COMMITTED**.
- Oracle uses **row-level locks** for DML and **multiversion read consistency** for queries.
- Table locks are usually:
  - **Lightweight** and compatible with many operations.
  - Used to protect **metadata** or to coordinate certain operations (e.g., DDL).

---

## 4. Explicit table locking: `LOCK TABLE`

You can explicitly lock a table to control concurrency.

### Example 1: ROW SHARE lock

**Session 1**

```sql
LOCK TABLE employees IN ROW SHARE MODE;
-- Now Session 1 holds a ROW SHARE lock on EMPLOYEES
```

- Other sessions can still:
  - SELECT from `employees`
  - INSERT/UPDATE/DELETE rows in `employees`
- But some stronger locks (like EXCLUSIVE) may be blocked.

---

### Example 2: EXCLUSIVE lock

**Session 1**

```sql
LOCK TABLE employees IN EXCLUSIVE MODE;
-- Session 1 has exclusive access to the table
```

**Session 2**

```sql
UPDATE employees
SET salary = salary * 1.1
WHERE department_id = 10;
```

**Result (Session 2):**

- The `UPDATE` will **wait** (block) until Session 1:
  - Commits or rolls back (releasing the lock), or
  - The waiting times out (if configured).

---

### Example 3: SHARE lock with SELECT

**Session 1**

```sql
LOCK TABLE employees IN SHARE MODE;
-- Session 1 can query; DML from others may be restricted
SELECT * FROM employees WHERE department_id = 20;
```

**Session 2**

```sql
INSERT INTO employees (employee_id, first_name, last_name, department_id)
VALUES (300, 'John', 'Smith', 20);
```

**Result (Session 2):**

- Depending on the exact lock compatibility, the INSERT may:
  - Wait until Session 1 releases the SHARE lock, or
  - Be blocked by Oracle with an error if incompatible.

---

## 5. Viewing locks

You can inspect locks using data dictionary views.

```sql
SELECT
    l.session_id,
    l.locked_mode,
    o.object_name,
    o.object_type
FROM
    v$locked_object l
    JOIN dba_objects o ON l.object_id = o.object_id;
```

**Output (example):**

| SESSION_ID | LOCKED_MODE | OBJECT_NAME | OBJECT_TYPE |
|-----------:|------------:|------------|------------|
|       123  |           6 | EMPLOYEES  | TABLE      |

- `LOCKED_MODE` is a numeric code for the lock mode (e.g., 6 might represent EXCLUSIVE).

---

## 6. Key ideas to remember

- Oracle prefers **row-level locks**; table locks are used when necessary.
- Table locks help maintain **consistency** and **prevent conflicts**.
- You can use `LOCK TABLE` to:
  - Ensure no one else modifies a table while you perform a critical operation.
  - Coordinate application logic that must be serialized.
- Always balance explicit locks with the risk of **blocking** other sessions.
