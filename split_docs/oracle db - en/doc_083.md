
Here are the key ideas for designing Oracle code to minimize locking issues, plus some example patterns.

---

## 1. Keep Transactions Short

- Do only what is necessary between `BEGIN` and `COMMIT`/`ROLLBACK`.
- Avoid user interaction (e.g., waiting for input) inside a transaction.
- The longer a transaction runs, the longer it holds locks.

**Bad pattern:**
```sql
-- Pseudo-code
BEGIN
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  -- wait for user confirmation in the app...
  -- user takes 30 seconds
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
  COMMIT;
END;
```

**Better pattern:**
```sql
-- Do all user interaction first, then run a short transaction
BEGIN
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
  COMMIT;
END;
```

---

## 2. Access Objects in a Consistent Order

- When multiple sessions update the same set of tables/rows, always access them in the same order to avoid deadlocks.

**Example:**
- Rule: Always update `account` with smaller `id` first.

```sql
-- Session 1
BEGIN
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
  COMMIT;
END;
```

```sql
-- Session 2 (same order)
BEGIN
  UPDATE accounts SET balance = balance - 50 WHERE id = 1;
  UPDATE accounts SET balance = balance + 50 WHERE id = 2;
  COMMIT;
END;
```

If Session 2 updated `id = 2` first and Session 1 updated `id = 1` first, they could deadlock.

---

## 3. Use Appropriate Isolation and Locking

- Oracle’s default isolation level is `READ COMMITTED`, which is usually good for OLTP.
- Avoid unnecessary explicit locks like `SELECT ... FOR UPDATE` unless you really need them.

**Example: SELECT FOR UPDATE**
```sql
-- Session 1
SELECT balance
FROM accounts
WHERE id = 1
FOR UPDATE;  -- locks this row

-- Later
UPDATE accounts
SET balance = balance - 100
WHERE id = 1;
COMMIT;
```

**Session 2 (input/output example):**
```sql
-- Session 2 tries to lock the same row
SELECT balance
FROM accounts
WHERE id = 1
FOR UPDATE;
```
**Output:** Session 2 will **wait** until Session 1 commits or rolls back.

To avoid long waits, use `NOWAIT` or `WAIT n`:

```sql
SELECT balance
FROM accounts
WHERE id = 1
FOR UPDATE NOWAIT;
```

If the row is locked, Oracle immediately returns:
```text
ORA-00054: resource busy and acquire with NOWAIT specified or timeout expired
```

---

## 4. Commit Frequently but Safely

- Commit after logical units of work, not after every single row.
- Too many commits can hurt performance, but too few can cause long locks.

**Example: batch processing with periodic commits**
```plsql
DECLARE
  v_count NUMBER := 0;
BEGIN
  FOR r IN (SELECT id FROM orders WHERE status = 'NEW') LOOP
    UPDATE orders
    SET status = 'PROCESSED'
    WHERE id = r.id;

    v_count := v_count + 1;

    IF v_count >= 1000 THEN
      COMMIT;       -- release locks every 1000 rows
      v_count := 0;
    END IF;
  END LOOP;

  COMMIT;
END;
/
```

---

## 5. Avoid “Lock Escalation” Patterns

- Don’t scan and update large sets of rows without proper indexing and predicates.
- Poor queries can cause many row locks and long-running transactions.

**Bad pattern:**
```sql
UPDATE orders
SET status = 'CLOSED'
WHERE status = 'OPEN';  -- may touch millions of rows
```

**Better pattern (chunking):**
```plsql
DECLARE
  CURSOR c_orders IS
    SELECT id
    FROM orders
    WHERE status = 'OPEN'
    AND ROWNUM <= 1000;  -- process in chunks
BEGIN
  LOOP
    FOR r IN c_orders LOOP
      UPDATE orders
      SET status = 'CLOSED'
      WHERE id = r.id;
    END LOOP;

    EXIT WHEN SQL%ROWCOUNT = 0;
    COMMIT;  -- release locks for this chunk
  END LOOP;
END;
/
```

---

## 6. Use Optimistic Concurrency Where Possible

- Instead of locking early, check for changes at update time using a version column or timestamp.

**Table:**
```sql
CREATE TABLE products (
  id          NUMBER PRIMARY KEY,
  name        VARCHAR2(100),
  price       NUMBER,
  version_no  NUMBER DEFAULT 0
);
```

**Read (no lock):**
```sql
SELECT id, name, price, version_no
FROM products
WHERE id = 10;
```

**Update with optimistic check (input/output example):**
```sql
-- Input: user sends id=10, new_price=120, version_no=3

UPDATE products
SET price      = 120,
    version_no = version_no + 1
WHERE id = 10
  AND version_no = 3;
```

**Output:**
- If `SQL%ROWCOUNT = 1`: update succeeded, no conflict.
- If `SQL%ROWCOUNT = 0`: someone else changed the row; handle conflict in the app.

---

## 7. Avoid Unnecessary Locking in Application Logic

- Don’t hold database connections open with uncommitted work while doing long operations in the middle tier.
- Use connection pooling correctly: return connections to the pool only after commit/rollback.

---

## 8. Monitor and Diagnose Locks

Use Oracle views to see locks and blocking sessions:

```sql
-- See current locks
SELECT s.sid, s.serial#, l.type, l.id1, l.id2, l.lmode, l.request
FROM   v$lock l
JOIN   v$session s ON l.sid = s.sid;

-- See blocking sessions
SELECT s1.sid AS blocker_sid,
       s2.sid AS waiter_sid
FROM   v$lock l1
JOIN   v$lock l2 ON l1.id1 = l2.id1 AND l1.id2 = l2.id2
JOIN   v$session s1 ON l1.sid = s1.sid
JOIN   v$session s2 ON l2.sid = s2.sid
WHERE  l1.block = 1
AND    l2.request > 0;
```

---

### Summary

To minimize locking issues in Oracle:

1. Keep transactions short and focused.
2. Access tables/rows in a consistent order.
3. Use `SELECT ... FOR UPDATE` only when needed, with `NOWAIT`/`WAIT` if appropriate.
4. Commit at logical boundaries; avoid very long transactions.
5. Process large updates in chunks.
6. Use optimistic concurrency (version columns) where possible.
7. Design application logic to avoid holding locks while doing non-database work.
8. Monitor locks and blocking sessions using Oracle’s dynamic views.
