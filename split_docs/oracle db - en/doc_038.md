
Here’s a concise summary and some practical examples around **Advanced PL/SQL: Global variables**.

---

## Key Ideas

1. **PL/SQL has no true “global variables” shared across all sessions**  
   Each database session has its own memory (PGA), so normal PL/SQL variables are local to:
   - A block
   - A package
   - A procedure/function

2. **Package variables act as “session‑global” variables**  
   - Variables declared in a package specification are visible to any PL/SQL code in the same session that uses the package.
   - Their values persist for the duration of the session.
   - They are not shared between sessions.

3. **Database‑global state is usually stored in tables**  
   - If you need values shared across all sessions, you typically store them in:
     - Tables (configuration, parameters, counters)
     - Contexts (via `DBMS_SESSION.SET_CONTEXT`)
   - This makes the data persistent and visible to all sessions.

4. **Initialization and encapsulation**  
   - Use package initialization sections to set default values for “global” (package) variables.
   - Provide getter/setter procedures/functions to control access and validation.

---

## Example 1: Session‑Global Variables via Package

### Package Specification

```sql
CREATE OR REPLACE PACKAGE app_state IS
  -- Session-global variables
  g_user_id      NUMBER;
  g_request_id   NUMBER;
  g_debug_mode   BOOLEAN := FALSE;

  -- Accessor procedures/functions
  PROCEDURE set_user(p_user_id IN NUMBER);
  PROCEDURE set_request(p_request_id IN NUMBER);
  PROCEDURE set_debug(p_debug IN BOOLEAN);

  FUNCTION get_user RETURN NUMBER;
  FUNCTION get_request RETURN NUMBER;
  FUNCTION is_debug RETURN BOOLEAN;
END app_state;
/
```

### Package Body

```sql
CREATE OR REPLACE PACKAGE BODY app_state IS

  PROCEDURE set_user(p_user_id IN NUMBER) IS
  BEGIN
    g_user_id := p_user_id;
  END;

  PROCEDURE set_request(p_request_id IN NUMBER) IS
  BEGIN
    g_request_id := p_request_id;
  END;

  PROCEDURE set_debug(p_debug IN BOOLEAN) IS
  BEGIN
    g_debug_mode := p_debug;
  END;

  FUNCTION get_user RETURN NUMBER IS
  BEGIN
    RETURN g_user_id;
  END;

  FUNCTION get_request RETURN NUMBER IS
  BEGIN
    RETURN g_request_id;
  END;

  FUNCTION is_debug RETURN BOOLEAN IS
  BEGIN
    RETURN g_debug_mode;
  END;

BEGIN
  -- Package initialization (runs once per session, on first reference)
  g_debug_mode := FALSE;
END app_state;
/
```

### Usage Example (Session 1)

```sql
-- Input
BEGIN
  app_state.set_user(101);
  app_state.set_request(5001);
  app_state.set_debug(TRUE);

  DBMS_OUTPUT.PUT_LINE('User: '    || app_state.get_user);
  DBMS_OUTPUT.PUT_LINE('Request: ' || app_state.get_request);
  DBMS_OUTPUT.PUT_LINE('Debug: '   || CASE WHEN app_state.is_debug THEN 'ON' ELSE 'OFF' END);
END;
/
```

**Output (Session 1)**

```text
User: 101
Request: 5001
Debug: ON
```

### Usage Example (Session 2 – separate session)

```sql
-- Input (new session, same code)
BEGIN
  DBMS_OUTPUT.PUT_LINE('User: '    || NVL(TO_CHAR(app_state.get_user), 'NULL'));
  DBMS_OUTPUT.PUT_LINE('Request: ' || NVL(TO_CHAR(app_state.get_request), 'NULL'));
END;
/
```

**Output (Session 2)**

```text
User: NULL
Request: NULL
```

This shows that package variables are **per session**, not shared.

## Example 2: Database‑Global Configuration via Table

```sql
CREATE TABLE app_config (
  config_name  VARCHAR2(30) PRIMARY KEY,
  config_value VARCHAR2(4000)
);

INSERT INTO app_config (config_name, config_value)
VALUES ('MAX_LOGIN_ATTEMPTS', '5');

COMMIT;
```

### Access via PL/SQL

```sql
CREATE OR REPLACE FUNCTION get_config(p_name IN VARCHAR2)
  RETURN VARCHAR2
IS
  l_value app_config.config_value%TYPE;
BEGIN
  SELECT config_value
    INTO l_value
    FROM app_config
   WHERE config_name = p_name;

  RETURN l_value;
EXCEPTION
  WHEN NO_DATA_FOUND THEN
    RETURN NULL;
END;
/
```

**Usage**

```sql
-- Input
DECLARE
  l_max_attempts NUMBER;
BEGIN
  l_max_attempts := TO_NUMBER(get_config('MAX_LOGIN_ATTEMPTS'));
  DBMS_OUTPUT.PUT_LINE('Max login attempts: ' || l_max_attempts);
END;
/
```

**Output**

```text
Max login attempts: 5
```

This value is **shared across all sessions** because it’s stored in a table.

## Example 3: Using Application Context (Logical Global Variables)

```sql
-- Create a context namespace
CREATE CONTEXT app_ctx USING set_app_ctx;
/

-- Package to set context values
CREATE OR REPLACE PACKAGE set_app_ctx IS
  PROCEDURE set_user(p_user_id IN NUMBER);
END set_app_ctx;
/

CREATE OR REPLACE PACKAGE BODY set_app_ctx IS
  PROCEDURE set_user(p_user_id IN NUMBER) IS
  BEGIN
    DBMS_SESSION.SET_CONTEXT(
      namespace => 'APP_CTX',
      attribute => 'USER_ID',
      value     => TO_CHAR(p_user_id)
    );
  END;
END set_app_ctx;
/
```

**Usage**

```sql
-- Input
BEGIN
  set_app_ctx.set_user(200);
  DBMS_OUTPUT.PUT_LINE('User from context: ' ||
    SYS_CONTEXT('APP_CTX', 'USER_ID'));
END;
/
```

**Output**

```text
User from context: 200
```

Context values are also **per session**, but accessible from SQL and PL/SQL.

---

## Summary

- Use **package variables** for session‑global state in PL/SQL.
- Use **tables** (and sometimes application contexts) for data that must be shared across sessions and persist.
- Encapsulate “global” state with packages (getters/setters, initialization) to keep code maintainable and safe.
