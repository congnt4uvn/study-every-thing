
Here’s a concise summary of Oracle Database **partitioning** in the context of **physical design**, plus some code examples.

---

## Key Ideas of Partitioning

1. **What is Partitioning?**  
   - Splitting a large table (and its indexes) into smaller, more manageable pieces called **partitions**.  
   - Logically one table, physically many segments.

2. **Why Use Partitioning?**
   - **Performance**: Queries that access only some partitions can skip others (partition pruning).  
   - **Manageability**: Easier to manage large tables (backup, move, compress, archive per partition).  
   - **Availability**: Operations on one partition don’t lock the entire table (e.g., loading or rebuilding a partition).

3. **Common Partitioning Methods**
   - **Range partitioning**: Based on a range of values (e.g., dates, numeric ranges).  
   - **List partitioning**: Based on a list of discrete values (e.g., region codes).  
   - **Hash partitioning**: Evenly distributes rows using a hash function (good for load balancing).  
   - **Composite partitioning**: Combination (e.g., range-hash, range-list).

4. **Design Considerations**
   - Choose partition key that matches common query predicates (e.g., `WHERE order_date BETWEEN ...`).  
   - Keep partitions at a manageable size (not too many, not too few).  
   - Align local indexes with partitions for easier maintenance.  
   - Plan lifecycle: how to add/drop/split/merge partitions over time.

## Code Examples

### 1. Range Partitioned Table (by Date)

**Create table:**

```sql
CREATE TABLE sales (
    sale_id      NUMBER,
    sale_date    DATE,
    amount       NUMBER,
    region       VARCHAR2(10)
)
PARTITION BY RANGE (sale_date) (
    PARTITION p_2023_q1 VALUES LESS THAN (TO_DATE('2023-04-01','YYYY-MM-DD')),
    PARTITION p_2023_q2 VALUES LESS THAN (TO_DATE('2023-07-01','YYYY-MM-DD')),
    PARTITION p_2023_q3 VALUES LESS THAN (TO_DATE('2023-10-01','YYYY-MM-DD')),
    PARTITION p_2023_q4 VALUES LESS THAN (TO_DATE('2024-01-01','YYYY-MM-DD')),
    PARTITION p_max     VALUES LESS THAN (MAXVALUE)
);
```

**Insert sample data:**

```sql
INSERT INTO sales (sale_id, sale_date, amount, region)
VALUES (1, DATE '2023-02-15', 100, 'US');

INSERT INTO sales (sale_id, sale_date, amount, region)
VALUES (2, DATE '2023-05-10', 200, 'EU');
COMMIT;
```

**Check which partition rows are in:**

```sql
SELECT sale_id, sale_date, amount, region,
       PARTITION_NAME
FROM   (
  SELECT s.*, 
         SUBSTR(DBMS_ROWID.ROWID_OBJECT(ROWID),1,10) AS obj_id
  FROM   sales s
) s
JOIN   user_tab_partitions p
ON     p.table_name = 'SALES'
AND    p.partition_position = 
       (SELECT partition_position
        FROM   user_tab_partitions
        WHERE  table_name = 'SALES'
        AND    partition_name = p.partition_name);
```

(Exact mapping of row to partition is usually checked via `EXPLAIN PLAN` or `DBA_EXTENTS`; above is illustrative.)

**Query using partition pruning (input → output):**

```sql
-- Input query:
SELECT SUM(amount) AS total_q2
FROM   sales
WHERE  sale_date BETWEEN DATE '2023-04-01' AND DATE '2023-06-30';
```

**Expected behavior (output conceptually):**

- Oracle will only scan partition `p_2023_q2` (partition pruning).
- Example result:

```text
TOTAL_Q2
--------
     200
```

### 2. List Partitioned Table (by Region)

```sql
CREATE TABLE customers (
    cust_id   NUMBER,
    name      VARCHAR2(100),
    region    VARCHAR2(10)
)
PARTITION BY LIST (region) (
    PARTITION p_us VALUES ('US'),
    PARTITION p_eu VALUES ('EU', 'UK'),
    PARTITION p_apac VALUES ('APAC', 'AU', 'NZ'),
    PARTITION p_other VALUES (DEFAULT)
);
```

**Insert and query:**

```sql
INSERT INTO customers VALUES (1, 'Alice', 'US');
INSERT INTO customers VALUES (2, 'Bob',   'EU');
COMMIT;

SELECT * FROM customers WHERE region = 'US';
```

**Output:**

```text
CUST_ID  NAME   REGION
-------  -----  ------
1        Alice  US
```

Oracle will only scan partition `P_US`.

### 3. Hash Partitioned Table

```sql
CREATE TABLE log_data (
    log_id    NUMBER,
    log_time  DATE,
    message   VARCHAR2(4000)
)
PARTITION BY HASH (log_id)
PARTITIONS 4;
```

- Rows are distributed across 4 partitions to balance I/O and storage.

---

### 4. Composite Range-Hash Partitioning

```sql
CREATE TABLE orders (
    order_id    NUMBER,
    order_date  DATE,
    customer_id NUMBER,
    amount      NUMBER
)
PARTITION BY RANGE (order_date)
SUBPARTITION BY HASH (customer_id)
SUBPARTITIONS 4
(
    PARTITION p_2023_q1 VALUES LESS THAN (DATE '2023-04-01'),
    PARTITION p_2023_q2 VALUES LESS THAN (DATE '2023-07-01'),
    PARTITION p_2023_q3 VALUES LESS THAN (DATE '2023-10-01'),
    PARTITION p_2023_q4 VALUES LESS THAN (DATE '2024-01-01')
);
```

- First partitioned by time (range), then each partition is hash-subpartitioned by `customer_id`.

---

### 5. Managing Partitions

**Add a new partition:**

```sql
ALTER TABLE sales
ADD PARTITION p_2024_q1
VALUES LESS THAN (DATE '2024-04-01');
```

**Drop an old partition (e.g., archive and then drop):**

```sql
ALTER TABLE sales
DROP PARTITION p_2023_q1;
```

**Split a partition:**

```sql
ALTER TABLE sales
SPLIT PARTITION p_2023_q4 AT (DATE '2023-11-01')
INTO (
    PARTITION p_2023_q4a,
    PARTITION p_2023_q4b
);
```
