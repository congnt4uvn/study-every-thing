
Here’s a concise summary of “Advanced PL/SQL: Encapsulation” with examples.

---

## Key Ideas

1. **Encapsulation in PL/SQL**  
   - Encapsulation = grouping related data and behavior together and hiding internal details.  
   - In PL/SQL, this is mainly done with:
     - **Packages**
     - **Object types**
     - **Local (nested) procedures/functions**

2. **Packages as Encapsulation Units**  
   - A package has:
     - **Specification**: public interface (what is visible to other code).
     - **Body**: implementation (can contain private elements).
   - You expose only what callers need; keep helper logic private.

3. **Information Hiding**  
   - Use the package spec for public procedures/functions/types/constants.
   - Put internal variables, cursors, and helper subprograms in the package body only.
   - This allows you to change implementation without affecting callers.

4. **Modularity and Maintainability**  
   - Encapsulation makes code:
     - Easier to understand (clear interface).
     - Easier to maintain (change internals without breaking callers).
     - More secure (prevent direct access to sensitive data/logic).

5. **Object Types for Encapsulation**  
   - PL/SQL object types can encapsulate attributes (data) and methods (procedures/functions).
   - Similar to classes in OO languages.

## Example 1: Package Encapsulation

### Package Specification (Public Interface)

```sql
CREATE OR REPLACE PACKAGE employee_api AS
  -- Public type
  TYPE t_employee_rec IS RECORD (
    emp_id    employees.employee_id%TYPE,
    full_name VARCHAR2(200),
    salary    employees.salary%TYPE
  );

  -- Public procedure
  PROCEDURE give_raise (
    p_emp_id   IN employees.employee_id%TYPE,
    p_percent  IN NUMBER
  );

  -- Public function
  FUNCTION get_employee (
    p_emp_id IN employees.employee_id%TYPE
  ) RETURN t_employee_rec;
END employee_api;
/
```

### Package Body (Implementation + Private Details)

```sql
CREATE OR REPLACE PACKAGE BODY employee_api AS

  -- Private constant (not visible outside)
  c_max_raise_percent CONSTANT NUMBER := 20;

  -- Private helper procedure
  PROCEDURE log_change (
    p_emp_id IN employees.employee_id%TYPE,
    p_old_sal IN employees.salary%TYPE,
    p_new_sal IN employees.salary%TYPE
  ) IS
  BEGIN
    INSERT INTO salary_log (emp_id, old_salary, new_salary, change_date)
    VALUES (p_emp_id, p_old_sal, p_new_sal, SYSDATE);
  END log_change;

  PROCEDURE give_raise (
    p_emp_id   IN employees.employee_id%TYPE,
    p_percent  IN NUMBER
  ) IS
    v_old_sal employees.salary%TYPE;
    v_new_sal employees.salary%TYPE;
  BEGIN
    IF p_percent > c_max_raise_percent THEN
      RAISE_APPLICATION_ERROR(-20001, 'Raise too high');
    END IF;

    SELECT salary INTO v_old_sal
    FROM employees
    WHERE employee_id = p_emp_id
    FOR UPDATE;

    v_new_sal := v_old_sal * (1 + p_percent / 100);

    UPDATE employees
    SET salary = v_new_sal
    WHERE employee_id = p_emp_id;

    log_change(p_emp_id, v_old_sal, v_new_sal); -- private helper
  END give_raise;

  FUNCTION get_employee (
    p_emp_id IN employees.employee_id%TYPE
  ) RETURN t_employee_rec IS
    v_emp t_employee_rec;
  BEGIN
    SELECT employee_id,
           first_name || ' ' || last_name,
           salary
    INTO v_emp.emp_id, v_emp.full_name, v_emp.salary
    FROM employees
    WHERE employee_id = p_emp_id;

    RETURN v_emp;
  END get_employee;

END employee_api;
/
```

### Example Usage (Input/Output)

```sql
-- Input: give employee 100 a 10% raise
BEGIN
  employee_api.give_raise(100, 10);
END;
/

-- Input: get employee 100 data
DECLARE
  v_emp employee_api.t_employee_rec;
BEGIN
  v_emp := employee_api.get_employee(100);
  DBMS_OUTPUT.PUT_LINE('ID: ' || v_emp.emp_id);
  DBMS_OUTPUT.PUT_LINE('Name: ' || v_emp.full_name);
  DBMS_OUTPUT.PUT_LINE('Salary: ' || v_emp.salary);
END;
/
```

**Output (example):**
```
ID: 100
Name: Steven King
Salary: 26400
```

The caller cannot see or call `log_change` or access `c_max_raise_percent` directly; they are encapsulated inside the package body.

## Example 2: Encapsulation with Object Types

```sql
CREATE OR REPLACE TYPE account_t AS OBJECT (
  account_id   NUMBER,
  balance      NUMBER,

  MEMBER PROCEDURE deposit(p_amount IN NUMBER),
  MEMBER PROCEDURE withdraw(p_amount IN NUMBER),
  MEMBER FUNCTION get_balance RETURN NUMBER
);
/

CREATE OR REPLACE TYPE BODY account_t AS

  MEMBER PROCEDURE deposit(p_amount IN NUMBER) IS
  BEGIN
    IF p_amount <= 0 THEN
      RAISE_APPLICATION_ERROR(-20010, 'Amount must be positive');
    END IF;
    SELF.balance := SELF.balance + p_amount;
  END deposit;

  MEMBER PROCEDURE withdraw(p_amount IN NUMBER) IS
  BEGIN
    IF p_amount <= 0 THEN
      RAISE_APPLICATION_ERROR(-20011, 'Amount must be positive');
    ELSIF p_amount > SELF.balance THEN
      RAISE_APPLICATION_ERROR(-20012, 'Insufficient funds');
    END IF;
    SELF.balance := SELF.balance - p_amount;
  END withdraw;

  MEMBER FUNCTION get_balance RETURN NUMBER IS
  BEGIN
    RETURN SELF.balance;
  END get_balance;

END;
/
```

### Usage (Input/Output)

```sql
DECLARE
  v_acc account_t := account_t(1, 1000); -- id=1, balance=1000
BEGIN
  v_acc.deposit(200);      -- balance = 1200
  v_acc.withdraw(150);     -- balance = 1050

  DBMS_OUTPUT.PUT_LINE('Balance: ' || v_acc.get_balance);
END;
/
```

**Output:**
```
Balance: 1050
```

The internal rule “no negative or overdrawn withdrawals” is encapsulated inside the object methods.

---

**In summary:**  
Encapsulation in advanced PL/SQL is about using packages, object types, and local subprograms to expose a clean, stable interface while hiding implementation details, improving modularity, security, and maintainability.
