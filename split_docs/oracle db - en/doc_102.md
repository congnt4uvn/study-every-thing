## Key Ideas: Version Control & CI/CD for Oracle Database Code

1. **Store all database code in Git**
   - Include: `CREATE/ALTER` scripts for tables, views, packages, procedures, functions, triggers, types, synonyms, grants, etc.
   - Treat database code like application code: commit, branch, merge, review (PRs), tag releases.

2. **Use migration scripts instead of manual changes**
   - Every change to the database is a script checked into Git.
   - Scripts are applied in order (e.g., numbered or timestamped).
   - This creates a reproducible history of schema changes.

3. **Automate deployment with CI/CD**
   - CI: On each commit, run:
     - Syntax checks (e.g., compile PL/SQL in a test DB).
     - Unit tests (e.g., utPLSQL).
   - CD: On approved changes, automatically deploy scripts to:
     - Dev → Test → Staging → Production
   - Use tools like Jenkins, GitLab CI, GitHub Actions, Azure DevOps, etc.

4. **Benefits**
   - Traceability: Who changed what, when, and why.
   - Rollback: Easier to revert to a previous version.
   - Consistency: Same scripts run in all environments.
   - Collaboration: Code reviews and branching strategies.

### 2. PL/SQL package script (V002__create_customer_api.sql)

**Input (script in Git):**
```sql
-- V002__create_customer_api.sql
CREATE OR REPLACE PACKAGE customer_api AS
    FUNCTION create_customer(
        p_first_name IN VARCHAR2,
        p_last_name  IN VARCHAR2,
        p_email      IN VARCHAR2
    ) RETURN NUMBER;

    FUNCTION get_customer_email(
        p_customer_id IN NUMBER
    ) RETURN VARCHAR2;
END customer_api;
/

CREATE OR REPLACE PACKAGE BODY customer_api AS

    FUNCTION create_customer(
        p_first_name IN VARCHAR2,
        p_last_name  IN VARCHAR2,
        p_email      IN VARCHAR2
    ) RETURN NUMBER IS
        v_id NUMBER;
    BEGIN
        INSERT INTO customers (first_name, last_name, email)
        VALUES (p_first_name, p_last_name, p_email)
        RETURNING customer_id INTO v_id;

        RETURN v_id;
    END create_customer;

    FUNCTION get_customer_email(
        p_customer_id IN NUMBER
    ) RETURN VARCHAR2 IS
        v_email customers.email%TYPE;
    BEGIN
        SELECT email
          INTO v_email
          FROM customers
         WHERE customer_id = p_customer_id;

        RETURN v_email;
    EXCEPTION
        WHEN NO_DATA_FOUND THEN
            RETURN NULL;
    END get_customer_email;

END customer_api;
/
```

**Example usage (input) and output:**
```sql
-- Insert a customer via the API
DECLARE
    v_id NUMBER;
BEGIN
    v_id := customer_api.create_customer(
        p_first_name => 'John',
        p_last_name  => 'Doe',
        p_email      => 'john.doe@example.com'
    );
    DBMS_OUTPUT.PUT_LINE('New customer ID: ' || v_id);
END;
/

-- Possible output:
-- New customer ID: 1

-- Get the email back
SET SERVEROUTPUT ON;
DECLARE
    v_email VARCHAR2(100);
BEGIN
    v_email := customer_api.get_customer_email(1);
    DBMS_OUTPUT.PUT_LINE('Email: ' || v_email);
END;
/

-- Output:
-- Email: john.doe@example.com
```

## Example: Simple CI/CD Pipeline (GitHub Actions)

**.github/workflows/db-ci.yml**

```yaml
name: Oracle DB CI

on:
  push:
    branches: [ main ]
  pull_request:

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Set up SQL*Plus (example using Docker)
        run: |
          docker pull container-registry/oracle-client:latest

      - name: Run migration scripts on test DB
        env:
          ORA_USER: ${{ secrets.ORA_USER }}
          ORA_PASS: ${{ secrets.ORA_PASS }}
          ORA_CONN: ${{ secrets.ORA_CONN }} # e.g. host:port/service
        run: |
          docker run --rm \
            -v $PWD/db:/db \
            container-registry/oracle-client:latest \
            bash -c "
              for f in /db/migrations/*.sql; do
                echo 'Running' \$f
                sqlplus -s ${ORA_USER}/${ORA_PASS}@${ORA_CONN} @\$f
              done
            "

      - name: Run unit tests (utPLSQL example)
        env:
          ORA_USER: ${{ secrets.ORA_USER }}
          ORA_PASS: ${{ secrets.ORA_PASS }}
          ORA_CONN: ${{ secrets.ORA_CONN }}
        run: |
          docker run --rm \
            container-registry/oracle-client:latest \
            sqlplus -s ${ORA_USER}/${ORA_PASS}@${ORA_CONN} @/db/tests/run_all_tests.sql
```

- **Input:** Push to `main` or open a PR.
- **Output:** Pipeline:
  - Applies all migration scripts to a test Oracle DB.
  - Runs unit tests.
  - Fails the build if any script or test fails.
