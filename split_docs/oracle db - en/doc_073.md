
Here’s a concise summary of ACID properties in Oracle (and relational databases in general), with simple examples.

---

## ACID Properties

ACID describes the key guarantees of a database transaction:

1. **Atomicity**  
   - A transaction is “all or nothing.”  
   - Either all operations in the transaction succeed, or none are applied.
   - If something fails, the database rolls back to the state before the transaction started.

2. **Consistency**  
   - A transaction takes the database from one valid state to another valid state.  
   - All integrity constraints (PK, FK, CHECK, etc.) must be satisfied before commit.  
   - Invalid data is not allowed to be committed.

3. **Isolation**  
   - Concurrent transactions do not interfere with each other’s intermediate states.  
   - Each transaction behaves as if it is running alone, even when many run at the same time.  
   - Oracle uses mechanisms like locks and multiversion read consistency to achieve this.

4. **Durability**  
   - Once a transaction is committed, its changes are permanent, even if the system crashes.  
   - Oracle ensures this via redo logs and writing changes to disk.

## Simple Oracle SQL Example

### Example 1: Atomicity & Consistency

```sql
-- Setup: create a simple accounts table
CREATE TABLE accounts (
    account_id NUMBER PRIMARY KEY,
    balance    NUMBER CHECK (balance >= 0)
);

INSERT INTO accounts VALUES (1, 1000);
INSERT INTO accounts VALUES (2, 500);
COMMIT;
```

#### Transaction: Transfer 200 from account 1 to account 2

```sql
-- Start transaction (implicit when you run DML)
UPDATE accounts
SET balance = balance - 200
WHERE account_id = 1;

UPDATE accounts
SET balance = balance + 200
WHERE account_id = 2;

COMMIT;  -- make changes permanent
```

**Result (after COMMIT):**

```sql
SELECT * FROM accounts;

-- Output:
-- ACCOUNT_ID | BALANCE
--     1      | 800
--     2      | 700
```

If something goes wrong before `COMMIT` (e.g., second update fails), you can:

```sql
ROLLBACK;  -- undo all changes in this transaction
```

Then:

```sql
SELECT * FROM accounts;

-- Output (original state restored):
-- ACCOUNT_ID | BALANCE
--     1      | 1000
--     2      | 500
```

This shows **atomicity** (both updates succeed or none) and **consistency** (no negative balances, constraints respected).

---

### Example 2: Isolation (Two Sessions)

**Session A:**

```sql
-- Start transaction
UPDATE accounts
SET balance = balance - 100
WHERE account_id = 1;

-- Do NOT commit yet
```

**Session B (at the same time):**

```sql
SELECT * FROM accounts WHERE account_id = 1;

-- Output in Session B (Oracle read consistency):
-- ACCOUNT_ID | BALANCE
--     1      | 800   -- still sees old committed value, not 700
```

Session B does not see Session A’s uncommitted change. This demonstrates **isolation**.

When Session A finally does:

```sql
COMMIT;
```

New queries in Session B will see the updated balance.

---

### Example 3: Durability

After a `COMMIT`:

```sql
UPDATE accounts
SET balance = balance + 50
WHERE account_id = 2;

COMMIT;
```

Even if the database instance crashes right after the commit, when it restarts:

```sql
SELECT * FROM accounts WHERE account_id = 2;

-- Output will still include the +50 change
```

This shows **durability**: committed changes survive crashes.

---

**Summary:**  
- **Atomicity**: all-or-nothing execution of a transaction.  
- **Consistency**: constraints and rules are preserved.  
- **Isolation**: concurrent transactions don’t see each other’s intermediate states.  
- **Durability**: committed data is permanent.
