
Here’s a concise summary and examples based on the topic:

---

## Key Ideas

1. **Avoid unnecessary `DISTINCT`**
   - `DISTINCT` forces Oracle to remove duplicate rows, which usually requires sorting or hashing.
   - This adds extra CPU and memory usage.
   - If you know your query already returns unique rows (e.g., using primary keys or unique constraints), you don’t need `DISTINCT`.

2. **Prefer `UNION ALL` over `UNION` when possible**
   - `UNION` = `UNION ALL` + duplicate elimination.
   - `UNION` sorts or hashes the combined result to remove duplicates, which is more expensive.
   - If you don’t need to remove duplicates, use `UNION ALL` for better performance.

---

## `DISTINCT` Example

### Bad (unnecessary DISTINCT)

```sql
-- Table: employees(emp_id PRIMARY KEY, first_name, last_name, department_id)

SELECT DISTINCT emp_id, first_name, last_name
FROM employees
WHERE department_id = 10;
```

**Why unnecessary?**  
`emp_id` is a primary key, so each row is already unique. `DISTINCT` adds extra work.

### Better

```sql
SELECT emp_id, first_name, last_name
FROM employees
WHERE department_id = 10;
```

**Input (employees table sample)**

| emp_id | first_name | last_name | department_id |
|--------|------------|-----------|---------------|
| 1      | John       | Smith     | 10            |
| 2      | Mary       | Jones     | 10            |
| 3      | John       | Smith     | 20            |

**Output of both queries**

| emp_id | first_name | last_name |
|--------|------------|-----------|
| 1      | John       | Smith     |
| 2      | Mary       | Jones     |

Same result, but the second query avoids unnecessary work.

## `UNION` vs `UNION ALL` Example

### Using `UNION` (removes duplicates, more expensive)

```sql
SELECT department_id
FROM employees
WHERE salary > 10000
UNION
SELECT department_id
FROM employees
WHERE job_id = 'MANAGER';
```

- Oracle combines the two result sets and removes duplicate `department_id` values.

**Sample Inputs**

`employees` (simplified):

| emp_id | department_id | salary | job_id  |
|--------|---------------|--------|--------|
| 1      | 10            | 12000  | DEV    |
| 2      | 20            | 9000   | MANAGER|
| 3      | 10            | 15000  | MANAGER|
| 4      | 30            | 11000  | DEV    |

**Intermediate results**

- `salary > 10000` → dept: 10, 10, 30
- `job_id = 'MANAGER'` → dept: 20, 10

**UNION output (duplicates removed)**

| department_id |
|---------------|
| 10            |
| 20            |
| 30            |

### Using `UNION ALL` (keeps duplicates, faster)

```sql
SELECT department_id
FROM employees
WHERE salary > 10000
UNION ALL
SELECT department_id
FROM employees
WHERE job_id = 'MANAGER';
```

**Output (duplicates kept)**

| department_id |
|---------------|
| 10            |
| 10            |
| 30            |
| 20            |
| 10            |

Use `UNION ALL` when:
- You don’t care about duplicates, or
- You know the two result sets cannot overlap.

---

## Practical Guidelines

- Remove `DISTINCT` if:
  - The columns are already guaranteed unique (primary key, unique index).
- Replace `UNION` with `UNION ALL` if:
  - Duplicates are acceptable, or
  - You know there will be no duplicates between the queries.

These changes often reduce sorting/hashing and improve query performance in Oracle.
