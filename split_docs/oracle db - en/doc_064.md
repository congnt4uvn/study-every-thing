
Here’s a concise summary of the topic “Relational Design: Referential Integrity and Cascading” with examples.

---

## 1. Referential Integrity: Key Idea

**Referential integrity** ensures that relationships between tables remain valid.  
Typically enforced with **foreign keys**:

- A **foreign key** in a child table must match an existing **primary key** (or unique key) in a parent table, or be NULL (if allowed).
- Prevents “orphan” rows (child rows that reference non‑existent parent rows).

### Example Tables

```sql
-- Parent table
CREATE TABLE departments (
    dept_id   NUMBER PRIMARY KEY,
    dept_name VARCHAR2(100) NOT NULL
);

-- Child table
CREATE TABLE employees (
    emp_id    NUMBER PRIMARY KEY,
    emp_name  VARCHAR2(100) NOT NULL,
    dept_id   NUMBER,
    CONSTRAINT fk_emp_dept
        FOREIGN KEY (dept_id)
        REFERENCES departments (dept_id)
);
```

**Behavior:**

- You cannot insert an employee with `dept_id = 10` if there is no row in `departments` with `dept_id = 10`.
- You cannot delete a department if there are employees referencing it (unless cascading is defined).

---

## 2. Cascading: What Happens on UPDATE/DELETE

**Cascading** defines what Oracle should do to child rows when a parent row is updated or deleted.

Common options:

1. `ON DELETE CASCADE`  
   - When a parent row is deleted, all matching child rows are automatically deleted.

2. `ON DELETE SET NULL`  
   - When a parent row is deleted, the foreign key in child rows is set to NULL.

> Note: Oracle does **not** support `ON UPDATE CASCADE` on foreign keys (unlike some other DBMSs). Updates to primary keys must be handled manually or avoided.

### Example: ON DELETE CASCADE

```sql
CREATE TABLE departments (
    dept_id   NUMBER PRIMARY KEY,
    dept_name VARCHAR2(100) NOT NULL
);

CREATE TABLE employees (
    emp_id    NUMBER PRIMARY KEY,
    emp_name  VARCHAR2(100) NOT NULL,
    dept_id   NUMBER,
    CONSTRAINT fk_emp_dept
        FOREIGN KEY (dept_id)
        REFERENCES departments (dept_id)
        ON DELETE CASCADE
);
```

#### Sample Data (Input)

```sql
INSERT INTO departments (dept_id, dept_name)
VALUES (10, 'Sales');

INSERT INTO employees (emp_id, emp_name, dept_id)
VALUES (1, 'Alice', 10);

INSERT INTO employees (emp_id, emp_name, dept_id)
VALUES (2, 'Bob', 10);
COMMIT;
```

#### Operation

```sql
DELETE FROM departments WHERE dept_id = 10;
COMMIT;
```

#### Result (Output)

```sql
SELECT * FROM employees;
-- No rows returned, because both Alice and Bob were deleted automatically.
```

---

### Example: ON DELETE SET NULL

```sql
CREATE TABLE departments (
    dept_id   NUMBER PRIMARY KEY,
    dept_name VARCHAR2(100) NOT NULL
);

CREATE TABLE employees (
    emp_id    NUMBER PRIMARY KEY,
    emp_name  VARCHAR2(100) NOT NULL,
    dept_id   NUMBER,
    CONSTRAINT fk_emp_dept
        FOREIGN KEY (dept_id)
        REFERENCES departments (dept_id)
        ON DELETE SET NULL
);
```

#### Sample Data (Input)

```sql
INSERT INTO departments (dept_id, dept_name)
VALUES (20, 'HR');

INSERT INTO employees (emp_id, emp_name, dept_id)
VALUES (3, 'Carol', 20);

INSERT INTO employees (emp_id, emp_name, dept_id)
VALUES (4, 'Dave', 20);
COMMIT;
```

#### Operation

```sql
DELETE FROM departments WHERE dept_id = 20;
COMMIT;
```

#### Result (Output)

```sql
SELECT emp_id, emp_name, dept_id FROM employees;

-- emp_id | emp_name | dept_id
--   3    | Carol    |  NULL
--   4    | Dave     |  NULL
```

Employees remain, but their `dept_id` is set to NULL.

---

## 3. Design Considerations

- Use **foreign keys** to enforce referential integrity instead of relying on application logic.
- Choose cascading options based on business rules:
  - `ON DELETE CASCADE` when child data is meaningless without the parent.
  - `ON DELETE SET NULL` when child data can exist without the parent.
- Avoid updating primary keys; design stable keys to reduce the need for cascaded updates.
