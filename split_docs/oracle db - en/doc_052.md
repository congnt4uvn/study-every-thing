**Topic: Indexing Strategy – Leading Column in Composite Index (Oracle Database)**

---

## Key Ideas

1. **Composite Index = Multi-column Index**  
   A composite (multi-column) index is created on more than one column:
   ```sql
   CREATE INDEX idx_emp_dept_job
   ON employees (department_id, job_id);
   ```

2. **Leading Column = First Column in the Index Definition**  
   - In `ON employees (department_id, job_id)`, the **leading column** is `department_id`.
   - Oracle can use this index efficiently when the query’s `WHERE` clause filters on:
     - `department_id` alone, or
     - `department_id` and `job_id`, or
     - `department_id` plus more columns in the index order.

3. **Without the Leading Column, Index Often Not Used**  
   - If you filter only on `job_id` (the second column), Oracle usually **cannot** use the composite index efficiently, because the index is ordered first by `department_id`.
   - This is why choosing the right leading column is critical.

4. **Choose Leading Column Based on Selectivity & Query Patterns**  
   - Put the **most selective** and **most frequently filtered** column first.
   - Analyze typical queries: which column appears most often in `WHERE` conditions?

---

## Example 1: Composite Index Usage

### Index Definition
```sql
CREATE INDEX idx_emp_dept_job
ON employees (department_id, job_id);
```

### Query 1 – Uses Leading Column (Good)
```sql
SELECT employee_id, first_name, last_name
FROM employees
WHERE department_id = 50;
```

**Expected behavior:**  
Oracle can use `idx_emp_dept_job` because the predicate includes the **leading column** `department_id`.

**Simplified output example:**
```text
EMPLOYEE_ID | FIRST_NAME | LAST_NAME
----------- | ---------- | ---------
        101 | John       | Smith
        102 | Mary       | Brown
```

### Query 2 – Uses Leading + Next Column (Good)
```sql
SELECT employee_id, first_name, last_name
FROM employees
WHERE department_id = 50
  AND job_id = 'SA_REP';
```

**Expected behavior:**  
Oracle can use the same index even more efficiently (range scan on `(50, 'SA_REP')`).

**Simplified output example:**
```text
EMPLOYEE_ID | FIRST_NAME | LAST_NAME
----------- | ---------- | ---------
        102 | Mary       | Brown
```

### Query 3 – Skips Leading Column (Often Bad)
```sql
SELECT employee_id, first_name, last_name
FROM employees
WHERE job_id = 'SA_REP';
```

**Expected behavior:**  
Oracle usually **cannot** use `idx_emp_dept_job` efficiently because `department_id` (the leading column) is not in the predicate. It may do a full table scan or use another index if available.

---

## Example 2: Better Leading Column Choice

Assume:
- `department_id` has only a few distinct values (low selectivity).
- `job_id` has many distinct values and is often used in filters (high selectivity).

Then a better index might be:
```sql
CREATE INDEX idx_emp_job_dept
ON employees (job_id, department_id);
```

Now queries like:
```sql
SELECT *
FROM employees
WHERE job_id = 'SA_REP';
```
can use the index efficiently, because `job_id` is the **leading column**.

---

## Practical Guidelines

- Always think: **“Which column should be first?”**  
  That column should:
  - Be frequently used in `WHERE` clauses.
  - Be selective (filters out many rows).
- Composite index order matters: `(A, B)` is not the same as `(B, A)`.
- If you often query by `B` alone, consider:
  - A separate index on `B`, or
  - A composite index with `B` as the leading column.
