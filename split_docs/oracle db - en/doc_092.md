
Here’s a concise, high‑level summary of Oracle VPD / RLS, with simple code examples.

---

## Key Ideas

**VPD (Virtual Private Database)** and **RLS (Row-Level Security)** are Oracle features that:

- Automatically add security predicates (e.g., `WHERE` clauses) to queries.
- Restrict which rows a user can see or modify, based on:
  - User identity (`USER`, `SYS_CONTEXT`)
  - Session attributes (e.g., department, application, IP)
- Work transparently: applications run normal SQL; Oracle rewrites it behind the scenes.

RLS is implemented via **security policies** attached to tables/views. These policies call a **policy function** that returns a predicate (string) like:

```sql
'DEPARTMENT_ID = 10'
```

Oracle then rewrites:

```sql
SELECT * FROM employees;
```

to:

```sql
SELECT * FROM employees WHERE DEPARTMENT_ID = 10;
```

for that user/session.

---

## Basic Components

1. **Policy Function**  
   PL/SQL function that returns a `VARCHAR2` predicate.

2. **Policy Definition**  
   Use `DBMS_RLS.ADD_POLICY` to attach the function to a table/view and specify which operations it applies to (SELECT, INSERT, UPDATE, DELETE).

## Simple Example

### Scenario

- Table: `EMPLOYEES(DEPARTMENT_ID, EMP_NAME, SALARY, ...)`
- Requirement:  
  - Users only see rows where `DEPARTMENT_ID` = their own department.
  - Department is stored in a context variable `MY_CTX.DEPARTMENT_ID`.

### 1. Create an Application Context

```sql
-- As SYS or a privileged user
CREATE CONTEXT my_ctx USING set_my_ctx_pkg;
```

Package to set the context:

```sql
CREATE OR REPLACE PACKAGE set_my_ctx_pkg AS
  PROCEDURE set_dept(p_dept_id NUMBER);
END;
/

CREATE OR REPLACE PACKAGE BODY set_my_ctx_pkg AS
  PROCEDURE set_dept(p_dept_id NUMBER) IS
  BEGIN
    DBMS_SESSION.SET_CONTEXT(
      namespace => 'MY_CTX',
      attribute => 'DEPARTMENT_ID',
      value     => p_dept_id
    );
  END;
END;
/
```

Example of setting the context in a session:

```sql
BEGIN
  set_my_ctx_pkg.set_dept(10);  -- This user belongs to department 10
END;
/
```

### 2. Create the Policy Function

```sql
CREATE OR REPLACE FUNCTION emp_dept_policy(
  p_schema IN VARCHAR2,
  p_object IN VARCHAR2
) RETURN VARCHAR2 AS
  l_dept_id VARCHAR2(30);
BEGIN
  l_dept_id := SYS_CONTEXT('MY_CTX', 'DEPARTMENT_ID');

  -- If no department set, return a predicate that matches nothing
  IF l_dept_id IS NULL THEN
    RETURN '1 = 0';
  END IF;

  RETURN 'DEPARTMENT_ID = ' || l_dept_id;
END;
/
```

### 3. Attach the Policy to the Table

```sql
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'HR',
    object_name     => 'EMPLOYEES',
    policy_name     => 'EMP_DEPT_RLS',
    function_schema => 'HR',
    policy_function => 'EMP_DEPT_POLICY',
    statement_types => 'SELECT'
  );
END;
/
```

## How It Works (Input/Output Example)

**Input SQL (from application/user):**

```sql
SELECT EMP_NAME, DEPARTMENT_ID
FROM HR.EMPLOYEES;
```

**Context for this session:**

```sql
BEGIN
  set_my_ctx_pkg.set_dept(10);
END;
/
```

**What Oracle effectively runs (rewritten SQL):**

```sql
SELECT EMP_NAME, DEPARTMENT_ID
FROM HR.EMPLOYEES
WHERE DEPARTMENT_ID = 10;
```

**Output (visible to this user):**

```text
EMP_NAME     DEPARTMENT_ID
-----------  -------------
Alice        10
Bob          10
...
```

Rows from other departments are automatically hidden.

---

## Variations

- Apply to `INSERT`, `UPDATE`, `DELETE` as well (e.g., prevent updates outside user’s department).
- Use `SYS_CONTEXT('USERENV', 'SESSION_USER')` to filter by username:

  ```sql
  RETURN 'OWNER = ''' || SYS_CONTEXT('USERENV', 'SESSION_USER') || '''';
  ```

- Combine multiple conditions:

  ```sql
  RETURN 'DEPARTMENT_ID = ' || l_dept_id || ' AND ACTIVE_FLAG = ''Y''';
  ```

---

## Summary

- VPD/RLS = transparent row-level filtering enforced by the database.
- Implemented via:
  - A policy function that returns a `WHERE` clause string.
  - `DBMS_RLS.ADD_POLICY` to bind that function to a table/view and operations.
- Ensures consistent, centralized security regardless of how the data is accessed.
