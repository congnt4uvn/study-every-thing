
Key idea  
They want to see that you can think in terms of trade‑offs when designing or tuning an Oracle database:  
- Performance vs. flexibility vs. complexity  
- You understand there is no “perfect” solution; each choice has pros and cons.

Typical trade‑offs  
1. **Performance vs. Flexibility**  
   - Denormalization or precomputed data can speed up queries but reduces flexibility and increases maintenance.
2. **Performance vs. Complexity**  
   - Advanced features (partitioning, materialized views, complex PL/SQL) can improve performance but make the system harder to understand and maintain.
3. **Flexibility vs. Simplicity**  
   - Highly generic schemas (EAV models, dynamic structures) are flexible but often slow and complex.

Example 1: Denormalized summary table  
Trade‑off: Faster reads vs. more complex writes/maintenance.

```sql
-- Base table: SALES
CREATE TABLE sales (
    sale_id      NUMBER PRIMARY KEY,
    customer_id  NUMBER,
    sale_date    DATE,
    amount       NUMBER
);

-- Summary table for faster reporting
CREATE TABLE sales_daily_summary (
    summary_date DATE PRIMARY KEY,
    total_amount NUMBER
);

-- Procedure to maintain summary (complexity added)
CREATE OR REPLACE PROCEDURE update_sales_summary(p_date DATE) AS
BEGIN
    MERGE INTO sales_daily_summary s
    USING (
        SELECT TRUNC(p_date) AS summary_date,
               SUM(amount)   AS total_amount
        FROM   sales
        WHERE  TRUNC(sale_date) = TRUNC(p_date)
        GROUP  BY TRUNC(p_date)
    ) src
    ON (s.summary_date = src.summary_date)
    WHEN MATCHED THEN
        UPDATE SET s.total_amount = src.total_amount
    WHEN NOT MATCHED THEN
        INSERT (summary_date, total_amount)
        VALUES (src.summary_date, src.total_amount);
END;
/

-- Input: new sales rows for 2025-05-20
INSERT INTO sales VALUES (1, 101, DATE '2025-05-20', 100);
INSERT INTO sales VALUES (2, 102, DATE '2025-05-20', 200);

BEGIN
    update_sales_summary(DATE '2025-05-20');
END;
/

-- Output: fast reporting query
SELECT * FROM sales_daily_summary;

-- Example output:
-- SUMMARY_DATE  TOTAL_AMOUNT
-- 2025-05-20    300
```

Example 2: Flexible vs. performant design (EAV vs. normal columns)  
Trade‑off: Flexibility (dynamic attributes) vs. performance and simplicity.

**Flexible (EAV-style) design:**

```sql
CREATE TABLE product (
    product_id   NUMBER PRIMARY KEY,
    name         VARCHAR2(100)
);

CREATE TABLE product_attribute (
    product_id   NUMBER,
    attr_name    VARCHAR2(50),
    attr_value   VARCHAR2(100),
    CONSTRAINT fk_product_attr
        FOREIGN KEY (product_id) REFERENCES product(product_id)
);

-- Input: store arbitrary attributes
INSERT INTO product VALUES (1, 'Laptop');
INSERT INTO product_attribute VALUES (1, 'RAM', '16GB');
INSERT INTO product_attribute VALUES (1, 'Color', 'Black');

-- Output: query attributes
SELECT p.name, a.attr_name, a.attr_value
FROM   product p
JOIN   product_attribute a
ON     p.product_id = a.product_id
WHERE  p.product_id = 1;

-- Example output:
-- NAME   ATTR_NAME  ATTR_VALUE
-- Laptop RAM        16GB
-- Laptop Color      Black
```

**Simpler, more performant design (fixed columns):**

```sql
CREATE TABLE product_fixed (
    product_id   NUMBER PRIMARY KEY,
    name         VARCHAR2(100),
    ram          VARCHAR2(20),
    color        VARCHAR2(20)
);

-- Input:
INSERT INTO product_fixed
VALUES (1, 'Laptop', '16GB', 'Black');

-- Output: simple, fast query
SELECT name, ram, color
FROM   product_fixed
WHERE  product_id = 1;

-- Example output:
-- NAME   RAM   COLOR
-- Laptop 16GB  Black
```

In interviews or design discussions, explicitly state:  
- What you’re optimizing for (e.g., read performance, ease of change, maintainability).  
- What you’re sacrificing (e.g., flexibility, simplicity).  
- Why that trade‑off is appropriate for the specific Oracle database use case.
