
Here’s a concise summary of relational design normalization (1NF, 2NF, 3NF, BCNF), with simple Oracle-style SQL examples.

---

## 1. First Normal Form (1NF)

**Idea:**  
- All attributes are atomic (no repeating groups, no arrays, no comma-separated lists).
- Each field holds a single value.

**Bad (not 1NF):**

```sql
-- Orders table with repeating group
CREATE TABLE orders_bad (
    order_id     NUMBER PRIMARY KEY,
    customer_id  NUMBER,
    product_ids  VARCHAR2(100)  -- e.g. '10,11,12'
);
```

**Good (1NF):**

```sql
CREATE TABLE orders (
    order_id    NUMBER,
    customer_id NUMBER,
    product_id  NUMBER,
    PRIMARY KEY (order_id, product_id)
);
```

**Example data (input):**

```sql
INSERT INTO orders VALUES (1, 101, 10);
INSERT INTO orders VALUES (1, 101, 11);
INSERT INTO orders VALUES (1, 101, 12);
```

**Query (output):**

```sql
SELECT * FROM orders;

-- order_id | customer_id | product_id
--    1     |    101      |    10
--    1     |    101      |    11
--    1     |    101      |    12
```

---

## 2. Second Normal Form (2NF)

**Idea (for tables with composite primary keys):**  
- Must be in 1NF.  
- No **partial dependency**: every non-key attribute depends on the **whole** primary key, not just part of it.

**Bad (not 2NF):**

```sql
-- Composite key: (order_id, product_id)
CREATE TABLE order_items_bad (
    order_id     NUMBER,
    product_id   NUMBER,
    order_date   DATE,        -- depends only on order_id
    product_name VARCHAR2(50),-- depends only on product_id
    PRIMARY KEY (order_id, product_id)
);
```

Here, `order_date` depends only on `order_id`, and `product_name` only on `product_id` → partial dependencies.

**Good (2NF):** split into separate tables:

```sql
CREATE TABLE orders (
    order_id   NUMBER PRIMARY KEY,
    order_date DATE
);

CREATE TABLE products (
    product_id   NUMBER PRIMARY KEY,
    product_name VARCHAR2(50)
);

CREATE TABLE order_items (
    order_id   NUMBER,
    product_id NUMBER,
    quantity   NUMBER,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES orders(order_id),
    FOREIGN KEY (product_id) REFERENCES products(product_id)
);
```

---

## 3. Third Normal Form (3NF)

**Idea:**  
- Must be in 2NF.  
- No **transitive dependency**: non-key attributes must depend **only** on the key, not on other non-key attributes.

**Bad (not 3NF):**

```sql
CREATE TABLE employees_bad (
    emp_id      NUMBER PRIMARY KEY,
    emp_name    VARCHAR2(50),
    dept_id     NUMBER,
    dept_name   VARCHAR2(50)  -- depends on dept_id, not directly on emp_id
);
```

`dept_name` depends on `dept_id`, which depends on `emp_id` → transitive dependency.

**Good (3NF):**

```sql
CREATE TABLE departments (
    dept_id   NUMBER PRIMARY KEY,
    dept_name VARCHAR2(50)
);

CREATE TABLE employees (
    emp_id   NUMBER PRIMARY KEY,
    emp_name VARCHAR2(50),
    dept_id  NUMBER,
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);
```

**Example query (output):**

```sql
SELECT e.emp_name, d.dept_name
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

-- emp_name | dept_name
-- Alice    | Sales
-- Bob      | IT
```

## 4. Boyce–Codd Normal Form (BCNF)

**Idea:**  
- Stronger than 3NF.  
- For **every** functional dependency X → Y, X must be a **superkey**.  
- Eliminates anomalies when there are multiple candidate keys.

**Typical violation example:**

```sql
-- Assume: each course has one room, and each room has one course at a time
-- So: course_id → room_id and room_id → course_id
CREATE TABLE schedule_bad (
    course_id VARCHAR2(10),
    room_id   VARCHAR2(10),
    time_slot VARCHAR2(20),
    PRIMARY KEY (course_id, time_slot)
);
```

Here, `room_id` is not a key, but `room_id → course_id` holds, so BCNF is violated.

**BCNF fix (conceptually):**  
Split so that each determinant is a key:

```sql
CREATE TABLE course_room (
    course_id VARCHAR2(10) PRIMARY KEY,
    room_id   VARCHAR2(10) UNIQUE
);

CREATE TABLE course_time (
    course_id VARCHAR2(10),
    time_slot VARCHAR2(20),
    PRIMARY KEY (course_id, time_slot),
    FOREIGN KEY (course_id) REFERENCES course_room(course_id)
);
```

Now any functional dependency’s left side is a key (or superkey).

---

## Summary of Normal Forms

- **1NF:** No repeating groups; atomic values.
- **2NF:** 1NF + no partial dependency on part of a composite key.
- **3NF:** 2NF + no transitive dependency (non-key attributes depend only on the key).
- **BCNF:** For every functional dependency X → Y, X is a superkey (stronger than 3NF).
