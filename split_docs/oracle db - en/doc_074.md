
Here’s a concise summary of the key transaction concepts in Oracle: **COMMIT, ROLLBACK, SAVEPOINT**, with simple code examples and expected behavior.

---

## 1. Transaction Basics

- A **transaction** is a logical unit of work (one or more SQL statements) that must be **all successful or all undone**.
- In Oracle, a transaction starts with the first DML statement (INSERT/UPDATE/DELETE) and ends when you:
  - `COMMIT`
  - `ROLLBACK`
  - or the session ends (disconnect, crash, etc.).

---

## 2. COMMIT

**Purpose:**  
Permanently save all changes made in the current transaction.

**Key points:**
- Makes all changes **visible to other sessions**.
- Releases locks held by the transaction.
- After `COMMIT`, you **cannot undo** those changes with `ROLLBACK`.

**Example:**

```sql
-- Input
INSERT INTO employees (employee_id, first_name, last_name, salary)
VALUES (1001, 'John', 'Smith', 5000);

UPDATE employees
SET salary = 5500
WHERE employee_id = 1001;

COMMIT;
```

**Effect / Output:**
- A new employee with ID 1001 is inserted.
- Salary is updated to 5500.
- Changes are permanently stored and visible to other users.

---

## 3. ROLLBACK

**Purpose:**  
Undo changes in the current transaction.

**Key points:**
- `ROLLBACK` with no options: undo **all** changes since the last `COMMIT` (or since the transaction started).
- Releases locks.
- Restores data to the state at the last commit.

**Example (full rollback):**

```sql
-- Input
INSERT INTO employees (employee_id, first_name, last_name, salary)
VALUES (1002, 'Alice', 'Brown', 6000);

UPDATE employees
SET salary = 6500
WHERE employee_id = 1002;

ROLLBACK;
```

**Effect / Output:**
- The insert and update are both undone.
- Employee 1002 does not exist after the rollback.

---

## 4. SAVEPOINT

**Purpose:**  
Create a **named point** within a transaction to which you can partially roll back.

**Key points:**
- Does **not** end the transaction.
- You can have multiple savepoints.
- `ROLLBACK TO SAVEPOINT` undoes changes **after** that savepoint, but keeps earlier changes in the same transaction.
- Changes kept after a `ROLLBACK TO SAVEPOINT` are still uncommitted until you `COMMIT`.

**Example with SAVEPOINT:**

```sql
-- Input
-- Start transaction (first DML)
INSERT INTO employees (employee_id, first_name, last_name, salary)
VALUES (2001, 'Bob', 'Lee', 4000);

SAVEPOINT sp1;

UPDATE employees
SET salary = 4500
WHERE employee_id = 2001;

SAVEPOINT sp2;

UPDATE employees
SET salary = 5000
WHERE employee_id = 2001;

-- Decide to undo last change only
ROLLBACK TO sp2;  -- (this actually undoes changes after sp2; here it's a no-op)
ROLLBACK TO sp1;  -- undo salary = 4500 and salary = 5000

COMMIT;
```

**Step-by-step effect / output:**

1. After first `INSERT`:  
   - Employee 2001 exists with salary 4000.

2. After `SAVEPOINT sp1`:  
   - Mark this state (salary 4000).

3. After `UPDATE` to 4500:  
   - Salary is 4500.

4. After `SAVEPOINT sp2`:  
   - Mark this state (salary 4500).

5. After `UPDATE` to 5000:  
   - Salary is 5000.

6. `ROLLBACK TO sp2`:  
   - Undo changes after `sp2`.  
   - In this example, only the last update (to 5000) is undone.  
   - Salary returns to 4500.

7. `ROLLBACK TO sp1`:  
   - Undo changes after `sp1`.  
   - Salary returns to 4000.

8. `COMMIT`:  
   - Employee 2001 with salary 4000 is permanently saved.

---

## 5. Combined Example (COMMIT, ROLLBACK, SAVEPOINT)

```sql
-- Input
INSERT INTO accounts (account_id, balance) VALUES (1, 1000);
INSERT INTO accounts (account_id, balance) VALUES (2, 2000);

SAVEPOINT before_transfer;

UPDATE accounts
SET balance = balance - 300
WHERE account_id = 1;

UPDATE accounts
SET balance = balance + 300
WHERE account_id = 2;

-- Suppose we detect an error in the transfer logic
ROLLBACK TO before_transfer;

-- Now we decide to cancel everything
ROLLBACK;
```

**Effect / Output:**
- After `ROLLBACK TO before_transfer`:  
  - Both accounts are back to their balances right after the two INSERTs.
- After final `ROLLBACK`:  
  - Both INSERTs are undone.  
  - Accounts 1 and 2 do not exist (or are back to their previous state before this transaction).

---

### Summary

- **COMMIT**: End transaction, permanently save changes.
- **ROLLBACK**: Undo all changes in the current transaction (since last commit).
- **SAVEPOINT**: Mark a point inside a transaction; `ROLLBACK TO SAVEPOINT` allows **partial undo** within the same transaction.
