
Here’s a concise summary and some practical examples around **SQL tuning using `DBMS_STATS`** in Oracle.

---

## Key Points & Main Ideas

1. **Accurate statistics are critical for SQL performance**  
   - Oracle’s **Cost-Based Optimizer (CBO)** relies on statistics about tables, indexes, and columns to choose the best execution plan.  
   - Poor or stale statistics often lead to bad plans (full scans instead of index scans, wrong join methods, etc.).

2. **`DBMS_STATS` is the recommended package for statistics**  
   - Replaces older `ANALYZE` commands.  
   - Used to gather, modify, export/import, and lock/unlock statistics.

3. **Types of statistics managed by `DBMS_STATS`**
   - **Table stats**: number of rows, blocks, average row length.  
   - **Column stats**: number of distinct values, nulls, histograms.  
   - **Index stats**: leaf blocks, clustering factor, distinct keys.  
   - **System stats**: CPU speed, I/O performance (affects optimizer’s cost model).

4. **When to gather statistics**
   - After **bulk loads**, **massive deletes/updates**, or **schema changes**.  
   - Regularly via **scheduled jobs** (e.g., nightly or weekly).  
   - More frequently on **volatile tables**, less frequently on **static tables**.

5. **Histograms for skewed data**
   - Use histograms when column values are **skewed** and queries filter on those columns.  
   - Helps the optimizer estimate selectivity more accurately.

6. **Locking and restoring statistics**
   - You can **lock stats** on stable tables to avoid unwanted changes.  
   - You can **export/import** or **restore** stats to revert bad changes.

---

## Core `DBMS_STATS` Usage Examples

### 1. Gather Table Statistics

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',
    tabname          => 'EMPLOYEES',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE  -- gather index stats too
  );
END;
/
```

**Effect (conceptual “input/output”):**

- Input: Table `HR.EMPLOYEES` with current data.  
- Output: Updated stats in the data dictionary (e.g., `NUM_ROWS`, `BLOCKS`, column NDV, histograms).  
- Result: Optimizer may choose a different, more efficient plan for queries on `HR.EMPLOYEES`.

---

### 2. Gather Schema Statistics

```sql
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(
    ownname          => 'HR',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE
  );
END;
/
```

- Input: All objects in schema `HR`.  
- Output: Fresh stats for all tables and indexes in `HR`.

---

### 3. Gather Statistics on a Specific Index

```sql
BEGIN
  DBMS_STATS.GATHER_INDEX_STATS(
    ownname => 'HR',
    indname => 'EMP_LAST_NAME_IDX'
  );
END;
/
```

- Useful if you rebuilt or created an index and want the optimizer to use it correctly.

---

### 4. Creating Histograms for Skewed Columns

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname    => 'SALES',
    tabname    => 'ORDERS',
    method_opt => 'FOR COLUMNS SIZE 254 STATUS',
    cascade    => TRUE
  );
END;
/
```

- Input: `SALES.ORDERS` table with skewed `STATUS` column.  
- Output: Histogram on `STATUS` so queries like  
  `SELECT * FROM SALES.ORDERS WHERE STATUS = 'CANCELLED';`  
  get better selectivity estimates.

---

### 5. Locking and Unlocking Statistics

```sql
-- Lock stats to prevent automatic changes
BEGIN
  DBMS_STATS.LOCK_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES'
  );
END;
/

-- Unlock stats when you want to refresh them
BEGIN
  DBMS_STATS.UNLOCK_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES'
  );
END;
/
```

- Use this for **static reference tables** where you want stable plans.

---

### 6. Export and Import Statistics (Backup/Restore)

```sql
-- Create a stats table
BEGIN
  DBMS_STATS.CREATE_STAT_TABLE(
    ownname    => 'HR',
    stattab    => 'HR_STATS_BACKUP'
  );
END;
/

-- Export stats
BEGIN
  DBMS_STATS.EXPORT_SCHEMA_STATS(
    ownname => 'HR',
    stattab => 'HR_STATS_BACKUP'
  );
END;
/

-- Import stats (restore)
BEGIN
  DBMS_STATS.IMPORT_SCHEMA_STATS(
    ownname => 'HR',
    stattab => 'HR_STATS_BACKUP'
  );
END;
/
```

- Useful before major changes; you can revert if performance degrades.

---

### 7. System Statistics (Optional but Important)

```sql
BEGIN
  DBMS_STATS.GATHER_SYSTEM_STATS(
    gathering_mode => 'NOWORKLOAD'
  );
END;
/
```

- Helps the optimizer model CPU vs I/O cost more accurately.

---

## How This Helps SQL Tuning

- **Better cardinality estimates** → better join orders and join methods.  
- **Correct index usage** → avoids unnecessary full table scans.  
- **Stable plans** via locked or backed-up stats.  
- **Adaptive behavior** with histograms for skewed data.
