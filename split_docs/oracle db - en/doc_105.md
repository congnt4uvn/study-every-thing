## Key Idea: Rollback Scripts in Version Control & CI/CD (Oracle DB)

1. **Purpose of rollback scripts**
   - Allow you to safely revert database changes (schema or data) when a deployment fails or needs to be undone.
   - Must be versioned alongside the forward (upgrade) scripts in your VCS (Git, etc.).
   - Integrated into CI/CD pipelines so rollbacks can be automated or semi-automated.

2. **Versioning strategy**
   - Each migration script (forward) has a corresponding rollback script.
   - Use a consistent naming convention, e.g.:
     - `V001__create_users_table.sql`
     - `V001__rollback_create_users_table.sql`
   - Store both in the same repo so they are traceable and auditable.

3. **CI/CD integration**
   - Pipelines run forward scripts on deploy.
   - On failure or manual trigger, pipelines run the matching rollback script.
   - Use a migration tracking table (e.g. `SCHEMA_VERSION`) to know which scripts were applied and which rollback to run.

## Example: Simple Schema Change + Rollback

### Forward script: create a table

**File:** `V001__create_users_table.sql`

```sql
-- Forward migration: create USERS table
CREATE TABLE users (
    user_id      NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username     VARCHAR2(50) NOT NULL UNIQUE,
    email        VARCHAR2(100) NOT NULL,
    created_at   DATE DEFAULT SYSDATE
);

-- Record migration in tracking table
INSERT INTO schema_version (version, description, applied_at)
VALUES ('001', 'create_users_table', SYSDATE);
COMMIT;
```

**Input:** Run this script via CI/CD (e.g., `sqlplus` or `sqlcl`).

**Output (DB state):**
- New table `USERS` created.
- Row inserted into `SCHEMA_VERSION` with version `001`.

---

### Rollback script: drop the table

**File:** `V001__rollback_create_users_table.sql`

```sql
-- Rollback migration: drop USERS table
DROP TABLE users CASCADE CONSTRAINTS;

-- Remove migration record
DELETE FROM schema_version
 WHERE version = '001';
COMMIT;
```

**Input:** Run this script if deployment needs to be reverted.

**Output (DB state):**
- Table `USERS` removed.
- Version `001` removed from `SCHEMA_VERSION`.

## Example: Column Addition + Rollback

### Forward script: add column

**File:** `V002__add_status_to_users.sql`

```sql
ALTER TABLE users ADD (status VARCHAR2(20) DEFAULT 'ACTIVE' NOT NULL);

INSERT INTO schema_version (version, description, applied_at)
VALUES ('002', 'add_status_to_users', SYSDATE);
COMMIT;
```

**Output (DB state):**
- Column `STATUS` added to `USERS`.
- Version `002` recorded.

### Rollback script: remove column

**File:** `V002__rollback_add_status_to_users.sql`

```sql
ALTER TABLE users DROP COLUMN status;

DELETE FROM schema_version
 WHERE version = '002';
COMMIT;
```

**Output (DB state):**
- Column `STATUS` removed.
- Version `002` removed from tracking.

## Example: CI/CD Pipeline (Pseudo YAML)

```yaml
stages:
  - migrate
  - rollback

migrate:
  stage: migrate
  script:
    - sqlplus user/pass@db @V001__create_users_table.sql
    - sqlplus user/pass@db @V002__add_status_to_users.sql
  when: on_success

rollback:
  stage: rollback
  script:
    - sqlplus user/pass@db @V002__rollback_add_status_to_users.sql
    - sqlplus user/pass@db @V001__rollback_create_users_table.sql
  when: manual   # or on_failure
```

---

## Main Takeaways

- Always pair each forward migration with a rollback script.
- Keep both under version control with clear naming and ordering.
- Use a migration tracking table to know what’s applied.
- Integrate both forward and rollback scripts into your CI/CD pipeline so you can reliably deploy and revert Oracle database changes.
