
Here’s a concise summary and some practical examples about **SQL tuning using hints in Oracle**, based on your topic:

---

## Key Ideas

1. **Hints are optimizer directives**  
   - Oracle SQL hints are comments embedded in SQL that influence the Cost-Based Optimizer (CBO).  
   - They can change join methods, access paths, parallelism, etc.

2. **Use hints only when needed**  
   - First try: good statistics, proper indexing, and query rewrite.  
   - Use hints when:
     - The optimizer consistently chooses a suboptimal plan.
     - You need to enforce a specific plan for critical queries.
   - Overuse of hints can:
     - Make SQL harder to maintain.
     - Lock you into a plan that may become bad as data changes.

3. **Hints are not guaranteed**  
   - If a hint is invalid, conflicts with others, or cannot be applied, Oracle may ignore it.  
   - Always verify the actual execution plan.

4. **Common categories of hints**
   - **Join order**: `ORDERED`, `LEADING`
   - **Join methods**: `USE_NL`, `USE_HASH`, `USE_MERGE`
   - **Access paths**: `INDEX`, `FULL`, `INDEX_FFS`
   - **Parallelism**: `PARALLEL`, `NOPARALLEL`
   - **Result caching**: `RESULT_CACHE`

---

## Basic Syntax

Hints go right after `SELECT`, `UPDATE`, `DELETE`, or `INSERT` keyword:

```sql
SELECT /*+ HINT_NAME */ col1, col2
FROM   table_name
WHERE  ...
```

---

## Example 1: Forcing an Index

**Goal:** Force the optimizer to use a specific index.

```sql
-- Table: employees(emp_id, dept_id, last_name, salary)
-- Index: emp_dept_idx on employees(dept_id)

SELECT /*+ INDEX(e emp_dept_idx) */
       e.emp_id, e.last_name, e.salary
FROM   employees e
WHERE  e.dept_id = 10;
```

**Expected effect:**  
- Without the hint, Oracle might choose a full table scan if it thinks it’s cheaper.  
- With `INDEX(e emp_dept_idx)`, it is directed to use `emp_dept_idx` on `employees`.

**Check the plan:**

```sql
EXPLAIN PLAN FOR
SELECT /*+ INDEX(e emp_dept_idx) */
       e.emp_id, e.last_name, e.salary
FROM   employees e
WHERE  e.dept_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Output (simplified):

```text
--------------------------------------------------------------------------------
| Id  | Operation                   | Name          |
--------------------------------------------------------------------------------
|   0 | SELECT STATEMENT            |               |
|   1 |  TABLE ACCESS BY INDEX ROWID| EMPLOYEES     |
|   2 |   INDEX RANGE SCAN          | EMP_DEPT_IDX  |
--------------------------------------------------------------------------------
```

## Example 2: Forcing a Nested Loop Join

**Goal:** Use a nested loop join instead of a hash join.

```sql
SELECT /*+ USE_NL(e d) */
       e.emp_id, e.last_name, d.dept_name
FROM   employees e
       JOIN departments d
         ON e.dept_id = d.dept_id
WHERE  e.salary > 10000;
```

**Expected effect:**  
- `USE_NL(e d)` tells Oracle to use a nested loop join between `employees` and `departments`.

---

## Example 3: Controlling Join Order

**Goal:** Force a specific join order.

```sql
SELECT /*+ LEADING(d e) */
       d.dept_name, e.emp_id, e.last_name
FROM   departments d
       JOIN employees e
         ON e.dept_id = d.dept_id
       JOIN locations l
         ON d.location_id = l.location_id
WHERE  l.country_id = 'US';
```

**Expected effect:**  
- `LEADING(d e)` tells Oracle to start the join with `departments` then `employees`, instead of letting the optimizer choose freely.

---

## Example 4: Parallel Query Hint

**Goal:** Run a query in parallel.

```sql
SELECT /*+ PARALLEL(e 4) */
       COUNT(*)
FROM   employees e
WHERE  salary > 5000;
```

**Expected effect:**  
- `PARALLEL(e 4)` suggests using 4 parallel execution servers for table `employees`.

---

## When to Avoid Hints

- When the optimizer already chooses a good plan.
- When data volume and distribution are changing rapidly.
- When you don’t have a clear, measured performance problem.

---

## Practical Workflow

1. Capture the slow SQL.
2. Check the execution plan (`DBMS_XPLAN.DISPLAY_CURSOR`).
3. Fix statistics, indexes, and query structure first.
4. If still bad, test hints in a controlled environment.
5. Measure performance before/after.
6. Only then, consider deploying hints in production.
