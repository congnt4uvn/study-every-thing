## Key Ideas

1. **Locking & Isolation**
   - Oracle uses locks to protect data integrity when multiple sessions access the same data.
   - Isolation level (usually `READ COMMITTED` by default) controls how and when changes become visible to other sessions.
   - Blocking occurs when one session holds a lock that another session needs, causing the second session to wait.

2. **Blocking Session**
   - A *blocking session* is a session that holds a lock on a resource (row, table, etc.) that another session is trying to modify or lock.
   - A *blocked session* is the one waiting for the lock to be released.

3. **Detecting Blocking Sessions**
   - You can detect blocking sessions using:
     - Data dictionary views (`V$SESSION`, `V$LOCK`, `V$SESSION_WAIT`, `V$ACTIVE_SESSION_HISTORY`)
     - Enterprise Manager / Cloud Control
     - Custom queries joining these views

4. **Resolving Blocking**
   - Identify the blocking session and what SQL it is running.
   - Decide whether to:
     - Wait for it to complete
     - Ask the user to commit/rollback
     - Kill the session (as a last resort)

## Core Queries to Detect Blocking Sessions

### 1. Simple “who blocks whom” query

```sql
SELECT
    s1.sid        AS blocking_sid,
    s1.serial#    AS blocking_serial,
    s1.username   AS blocking_user,
    s1.machine    AS blocking_machine,
    s1.program    AS blocking_program,
    s2.sid        AS blocked_sid,
    s2.serial#    AS blocked_serial,
    s2.username   AS blocked_user,
    s2.machine    AS blocked_machine,
    s2.program    AS blocked_program
FROM v$lock l1
JOIN v$session s1 ON l1.sid = s1.sid
JOIN v$lock l2 ON l1.id1 = l2.id1
               AND l1.id2 = l2.id2
JOIN v$session s2 ON l2.sid = s2.sid
WHERE l1.block = 1
  AND l2.request > 0;
```

**Output (example):**

| BLOCKING_SID | BLOCKING_SERIAL | BLOCKING_USER | BLOCKING_MACHINE | BLOCKING_PROGRAM | BLOCKED_SID | BLOCKED_SERIAL | BLOCKED_USER | BLOCKED_MACHINE | BLOCKED_PROGRAM |
|-------------:|----------------:|---------------|------------------|------------------|------------:|---------------:|-------------|-----------------|-----------------|
| 123          | 4567            | APPUSER       | APP-SRV-01       | myapp.exe        | 234         | 8910           | APPUSER     | APP-SRV-02      | myapp.exe       |

Interpretation: session 123 is blocking session 234.

### 2. Show blocking sessions with SQL text

```sql
SELECT
    s.sid,
    s.serial#,
    s.username,
    s.machine,
    s.program,
    q.sql_text
FROM v$session s
LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE s.sid IN (
    SELECT DISTINCT l1.sid
    FROM v$lock l1
    WHERE l1.block = 1
);
```

**Output (example):**

| SID | SERIAL# | USERNAME | MACHINE    | PROGRAM   | SQL_TEXT                                  |
|----:|--------:|----------|-----------|-----------|-------------------------------------------|
| 123 | 4567    | APPUSER  | APP-SRV-01| myapp.exe | UPDATE orders SET status='P' WHERE id=10  |

Interpretation: session 123 is blocking others while running that `UPDATE`.

---

### 3. Show blocked sessions and what they are waiting for

```sql
SELECT
    s.sid,
    s.serial#,
    s.username,
    s.machine,
    s.program,
    s.event,
    s.wait_class,
    s.seconds_in_wait,
    q.sql_text
FROM v$session s
LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE s.blocking_session IS NOT NULL;
```

**Output (example):**

| SID | SERIAL# | USERNAME | EVENT              | WAIT_CLASS | SECONDS_IN_WAIT | SQL_TEXT                                      |
|----:|--------:|----------|--------------------|-----------|----------------:|-----------------------------------------------|
| 234 | 8910    | APPUSER  | enq: TX - row lock | Application| 120             | UPDATE orders SET status='S' WHERE id=10      |

Interpretation: session 234 is waiting 120 seconds on a row lock (TX) held by another session.

### 4. Using `V$SESSION` blocking columns (11g+)

```sql
SELECT
    sid,
    serial#,
    username,
    blocking_session,
    blocking_instance,
    blocking_session_status
FROM v$session
WHERE blocking_session IS NOT NULL;
```

**Output (example):**

| SID | SERIAL# | USERNAME | BLOCKING_SESSION | BLOCKING_INSTANCE | BLOCKING_SESSION_STATUS |
|----:|--------:|----------|------------------|-------------------|-------------------------|
| 234 | 8910    | APPUSER  | 123              | 1                 | VALID                   |

## Example Scenario (Input/Output)

**Session A (blocking):**

```sql
-- Session A
UPDATE employees
SET salary = salary + 100
WHERE employee_id = 100;

-- No COMMIT or ROLLBACK yet
```

**Session B (blocked):**

```sql
-- Session B
UPDATE employees
SET salary = salary + 200
WHERE employee_id = 100;
-- This will hang, waiting for Session A's lock
```

**Admin session:**

Run the blocking query:

```sql
SELECT
    s1.sid AS blocking_sid,
    s2.sid AS blocked_sid
FROM v$lock l1
JOIN v$session s1 ON l1.sid = s1.sid
JOIN v$lock l2 ON l1.id1 = l2.id1 AND l1.id2 = l2.id2
JOIN v$session s2 ON l2.sid = s2.sid
WHERE l1.block = 1
  AND l2.request > 0;
```

**Output:**

| BLOCKING_SID | BLOCKED_SID |
|-------------:|------------:|
| 10           | 20          |

Interpretation: Session 10 (Session A) is blocking Session 20 (Session B).

## Killing a Blocking Session (last resort)

```sql
ALTER SYSTEM KILL SESSION '10,12345' IMMEDIATE;
-- 'sid,serial#' from the queries above
```

After this, the blocked session will either:
- Get an error and rollback, or
- Be able to proceed if it retries the operation.
