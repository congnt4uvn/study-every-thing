
Here’s a concise summary of **Physical Design: Materialized Views** in Oracle Database, plus some practical code examples.

---

## Key Ideas

1. **What a Materialized View Is**
   - A **materialized view (MV)** is a database object that stores the **result of a query physically on disk**.
   - Unlike a normal view (which is just a stored query), an MV keeps the data, so queries can be much faster.

2. **Why Use Materialized Views (Physical Design Perspective)**
   - Improve performance for:
     - Complex joins
     - Aggregations (SUM, COUNT, AVG, etc.)
     - Data warehouse / reporting queries
   - Reduce load on base tables by serving results from precomputed data.
   - Can be used for **query rewrite**: Oracle automatically uses the MV instead of the base tables when possible.

3. **Refresh Options**
   - **ON COMMIT**: Refreshes the MV whenever the underlying tables are committed (near real-time, more overhead).
   - **ON DEMAND**: Refresh only when explicitly requested (e.g., via a job).
   - **COMPLETE**: Rebuilds the MV from scratch.
   - **FAST**: Applies only the changes (requires materialized view logs or other conditions).
   - **FORCE**: Tries FAST; if not possible, uses COMPLETE.

4. **Storage & Physical Design Considerations**
   - MVs consume **disk space** (they store data).
   - Need **indexes** on MV columns used in filters/joins to maximize performance.
   - Choose **partitioning** for large MVs to improve manageability and refresh performance.
   - Decide which queries are most expensive and frequent; design MVs to match those patterns.

5. **Query Rewrite**
   - If enabled, Oracle can automatically rewrite a user’s query to use the MV instead of base tables.
   - Requires:
     - `QUERY_REWRITE_ENABLED = TRUE`
     - `QUERY_REWRITE_INTEGRITY` set appropriately
     - MV defined with `ENABLE QUERY REWRITE`

## Basic Code Examples

### 1. Simple Materialized View (Aggregation)

**Goal:** Precompute total sales per product.

**Base table:**
```sql
CREATE TABLE sales (
    sale_id      NUMBER PRIMARY KEY,
    product_id   NUMBER,
    sale_date    DATE,
    amount       NUMBER
);
```

**Materialized view:**
```sql
CREATE MATERIALIZED VIEW mv_sales_by_product
BUILD IMMEDIATE
REFRESH FAST ON DEMAND
ENABLE QUERY REWRITE
AS
SELECT
    product_id,
    SUM(amount) AS total_amount,
    COUNT(*)    AS sale_count
FROM sales
GROUP BY product_id;
```

**Example usage (input query):**
```sql
SELECT product_id, SUM(amount) AS total_amount
FROM sales
GROUP BY product_id;
```

**Output (served from MV if query rewrite is enabled):**
```text
PRODUCT_ID | TOTAL_AMOUNT
-----------+-------------
1          | 1500
2          |  800
3          | 2200
...
```

Oracle may transparently use `mv_sales_by_product` instead of scanning `sales`.

---

### 2. Creating a Materialized View Log (for FAST refresh)

To support **FAST** refresh, you often need a **materialized view log** on the base table:

```sql
CREATE MATERIALIZED VIEW LOG ON sales
WITH ROWID, SEQUENCE (product_id, amount)
INCLUDING NEW VALUES;
```

This log tracks changes so the MV can be incrementally refreshed.

---

### 3. Refreshing a Materialized View

**Manual refresh (ON DEMAND):**
```sql
EXEC DBMS_MVIEW.REFRESH('MV_SALES_BY_PRODUCT', 'F');  -- 'F' = FAST
-- or
EXEC DBMS_MVIEW.REFRESH('MV_SALES_BY_PRODUCT', 'C');  -- 'C' = COMPLETE
```

---

### 4. ON COMMIT Refresh Example

**Near real-time MV:**
```sql
CREATE MATERIALIZED VIEW mv_sales_daily
BUILD IMMEDIATE
REFRESH FAST ON COMMIT
ENABLE QUERY REWRITE
AS
SELECT
    TRUNC(sale_date) AS sale_day,
    SUM(amount)      AS total_amount
FROM sales
GROUP BY TRUNC(sale_date);
```

Whenever a transaction commits changes to `sales`, `mv_sales_daily` is updated.

---

### 5. Checking Which MVs Exist

```sql
SELECT mview_name, refresh_mode, refresh_method
FROM   user_mviews;
```

Example output:
```text
MVIEW_NAME           REFRESH_MODE  REFRESH_METHOD
-------------------- ------------  --------------
MV_SALES_BY_PRODUCT  DEMAND        FAST
MV_SALES_DAILY       COMMIT        FAST
```
