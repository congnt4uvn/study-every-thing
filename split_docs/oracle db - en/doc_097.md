## Key Points & Main Ideas (for a developer)

1. **High Availability (HA) = keep the app running**
   - Goal: Minimize downtime when hardware, network, or software fails.
   - For developers: Your app should survive database node failures, planned maintenance, and network glitches without major user impact.

2. **Backup & Recovery = protect data**
   - Goal: Ensure you can restore data after corruption, user errors (e.g., accidental DELETE), or disasters.
   - For developers: Your code and processes must assume that data can be restored to a previous point in time and that some operations may be rolled back.

3. **Why developers should care**
   - **Error handling**: Your code must handle transient errors (e.g., connection drops, failover) and retry safely.
   - **Idempotency**: Operations may be retried after failover; your logic should not double-charge, double-insert, etc.
   - **Transactions**: Use proper transactions so that partial work doesn’t get committed during failures.
   - **Performance vs. durability**: Some features (e.g., synchronous replication) affect latency; design queries and patterns with this in mind.
   - **Testing**: You should test how your app behaves during failover and restore scenarios.

4. **Oracle-specific HA concepts (high-level)**
   - **RAC (Real Application Clusters)**: Multiple instances on different servers accessing the same database → instance failure doesn’t mean database down.
   - **Data Guard**: Primary + standby databases (physical or logical) for disaster recovery and read-only offloading.
   - **FAN/Fast Connection Failover**: Notifies clients of node failures so they can quickly reconnect.
   - **Services**: Logical groupings of workloads that can move between instances.

5. **Oracle backup & recovery basics**
   - **RMAN (Recovery Manager)**: Oracle’s tool for backups and restores.
   - **Backup types**: Full, incremental, archived redo logs.
   - **Point-in-time recovery**: Restore database to a specific SCN/time before a bad operation.

## Example: Handling transient DB errors in application code

### Scenario
Your app connects to an Oracle RAC or Data Guard primary. One node fails; connections drop. Your code should:
- Catch the error
- Roll back the transaction
- Reconnect
- Retry if safe

### Pseudocode (Java with JDBC)

```java
public void transferFunds(Connection conn, long fromId, long toId, BigDecimal amount) throws SQLException {
    String debitSql = "UPDATE accounts SET balance = balance - ? WHERE id = ?";
    String creditSql = "UPDATE accounts SET balance = balance + ? WHERE id = ?";

    PreparedStatement debitStmt = null;
    PreparedStatement creditStmt = null;

    try {
        conn.setAutoCommit(false);

        debitStmt = conn.prepareStatement(debitSql);
        debitStmt.setBigDecimal(1, amount);
        debitStmt.setLong(2, fromId);
        debitStmt.executeUpdate();

        creditStmt = conn.prepareStatement(creditSql);
        creditStmt.setBigDecimal(1, amount);
        creditStmt.setLong(2, toId);
        creditStmt.executeUpdate();

        conn.commit();
    } catch (SQLTransientConnectionException e) {
        // Connection lost (e.g., failover)
        conn.rollback();
        // Re-throw or trigger retry logic at a higher level
        throw e;
    } catch (SQLException e) {
        conn.rollback();
        throw e;
    } finally {
        if (debitStmt != null) debitStmt.close();
        if (creditStmt != null) creditStmt.close();
        conn.setAutoCommit(true);
    }
}
```

**Input (conceptual):**
- `fromId = 1`, `toId = 2`, `amount = 100.00`

**Output (database state):**
- Before: `acct 1 = 1000`, `acct 2 = 500`
- After: `acct 1 = 900`, `acct 2 = 600`  
If a failover happens mid-transaction, the rollback ensures no partial transfer.

## Example: Idempotent operation with a unique business key

To safely retry after a failure, use a unique key to ensure the same logical operation isn’t applied twice.

### Table design

```sql
CREATE TABLE payments (
    payment_id      NUMBER PRIMARY KEY,
    business_key    VARCHAR2(64) UNIQUE, -- e.g., "order-12345"
    amount          NUMBER(10,2),
    status          VARCHAR2(20),
    created_at      TIMESTAMP DEFAULT SYSTIMESTAMP
);
```

### Insert with idempotency

```sql
INSERT INTO payments (payment_id, business_key, amount, status)
VALUES (payments_seq.NEXTVAL, :business_key, :amount, 'PENDING');
```

If the client retries with the same `:business_key`, the second attempt will fail with a unique constraint error, which you can treat as “already processed”.

**Input:**
```sql
-- First call
:business_key = 'order-12345', :amount = 50.00

-- Second call (retry after timeout)
:business_key = 'order-12345', :amount = 50.00
```

**Output:**
- First call: row inserted.
- Second call: ORA-00001 unique constraint violation → app interprets as “already exists”.

## Example: Basic RMAN backup command (DBA side, but devs should know it exists)

```bash
rman target /

RMAN> BACKUP DATABASE PLUS ARCHIVELOG;
```

This creates a backup of the entire database plus archived redo logs, enabling point-in-time recovery.

## Example: Point-in-time recovery scenario (developer perspective)

1. At 10:00, a batch job accidentally runs:
   ```sql
   DELETE FROM orders WHERE order_date < DATE '2020-01-01';
   COMMIT;
   ```
2. At 10:30, you discover the mistake.
3. DBA uses RMAN to restore the database to 09:59 (before the DELETE).
4. Your application must:
   - Handle a brief downtime.
   - Not assume that data deleted at 10:00 is still gone after recovery.
   - Possibly re-run some jobs safely (idempotent design).

## Summary for developers

- Design for **failures and retries**: HA means your app will see transient errors; handle them cleanly.
- Make operations **idempotent**: So retries after failover don’t corrupt data.
- Use **transactions** correctly: Ensure atomicity and consistency during failures.
- Understand **Oracle HA features** (RAC, Data Guard, services, FAN) enough to cooperate with them.
- Be aware of **backup & recovery**: Data can be rolled back in time; your logic and processes must tolerate that.
