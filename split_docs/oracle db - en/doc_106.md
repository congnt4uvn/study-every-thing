## Summary: Version Control & CI/CD for Oracle Schema Changes

1. **Schema as Code**
   - Treat database schema (tables, views, packages, etc.) like application code.
   - Store all DDL and PL/SQL in version control (e.g., Git).
   - Use migration scripts with clear ordering (e.g., `V001__create_users.sql`, `V002__add_index.sql`).

2. **Migration Strategy**
   - Use incremental, forward-only migration scripts instead of editing existing scripts.
   - Each change has:
     - A unique ID
     - A description
     - A script that can be applied once and tracked in a schema history table.

3. **CI/CD Pipeline Integration**
   - On each commit:
     - Run static checks (linting, formatting) on SQL/PLSQL.
     - Run unit tests (e.g., utPLSQL) against a test database.
     - Apply migrations to a test/stage database automatically.
   - On release:
     - Apply the same migration set to production in a controlled, automated way.
     - Use rollback/backup strategy (e.g., restore point, flashback, or down-migration scripts).

4. **Safe Production Changes**
   - Prefer **backward-compatible** changes:
     - Add new columns instead of dropping/renaming immediately.
     - Deploy code that uses both old and new structures during transition.
   - Use **online operations** where possible (e.g., `ONLINE` index rebuilds).
   - Monitor performance and errors after deployment.

5. **Tracking & Auditing**
   - Maintain a schema version table in Oracle to record which migrations have been applied.
   - Ensure every environment (dev/test/stage/prod) can report its current schema version.

## Example: Simple Migration System

### 1. Schema Version Table

```sql
CREATE TABLE schema_version (
    version_id      NUMBER       NOT NULL,
    description     VARCHAR2(200),
    applied_at      DATE         DEFAULT SYSDATE,
    applied_by      VARCHAR2(30) DEFAULT USER,
    CONSTRAINT pk_schema_version PRIMARY KEY (version_id)
);
```

### 2. Migration Script Example

**File:** `V001__create_users_table.sql`

```sql
-- V001__create_users_table.sql
DECLARE
    v_version NUMBER := 1;
BEGIN
    -- Check if this version is already applied
    SELECT COUNT(*)
      INTO v_version
      FROM schema_version
     WHERE version_id = 1;

    IF v_version > 0 THEN
        DBMS_OUTPUT.PUT_LINE('Version 1 already applied.');
        RETURN;
    END IF;
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        NULL;
END;
/

-- Actual schema change
CREATE TABLE app_users (
    user_id      NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username     VARCHAR2(50) NOT NULL UNIQUE,
    email        VARCHAR2(100),
    created_at   DATE DEFAULT SYSDATE
);

-- Record the migration
INSERT INTO schema_version (version_id, description)
VALUES (1, 'Create app_users table');

COMMIT;
```

**Input:** Run the script in SQL*Plus or SQLcl:

```bash
sqlplus app_user/password@//dbhost:1521/ORCLPDB1 @V001__create_users_table.sql
```

**Output (example):**

```text
Version 1 already applied.  -- or nothing if first time
Table created.
1 row created.
Commit complete.
```

### 3. Next Migration: Add Column

**File:** `V002__add_status_to_users.sql`

```sql
-- V002__add_status_to_users.sql
DECLARE
    v_count NUMBER;
BEGIN
    SELECT COUNT(*)
      INTO v_count
      FROM schema_version
     WHERE version_id = 2;

    IF v_count > 0 THEN
        DBMS_OUTPUT.PUT_LINE('Version 2 already applied.');
        RETURN;
    END IF;
END;
/

ALTER TABLE app_users ADD (status VARCHAR2(20) DEFAULT 'ACTIVE' NOT NULL);

INSERT INTO schema_version (version_id, description)
VALUES (2, 'Add status column to app_users');

COMMIT;
```

## Example: CI Step (Pseudo-Script)

A CI job (e.g., in Jenkins, GitLab CI) might:

```bash
# 1. Run unit tests
sqlplus test_user/test@//testdb:1521/ORCLPDB1 @run_utplsql_tests.sql

# 2. Apply all pending migrations in order
for file in migrations/V*.sql; do
  echo "Applying $file"
  sqlplus deploy_user/deploy@//testdb:1521/ORCLPDB1 @"$file"
done
```

**Expected behavior:**
- If a migration is already applied, it prints a message and skips.
- New migrations are applied and recorded in `schema_version`.
