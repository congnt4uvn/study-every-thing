## Key Points (Physical Design in Oracle)

**Physical Design** in an Oracle database focuses on how data is stored on disk (tablespaces, datafiles, partitions, indexes, etc.). Good physical design provides:

1. **Manageability**  
   - Easier backup, recovery, and maintenance.  
   - Separation of data by function or usage (e.g., OLTP vs. reporting).  
   - Clear structure for growth and monitoring.

2. **Performance**  
   - Faster queries through proper indexing, partitioning, and storage layout.  
   - Reduced I/O and better use of memory and CPU.  
   - Ability to tune hot objects separately (e.g., frequently accessed tables).

3. **Pruning (Partition Pruning)**  
   - Oracle can skip entire partitions that are not needed for a query.  
   - Reduces the amount of data scanned, improving query performance.  
   - Especially useful for large tables (e.g., time-based data).

## Example: Partitioning for Performance & Pruning

### 1. Create a Partitioned Table

```sql
-- Input: DDL to create a partitioned sales table
CREATE TABLE sales (
    sale_id      NUMBER,
    sale_date    DATE,
    amount       NUMBER,
    customer_id  NUMBER
)
PARTITION BY RANGE (sale_date) (
    PARTITION sales_2023_q1 VALUES LESS THAN (DATE '2023-04-01'),
    PARTITION sales_2023_q2 VALUES LESS THAN (DATE '2023-07-01'),
    PARTITION sales_2023_q3 VALUES LESS THAN (DATE '2023-10-01'),
    PARTITION sales_2023_q4 VALUES LESS THAN (DATE '2024-01-01')
);
```

**What this achieves:**

- **Manageability**: You can archive/drop old partitions (e.g., `sales_2023_q1`) without affecting others.  
- **Performance**: Queries on a specific date range only touch relevant partitions.  
- **Pruning**: Oracle automatically prunes partitions that don’t match the `WHERE` clause.

### 2. Query That Uses Partition Pruning

```sql
-- Input: Query for Q2 2023 sales
SELECT SUM(amount) AS total_amount
FROM   sales
WHERE  sale_date BETWEEN DATE '2023-04-01' AND DATE '2023-06-30';
```

**Expected behavior (Output conceptually):**

- Oracle will only scan partition `sales_2023_q2`.  
- Other partitions (`sales_2023_q1`, `sales_2023_q3`, `sales_2023_q4`) are **pruned** (skipped).  
- Result example:

```text
TOTAL_AMOUNT
-----------
  1234567.89
```

(Exact value depends on data.)

## Example: Physical Design for Manageability (Tablespaces)

```sql
-- Input: Create separate tablespaces for OLTP and reporting
CREATE TABLESPACE oltp_data
  DATAFILE 'oltp_data01.dbf' SIZE 10G AUTOEXTEND ON;

CREATE TABLESPACE rpt_data
  DATAFILE 'rpt_data01.dbf' SIZE 20G AUTOEXTEND ON;

-- Create tables in specific tablespaces
CREATE TABLE orders (
    order_id   NUMBER PRIMARY KEY,
    order_date DATE,
    amount     NUMBER
) TABLESPACE oltp_data;

CREATE TABLE orders_report (
    order_id   NUMBER,
    order_date DATE,
    amount     NUMBER
) TABLESPACE rpt_data;
```

**Benefits:**

- **Manageability**: Different backup/maintenance strategies for OLTP vs. reporting data.  
- **Performance**: You can place tablespaces on different disks/storage tiers.

## Example: Index for Performance

```sql
-- Input: Create an index to speed up lookups by customer_id
CREATE INDEX idx_sales_customer
ON sales (customer_id);
```

**Effect:**

- **Performance**: Queries filtering by `customer_id` will be faster.  
- Combined with partitioning, you can also use **local indexes** per partition for better manageability and performance.

---

**In summary:**  
Physical design in Oracle (tablespaces, partitioning, indexing, storage layout) directly impacts **manageability**, **performance**, and **pruning**. Good design lets Oracle read less data, manage growth more easily, and respond faster to queries.
