
Here’s a concise summary of `EXPLAIN PLAN` and execution in Oracle, with code examples.

---

## 1. What `EXPLAIN PLAN` Does

- `EXPLAIN PLAN` shows how Oracle *would* execute a SQL statement (the execution plan) **without actually running it**.
- It reveals:
  - Join methods (NESTED LOOPS, HASH JOIN, MERGE JOIN)
  - Access paths (INDEX RANGE SCAN, FULL TABLE SCAN)
  - Order of operations
  - Estimated cost, cardinality (rows), and bytes

---

## 2. Basic Syntax

```sql
EXPLAIN PLAN
SET STATEMENT_ID = 'test_plan'
FOR
SELECT e.employee_id, e.first_name, d.department_name
FROM   employees e
       JOIN departments d ON e.department_id = d.department_id
WHERE  e.salary > 5000;
```

This stores the plan in the `PLAN_TABLE` (a special table used by Oracle to hold plan data).

---

## 3. Viewing the Plan

### 3.1 Using `DBMS_XPLAN.DISPLAY`

```sql
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY(
    table_name   => 'PLAN_TABLE',
    statement_id => 'test_plan',
    format       => 'TYPICAL'
));
```

**Sample Output (simplified):**

```text
Plan hash value: 123456789

--------------------------------------------------------------------------------
| Id  | Operation                    | Name        | Rows | Bytes | Cost (%CPU)|
--------------------------------------------------------------------------------
|   0 | SELECT STATEMENT             |             |   10 |   520 |    5   (0) |
|   1 |  NESTED LOOPS                |             |   10 |   520 |    5   (0) |
|   2 |   TABLE ACCESS BY INDEX ROWID| EMPLOYEES   |   10 |   260 |    3   (0) |
|   3 |    INDEX RANGE SCAN          | EMP_SAL_IDX |   10 |       |    2   (0) |
|   4 |   TABLE ACCESS BY INDEX ROWID| DEPARTMENTS |   10 |   260 |    2   (0) |
|   5 |    INDEX UNIQUE SCAN         | DEPT_PK     |   10 |       |    1   (0) |
--------------------------------------------------------------------------------
```

Key columns:
- **Operation**: What Oracle does (e.g., TABLE ACCESS, INDEX SCAN, JOIN)
- **Name**: Table or index name
- **Rows**: Estimated rows
- **Cost**: Relative cost (lower is generally better)

---

## 4. Using `EXPLAIN PLAN` Without `STATEMENT_ID`

You can omit `STATEMENT_ID` and just use the default:

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

---

## 5. Comparing `EXPLAIN PLAN` vs Actual Execution

- `EXPLAIN PLAN` shows the **optimizer’s estimate**.
- The **actual** plan used at runtime can differ due to:
  - Bind variable values
  - Different statistics
  - Adaptive plans, etc.

To see the **actual** plan after running a query:

```sql
-- Run the query
SELECT * FROM employees WHERE department_id = 10;

-- Then check the last cursor’s actual plan
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(
    sql_id => '<SQL_ID_FROM_V$SQL>',
    format => 'ALLSTATS LAST'
));
```

This shows actual row counts and execution statistics.

---

## 6. Example: Input/Output Flow

**Input SQL:**

```sql
EXPLAIN PLAN FOR
SELECT e.employee_id, e.first_name
FROM   employees e
WHERE  e.salary > 10000
AND    e.department_id = 50;
```

**Output Plan (simplified):**

```text
--------------------------------------------------------------------------------
| Id  | Operation                    | Name        | Rows | Cost |
--------------------------------------------------------------------------------
|   0 | SELECT STATEMENT             |             |    5 |    3 |
|   1 |  TABLE ACCESS BY INDEX ROWID | EMPLOYEES   |    5 |    3 |
|   2 |   INDEX RANGE SCAN           | EMP_SAL_DEPT_IDX | 5 | 2 |
--------------------------------------------------------------------------------
```

Interpretation:
- Oracle uses an index (`EMP_SAL_DEPT_IDX`) to find rows by `salary` and `department_id`.
- Then accesses the table rows via ROWID.

---

## 7. Key Points to Remember

- `EXPLAIN PLAN`:
  - Does not execute the query.
  - Stores the plan in `PLAN_TABLE`.
  - Is used with `DBMS_XPLAN.DISPLAY` to view the plan.
- For **real** runtime behavior, use `DBMS_XPLAN.DISPLAY_CURSOR` after executing the query.
- Use plans to:
  - Diagnose performance issues.
  - Check if indexes are used.
  - Understand join order and methods.
