
Here’s a concise summary of the key ideas around **DBMS_SCHEDULER vs DBMS_JOB** in Oracle, plus example code.

---

## 1. Purpose of Each

- **DBMS_JOB**
  - Older, legacy job scheduling package.
  - Simple, limited functionality.
  - Still supported for backward compatibility, but not recommended for new development.

- **DBMS_SCHEDULER**
  - Newer, more powerful and flexible scheduler.
  - Supports complex schedules, job classes, windows, resource management, file watchers, chains, etc.
  - Recommended for all new scheduling work.

---

## 2. Main Differences (High-Level)

- **Flexibility**
  - DBMS_JOB: Basic interval-based scheduling.
  - DBMS_SCHEDULER: Calendaring expressions, time zones, windows, job chains, event-based jobs.

- **Management**
  - DBMS_JOB: Jobs stored in `USER_JOBS`, `DBA_JOBS`.
  - DBMS_SCHEDULER: Jobs stored in `*_SCHEDULER_JOBS` views; more metadata and control.

- **Features**
  - DBMS_JOB: Run PL/SQL blocks only.
  - DBMS_SCHEDULER: Run PL/SQL, stored procedures, external executables, scripts, etc.

- **Control**
  - DBMS_SCHEDULER supports:
    - Job classes
    - Resource consumer groups
    - Windows (time-based resource plans)
    - Job chains (workflow of multiple steps)
    - Event-based jobs (file arrival, AQ messages, etc.)

---

## 3. Basic Usage Examples

### 3.1 DBMS_JOB Example

**Goal:** Run a PL/SQL procedure every day at 2 AM.

```sql
-- Procedure to be scheduled
CREATE OR REPLACE PROCEDURE p_daily_task IS
BEGIN
  INSERT INTO log_table (log_date, message)
  VALUES (SYSDATE, 'Daily task executed');
  COMMIT;
END;
/

DECLARE
  l_job NUMBER;
BEGIN
  DBMS_JOB.SUBMIT(
    job       => l_job,
    what      => 'p_daily_task;',
    next_date => TRUNC(SYSDATE + 1) + 2/24,  -- tomorrow at 02:00
    interval  => 'TRUNC(SYSDATE + 1) + 2/24' -- every day at 02:00
  );
  COMMIT;
END;
/
```

**Check job:**

```sql
SELECT job, what, next_date, interval
FROM   user_jobs;
```

**Sample output:**

| JOB | WHAT           | NEXT_DATE           | INTERVAL                               |
|-----|----------------|---------------------|----------------------------------------|
|  42 | p_daily_task;  | 2026-05-21 02:00:00 | TRUNC(SYSDATE + 1) + 2/24              |

---

### 3.2 DBMS_SCHEDULER Example (Basic)

**Goal:** Same task, but using DBMS_SCHEDULER.

```sql
BEGIN
  DBMS_SCHEDULER.CREATE_JOB(
    job_name        => 'job_daily_task',
    job_type        => 'STORED_PROCEDURE',
    job_action      => 'P_DAILY_TASK',
    start_date      => TRUNC(SYSDATE + 1) + 2/24, -- tomorrow 02:00
    repeat_interval => 'FREQ=DAILY;BYHOUR=2;BYMINUTE=0;BYSECOND=0',
    enabled         => TRUE,
    comments        => 'Daily task at 2 AM'
  );
END;
/
```

**Check job:**

```sql
SELECT job_name, state, start_date, repeat_interval
FROM   user_scheduler_jobs
WHERE  job_name = 'JOB_DAILY_TASK';
```

**Sample output:**

| JOB_NAME        | STATE   | START_DATE           | REPEAT_INTERVAL                               |
|-----------------|---------|----------------------|-----------------------------------------------|
| JOB_DAILY_TASK  | SCHEDULED | 2026-05-21 02:00:00 | FREQ=DAILY;BYHOUR=2;BYMINUTE=0;BYSECOND=0    |

## 4. More Advanced DBMS_SCHEDULER Features

### 4.1 Running an External Script

```sql
BEGIN
  DBMS_SCHEDULER.CREATE_JOB(
    job_name        => 'job_run_shell_script',
    job_type        => 'EXECUTABLE',
    job_action      => '/usr/local/bin/myscript.sh',
    start_date      => SYSTIMESTAMP,
    repeat_interval => 'FREQ=HOURLY;INTERVAL=1',
    enabled         => TRUE
  );
END;
/
```

### 4.2 Job Chain (Multiple Steps)

```sql
BEGIN
  -- Create chain
  DBMS_SCHEDULER.CREATE_CHAIN(
    chain_name => 'chain_etl',
    rule_set_name => NULL,
    evaluation_interval => NULL
  );

  -- Add steps
  DBMS_SCHEDULER.DEFINE_CHAIN_STEP('chain_etl', 'step_extract', 'P_EXTRACT');
  DBMS_SCHEDULER.DEFINE_CHAIN_STEP('chain_etl', 'step_transform', 'P_TRANSFORM');
  DBMS_SCHEDULER.DEFINE_CHAIN_STEP('chain_etl', 'step_load', 'P_LOAD');

  -- Define rules: extract -> transform -> load
  DBMS_SCHEDULER.DEFINE_CHAIN_RULE(
    chain_name => 'chain_etl',
    condition  => 'TRUE',
    action     => 'START step_extract'
  );

  DBMS_SCHEDULER.DEFINE_CHAIN_RULE(
    chain_name => 'chain_etl',
    condition  => 'step_extract COMPLETED',
    action     => 'START step_transform'
  );

  DBMS_SCHEDULER.DEFINE_CHAIN_RULE(
    chain_name => 'chain_etl',
    condition  => 'step_transform COMPLETED',
    action     => 'START step_load'
  );

  DBMS_SCHEDULER.ENABLE('chain_etl');

  -- Create a job to run the chain
  DBMS_SCHEDULER.CREATE_JOB(
    job_name        => 'job_chain_etl',
    job_type        => 'CHAIN',
    job_action      => 'chain_etl',
    start_date      => SYSTIMESTAMP,
    enabled         => TRUE
  );
END;
/
```

---

## 5. When to Use Which

- Use **DBMS_SCHEDULER**:
  - For new applications.
  - When you need complex schedules, external programs, chains, or resource control.
- Use **DBMS_JOB**:
  - Only for maintaining legacy systems where it’s already heavily used and migration is not yet possible.
