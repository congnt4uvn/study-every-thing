
Here’s a concise summary and some practical examples around Oracle security, data masking, and redaction.

---

## Key Ideas

1. **Goal**: Protect sensitive data (e.g., SSN, credit card, salary) so that:
   - It is hidden or obfuscated for certain users.
   - It may still be visible to privileged users (e.g., admins).

2. **Two Main Approaches**:
   - **Data Masking** (usually done in non‑production):
     - Physically changes data in the database (e.g., scramble or replace real values).
     - Used to create safe test/dev copies of production data.
   - **Data Redaction** (at runtime, in production):
     - Does *not* change stored data.
     - Masks data only in query results, based on user/role/policy.

3. **User/Role-Based Control**:
   - Different users see different versions of the same data.
   - Policies can be based on:
     - User name
     - Role
     - Application context
     - IP, time of day, etc.

---

## Example: Data Redaction (Oracle 12c+)

Use `DBMS_REDACT` to define redaction policies.

### 1. Setup Example Table

```sql
CREATE TABLE customers (
    customer_id   NUMBER PRIMARY KEY,
    name          VARCHAR2(100),
    ssn           VARCHAR2(11),
    credit_card   VARCHAR2(16),
    salary        NUMBER
);

INSERT INTO customers VALUES (1, 'Alice', '123-45-6789', '4111111111111111', 90000);
INSERT INTO customers VALUES (2, 'Bob',   '987-65-4321', '5555555555554444', 60000);
COMMIT;
```

### 2. Create Users

```sql
-- Privileged user (can see full data)
CREATE USER app_admin IDENTIFIED BY adminpwd;
GRANT CONNECT, RESOURCE TO app_admin;

-- Restricted user (should see masked data)
CREATE USER app_user IDENTIFIED BY userpwd;
GRANT CONNECT TO app_user;

GRANT SELECT ON customers TO app_admin, app_user;
```

### 3. Add Redaction Policy

Mask SSN and credit card for `APP_USER`, but not for `APP_ADMIN`.

```sql
BEGIN
  DBMS_REDACT.ADD_POLICY(
    object_schema   => 'YOUR_SCHEMA',
    object_name     => 'CUSTOMERS',
    column_name     => 'SSN',
    policy_name     => 'SSN_REDACT_POLICY',
    function_type   => DBMS_REDACT.FULL,  -- full masking
    expression      => 'SYS_CONTEXT(''USERENV'',''SESSION_USER'') = ''APP_USER'''
  );

  DBMS_REDACT.ADD_POLICY(
    object_schema   => 'YOUR_SCHEMA',
    object_name     => 'CUSTOMERS',
    column_name     => 'CREDIT_CARD',
    policy_name     => 'CC_REDACT_POLICY',
    function_type   => DBMS_REDACT.PARTIAL, -- partial masking
    function_parameters => '1,12,''X''',    -- mask first 12 chars with 'X'
    expression      => 'SYS_CONTEXT(''USERENV'',''SESSION_USER'') = ''APP_USER'''
  );
END;
/
```

### 4. Query Results (Redaction in Action)

**As `APP_ADMIN` (full access):**

```sql
-- Login as APP_ADMIN
SELECT customer_id, name, ssn, credit_card, salary
FROM your_schema.customers;
```

**Output:**

| CUSTOMER_ID | NAME  | SSN         | CREDIT_CARD       | SALARY |
|------------|-------|-------------|-------------------|--------|
| 1          | Alice | 123-45-6789 | 4111111111111111  | 90000  |
| 2          | Bob   | 987-65-4321 | 5555555555554444  | 60000  |

---

**As `APP_USER` (restricted):**

```sql
-- Login as APP_USER
SELECT customer_id, name, ssn, credit_card, salary
FROM your_schema.customers;
```

**Output (redacted):**

| CUSTOMER_ID | NAME  | SSN         | CREDIT_CARD       | SALARY |
|------------|-------|-------------|-------------------|--------|
| 1          | Alice | 000-00-0000 | XXXXXXXXXXXX1111  | 90000  |
| 2          | Bob   | 000-00-0000 | XXXXXXXXXXXX4444  | 60000  |

Note: Actual mask format depends on the redaction function type and parameters.

## Example: Data Masking (High-Level)

Oracle Data Masking is usually done with **Enterprise Manager** or **Data Masking Pack**, but conceptually:

- You define a **masking format** (e.g., random SSN, random credit card).
- You run a masking job on a copy of production.
- The data in the table is permanently changed.

A simple manual masking example (not using the pack):

```sql
UPDATE customers
SET ssn = TO_CHAR(TRUNC(DBMS_RANDOM.VALUE(100000000, 999999999))),
    credit_card = TO_CHAR(TRUNC(DBMS_RANDOM.VALUE(4000000000000000, 4999999999999999)));

COMMIT;
```

After this, even admins see only fake values.

---

## Summary

- **Data Masking**: permanently changes data (used for test/dev).
- **Data Redaction**: masks data at query time based on policies (used in production).
- **User/Role Context**: determines who sees real vs masked data.
- Oracle provides built-in packages (`DBMS_REDACT`, Data Masking Pack) to implement these controls.
