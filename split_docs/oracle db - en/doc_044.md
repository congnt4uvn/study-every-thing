
HereтАЩs a concise summary of the key ideas around **EXPLAIN PLAN & Execution** focusing on **FILTER, SORT, VIEW, MATERIALIZE** in Oracle, with simple examples.

---

## 1. EXPLAIN PLAN тАУ What it is

`EXPLAIN PLAN` shows how Oracle *plans* to execute a SQL statement:  
- Which tables and indexes it will use  
- In what order operations occur  
- What operations (FILTER, SORT, VIEW, etc.) are applied  

You typically see this via:

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

---

## 2. FILTER

**Meaning:** A `FILTER` operation applies a condition to rows coming from its child operations. It doesnтАЩt access data by itself; it just removes rows that donтАЩt match the predicate.

**Example:**

```sql
EXPLAIN PLAN FOR
SELECT *
FROM employees
WHERE department_id = 10
  AND salary > 5000;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Typical plan fragment:**

```text
| Id | Operation          | Name      |
|----|--------------------|-----------|
|  0 | SELECT STATEMENT   |           |
|  1 |  FILTER            |           |
|  2 |   TABLE ACCESS FULL| EMPLOYEES |
```

- `TABLE ACCESS FULL` reads all rows from `EMPLOYEES`.
- `FILTER` applies `department_id = 10 AND salary > 5000`.

**Input/Output conceptually:**

- Input to FILTER: all rows from `EMPLOYEES`
- Output from FILTER: only rows where `department_id = 10` and `salary > 5000`

---

## 3. SORT

**Meaning:** `SORT` operations order or group data. Common variants:
- `SORT ORDER BY` тАУ for `ORDER BY`
- `SORT GROUP BY` тАУ for `GROUP BY`
- `SORT AGGREGATE` тАУ for aggregate functions without GROUP BY

**Example (ORDER BY):**

```sql
EXPLAIN PLAN FOR
SELECT employee_id, salary
FROM employees
WHERE department_id = 10
ORDER BY salary DESC;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Typical plan fragment:**

```text
| Id | Operation           | Name      |
|----|---------------------|-----------|
|  0 | SELECT STATEMENT    |           |
|  1 |  SORT ORDER BY      |           |
|  2 |   TABLE ACCESS FULL | EMPLOYEES |
```

**Input/Output conceptually:**

- Input to SORT: all rows for `department_id = 10`
- Output from SORT: same rows, but ordered by `salary DESC`

**Example (GROUP BY):**

```sql
EXPLAIN PLAN FOR
SELECT department_id, AVG(salary)
FROM employees
GROUP BY department_id;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Plan will show `SORT GROUP BY`.

---

## 4. VIEW

**Meaning:** `VIEW` in the plan represents a logical subquery or view. Oracle treats the subquery as a separate тАЬviewтАЭ node in the plan.

This appears for:
- Real database views
- Inline views (subqueries in the FROM clause)
- Some complex subqueries

**Example (inline view):**

```sql
EXPLAIN PLAN FOR
SELECT v.department_id, v.avg_sal
FROM (
    SELECT department_id, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY department_id
) v
WHERE v.avg_sal > 5000;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Typical plan fragment:**

```text
| Id | Operation           | Name |
|----|---------------------|------|
|  0 | SELECT STATEMENT    |      |
|  1 |  FILTER             |      |
|  2 |   VIEW              |      |
|  3 |    SORT GROUP BY    |      |
|  4 |     TABLE ACCESS FULL|EMPLOYEES|
```

- `VIEW` node corresponds to the inline view `v`.
- Inside the VIEW, Oracle does `SORT GROUP BY` on `EMPLOYEES`.

**Input/Output conceptually:**

- Input to VIEW: rows from its internal operations (here, employees)
- Output from VIEW: aggregated rows (department_id, avg_sal)

---

## 5. MATERIALIZE

**Meaning:** `MATERIALIZE` indicates that Oracle will physically store the intermediate result of a subquery/CTE (Common Table Expression) in temporary storage, then reuse it.  
This is often used with `WITH` clauses or when hints force materialization.

**Example (CTE with MATERIALIZE hint):**

```sql
EXPLAIN PLAN FOR
WITH emp_dept AS (
    SELECT /*+ MATERIALIZE */
           department_id, COUNT(*) AS cnt
    FROM employees
    GROUP BY department_id
)
SELECT e.employee_id, d.cnt
FROM employees e
JOIN emp_dept d
  ON e.department_id = d.department_id;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Typical plan fragment:**

```text
| Id | Operation              | Name      |
|----|------------------------|-----------|
|  0 | SELECT STATEMENT       |           |
|  1 |  NESTED LOOPS          |           |
|  2 |   TABLE ACCESS FULL    | EMPLOYEES |
|  3 |   VIEW                 | EMP_DEPT  |
|  4 |    MATERIALIZE         |           |
|  5 |     SORT GROUP BY      |           |
|  6 |      TABLE ACCESS FULL | EMPLOYEES |
```

- `MATERIALIZE` means the result of the CTE `emp_dept` is computed once and stored.
- Then the outer query joins to that stored result.

**Input/Output conceptually:**

- Input to MATERIALIZE: rows from `EMPLOYEES` (for the CTE)
- Output from MATERIALIZE: a temporary stored set of (department_id, cnt) used by the join

---

## 6. Putting it together

In an execution plan:

- **FILTER**: applies conditions to rows (WHERE, HAVING, join predicates).
- **SORT**: orders or groups rows (ORDER BY, GROUP BY, aggregates).
- **VIEW**: represents a subquery or view as a logical node.
- **MATERIALIZE**: stores intermediate results (often from CTEs/subqueries) for reuse.

Use:

```sql
EXPLAIN PLAN FOR
<your SQL here>;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

to see how these operations appear and understand how Oracle will execute your query.
