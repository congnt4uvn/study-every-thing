
Key idea  
Interviewers want to see that you understand how **database design choices** (schema, indexes, queries, etc.) impact both **performance** and **maintainability** of an Oracle database.

They look for whether you can:

1. **Choose appropriate data structures**
   - Normalize enough to avoid redundancy, but denormalize when needed for performance.
   - Use proper data types (e.g., `NUMBER`, `DATE`, `VARCHAR2` with correct sizes).

2. **Design efficient indexing strategies**
   - Create indexes that support frequent queries.
   - Avoid over-indexing (too many indexes slow down DML).
   - Use composite indexes when queries filter on multiple columns.

3. **Write performant, maintainable SQL/PLSQL**
   - Prefer set-based operations over row-by-row processing.
   - Use clear, readable SQL and PL/SQL with comments.
   - Avoid unnecessary complexity that makes future changes hard.

4. **Consider scalability and future changes**
   - Design tables and constraints so they can evolve.
   - Think about partitioning, archiving, and growth.

### Example 1: Table and Index Design

```sql
-- Table design
CREATE TABLE orders (
    order_id      NUMBER        PRIMARY KEY,
    customer_id   NUMBER        NOT NULL,
    order_date    DATE          NOT NULL,
    status        VARCHAR2(20)  NOT NULL,
    total_amount  NUMBER(10,2)  NOT NULL
);

-- Index to speed up common query: find orders by customer and date
CREATE INDEX idx_orders_cust_date
    ON orders (customer_id, order_date);
```

**Input query (common use case):**
```sql
SELECT order_id, order_date, total_amount
FROM   orders
WHERE  customer_id = :cust_id
AND    order_date  >= :from_date
AND    order_date  <  :to_date;
```

**Expected behavior/output:**
- Oracle uses `idx_orders_cust_date` to quickly find matching rows.
- Query returns only the orders for that customer in the date range.
- Design is easy to understand and maintain (clear column names, simple structure).

### Example 2: Maintainable vs. Hard-to-Maintain Query

**Hard to maintain (poor design):**
```sql
SELECT *
FROM   orders
WHERE  TO_CHAR(order_date, 'YYYYMMDD') = :p_date;
```

- Performance: Index on `order_date` may not be used because of the function on the column.
- Maintainability: Logic is hidden inside `TO_CHAR`, harder to change.

**Better design:**
```sql
SELECT *
FROM   orders
WHERE  order_date >= TO_DATE(:p_date, 'YYYYMMDD')
AND    order_date <  TO_DATE(:p_date, 'YYYYMMDD') + 1;
```

- Performance: Can use index on `order_date`.
- Maintainability: Date logic is explicit and easier to adjust.

---

### Example 3: PL/SQL – Row-by-Row vs. Set-Based

**Less performant, harder to maintain:**
```plsql
FOR rec IN (SELECT order_id FROM orders WHERE status = 'NEW') LOOP
    UPDATE orders
    SET    status = 'PROCESSED'
    WHERE  order_id = rec.order_id;
END LOOP;
```

**Better (set-based, simpler):**
```sql
UPDATE orders
SET    status = 'PROCESSED'
WHERE  status = 'NEW';
```

- Performance: Single set-based update is much faster.
- Maintainability: One statement, easier to read and modify.

---

In summary, they want you to show that you can design Oracle database structures and code that are both **fast** and **easy to change**, and that you understand the trade-offs between performance and maintainability.
