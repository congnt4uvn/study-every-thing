
Key idea:  
When a report query takes ~30 minutes, you need to identify and remove performance bottlenecks in SQL, indexes, and execution plans.

Main points to focus on:

1. **Identify the slow SQL**
   - Find the exact query (or queries) used by the report.
   - Use AWR/ASH/SQL Monitor or application logs to locate the SQL with the highest elapsed time.

   ```sql
   -- Example: find top SQL by elapsed time in AWR (requires Diagnostics Pack)
   SELECT *
   FROM   dba_hist_sqlstat s
   JOIN   dba_hist_sqltext t USING (sql_id)
   WHERE  s.snap_id BETWEEN :start_snap AND :end_snap
   ORDER BY s.elapsed_time_delta DESC
   FETCH FIRST 10 ROWS ONLY;
   ```

2. **Check the execution plan**
   - Look for full table scans on large tables, nested loops on huge rowsets, or bad join orders.
   - Use `EXPLAIN PLAN` or `DBMS_XPLAN.DISPLAY_CURSOR`.

   ```sql
   EXPLAIN PLAN FOR
   SELECT ...
   FROM   big_table bt
   JOIN   other_table ot ON bt.id = ot.id
   WHERE  bt.status = 'ACTIVE';

   SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
   ```

   Example output (simplified):

   ```
   Plan hash value: 123456789

   ---------------------------------------------------------
   | Id | Operation           | Name      | Rows | Cost    |
   ---------------------------------------------------------
   |  0 | SELECT STATEMENT    |           |      | 100000  |
   |  1 |  HASH JOIN          |           | 1M   | 100000  |
   |  2 |   TABLE ACCESS FULL | BIG_TABLE | 1M   |  80000  |
   |  3 |   TABLE ACCESS FULL | OTHER_T   | 1M   |  20000  |
   ---------------------------------------------------------
   ```

3. **Check and add appropriate indexes**
   - Index columns used in `WHERE`, `JOIN`, and `ORDER BY` clauses if they are selective.
   - Avoid over-indexing; focus on predicates that filter a lot of rows.

   ```sql
   -- Example: add index to support a common filter
   CREATE INDEX idx_big_table_status_date
       ON big_table (status, created_date);
   ```

   After adding the index, re-check the plan:

   ```sql
   SELECT * 
   FROM   big_table
   WHERE  status = 'ACTIVE'
   AND    created_date >= DATE '2024-01-01';
   ```

   Expected plan change (simplified):

   ```
   | Id | Operation                   | Name                    |
   ------------------------------------------------------------
   |  0 | SELECT STATEMENT            |                         |
   |  1 |  TABLE ACCESS BY INDEX ROWID| BIG_TABLE               |
   |  2 |   INDEX RANGE SCAN          | IDX_BIG_TABLE_STATUS_DATE |
   ```

4. **Rewrite or simplify the SQL**
   - Remove unnecessary subqueries, functions on indexed columns, and `SELECT *`.
   - Replace correlated subqueries with joins or analytic functions when appropriate.

   Example: correlated subquery (slow):

   ```sql
   SELECT c.customer_id,
          c.name,
          (SELECT MAX(o.order_date)
           FROM   orders o
           WHERE  o.customer_id = c.customer_id) AS last_order_date
   FROM   customers c;
   ```

   Rewritten with analytic function (often faster):

   ```sql
   SELECT DISTINCT customer_id, name, last_order_date
   FROM (
       SELECT c.customer_id,
              c.name,
              MAX(o.order_date) OVER (PARTITION BY c.customer_id) AS last_order_date
       FROM   customers c
       LEFT JOIN orders o
              ON o.customer_id = c.customer_id
   );
   ```

5. **Use proper statistics**
   - Ensure table and index statistics are up to date so the optimizer can choose good plans.

   ```sql
   BEGIN
     DBMS_STATS.GATHER_TABLE_STATS(
       ownname          => 'REPORT_SCHEMA',
       tabname          => 'BIG_TABLE',
       estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
       cascade          => TRUE
     );
   END;
   /
   ```

6. **Consider partitioning and parallelism**
   - Partition large tables by date or key to reduce scanned data.
   - Use parallel query for large, read-only reporting workloads.

   ```sql
   -- Example: parallel hint
   SELECT /*+ PARALLEL(bt 4) */
          ...
   FROM   big_table bt
   WHERE  bt.created_date >= DATE '2024-01-01';
   ```

7. **Measure before and after**
   - Compare elapsed time, logical reads, and CPU before and after changes.

   Example using `SET TIMING ON` in SQL*Plus:

   ```sql
   SET TIMING ON
   SELECT ... -- report query
   FROM   ...;
   ```

   Output (simplified):

   - Before tuning: `Elapsed: 00:30:12.34`
   - After tuning:  `Elapsed: 00:02:05.67`

Summary:  
To improve a 30‑minute report, you (1) find the slow SQL, (2) inspect and fix the execution plan via indexes and statistics, (3) rewrite inefficient SQL patterns, and (4) optionally use partitioning/parallelism, always validating improvements with measured timings.
