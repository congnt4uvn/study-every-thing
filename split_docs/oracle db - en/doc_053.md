
Here’s a concise summary of the topic “SQL Tuning Techniques: Bind variables vs literals (and bind peeking)” with examples.

---

## 1. Bind Variables vs Literals

### Literals
- SQL text contains actual values.
- Each different value usually produces a different SQL statement in the shared pool.
- Can cause:
  - **Hard parsing** for each distinct value.
  - **Shared pool bloat** (many similar SQLs).
  - Higher CPU usage.

**Example (literals):**
```sql
SELECT *
FROM orders
WHERE customer_id = 101;
```

Another query:
```sql
SELECT *
FROM orders
WHERE customer_id = 102;
```

Even though they are logically the same, Oracle treats them as different SQL statements because the text is different.

---

### Bind Variables
- Use placeholders instead of hard-coded values.
- Same SQL text reused with different values.
- Reduces:
  - Number of hard parses.
  - Shared pool memory usage.
  - CPU overhead.

**Example (bind variables):**
```sql
SELECT *
FROM orders
WHERE customer_id = :cust_id;
```

**Inputs/Outputs example (in a client program):**

```plsql
-- Input
VARIABLE cust_id NUMBER;
EXEC :cust_id := 101;

-- Execution
SELECT *
FROM orders
WHERE customer_id = :cust_id;

-- Output (example)
-- ORDER_ID | CUSTOMER_ID | ORDER_DATE
-- 1001     | 101         | 2024-01-10
-- 1005     | 101         | 2024-02-03
```

If you later set `:cust_id := 102`, the same parsed statement is reused.

---

## 2. Performance Impact

**Using literals:**
- Many similar SQLs:
  ```sql
  SELECT * FROM orders WHERE customer_id = 101;
  SELECT * FROM orders WHERE customer_id = 102;
  SELECT * FROM orders WHERE customer_id = 103;
  ```
- Oracle must:
  - Parse each statement.
  - Generate/lookup a separate execution plan.
- Bad for high-throughput OLTP systems.

**Using bind variables:**
- One SQL:
  ```sql
  SELECT * FROM orders WHERE customer_id = :cust_id;
  ```
- Many executions with different values.
- One parse, many executions → better scalability.

---

## 3. Bind Peeking

### What is Bind Peeking?
- When a statement with bind variables is parsed, Oracle may **peek** at the actual bind values used in that first execution.
- It uses those values to estimate cardinality and choose an execution plan.
- That plan is then reused for later executions, even if later bind values are very different.

### Potential Problem
- If data distribution is skewed, one plan may not be good for all values.

**Example scenario:**
```sql
SELECT *
FROM orders
WHERE customer_id = :cust_id;
```

- Suppose:
  - `customer_id = 101` has 1,000,000 rows.
  - Most other customers have only 10 rows.

1. First execution:
   - `:cust_id = 101`
   - Optimizer peeks at 101, sees many rows, chooses a **full table scan**.
   - Plan is stored and reused.

2. Later execution:
   - `:cust_id = 202`
   - Only 10 rows match, but Oracle still uses the full table scan plan chosen for 101.
   - This is inefficient; an index range scan would be better.

This mismatch is a **bind peeking** issue: one plan for all values, even when data is skewed.

---

## 4. Techniques to Handle Bind Peeking Issues

1. **Adaptive Cursor Sharing (newer Oracle versions)**
   - Oracle can create multiple child cursors with different plans for different bind value ranges.
   - Helps when data is skewed.

2. **Use Histograms**
   - Collect statistics with histograms on skewed columns.
   - Helps the optimizer understand data distribution.

   ```sql
   BEGIN
     DBMS_STATS.GATHER_TABLE_STATS(
       ownname          => 'HR',
       tabname          => 'ORDERS',
       method_opt       => 'FOR COLUMNS customer_id SIZE 254'
     );
   END;
   /
   ```

3. **Use literals selectively**
   - For highly skewed predicates where you really need different plans per value, you might use literals instead of binds.
   - But this can increase parsing overhead, so use carefully.

4. **Hints or SQL Profiles/Baselines**
   - Force or stabilize a good plan when necessary.

---

## 5. Summary of Key Points

- **Bind variables**:
  - Reduce hard parses and shared pool usage.
  - Improve scalability and performance in OLTP systems.
- **Literals**:
  - Can cause many similar SQLs and high parse overhead.
  - But allow the optimizer to choose different plans per value.
- **Bind peeking**:
  - Oracle peeks at bind values during first parse to choose a plan.
  - One plan may be reused for all values, which can be bad with skewed data.
- **Mitigation**:
  - Adaptive cursor sharing, histograms, careful use of literals, and plan management tools.
