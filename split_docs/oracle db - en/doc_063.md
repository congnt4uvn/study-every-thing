
Here’s a concise summary of the topic “Relational Design: Surrogate vs Natural Keys,” with examples.

---

## 1. What are Natural Keys?

**Natural key**: A primary key that comes from real-world data and has meaning in the business domain.

**Examples**  
- `SSN` for a person (in some systems)  
- `email` for a user  
- `VIN` for a car  
- `ISBN` for a book  

**Pros**
- Already meaningful to users.
- Sometimes unique and stable enough.
- Can reduce joins if the natural key is what you search by anyway.

**Cons**
- Can change (email, last name, phone number, etc.).
- May be large or composite (e.g., `(country_code, passport_number)`).
- Business rules can change (e.g., what is considered unique).

**Example table with a natural key**

```sql
CREATE TABLE Customer (
    email        VARCHAR2(255) PRIMARY KEY,
    first_name   VARCHAR2(100),
    last_name    VARCHAR2(100),
    created_at   DATE DEFAULT SYSDATE
);
```

**Insert / Select**

```sql
INSERT INTO Customer (email, first_name, last_name)
VALUES ('alice@example.com', 'Alice', 'Smith');

SELECT * FROM Customer
WHERE email = 'alice@example.com';
```

---

## 2. What are Surrogate Keys?

**Surrogate key**: A primary key with no business meaning, usually an auto-generated numeric ID.

**Examples**
- `customer_id NUMBER` generated by a sequence
- `order_id` as a numeric identity
- UUID-based keys

**Pros**
- Stable: doesn’t change when business data changes.
- Simple, short, and efficient for joins and indexing.
- Decouples database design from business rules.

**Cons**
- Not meaningful to users.
- You still need unique constraints on natural attributes to prevent duplicates.
- Can hide data-quality issues if misused.

**Example table with a surrogate key (Oracle)**

```sql
CREATE TABLE Customer (
    customer_id  NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email        VARCHAR2(255) NOT NULL,
    first_name   VARCHAR2(100),
    last_name    VARCHAR2(100),
    created_at   DATE DEFAULT SYSDATE,
    CONSTRAINT uq_customer_email UNIQUE (email)
);
```

**Insert / Select**

```sql
INSERT INTO Customer (email, first_name, last_name)
VALUES ('alice@example.com', 'Alice', 'Smith')
RETURNING customer_id INTO :new_id;

-- Example output:
-- :new_id = 1

SELECT customer_id, email
FROM Customer
WHERE customer_id = 1;
```

---

## 3. Design Trade-offs

**When to prefer natural keys**
- The natural key is:
  - Truly stable (e.g., ISO country code, ISO currency code).
  - Short and simple.
  - Guaranteed unique by external standards.

**When to prefer surrogate keys**
- Natural key can change (email, phone, name).
- Natural key is long or composite.
- Business rules around uniqueness may change.
- You want consistent design across many tables.

**Typical pattern in Oracle**

1. Use a **surrogate primary key** (e.g., `id`).
2. Add **unique constraints** on important natural attributes.

```sql
CREATE TABLE Product (
    product_id   NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    sku          VARCHAR2(50) NOT NULL,
    name         VARCHAR2(200) NOT NULL,
    CONSTRAINT uq_product_sku UNIQUE (sku)
);
```

---

## 4. Foreign Keys: Surrogate vs Natural

**With surrogate keys**

```sql
CREATE TABLE Orders (
    order_id     NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_id  NUMBER NOT NULL,
    order_date   DATE DEFAULT SYSDATE,
    CONSTRAINT fk_orders_customer
        FOREIGN KEY (customer_id)
        REFERENCES Customer(customer_id)
);
```

**Insert / Select**

```sql
INSERT INTO Orders (customer_id)
VALUES (1)
RETURNING order_id INTO :order_id;

-- :order_id = 1001

SELECT o.order_id, c.email
FROM Orders o
JOIN Customer c ON c.customer_id = o.customer_id
WHERE o.order_id = 1001;
```

**With natural keys as foreign keys** (less common in large systems):

```sql
CREATE TABLE Orders (
    order_id     NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_email VARCHAR2(255) NOT NULL,
    order_date   DATE DEFAULT SYSDATE,
    CONSTRAINT fk_orders_customer_email
        FOREIGN KEY (customer_email)
        REFERENCES Customer(email)
);
```

---

## 5. Practical Guideline

- In most Oracle-based enterprise systems:
  - Use **surrogate keys** as primary keys.
  - Enforce **business uniqueness** with **unique constraints** on natural attributes.
- Use **natural keys as primary keys** only when:
  - They are standardized, short, and extremely stable (e.g., ISO codes).

This balances performance, flexibility, and data integrity while keeping the schema resilient to changing business rules.
