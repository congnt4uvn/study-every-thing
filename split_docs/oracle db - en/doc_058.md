
Here’s a concise, self‑contained summary of **SQL tuning with statistics in Oracle**, with example code and sample outputs.

---

## 1. Why Statistics Matter

Oracle’s **Cost-Based Optimizer (CBO)** relies on **statistics** to choose the best execution plan.  
If statistics are:

- Missing  
- Stale (out of date)  
- Inaccurate  

then the optimizer may choose poor plans (full scans instead of index scans, wrong join methods, etc.), causing slow SQL.

---

## 2. Types of Statistics

Key types used for SQL tuning:

1. **Table statistics**
   - Number of rows
   - Number of blocks
   - Average row length

2. **Column statistics**
   - Number of distinct values (NDV)
   - Null count
   - Data distribution (histograms)

3. **Index statistics**
   - Leaf blocks
   - Clustering factor
   - Distinct keys

4. **System statistics**
   - CPU speed
   - I/O performance

---

## 3. Gathering Statistics

Use `DBMS_STATS` (recommended) instead of old `ANALYZE` command.

### 3.1 Gather Table and Column Statistics

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',
    tabname          => 'EMPLOYEES',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE  -- also gather index stats
  );
END;
/
```

**Effect:**  
- Updates stats for `HR.EMPLOYEES` and its indexes.  
- Lets optimizer know current row counts, data distribution, etc.

---

### 3.2 Gather Schema Statistics

```sql
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(
    ownname          => 'HR',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    options          => 'GATHER AUTO'
  );
END;
/
```

**Effect:**  
- Gathers stats only where Oracle thinks they are missing or stale.

---

### 3.3 Gather System Statistics

```sql
BEGIN
  DBMS_STATS.GATHER_SYSTEM_STATS('NOWORKLOAD');
END;
/
```

**Effect:**  
- Helps optimizer estimate CPU vs I/O cost more accurately.

---

## 4. Checking Statistics

### 4.1 Check Table Statistics

```sql
SELECT table_name,
       num_rows,
       blocks,
       last_analyzed
FROM   dba_tab_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

**Sample output:**

| TABLE_NAME | NUM_ROWS | BLOCKS | LAST_ANALYZED        |
|-----------|----------|--------|----------------------|
| EMPLOYEES | 107      | 8      | 2026-05-20 10:15:32  |

---

### 4.2 Check Column Statistics

```sql
SELECT column_name,
       num_distinct,
       num_nulls,
       histogram
FROM   dba_tab_col_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

**Sample output:**

| COLUMN_NAME | NUM_DISTINCT | NUM_NULLS | HISTOGRAM   |
|-------------|--------------|-----------|-------------|
| EMPLOYEE_ID | 107          | 0         | NONE        |
| JOB_ID      | 19           | 0         | FREQUENCY   |
| MANAGER_ID  | 18           | 1         | HEIGHT BALANCED |

## 5. Histograms and Skewed Data

Histograms help when data is **skewed** (some values are much more common than others).

Example: `JOB_ID` has one very common value and many rare ones.

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname    => 'HR',
    tabname    => 'EMPLOYEES',
    method_opt => 'FOR COLUMNS JOB_ID SIZE 254'
  );
END;
/
```

**Effect:**  
- Creates a detailed histogram on `JOB_ID`.  
- Optimizer can estimate selectivity better for predicates like:

```sql
SELECT * 
FROM   hr.employees
WHERE  job_id = 'AD_PRES';
```

---

## 6. Stale or Missing Statistics

Oracle tracks whether stats are stale.

```sql
SELECT table_name,
       stale_stats
FROM   dba_tab_statistics
WHERE  owner = 'HR';
```

**Sample output:**

| TABLE_NAME | STALE_STATS |
|-----------|-------------|
| EMPLOYEES | NO          |
| JOBS      | YES         |

If `STALE_STATS = 'YES'`, gather stats again.

## 7. Locking and Restoring Statistics

### 7.1 Lock Statistics (prevent auto changes)

```sql
BEGIN
  DBMS_STATS.LOCK_TABLE_STATS('HR', 'EMPLOYEES');
END;
/
```

### 7.2 Unlock Statistics

```sql
BEGIN
  DBMS_STATS.UNLOCK_TABLE_STATS('HR', 'EMPLOYEES');
END;
/
```

### 7.3 Export / Import Statistics (for backup or migration)

```sql
-- Export stats to a stats table
BEGIN
  DBMS_STATS.CREATE_STAT_TABLE('HR', 'HR_STATS_BACKUP');
  DBMS_STATS.EXPORT_SCHEMA_STATS(
    ownname   => 'HR',
    stattab   => 'HR_STATS_BACKUP'
  );
END;
/

-- Later, restore them
BEGIN
  DBMS_STATS.IMPORT_SCHEMA_STATS(
    ownname   => 'HR',
    stattab   => 'HR_STATS_BACKUP'
  );
END;
/
```

## 8. Using Statistics for SQL Tuning

1. **Identify slow SQL** (AWR, ASH, SQL Monitor).
2. **Check execution plan** (`EXPLAIN PLAN` or `DBMS_XPLAN.DISPLAY_CURSOR`).
3. **Verify statistics** on involved tables/columns/indexes.
4. **Gather or refine statistics** (especially histograms for skewed columns).
5. **Compare plans before/after** stats changes.

Example:

```sql
EXPLAIN PLAN FOR
SELECT * 
FROM   hr.employees
WHERE  department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Before good stats:**  
- Plan might show `TABLE ACCESS FULL HR.EMPLOYEES`.

**After good stats + index:**  
- Plan might show `INDEX RANGE SCAN` on an index on `DEPARTMENT_ID`.

---

**Core idea:**  
Accurate, up‑to‑date statistics are essential for the Oracle optimizer to choose efficient execution plans. SQL tuning often starts with verifying and correcting statistics using `DBMS_STATS`.
