
Here’s a concise, high‑level summary of Oracle Data Guard, plus simple code examples.

---

## Key Ideas: Oracle Data Guard (Conceptual)

**Purpose:**  
Oracle Data Guard provides **high availability**, **data protection**, and **disaster recovery** for Oracle databases.

**Core Concept:**  
You have:

- **Primary database** – the main production database where all changes occur.
- **Standby database(s)** – one or more copies of the primary that receive and apply redo data.

If the primary fails, a standby can be **switched over** or **failed over** to become the new primary.

---

## Main Components & Concepts

1. **Redo Transport**
   - Primary database sends **redo** (change records) to standby.
   - Transport can be:
     - **Synchronous (SYNC)** – zero or near-zero data loss, but can add latency.
     - **Asynchronous (ASYNC)** – less impact on performance, but possible small data loss.

2. **Apply Services**
   - Standby applies redo to keep itself in sync:
     - **Physical standby** – applies redo at block level (exact copy of primary).
     - **Logical standby** – applies SQL-level changes (can have different structures, indexes, etc.).

3. **Protection Modes**
   - **Maximum Protection** – no data loss allowed; primary stops if standby not available.
   - **Maximum Availability** – aims for no data loss; primary continues if standby temporarily unavailable.
   - **Maximum Performance** – best performance; allows some data loss.

4. **Role Transitions**
   - **Switchover** – planned role reversal (primary ↔ standby) with no data loss.
   - **Failover** – unplanned transition when primary is lost; standby becomes new primary.

5. **Automatic Management**
   - Can be managed via:
     - SQL*Plus / SQL commands
     - **Data Guard Broker** (DGMGRL, GUI tools) for easier configuration and monitoring.

## Simple Code Examples

> Note: These are simplified examples to illustrate the concepts, not full production scripts.

### 1. Configure Redo Transport (Primary Side)

**Input (on primary):**
```sql
-- Set the standby destination
ALTER SYSTEM SET LOG_ARCHIVE_DEST_2 =
  'SERVICE=standby_db SYNC AFFIRM
   VALID_FOR=(ONLINE_LOGFILES,PRIMARY_ROLE)
   DB_UNIQUE_NAME=standby_db'
  SCOPE=BOTH;

-- Set the unique name for the primary
ALTER SYSTEM SET DB_UNIQUE_NAME='primary_db' SCOPE=SPFILE;
```

**Conceptual Output:**
- Primary now knows to send redo to the standby service `standby_db` synchronously.
- Standby will receive redo logs for application.

---

### 2. Start Managed Recovery on Physical Standby

**Input (on standby):**
```sql
-- Mount the standby database
STARTUP MOUNT;

-- Start managed recovery (apply redo)
ALTER DATABASE RECOVER MANAGED STANDBY DATABASE DISCONNECT FROM SESSION;
```

**Conceptual Output:**
- Standby begins applying redo from the primary.
- Standby stays in sync (or near sync) with primary.

---

### 3. Switchover (Planned Role Change)

**Step 1 – On Primary:**
```sql
-- Verify switchover status
SELECT SWITCHOVER_STATUS FROM V$DATABASE;

-- If status is 'TO STANDBY', perform switchover
ALTER DATABASE COMMIT TO SWITCHOVER TO STANDBY WITH SESSION SHUTDOWN;
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
```

**Step 2 – On Standby:**
```sql
-- Convert standby to primary
ALTER DATABASE COMMIT TO SWITCHOVER TO PRIMARY;
ALTER DATABASE OPEN;
```

**Conceptual Output:**
- Old primary becomes standby.
- Old standby becomes the new primary.
- No data loss in a clean switchover.

---

### 4. Simple Failover (Unplanned)

**Input (on standby after primary is lost):**
```sql
-- Immediately activate standby as primary
ALTER DATABASE ACTIVATE STANDBY DATABASE;
ALTER DATABASE OPEN;
```

**Conceptual Output:**
- Standby becomes the new primary.
- Some data loss may occur depending on protection mode and last received redo.

---

## Summary

- **Oracle Data Guard** = primary + standby databases with redo transport and apply services.
- Provides **high availability**, **disaster recovery**, and **data protection**.
- Uses **redo transport (SYNC/ASYNC)**, **apply services (physical/logical)**, and **role transitions (switchover/failover)**.
- Protection modes balance **data safety vs performance**.
