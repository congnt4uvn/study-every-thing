
Here’s a concise summary of the topic “Explain Plan & Execution: Full table scan vs index range scan,” with simple examples.

---

## 1. What is an Explain Plan?

`EXPLAIN PLAN` shows how Oracle *plans* to execute a SQL statement:  
- Which access paths it will use (full table scan, index range scan, etc.)  
- Join methods, order of operations, estimated cost, rows, etc.

You typically view it with `DBMS_XPLAN.DISPLAY`.

**Example:**

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Sample output (simplified):**

```text
Plan hash value: 123456789

---------------------------------------------------------
| Id  | Operation                   | Name              |
---------------------------------------------------------
|   0 | SELECT STATEMENT            |                   |
|*  1 |  TABLE ACCESS BY INDEX ROWID| EMPLOYEES         |
|*  2 |   INDEX RANGE SCAN          | EMP_DEPT_ID_IDX   |
---------------------------------------------------------
Predicate Information (identified by operation id):
---------------------------------------------------------
   1 - filter("DEPARTMENT_ID"=10)
   2 - access("DEPARTMENT_ID"=10)
```

This shows Oracle will use an **index range scan** on `EMP_DEPT_ID_IDX`.

---

## 2. Full Table Scan (FTS)

**What it is:**  
Oracle reads *all* blocks of the table sequentially, applying the filter to each row.

**When it happens:**
- No usable index on the filter columns.
- The query needs a large portion of the table (e.g., > ~10–20% of rows, depending on stats).
- The optimizer estimates that scanning the whole table is cheaper than using an index.
- Functions or expressions on indexed columns make the index unusable (unless function-based index exists).

**Example:**

```sql
EXPLAIN PLAN FOR
SELECT * FROM employees WHERE salary > 1000;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

If there is **no index** on `salary`, you might see:

```text
---------------------------------------------------------
| Id  | Operation          | Name                      |
---------------------------------------------------------
|   0 | SELECT STATEMENT   |                           |
|*  1 |  TABLE ACCESS FULL | EMPLOYEES                 |
---------------------------------------------------------
```

**Input:**  
```sql
SELECT * FROM employees WHERE salary > 1000;
```

**Output (logical result):**  
All rows from `employees` where `salary > 1000`, but Oracle reads the entire table to find them.

---

## 3. Index Range Scan

**What it is:**  
Oracle uses an index to read only the index entries that match a range of values, then fetches the corresponding table rows.

**When it happens:**
- There is a B-tree index on the column(s) in the `WHERE` clause.
- The condition is selective (returns relatively few rows).
- Predicates like `=`, `BETWEEN`, `>`, `<`, `>=`, `<=`, or range on leading index column.

**Example:**

Assume an index:

```sql
CREATE INDEX emp_dept_id_idx ON employees(department_id);
```

Query:

```sql
EXPLAIN PLAN FOR
SELECT employee_id, first_name, last_name
FROM employees
WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Possible plan:

```text
---------------------------------------------------------
| Id  | Operation                   | Name              |
---------------------------------------------------------
|   0 | SELECT STATEMENT            |                   |
|*  1 |  TABLE ACCESS BY INDEX ROWID| EMPLOYEES         |
|*  2 |   INDEX RANGE SCAN          | EMP_DEPT_ID_IDX   |
---------------------------------------------------------
```

**Input:**  
```sql
SELECT employee_id, first_name, last_name
FROM employees
WHERE department_id = 10;
```

**Output (logical result):**  
Only employees in department 10, with Oracle reading a small range of index entries instead of the whole table.

---

## 4. Comparing Full Table Scan vs Index Range Scan

**Full Table Scan:**
- Reads all table blocks.
- Efficient for:
  - Very large result sets.
  - Small tables (where scanning all rows is cheap).
- Can use multiblock I/O and benefit from sequential reads.

**Index Range Scan:**
- Reads only relevant index entries + corresponding table rows.
- Efficient for:
  - Highly selective predicates (small subset of rows).
  - Queries that filter on indexed columns.

**Key idea:**  
Oracle’s optimizer chooses between FTS and index range scan based on:
- Table and index statistics (number of rows, distinct values, clustering factor).
- Predicate selectivity.
- Available indexes.
- System parameters (e.g., `db_file_multiblock_read_count`).

## 5. Simple Demo: Forcing Different Plans

### 5.1 Using an index (range scan)

```sql
-- Index on department_id
CREATE INDEX emp_dept_id_idx ON employees(department_id);

EXPLAIN PLAN FOR
SELECT * FROM employees WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

You should see `INDEX RANGE SCAN` if the optimizer thinks it’s selective.

### 5.2 Forcing a full table scan with a hint

```sql
EXPLAIN PLAN FOR
SELECT /*+ FULL(e) */ *
FROM employees e
WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Plan (simplified):

```text
---------------------------------------------------------
| Id  | Operation          | Name                      |
---------------------------------------------------------
|   0 | SELECT STATEMENT   |                           |
|*  1 |  TABLE ACCESS FULL | EMPLOYEES                 |
---------------------------------------------------------
```

## 6. How to Read and Use Explain Plan

1. Run `EXPLAIN PLAN FOR <your query>`.
2. Display with `DBMS_XPLAN.DISPLAY`.
3. Look at:
   - `Operation`: `TABLE ACCESS FULL` vs `INDEX RANGE SCAN`.
   - `Cost`, `Cardinality` (estimated rows).
4. If you see a full table scan where you expect an index:
   - Check if an index exists on the filter columns.
   - Check if functions or implicit conversions are disabling the index.
   - Check statistics (gather with `DBMS_STATS` if outdated).
