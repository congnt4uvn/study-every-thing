**“SQL Tuning Techniques: Stale/missing statistics impact”**

---

## Key Points & Main Ideas

1. **Optimizer statistics are critical**  
   - Oracle’s Cost-Based Optimizer (CBO) relies on table, index, and column statistics (row counts, data distribution, etc.) to choose an execution plan.  
   - Good statistics → better cardinality estimates → better plans.

2. **Missing statistics**  
   - If statistics are missing, Oracle may:
     - Use dynamic sampling (at parse time), or  
     - Fall back to default assumptions.  
   - This often leads to poor join methods, wrong index usage, or full table scans when not appropriate.

3. **Stale statistics**  
   - Statistics become stale when data changes significantly (inserts/updates/deletes) but stats are not refreshed.  
   - Stale stats cause the optimizer to think the table is smaller/larger or more/less selective than it really is, leading to:
     - Wrong join order  
     - Wrong join method (e.g., nested loop vs hash join)  
     - Wrong index choice or no index usage

4. **Symptoms of stale/missing stats**  
   - Sudden performance degradation after data growth or batch loads.  
   - Execution plan changes without code changes.  
   - Large differences between estimated rows and actual rows in execution plans.

5. **Prevention and tuning approach**  
   - Regularly gather statistics on tables and indexes, especially after large data changes.  
   - Use automatic statistics gathering, but override for critical objects when needed.  
   - Monitor for stale stats and fix them proactively.  
   - Compare execution plans before/after stats refresh.

## Useful Views & Checks

### 1. Check if statistics are stale

```sql
SELECT owner, table_name, stale_stats
FROM   dba_tab_statistics
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

**Output (example):**

| OWNER | TABLE_NAME | STALE_STATS |
|-------|------------|------------|
| HR    | EMPLOYEES  | YES        |

`STALE_STATS = YES` indicates stats should be refreshed.

---

### 2. Check when statistics were last gathered

```sql
SELECT owner, table_name, num_rows, last_analyzed
FROM   dba_tables
WHERE  owner = 'HR'
AND    table_name = 'EMPLOYEES';
```

**Output (example):**

| OWNER | TABLE_NAME | NUM_ROWS | LAST_ANALYZED        |
|-------|------------|----------|----------------------|
| HR    | EMPLOYEES  | 100000   | 2024-05-01 10:23:45  |

If `LAST_ANALYZED` is old and data changed a lot, stats may be stale.

## Gathering Statistics

### 3. Gather statistics for a specific table

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname          => 'HR',
    tabname          => 'EMPLOYEES',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    method_opt       => 'FOR ALL COLUMNS SIZE AUTO',
    cascade          => TRUE  -- also gather index stats
  );
END;
/
```

**Effect:**  
- Updates table and index statistics for `HR.EMPLOYEES`.  
- Optimizer will use new stats for future parses.

---

### 4. Gather statistics for a schema

```sql
BEGIN
  DBMS_STATS.GATHER_SCHEMA_STATS(
    ownname          => 'HR',
    estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
    options          => 'GATHER AUTO'
  );
END;
/
```

**Effect:**  
- Gathers stats for objects in schema `HR` that need it (e.g., missing or stale).

## Impact on Execution Plans (Example)

### 5. Query before statistics (missing or stale)

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   hr.employees e
WHERE  e.department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Possible output (simplified):**

| Id | Operation        | Name       | Rows (Est) |
|----|------------------|-----------|-----------|
|  0 | SELECT STATEMENT |           | 100000    |
|  1 | TABLE ACCESS FULL| EMPLOYEES | 100000    |

- Optimizer thinks many rows match, so it chooses a full table scan.

---

### 6. After gathering statistics

```sql
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'HR',
    tabname => 'EMPLOYEES',
    cascade => TRUE
  );
END;
/

EXPLAIN PLAN FOR
SELECT *
FROM   hr.employees e
WHERE  e.department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Possible output (simplified):**

| Id | Operation                   | Name       | Rows (Est) |
|----|-----------------------------|-----------|-----------|
|  0 | SELECT STATEMENT            |           | 100       |
|  1 | TABLE ACCESS BY INDEX ROWID | EMPLOYEES | 100       |
|  2 | INDEX RANGE SCAN            | EMP_DEPT_I| 100       |

- With accurate stats, optimizer estimates fewer rows and uses an index, improving performance.

## Automatic Statistics Gathering

Oracle’s automatic job (e.g., `GATHER_STATS_JOB`) runs in maintenance windows:

```sql
SELECT client_name, status
FROM   dba_autotask_client
WHERE  client_name = 'auto optimizer stats collection';
```

**Output (example):**

| CLIENT_NAME                     | STATUS  |
|---------------------------------|---------|
| auto optimizer stats collection | ENABLED |

- Ensure it’s enabled, but for critical tables, you may still gather stats manually after big data changes.

---

## Summary

- Missing or stale statistics can severely degrade SQL performance by causing bad execution plans.
- Regularly monitor and refresh statistics, especially after large data changes.
- Use `DBMS_STATS` and dictionary views (`DBA_TAB_STATISTICS`, `DBA_TABLES`) to detect and fix stale/missing stats.
- Always compare execution plans before and after statistics changes to confirm the impact.
