
**Summary: `enq: TX - row lock contention` (Oracle)**

This wait event indicates **session-level blocking on row locks**.  
A session is waiting because **another session holds a lock on the same row(s)** and has not yet committed or rolled back.

---

## Key Points

1. **What it is**
   - `enq: TX - row lock contention` = a session is waiting for a **TX (transaction) enqueue**.
   - Typically caused by **two or more sessions updating/deleting the same rows** at the same time.

2. **Why it happens**
   - Session A updates a row and does not commit/rollback.
   - Session B tries to update/delete the same row → must wait on `enq: TX - row lock contention`.
   - Long-running transactions, missing commits, or poor application design can increase this wait.

3. **Impact**
   - Increased response time.
   - Possible “chain” of blocked sessions.
   - Can appear as performance slowdown or “hang” from the application’s perspective.

4. **How to diagnose**
   - Identify blocking and blocked sessions.
   - Check which SQL is holding locks and which SQL is waiting.
   - Look at transaction duration and application logic.

## Basic Diagnostic Queries

### 1. Find sessions waiting on TX row lock

```sql
SELECT sid, serial#, event, wait_class, blocking_session, seconds_in_wait
FROM   v$session
WHERE  event = 'enq: TX - row lock contention';
```

**Example output (simplified):**

| SID | SERIAL# | EVENT                          | BLOCKING_SESSION | SECONDS_IN_WAIT |
|-----|---------|--------------------------------|------------------|-----------------|
| 123 | 4567    | enq: TX - row lock contention | 130              | 120             |

This shows session 123 is waiting on a row lock held by session 130.

---

### 2. See what the blocking session is doing

```sql
-- Replace 130 with the BLOCKING_SESSION from above
SELECT s.sid,
       s.serial#,
       s.username,
       s.status,
       s.logon_time,
       q.sql_text
FROM   v$session s
       LEFT JOIN v$sql q ON s.sql_id = q.sql_id
WHERE  s.sid = 130;
```

**Example output (simplified):**

| SID | SERIAL# | USERNAME | STATUS | SQL_TEXT                          |
|-----|---------|----------|--------|-----------------------------------|
| 130 | 9999    | APPUSER  | ACTIVE | UPDATE orders SET status = :b1... |

This tells you which SQL is holding the lock.

---

### 3. Show locked objects and sessions

```sql
SELECT s.sid,
       s.serial#,
       s.username,
       o.object_name,
       l.locked_mode
FROM   v$locked_object l
       JOIN dba_objects o ON l.object_id = o.object_id
       JOIN v$session s   ON l.session_id = s.sid
ORDER BY s.sid;
```

**Example output (simplified):**

| SID | SERIAL# | USERNAME | OBJECT_NAME | LOCKED_MODE |
|-----|---------|----------|-------------|-------------|
| 130 | 9999    | APPUSER  | ORDERS      | 6           |

`LOCKED_MODE = 6` means exclusive lock (row-level in this context).

## Simple Reproduction Example

### Session 1

```sql
-- Session 1: start transaction and lock a row
UPDATE orders
SET    status = 'PENDING'
WHERE  order_id = 1001;

-- Do NOT commit or rollback yet
-- (e.g., leave the session idle)
```

### Session 2

```sql
-- Session 2: try to update the same row
UPDATE orders
SET    status = 'SHIPPED'
WHERE  order_id = 1001;
```

**Result:**

- Session 2 will **hang** (wait) on `enq: TX - row lock contention` until:
  - Session 1 issues `COMMIT;` or `ROLLBACK;`, or
  - Session 1 is killed and its transaction is rolled back.

---

## How to Reduce / Avoid This Wait

1. **Short transactions**
   - Commit or rollback as soon as logical work is done.
2. **Consistent access patterns**
   - Update rows in a consistent order to avoid deadlocks and reduce contention.
3. **Application design**
   - Avoid “chatty” transactions that hold locks while waiting for user input.
4. **Indexing**
   - Ensure WHERE clauses are selective and use indexes, so fewer rows are locked.
5. **Monitoring**
   - Regularly check for long-running transactions and blocking sessions.
