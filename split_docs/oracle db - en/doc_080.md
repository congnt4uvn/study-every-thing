
Here’s a concise summary and some practical examples around locking & isolation using `V$SESSION`, `V$LOCK`, and `V$SQL` in Oracle.

---

## Key Ideas

1. **Locking & Isolation**
   - Oracle uses locks to protect data consistency and enforce transaction isolation.
   - Common issues: blocking sessions, deadlocks, long-running transactions.

2. **V$SESSION**
   - Shows information about current database sessions.
   - Useful to identify:
     - Who is connected
     - What they are doing
     - Blocking / blocked sessions

3. **V$LOCK**
   - Shows lock information at the database level.
   - Used to see:
     - Which sessions hold locks
     - Which sessions are waiting for locks
     - Types of locks (TM, TX, etc.)

4. **V$SQL**
   - Shows SQL statements currently in the shared pool.
   - Used to:
     - Find the SQL text for a given session
     - Analyze problematic or blocking queries

---

## Common Queries & Examples

### 1. Find Blocking and Blocked Sessions

**Query:**

```sql
SELECT
    s1.sid        AS blocking_sid,
    s1.serial#    AS blocking_serial,
    s1.username   AS blocking_user,
    s2.sid        AS blocked_sid,
    s2.serial#    AS blocked_serial,
    s2.username   AS blocked_user,
    l1.type       AS lock_type,
    l1.id1, l1.id2,
    l1.lmode      AS blocking_lmode,
    l2.request    AS blocked_request
FROM v$lock l1
JOIN v$session s1 ON l1.sid = s1.sid
JOIN v$lock l2 ON l1.id1 = l2.id1
               AND l1.id2 = l2.id2
               AND l1.type = l2.type
JOIN v$session s2 ON l2.sid = s2.sid
WHERE l1.block = 1
  AND l2.request > 0;
```

**Output (example):**

| BLOCKING_SID | BLOCKING_SERIAL | BLOCKING_USER | BLOCKED_SID | BLOCKED_SERIAL | BLOCKED_USER | LOCK_TYPE | ID1  | ID2 | BLOCKING_LMODE | BLOCKED_REQUEST |
|-------------|-----------------|---------------|-------------|----------------|--------------|----------|------|-----|----------------|-----------------|
| 123         | 4567            | APP_USER      | 130         | 8910           | APP_USER     | TX       | 1234 | 0   | 6              | 4               |

Interpretation:
- Session 123 is blocking session 130 on a transaction lock (`TX`).
- `LMODE` and `REQUEST` show what lock modes are held/requested.

---

### 2. See What a Session Is Running (Using V$SESSION + V$SQL)

**Step 1: Find the session**

```sql
SELECT sid, serial#, username, status, machine, program, sql_id
FROM v$session
WHERE username = 'APP_USER';
```

**Example output:**

| SID | SERIAL# | USERNAME | STATUS | MACHINE     | PROGRAM          | SQL_ID       |
|-----|---------|----------|--------|-------------|------------------|--------------|
| 123 | 4567    | APP_USER | ACTIVE | APP-SERVER1 | JDBC Thin Client | 5t9k3u1b2x3y |

**Step 2: Get the SQL text**

```sql
SELECT sql_id, sql_text
FROM v$sql
WHERE sql_id = '5t9k3u1b2x3y';
```

**Example output:**

| SQL_ID       | SQL_TEXT                                      |
|--------------|-----------------------------------------------|
| 5t9k3u1b2x3y | UPDATE orders SET status = 'P' WHERE id = :1  |

### 3. List All Locks and Their Sessions

```sql
SELECT
    s.sid,
    s.serial#,
    s.username,
    l.type,
    l.id1,
    l.id2,
    l.lmode,
    l.request,
    l.block
FROM v$lock l
JOIN v$session s ON l.sid = s.sid
ORDER BY s.sid, l.type;
```

**Example output:**

| SID | SERIAL# | USERNAME | TYPE | ID1  | ID2 | LMODE | REQUEST | BLOCK |
|-----|---------|----------|------|------|-----|-------|---------|-------|
| 123 | 4567    | APP_USER | TX   | 1234 | 0   | 6     | 0       | 1     |
| 130 | 8910    | APP_USER | TX   | 1234 | 0   | 0     | 4       | 0     |

- `LMODE` > 0: lock held.
- `REQUEST` > 0: lock requested (waiting).
- `BLOCK = 1`: this lock is blocking others.

---

### 4. Kill a Blocking Session (Admin)

Once you identify a problematic session (e.g., SID=123, SERIAL#=4567):

```sql
ALTER SYSTEM KILL SESSION '123,4567' IMMEDIATE;
```

---

### 5. Check Isolation-Related Info (Current Transaction)

To see if a session has uncommitted changes:

```sql
SELECT s.sid, s.serial#, s.username, t.xidusn, t.xidslot, t.xidsqn
FROM v$transaction t
JOIN v$session s ON t.ses_addr = s.saddr;
```

This helps correlate active transactions with sessions and locks.

---

These views (`V$SESSION`, `V$LOCK`, `V$SQL`) together let you:
- Detect blocking/blocked sessions
- See what SQL is causing locks
- Take corrective actions (tuning, killing sessions, etc.) while understanding Oracle’s locking and isolation behavior.
