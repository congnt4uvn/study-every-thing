## Key Ideas: Version Control & CI/CD for Oracle Migration Scripts

1. **Migration scripts as source of truth**  
   - All schema changes (DDL) and data fixes (DML) are stored as ordered scripts in version control (e.g., Git).  
   - Each script represents a small, incremental change (migration).

2. **Immutable, ordered migrations**  
   - Scripts are usually named with an increasing version or timestamp, e.g.:  
     - `V001__create_users_table.sql`  
     - `V002__add_status_to_users.sql`  
   - Once applied to production, scripts are not edited; new changes get new scripts.

3. **Environment consistency**  
   - The same migration scripts are applied in dev, test, staging, and production.  
   - This ensures all environments share the same schema version.

4. **Automated CI/CD pipeline**  
   - CI runs migrations against a test Oracle database on each commit/merge.  
   - CD (deployment) runs the same migration set against staging/production.  
   - Failures in migration (syntax errors, conflicts) break the pipeline early.

5. **Schema version tracking**  
   - A dedicated table (e.g., `SCHEMA_VERSION`) in Oracle tracks which migrations have been applied.  
   - The migration tool or custom script checks this table to know what to run next.

6. **Rollback / recovery strategy**  
   - For each migration, you may define a rollback script or a manual recovery plan.  
   - Not all changes are easily reversible (e.g., data loss), so rollbacks must be designed carefully.

## Example: Migration Script Files

### 1. Initial schema creation

**File:** `V001__create_users_table.sql`

```sql
-- V001__create_users_table.sql
CREATE TABLE users (
    id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    username    VARCHAR2(50) NOT NULL UNIQUE,
    email       VARCHAR2(100) NOT NULL UNIQUE,
    created_at  DATE DEFAULT SYSDATE NOT NULL
);
```

**Effect (output in DB):**

```sql
DESC users;

-- Result:
-- Name        Null?    Type
-- ----------  -------  --------------------
-- ID         NOT NULL  NUMBER
-- USERNAME   NOT NULL  VARCHAR2(50)
-- EMAIL      NOT NULL  VARCHAR2(100)
-- CREATED_AT NOT NULL  DATE
```

---

### 2. Add a new column

**File:** `V002__add_status_to_users.sql`

```sql
-- V002__add_status_to_users.sql
ALTER TABLE users
  ADD (status VARCHAR2(20) DEFAULT 'ACTIVE' NOT NULL);
```

**Effect (output in DB):**

```sql
DESC users;

-- Result (new column added):
-- STATUS     NOT NULL  VARCHAR2(20)
```

---

### 3. Data migration / fix

**File:** `V003__normalize_emails.sql`

```sql
-- V003__normalize_emails.sql
UPDATE users
   SET email = LOWER(email)
 WHERE email != LOWER(email);
```

**Effect (output in DB):**

```sql
SELECT id, email FROM users;

-- All emails are now lowercase.
```

## Example: Schema Version Table

**File:** `V000__create_schema_version_table.sql`

```sql
CREATE TABLE schema_version (
    version        NUMBER       NOT NULL,
    description    VARCHAR2(200),
    applied_at     DATE         DEFAULT SYSDATE NOT NULL,
    CONSTRAINT pk_schema_version PRIMARY KEY (version)
);
```

When a migration runs, your tool or script inserts a row:

```sql
INSERT INTO schema_version (version, description)
VALUES (1, 'create_users_table');

INSERT INTO schema_version (version, description)
VALUES (2, 'add_status_to_users');

INSERT INTO schema_version (version, description)
VALUES (3, 'normalize_emails');
```

**Check applied versions:**

```sql
SELECT * FROM schema_version ORDER BY version;

-- Example output:
-- VERSION  DESCRIPTION              APPLIED_AT
-- -------  ------------------------ -------------------
-- 1        create_users_table       2026-05-20 10:01:23
-- 2        add_status_to_users      2026-05-20 10:02:10
-- 3        normalize_emails         2026-05-20 10:03:45
```

## Example: Simple CI Step (Shell + SQL*Plus)

```bash
# run_migrations.sh
ORACLE_CONN="user/password@//host:1521/ORCLPDB1"

for file in migrations/V*.sql; do
  echo "Running $file"
  sqlplus -s "$ORACLE_CONN" @"$file" || exit 1
done
```

In CI (e.g., GitHub Actions, GitLab CI, Jenkins), you call:

```bash
./run_migrations.sh
```

If any script fails, the exit code is non‑zero and the pipeline stops.
