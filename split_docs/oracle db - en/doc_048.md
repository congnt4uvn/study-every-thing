
Here’s a concise summary and some practical examples around **index selectivity** and **cardinality** in Oracle.

---

## Key Concepts

### 1. Cardinality
- **Definition**: The number of distinct values in a column.
- **High cardinality**: Many distinct values (e.g., `EMP_ID`, `EMAIL`).
- **Low cardinality**: Few distinct values (e.g., `GENDER`, `STATUS` with Y/N).

Oracle uses cardinality to estimate how many rows a query will return and whether an index is useful.

```sql
-- Check number of distinct values (cardinality) for a column
SELECT COUNT(DISTINCT dept_id) AS distinct_dept
FROM   employees;
```

**Example output:**

| DISTINCT_DEPT |
|---------------|
| 10            |

---

### 2. Index Selectivity
- **Definition**: How well an index distinguishes between rows.
- Often approximated as:  
  `selectivity = distinct_values / total_rows`
- **High selectivity** (close to 1): Index is very selective → good for equality predicates.
- **Low selectivity** (close to 0): Index is not selective → Oracle may prefer full table scan.

```sql
-- Get table row count and distinct values for a column
SELECT
    (SELECT COUNT(*) FROM employees) AS total_rows,
    (SELECT COUNT(DISTINCT dept_id) FROM employees) AS distinct_dept
FROM dual;
```

**Example output:**

| TOTAL_ROWS | DISTINCT_DEPT |
|-----------:|--------------:|
|     100000 |            10 |

Selectivity for `DEPT_ID` = 10 / 100000 = 0.0001 (very low).

---

## Indexing Strategy Guidelines

1. **Use B-tree indexes for high-cardinality columns**
   - Good for `=` and range conditions.
   - Example: `EMP_ID`, `ORDER_ID`, `EMAIL`.

   ```sql
   CREATE INDEX emp_empid_idx ON employees(emp_id);
   ```

2. **Be careful with low-cardinality columns**
   - Simple B-tree index often not useful.
   - Consider:
     - **Bitmap indexes** (mainly in data warehouses, not OLTP).
     - Or no index at all if predicates are not selective.

   ```sql
   -- Bitmap index (avoid in heavy OLTP due to locking)
   CREATE BITMAP INDEX emp_gender_bix ON employees(gender);
   ```

3. **Composite (multi-column) indexes**
   - Put the **most selective column first**.
   - Helps queries that filter on multiple columns.

   ```sql
   -- Suppose dept_id has low cardinality, job_id has higher cardinality
   CREATE INDEX emp_job_dept_idx ON employees(job_id, dept_id);
   ```

4. **Use statistics to help the optimizer**
   - Oracle needs up-to-date stats to estimate cardinality and selectivity correctly.

   ```sql
   BEGIN
     DBMS_STATS.GATHER_TABLE_STATS(
       ownname          => 'HR',
       tabname          => 'EMPLOYEES',
       cascade          => TRUE
     );
   END;
   /
   ```

## Example: When an Index Helps vs. Not

Assume:

- `EMPLOYEES` has 1,000,000 rows.
- `STATUS` column has only 2 values: 'A' (active), 'I' (inactive).
- Cardinality of `STATUS` = 2 → very low selectivity.

```sql
-- Query 1: Low selectivity predicate
SELECT *
FROM   employees
WHERE  status = 'A';
```

- If 90% of rows are 'A', Oracle will likely do a **full table scan**, even if there is an index on `STATUS`.

```sql
-- Query 2: High selectivity predicate
SELECT *
FROM   employees
WHERE  emp_id = 12345;
```

- `EMP_ID` is unique → very high selectivity.
- Oracle will almost certainly use the index on `EMP_ID`.

---

## Checking What Oracle Chooses

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   employees
WHERE  emp_id = 12345;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

**Example output (simplified):**

| Operation         | Name          |
|-------------------|---------------|
| INDEX UNIQUE SCAN | EMP_EMPID_IDX |
| TABLE ACCESS BY INDEX ROWID | EMPLOYEES |

This shows Oracle using the index because of high selectivity.

---

**In summary**:  
- **Cardinality** = how many distinct values a column has.  
- **Selectivity** = how well an index filters rows (distinct / total).  
- High-cardinality columns → good B-tree index candidates.  
- Low-cardinality columns → consider bitmap indexes (DW) or no index.  
- Order composite indexes by most selective column first.
