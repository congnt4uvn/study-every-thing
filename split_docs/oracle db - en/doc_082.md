
Here’s a concise summary and practical guide for handling:

- `ORA-00054: resource busy and acquire with NOWAIT specified or timeout expired`
- `ORA-00060: deadlock detected while waiting for resource`

---

## 1. ORA-00054: Resource Busy

**Meaning:**  
You tried to lock/modify an object or row that is already locked by another session, and you either:

- Used `NOWAIT`, or  
- Hit a timeout.

### Typical Causes

- Running `ALTER TABLE`, `DROP TABLE`, `CREATE INDEX`, etc., while another session is using the table.
- Updating rows that are locked by another transaction.

### Common Patterns & Fixes

#### A. Avoid `NOWAIT` or Add Retry Logic

**Example: DDL with NOWAIT (fails if locked)**

```sql
ALTER TABLE employees ADD (temp_col NUMBER) NOWAIT;
-- ORA-00054 if another session is using employees
```

**Better: retry with a loop (PL/SQL)**

```sql
DECLARE
  l_attempts PLS_INTEGER := 5;
  l_sleep    PLS_INTEGER := 2; -- seconds
BEGIN
  FOR i IN 1 .. l_attempts LOOP
    BEGIN
      EXECUTE IMMEDIATE 'ALTER TABLE employees ADD (temp_col NUMBER)';
      DBMS_OUTPUT.PUT_LINE('Success on attempt ' || i);
      RETURN;
    EXCEPTION
      WHEN OTHERS THEN
        IF SQLCODE = -54 THEN
          DBMS_OUTPUT.PUT_LINE('Resource busy, retry ' || i);
          DBMS_LOCK.SLEEP(l_sleep);
        ELSE
          RAISE;
        END IF;
    END;
  END LOOP;

  RAISE_APPLICATION_ERROR(-20001, 'Could not get lock after retries');
END;
/
```

**Input:** none (run as a script)  
**Output (example):**
```
Resource busy, retry 1
Resource busy, retry 2
Success on attempt 3
```

#### B. Schedule DDL During Low Activity

- Run schema changes during maintenance windows.
- Ensure no long-running transactions on the target objects.

#### C. Identify Blocking Sessions

```sql
-- Sessions waiting on locks
SELECT s.sid, s.serial#, s.username, s.status, l.type, l.lmode, l.request,
       o.object_name, o.object_type
FROM   v$lock l
JOIN   v$session s ON l.sid = s.sid
LEFT JOIN dba_objects o ON l.id1 = o.object_id
WHERE  l.block = 0 AND l.request > 0;
```

Use `ALTER SYSTEM KILL SESSION 'sid,serial#';` carefully if you must free the lock.

## 2. ORA-00060: Deadlock Detected

**Meaning:**  
Two or more sessions are waiting on each other’s locks in a cycle. Oracle detects this and aborts one statement with ORA-00060 to break the deadlock.

### Typical Causes

- Two sessions update the same tables but in different order.
- Complex transactions with multiple tables and foreign keys.
- Application code that locks rows inconsistently.

### Example Deadlock Scenario

**Session 1:**

```sql
-- Step 1
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- Step 2 (waits for Session 2)
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
```

**Session 2:**

```sql
-- Step 1
UPDATE accounts SET balance = balance - 50 WHERE id = 2;

-- Step 2 (waits for Session 1)
UPDATE accounts SET balance = balance + 50 WHERE id = 1;
```

Both sessions wait on each other → Oracle detects deadlock → one gets ORA-00060.

**Output (for one session):**
```text
ORA-00060: deadlock detected while waiting for resource
```

---

### How to Diagnose ORA-00060

1. **Check the trace file / alert log**  
   Oracle writes a deadlock graph to a trace file. It shows:
   - Sessions involved
   - SQL statements
   - Objects and locks

2. **Find the trace file (example on newer versions):**

```sql
SELECT value
FROM   v$diag_info
WHERE  name = 'Default Trace File';
```

Open that file on the server and look for `Deadlock graph:`.

---

### How to Fix / Prevent Deadlocks

#### A. Enforce a Consistent Locking Order

Always access tables/rows in the same order in all transactions.

**Bad (different order):**

- Tx A: `UPDATE table1 ...` then `UPDATE table2 ...`
- Tx B: `UPDATE table2 ...` then `UPDATE table1 ...`

**Good (same order):**

- All transactions: `UPDATE table1 ...` then `UPDATE table2 ...`

#### B. Keep Transactions Short

- Commit or rollback as soon as possible.
- Avoid user interaction (e.g., waiting for input) inside a transaction.

#### C. Use SELECT FOR UPDATE Carefully

```sql
-- Consistent order and key set
SELECT * FROM accounts
WHERE id IN (1, 2)
FOR UPDATE;
```

Then update in a deterministic order (e.g., ascending ID).

#### D. Add Application-Level Retry

If a statement fails with ORA-00060, retry the transaction.

**PL/SQL example:**

```sql
DECLARE
  l_attempts PLS_INTEGER := 3;
BEGIN
  FOR i IN 1 .. l_attempts LOOP
    BEGIN
      -- Your transactional work
      UPDATE accounts SET balance = balance - 100 WHERE id = 1;
      UPDATE accounts SET balance = balance + 100 WHERE id = 2;
      COMMIT;
      DBMS_OUTPUT.PUT_LINE('Transaction succeeded on attempt ' || i);
      RETURN;
    EXCEPTION
      WHEN OTHERS THEN
        IF SQLCODE = -60 THEN
          DBMS_OUTPUT.PUT_LINE('Deadlock, retry ' || i);
          ROLLBACK;
        ELSE
          RAISE;
        END IF;
    END;
  END LOOP;

  RAISE_APPLICATION_ERROR(-20002, 'Transaction failed after retries');
END;
/
```

**Output (example):**
```
Deadlock, retry 1
Transaction succeeded on attempt 2
```

---

## 3. Summary of Handling

- **ORA-00054 (resource busy)**  
  - Cause: object/row locked by another session.  
  - Handle by:  
    - Avoiding `NOWAIT` or adding retry logic.  
    - Running DDL during low activity.  
    - Identifying and resolving blocking sessions.

- **ORA-00060 (deadlock)**  
  - Cause: cyclic lock dependency between sessions.  
  - Handle by:  
    - Consistent locking order.  
    - Short transactions and careful `SELECT FOR UPDATE`.  
    - Application-level retry on ORA-00060.  
    - Using trace file deadlock graph to find and fix the root cause.
