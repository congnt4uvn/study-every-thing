
Here’s a concise summary of **Index-Organized Tables (IOTs)** in Oracle, with key points and code examples.

---

## 1. What is an Index-Organized Table?

- An **Index-Organized Table (IOT)** stores table data **in a B-tree index structure**, ordered by the **primary key**.
- Unlike a heap table (default), where rows are stored unordered and indexes are separate, in an IOT:
  - The **primary key index and the table data are the same structure**.
  - Access by primary key is very fast and space-efficient.

---

## 2. When to Use an IOT

Use an IOT when:

- Most queries access rows **by primary key** or by a **range of primary keys**.
- You want **faster primary key lookups** and **better clustering** of data.
- You want to **save space** by avoiding a separate primary key index.
- The table is **read-heavy** and relatively stable (not extremely high insert/update volume on the key).

Avoid or be careful when:

- You frequently update the **primary key** (this can be expensive).
- You often need **full table scans** or access by non-key columns only.

---

## 3. Basic IOT Creation Example

### Example: Create a simple IOT

```sql
CREATE TABLE customers_iot (
    customer_id   NUMBER,
    first_name    VARCHAR2(50),
    last_name     VARCHAR2(50),
    email         VARCHAR2(100),
    CONSTRAINT customers_iot_pk PRIMARY KEY (customer_id)
)
ORGANIZATION INDEX;
```

Key points:

- `ORGANIZATION INDEX` makes it an IOT.
- The table is physically ordered by `customer_id`.

---

## 4. Inserting and Selecting from an IOT

### Insert data

```sql
INSERT INTO customers_iot (customer_id, first_name, last_name, email)
VALUES (101, 'John', 'Smith', 'john.smith@example.com');

INSERT INTO customers_iot (customer_id, first_name, last_name, email)
VALUES (102, 'Mary', 'Jones', 'mary.jones@example.com');
```

### Query by primary key (very efficient)

```sql
SELECT *
FROM customers_iot
WHERE customer_id = 101;
```

**Sample output:**

| CUSTOMER_ID | FIRST_NAME | LAST_NAME | EMAIL                      |
|------------:|------------|-----------|----------------------------|
|         101 | John       | Smith     | john.smith@example.com     |

## 5. Including / Excluding Columns in the IOT Index

By default, all columns are stored in the IOT’s index structure.  
You can control this with **INCLUDING** and **OVERFLOW** to move some columns to an overflow segment.

### Example: IOT with overflow segment

```sql
CREATE TABLE orders_iot (
    order_id      NUMBER,
    customer_id   NUMBER,
    order_date    DATE,
    status        VARCHAR2(20),
    comments      VARCHAR2(4000),
    CONSTRAINT orders_iot_pk PRIMARY KEY (order_id)
)
ORGANIZATION INDEX
INCLUDING order_date
OVERFLOW;
```

Meaning:

- Columns up to and including `order_date` are stored in the main IOT index.
- Columns after `order_date` (`status`, `comments`) go to the **overflow segment**.
- This keeps the index entries smaller and faster for key lookups.

---

## 6. Secondary Indexes on IOTs

You can still create **secondary indexes** on non-key columns.

### Example: Secondary index on `customer_id`

```sql
CREATE INDEX orders_iot_cust_idx
ON orders_iot (customer_id);
```

- This index points to the IOT rows using the **primary key** as the logical rowid.

---

## 7. Comparing IOT vs Heap Table

### Heap table + primary key index

```sql
CREATE TABLE customers_heap (
    customer_id   NUMBER PRIMARY KEY,
    first_name    VARCHAR2(50),
    last_name     VARCHAR2(50),
    email         VARCHAR2(100)
);

-- Oracle automatically creates a separate index on customer_id
```

- Data is unordered; primary key index is separate.

### IOT

```sql
CREATE TABLE customers_iot (
    customer_id   NUMBER,
    first_name    VARCHAR2(50),
    last_name     VARCHAR2(50),
    email         VARCHAR2(100),
    CONSTRAINT customers_iot_pk PRIMARY KEY (customer_id)
)
ORGANIZATION INDEX;
```

- Data is stored in the primary key index structure itself.

---

## 8. Simple Performance Illustration (Conceptual)

Assume `customers_iot` has 1 million rows.

### Query 1: Primary key lookup

```sql
SELECT email
FROM customers_iot
WHERE customer_id = 500000;
```

- IOT: **Single index lookup** (data is in the index).
- Heap: Index lookup + table row fetch (two steps).

### Query 2: Range query on primary key

```sql
SELECT customer_id, email
FROM customers_iot
WHERE customer_id BETWEEN 1000 AND 2000
ORDER BY customer_id;
```

- IOT: Data already ordered by `customer_id`, efficient range scan.
- Heap: Index range scan + table access by rowid.
