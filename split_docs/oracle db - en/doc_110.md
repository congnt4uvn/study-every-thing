
Here are the main ideas and typical patterns for handling **multi‑tenant (many customers) in one Oracle database**:

---

## 1. Main Approaches to Multi‑Tenancy

### A. Separate Schema per Tenant
- **Each customer has its own schema** (set of tables, views, etc.).
- Same table structure, different owners.

**Pros**
- Good isolation between tenants.
- Easier per‑tenant backup/restore and management.

**Cons**
- Many schemas to manage (hundreds/thousands can be complex).
- Harder to change schema (must apply changes to all schemas).

**Example**

```sql
-- Tenant A schema
CREATE USER TENANT_A IDENTIFIED BY "passwordA";
GRANT CONNECT, RESOURCE TO TENANT_A;

CREATE TABLE TENANT_A.ORDERS (
    ORDER_ID     NUMBER PRIMARY KEY,
    ORDER_DATE   DATE,
    AMOUNT       NUMBER(10,2)
);

-- Tenant B schema
CREATE USER TENANT_B IDENTIFIED BY "passwordB";
GRANT CONNECT, RESOURCE TO TENANT_B;

CREATE TABLE TENANT_B.ORDERS (
    ORDER_ID     NUMBER PRIMARY KEY,
    ORDER_DATE   DATE,
    AMOUNT       NUMBER(10,2)
);
```

**Usage example**

```sql
-- Tenant A inserts an order
INSERT INTO TENANT_A.ORDERS (ORDER_ID, ORDER_DATE, AMOUNT)
VALUES (1, DATE '2024-05-20', 100.00);

-- Tenant B inserts an order
INSERT INTO TENANT_B.ORDERS (ORDER_ID, ORDER_DATE, AMOUNT)
VALUES (1, DATE '2024-05-20', 200.00);
```

---

### B. Shared Schema with Tenant Column (Row‑Level)
- **All tenants share the same tables**, distinguished by a `TENANT_ID` column.
- Application always filters by `TENANT_ID`.

**Pros**
- One schema to manage.
- Schema changes are simple (one place).
- Easy to add new tenants (just new `TENANT_ID` values).

**Cons**
- Must strictly enforce row‑level security.
- Large tables; need good indexing and partitioning.

**Example table**

```sql
CREATE TABLE ORDERS (
    TENANT_ID   NUMBER NOT NULL,
    ORDER_ID    NUMBER NOT NULL,
    ORDER_DATE  DATE,
    AMOUNT      NUMBER(10,2),
    CONSTRAINT PK_ORDERS PRIMARY KEY (TENANT_ID, ORDER_ID)
);
```

**Example inserts**

```sql
-- Tenant 101
INSERT INTO ORDERS (TENANT_ID, ORDER_ID, ORDER_DATE, AMOUNT)
VALUES (101, 1, DATE '2024-05-20', 150.00);

-- Tenant 202
INSERT INTO ORDERS (TENANT_ID, ORDER_ID, ORDER_DATE, AMOUNT)
VALUES (202, 1, DATE '2024-05-20', 300.00);
```

**Example query (application must filter by tenant)**

```sql
-- Get orders for tenant 101
SELECT ORDER_ID, ORDER_DATE, AMOUNT
FROM ORDERS
WHERE TENANT_ID = 101;
```

**Sample output**

| ORDER_ID | ORDER_DATE  | AMOUNT |
|---------:|-------------|--------|
| 1        | 2024‑05‑20  | 150.00 |

### C. Oracle VPD / RLS (Row‑Level Security)
- Still use a shared schema with `TENANT_ID`.
- Use **Virtual Private Database (VPD)** or **Row‑Level Security** to automatically add `TENANT_ID` filters to queries.

**Pros**
- Security enforced in the database, not only in application code.
- Transparent to application (no need to always add `WHERE TENANT_ID = ...`).

**Cons**
- More complex to configure and debug.
- Performance overhead if not designed carefully.

**Example: VPD policy**

```sql
-- Assume we store current tenant in a context
CREATE CONTEXT tenant_ctx USING set_tenant_ctx;

CREATE OR REPLACE PROCEDURE set_tenant_ctx (p_tenant_id NUMBER) AS
BEGIN
  DBMS_SESSION.SET_CONTEXT('TENANT_CTX', 'TENANT_ID', p_tenant_id);
END;
/

-- Policy function
CREATE OR REPLACE FUNCTION orders_rls_policy (
  schema_name IN VARCHAR2,
  object_name IN VARCHAR2
) RETURN VARCHAR2 AS
BEGIN
  RETURN 'TENANT_ID = SYS_CONTEXT(''TENANT_CTX'', ''TENANT_ID'')';
END;
/

-- Attach policy to ORDERS table
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'APP',
    object_name     => 'ORDERS',
    policy_name     => 'ORDERS_TENANT_POLICY',
    function_schema => 'APP',
    policy_function => 'ORDERS_RLS_POLICY'
  );
END;
/
```

**Usage**

```sql
-- Set tenant context for this session
BEGIN
  set_tenant_ctx(101);
END;
/

-- Now this query automatically becomes:
-- SELECT * FROM ORDERS WHERE TENANT_ID = 101
SELECT ORDER_ID, ORDER_DATE, AMOUNT
FROM ORDERS;
```

### D. Oracle Multitenant (CDB / PDB)
- At the **database level**, use **Container Database (CDB)** with multiple **Pluggable Databases (PDBs)**.
- Each tenant can have its own PDB (or group of tenants per PDB).

**Pros**
- Strong isolation (almost like separate databases).
- Can clone, backup, and move PDBs independently.

**Cons**
- More resource usage per tenant.
- License and operational complexity.

**Example (conceptual)**

```sql
-- Create a pluggable database for tenant A
CREATE PLUGGABLE DATABASE TENANT_A_PDB
  ADMIN USER pdb_admin IDENTIFIED BY "password"
  FILE_NAME_CONVERT = ('/oradata/CDB1/pdbseed/', '/oradata/CDB1/tenant_a_pdb/');

-- Open the PDB
ALTER PLUGGABLE DATABASE TENANT_A_PDB OPEN;
```

---

## 2. Choosing an Approach

- **Few large tenants, strong isolation needed**  
  → Separate schemas or separate PDBs.
- **Many small/medium tenants, need easy scaling and schema changes**  
  → Shared schema with `TENANT_ID` + VPD/RLS.
- **Regulatory or security isolation requirements**  
  → PDB per tenant or schema per tenant.
