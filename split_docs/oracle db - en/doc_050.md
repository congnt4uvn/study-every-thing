## Key Points: Common Indexing Pitfalls in Oracle

1. **Creating too many indexes**
   - Each `INSERT/UPDATE/DELETE` must maintain all related indexes.
   - Too many indexes slow down DML and increase storage and maintenance cost.

2. **Indexing low-selectivity columns**
   - Columns with few distinct values (e.g., `STATUS` = 'Y'/'N') often don’t benefit from normal B-tree indexes.
   - Oracle may ignore such indexes and do full table scans.

3. **Not matching index to query predicates**
   - Index columns and order should match how they’re used in `WHERE`, `JOIN`, and `ORDER BY`.
   - A composite index on `(col1, col2)` is not the same as `(col2, col1)`.

4. **Ignoring function usage in predicates**
   - `WHERE UPPER(name) = 'JOHN'` cannot use an index on `name` unless you create a function-based index.
   - Functions on indexed columns often make the index unusable.

5. **Using indexes on very small tables**
   - For small tables, a full table scan is often faster than using an index.
   - Unnecessary indexes add overhead without benefit.

6. **Not gathering statistics**
   - Outdated or missing statistics cause the optimizer to choose bad plans (e.g., ignoring good indexes).
   - Regular `DBMS_STATS` is essential.

7. **Wrong index type**
   - Using B-tree where bitmap or full-text (Oracle Text) would be better, and vice versa.
   - Bitmap indexes are good for low-cardinality columns in read-mostly data, but bad for heavy OLTP.

8. **Ignoring index selectivity and cardinality**
   - Index is useful when it significantly reduces the number of rows to read.
   - Poor selectivity → Oracle may prefer full table scan.

9. **Over-indexing for every query**
   - Designing an index for each individual query leads to index explosion.
   - Focus on the most critical and frequent queries.

10. **Not monitoring index usage**
    - Some indexes are never used.
    - Unused indexes can often be dropped after verification.

## Code Examples (Oracle)

### 1. Creating a Basic Index

```sql
-- Table
CREATE TABLE orders (
    order_id     NUMBER PRIMARY KEY,
    customer_id  NUMBER,
    order_date   DATE,
    status       VARCHAR2(10),
    amount       NUMBER
);

-- Index on customer_id (common filter/join column)
CREATE INDEX idx_orders_customer
    ON orders (customer_id);
```

**Usage:**

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   orders
WHERE  customer_id = 101;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

You should see `INDEX RANGE SCAN` on `IDX_ORDERS_CUSTOMER` if the index is used.

---

### 2. Pitfall: Indexing Low-Selectivity Column

```sql
-- status has very few distinct values: 'NEW', 'PAID', 'CANCELLED'
CREATE INDEX idx_orders_status
    ON orders (status);
```

Query:

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   orders
WHERE  status = 'NEW';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

If most rows are `status = 'NEW'`, Oracle may choose a `TABLE ACCESS FULL` instead of using `IDX_ORDERS_STATUS`, making the index mostly useless.

---

### 3. Composite Index and Column Order

```sql
-- Composite index
CREATE INDEX idx_orders_cust_date
    ON orders (customer_id, order_date);
```

**Good usage (matches leading column):**

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   orders
WHERE  customer_id = 101
AND    order_date  >= DATE '2024-01-01';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Likely uses `INDEX RANGE SCAN` on `IDX_ORDERS_CUST_DATE`.

**Less effective usage (no predicate on leading column):**

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   orders
WHERE  order_date >= DATE '2024-01-01';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

Oracle may not use `IDX_ORDERS_CUST_DATE` because `customer_id` (the leading column) is not in the predicate.

---

### 4. Function-Based Index to Avoid a Pitfall

Pitfall: Using a function on a column without a matching function-based index.

```sql
-- Query
SELECT *
FROM   customers
WHERE  UPPER(last_name) = 'SMITH';
```

Index on `last_name` alone:

```sql
CREATE INDEX idx_customers_lastname
    ON customers (last_name);
```

This index may not be used because of `UPPER(last_name)`.

**Fix: function-based index**

```sql
CREATE INDEX idx_customers_upper_lastname
    ON customers (UPPER(last_name));
```

Now:

```sql
EXPLAIN PLAN FOR
SELECT *
FROM   customers
WHERE  UPPER(last_name) = 'SMITH';

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```

You should see `INDEX RANGE SCAN` on `IDX_CUSTOMERS_UPPER_LASTNAME`.

### 5. Gathering Statistics

```sql
BEGIN
    DBMS_STATS.GATHER_TABLE_STATS(
        ownname          => 'HR',
        tabname          => 'ORDERS',
        cascade          => TRUE,   -- also gather index stats
        estimate_percent => DBMS_STATS.AUTO_SAMPLE_SIZE,
        method_opt       => 'FOR ALL COLUMNS SIZE AUTO'
    );
END;
/
```

Up-to-date stats help the optimizer decide whether to use indexes or full scans.

---

### 6. Monitoring Index Usage

Enable monitoring:

```sql
ALTER INDEX idx_orders_customer MONITORING USAGE;
```

Run your workload, then check:

```sql
SELECT index_name, used
FROM   v$object_usage
WHERE  index_name = 'IDX_ORDERS_CUSTOMER';
```

If `USED = 'NO'` over a long period, the index might be a candidate for removal (after careful analysis).

Stop monitoring:

```sql
ALTER INDEX idx_orders_customer NOMONITORING USAGE;
```
