
Here’s a concise summary of common Oracle SQL tuning hints and how they’re used, with simple examples.

---

## 1. `/*+ INDEX */` Hint

**Purpose:**  
Force the optimizer to use a specific index (or any index on a table) instead of a full table scan.

**Syntax:**
```sql
SELECT /*+ INDEX(table_alias index_name) */
       columns
FROM   table_name table_alias
WHERE  conditions;
```

**Example:**
```sql
-- Table: EMP (EMP_ID PK, DEPT_ID, SALARY, etc.)
-- Index: EMP_DEPT_IDX on EMP(DEPT_ID)

SELECT /*+ INDEX(e EMP_DEPT_IDX) */
       e.emp_id, e.dept_id, e.salary
FROM   emp e
WHERE  e.dept_id = 10;
```

**Effect (conceptual):**
- Input: Query with `WHERE dept_id = 10`
- Without hint: Optimizer might choose a full table scan.
- With `INDEX` hint: Optimizer is pushed to use `EMP_DEPT_IDX` to find rows with `dept_id = 10`.

---

## 2. `/*+ FULL */` Hint

**Purpose:**  
Force a full table scan, even if an index exists.

**Syntax:**
```sql
SELECT /*+ FULL(table_alias) */
       columns
FROM   table_name table_alias
WHERE  conditions;
```

**Example:**
```sql
SELECT /*+ FULL(e) */
       e.emp_id, e.dept_id, e.salary
FROM   emp e
WHERE  e.salary > 1000;
```

**When useful:**
- Large portion of the table is being read.
- Index access would cause many random I/Os, making a full scan cheaper.

**Effect (conceptual):**
- Input: Query with `WHERE salary > 1000` (many rows match).
- With `FULL(e)`: Oracle reads the whole `EMP` table sequentially.

---

## 3. `/*+ PARALLEL */` Hint

**Purpose:**  
Enable parallel execution for a table or query, using multiple processes to speed up large operations.

**Syntax:**
```sql
SELECT /*+ PARALLEL(table_alias degree) */
       columns
FROM   table_name table_alias
WHERE  conditions;
```

- `degree` = number of parallel execution servers (e.g., 4, 8).

**Example:**
```sql
SELECT /*+ PARALLEL(e 4) */
       e.dept_id, COUNT(*) AS emp_count
FROM   emp e
GROUP  BY e.dept_id;
```

**Effect (conceptual):**
- Input: Large `EMP` table.
- With `PARALLEL(e 4)`: Oracle splits the work across 4 parallel processes, then merges results.

**Parallel DML example:**
```sql
INSERT /*+ PARALLEL(e 8) */
INTO   emp_archive e
SELECT /*+ PARALLEL(s 8) */
       s.*
FROM   emp s;
```

---

## 4. Combining Hints

You can combine hints in one comment:

```sql
SELECT /*+ FULL(e) PARALLEL(e 4) */
       e.dept_id, AVG(e.salary) avg_sal
FROM   emp e
GROUP  BY e.dept_id;
```

- `FULL(e)`: Use full table scan on `EMP`.
- `PARALLEL(e 4)`: Do that full scan in parallel with degree 4.

---

## 5. General Notes

- Hints are written in a special comment right after `SELECT`, `INSERT`, `UPDATE`, or `DELETE`:
  ```sql
  SELECT /*+ HINT_NAME(...) */ ...
  ```
- They **influence** the optimizer but do not guarantee performance improvement; always test with `EXPLAIN PLAN` or `AUTOTRACE`.
- Use hints only when the optimizer’s default plan is consistently suboptimal and you understand the data and workload.
