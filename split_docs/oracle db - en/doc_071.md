
Here’s a concise summary of Oracle materialized view refresh types, with simple examples.

---

## 1. Complete Refresh

**Idea:**  
Rebuilds the materialized view (MV) from scratch by re-running the defining query.

**When to use:**
- Source tables change a lot.
- No materialized view logs or fast refresh is not possible.
- Data volume is manageable for full rebuild.

**Create example:**
```sql
CREATE MATERIALIZED VIEW sales_mv
BUILD IMMEDIATE
REFRESH COMPLETE
START WITH SYSDATE
NEXT SYSDATE + 1/24  -- every hour
AS
SELECT product_id, SUM(amount) total_amount
FROM sales
GROUP BY product_id;
```

**Manual refresh (on demand):**
```sql
EXEC DBMS_MVIEW.REFRESH('SALES_MV', 'C');  -- 'C' = COMPLETE
```

**Effect (conceptual):**
- Input: `sales` table with updated rows.
- Output: `sales_mv` is fully recalculated from `sales`.

---

## 2. Fast Refresh

**Idea:**  
Applies only the changes (inserts/updates/deletes) since the last refresh using **materialized view logs**.

**Requirements (simplified):**
- Materialized view logs on base tables.
- Query must be “fast refreshable” (no unsupported constructs).

**Create materialized view log:**
```sql
CREATE MATERIALIZED VIEW LOG ON sales
WITH ROWID, SEQUENCE (product_id, amount)
INCLUDING NEW VALUES;
```

**Create MV with fast refresh:**
```sql
CREATE MATERIALIZED VIEW sales_mv_fast
BUILD IMMEDIATE
REFRESH FAST
START WITH SYSDATE
NEXT SYSDATE + 1/24
AS
SELECT product_id, SUM(amount) total_amount
FROM sales
GROUP BY product_id;
```

**Manual fast refresh:**
```sql
EXEC DBMS_MVIEW.REFRESH('SALES_MV_FAST', 'F');  -- 'F' = FAST
```

**Effect (conceptual):**
- Input: Only changed rows in `sales` since last refresh (tracked in logs).
- Output: `sales_mv_fast` is updated incrementally, usually much faster than complete.

---

## 3. ON COMMIT Refresh

**Idea:**  
Refreshes the MV automatically at the end of a transaction that changes the base tables.

**When to use:**
- Need near real-time consistency.
- Data volume per transaction is small enough that refresh cost is acceptable.

**Example:**
```sql
CREATE MATERIALIZED VIEW sales_mv_commit
BUILD IMMEDIATE
REFRESH FAST ON COMMIT
AS
SELECT product_id, SUM(amount) total_amount
FROM sales
GROUP BY product_id;
```

**Behavior:**
- Input: A transaction that inserts/updates/deletes rows in `sales`.
- Output: At `COMMIT`, `sales_mv_commit` is refreshed (fast, using logs).

**Example transaction:**
```sql
INSERT INTO sales (sale_id, product_id, amount)
VALUES (1001, 10, 500);

COMMIT;  -- triggers refresh of sales_mv_commit
```

---

## 4. ON DEMAND Refresh

**Idea:**  
Refresh happens only when explicitly requested (via `DBMS_MVIEW.REFRESH` or a scheduled job).

**When to use:**
- You control when refresh happens (e.g., nightly batch).
- You want to avoid overhead during normal transactions.

**Example:**
```sql
CREATE MATERIALIZED VIEW sales_mv_demand
BUILD IMMEDIATE
REFRESH FAST ON DEMAND
AS
SELECT product_id, SUM(amount) total_amount
FROM sales
GROUP BY product_id;
```

**Manual refresh (fast or complete):**
```sql
-- Fast refresh
EXEC DBMS_MVIEW.REFRESH('SALES_MV_DEMAND', 'F');

-- Complete refresh
EXEC DBMS_MVIEW.REFRESH('SALES_MV_DEMAND', 'C');
```

**Effect (conceptual):**
- Input: All changes since last refresh.
- Output: MV is updated only when you call refresh (or via a scheduled job).

---

## Summary Table

| Aspect          | FAST                     | COMPLETE                     | ON COMMIT                     | ON DEMAND                         |
|----------------|--------------------------|------------------------------|--------------------------------|-----------------------------------|
| What it does   | Incremental changes only | Full rebuild of MV           | Refresh at each COMMIT        | Refresh only when requested       |
| Needs logs     | Yes                      | No                           | Usually yes (for FAST)        | Depends (FAST vs COMPLETE)        |
| Performance    | Usually faster           | Slower for large data        | Overhead on each commit       | You control when cost is paid     |
| Use case       | Frequent small changes   | Periodic full recalculation  | Near real-time consistency    | Batch/controlled refresh timing   |
