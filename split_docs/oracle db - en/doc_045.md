## Key Ideas: Indexing Strategy in Oracle

1. **Purpose of Indexes**
   - Speed up data retrieval (SELECT).
   - Can slow down DML (INSERT/UPDATE/DELETE) because indexes must be maintained.
   - Good strategy: index columns used in `WHERE`, `JOIN`, and `ORDER BY` clauses, but avoid over-indexing.

2. **Main Types of Indexes in Oracle**

### 1. B-tree Index (default)
- **Use when**: Column has many distinct values (high cardinality).
- **Good for**: Equality and range searches.

**Create:**
```sql
CREATE INDEX idx_emp_lastname
ON employees (last_name);
```

**Example query using it:**
```sql
SELECT employee_id, last_name
FROM employees
WHERE last_name = 'KING';
```

**Conceptual input/output:**
- Input: `last_name = 'KING'`
- Output: rows where `last_name` is KING, found quickly via index instead of full table scan.

---

### 2. Bitmap Index
- **Use when**: Column has few distinct values (low cardinality), e.g., gender, status flags.
- **Good for**: Data warehouses, complex ad-hoc queries with many AND/OR conditions.
- **Not ideal for**: Heavy OLTP with many updates.

**Create:**
```sql
CREATE BITMAP INDEX idx_emp_gender
ON employees (gender);
```

**Example query:**
```sql
SELECT COUNT(*)
FROM employees
WHERE gender = 'F'
  AND status = 'ACTIVE';
```

Oracle can combine multiple bitmap indexes efficiently for such conditions.

---

### 3. Unique Index
- **Ensures**: No duplicate values in the indexed column(s).
- Usually created automatically by `PRIMARY KEY` or `UNIQUE` constraints.

**Create via constraint (recommended):**
```sql
ALTER TABLE employees
ADD CONSTRAINT emp_email_uk UNIQUE (email);
```

This creates a unique index under the hood.

**Behavior (input/output):**
```sql
INSERT INTO employees (employee_id, email)
VALUES (1001, 'john.doe@example.com');  -- succeeds

INSERT INTO employees (employee_id, email)
VALUES (1002, 'john.doe@example.com');  -- fails: ORA-00001 unique constraint violated
```

---

### 4. Composite (Multi-column) Index
- **Use when**: Queries filter on multiple columns together.
- Column order matters.

**Create:**
```sql
CREATE INDEX idx_emp_dept_job
ON employees (department_id, job_id);
```

**Best used when queries look like:**
```sql
SELECT *
FROM employees
WHERE department_id = 10
  AND job_id = 'SA_REP';
```

Oracle can use the index efficiently when filtering by `department_id` alone or by both columns, but not by `job_id` alone (because of the leading column rule).

---

### 5. Function-based Index
- **Use when**: Queries use expressions or functions on columns in `WHERE` clause.

**Create:**
```sql
CREATE INDEX idx_emp_upper_lastname
ON employees (UPPER(last_name));
```

**Query that can use it:**
```sql
SELECT *
FROM employees
WHERE UPPER(last_name) = 'KING';
```

Without a function-based index, Oracle might not use a normal index on `last_name` because of the function.

---

### 6. Reverse Key Index
- **Use when**: Avoid index hot spots on monotonically increasing keys (e.g., sequence-based IDs).
- Reverses the bytes of the index key.

**Create:**
```sql
CREATE INDEX idx_emp_id_rev
ON employees (employee_id) REVERSE;
```

Helps distribute inserts across index leaf blocks in high-concurrency environments.

---

### 7. Partitioned Index
- **Use when**: Table is partitioned (large tables).
- Types: local (aligned with table partitions) or global.

**Local partitioned index:**
```sql
CREATE INDEX idx_sales_date_local
ON sales (sale_date)
LOCAL;
```

Improves manageability and performance for large partitioned tables.

## Simple Example: Before/After Index

**Table:**
```sql
CREATE TABLE employees (
  employee_id   NUMBER PRIMARY KEY,
  last_name     VARCHAR2(50),
  department_id NUMBER
);
```

**Query without index on `department_id`:**
```sql
EXPLAIN PLAN FOR
SELECT *
FROM employees
WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```
- Likely shows `TABLE ACCESS FULL`.

**Create index:**
```sql
CREATE INDEX idx_emp_dept
ON employees (department_id);
```

**Run again:**
```sql
EXPLAIN PLAN FOR
SELECT *
FROM employees
WHERE department_id = 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);
```
- Now likely shows `INDEX RANGE SCAN` on `IDX_EMP_DEPT` + `TABLE ACCESS BY INDEX ROWID`.
