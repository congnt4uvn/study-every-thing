
Here’s a concise summary and examples for:

**Advanced SQL: `SUM/AVG OVER (PARTITION BY … ORDER BY …)`**

---

## 1. What this does

`SUM(...) OVER (PARTITION BY ... ORDER BY ...)` and  
`AVG(...) OVER (PARTITION BY ... ORDER BY ...)` are **analytic (window) functions**.

They let you:

1. **Partition** rows into groups (like `GROUP BY`, but without collapsing rows).
2. **Order** rows within each partition.
3. Compute **running / cumulative** or **moving** totals/averages across those ordered rows.

Unlike `GROUP BY`, analytic functions:
- Do **not** reduce the number of rows.
- Add a calculated value **per row**.

---

## 2. Basic syntax

```sql
SUM(expr) OVER (
  PARTITION BY partition_col1, partition_col2, ...
  ORDER BY order_col1, order_col2, ...
  [ROWS/RANGE frame_clause]
)

AVG(expr) OVER (
  PARTITION BY partition_col1, partition_col2, ...
  ORDER BY order_col1, order_col2, ...
  [ROWS/RANGE frame_clause]
)
```

- `PARTITION BY` = how to split data into groups (e.g., by customer, department).
- `ORDER BY` = order within each group (e.g., by date).
- Frame clause (optional) controls the window (e.g., cumulative vs last N rows).

---

## 3. Example: Running (cumulative) sum per customer

**Table**: `sales`

| sale_id | customer_id | sale_date  | amount |
|--------:|------------:|------------|-------:|
| 1       | 101         | 2024-01-01 | 100    |
| 2       | 101         | 2024-01-05 | 200    |
| 3       | 101         | 2024-01-10 | 50     |
| 4       | 102         | 2024-01-02 | 300    |
| 5       | 102         | 2024-01-06 | 150    |

**Query:**

```sql
SELECT
    customer_id,
    sale_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY sale_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
FROM sales
ORDER BY customer_id, sale_date;
```

**Output:**

| customer_id | sale_date  | amount | running_total |
|------------:|------------|-------:|--------------:|
| 101         | 2024-01-01 | 100    | 100           |
| 101         | 2024-01-05 | 200    | 300           |
| 101         | 2024-01-10 | 50     | 350           |
| 102         | 2024-01-02 | 300    | 300           |
| 102         | 2024-01-06 | 150    | 450           |

Key idea: `SUM OVER` gives a **cumulative sum** per customer, ordered by date.

## 4. Example: Running average per department

**Table**: `employee_salary`

| emp_id | dept | month      | salary |
|-------:|-----:|------------|-------:|
| 1      | A    | 2024-01-01 | 3000   |
| 2      | A    | 2024-02-01 | 3200   |
| 3      | A    | 2024-03-01 | 3100   |
| 4      | B    | 2024-01-01 | 4000   |
| 5      | B    | 2024-02-01 | 4200   |

**Query:**

```sql
SELECT
    dept,
    month,
    salary,
    AVG(salary) OVER (
        PARTITION BY dept
        ORDER BY month
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_avg_salary
FROM employee_salary
ORDER BY dept, month;
```

**Output:**

| dept | month      | salary | running_avg_salary |
|-----:|------------|-------:|-------------------:|
| A    | 2024-01-01 | 3000   | 3000.00            |
| A    | 2024-02-01 | 3200   | 3100.00            |
| A    | 2024-03-01 | 3100   | 3100.00            |
| B    | 2024-01-01 | 4000   | 4000.00            |
| B    | 2024-02-01 | 4200   | 4100.00            |

## 5. Example: Moving 3-row average (sliding window)

**Same `employee_salary` table**, but we want the average of the **current and previous 2 rows** per department.

**Query:**

```sql
SELECT
    dept,
    month,
    salary,
    AVG(salary) OVER (
        PARTITION BY dept
        ORDER BY month
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS moving_3row_avg
FROM employee_salary
ORDER BY dept, month;
```

**Output (conceptual):**

| dept | month      | salary | moving_3row_avg |
|-----:|------------|-------:|----------------:|
| A    | 2024-01-01 | 3000   | 3000.00         |  -- only 1 row in window
| A    | 2024-02-01 | 3200   | 3100.00         |  -- (3000+3200)/2
| A    | 2024-03-01 | 3100   | 3100.00         |  -- (3000+3200+3100)/3
| B    | 2024-01-01 | 4000   | 4000.00         |
| B    | 2024-02-01 | 4200   | 4100.00         |

---

## 6. Key points to remember

- `PARTITION BY` = group scope (like a per-customer or per-department window).
- `ORDER BY` = defines sequence for running/moving calculations.
- Frame clause:
  - `UNBOUNDED PRECEDING` to `CURRENT ROW` → cumulative (running) total/average.
  - `N PRECEDING` to `CURRENT ROW` → moving window of last N rows.
- Analytic functions **do not** collapse rows; they add values per row.

These patterns are common for:
- Running totals (sales, balances).
- Running averages (performance, salaries).
- Moving averages (trend smoothing).
