## Key Points & Main Ideas

### 1. Version Control for Oracle Database

- Store **all database-related artifacts** in Git (or similar):
  - DDL: `CREATE TABLE`, `ALTER TABLE`, `CREATE INDEX`, etc.
  - PL/SQL: packages, procedures, functions, triggers.
  - Seed data / reference data.
  - Migration scripts (incremental changes).
- Use a **clear structure**, for example:
  ```text
  db/
    schema/
      tables/
      views/
      indexes/
    plsql/
      packages/
      procedures/
    migrations/
      V001__init_schema.sql
      V002__add_customer_table.sql
  ```

### 2. CI/CD for Oracle Database

- CI (Continuous Integration):
  - On each commit/merge:
    - Run static checks (lint PL/SQL).
    - Run unit tests (e.g., utPLSQL).
    - Optionally spin up a test Oracle DB (container or shared instance) and apply migrations.
- CD (Continuous Delivery/Deployment):
  - Automatically or semi‑automatically deploy to:
    - Dev → Test → Staging → Production
  - Use **migration tools** (Liquibase, Flyway, custom scripts) to apply changes in order.

### 3. Deployment Strategies (Database)

Common strategies adapted to Oracle:

1. **Migration-based (Incremental)**
   - Each change is a script with a version.
   - Applied in order; schema evolves over time.
   - Works well with CI/CD and roll-forward deployments.

2. **Blue-Green / Shadow Schema**
   - Maintain two schemas or databases:
     - Blue (current production)
     - Green (new version)
   - Sync data, switch application connection from Blue to Green.
   - More complex for large Oracle DBs but useful for high availability.

3. **Rolling / Zero-Downtime Changes**
   - Backward-compatible changes:
     - Add new columns (nullable), don’t drop old ones immediately.
     - Deploy app that uses both old and new structures.
     - Migrate data in background.
     - Remove old columns in a later release.

4. **Feature Toggles**
   - Deploy DB changes ahead of time.
   - Application uses feature flags to enable new behavior gradually.

-- Input: existing schema without CUSTOMER table

    customer_id   NUMBER GENERATED BY DEFAULT AS IDENTITY,

## Example: Simple CI Step (Shell + SQL*Plus)

**Input:** Run all migration scripts against a test DB in CI.

```bash
#!/usr/bin/env bash
set -e

ORACLE_CONN="user/password@//host:1521/ORCLPDB1"

for script in db/migrations/*.sql; do
  echo "Applying $script"
  sqlplus -s "$ORACLE_CONN" @"$script"
done
```

**Output (CI log excerpt):**

```text
Applying db/migrations/V001__init_schema.sql
Applying db/migrations/V002__add_customer_table.sql
Applying db/migrations/V003__add_order_table.sql
...
```

If a script fails, CI job fails, preventing a broken schema from being deployed.
