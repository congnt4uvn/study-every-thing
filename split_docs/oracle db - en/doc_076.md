
Here’s a concise summary of the topic “Locking & Isolation: Row-level locking” in Oracle, with code examples.

---

## Key Ideas

1. **Row-level locking**  
   - Oracle locks only the specific rows being modified, not the whole table.  
   - This allows many users to work on the same table concurrently, as long as they don’t modify the same rows.

2. **Automatic locking**  
   - Locks are acquired automatically when you run `INSERT`, `UPDATE`, `DELETE`, `SELECT ... FOR UPDATE`.  
   - You don’t manually lock rows in normal DML; Oracle handles it.

3. **Isolation & consistency**  
   - Default isolation level is **READ COMMITTED**.  
   - Readers don’t block writers and writers don’t block readers:  
     - Readers see a **consistent snapshot** of data (via undo), not blocked by row locks.  
     - Writers lock only the rows they change, blocking other writers on those same rows.

4. **Blocking & waiting**  
   - If two sessions try to update the same row, the second session waits until the first commits or rolls back.  
   - You can use `FOR UPDATE NOWAIT` or `FOR UPDATE WAIT n` to control waiting behavior.

5. **Deadlocks**  
   - Can occur if sessions lock rows in different orders.  
   - Oracle detects deadlocks and raises an error; one transaction is rolled back.

## Basic Row-level Lock Example

Assume a table:

```sql
CREATE TABLE accounts (
    account_id   NUMBER PRIMARY KEY,
    balance      NUMBER
);

INSERT INTO accounts VALUES (1, 1000);
INSERT INTO accounts VALUES (2, 2000);
COMMIT;
```

### Session 1

```sql
-- Session 1: start transaction (implicit)
UPDATE accounts
   SET balance = balance - 100
 WHERE account_id = 1;
```

- This locks **only** the row where `account_id = 1`.
- Other sessions can still read that row (from undo) but cannot update it.

### Session 2

```sql
-- Session 2: try to update the same row
UPDATE accounts
   SET balance = balance + 50
 WHERE account_id = 1;
```

- Session 2 will **wait** until Session 1 commits or rolls back.

#### If Session 1 does:

```sql
COMMIT;
```

Then Session 2’s update proceeds and completes.

**Resulting balances (after both commits):**

- Initial: `account_id=1, balance=1000`
- After Session 1: `1000 - 100 = 900`
- After Session 2: `900 + 50 = 950`

Final row:

```sql
SELECT * FROM accounts WHERE account_id = 1;

-- Output:
-- ACCOUNT_ID | BALANCE
--     1      |   950
```

## SELECT FOR UPDATE (Explicit Row Lock)

Use `SELECT ... FOR UPDATE` to lock rows you plan to modify:

### Session 1

```sql
SELECT *
  FROM accounts
 WHERE account_id = 2
 FOR UPDATE;
```

- Locks the row with `account_id = 2` for update by this session.

### Session 2

```sql
SELECT *
  FROM accounts
 WHERE account_id = 2
 FOR UPDATE;
```

- Session 2 will **wait** until Session 1 commits/rolls back.

### Avoid waiting: NOWAIT / WAIT

```sql
-- Session 2: fail immediately if row is locked
SELECT *
  FROM accounts
 WHERE account_id = 2
 FOR UPDATE NOWAIT;
```

If locked, Session 2 gets:

```text
ORA-00054: resource busy and acquire with NOWAIT specified or timeout expired
```

Or wait up to 5 seconds:

```sql
SELECT *
  FROM accounts
 WHERE account_id = 2
 FOR UPDATE WAIT 5;
```

## Isolation Level Example (READ COMMITTED)

### Session 1

```sql
UPDATE accounts
   SET balance = 3000
 WHERE account_id = 2;
-- no commit yet
```

### Session 2

```sql
-- Session 2 reads the row
SELECT balance
  FROM accounts
 WHERE account_id = 2;
```

**Output in Session 2 (READ COMMITTED):**

- Session 2 sees the **old committed value**, not 3000, because Session 1 hasn’t committed yet.
- This is provided by Oracle’s **multi-version read consistency** using undo data.

After Session 1 does:

```sql
COMMIT;
```

A new `SELECT` in Session 2 will see the updated value (3000).

---

## Summary

- Oracle uses **row-level locks** to maximize concurrency.
- DML automatically locks only the rows being changed.
- Readers are not blocked by writers; they see a consistent snapshot.
- Writers block other writers on the same rows.
- `SELECT ... FOR UPDATE` explicitly locks rows for later update, with `NOWAIT`/`WAIT` options to control blocking.
