
Here’s a concise summary of the key ideas around **transaction concepts, read consistency, and undo segments** in Oracle Database, with simple examples.

---

## 1. Transactions in Oracle

- A **transaction** is a logical unit of work, typically from one `BEGIN` (implicit) to `COMMIT` or `ROLLBACK`.
- Changes made in a transaction are:
  - **Not visible** to other sessions until `COMMIT`.
  - **Fully undone** if `ROLLBACK` is issued.

**Example:**

Session A:
```sql
-- Start transaction (implicit)
UPDATE employees
   SET salary = salary * 1.1
 WHERE department_id = 10;

-- Changes visible only to Session A
SELECT employee_id, salary
  FROM employees
 WHERE department_id = 10;
```

Session B (at the same time):
```sql
SELECT employee_id, salary
  FROM employees
 WHERE department_id = 10;
```

**Output:**
- Session A sees the **updated** salaries.
- Session B still sees the **old** salaries until Session A commits.

Session A:
```sql
COMMIT;  -- Now changes become visible to others
```

---

## 2. Read Consistency

- Oracle guarantees that a query sees a **consistent snapshot** of data as of the time the query started.
- Even if other sessions are updating rows, your query:
  - Does **not** see partial changes.
  - Sees data as it existed at the query’s start time.

**Example:**

Session A:
```sql
UPDATE accounts
   SET balance = balance - 100
 WHERE account_id = 1;

-- Not committed yet
```

Session B:
```sql
-- Long-running query
SELECT account_id, balance
  FROM accounts
 WHERE account_type = 'SAVINGS';
```

While Session B’s query is running, Session A commits:
```sql
COMMIT;
```

**Behavior:**
- Session B’s query continues to see data as it was when the query started (before Session A’s commit).
- This is achieved using **undo segments**.

---

## 3. Undo Segments

- **Undo segments** store the **before-image** of data that is being modified.
- They are used for:
  1. **Transaction rollback** – to undo changes if `ROLLBACK` or error.
  2. **Read consistency** – to reconstruct old versions of rows for queries.
  3. **Recovery** – after instance failure.

**How it works conceptually:**

1. When you run:
   ```sql
   UPDATE employees
      SET salary = salary * 1.1
    WHERE employee_id = 100;
   ```
2. Oracle:
   - Writes the **old salary** to an undo segment.
   - Writes the **new salary** to the data block (in buffer cache).
3. If another session needs the old version (for read consistency), Oracle reads it from the undo segment.

---

## 4. Simple Demonstration of Undo Use

### Step 1: Create a test table

```sql
CREATE TABLE t_test (
    id   NUMBER PRIMARY KEY,
    val  NUMBER
);

INSERT INTO t_test VALUES (1, 100);
COMMIT;
```

### Step 2: Session A updates but does not commit

Session A:
```sql
UPDATE t_test
   SET val = 200
 WHERE id = 1;

SELECT id, val FROM t_test WHERE id = 1;
```

**Output (Session A):**
```text
ID  VAL
--  ---
1   200
```

### Step 3: Session B reads the same row

Session B:
```sql
SELECT id, val FROM t_test WHERE id = 1;
```

**Output (Session B):**
```text
ID  VAL
--  ---
1   100
```

- Session B sees `100` because Oracle uses the **undo** to reconstruct the old version for read consistency.

### Step 4: Session A rolls back

Session A:
```sql
ROLLBACK;
```

Session B (again):
```sql
SELECT id, val FROM t_test WHERE id = 1;
```

**Output (Session B):**
```text
ID  VAL
--  ---
1   100
```

- After rollback, the table data matches the undo (old value).

---

## 5. Key Points Summary

- **Transaction**: group of operations that are committed or rolled back as a unit.
- **Read consistency**: every query sees a stable snapshot of data as of its start time, regardless of concurrent updates.
- **Undo segments**:
  - Store old versions of data.
  - Enable rollback, read consistency, and recovery.
- Other sessions do not see your uncommitted changes; they see data reconstructed from undo if needed.
