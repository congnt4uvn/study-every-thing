
Here’s a concise summary of the main join methods in Oracle execution plans, with simple examples.

---

## 1. NESTED LOOPS JOIN

**Idea:**  
For each row from the outer (driving) table, Oracle probes the inner table (usually via an index).  
Best when:
- Outer table is small (or well-filtered)
- Inner table has a good index on the join column
- You expect few matching rows per outer row

**Typical plan snippet:**
```sql
SELECT /*+ USE_NL(e d) */ e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id
WHERE  e.emp_id < 10;
```

Possible `EXPLAIN PLAN` (simplified):
```sql
EXPLAIN PLAN FOR
SELECT /*+ USE_NL(e d) */ e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id
WHERE  e.emp_id < 10;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- Output (simplified)
--------------------------------------------------------------------------------
| Id | Operation                    | Name        |
--------------------------------------------------------------------------------
|  0 | SELECT STATEMENT             |             |
|  1 |  NESTED LOOPS                |             |
|  2 |   TABLE ACCESS BY INDEX ROWID| EMPLOYEES   |
|  3 |    INDEX RANGE SCAN          | EMP_EMP_ID_I|
|  4 |   TABLE ACCESS BY INDEX ROWID| DEPARTMENTS |
|  5 |    INDEX UNIQUE SCAN         | DEPT_PK     |
--------------------------------------------------------------------------------
```

**Execution conceptually:**
- Get a row from `EMPLOYEES` (outer)
- Use `DEPT_PK` index to find matching row in `DEPARTMENTS` (inner)
- Repeat for each outer row

---

## 2. HASH JOIN

**Idea:**  
Build a hash table in memory from the smaller (build) input, then probe it with the larger (probe) input.  
Best when:
- Joining large sets
- No suitable indexes, or full scans are cheaper
- Equality joins (`=`) on large volumes

**Typical plan snippet:**
```sql
SELECT /*+ USE_HASH(e d) */ e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id;
```

Possible `EXPLAIN PLAN`:
```sql
EXPLAIN PLAN FOR
SELECT /*+ USE_HASH(e d) */ e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- Output (simplified)
--------------------------------------------------------------------------------
| Id | Operation           | Name        |
--------------------------------------------------------------------------------
|  0 | SELECT STATEMENT    |             |
|  1 |  HASH JOIN          |             |
|  2 |   TABLE ACCESS FULL | DEPARTMENTS |
|  3 |   TABLE ACCESS FULL | EMPLOYEES   |
--------------------------------------------------------------------------------
```

**Execution conceptually:**
- Read all rows from the smaller table (e.g. `DEPARTMENTS`), build an in-memory hash table on `DEPT_ID`
- Scan `EMPLOYEES`, for each row hash `DEPT_ID` and probe the hash table
- Return matches

---

## 3. MERGE JOIN (SORT MERGE JOIN)

**Idea:**  
Both inputs are sorted on the join key, then merged like a “merge” step in merge sort.  
Best when:
- Both sides are already sorted on join columns (e.g. due to indexes or ORDER BY)
- Large data sets
- Equality or range joins

**Typical plan snippet:**
```sql
SELECT /*+ USE_MERGE(e d) */ e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id;
```

Possible `EXPLAIN PLAN`:
```sql
EXPLAIN PLAN FOR
SELECT /*+ USE_MERGE(e d) */ e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id;

SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- Output (simplified)
--------------------------------------------------------------------------------
| Id | Operation           | Name        |
--------------------------------------------------------------------------------
|  0 | SELECT STATEMENT    |             |
|  1 |  MERGE JOIN         |             |
|  2 |   SORT JOIN         | DEPARTMENTS |
|  3 |    TABLE ACCESS FULL| DEPARTMENTS |
|  4 |   SORT JOIN         | EMPLOYEES   |
|  5 |    TABLE ACCESS FULL| EMPLOYEES   |
--------------------------------------------------------------------------------
```

**Execution conceptually:**
- Sort `DEPARTMENTS` by `DEPT_ID`
- Sort `EMPLOYEES` by `DEPT_ID`
- Walk through both sorted streams, matching rows with the same `DEPT_ID`

---

## 4. Comparing the Join Methods

- **Nested Loops**
  - Good for: small outer set, indexed inner table, OLTP-style queries
  - Bad for: large unindexed joins

- **Hash Join**
  - Good for: large tables, data warehouse queries, no good indexes
  - Needs memory (PGA) for hash table; may spill to disk if too big

- **Merge Join**
  - Good for: large sorted inputs, or when sort cost is acceptable
  - Can handle range joins better than hash join

## Simple Input/Output Example

Assume:

```sql
-- EMPLOYEES
EMP_ID | DEPT_ID
-------+--------
1      | 10
2      | 20
3      | 10

-- DEPARTMENTS
DEPT_ID | DEPT_NAME
--------+----------
10      | SALES
20      | HR
```

Query:
```sql
SELECT e.emp_id, e.dept_id, d.dept_name
FROM   employees e
       JOIN departments d ON e.dept_id = d.dept_id
ORDER BY e.emp_id;
```

Output (same regardless of join method):
```text
EMP_ID | DEPT_ID | DEPT_NAME
-------+---------+----------
1      | 10      | SALES
2      | 20      | HR
3      | 10      | SALES
```

The **result is identical**; only the internal **execution method** (nested loops, hash join, merge join) differs based on statistics, indexes, and optimizer decisions.
