
Dưới đây là tóm tắt nội dung bài giảng:

1. Khởi động các container bằng Docker Compose  
   - Đảm bảo không còn container nào đang chạy hoặc bị dừng; xóa hết để giải phóng tài nguyên.  
   - Di chuyển vào thư mục chứa file `docker-compose.yml`.  
   - Chạy lệnh: `docker compose up -d`.  
   - Thứ tự khởi động: `rabbitmq` → `config-server` → các microservice `accounts`, `loans`, `cards`.  
   - Dùng `docker ps` để kiểm tra các container đang chạy.

2. Kiểm tra cấu hình mặc định trong Postman  
   - Gọi API `localhost:8080/api/build-info` của service accounts.  
   - Nhận version `3.0` (cấu hình mặc định).  
   - Các thuộc tính cấu hình được lấy từ `commonconfig.yml` thông qua biến môi trường (ưu tiên cao hơn `application.yml`).  
   - Kiểm tra thêm:  
     - `JAVA_HOME` bên trong container.  
     - Endpoint thông tin liên hệ (contact info) → lấy từ profile `default`.  
   - Có thể kiểm tra tương tự cho services `cards` và `loans`.

3. Thiết lập webhook để auto-refresh cấu hình  
   - Cần webhook đang chạy để GitHub gửi sự kiện khi có commit mới.  
   - Sử dụng Hookdeck:  
     - Vào website Hookdeck, chọn “Test webhook”, “Add destination” để tạo session mới.  
     - Cài và dùng CLI:  
       - Nếu bị lỗi 401 khi login → chạy `hookdeck logout` rồi login lại.  
       - Chạy `hookdeck listen` với cổng `8071`.  
       - Cấu hình đường dẫn với host `localhost`.  
   - Lấy URL webhook từ Hookdeck, vào GitHub repo `eazybytes-config`:  
     - `Settings` → `Webhooks`.  
     - Chỉnh sửa webhook hiện có (nhập mật khẩu GitHub nếu cần).  
     - Thay URL payload bằng URL webhook mới → `Update webhook`.

4. Thử thay đổi cấu hình và lỗi 500  
   - Mở file `accounts.yml` trong repo config trên GitHub.  
   - Sửa một thuộc tính, ví dụ đổi `local` thành `docker`, rồi commit.  
   - Quan sát terminal Hookdeck: nhận trạng thái `500` → refresh thất bại.  

5. Phân tích lỗi: Config Server không kết nối được RabbitMQ  
   - Mở Docker Desktop → chọn container `config-server` → xem logs.  
   - Thấy config server cố kết nối RabbitMQ với `localhost:5672`.  
   - Điều này sai vì RabbitMQ chạy trong container riêng, không phải cùng container với config server.  
   - Trong `application.yml` có cấu hình:  
     - `spring.rabbitmq.host=localhost`  
     - `spring.rabbitmq.port=5672`  
     - username/password mặc định vẫn dùng được.  
   - Health check:  
     - `/actuator/health` → trạng thái `DOWN` (do RabbitMQ).  
     - `/actuator/health/readiness` → `UP` (RabbitMQ là optional trong cấu hình readiness).  
   - Vì readiness `UP`, các microservice `accounts`, `loans`, `cards` vẫn khởi động được.

6. Sửa lỗi: cấu hình RabbitMQ trong Docker Compose  
   - Cần cung cấp thông tin kết nối RabbitMQ cho các container (config server, accounts, loans, cards).  
   - Trong `docker-compose.yml`, thêm phần `environment` cho các service cần RabbitMQ.  
   - Thay vì dùng `localhost` cho `spring.rabbitmq.host`, dùng tên service của RabbitMQ trong compose, ví dụ `rabbit`.  
   - Trong `commonconfig.yml` (hoặc config chung), tạo biến môi trường:  
     - `spring_rabbitmq_host=rabbit` (có thể đặt trong dấu ngoặc kép).  
   - Lưu file, sau đó:  
     - `docker compose down` để dừng và xóa container.  
     - `docker compose up -d` để khởi động lại với cấu hình mới.

7. Xác nhận kết nối RabbitMQ và trạng thái hệ thống  
   - Xem lại logs của `config-server` trong Docker Desktop:  
     - Thấy log kết nối thành công tới RabbitMQ `rabbit:5672`.  
   - Kiểm tra health:  
     - `/actuator/health` → `UP`.  
     - `/actuator/health/readiness` → `UP`.  
   - Xác nhận các microservice:  
     - `accounts`, `cards`, `loans` đều khởi động thành công.

8. Kiểm tra lại auto-refresh cấu hình  
   - Trong Postman, gọi lại endpoint contact info của `accounts`:  
     - Trước đó giá trị là `docker` (do cấu hình mới).  
   - Đảm bảo terminal Hookdeck (webhook listener) vẫn đang chạy.  
   - Vào GitHub repo, mở `accounts.yml`:  
     - Đổi lại giá trị từ `docker` về `local`.  
     - Commit thay đổi.  
   - Quan sát terminal Hookdeck: lần này nhận trạng thái `200` → refresh thành công.  
   - Quay lại Postman, gọi lại endpoint contact info:  
     - Trước là `docker`, giờ trả về `local` → chứng minh auto-refresh hoạt động mà không cần restart container hay gọi actuator refresh thủ công.

9. Kết luận  
   - Thiết lập Docker Compose hiện đã hoạt động đúng với cấu hình mặc định.  
   - RabbitMQ được cấu hình đúng qua biến môi trường và tên service trong Docker network.  
   - Cơ chế auto-refresh cấu hình (qua GitHub + webhook + config server + RabbitMQ) hoạt động trong môi trường Docker.  
   - Bài giảng kết thúc bằng gợi ý: sẽ tiếp tục thiết lập tương tự cho các profile `prod` và `qa` trong bài tiếp theo.
