
Bài giảng mô tả cách triển khai **Retry Pattern** (mẫu thử lại) trong **Spring Cloud Gateway** cho microservice Loans:

1. **Cấu hình Retry trong Gateway**
   - Trong lớp cấu hình Gateway, sau `addResponseHeaderFilter()` thêm một filter mới `retry`.
   - Dùng dạng filter nhận **RetryConfig** (lambda `retryConfig -> { ... }`) để cấu hình chi tiết.

2. **Các thiết lập chính trong RetryConfig**
   - `setRetries(3)`: cho phép Gateway thử lại tối đa 3 lần (tổng cộng 4 request gồm cả lần đầu).
   - `setMethods(HttpMethod.GET)`: chỉ áp dụng retry cho các request **GET** (idempotent), tránh áp dụng cho POST, PATCH, PUT, DELETE để không gây tác dụng phụ.
   - `setBackoff(...)`: cấu hình backoff (thời gian chờ giữa các lần retry) với 4 tham số:
     1. **firstBackoff**: ví dụ 100 ms – thời gian chờ trước lần retry đầu tiên.
     2. **maxBackoff**: ví dụ 1000 ms – giới hạn tối đa thời gian chờ giữa hai lần retry.
     3. **factor**: hệ số nhân để tăng dần thời gian chờ dựa trên lần trước.
     4. **boolean**: cho biết có áp dụng hệ số nhân dựa trên backoff trước đó hay không (true = dùng backoff trước đó để nhân).

3. **Lý do chỉ retry GET**
   - GET thường là bất biến (idempotent), gọi nhiều lần không gây thay đổi dữ liệu.
   - Các phương thức như POST/PATCH/PUT có thể gây tác dụng phụ (tạo/sửa dữ liệu).
   - DELETE cũng có thể nguy hiểm nếu điều kiện truy vấn sai, nên không bật retry.

4. **Chuẩn bị demo trong LoansController**
   - Thêm log: `logger.debug("Invoked Loans contact-info API")` trong API GET `/contact-info`.
   - Dùng log để đếm số lần API được gọi khi Gateway retry.

5. **Demo 1 – Timeout & Retry**
   - Đặt breakpoint trong LoansController để cố tình làm chậm xử lý.
   - Gửi request GET `/contact-info` qua Gateway (Postman).
   - Gateway có global timeout ~2 giây, nhưng do retry 3 lần:
     - Tổng thời gian ~9 giây (1 request ban đầu + 3 lần retry).
   - Quan sát:
     - Phía client nhận lỗi `Gateway Timeout`.
     - Trong log LoansApplication, khi bỏ breakpoint, thấy **4 dòng log** “Invoked Loans contact-info API” → chứng minh 1 request + 3 lần retry.

6. **Demo 2 – RuntimeException & Retry**
   - Xóa breakpoint, thay vào đó:
     ```java
     throw new RuntimeException();
     ```
   - Comment các dòng code phía dưới vì không còn reachable.
   - Build lại, restart LoansApplication và Gateway.
   - Gửi lại request GET `/contact-info`:
     - Client nhận lỗi `Internal Server Error`.
     - Log LoansApplication hiển thị **4 lần** “Invoked Loans contact-info API” → Gateway vẫn retry khi gặp RuntimeException.
   - Điều này cho thấy Retry Pattern giúp dịch vụ có cơ hội phục hồi khi gặp lỗi tạm thời (network, mất điện, lỗi ngắn hạn).

7. **Dọn dẹp & kết luận**
   - Khôi phục LoansController, bỏ phần `RuntimeException` vì không muốn API luôn lỗi.
   - Kết luận:
     - Đã cấu hình **Retry Pattern** trong Gateway cho các API GET của microservice Loans.
     - Gateway tự động retry nhiều lần khi gặp lỗi (timeout, RuntimeException), tăng khả năng chịu lỗi mà không cần can thiệp thủ công từ client.
