
Dưới đây là tóm tắt ngắn gọn nội dung trên:

1. **Lý do cần validation**  
   - REST API đang nhận nhiều loại dữ liệu (name, email, mobile, accountNo, accountType, branchAddress…).  
   - Nếu dữ liệu sai (số điện thoại không đủ 10 số, email sai format, tên quá ngắn, accountNo không đủ 10 số…), vẫn gửi query xuống DB → tốn tài nguyên, không chuẩn.  
   - Chuẩn tốt là phải **từ chối ngay** dữ liệu không hợp lệ ở tầng API.

2. **Thêm dependency validation**  
   - Sử dụng dependency `spring-boot-starter-validation` (Jakarta Validation) trong `pom.xml`.  
   - Dependency này cung cấp các annotation như `@NotEmpty`, `@Email`, `@Pattern`, `@Size`,…

3. **Validation trong DTO**  
   - Thực hiện validation trên các lớp DTO nhận input từ client: `CustomerDto`, `AccountsDto`.  
   - **CustomerDto**:  
     - `name`:  
       - `@NotEmpty(message = "Tên không được để trống hoặc rỗng")`  
       - `@Size(min = 5, max = 30, message = "Độ dài tên khách hàng phải từ 5 đến 30")`  
     - `email`:  
       - `@NotEmpty(message = "Địa chỉ email không được để trống hoặc rỗng")`  
       - `@Email(message = "Địa chỉ email phải là giá trị hợp lệ")`  
     - `mobileNumber`:  
       - `@Pattern(regexp = "^[0-9]{10}$", message = "Số điện thoại di động phải có mười chữ số")`  
   - **AccountsDto**:  
     - `accountNumber`:  
       - `@NotEmpty(message = "Số tài khoản không được để trống hoặc rỗng")`  
       - `@Pattern(regexp = "^[0-9]{10}$", message = "Số tài khoản phải có mười chữ số")`  
     - `accountType`:  
       - `@NotEmpty(message = "Loại tài khoản không được để trống hoặc rỗng")`  
     - `branchAddress`:  
       - `@NotEmpty(message = "Địa chỉ chi nhánh không được để trống hoặc rỗng")`  
   - `ResponseDto` và `ErrorResponseDto` không cần validation vì chỉ dùng để trả về response, không nhận input.

4. **Kích hoạt validation trong Controller**  
   - Trong `AccountsController`:  
     - Thêm `@Validated` trên class để bật validation cho toàn bộ REST API trong controller.  
     - Với các method nhận body (`CustomerDto`, `AccountsDto`):  
       - Thêm `@Valid` trước `@RequestBody`, ví dụ:  
         `public ResponseEntity<?> createAccount(@Valid @RequestBody CustomerDto customerDto)`  
     - Với các API dùng query param (ví dụ `mobileNumber` trong get/delete):  
       - Thêm `@Pattern` trực tiếp trên tham số method:  
         `@RequestParam @Pattern(regexp = "^[0-9]{10}$", message = "Số điện thoại di động phải có mười chữ số") String mobileNumber`

5. **Xử lý lỗi validation toàn cục**  
   - Trong `GlobalExceptionHandler`:  
     - Class extends `ResponseEntityExceptionHandler`.  
     - Override method `handleMethodArgumentNotValid(...)`.  
     - Bên trong:  
       - Tạo `Map<String, String> validationErrors = new HashMap<>();`  
       - Lấy danh sách lỗi: `ex.getBindingResult().getAllErrors()`  
       - Với mỗi lỗi, lấy:  
         - Tên field (fieldName)  
         - Thông báo lỗi (defaultMessage – chính là message đã khai báo trong annotation)  
       - Đưa vào map `validationErrors`.  
       - Trả về `ResponseEntity` chứa map này với HTTP status `BAD_REQUEST (400)`.

6. **Kiểm thử bằng Postman**  
   - Gửi request tạo tài khoản với dữ liệu sai:  
     - Email thiếu `@` hoặc `.com`,  
     - Số điện thoại 9 chữ số,  
     - Tên chỉ 1–2 ký tự…  
   - Nhận lại JSON chứa các lỗi validation tương ứng (name, email, mobileNumber…).  
   - Khi gửi dữ liệu hợp lệ → API hoạt động bình thường.  
   - Tương tự kiểm thử cho API get, update, delete:  
     - Nếu `mobileNumber` không đủ 10 số → trả về lỗi validation, không gọi DB.  
     - Nếu đủ 10 số → xử lý bình thường.

7. **Kết luận**  
   - Đã áp dụng chuẩn validation đầu vào cho tất cả REST API:  
     - Validation ở DTO + annotation trong Controller + GlobalExceptionHandler.  
   - Đây là cách tiếp cận chuẩn, có thể mở rộng thêm tùy yêu cầu nghiệp vụ.
