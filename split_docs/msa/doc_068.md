
Bài giảng nói về cách **quản lý và ngoại hóa cấu hình** trong ứng dụng **Spring Boot** cho các dịch vụ vi mô.

1. **Mục tiêu**  
   - Tách cấu hình ra khỏi mã nguồn để cùng một code chạy được trên nhiều môi trường (dev, QA, production) mà không cần build lại.  
   - Giải quyết bài toán: làm sao thay đổi các thuộc tính (DB URL, username, password, …) theo môi trường.

2. **Các nguồn cấu hình mà Spring Boot hỗ trợ**  
   - `application.properties` hoặc `application.yml` (trong classpath) – mức ưu tiên thấp nhất.  
   - Các file properties/yaml khác (có thể cấu hình để Spring Boot đọc).  
   - Java System Properties.  
   - JNDI Properties.  
   - Servlet config / init params.  
   - **Environment variables** (biến môi trường hệ điều hành).  
   - **Command-line arguments** (đối số dòng lệnh) – mức ưu tiên cao nhất.

   Nếu cùng một thuộc tính xuất hiện ở nhiều nơi, Spring Boot sẽ áp dụng **thứ tự ưu tiên**:  
   - Cái ở “dưới” (ví dụ: command-line args) sẽ **ghi đè** cái ở “trên” (ví dụ: application.properties).

3. **Vấn đề với việc để cấu hình trong file ứng dụng**  
   - Khi cấu hình nằm trong `application.properties`/`application.yml` trong code, ta đang “gộp” cấu hình với mã nguồn.  
   - Điều này không phù hợp khi triển khai nhiều môi trường, vì mỗi môi trường cần giá trị khác nhau.  
   - Do đó, ta thường dùng biến môi trường, command-line args… để **ghi đè** cấu hình mặc định.

4. **Ba cách đọc thuộc tính trong code Spring Boot**  

   a. **Dùng annotation `@Value`**  
      - Khai báo một field trong class và gắn `@Value("${ten.thuoc.tinh}")`.  
      - Khi ứng dụng khởi động, Spring Boot tìm thuộc tính đó trong tất cả nguồn cấu hình (properties, env, args, …) theo thứ tự ưu tiên.  
      - Phù hợp khi chỉ cần đọc **ít thuộc tính**.

   b. **Dùng `Environment` interface**  
      - Autowire `Environment` vào class.  
      - Dùng `environment.getProperty("ten.thuoc.tinh")` để đọc giá trị.  
      - Thường dùng khi nhiều cấu hình được đặt trong **biến môi trường** (do admin/server script thiết lập).  
      - Cũng chỉ tiện khi số lượng thuộc tính không quá nhiều.

   c. **Dùng `@ConfigurationProperties`**  
      - Giải quyết nhược điểm của hai cách trên:  
        + Không phải hard-code key thuộc tính nhiều lần.  
        + Không phải đọc từng thuộc tính một.  
      - Cách làm:  
        + Định nghĩa các thuộc tính trong file cấu hình với **cùng một prefix** (ví dụ: `accounts.*`).  
        + Tạo một class Java, gắn `@ConfigurationProperties(prefix = "accounts")`.  
        + Khai báo các field tương ứng với các thuộc tính (tên field và kiểu dữ liệu phải khớp).  
        + Khi ứng dụng khởi động, Spring Boot tự động **bind** các giá trị cấu hình vào các field.  
        + Chỉ cần dùng các getter để lấy giá trị.  
      - Đây là **cách được khuyến nghị** khi có **nhiều thuộc tính**.

5. **Kết luận**  
   - Spring Boot hỗ trợ nhiều cách để ngoại hóa cấu hình và có một **thứ tự ưu tiên rõ ràng** giữa các nguồn cấu hình.  
   - Có ba cách phổ biến để đọc cấu hình trong code: `@Value`, `Environment`, `@ConfigurationProperties`.  
   - `@ConfigurationProperties` là lựa chọn tốt nhất khi số lượng thuộc tính lớn.  
   - Các bài giảng tiếp theo sẽ demo chi tiết từng cách và chỉ ra nhược điểm của các phương pháp cơ bản.
