
Bài giảng giải thích vì sao nên có một **Edge Server / API Gateway riêng** trong kiến trúc microservices.

1. **Vấn đề khi không dùng Edge Server**  
   - Mọi client bên ngoài gọi trực tiếp vào từng microservice (account, loan, card, …).  
   - Có thể có hàng trăm service, mỗi service phải tự xử lý:
     - Bảo mật (authentication/authorization)  
     - Logging, auditing  
     - Routing  
     - Các yêu cầu phi chức năng khác  
   - Dẫn đến:
     - Lặp lại cùng một logic ở nhiều nơi  
     - Nhiều developer triển khai khác nhau → thiếu nhất quán về bảo mật, logging, kiểm tra, routing…

2. **Giải pháp thư viện chung và nhược điểm**  
   - Ý tưởng: gom các mối quan tâm xuyên suốt vào một **thư viện dùng chung** rồi add vào tất cả service.  
   - Nhược điểm:
     - Tạo **kết nối chặt chẽ** giữa thư viện và các service.  
     - Mỗi lần thay đổi logic bảo mật, logging… phải:
       - Kiểm tra tương thích với tất cả service  
       - Làm regression test, phân tích tác động trên rất nhiều service  
     - Không khả thi khi số lượng service lớn.  
   → Vì vậy, cách này cũng không được khuyến khích.

3. **Giải pháp: Edge Server / API Gateway**  
   - Đặt một **cổng duy nhất** giữa client và toàn bộ mạng microservices.  
   - Edge Server nhận mọi request từ bên ngoài, xử lý các mối quan tâm xuyên suốt, sau đó mới forward đến service thực.  
   - Lợi ích:
     - Tập trung triển khai:
       - Bảo mật, logging, auditing  
       - Ngăn lỗi dây chuyền, tăng khả năng chịu lỗi và phục hồi cho downstream services  
       - Timeout, retry cho các call nội bộ  
       - Chính sách quota / rate limit theo từng gói (standard, premium, advanced…)  
     - Dễ quản lý, dễ thay đổi, đảm bảo **tính nhất quán**.

4. **Các chức năng cụ thể của API Gateway**  
   - Xác thực, ủy quyền.  
   - Whitelist/blacklist IP, danh sách cho phép/loại trừ.  
   - Rate limiting: giới hạn số request theo client.  
   - Dynamic routing, service discovery.  
   - Sửa đổi request/response.  
   - Chuyển đổi giao thức (ví dụ: ngoài dùng HTTPS, bên trong dùng HTTP).  
   - Không bắt buộc phải bật hết; tùy nhu cầu kinh doanh.  
   - Có thể:
     - Xử lý ngoại lệ tập trung.  
     - Triển khai **circuit breaker** để tăng khả năng chịu lỗi.  
     - Gửi log/metrics sang các công cụ như Grafana để giám sát.  
     - Tích hợp cache như Redis để tối ưu hiệu năng.

5. **Vì sao Eureka không thay thế được API Gateway**  
   - Eureka Server chỉ phục vụ:
     - Service discovery  
     - Service registration  
   - Không có khả năng xử lý các yêu cầu phi chức năng, mối quan tâm xuyên suốt như API Gateway.  
   - Tùy tổ chức:
     - Có thể chỉ dùng Eureka  
     - Hoặc dùng cả Eureka và API Gateway, mỗi thành phần giải quyết một nhóm vấn đề khác nhau.

6. **Kết luận**  
   - Việc có một Edge Server / API Gateway riêng là lựa chọn hợp lý để:
     - Tránh lặp lại logic  
     - Đảm bảo nhất quán  
     - Tăng khả năng chịu lỗi, bảo mật, giám sát  
   - Dù nghe có vẻ phức tạp, nhưng với **Spring Boot** và **Spring Cloud**, việc xây dựng API Gateway/Edge Server cho microservices trở nên dễ dàng.
