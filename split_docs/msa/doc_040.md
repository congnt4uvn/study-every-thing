
Dưới đây là tóm tắt nội dung bài giảng:

1. Bối cảnh trước đây  
- Trước khi có điện toán đám mây, mỗi tổ chức phải mua máy chủ vật lý, cài hệ điều hành, triển khai ứng dụng lên đó, gán IP public và ánh xạ với domain để người dùng truy cập.  

2. Sự ra đời của máy ảo (VM)  
- Với cloud (AWS, Azure, GCP…), nhà cung cấp có các máy chủ vật lý lớn trong data center.  
- Họ dùng phần mềm “trình quản lý ảo” (hypervisor) để chia nhỏ một máy vật lý thành nhiều máy ảo (VM1, VM2, VM3…).  
- Mỗi VM có thể cài hệ điều hành riêng (Windows, Linux, Mac…), RAM và ổ cứng được chia ảo từ máy vật lý.  
- Mỗi VM được gán IP public, ánh xạ với domain để triển khai và truy cập microservice.

3. Vấn đề khi dùng VM cho microservices  
- Nếu có nhiều microservice (ví dụ 100), không thể tạo 100 VM: tốn kém, phức tạp, chi phí cloud cao.  
- Một VM thường khá “to” (nhiều RAM, disk), nhưng mỗi microservice lại nhỏ → lãng phí tài nguyên.  
- Nếu dồn nhiều microservice vào 1 VM:  
  - Mỗi service có thể cần phiên bản Java/ngôn ngữ khác nhau (Java 8, Java 17, Python…) → xung đột môi trường.  
  - Restart VM sẽ làm tất cả service trên VM đó dừng theo → rủi ro lớn.  
- Khả năng mở rộng (scaling) kém:  
  - Tạo thêm VM mới để tăng số instance của một microservice mất nhiều thời gian (khoảng 15 phút để tạo VM, cài OS, cài JDK, server, thư viện…).  
  - Thu nhỏ (xóa VM) cũng mất thêm 5–10 phút.  
  - Như vậy không đáp ứng được yêu cầu “scale nhanh” của microservices.

4. Container và sự khác biệt với VM  
- Hạ tầng vật lý vẫn giống: máy chủ vật lý + hệ điều hành host (Windows/Linux/Mac).  
- Thay vì dùng hypervisor + guest OS cho mỗi VM, ta cài công cụ container (Docker).  
- Docker cho phép tạo nhiều container trên cùng một host:  
  - Không cần hệ điều hành riêng cho từng container, các container dùng chung OS của host → nhẹ hơn rất nhiều.  
  - Tạo, xóa, khởi động lại container chỉ mất vài giây.  

5. Ưu điểm của container  
- Nhẹ, khởi động nhanh, hủy nhanh → phù hợp cho scale up/down tức thời.  
- Mỗi container là một “gói đóng” chứa:  
  - Ứng dụng (microservice) + tất cả dependency (JDK, thư viện Spring Boot, runtime, v.v.).  
  - Khi chuyển sang môi trường khác, chỉ cần mang “gói” đó sang và chạy một lệnh để tạo container → giải quyết vấn đề tính di động (portability).  
- Cô lập môi trường:  
  - Trên cùng một host, container 1 có thể dùng Java 8, container 2 dùng Java 17, container 3 dùng Python… mà không xung đột.  
  - Mỗi container có mạng ảo, tài nguyên, bộ nhớ, lưu trữ riêng; container khác không truy cập được nếu không được cho phép.  

6. Kết luận  
- Khác biệt cốt lõi:  
  - VM: cần hypervisor, mỗi VM có guest OS riêng → nặng, khởi động chậm, scale chậm.  
  - Container: không có guest OS, dùng chung OS host, dùng Docker để quản lý → nhẹ, nhanh, dễ triển khai và mở rộng.  
- Vì vậy, khi triển khai microservices, nên tránh dùng VM cho từng service, thay vào đó sử dụng container (Docker) để tận dụng các lợi ích về hiệu năng, chi phí, khả năng mở rộng và tính di động.
