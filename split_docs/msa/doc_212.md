
Dưới đây là tóm tắt ngắn gọn nội dung slide/bài giảng:

1. **Trạng thái hiện tại của hệ thống sự kiện**  
   - Trước đó mới làm xong luồng sự kiện: Account Microservice → Messaging Microservice (tài khoản → hàng đợi → messaging).  
   - Bây giờ mở rộng thêm luồng ngược lại: Messaging Microservice → Account Microservice để hoàn thiện quy trình nghiệp vụ.

2. **Cấu hình Messaging Microservice (gửi sự kiện đi)**  
   - Đã có ràng buộc đầu vào: `emailsms-in-0` đọc từ hàng đợi `send-communication`.  
   - Cần thêm ràng buộc đầu ra trong `application.yml`:
     - Tên binding: `emailsms-out-0`.  
     - `spring.cloud.function.definition` chứa prefix `emailsms|...` rồi thêm `out-0` để chỉ output binding.  
     - `destination` mới: `communication-send` (tên exchange trong RabbitMQ, được tạo tự động).  
   - Nhờ Spring Cloud Function + Spring Cloud Stream:
     - Không cần dùng `StreamBridge` hay code thủ công để gửi message.  
     - Output của hàm `emailsms` (hàm thứ hai `sms` trả về `Long accountNumber`) sẽ tự động được gửi đến exchange `communication-send`.

3. **Ý nghĩa nghiệp vụ ở Messaging Microservice**  
   - Hàm `sms` trả về `Long accountNumber`.  
   - Messaging service phát ra message chứa `accountNumber`.  
   - Account service sẽ dùng `accountNumber` để lấy thông tin tài khoản và cập nhật trạng thái “communication sent” trong DB.

4. **Cấu hình Account Microservice (nhận sự kiện)**  

   a. **Tạo hàm xử lý message**  
   - Tạo package: `com.eazybyte.accounts.function`.  
   - Tạo class `AccountsFunctions` với:  
     - `@Configuration`  
     - Logger  
     - Định nghĩa một bean hàm:  
       ```java
       @Bean
       public Consumer<Long> updateCommunication(IAccountService accountService) {
           return accountNumber -> {
               log.info("Updating communication status for account: {}", accountNumber);
               accountService.updateCommunicationStatus(accountNumber);
           };
       }
       ```
     - Dùng `Consumer<Long>` vì:
       - Chỉ nhận input (accountNumber), không trả output.  
       - Phù hợp với kịch bản: nhận message, cập nhật DB, không phản hồi.

   b. **Chuẩn bị DB và entity**  
   - Trong `schema.sql`:
     - Thêm cột mới trong bảng `accounts`: `communication_switch BOOLEAN`.  
   - Trong `AccountsEntity`:
     - Thêm field `Boolean communicationSwitch` và map với cột `communication_switch`.

   c. **Logic service cập nhật trạng thái**  
   - Trong `IAccountService`:
     - Thêm method: `Boolean updateCommunicationStatus(Long accountNumber);`  
   - Trong `AccountServiceImpl`:
     - Cài đặt:
       - Kiểm tra `accountNumber` khác null.  
       - `findById(accountNumber)`:
         - Nếu không có: ném `ResourceNotFoundException`.  
         - Nếu có: set `communicationSwitch = true`, `save` lại.  
       - Trả về `true` nếu cập nhật thành công.

   d. **Cấu hình Spring Cloud Function & Stream trong Accounts**  
   - Trong `application.yml` của Accounts:
     - Khai báo:
       ```yaml
       spring:
         cloud:
           function:
             definition: updateCommunication
       ```
       (Tên phải trùng với tên hàm `updateCommunication`).
     - Khai báo binding input:
       ```yaml
       spring:
         cloud:
           stream:
             bindings:
               updateCommunication-in-0:
                 destination: communication-send
                 group: <ten-nhom>
       ```
       - `updateCommunication-in-0`: tên binding input, theo chuẩn: `<functionName>-in-0`.  
       - `destination: communication-send`: phải trùng với destination mà Messaging service gửi tới.  
       - `group`: để tên hàng đợi cố định, không bị thêm hậu tố ngẫu nhiên.

5. **Cách Spring Cloud Stream nối hai microservice**  
   - Messaging service:
     - Output binding: `emailsms-out-0` → `destination: communication-send` (exchange).  
   - Account service:
     - Input binding: `updateCommunication-in-0` → `destination: communication-send` (queue được bind vào exchange).  
   - RabbitMQ:
     - Exchange `communication-send` nhận message từ Messaging service.  
     - Forward message đến queue tương ứng với `destination + "." + group`.  
   - Nhờ đó:
     - Messaging gửi `accountNumber`.  
     - Accounts nhận `accountNumber` qua hàm `updateCommunication`, cập nhật `communication_switch = true`.

6. **Lợi ích của Spring Cloud Function & Stream**  
   - Không cần code thủ công để gửi/nhận message (không cần `StreamBridge`, không cần nhiều cấu hình phức tạp).  
   - Chỉ cần:
     - Định nghĩa hàm (Function/Consumer/Supplier).  
     - Khai báo `spring.cloud.function.definition`.  
     - Khai báo bindings (in/out) với `destination`.  
   - Dễ dàng chuyển đổi giữa các broker (RabbitMQ, Kafka, v.v.) mà không phải sửa nhiều code.

Tóm lại:  
- Messaging Microservice được cấu hình để phát `accountNumber` ra exchange `communication-send` thông qua `emailsms-out-0`.  
- Account Microservice định nghĩa hàm `updateCommunication` (Consumer<Long>) nhận `accountNumber` từ queue `communication-send`, cập nhật cột `communication_switch` trong DB.  
- Tất cả được kết nối tự động nhờ Spring Cloud Function + Spring Cloud Stream.
