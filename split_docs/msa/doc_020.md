
Dưới đây là tóm tắt ngắn gọn nội dung bài giảng:

1. Mục tiêu  
   - Xây dựng một REST API để lấy thông tin khách hàng và tài khoản ngân hàng từ DB, dựa trên số điện thoại di động (mobileNumber) được truyền từ client.

2. Thay đổi ở AccountsController  
   - Tạo phương thức mới: `fetchAccountDetails()`.
   - Dùng:
     - `@GetMapping("/fetch")` vì đây là API đọc dữ liệu.
     - `@RequestParam String mobileNumber` để nhận số điện thoại qua query param.
   - Gọi service `IAccountService.fetchAccount(mobileNumber)` và trả về `ResponseEntity.ok(customerDto)`.

3. Thay đổi ở IAccountService và AccountServiceImpl  
   - Trong `IAccountService`: khai báo phương thức trừu tượng `CustomerDto fetchAccount(String mobileNumber)`.
   - Trong `AccountServiceImpl`: implement `fetchAccount()`:
     1. Dùng `customerRepository.findByMobileNumber(mobileNumber)`:
        - Nếu không có, dùng `orElseThrow()` để ném `ResourceNotFound` (resource = "Customer", fieldName = "mobileNumber", fieldValue = mobileNumber).
        - Nếu có, lưu vào biến `customer`.
     2. Dùng `accountsRepository.findByCustomerId(customer.getCustomerId())`:
        - Nếu không có, tiếp tục ném `ResourceNotFound` (resource = "Account", fieldName = "customerId", fieldValue = customerId).
     3. Chuyển đổi entity sang DTO:
        - Dùng `CustomerMapper.mapToCustomerDto(customer, new CustomerDto())` → `customerDto`.
        - Dùng `AccountsMapper.mapToAccountsDto(accounts, new AccountsDto())`.
        - Gán `customerDto.setAccountsDto(accountsDto)`.
     4. Trả về `customerDto`.

4. Tạo ngoại lệ tùy chỉnh ResourceNotFound  
   - Tạo class `ResourceNotFound` trong package exception.
   - Kế thừa `RuntimeException`.
   - Constructor nhận 3 tham số:
     - `String resourceName` (Customer/Account),
     - `String fieldName` (mobileNumber/customerId),
     - `String fieldValue`.
   - Dùng `String.format()` để tạo một thông điệp lỗi duy nhất, ví dụ:
     - “Resource %s not found for field %s with value %s”.
   - Đánh dấu HTTP status là `HttpStatus.NOT_FOUND`.

5. Cập nhật Global Exception Handler  
   - Thêm một phương thức handler mới cho `ResourceNotFound`.
   - Tương tự handler của `CustomerException`, nhưng trả về `HttpStatus.NOT_FOUND` thay vì `BAD_REQUEST`.
   - Trả về một response chứa:
     - mã lỗi,
     - thông điệp lỗi,
     - thông tin input (ví dụ số điện thoại).

6. Thay đổi ở AccountsRepository  
   - Thêm phương thức mới: `Optional<Accounts> findByCustomerId(Long customerId)`.
   - Dùng phương thức này trong `AccountServiceImpl` để lấy tài khoản dựa trên `customerId` lấy được từ entity `Customer`.

7. Thiết kế DTO  
   - Đã có `CustomerDto` và `AccountsDto`.
   - Thay vì tạo một DTO mới tổng hợp, tái sử dụng:
     - Thêm một field `AccountsDto accountsDto` trong `CustomerDto`.
     - Khi trả về, `CustomerDto` chứa thông tin khách hàng + một object con chứa thông tin tài khoản.

8. Kiểm thử API  
   - Vì dùng H2 in-memory DB, mỗi lần server restart dữ liệu bị mất → phải gọi lại API tạo tài khoản để seed dữ liệu.
   - Gọi API:
     - `GET http://localhost:8080/api/fetch?mobileNumber=...`
   - Trường hợp số điện thoại tồn tại:
     - Nhận JSON chứa:
       - tên khách hàng, email, mobileNumber,
       - object `accountsDto` với accountNumber, accountType, branchAddress.
   - Trường hợp số điện thoại không tồn tại:
     - Nhận response lỗi với:
       - mã lỗi,
       - thông điệp “Customer not found for mobileNumber ...”,
       - giá trị mobileNumber đã gửi.

9. Sửa lỗi trong quá trình demo  
   - Ban đầu handler ngoại lệ bị viết nhầm, dùng sai loại exception.
   - Sau khi sửa lại để handler đúng `ResourceNotFound`, API trả về lỗi đúng như thiết kế.
   - Nhấn mạnh tầm quan trọng của debug và unit test để phát hiện các lỗi nhỏ kiểu này.

Tóm lại, bài giảng hướng dẫn hoàn chỉnh việc:
- thêm API GET `/fetch` nhận `mobileNumber`,
- truy vấn Customer và Account theo mobileNumber,
- chuyển entity → DTO,
- xử lý trường hợp không tìm thấy bằng `ResourceNotFound`,
- và trả về response chuẩn REST với HTTP status phù hợp.
