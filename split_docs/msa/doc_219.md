
Dưới đây là tóm tắt nội dung:

1. Chuẩn bị mã cho Section 14  
   - Tạo thư mục mới trong workspace, sao chép mã từ Section 13 sang, rồi đổi tên thành `section14`.  
   - Mở project `section14` bằng IntelliJ, load lại toàn bộ Maven projects.

2. Chuyển từ RabbitMQ sang Kafka trong AccountsMicroservice và MessageMicroservice  
   - Mở `pom.xml` của AccountsMicroservice:  
     - Thay dependency Spring Cloud Stream Binder từ RabbitMQ sang Kafka (`rabbit` → `kafka`).  
     - Cập nhật tag version từ `s13` sang `s14`.  
   - Làm tương tự với `pom.xml` của MessageMicroservice.  
   - Các microservice khác chỉ cần đổi tag version khi build Docker image (sẽ làm sau).

3. Cấu hình application.yml cho Kafka  
   - Trong MessageMicroservice:  
     - Xóa toàn bộ cấu hình liên quan đến RabbitMQ.  
     - Thêm cấu hình Kafka binder:  
       - `spring.cloud.stream.kafka.binder.brokers: localhost:9092` (Kafka broker local).  
   - Trong AccountsMicroservice:  
     - Xóa cấu hình RabbitMQ.  
     - Thêm cấu hình tương tự: `spring.cloud.stream.kafka.binder.brokers: localhost:9092`.  
   - Không cần thay đổi code business; chỉ thay dependency và cấu hình kết nối là đủ nhờ Spring Cloud Stream.

4. Khởi động hệ thống microservices với Kafka  
   - Khởi động: Config Server, Eureka Server, AccountsApplication, MessageApplication, GatewayServerApplication.  
   - Kafka đang chạy local ở background, các service đã kết nối tới Kafka.

5. Kiểm tra topic và consumer bằng plugin Kafkalytic trong IntelliJ  
   - Cài plugin Kafkalytic từ Marketplace.  
   - Thêm Kafka broker: `localhost:9092`, test connection thành công.  
   - Quan sát cluster:  
     - Có 1 broker.  
     - Có 2 consumer (Accounts và Message) vì giao tiếp hai chiều.  
     - Các topic được tạo: `communication-sent`, `send-communication` (theo cấu hình destination trong `application.yml`).  
     - Có thêm topic offset dành cho nội bộ Kafka.

6. Kiểm thử luồng bất đồng bộ giữa Accounts và Message  
   - Khởi động Keycloak (qua Docker) vì Gateway dùng OAuth2:  
     - Truy cập `localhost:7080`, đăng nhập admin/admin, vào `clients`, lấy client `eazybank-callcenter-cc` để dùng trong Postman.  
   - Đặt breakpoint trong `MessageFunction` (hàm `email`) của MessageMicroservice.  
   - Trong Postman:  
     - Lấy access token từ Keycloak bằng client vừa lấy.  
     - Gọi API tạo account trong AccountsMicroservice qua Gateway.  
     - Nhận response thành công; đồng thời message được publish lên Kafka, nên breakpoint dừng tại hàm `email`.  
   - Mở H2 console của AccountsMicroservice:  
     - Trước khi tiếp tục breakpoint, cột `communication` trong bảng `accounts` là `false`/null.  
     - Sau khi resume breakpoint, chạy lại query: cột `communication` được cập nhật thành `true`.  
   - Điều này chứng minh luồng end-to-end bất đồng bộ giữa Accounts và Message hoạt động tốt với Kafka.

7. Ý nghĩa của Spring Cloud Stream  
   - Việc chuyển từ RabbitMQ sang Kafka chỉ cần:  
     - Thay dependency binder.  
     - Thay cấu hình kết nối.  
   - Không cần sửa logic code nhờ Spring Cloud Stream và Spring Cloud Function.  
   - So với cách kết nối “cổ điển” (tự dùng client Kafka/RabbitMQ), Spring Cloud Stream giúp chuyển đổi giữa các nền tảng messaging rất nhanh và đơn giản.

8. Bước tiếp theo (chuẩn bị cho môi trường Docker)  
   - Tạo Docker image mới cho Section 14, push lên Docker Hub.  
   - Cập nhật file `docker-compose` để dùng image Section 14 và cấu hình Kafka tương ứng.  
   - Kiểm thử lại toàn bộ luồng trong môi trường Docker.

Đó là toàn bộ nội dung chính: chuyển hệ thống từ RabbitMQ sang Kafka bằng Spring Cloud Stream, kiểm thử thành công trên môi trường local, và chuẩn bị bước sang môi trường Docker.
