
Dưới đây là tóm tắt nội dung bài giảng:

1. **Thay H2 bằng MySQL trong các microservice**  
   - Mở `pom.xml` của từng service (accounts, cards, loans).  
   - Xóa dependency H2.  
   - Thêm dependency MySQL:  
     - `groupId`: `com.mysql`  
     - `artifactId`: `mysql-connector-j`  
   - Reload Maven cho từng project.

2. **Cấu hình MySQL trong `application.yml` của Accounts**  
   - Thay URL H2 bằng URL MySQL, ví dụ:  
     `jdbc:mysql://localhost:3306/accountsdb`  
   - Xóa `driver-class-name` (Spring Boot tự nhận).  
   - Đặt `username: root`, `password: root` (chỉ dùng cho môi trường local/demo).  
   - Xóa các cấu hình H2 và một số cấu hình JPA không cần thiết như `hibernate.ddl-auto`.  
   - Giữ `show-sql: true` để xem câu lệnh SQL trên console.  
   - Giải thích:  
     - Trong thực tế, không hard-code credential DB production trong `application.yml`.  
     - Nên dùng biến môi trường, tham số CLI, JVM system properties, Docker Compose, hoặc trong Kubernetes thì dùng ConfigMap và Secret.

3. **Tự động chạy `schema.sql` với MySQL**  
   - Với H2, Spring Boot tự động chạy `schema.sql` khi khởi động.  
   - Với MySQL (hoặc DB “thật”), Spring Boot không tự làm như vậy.  
   - Để Spring Boot luôn chạy `schema.sql`, cấu hình:  
     ```yaml
     spring:
       datasource:
         url: ...
         username: ...
         password: ...
         sql:
           init:
             mode: always
     ```  
   - Lưu ý: `spring.sql.init.mode` phải nằm dưới `spring.datasource` (trong bài giảng ban đầu đặt nhầm dưới `jpa` nên không chạy).  
   - Trong `schema.sql` nên dùng `IF NOT EXISTS` để tránh lỗi khi bảng đã tồn tại.

4. **Sao chép cấu hình sang Cards và Loans**  
   - Copy phần `spring.datasource` và `spring.jpa` từ Accounts sang Cards và Loans.  
   - Chỉ thay đổi:  
     - Cards: `jdbc:mysql://localhost:3308/cardsdb`  
     - Loans: `jdbc:mysql://localhost:3307/loansdb`  
   - Đảm bảo `spring.datasource.sql.init.mode: always` cũng được cấu hình đúng cho hai service này.

5. **Khởi động hệ thống**  
   - Khởi động Config Server trước (vì các microservice phụ thuộc vào nó).  
   - Sau đó khởi động lần lượt: Accounts, Cards, Loans.  
   - Kiểm tra trong client SQL (ví dụ SQL Electron):  
     - Accounts DB: có bảng `accounts`, `customers`.  
     - Loans DB: có bảng `loans`.  
     - Cards DB: có bảng `cards`.  
   - Xác nhận các bảng được tạo từ `schema.sql` của từng microservice.

6. **Kiểm thử với Postman**  
   - Gửi request tạo account (Accounts service) với một số điện thoại.  
   - Dùng cùng số điện thoại để tạo card (Cards service).  
   - Tạo loan (Loans service).  
   - Kiểm tra lại trong DB:  
     - Cards DB: bảng `cards` có bản ghi vừa tạo.  
     - Loans DB: bảng `loans` có bản ghi vừa tạo.  
     - Accounts DB: bảng `customers` và `accounts` có dữ liệu tương ứng.

7. **Hành vi của dữ liệu khi dùng Docker container**  
   - Dừng tất cả microservice.  
   - Dừng các container DB (accounts, loans, cards).  
   - Xóa container DB của cards.  
   - Khởi động lại container DB accounts và loans, tạo lại container DB cards mới.  
   - Kết quả:  
     - Accounts DB và Loans DB vẫn còn dữ liệu (vì chỉ dừng, không xóa container).  
     - Cards DB mất toàn bộ bảng và dữ liệu (vì container bị xóa và tạo mới).  
   - Khởi động lại các microservice.  
   - Gọi API fetch:  
     - Accounts: vẫn trả về dữ liệu.  
     - Loans: vẫn trả về dữ liệu.  
     - Cards: không có dữ liệu (vì DB mới hoàn toàn).

8. **Kết luận về việc dùng MySQL trong Docker**  
   - Ưu điểm: khởi động DB nhanh, dễ tạo môi trường local.  
   - Nhược điểm: nếu xóa container, toàn bộ dữ liệu bên trong sẽ mất vĩnh viễn.  
   - Khuyến nghị:  
     - Có thể stop container nhưng tránh xóa nếu muốn giữ dữ liệu.  
     - Hiểu rõ rủi ro khi dùng container DB trong môi trường local/demo.  

Bài giảng kết thúc bằng việc nhấn mạnh cách cấu hình MySQL cho các microservice, cách tự động chạy `schema.sql`, và lưu ý về việc bảo toàn dữ liệu khi sử dụng Docker container.
