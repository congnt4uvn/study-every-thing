
Dưới đây là tóm tắt nội dung:

1. Chuẩn bị và chạy hệ thống
- Đã build 6 Docker image (tag s11) cho các microservice.
- Sử dụng docker-compose (profile prod) để chạy toàn bộ hệ thống.
- Gặp lỗi do sai lệch thụt lề (indent) trong file docker-compose YAML, chỉnh lại cho các service thẳng hàng rồi chạy lại.
- Tất cả container (Minio read/write, Promtail, Grafana, gateway, các microservice) khởi động thành công sau 1–2 phút.
- Nếu máy yếu (CPU/RAM thấp), có thể tăng thời gian chờ (delay) để container có đủ thời gian khởi động và báo “healthy”.

2. Tạo log bằng cách gọi API
- Dùng Postman gọi các API:
  - fetchCustomerDetails (account service) → ban đầu NotFound vì DB H2 trống.
  - create account → tạo dữ liệu tài khoản.
  - create card → ban đầu timeout, sau đó request invalid nhưng thực tế card đã được tạo.
  - create loan → trả về 201.
  - fetchCustomerDetails → ban đầu lỗi do circuit breaker (tải hệ thống cao), gửi lại thì thành công.
- Các thao tác này tạo ra nhiều log trong các microservice.

3. Vấn đề khi không có log aggregation
- Nếu không có hệ thống tổng hợp log, developer phải:
  - Vào từng container (gateway, accounts, loans, cards, Eureka, …).
  - Xem log riêng lẻ cho từng service.
- Điều này rất phức tạp và tốn thời gian.

4. Thiết lập Grafana + Loki + Promtail
- Mục tiêu: tổng hợp log tập trung mà không cần sửa code trong microservice.
- Grafana chạy trên localhost:3000.
- Trong Grafana:
  - Data source Loki được cấu hình sẵn (URL, headers) thông qua docker-compose (entrypoint command của service Grafana).
  - Nhờ đó, khi container Grafana khởi động, data source Loki được tạo tự động; không cần cấu hình thủ công trong UI.

5. Xem log trong Grafana
- Vào tab “Explore”:
  - Chọn label “container” (được Promtail gán cho log).
  - Giá trị label là tên các Docker container (accounts, cards, loans, gateway, Eureka, …).
  - Chọn container tương ứng (ví dụ accounts) → Run query → xem toàn bộ log của service đó.
  - Có thể bật “Live” để xem log realtime (cập nhật mỗi 5 giây).
- Có thể lọc log theo nội dung:
  - Ví dụ: tìm các log chứa “easy bank correlation id”.
  - Chọn “line contains”, nhập chuỗi cần tìm, Run query → chỉ hiển thị log có chứa chuỗi đó.
  - Có nhiều tùy chọn khác: line does not contain, case-sensitive/insensitive, regex, line filters, …
- Grafana cũng hiển thị mật độ log theo thời gian (khi test nhiều API thì số log tăng mạnh ở khoảng thời gian đó).

6. Lưu trữ log (Minio và volume)
- Trong môi trường local:
  - Log được lưu trong Minio.
  - Trong thư mục dự án (profile prod) có thư mục `.data` chứa:
    - `loki-data`, `loki-ruler`, …
  - Trong docker-compose:
    - Mount thư mục `.data/minio` trên host vào thư mục `/data` trong container Minio.
    - Nhờ đó, log lưu trong host được gắn vào container Minio.
- Trong môi trường production thực tế:
  - Có thể dùng storage cloud như AWS S3 hoặc hệ thống lưu trữ khác để lưu trữ log lâu dài, với số lượng lớn và nhiều microservice.

7. Không cần thay đổi code microservice
- Toàn bộ việc thu thập và tổng hợp log được thực hiện:
  - Bằng Grafana, Loki, Promtail, Minio, Docker.
  - Không sửa bất kỳ dòng code nào trong các microservice.
- Đây là giải pháp cho môi trường dev/test/prod (server), không dành cho phát triển local trong IDE:
  - Khi dev local với IntelliJ, log đã có sẵn trong console, có thể debug, đặt breakpoint, nên không cần setup phức tạp này.

8. Kết luận
- Đã triển khai thành công một trụ cột của observability: logging.
- Tất cả log từ nhiều microservice được tập trung về một nơi, giúp developer dễ dàng:
  - Xem log theo service.
  - Tìm kiếm theo nội dung, correlation id, v.v.
- Bài tiếp theo sẽ tập trung vào metrics (số liệu giám sát).
