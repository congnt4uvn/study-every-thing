
Dưới đây là tóm tắt nội dung:

1. Mục tiêu  
   - Đào sâu về OAuth2 và OpenID Connect, đặc biệt là luồng cấp quyền “Client Credentials Grant” (luồng thông tin xác thực của khách hàng) để áp dụng bảo mật trong mạng microservices.

2. Khi nào dùng Client Credentials Grant  
   - Dùng khi **hai API / hai backend / hai microservice** giao tiếp với nhau qua API.  
   - **Không có người dùng cuối, không có UI** tham gia.  
   - Phù hợp cho các cuộc gọi backend-to-backend trong hệ thống microservices.

3. Ba thành phần chính trong luồng  
   - **Authorization Server (Máy chủ xác thực)**:  
     - Xác thực client (ứng dụng) dựa trên clientId và clientSecret.  
     - Cấp access token.  
     - Trong ví dụ, sẽ dùng **Keycloak** làm authorization server.
   - **Resource Server (Máy chủ tài nguyên)**:  
     - Nhận request kèm access token.  
     - Xác thực token với Authorization Server.  
     - Chỉ trả dữ liệu nếu token hợp lệ và có đủ quyền.  
     - Trong kiến trúc microservices, **API Gateway** sẽ đóng vai trò Resource Server.
   - **Client (Ứng dụng khách)**:  
     - Là một backend khác, một API khác, hoặc một microservice khác.  
     - Không phải ứng dụng UI, không có người dùng cuối.

4. Luồng hoạt động tổng quát  
   1. Client gửi yêu cầu đến Authorization Server để xin **access token**, kèm:  
      - clientId  
      - clientSecret  
      - scope (phạm vi quyền, ví dụ: email)  
      - grant_type = client_credentials  
   2. Authorization Server xác thực clientId, clientSecret, kiểm tra scope có hợp lệ với cấu hình đã đăng ký hay không.  
   3. Nếu hợp lệ, Authorization Server cấp **access token** cho client.  
   4. Client dùng access token này để gọi Resource Server (API Gateway), kèm thông tin tài nguyên muốn truy cập.  
   5. Resource Server xác thực token với Authorization Server.  
   6. Nếu token hợp lệ và có đủ quyền, Resource Server trả về tài nguyên được bảo vệ cho client.  
   7. Trong nền, Gateway có thể tiếp tục gọi các microservice khác (ví dụ: account-service) và trả kết quả cuối cùng về cho client.

5. Về scope và cấu hình  
   - Scope giống như “quyền hạn” (authorities).  
   - Client chỉ được yêu cầu những scope đã được admin cấu hình sẵn trong Authorization Server (ví dụ trong Keycloak).  
   - Nếu client yêu cầu scope không được phép, Authorization Server sẽ trả lỗi.

6. Đặc điểm của Client Credentials Grant  
   - Là luồng **đơn giản nhất** trong OAuth2.  
   - Không có resource owner (người dùng) trong luồng.  
   - Chỉ dùng cho **machine-to-machine / service-to-service**.  
   - Không dùng khi có người dùng cuối hoặc UI; các luồng phức tạp hơn sẽ được dùng trong những trường hợp đó.

Nội dung tiếp theo (trong bài giảng sau) sẽ minh họa luồng này bằng ví dụ cụ thể với Easy Bank Microservices để mọi thứ trở nên rõ ràng hơn.
