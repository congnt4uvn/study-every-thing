
Dưới đây là tóm tắt nội dung bài giảng:

1. So sánh Service type LoadBalancer và Ingress  
   - Service type LoadBalancer: mỗi microservice gắn type=LoadBalancer sẽ được expose ra ngoài với một public IP riêng (do cloud provider cấp). Nếu có 5 service thì sẽ có 5 LoadBalancer, 5 IP.  
   - Nhược điểm: mỗi service là một điểm vào riêng, khó quản lý khi số lượng service tăng.

2. Khái niệm Kubernetes Ingress  
   - Ingress cung cấp **một điểm vào duy nhất** cho toàn bộ cụm Kubernetes, nhận mọi HTTP/HTTPS request từ bên ngoài và định tuyến đến các service bên trong.  
   - Hoạt động như một **API Gateway / edge server** cho hệ thống microservices.  
   - Có thể:
     - Định tuyến theo host, path (virtual hosting theo tên miền).  
     - Cân bằng tải.  
     - Terminate SSL/TLS.  
     - Tích hợp xác thực/ủy quyền (OAuth2/OIDC) với Keycloak, Okta, v.v.

3. So sánh Spring Cloud Gateway và Kubernetes Ingress  
   - Cả hai đều làm nhiệm vụ tương tự: gateway/edge server, định tuyến request đến các service.  
   - Khác biệt chủ yếu:
     - **Spring Cloud Gateway**:
       - Là ứng dụng Spring Boot do developer xây dựng, cấu hình, triển khai.  
       - Cho phép viết **logic nghiệp vụ tùy chỉnh** bằng Java (filter, rule phức tạp, logic domain).  
       - Phù hợp khi cần nhiều custom logic mà Ingress không làm được.
     - **Kubernetes Ingress**:
       - Do team DevOps/Kubernetes admin thiết lập và quản lý.  
       - Tập trung vào cấu hình hạ tầng, bảo mật, routing, tích hợp với các sản phẩm khác.  
       - Phù hợp khi tổ chức có đội DevOps mạnh, muốn chuẩn hóa gateway ở tầng Kubernetes.

   - Quyết định dùng cái nào phụ thuộc:
     - Cơ cấu tổ chức: giao trách nhiệm “cửa ngõ ra/vào” cho team dev hay team DevOps.  
     - Nhu cầu nghiệp vụ: nếu cần logic phức tạp → Spring Cloud Gateway; nếu chủ yếu là routing/hạ tầng → Ingress.

4. Cấu hình Ingress (Ingress Resource)  
   - Là một resource trong Kubernetes với `kind: Ingress`.  
   - Phần `metadata.name`: tên resource.  
   - Phần `spec.rules`:
     - Khai báo `host` (tên miền).  
     - Khai báo các `paths`: nếu request vào path X thì forward đến service backend Y trong cluster.  
   - DevOps sẽ viết các file YAML Ingress để định nghĩa các rule này.

5. Ingress Controller  
   - Bản thân Ingress resource chỉ là cấu hình, **không tự hoạt động**.  
   - Cần cài **Ingress Controller** trong cluster để:
     - Theo dõi các Ingress resource.  
     - Cấu hình các thành phần mạng (proxy, load balancer) tương ứng.  
   - Một số Ingress Controller phổ biến:
     - NGINX Ingress Controller (mã nguồn mở, dùng nhiều nhất).  
     - Traefik, HAProxy Ingress, v.v.  
   - Có hơn 30 Ingress Controller khác nhau; tổ chức chọn tùy theo nhu cầu.

6. Luồng lưu lượng khi dùng Ingress  
   - Các microservice (accounts, loans, cards, …) chạy trong pod, expose qua **Service type ClusterIP** (chỉ nội bộ cluster).  
   - Khi thêm Ingress:
     - Client bên ngoài gửi request đến LoadBalancer (do Ingress/Ingress Controller quản lý).  
     - LoadBalancer phân phối đến một trong các instance của Ingress Controller.  
     - Ingress Controller dựa trên rule (host/path) để:
       - Gửi request đến Service tương ứng (accounts, loans, cards…).  
       - Service chuyển tiếp đến pod/container tương ứng.  
   - Ingress Controller vì vậy đóng vai trò **gateway/edge server** cho cluster, tương tự vai trò của Spring Cloud Gateway nhưng ở tầng Kubernetes.

7. Kết luận  
   - Spring Cloud Gateway và Kubernetes Ingress đều là giải pháp gateway, nhưng:
     - Ingress thiên về hạ tầng, DevOps, routing, bảo mật ở tầng cluster.  
     - Spring Cloud Gateway thiên về logic ứng dụng, do developer kiểm soát.  
   - Tùy tổ chức, đội ngũ và yêu cầu nghiệp vụ mà chọn giải pháp phù hợp, hoặc kết hợp cả hai.
