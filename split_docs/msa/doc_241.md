
Dưới đây là tóm tắt nội dung bài giảng:

1. Bối cảnh  
- Trước đó giảng viên dùng một Helm chart có sẵn (public repo) để triển khai WordPress.  
- Trong thực tế/doanh nghiệp, thường phải tự xây dựng Helm chart riêng cho hệ thống microservices.  
- Mục tiêu: tạo một “common Helm chart” dùng chung cho nhiều microservice, giúp:
  - Deploy tất cả microservice bằng một lệnh.
  - Uninstall bằng một lệnh.
  - Dễ quản lý nhiều file manifest Kubernetes cho nhiều dịch vụ.

2. Chuẩn bị thư mục & tạo chart  
- Tạo thư mục Section_16, bên trong tạo thư mục Helm.  
- Mở terminal, cd vào Section_16/Helm.  
- Chạy lệnh:  
  - `helm create eazybank-common`  
- Helm tạo ra một chart mới với cấu trúc mặc định (templates, values.yaml, Chart.yaml, …).

3. Dọn dẹp chart mặc định  
- Xóa tất cả file trong thư mục templates của chart mặc định (vì không dùng nội dung NGINX có sẵn).  
- Mở values.yaml:
  - Xóa toàn bộ nội dung liên quan đến NGINX (vì sẽ dùng giá trị riêng).  
- Mở Chart.yaml:
  - apiVersion: v2 (giữ nguyên).  
  - name, description, type: application (giữ nguyên).  
  - version: 0.1.0 (phiên bản chart, giữ nguyên).  
  - appVersion: đổi thành 1.0.0 (phiên bản ứng dụng do ta quy định).  
- Kiểm tra file Chart.yaml không có dependencies (chart này không phụ thuộc chart khác).

4. Tạo các file template dùng chung  
Giảng viên tạo 3 file template trong thư mục templates của eazybank-common:

- service.yaml  
- deployment.yaml  
- configmap.yaml  

Các file này là template Helm (không phải manifest tĩnh), dùng cho nhiều microservice (accounts, loans, cards, …).

---

### 4.1. Template service.yaml

- Dùng cú pháp template của Helm (Go template), có `{{ }}` và các khối `{{- define }}` / `{{- end }}`.  
- Đặt tên template: `common.service`:
  ```yaml
  {{- define "common.service" -}}
  ...
  {{- end }}
  ```
  - `define` đặt tên cho template để các chart khác có thể “gọi” template này.  
  - Dấu `-` trước/sau giúp trim khoảng trắng.

- Nội dung chính:
  - apiVersion: v1  
  - kind: Service  
  - metadata.name: lấy từ `.Values.serviceName`  
  - spec.selector.app: lấy từ `.Values.appLabel`  
  - spec.type: lấy từ `.Values.service.type` (hoặc tương tự)  
  - spec.ports:
    - port: `.Values.servicePort`  
    - targetPort: `.Values.serviceTargetPort`  
    - protocol: TCP  
- Tất cả giá trị động được lấy từ `.Values` (tức là từ values.yaml của chart sử dụng).

---

### 4.2. Template deployment.yaml

- Đặt tên template: `common.deployment`:
  ```yaml
  {{- define "common.deployment" -}}
  ...
  {{- end }}
  ```

- Nội dung chính:
  - apiVersion: apps/v1  
  - kind: Deployment  
  - metadata:
    - name: giá trị động (từ `.Values`)  
    - labels.app: `.Values.appLabel`  
  - spec:
    - replicas: `.Values.replicas`  
    - selector.matchLabels.app: `.Values.appLabel`  
    - template.metadata.labels.app: `.Values.appLabel`  
    - template.spec.containers:
      - name: `.Values.appLabel`  
      - image: kết hợp `.Values.image.repository` và `.Values.image.tag`  
      - ports:
        - containerPort: `.Values.containerPort`  
        - protocol: TCP  

- Phần env (biến môi trường):
  - Sử dụng nhiều khối `if` để điều khiển việc có inject biến môi trường hay không, ví dụ:
    - `if .Values.appName_enabled`:
      - Thêm env `SPRING_APPLICATION_NAME` với giá trị từ `.Values.appName`.
    - `if .Values.profile_enabled`:
      - Thêm env `SPRING_PROFILES_ACTIVE`, lấy giá trị từ ConfigMap:
        - ConfigMap name: `.Values.global.configMapName`
        - key: `SPRING_PROFILES_ACTIVE`
    - `if .Values.config_enabled`:
      - Thêm env `SPRING_CONFIG_IMPORT`, lấy từ ConfigMap.
    - `if .Values.eureka_enabled`:
      - Thêm env chứa URL Eureka service.
    - `if .Values.resourceserver_enabled`:
      - Thêm env liên quan đến resource server (dùng cho API Gateway).
    - `if .Values.otel_enabled`:
      - Thêm các env liên quan OpenTelemetry:
        - Đường dẫn agent Java (JAVA_TOOL_OPTIONS hoặc tương tự).
        - `OTEL_EXPORTER_OTLP_ENDPOINT`
        - `OTEL_METRICS_EXPORTER`
        - `OTEL_SERVICE_NAME`
    - `if .Values.kafka_enabled`:
      - Thêm các env cấu hình Kafka (bootstrap servers, topic, …) lấy từ ConfigMap.

- Ý tưởng:
  - Mỗi microservice có thể bật/tắt từng nhóm biến môi trường bằng cách set true/false trong values.yaml.
  - Các giá trị cụ thể (URL, profile, …) được lấy từ ConfigMap hoặc từ values.yaml của từng service.

---

### 4.3. Template configmap.yaml

- Đặt tên template: `common.configmap`:
  ```yaml
  {{- define "common.configmap" -}}
  ...
  {{- end }}
  ```

- Nội dung chính:
  - apiVersion: v1  
  - kind: ConfigMap  
  - metadata.name: `.Values.global.configMapName`  
  - data:
    - `SPRING_PROFILES_ACTIVE`: `.Values.global.activeProfile`
    - `SPRING_CONFIG_IMPORT`: `.Values.global.springConfigImport`
    - `eurekaServiceURL`: `.Values.global.eurekaServiceURL`
    - `resourceServerURL` (hoặc tương tự): `.Values.global.resourceServerURL`
    - `JAVA_TOOL_OPTIONS`: `.Values.global.javaToolOptions` (chứa đường dẫn agent OpenTelemetry)

- Quy ước:
  - Các thuộc tính dùng chung cho nhiều microservice được đặt dưới `.Values.global.*`.
  - Nhờ vậy, cùng một ConfigMap có thể dùng cho nhiều service, chỉ cần thay values.yaml theo môi trường (dev, QA, prod).

---

5. values.yaml của eazybank-common  
- Giữ trống (không khai báo gì trong values.yaml của chart eazybank-common).  
- Lý do:
  - Đây là “common chart” chỉ chứa template dùng chung.
  - Các chart microservice cụ thể (accounts, loans, cards, …) sẽ:
    - Khai báo dependency tới eazybank-common.
    - Cung cấp values.yaml riêng của chúng (serviceName, appLabel, image, env flags, global.* …).

---

6. Kết luận bài giảng  
- Đã tạo xong một Helm chart chung: `eazybank-common` với 3 template:
  - Service
  - Deployment
  - ConfigMap  
- Chart này không có values riêng, chỉ cung cấp “khung” template.  
- Bài tiếp theo: tạo các Helm chart cho từng microservice, sử dụng lại các template `common.service`, `common.deployment`, `common.configmap` và truyền values.yaml riêng theo nhu cầu từng dịch vụ.
