
Here’s a concise summary of “Advanced Concurrency: Parallel streams: when to use, pitfalls” with examples.

---

## Key Ideas

### What parallel streams are
- Java 8+ feature: `stream().parallel()` or `parallelStream()` uses the **ForkJoinPool.commonPool** to process stream operations in parallel.
- Goal: speed up CPU‑bound operations on large data sets by using multiple cores.

---

## When parallel streams can help

Use parallel streams when:

1. **CPU‑bound work**
   - Each element requires significant computation.
   - Example: heavy calculations, image processing, complex transformations.

2. **Large data sets**
   - Enough elements to amortize parallel overhead (task splitting, thread scheduling).
   - Often tens of thousands or more, but depends on the cost per element.

3. **Stateless, independent operations**
   - No shared mutable state.
   - Operations are pure functions: same input → same output, no side effects.

4. **Associative, order‑independent reductions**
   - Operations like `sum`, `max`, `min`, `count`, `collect` with associative combiners.
   - Order doesn’t matter or you don’t care about it.

---

## When to avoid parallel streams

1. **I/O‑bound tasks**
   - Network calls, disk access, database queries.
   - Threads will mostly wait; parallelism may not help and can overload external systems.

2. **Small collections or cheap operations**
   - Overhead of parallelism > benefit.
   - Simple mapping over a small list is usually faster sequentially.

3. **Shared mutable state**
   - Writing to shared data structures (e.g., `ArrayList`, `HashMap`) without proper synchronization.
   - Leads to race conditions, corrupted data, or subtle bugs.

4. **Order‑sensitive logic**
   - Code that depends on processing order (e.g., first match, stable ordering) can behave differently or be slower with `parallel()`.

5. **Custom thread management needs**
   - Parallel streams use the **common ForkJoinPool**, which:
     - Is shared across the JVM.
     - Can interfere with other parallel tasks.
   - Harder to control thread count and isolation.

---

## Common pitfalls

1. **Side effects in parallel operations**

```java
List<Integer> numbers = IntStream.range(0, 1_000).boxed().toList();
List<Integer> result = new ArrayList<>();

// BAD: shared mutable state
numbers.parallelStream()
       .forEach(n -> result.add(n)); // race condition, may throw or corrupt

System.out.println(result.size()); // Output: may be < 1000 or inconsistent
```

Use thread‑safe collectors instead:

```java
List<Integer> safeResult = numbers.parallelStream()
                                  .collect(Collectors.toList());

System.out.println(safeResult.size()); // Output: 1000
```

2. **Non‑associative operations in reductions**

```java
// BAD: using non-associative operation in reduce
double result = DoubleStream.of(1.0, 2.0, 3.0)
    .parallel()
    .reduce(0.0, (a, b) -> a - b); // subtraction is not associative

System.out.println(result); // Output: unpredictable across runs
```

Use associative operations (e.g., sum, max) or restructure logic.

3. **Blocking calls inside parallel streams**

```java
List<String> urls = List.of("https://a.com", "https://b.com");

// BAD: blocking I/O in parallel stream
urls.parallelStream()
    .map(url -> fetchContent(url)) // blocking HTTP call
    .forEach(System.out::println);
```

This can:
- Block ForkJoinPool threads.
- Starve other parallel tasks.
- Overload remote services.

Better: use dedicated thread pools or async APIs.

4. **Unexpected interaction with common pool**

If another library uses the common pool heavily, your parallel stream may be slow or starved. Also, your heavy parallel stream can slow down other components.

## Basic examples

### Example 1: CPU‑bound parallel computation

```java
import java.util.*;
import java.util.stream.*;

public class ParallelExample {
    public static void main(String[] args) {
        List<Integer> numbers = IntStream.rangeClosed(1, 10_000_000)
                                         .boxed()
                                         .toList();

        // Sequential sum
        long startSeq = System.currentTimeMillis();
        long seqSum = numbers.stream()
                             .mapToLong(ParallelExample::expensiveOperation)
                             .sum();
        long endSeq = System.currentTimeMillis();

        // Parallel sum
        long startPar = System.currentTimeMillis();
        long parSum = numbers.parallelStream()
                             .mapToLong(ParallelExample::expensiveOperation)
                             .sum();
        long endPar = System.currentTimeMillis();

        System.out.println("Sequential sum: " + seqSum +
                           " time(ms): " + (endSeq - startSeq));
        System.out.println("Parallel sum:   " + parSum +
                           " time(ms): " + (endPar - startPar));
    }

    private static long expensiveOperation(int n) {
        // Simulate CPU work
        long r = 0;
        for (int i = 0; i < 100; i++) {
            r += Math.sqrt(n * i + 1);
        }
        return r;
    }
}
```

**Typical output (example, not exact):**

```
Sequential sum: 3162277660 time(ms): 2500
Parallel sum:   3162277660 time(ms): 900
```

Parallel version is faster because:
- Work is CPU‑bound.
- Large data set.
- No shared mutable state.

### Example 2: Safe parallel grouping

```java
import java.util.*;
import java.util.stream.*;

public class GroupingExample {
    public static void main(String[] args) {
        List<String> words = List.of("apple", "ant", "banana", "boat", "cat", "car");

        Map<Character, List<String>> grouped =
            words.parallelStream()
                 .collect(Collectors.groupingByConcurrent(s -> s.charAt(0)));

        System.out.println(grouped);
    }
}
```

**Possible output (order not guaranteed):**

```text
{a=[apple, ant], b=[banana, boat], c=[cat, car]}
```

`groupingByConcurrent` is designed to work well with parallel streams.

---

## Practical guidelines

- Prefer **sequential streams** by default.
- Consider `parallel()` only when:
  - You have CPU‑heavy operations.
  - Data size is large.
  - Operations are stateless and thread‑safe.
- Avoid:
  - Shared mutable state.
  - Blocking I/O.
  - Order‑dependent logic.
- Measure performance (benchmark) instead of assuming parallel is faster.
