
Here’s a concise summary of the main Java lock types in `java.util.concurrent.locks`, with simple code examples and expected outputs.

---

## 1. `ReentrantLock`

**Key ideas:**
- A mutual exclusion lock similar to `synchronized`, but more flexible.
- “Reentrant”: the same thread can lock it multiple times and must unlock the same number of times.
- Supports:
  - `lock()` / `unlock()`
  - `tryLock()` (non-blocking attempt)
  - `lockInterruptibly()` (can be interrupted while waiting)
  - Fair vs. non-fair locking (constructor option)

**Basic example:**

```java
import java.util.concurrent.locks.ReentrantLock;

public class CounterWithReentrantLock {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock();

    public void increment() {
        lock.lock();              // acquire
        try {
            count++;
        } finally {
            lock.unlock();        // always release in finally
        }
    }

    public int getCount() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        CounterWithReentrantLock counter = new CounterWithReentrantLock();

        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) counter.increment();
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) counter.increment();
        });

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println("Final count = " + counter.getCount());
    }
}
```

**Expected output:**

```
Final count = 2000
```

## 2. `ReadWriteLock` (e.g., `ReentrantReadWriteLock`)

**Key ideas:**
- Separates lock into:
  - `readLock()` for shared read access (multiple readers allowed)
  - `writeLock()` for exclusive write access (only one writer, no readers)
- Good when reads are frequent and writes are rare.

**Basic example:**

```java
import java.util.concurrent.locks.ReadWriteLock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ReadWriteCache {
    private int value = 0;
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();

    public int read() {
        rwLock.readLock().lock();
        try {
            return value;
        } finally {
            rwLock.readLock().unlock();
        }
    }

    public void write(int newValue) {
        rwLock.writeLock().lock();
        try {
            value = newValue;
        } finally {
            rwLock.writeLock().unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        ReadWriteCache cache = new ReadWriteCache();
        cache.write(10);

        Thread reader1 = new Thread(() -> System.out.println("Reader1: " + cache.read()));
        Thread reader2 = new Thread(() -> System.out.println("Reader2: " + cache.read()));
        Thread writer = new Thread(() -> cache.write(20));

        reader1.start();
        reader2.start();
        writer.start();

        reader1.join();
        reader2.join();
        writer.join();

        System.out.println("Final value: " + cache.read());
    }
}
```

**Possible output (order may vary):**

```
Reader1: 10
Reader2: 10
Final value: 20
```

## 3. `StampedLock`

**Key ideas:**
- More modern lock for high-performance read-mostly scenarios.
- Provides:
  - **Write lock**: exclusive, like `writeLock()`.
  - **Read lock**: shared, like `readLock()`.
  - **Optimistic read**: non-blocking read that may be invalidated by writers.
- Returns a **stamp** (long) when locking; you must pass this stamp to unlock.
- `validate(stamp)` checks if an optimistic read is still valid.

**Basic example with optimistic read:**

```java
import java.util.concurrent.locks.StampedLock;

public class Point {
    private double x, y;
    private final StampedLock lock = new StampedLock();

    // Write: move the point
    public void move(double deltaX, double deltaY) {
        long stamp = lock.writeLock();
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            lock.unlockWrite(stamp);
        }
    }

    // Read: distance from origin using optimistic read
    public double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead();
        double currentX = x;
        double currentY = y;
        // validate: if a write occurred, stamp becomes invalid
        if (!lock.validate(stamp)) {
            // fall back to read lock
            stamp = lock.readLock();
            try {
                currentX = x;
                currentY = y;
            } finally {
                lock.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }

    public static void main(String[] args) {
        Point p = new Point();
        p.move(3, 4);
        System.out.println("Distance = " + p.distanceFromOrigin());
    }
}
```

**Expected output:**

```
Distance = 5.0
```

## Summary of When to Use What

- **`ReentrantLock`**  
  Use when you need:
  - More control than `synchronized` (tryLock, interruptible lock, fairness).
  - Reentrant mutual exclusion.

- **`ReadWriteLock` (`ReentrantReadWriteLock`)**  
  Use when:
  - Many threads read, few write.
  - You want multiple concurrent readers but only one writer.

- **`StampedLock`**  
  Use when:
  - Read-mostly workloads with high contention.
  - You can benefit from optimistic reads and are comfortable with more complex API.
