
Here’s a concise summary of **rate limiting** and **throttling** in system design, with Java-style examples.

---

## 1. Core Ideas

### Rate Limiting
- **What**: Restricts how many requests/actions are allowed in a given time window.
- **Why**:
  - Protect services from overload or abuse (e.g., API misuse, brute force).
  - Enforce fair usage and quotas (e.g., “1000 requests per hour per user”).
- **Where**:
  - At API gateways, load balancers, or inside services.
  - Often per user, per IP, per API key, or per service.

### Throttling
- **What**: Controls the **speed** of processing requests, often by delaying or queuing them.
- **Why**:
  - Smooth out traffic spikes.
  - Prevent downstream services (DB, external APIs) from being overwhelmed.
- **How**:
  - Queue requests and process at a controlled rate.
  - Sometimes reject or degrade service when limits are exceeded.

**Difference (informal)**:
- Rate limiting: “You can only make N requests per time window.”
- Throttling: “I’ll slow you down or queue you so I don’t exceed my safe processing rate.”

## 2. Common Algorithms

### a) Fixed Window Counter (Rate Limiting)
- Count requests in a fixed time window (e.g., per minute).
- If count > limit → reject.

**Pros**: Simple.  
**Cons**: Burst at window boundaries.

**Java-like example:**

```java
import java.time.Instant;
import java.util.concurrent.atomic.AtomicInteger;

public class FixedWindowRateLimiter {
    private final int limit;              // max requests per window
    private final long windowSizeMillis;  // e.g., 60_000 for 1 minute

    private AtomicInteger counter = new AtomicInteger(0);
    private volatile long windowStart = Instant.now().toEpochMilli();

    public FixedWindowRateLimiter(int limit, long windowSizeMillis) {
        this.limit = limit;
        this.windowSizeMillis = windowSizeMillis;
    }

    public synchronized boolean allowRequest() {
        long now = Instant.now().toEpochMilli();
        if (now - windowStart >= windowSizeMillis) {
            // reset window
            windowStart = now;
            counter.set(0);
        }

        if (counter.get() < limit) {
            counter.incrementAndGet();
            return true; // allowed
        } else {
            return false; // rate limit exceeded
        }
    }
}
```

**Example usage (input/output):**

```java
public static void main(String[] args) {
    FixedWindowRateLimiter limiter = new FixedWindowRateLimiter(3, 10_000); // 3 req / 10s

    for (int i = 1; i <= 5; i++) {
        boolean allowed = limiter.allowRequest();
        System.out.println("Request " + i + " allowed? " + allowed);
    }
}
```

**Possible output:**
```
Request 1 allowed? true
Request 2 allowed? true
Request 3 allowed? true
Request 4 allowed? false
Request 5 allowed? false
```

### b) Token Bucket (Rate Limiting + Bursting)
- Bucket holds tokens; each request consumes 1 token.
- Tokens are added at a fixed rate up to a max capacity.
- If no tokens → reject or delay.

**Pros**: Allows short bursts while enforcing average rate.

**Java-like example:**

```java
public class TokenBucket {
    private final long capacity;
    private final long refillTokensPerSecond;

    private double tokens;
    private long lastRefillTimestamp;

    public TokenBucket(long capacity, long refillTokensPerSecond) {
        this.capacity = capacity;
        this.refillTokensPerSecond = refillTokensPerSecond;
        this.tokens = capacity;
        this.lastRefillTimestamp = System.nanoTime();
    }

    public synchronized boolean allowRequest() {
        refill();

        if (tokens >= 1) {
            tokens -= 1;
            return true;
        }
        return false;
    }

    private void refill() {
        long now = System.nanoTime();
        double secondsSinceLast = (now - lastRefillTimestamp) / 1_000_000_000.0;
        double tokensToAdd = secondsSinceLast * refillTokensPerSecond;

        if (tokensToAdd > 0) {
            tokens = Math.min(capacity, tokens + tokensToAdd);
            lastRefillTimestamp = now;
        }
    }
}
```

**Example usage:**

```java
public static void main(String[] args) throws InterruptedException {
    TokenBucket bucket = new TokenBucket(5, 1); // capacity 5, refill 1 token/sec

    // Initial burst
    for (int i = 1; i <= 7; i++) {
        System.out.println("Request " + i + " allowed? " + bucket.allowRequest());
    }

    // Wait 3 seconds to refill
    Thread.sleep(3000);

    for (int i = 8; i <= 10; i++) {
        System.out.println("Request " + i + " allowed? " + bucket.allowRequest());
    }
}
```

**Possible output:**
```

Request 2 allowed? true
Request 3 allowed? true
Request 4 allowed? true
Request 5 allowed? true
Request 6 allowed? false
Request 7 allowed? false
Request 8 allowed? true
Request 9 allowed? true
Request 10 allowed? true
```

### c) Leaky Bucket (Throttling)
- Requests enter a queue (bucket).
- They “leak” out at a constant rate (processed at fixed speed).
- If queue is full → reject or drop.

**Pros**: Smooth, constant output rate.  
**Cons**: Adds latency; may drop when queue is full.

**Java-like example (simple throttler):**

```java
import java.util.concurrent.*;

public class LeakyBucketThrottler {
    private final BlockingQueue<Runnable> queue;
    private final ScheduledExecutorService worker = Executors.newSingleThreadScheduledExecutor();

    public LeakyBucketThrottler(int capacity, int ratePerSecond) {
        this.queue = new ArrayBlockingQueue<>(capacity);

        long periodMillis = 1000L / ratePerSecond;
        worker.scheduleAtFixedRate(() -> {
            Runnable task = queue.poll();
            if (task != null) {
                task.run();
            }
        }, 0, periodMillis, TimeUnit.MILLISECONDS);
    }

    public boolean submit(Runnable task) {
        // returns false if queue is full (throttled/rejected)
        return queue.offer(task);
    }

    public void shutdown() {
        worker.shutdown();
    }
}
```

**Example usage:**

```java
public static void main(String[] args) throws InterruptedException {
    LeakyBucketThrottler throttler = new LeakyBucketThrottler(5, 2); // capacity 5, 2 tasks/sec

    for (int i = 1; i <= 8; i++) {
        int id = i;
        boolean accepted = throttler.submit(() ->
            System.out.println("Processing task " + id + " at " + System.currentTimeMillis())
        );
        System.out.println("Task " + id + " accepted? " + accepted);
    }

    Thread.sleep(5000);
    throttler.shutdown();
}
```

## 3. Design Considerations

- **Scope**: Per user, per IP, per API key, per endpoint, or global.
- **Storage**:
  - In-memory (fast, but per-instance).
  - Distributed (Redis, etc.) for multi-instance services.
- **Behavior on limit**:
  - Return HTTP 429 (Too Many Requests).
  - Queue/delay (throttling).
  - Degrade gracefully (simpler response, cached data).
- **Client feedback**:
  - Include headers like `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `Retry-After`.
