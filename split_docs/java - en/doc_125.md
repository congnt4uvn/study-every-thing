
**Summary (English)**  
This content is about **Git**, a version control system commonly used in Java (and other) projects. It highlights four key concepts:

1. **Branching** – Create separate lines of development so you can work on features or fixes without affecting the main code.
2. **Merging** – Combine changes from one branch into another (e.g., from a feature branch into `main`).
3. **Rebasing** – Rewrite the base of your branch to create a cleaner, linear history (e.g., updating your feature branch on top of the latest `main`).
4. **Pull Requests (PRs)** – A workflow (on platforms like GitHub/GitLab/Bitbucket) where you propose changes from one branch to be reviewed and merged into another.

These tools are part of the **Java development ecosystem**, helping teams collaborate, review code, and manage changes safely.

---

## Basic Git Examples (with simple “input/output”)

### 1. Branching

**Command (input):**
```bash
# create a new branch for a feature
git checkout -b feature/login-page
```

**Effect (output conceptually):**
- A new branch `feature/login-page` is created.
- Your working directory is now on that branch.

You can check:
```bash
git branch
```
**Example output:**
```text
  main
* feature/login-page
```

---

### 2. Merging

Assume you finished work on `feature/login-page` and want to merge it into `main`.

**Commands (input):**
```bash
# switch to main
git checkout main

# merge the feature branch into main
git merge feature/login-page
```

**Possible output:**
```text
Updating a1b2c3d..d4e5f6g
Fast-forward
 src/Login.java | 10 ++++++++++
 1 file changed, 10 insertions(+)
```

Now `main` includes the changes from `feature/login-page`.

---

### 3. Rebasing

You want to update your feature branch with the latest changes from `main` but keep a clean history.

**Commands (input):**
```bash
# switch to your feature branch
git checkout feature/login-page

# rebase onto main
git rebase main
```

**Possible output:**
```text
First, rewinding head to replay your work on top of it...
Applying: Add login form
Applying: Validate user input
```

Your feature branch now appears as if it was developed on top of the latest `main`.

---

### 4. Pull Requests (conceptual example)

On GitHub (or similar):

1. Push your branch:
   ```bash
   git push origin feature/login-page
   ```
2. On the website, open a **Pull Request**:
   - **From**: `feature/login-page`
   - **Into**: `main`
3. Teammates review the code, comment, and approve.
4. After approval, the PR is merged (often using a merge or rebase strategy).

**Result (output conceptually):**
- `main` now contains your reviewed and approved changes.
- The PR is recorded with comments and history.

### Small Java Example in a Git Workflow

**Step 1 – Initial file on `main`:**

`Main.java`:
```java
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello, world!");
    }
}
```

**Step 2 – Create a feature branch and modify:**
```bash
git checkout -b feature/greeting
```

Edit `Main.java`:
```java
public class Main {
    public static void main(String[] args) {
        System.out.println("Hello, Java developer!");
    }
}
```

**Step 3 – Commit and push:**
```bash
git add Main.java
git commit -m "Change greeting message"
git push origin feature/greeting
```

**Step 4 – Open a Pull Request** to merge `feature/greeting` into `main`.  
After merge, running:

```bash
javac Main.java
java Main
```

**Program output:**
```text
Hello, Java developer!
```

This shows how Git branching, merging, and pull requests fit into a typical Java development workflow.
