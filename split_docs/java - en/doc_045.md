
Here’s a concise summary of the key ideas around JVM basics: class loading, classpath, and modules (Java 9+), with small code examples.

---

## 1. Class Loading

**Idea:**  
The JVM doesn’t load all classes at startup. It loads them **on demand** using **class loaders**.

**Main class loaders:**
- **Bootstrap ClassLoader**: Loads core Java classes (`java.lang.*`, `java.util.*`, etc.).
- **Extension / Platform ClassLoader**: Loads JDK extension or platform classes.
- **Application ClassLoader**: Loads classes from the application’s classpath.

**Key points:**
- Classes are loaded when first referenced.
- Each class is identified by its **fully qualified name + class loader**.
- Custom class loaders can load classes from non-standard sources (network, encrypted files, etc.).

**Example: Inspecting class loaders**

```java
public class ClassLoaderDemo {
    public static void main(String[] args) {
        ClassLoader appClassLoader = ClassLoaderDemo.class.getClassLoader();
        System.out.println("Application ClassLoader: " + appClassLoader);

        ClassLoader parent = appClassLoader.getParent();
        System.out.println("Parent (Platform) ClassLoader: " + parent);

        ClassLoader bootstrap = parent.getParent();
        System.out.println("Bootstrap ClassLoader: " + bootstrap); // usually null
    }
}
```

**Possible output (simplified):**
```
Application ClassLoader: jdk.internal.loader.ClassLoaders$AppClassLoader@...
Parent (Platform) ClassLoader: jdk.internal.loader.ClassLoaders$PlatformClassLoader@...
Bootstrap ClassLoader: null
```

## 2. Classpath

**Idea:**  
The **classpath** tells the JVM **where to find classes and resources** for your application.

**Key points:**
- It’s a list of directories, JAR files, or wildcard patterns.
- You can set it via:
  - Command line: `java -cp ...` or `java -classpath ...`
  - Environment variable: `CLASSPATH` (less recommended; can be confusing).
- The **Application ClassLoader** uses the classpath to locate classes.

**Example: Running a class with classpath**

Assume:
- `com.example.Main` is in `out/production/myapp`
- You use an external library `lib/mylib.jar`

Command:

```bash
java -cp out/production/myapp:lib/mylib.jar com.example.Main
```

(Use `;` instead of `:` on Windows.)

**Example Java code using a class from the classpath**

```java
// File: com/example/Main.java
package com.example;

import org.example.lib.Util; // from lib/mylib.jar

public class Main {
    public static void main(String[] args) {
        String result = Util.upper("hello");
        System.out.println(result);
    }
}
```

**Expected output:**
```
HELLO
```

## 3. Modules (Java 9+)

**Idea:**  
The **Java Platform Module System (JPMS)** organizes code into **modules** with explicit dependencies and exports. It’s a more structured alternative to a flat classpath.

**Key points:**
- A module is defined by a `module-info.java` file.
- You declare:
  - Which packages are exported (`exports`)
  - Which modules you depend on (`requires`)
- Modules are loaded via the **module path** (`--module-path`), not just the classpath.
- Helps with:
  - Strong encapsulation
  - Smaller runtimes (jlink)
  - Clear dependency graphs

**Example: Simple module**

Project structure:
```
src
 └─ com.example.app
     ├─ module-info.java
     └─ com/example/app/Main.java
```

`module-info.java`:

```java
module com.example.app {
    requires java.sql;      // depends on java.sql module
    exports com.example.app; // exports this package
}
```

`Main.java`:

```java
package com.example.app;

public class Main {
    public static void main(String[] args) {
        System.out.println("Hello from a module!");
    }
}
```

**Compile and run with modules:**

```bash
# Compile
javac -d out --module-source-path src $(find src -name "*.java")

# Run
java --module-path out --module com.example.app/com.example.app.Main
```

**Output:**
```
Hello from a module!
```

---

## Classpath vs Module Path (Java 9+)

- **Classpath**:  
  - Flat, no explicit dependencies or encapsulation.
  - All public classes are visible to all others.

- **Module path**:  
  - Structured, with `module-info.java`.
  - Only exported packages are visible.
  - Dependencies are declared (`requires`).

You can still use the classpath in Java 9+, but for modular applications, you use the module path and module descriptors.
