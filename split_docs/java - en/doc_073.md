
Here’s a concise summary of the key ideas around Spring Data & Persistence transactions: `@Transactional`, propagation, and isolation levels, with code examples and sample I/O behavior.

---

## 1. `@Transactional` – What It Does

`@Transactional` tells Spring that a method (or class) should run inside a database transaction.

Main effects:
- Starts a transaction before the method.
- Commits if the method finishes normally.
- Rolls back on runtime exceptions (by default).
- Can be placed on:
  - Methods in service layer (recommended)
  - Classes (applies to all public methods)

### Example: Basic `@Transactional`

```java
@Service
public class AccountService {

    @Autowired
    private AccountRepository accountRepository;

    @Transactional
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        Account from = accountRepository.findById(fromId).orElseThrow();
        Account to = accountRepository.findById(toId).orElseThrow();

        from.withdraw(amount);
        to.deposit(amount);

        accountRepository.save(from);
        accountRepository.save(to);
        // If any RuntimeException occurs here, the whole transfer is rolled back
    }
}
```

**Input (method call):**
```java
accountService.transfer(1L, 2L, new BigDecimal("100.00"));
```

**Output (DB state):**
- If no error:  
  - Account 1 balance: decreased by 100  
  - Account 2 balance: increased by 100
- If an exception occurs in the middle:  
  - Both balances remain unchanged (rollback).

## 2. Propagation – How Transactions Interact

Propagation defines what happens if a transactional method is called from another transactional method.

Common propagation types:

- `REQUIRED` (default):  
  - Use existing transaction if present; otherwise start a new one.
- `REQUIRES_NEW`:  
  - Always start a new transaction, suspending any existing one.
- `MANDATORY`:  
  - Must run inside an existing transaction; otherwise throw exception.
- `SUPPORTS`:  
  - Use transaction if present; otherwise run non-transactionally.
- `NOT_SUPPORTED`:  
  - Always run non-transactionally; suspend any existing transaction.
- `NEVER`:  
  - Must not run in a transaction; if there is one, throw exception.
- `NESTED`:  
  - Nested transaction using savepoints (depends on underlying DB/driver).

### Example: `REQUIRED` vs `REQUIRES_NEW`

```java
@Service
public class OrderService {

    @Autowired
    private PaymentService paymentService;

    @Autowired
    private OrderRepository orderRepository;

    @Transactional // propagation = REQUIRED (default)
    public void placeOrder(Order order) {
        orderRepository.save(order); // part of outer transaction
        paymentService.charge(order); // join same transaction by default
    }
}

@Service
public class PaymentService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void charge(Order order) {
        // This runs in a separate transaction
        // If this fails, only this transaction is rolled back
        // Outer transaction can still commit if you handle the exception
    }
}
```

**Behavior example:**

- `placeOrder()` starts Transaction A.
- `charge()` starts Transaction B (new), suspending A.
- If `charge()` fails and throws an exception:
  - Transaction B is rolled back.
  - If `placeOrder()` catches the exception and continues, Transaction A can still commit (order saved but payment failed).

## 3. Isolation Levels – How Concurrent Transactions See Data

Isolation level controls how one transaction sees changes made by others.

Common isolation levels:

- `READ_UNCOMMITTED`  
  - Can see uncommitted changes from other transactions (dirty reads).
- `READ_COMMITTED` (common default)  
  - Only sees committed data; prevents dirty reads.
- `REPEATABLE_READ`  
  - Same row read multiple times returns the same data within a transaction; prevents non-repeatable reads.
- `SERIALIZABLE`  
  - Highest isolation; transactions behave as if executed one by one; prevents phantom reads but reduces concurrency.

### Example: Setting Isolation Level

```java
@Service
public class ProductService {

    @Autowired
    private ProductRepository productRepository;

    @Transactional(isolation = Isolation.READ_COMMITTED)
    public Product getProduct(Long id) {
        return productRepository.findById(id).orElseThrow();
    }

    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void updateStock(Long id, int delta) {
        Product p = productRepository.findById(id).orElseThrow();
        p.setStock(p.getStock() + delta);
        productRepository.save(p);
    }
}
```

**Concurrent behavior example (simplified):**

- With `READ_COMMITTED`:
  - Transaction T1 reads product stock = 10.
  - T2 updates stock to 5 and commits.
  - T1 reads again: sees 5 (non-repeatable read allowed).
- With `REPEATABLE_READ`:
  - T1 reads stock = 10.
  - T2 updates stock to 5 and commits.
  - T1 reads again: still sees 10 until T1 commits.

## 4. Typical Configuration Example

```java
@Service
public class UserService {

    @Autowired
    private UserRepository userRepository;

    @Transactional(
        propagation = Propagation.REQUIRED,
        isolation = Isolation.READ_COMMITTED,
        rollbackFor = Exception.class // also rollback on checked exceptions
    )
    public User registerUser(User user) throws Exception {
        // validate, save, send email, etc.
        return userRepository.save(user);
    }
}
```

**Input:**
```java
User newUser = new User("alice", "alice@example.com");
userService.registerUser(newUser);
```

**Output:**
- On success: user row inserted and committed.
- On any `Exception`: insert is rolled back; no user row in DB.

---

**In summary:**
- `@Transactional` controls transaction boundaries and rollback behavior.
- Propagation defines how nested/related transactional calls interact.
- Isolation levels define how concurrent transactions see each other’s data.
