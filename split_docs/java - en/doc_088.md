
Here’s a concise summary of the main NoSQL types, with Java-oriented examples.

---

## 1. Key-Value Stores

**Idea:**  
Data is stored as simple key → value pairs.  
You look up values by key only (like a big `Map<String, Object>`).

**Use cases:**  
- Caching (sessions, tokens)  
- Simple user preferences  
- Fast lookups with known keys

**Examples:** Redis, Amazon DynamoDB (key-value usage), Riak

**Java-style example (pseudo-code using a Redis client):**

```java
// SET key
jedis.set("user:1001:name", "Alice");

// GET key
String name = jedis.get("user:1001:name");
System.out.println(name);
```

**Output:**
```text
Alice
```

---

## 2. Document Stores

**Idea:**  
Data is stored as documents (usually JSON/BSON).  
Each document can have flexible structure (schema-less or schema-light).  
You can query by fields inside the document.

**Use cases:**  
- User profiles  
- Product catalogs  
- Content management (articles, posts)

**Examples:** MongoDB, CouchDB, Couchbase

**Java + MongoDB example:**

```java
// Insert a document
Document user = new Document("userId", 1001)
        .append("name", "Alice")
        .append("email", "alice@example.com")
        .append("tags", Arrays.asList("premium", "beta"));

collection.insertOne(user);

// Query by field
Document found = collection.find(eq("userId", 1001)).first();
System.out.println(found.toJson());
```

**Possible output:**
```json
{
  "_id": { "$oid": "665f..." },
  "userId": 1001,
  "name": "Alice",
  "email": "alice@example.com",
  "tags": ["premium", "beta"]
}
```

---

## 3. Column-Family Stores (Wide-Column)

**Idea:**  
Data is organized into **column families** (like tables), but each row can have a different set of columns.  
Optimized for large-scale, distributed, write-heavy workloads.

**Use cases:**  
- Time-series data  
- Logging and metrics  
- Large-scale analytics

**Examples:** Apache Cassandra, HBase

**Conceptual model (Cassandra):**

- Keyspace: like a database  
- Table: column family  
- Row key: primary key  
- Columns: flexible, can vary per row

**Java + Cassandra example (using DataStax driver):**

```java
// Create table (CQL, not Java)
String createTable = """
CREATE TABLE IF NOT EXISTS users (
    user_id int PRIMARY KEY,
    name text,
    email text
);
""";
session.execute(createTable);

// Insert
session.execute("INSERT INTO users (user_id, name, email) VALUES (1001, 'Alice', 'alice@example.com')");

// Select
ResultSet rs = session.execute("SELECT user_id, name, email FROM users WHERE user_id = 1001");
Row row = rs.one();
System.out.println(row.getInt("user_id") + " " + row.getString("name") + " " + row.getString("email"));
```

**Output:**
```text
1001 Alice alice@example.com
```

## 4. Graph Databases

**Idea:**  
Data is stored as **nodes** and **edges** (relationships).  
You query based on relationships and paths (graph traversal).

**Use cases:**  
- Social networks (friends, followers)  
- Recommendation engines  
- Network/IT topology, fraud detection

**Examples:** Neo4j, JanusGraph, Amazon Neptune

**Conceptual model:**

- Node: entity (User, Product, City)  
- Edge: relationship (FRIEND_OF, PURCHASED, LIVES_IN)  
- Properties: key-value attributes on nodes/edges

**Java + Neo4j (using Cypher):**

```java
// Create nodes and relationship
session.run("CREATE (a:User {id: 1001, name: 'Alice'})");
session.run("CREATE (b:User {id: 1002, name: 'Bob'})");
session.run("""
MATCH (a:User {id: 1001}), (b:User {id: 1002})
CREATE (a)-[:FRIEND_OF]->(b)
""");

// Query friends of Alice
Result result = session.run("""
MATCH (a:User {id: 1001})-[:FRIEND_OF]->(friend)
RETURN friend.name AS name
""");

while (result.hasNext()) {
    Record record = result.next();
    System.out.println(record.get("name").asString());
}
```

**Output:**
```text
Bob
```

## Quick Comparison

- **Key-Value:**  
  - Model: key → value  
  - Strength: very fast, simple  
  - Weakness: limited querying (by key only)

- **Document:**  
  - Model: JSON-like documents  
  - Strength: flexible schema, rich queries  
  - Weakness: complex joins are harder

- **Column-Family:**  
  - Model: rows with flexible columns in families  
  - Strength: huge scale, high write throughput  
  - Weakness: query patterns must be designed up front

- **Graph:**  
  - Model: nodes + edges + properties  
  - Strength: relationship-heavy queries, traversals  
  - Weakness: not ideal for simple key-based lookups at massive scale
