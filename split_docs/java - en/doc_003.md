
Here’s a concise summary of the key ideas about Java generics wildcards and type erasure, with code examples and sample inputs/outputs.

---

## 1. Wildcards in Generics

Wildcards let you write more flexible methods that work with different generic types.

### 1.1 `? extends T` – Upper-Bounded Wildcard

Meaning: “Some unknown subtype of T”

Use when you **only need to read** from a structure, not add arbitrary T values.

```java
import java.util.*;

public class UpperBoundExample {
    // Can read Numbers, but cannot safely add Numbers
    public static double sumList(List<? extends Number> list) {
        double sum = 0.0;
        for (Number n : list) {
            sum += n.doubleValue();
        }
        return sum;
    }

    public static void main(String[] args) {
        List<Integer> ints = Arrays.asList(1, 2, 3);
        List<Double> doubles = Arrays.asList(1.5, 2.5);

        System.out.println(sumList(ints));    // Output: 6.0
        System.out.println(sumList(doubles)); // Output: 4.0
    }
}
```

Key points:
- You can pass `List<Integer>`, `List<Double>`, etc. to `sumList`.
- You **cannot** do `list.add(123)` inside `sumList` (except `null`), because the exact subtype is unknown.

---

### 1.2 `? super T` – Lower-Bounded Wildcard

Meaning: “Some unknown supertype of T”

Use when you **want to add** T values into a structure, and reading is usually as `Object`.

```java
import java.util.*;

public class LowerBoundExample {
    // Can safely add Integers
    public static void addNumbers(List<? super Integer> list) {
        list.add(1);
        list.add(2);
        list.add(3);
    }

    public static void main(String[] args) {
        List<Integer> ints = new ArrayList<>();
        List<Number> nums = new ArrayList<>();
        List<Object> objs = new ArrayList<>();

        addNumbers(ints);
        addNumbers(nums);
        addNumbers(objs);

        System.out.println(ints); // Output: [1, 2, 3]
        System.out.println(nums); // Output: [1, 2, 3]
        System.out.println(objs); // Output: [1, 2, 3]
    }
}
```

Key points:
- You can pass `List<Integer>`, `List<Number>`, `List<Object>`.
- You can safely add `Integer` values.
- When reading from `list`, you usually treat elements as `Object`.

---

### 1.3 PECS Rule

**PECS**: *Producer Extends, Consumer Super*

- If a parameter **produces** values for you to read → use `? extends T`
- If a parameter **consumes** values you put in → use `? super T`

## 2. Type Erasure

Java generics are implemented with **type erasure**: generic type information is removed at compile time.

### 2.1 What It Means

- At runtime, `List<String>` and `List<Integer>` are both just `List`.
- The compiler uses generic types to check correctness, then erases them.

Example:

```java
public class ErasureExample {
    public static void main(String[] args) {
        List<String> strings = new ArrayList<>();
        List<Integer> ints = new ArrayList<>();

        System.out.println(strings.getClass() == ints.getClass());
        // Output: true
    }
}
```

---

### 2.2 Consequences of Type Erasure

1. **No generic arrays**

```java
// Not allowed:
List<String>[] arr = new ArrayList<String>[10]; // Compile-time error
```

2. **No runtime checks with generic types**

```java
List<String> list = new ArrayList<>();
// if (list instanceof List<String>) { } // Compile-time error

if (list instanceof List) { // Allowed
    System.out.println("It's a List");
}
```

3. **Type casts are inserted by the compiler**

```java
List<String> list = new ArrayList<>();
list.add("hello");

// After erasure, roughly like:
List listRaw = new ArrayList();
listRaw.add("hello");
String s = (String) listRaw.get(0); // Cast inserted by compiler
```

4. **Type parameters cannot be used in static contexts**

```java
class Box<T> {
    // static T value; // Not allowed
}
```
