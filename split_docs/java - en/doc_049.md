
**Summary: Java Memory Model (JMM)**

The Java Memory Model defines how threads interact through memory. Its main concerns are:

1. **Visibility**  
   - When one thread writes to a variable, other threads may not immediately see the new value unless certain rules are followed.  
   - Tools for visibility:
     - `volatile` variables
     - Locks (`synchronized`)
     - Atomic classes (`AtomicInteger`, etc.)

2. **Reordering**  
   - The compiler, JVM, and CPU can reorder instructions to optimize performance.  
   - The JMM defines which reorderings are allowed so that correctly synchronized programs behave as if operations happen in a predictable order.  
   - Without proper synchronization, you can see surprising results due to reordering.

3. **Happens-Before Relationship**  
   - A *happens-before* relationship guarantees **visibility and ordering**:
     - If A happens-before B, then:
       - All effects of A are visible to B.
       - A is ordered before B.
   - Key happens-before rules:
     - **Program order**: In a single thread, earlier statements happen-before later ones.
     - **Monitor lock**: An unlock on a monitor (`synchronized` block exit) happens-before every subsequent lock on the same monitor.
     - **Volatile**: A write to a `volatile` variable happens-before every subsequent read of that same variable.
     - **Thread start/join**:
       - Actions in a thread happen-before another thread successfully returns from `Thread.join()` on that thread.
       - A call to `Thread.start()` happens-before any actions in the started thread.

### Example 1: Visibility Problem Without Synchronization

```java
public class VisibilityExample {
    private static boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (running) {
                // busy-wait
            }
            System.out.println("Stopped");
        });

        t.start();

        Thread.sleep(1000);
        running = false; // may NOT be visible to thread t
        System.out.println("Main set running = false");
    }
}
```

**Possible output:**
- The program may never print `"Stopped"` and may run forever, because the worker thread might keep seeing `running == true` due to lack of visibility guarantees.

**Fix with `volatile`:**

```java
public class VisibilityExampleFixed {
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (running) {
                // busy-wait
            }
            System.out.println("Stopped");
        });

        t.start();

        Thread.sleep(1000);
        running = false; // happens-before the read in thread t
        System.out.println("Main set running = false");
    }
}
```

**Typical output:**
```
Main set running = false
Stopped
```

The `volatile` write to `running` happens-before the read in the other thread, ensuring visibility.

### Example 2: Reordering and Data Race

```java
public class ReorderingExample {
    static int x = 0, y = 0;
    static int a = 0, b = 0;

    public static void main(String[] args) throws InterruptedException {
        for (int i = 0; i < 1_000_000; i++) {
            x = y = a = b = 0;

            Thread t1 = new Thread(() -> {
                a = 1;          // (1)
                x = b;          // (2)
            });

            Thread t2 = new Thread(() -> {
                b = 1;          // (3)
                y = a;          // (4)
            });

            t1.start();
            t2.start();
            t1.join();
            t2.join();

            if (x == 0 && y == 0) {
                System.out.println("Observed x=0, y=0");
                break;
            }
        }
    }
}
```

**Possible output:**
```
Observed x=0, y=0
```

Even though it looks like at least one of `x` or `y` should be `1`, due to **reordering and lack of happens-before**, both threads can see the old values (`a = 0`, `b = 0`).

**Fix with synchronization (establish happens-before):**

```java
public class ReorderingExampleFixed {
    static int x = 0, y = 0;
    static int a = 0, b = 0;
    static final Object lock = new Object();

    public static void main(String[] args) throws InterruptedException {
        for (int i = 0; i < 1_000_000; i++) {
            x = y = a = b = 0;

            Thread t1 = new Thread(() -> {
                synchronized (lock) {
                    a = 1;
                    x = b;
                }
            });

            Thread t2 = new Thread(() -> {
                synchronized (lock) {
                    b = 1;
                    y = a;
                }
            });

            t1.start();
            t2.start();
            t1.join();
            t2.join();

            if (x == 0 && y == 0) {
                System.out.println("Observed x=0, y=0"); // should never happen now
                break;
            }
        }
    }
}
```

With proper synchronization, the JMM prevents harmful reordering and ensures consistent visibility.

---

**Key Takeaways**

- The JMM defines how and when changes made by one thread become visible to others.
- **Happens-before** is the core concept: it guarantees both ordering and visibility.
- Use `volatile`, `synchronized`, and higher-level concurrency utilities to establish happens-before relationships and avoid bugs from visibility and reordering.
