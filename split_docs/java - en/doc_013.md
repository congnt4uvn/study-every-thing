
Here’s a concise summary of sealed classes in Java (Java 17+), with examples.

---

## What are sealed classes?

- **Sealed classes** (and interfaces) restrict which other classes or interfaces can extend or implement them.
- They provide **controlled inheritance**, improving:
  - API design (you know all possible subclasses)
  - Exhaustive handling in `switch` expressions
  - Security and maintainability

---

## Basic syntax

A sealed class:

```java
public sealed class Shape
    permits Circle, Rectangle, Square {
    // common members
}
```

Each permitted subclass must be one of:
- `final` (no further subclassing), or
- `sealed` (further restricted subclassing), or
- `non-sealed` (removes restrictions, allows normal inheritance)

Example:

```java
public sealed class Shape
    permits Circle, Rectangle, Square {
    public abstract double area();
}

public final class Circle extends Shape {
    private final double radius;
    public Circle(double radius) { this.radius = radius; }
    @Override
    public double area() { return Math.PI * radius * radius; }
}

public final class Rectangle extends Shape {
    private final double width, height;
    public Rectangle(double width, double height) {
        this.width = width;
        this.height = height;
    }
    @Override
    public double area() { return width * height; }
}

public non-sealed class Square extends Shape {
    private final double side;
    public Square(double side) { this.side = side; }
    @Override
    public double area() { return side * side; }
}
```

---

## Key rules

1. A `sealed` class must declare `permits` (or all permitted subclasses must be in the same file).
2. All permitted subclasses must:
   - Directly extend/implement the sealed type.
   - Be in the same module (or same package if no module).
   - Declare `final`, `sealed`, or `non-sealed`.
3. No other class can extend the sealed class except those listed in `permits`.

---

## Why use sealed classes?

### 1. Exhaustive `switch` on type

Because the set of subclasses is known, the compiler can help ensure you handle all cases:

```java
public String describeShape(Shape shape) {
    return switch (shape) {
        case Circle c    -> "Circle with area " + c.area();
        case Rectangle r -> "Rectangle with area " + r.area();
        case Square s    -> "Square with area " + s.area();
        // no default needed if all permitted types are covered
    };
}
```

**Example input/output:**

```java
public class Main {
    public static void main(String[] args) {
        Shape s1 = new Circle(2.0);
        Shape s2 = new Rectangle(3.0, 4.0);
        Shape s3 = new Square(5.0);

        System.out.println(describeShape(s1));
        System.out.println(describeShape(s2));
        System.out.println(describeShape(s3));
    }

    static String describeShape(Shape shape) {
        return switch (shape) {
            case Circle c    -> "Circle area: " + c.area();
            case Rectangle r -> "Rectangle area: " + r.area();
            case Square s    -> "Square area: " + s.area();
        };
    }
}
```

**Output:**

```text
Circle area: 12.566370614359172
Rectangle area: 12.0
Square area: 25.0
```

## Sealed interfaces

You can also seal interfaces:

```java
public sealed interface Vehicle
    permits Car, Truck {}

public final class Car implements Vehicle {
    // ...
}

public final class Truck implements Vehicle {
    // ...
}
```

---

## Summary of main ideas

- `sealed` restricts which classes can extend/implement a type.
- `permits` lists the allowed subclasses.
- Subclasses must be `final`, `sealed`, or `non-sealed`.
- Helps with:
  - Safer, more predictable inheritance hierarchies.
  - Better pattern matching and exhaustive `switch` handling.
  - Clearer API design and maintenance.
