
Here’s a concise summary of the Saga pattern for distributed transactions in Java, with code-style examples.

---

## Key Ideas of the Saga Pattern

1. **Problem**: In a microservices/distributed system, a single business operation often spans multiple services and databases. Traditional ACID transactions (2PC) are hard or impossible to use across services.

2. **Saga Pattern**:  
   A saga is a **sequence of local transactions**.  
   - Each local transaction updates data within a single service.  
   - If one step fails, previously completed steps are **compensated** by executing defined **compensating transactions** (undo logic).

3. **Two Main Styles**:
   - **Choreography**:  
     - No central coordinator.  
     - Services communicate via events.  
     - Each service listens for events, performs its local transaction, and publishes the next event.
   - **Orchestration**:  
     - A central **orchestrator** (Saga coordinator) tells each service what to do next.  
     - Orchestrator manages the flow and compensations.

4. **Compensation Instead of Rollback**:
   - You don’t roll back the whole distributed transaction.  
   - You **logically undo** previous steps (e.g., cancel order, refund payment, release inventory).

5. **Use Cases**:
   - Order processing (Order → Payment → Inventory → Shipping).  
   - Booking systems (Flight → Hotel → Car rental).  
   - Any long-running, multi-service business process.

---

## Simple Example Scenario

Business flow:  
1. Create Order  
2. Reserve Inventory  
3. Process Payment  

If payment fails, we must:  
- Cancel inventory reservation  
- Cancel order

## Orchestration Example (Java, pseudo-style)

### Saga Orchestrator

```java
public class OrderSagaOrchestrator {

    private final OrderService orderService;
    private final InventoryService inventoryService;
    private final PaymentService paymentService;

    public OrderSagaOrchestrator(OrderService orderService,
                                 InventoryService inventoryService,
                                 PaymentService paymentService) {
        this.orderService = orderService;
        this.inventoryService = inventoryService;
        this.paymentService = paymentService;
    }

    public SagaResult placeOrder(OrderRequest request) {
        Long orderId = null;
        boolean inventoryReserved = false;

        try {
            // Step 1: Create order
            orderId = orderService.createOrder(request);
            // Step 2: Reserve inventory
            inventoryService.reserveItems(orderId, request.getItems());
            inventoryReserved = true;
            // Step 3: Process payment
            paymentService.charge(orderId, request.getTotalAmount());

            // All steps succeeded
            orderService.markOrderCompleted(orderId);
            return new SagaResult(true, "Order completed, id=" + orderId);

        } catch (Exception e) {
            // Compensation logic
            if (inventoryReserved) {
                inventoryService.releaseItems(orderId);
            }
            if (orderId != null) {
                orderService.cancelOrder(orderId);
            }
            return new SagaResult(false, "Saga failed: " + e.getMessage());
        }
    }
}
```

### Example Input / Output

**Input (OrderRequest):**

```json
{
  "customerId": 123,
  "items": [
    { "productId": 10, "quantity": 2 },
    { "productId": 20, "quantity": 1 }
  ],
  "totalAmount": 150.00
}
```

**Possible Successful Output:**

```json
{
  "success": true,
  "message": "Order completed, id=1001"
}
```

**Possible Failure Output (e.g., payment fails):**

```json
{
  "success": false,
  "message": "Saga failed: Payment declined"
}
```

In the failure case, the orchestrator:
- Calls `inventoryService.releaseItems(orderId)`  
- Calls `orderService.cancelOrder(orderId)`  

So the system returns to a consistent state.

## Choreography Example (Event-Driven, simplified)

Each service reacts to events and publishes new ones.

### Order Service

```java
public class OrderService {

    public void handleCreateOrderCommand(CreateOrderCommand cmd) {
        Long orderId = createOrderInDb(cmd);
        // Publish event
        eventBus.publish(new OrderCreatedEvent(orderId, cmd.getItems(), cmd.getTotalAmount()));
    }

    public void handlePaymentFailedEvent(PaymentFailedEvent event) {
        cancelOrderInDb(event.getOrderId());
    }
}
```

### Inventory Service

```java
public class InventoryService {

    public void handleOrderCreatedEvent(OrderCreatedEvent event) {
        try {
            reserveItems(event.getOrderId(), event.getItems());
            eventBus.publish(new InventoryReservedEvent(event.getOrderId(), event.getTotalAmount()));
        } catch (InsufficientStockException e) {
            eventBus.publish(new InventoryReservationFailedEvent(event.getOrderId(), e.getMessage()));
        }
    }

    public void handlePaymentFailedEvent(PaymentFailedEvent event) {
        releaseItems(event.getOrderId());
    }
}
```

### Payment Service

```java
public class PaymentService {

    public void handleInventoryReservedEvent(InventoryReservedEvent event) {
        try {
            charge(event.getOrderId(), event.getTotalAmount());
            eventBus.publish(new PaymentCompletedEvent(event.getOrderId()));
        } catch (PaymentException e) {
            eventBus.publish(new PaymentFailedEvent(event.getOrderId(), e.getMessage()));
        }
    }
}
```

Here, the **sequence of events** forms the saga. Each service has:
- A **forward** action (reserve, charge, etc.)
- A **compensating** action (release, refund, cancel, etc.)

---

## Main Takeaways

- Saga pattern is a way to handle **distributed transactions** without 2PC.  
- It uses **local transactions + compensating actions**.  
- Two styles: **orchestration** (central coordinator) and **choreography** (event-driven).  
- It’s essential for **microservices** where a business process spans multiple services and databases.
