
Here’s a concise summary of the key concepts, with simple Java examples.

---

## 1. Race Condition

**Idea:**  
Multiple threads access and modify shared data without proper synchronization, causing unpredictable results.

**Example (incorrect counter):**

```java
public class RaceConditionDemo {
    private static int counter = 0;

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 1000; i++) {
                counter++; // not atomic, race condition here
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println("Final counter: " + counter);
    }
}
```

**Expected output (ideal):**  
`Final counter: 2000`

**Possible actual outputs:**  
`Final counter: 1734`, `1897`, `2000`, etc. (non-deterministic)

**Fix (synchronized):**

```java
public class RaceConditionFixed {
    private static int counter = 0;

    public static synchronized void increment() {
        counter++;
    }

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 1000; i++) {
                increment();
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println("Final counter: " + counter); // always 2000
    }
}
```

## 2. Deadlock

**Idea:**  
Two or more threads are blocked forever, each waiting for a lock held by another thread.

**Example:**

```java
public class DeadlockDemo {
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (LOCK_A) {
                System.out.println("T1: locked A");
                try { Thread.sleep(100); } catch (InterruptedException ignored) {}
                synchronized (LOCK_B) {
                    System.out.println("T1: locked B");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (LOCK_B) {
                System.out.println("T2: locked B");
                try { Thread.sleep(100); } catch (InterruptedException ignored) {}
                synchronized (LOCK_A) {
                    System.out.println("T2: locked A");
                }
            }
        });

        t1.start();
        t2.start();
    }
}
```

**Typical output:**
```
T1: locked A
T2: locked B
```
Then both threads freeze, each waiting for the other lock.

**Prevention idea:**  
Always acquire locks in the same global order (e.g., always lock A then B).

## 3. Livelock

**Idea:**  
Threads are not blocked, but keep changing state in response to each other and never make progress (they “dance around” the problem).

**Conceptual example (pseudo-style Java):**

```java
public class LivelockDemo {
    static class Worker {
        private boolean active;

        public Worker(boolean active) {
            this.active = active;
        }

        public void work(Worker other) {
            while (active) {
                if (other.active) {
                    System.out.println(Thread.currentThread().getName() +
                            ": letting the other worker go first");
                    try { Thread.sleep(50); } catch (InterruptedException ignored) {}
                    continue; // keeps yielding
                }
                System.out.println(Thread.currentThread().getName() + ": doing work");
                break;
            }
        }
    }

    public static void main(String[] args) {
        Worker w1 = new Worker(true);
        Worker w2 = new Worker(true);

        new Thread(() -> w1.work(w2), "Worker-1").start();
        new Thread(() -> w2.work(w1), "Worker-2").start();
    }
}
```

**Behavior:**  
Both threads keep “politely” yielding to each other and no real work is done. They are active but not progressing.

**Fix idea:**  
Introduce randomness, backoff, or a clear rule for who proceeds.

---

## 4. Starvation

**Idea:**  
A thread never gets CPU time or access to a resource because other threads keep taking it first.

**Example (high-priority threads starving a low-priority one):**

```java
public class StarvationDemo {
    public static void main(String[] args) {
        Runnable busyTask = () -> {
            while (true) {
                // do some work
            }
        };

        Thread high1 = new Thread(busyTask, "High-1");
        Thread high2 = new Thread(busyTask, "High-2");
        Thread low = new Thread(() -> {
            while (true) {
                System.out.println("Low priority thread running");
                try { Thread.sleep(100); } catch (InterruptedException ignored) {}
            }
        }, "Low");

        high1.setPriority(Thread.MAX_PRIORITY);
        high2.setPriority(Thread.MAX_PRIORITY);
        low.setPriority(Thread.MIN_PRIORITY);

        high1.start();
        high2.start();
        low.start();
    }
}
```

**Possible behavior:**  
The low-priority thread may rarely or never print, depending on the JVM/OS scheduler.

**Other starvation cases:**  
- A thread always losing in lock competition.  
- A thread waiting on a queue that is always dominated by others.

**Prevention ideas:**  
- Fair locks (`new ReentrantLock(true)` for fairness).  
- Avoid extreme priority differences.  
- Use well-designed concurrent data structures.

---

### Summary of the Four Concepts

- **Race condition:** Incorrect results due to unsynchronized access to shared data.  
- **Deadlock:** Threads permanently waiting on each other’s locks.  
- **Livelock:** Threads keep changing state but make no progress.  
- **Starvation:** A thread rarely or never gets resources/CPU to run.
