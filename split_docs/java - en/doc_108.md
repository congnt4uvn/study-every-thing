
Here’s a concise summary of the three distributed-system patterns, with simple Java-style examples.

---

## 1. Circuit Breaker Pattern

**Idea:**  
Protect your system from repeatedly calling a failing service.  
- **Closed:** Calls flow normally; failures are counted.  
- **Open:** Calls are blocked immediately (fail fast) after too many failures.  
- **Half-open:** Allow a few test calls to see if the service has recovered.

**Why:**  
Prevents cascading failures, reduces load on unhealthy services, improves resilience.

**Example (conceptual Java):**

```java
public class CircuitBreaker {
    private int failureCount = 0;
    private final int failureThreshold = 3;
    private long lastFailureTime = 0;
    private final long openStateTimeoutMs = 5000;
    private State state = State.CLOSED;

    enum State { CLOSED, OPEN, HALF_OPEN }

    public synchronized String call(Supplier<String> remoteCall) {
        long now = System.currentTimeMillis();

        // Move from OPEN to HALF_OPEN after timeout
        if (state == State.OPEN && now - lastFailureTime > openStateTimeoutMs) {
            state = State.HALF_OPEN;
        }

        if (state == State.OPEN) {
            return "Fallback: circuit is open";
        }

        try {
            String result = remoteCall.get();
            // Success: reset
            failureCount = 0;
            state = State.CLOSED;
            return result;
        } catch (Exception e) {
            failureCount++;
            lastFailureTime = now;
            if (failureCount >= failureThreshold) {
                state = State.OPEN;
            }
            return "Fallback: error calling service";
        }
    }
}
```

**Example usage:**

```java
CircuitBreaker cb = new CircuitBreaker();

String response = cb.call(() -> {
    // Simulate remote call
    if (Math.random() < 0.8) { // 80% chance of failure
        throw new RuntimeException("Remote service failed");
    }
    return "OK from remote service";
});

System.out.println(response);
```

**Possible outputs:**
- `"OK from remote service"` (when call succeeds)
- `"Fallback: error calling service"` (failure but circuit not yet open)
- `"Fallback: circuit is open"` (after repeated failures)

## 2. Retry Pattern

**Idea:**  
Automatically retry a failed operation a limited number of times, often with a delay between attempts.

**Why:**  
Handles transient errors (temporary network glitches, short outages) without failing the whole operation.

**Key points:**
- Limit max retries.
- Use delay or exponential backoff.
- Combine with timeout and circuit breaker.

**Example (simple retry with fixed delay):**

```java
public class RetryUtil {

    public static <T> T callWithRetry(Supplier<T> remoteCall,
                                      int maxRetries,
                                      long delayMs) throws Exception {
        int attempt = 0;
        while (true) {
            try {
                return remoteCall.get();
            } catch (Exception e) {
                attempt++;
                if (attempt > maxRetries) {
                    throw new Exception("All retries failed", e);
                }
                Thread.sleep(delayMs);
            }
        }
    }
}
```

**Example usage:**

```java
try {
    String result = RetryUtil.callWithRetry(() -> {
        if (Math.random() < 0.7) { // 70% chance of failure
            throw new RuntimeException("Temporary error");
        }
        return "Success!";
    }, 3, 500);

    System.out.println("Final result: " + result);
} catch (Exception e) {
    System.out.println("Operation failed after retries: " + e.getMessage());
}
```

**Possible outputs:**
- `"Final result: Success!"` (if one of the attempts succeeds)
- `"Operation failed after retries: All retries failed"` (if all attempts fail)

## 3. Timeout Pattern

**Idea:**  
Limit how long you wait for a remote call. If it takes too long, abort and treat it as a failure.

**Why:**  
Prevents threads from being blocked indefinitely, improves responsiveness, and avoids resource exhaustion.

**Key points:**
- Set reasonable timeout values (connect + read).
- Always handle timeout exceptions.
- Often used together with retry and circuit breaker.

**Example (using `HttpURLConnection` with timeouts):**

```java
public static String getWithTimeout(String urlStr, int timeoutMs) throws IOException {
    URL url = new URL(urlStr);
    HttpURLConnection conn = (HttpURLConnection) url.openConnection();
    conn.setConnectTimeout(timeoutMs);
    conn.setReadTimeout(timeoutMs);
    conn.setRequestMethod("GET");

    try (BufferedReader reader = new BufferedReader(
            new InputStreamReader(conn.getInputStream()))) {
        StringBuilder sb = new StringBuilder();
        String line;
        while ((line = reader.readLine()) != null) {
            sb.append(line);
        }
        return sb.toString();
    }
}
```

**Example usage:**

```java
try {
    String body = getWithTimeout("https://example.com/slow-api", 1000);
    System.out.println("Response: " + body);
} catch (SocketTimeoutException e) {
    System.out.println("Request timed out");
} catch (IOException e) {
    System.out.println("IO error: " + e.getMessage());
}
```

**Possible outputs:**
- `"Response: ..."` (if the server responds within 1 second)
- `"Request timed out"` (if it takes longer than 1 second)

---

## How They Work Together

In real systems, you often combine them:

1. **Timeout** on each remote call (don’t wait forever).
2. **Retry** a few times if the failure looks transient.
3. **Circuit breaker** to stop calling a service that is consistently failing.

This combination makes distributed Java applications more resilient, responsive, and fault-tolerant.
