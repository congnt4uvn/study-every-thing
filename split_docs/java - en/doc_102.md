
Here’s a concise summary of the key ideas, with simple examples.

---

## 1. Load Balancing

**Goal:** Distribute incoming traffic across multiple servers so no single server is overloaded.

**Key points:**
- Improves **availability** (if one server fails, others can handle traffic).
- Improves **scalability** (you can add more servers behind the load balancer).
- Can use different algorithms:
  - Round Robin
  - Least Connections
  - IP Hash, etc.

### Simple Conceptual Flow

Client → Load Balancer → Server A / Server B / Server C

### Example: Very Simple Java “Load Balancer” Logic

This is not production-grade, just to illustrate the idea of distributing requests.

```java
import java.util.List;
import java.util.concurrent.atomic.AtomicInteger;

class Server {
    private final String name;

    public Server(String name) {
        this.name = name;
    }

    public String handleRequest(String request) {
        return "Server " + name + " handled: " + request;
    }
}

class RoundRobinLoadBalancer {
    private final List<Server> servers;
    private final AtomicInteger index = new AtomicInteger(0);

    public RoundRobinLoadBalancer(List<Server> servers) {
        this.servers = servers;
    }

    public String route(String request) {
        int i = Math.abs(index.getAndIncrement() % servers.size());
        Server server = servers.get(i);
        return server.handleRequest(request);
    }
}

public class Main {
    public static void main(String[] args) {
        List<Server> servers = List.of(
                new Server("A"),
                new Server("B"),
                new Server("C")
        );

        RoundRobinLoadBalancer lb = new RoundRobinLoadBalancer(servers);

        for (int i = 1; i <= 6; i++) {
            String response = lb.route("Request " + i);
            System.out.println(response);
        }
    }
}
```

**Sample Output:**
```
Server A handled: Request 1
Server B handled: Request 2
Server C handled: Request 3
Server A handled: Request 4
Server B handled: Request 5
Server C handled: Request 6
```

This shows how requests are distributed across multiple servers.

## 2. Horizontal vs Vertical Scaling

### Vertical Scaling (Scale Up)

**Definition:** Add more power (CPU, RAM, disk) to a single machine.

**Characteristics:**
- Simpler to implement (no need to change application much).
- Limited by the maximum capacity of one machine.
- If that machine fails, everything goes down (single point of failure).

**Example (conceptual):**
- Old: 4 CPU, 8 GB RAM
- New: 16 CPU, 64 GB RAM

In Java, you might just increase JVM heap size:

```bash
# Before
java -Xms512m -Xmx512m -jar app.jar

# After (vertical scale)
java -Xms4g -Xmx4g -jar app.jar
```

No code change, just more resources.

---

### Horizontal Scaling (Scale Out)

**Definition:** Add more machines/instances and distribute load across them.

**Characteristics:**
- Better for high scalability.
- Requires load balancing and often stateless services.
- More complex: need to handle distributed state, caching, etc.

**Example (conceptual):**
- Instead of 1 big server, run 3 smaller instances of the same Java app:
  - app-instance-1
  - app-instance-2
  - app-instance-3  
  All behind a load balancer.

**Simple Java Example: Stateless Service (Good for Horizontal Scaling)**

```java
// A simple stateless controller (e.g., in Spring Boot)
@RestController
public class GreetingController {

    @GetMapping("/greet")
    public String greet(@RequestParam String name) {
        // No server-side session or local state
        return "Hello, " + name;
    }
}
```

You can run multiple instances of this same app; any instance can handle any request because it doesn’t rely on local state.

**Sample HTTP Request/Response:**

Request:
```http
GET /greet?name=Alice HTTP/1.1
Host: example.com
```

Response (from any instance):
```http
HTTP/1.1 200 OK
Content-Type: text/plain

Hello, Alice
```

---

## Summary

- **Load Balancing**: Distributes requests across multiple servers to improve availability and scalability.
- **Vertical Scaling**: Make one server stronger (more CPU/RAM). Simple but limited and can be a single point of failure.
- **Horizontal Scaling**: Add more servers and use a load balancer. More scalable but architecturally more complex; works best with stateless services.
