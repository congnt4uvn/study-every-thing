
Here’s a concise summary and comparison of JPA Criteria API vs JPQL, with code examples.

---

## 1. What They Are

**JPQL (Java Persistence Query Language)**  
- String-based query language similar to SQL but for entities.  
- Easier to read and write.  
- Compiled only at runtime (errors found when executing).

**Criteria API**  
- Type-safe, programmatic way to build queries using Java objects.  
- Harder to read, more verbose.  
- Compile-time safety (refactoring-friendly, IDE support).

---

## 2. When to Use Which

**Use JPQL when:**
- Queries are simple to medium complexity.
- You want readability and quick development.
- Queries are mostly static (don’t change much at runtime).

**Use Criteria API when:**
- You need dynamic queries (conditions added based on user input, filters, etc.).
- You want type safety and refactoring support.
- You build complex search screens with many optional parameters.

---

## 3. Basic Examples

Assume an entity:

```java
@Entity
public class Person {
    @Id
    private Long id;

    private String name;
    private int age;

    // getters/setters
}
```

### 3.1 Simple Select

**JPQL:**

```java
String jpql = "SELECT p FROM Person p WHERE p.age > :age";
TypedQuery<Person> query = em.createQuery(jpql, Person.class);
query.setParameter("age", 18);
List<Person> result = query.getResultList();
```

**Input:** `age = 18`  
**Output:** List of `Person` entities with `age > 18`.

**Criteria API:**

```java
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Person> cq = cb.createQuery(Person.class);
Root<Person> person = cq.from(Person.class);

cq.select(person)
  .where(cb.greaterThan(person.get("age"), 18));

TypedQuery<Person> query = em.createQuery(cq);
List<Person> result = query.getResultList();
```

Same input/output as above.

---

### 3.2 Dynamic Query with Optional Filters

Suppose you have optional filters: `name` and `minAge`.  
If a filter is null, it should not be applied.

**JPQL (often built as a string):**

```java
StringBuilder jpql = new StringBuilder("SELECT p FROM Person p WHERE 1=1");

if (name != null) {
    jpql.append(" AND p.name = :name");
}
if (minAge != null) {
    jpql.append(" AND p.age >= :minAge");
}

TypedQuery<Person> query = em.createQuery(jpql.toString(), Person.class);

if (name != null) {
    query.setParameter("name", name);
}
if (minAge != null) {
    query.setParameter("minAge", minAge);
}

List<Person> result = query.getResultList();
```

**Criteria API (more structured):**

```java
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Person> cq = cb.createQuery(Person.class);
Root<Person> person = cq.from(Person.class);

List<Predicate> predicates = new ArrayList<>();

if (name != null) {
    predicates.add(cb.equal(person.get("name"), name));
}
if (minAge != null) {
    predicates.add(cb.greaterThanOrEqualTo(person.get("age"), minAge));
}

cq.select(person)
  .where(predicates.toArray(new Predicate[0]));

TypedQuery<Person> query = em.createQuery(cq);
List<Person> result = query.getResultList();
```

**Example Input:**  
- `name = "Alice"`  
- `minAge = 20`

**Output:**  
List of `Person` where `name = "Alice"` and `age >= 20`.

---

### 3.3 Aggregation Example

Count how many persons are older than a given age.

**JPQL:**

```java
String jpql = "SELECT COUNT(p) FROM Person p WHERE p.age > :age";
TypedQuery<Long> query = em.createQuery(jpql, Long.class);
query.setParameter("age", 30L);
Long count = query.getSingleResult();
```

**Criteria API:**

```java
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Long> cq = cb.createQuery(Long.class);
Root<Person> person = cq.from(Person.class);

cq.select(cb.count(person))
  .where(cb.greaterThan(person.get("age"), 30));

TypedQuery<Long> query = em.createQuery(cq);
Long count = query.getSingleResult();
```

**Input:** `age = 30`  
**Output:** `count` = number of persons with `age > 30`.

---

## 4. Pros and Cons Summary

**JPQL Pros:**
- More concise and readable.
- Easier for developers familiar with SQL.
- Good for static queries.

**JPQL Cons:**
- String-based: no compile-time checking of field names.
- Harder to refactor safely.
- Dynamic query building can become messy.

**Criteria API Pros:**
- Type-safe and refactoring-friendly.
- Better for complex, dynamic queries.
- IDE autocompletion for attributes.

**Criteria API Cons:**
- Verbose and less readable.
- More boilerplate code.
- Harder to understand at a glance.

---

## 5. Practical Guideline

- Start with **JPQL** for most queries.
- Switch to **Criteria API** when:
  - You have many optional filters.
  - You need to build queries dynamically.
  - You want strong type safety and refactoring support.
