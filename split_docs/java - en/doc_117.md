
Here’s a concise summary of **JUnit 5 basics** for unit & integration testing in Java, with simple code examples and expected outputs.

---

## 1. What is JUnit 5?

- A modern testing framework for Java.
- Used for:
  - **Unit tests**: testing individual classes/methods.
  - **Integration tests**: testing how components work together.
- Main modules:
  - **JUnit Platform** – runs tests.
  - **JUnit Jupiter** – new programming model and annotations.
  - **JUnit Vintage** – backward compatibility with JUnit 3/4.

## 2. Basic Setup (Maven)

```xml
<dependencies>
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.10.0</version>
        <scope>test</scope>
    </dependency>

<build>
    <plugins>

            <configuration>
                <useModulePath>false</useModulePath>
            </configuration>
        </plugin>
    </plugins>
</build>
```

Run tests: `mvn test`

## 3. Core Annotations

- `@Test` – marks a test method.
- `@BeforeEach` – runs before each test.
- `@AfterEach` – runs after each test.
- `@BeforeAll` – runs once before all tests (must be `static`).
- `@AfterAll` – runs once after all tests (must be `static`).
- `@Disabled` – disables a test.
- `@DisplayName` – custom name for test.

### Example

```java
import org.junit.jupiter.api.*;

class CalculatorTest {

    private Calculator calculator;

    @BeforeAll
    static void initAll() {
        System.out.println("Before all tests");
    }

    @BeforeEach
    void init() {
        calculator = new Calculator();
        System.out.println("Before each test");
    }

    @Test
    @DisplayName("Addition works")
    void testAdd() {
        int result = calculator.add(2, 3);
        Assertions.assertEquals(5, result);
    }

    @AfterEach
    void tearDown() {
        System.out.println("After each test");
    }

    @AfterAll
    static void tearDownAll() {
        System.out.println("After all tests");
    }
}
```

**Example class under test:**

```java
class Calculator {
    int add(int a, int b) {
        return a + b;
    }
}
```

**Expected console output (simplified):**

```text
Before all tests
Before each test
After each test
After all tests
```

Test result: `testAdd` → **PASSED**

## 4. Assertions

Common assertions from `org.junit.jupiter.api.Assertions`:

- `assertEquals(expected, actual)`
- `assertNotEquals(unexpected, actual)`
- `assertTrue(condition)`
- `assertFalse(condition)`
- `assertNull(object)`
- `assertNotNull(object)`
- `assertThrows(Exception.class, executable)`
- `assertAll(...)` – group multiple assertions.

### Example with assertions

```java
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class StringUtilsTest {

    @Test
    void testIsBlank() {
        assertTrue(StringUtils.isBlank("   "));
        assertTrue(StringUtils.isBlank(null));
        assertFalse(StringUtils.isBlank("abc"));
    }

    @Test
    void testException() {
        Exception ex = assertThrows(
            IllegalArgumentException.class,
            () -> StringUtils.requireNonBlank("   ")
        );
        assertEquals("Text must not be blank", ex.getMessage());
    }
}
```

**Example implementation:**

```java
class StringUtils {
    static boolean isBlank(String s) {
        return s == null || s.trim().isEmpty();
    }

    static String requireNonBlank(String s) {
        if (isBlank(s)) {
            throw new IllegalArgumentException("Text must not be blank");
        }
        return s;
    }
}
```

**Expected behavior:**

- `testIsBlank`:
  - `isBlank("   ")` → `true`
  - `isBlank(null)` → `true`
  - `isBlank("abc")` → `false`
- `testException`:
  - `requireNonBlank("   ")` throws `IllegalArgumentException` with message `"Text must not be blank"`.

## 5. Parameterized Tests

Run the same test with different inputs.

Key annotations:
- `@ParameterizedTest`
- `@ValueSource`, `@CsvSource`, `@MethodSource`, etc.

### Example

```java
import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.ValueSource;
import static org.junit.jupiter.api.Assertions.*;

class EvenNumberTest {

    @ParameterizedTest
    @ValueSource(ints = {2, 4, 6, 8})
    void testIsEven(int number) {
        assertTrue(isEven(number));
    }

    boolean isEven(int n) {
        return n % 2 == 0;
    }
}
```

**Inputs/Outputs:**

- Input: `2` → `isEven(2)` → `true` → test passes
- Input: `4` → `true` → passes
- Input: `6` → `true` → passes
- Input: `8` → `true` → passes

## 6. Testing Exceptions and Timeouts

### Exceptions

```java
@Test
void testDivideByZero() {
    ArithmeticException ex = assertThrows(
        ArithmeticException.class,
        () -> divide(10, 0)
    );
    assertEquals("/ by zero", ex.getMessage());
}

int divide(int a, int b) {
    return a / b;
}
```

### Timeouts

```java
import static org.junit.jupiter.api.Assertions.assertTimeout;
import java.time.Duration;

@Test
void testShouldFinishQuickly() {
    assertTimeout(Duration.ofMillis(100), () -> {
        // code must finish within 100 ms
        Thread.sleep(50);
    });
}
```

## 7. Basic Integration Test Idea

- Use JUnit 5 to test multiple components together:
  - Example: service + repository + database (often with test containers or in-memory DB).
- Still uses the same annotations (`@Test`, `@BeforeAll`, etc.), but setup is more complex.

### Simple integration-style example (no real DB):

```java
class UserServiceIntegrationTest {

    private UserRepository repo;
    private UserService service;

    @BeforeEach
    void setup() {
        repo = new InMemoryUserRepository();
        service = new UserService(repo);
    }

    @Test
    void testCreateAndFindUser() {
        service.createUser("alice");
        User user = service.findUser("alice");

        assertNotNull(user);
        assertEquals("alice", user.getName());
    }
}
```

**Expected behavior:**

- After `createUser("alice")`, `findUser("alice")` returns a non-null `User` with name `"alice"`.

---

## 8. Key Takeaways

- JUnit 5 is the standard for Java unit & integration testing.
- Use annotations (`@Test`, `@BeforeEach`, etc.) to structure tests.
- Use assertions to verify behavior.
- Parameterized tests help avoid duplication.
- Same framework can be used for both unit and integration tests; the difference is in what you set up and test.
