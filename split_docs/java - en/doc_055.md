## 1. Memory Leaks in Java

### Key Ideas
- A *memory leak* in Java usually means: objects are no longer needed but are still referenced, so the Garbage Collector (GC) cannot free them.
- Common causes:
  - Static collections that grow and never shrink.
  - Caches without eviction policies.
  - Listeners or callbacks not removed.
  - Long-lived objects holding references to short-lived data.

### Simple Example of a Memory Leak Pattern

```java
import java.util.ArrayList;
import java.util.List;

public class MemoryLeakExample {
    // Static list lives for the entire application lifetime
    private static final List<byte[]> leakyList = new ArrayList<>();

    public static void main(String[] args) throws InterruptedException {
        while (true) {
            // Allocate 1 MB and store it in a static list
            byte[] data = new byte[1024 * 1024];
            leakyList.add(data);

            System.out.println("List size: " + leakyList.size());
            Thread.sleep(100); // slow down a bit
        }
    }
}
```

**What happens (output example):**
```text
List size: 1
List size: 2
List size: 3
...
List size: 1000
Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
```

**Why this leaks:**  
The static `leakyList` keeps references to all `byte[]` objects, so GC cannot reclaim them.

### How to Detect Memory Leaks

Tools and techniques:
- **Java Flight Recorder / Java Mission Control**
- **VisualVM** or **Eclipse MAT (Memory Analyzer Tool)**:
  - Take a heap dump.
  - Look for:
    - Dominator tree: which objects retain the most memory.
    - Collections that keep growing.
- **jmap / jcmd**:
  - `jmap -dump:live,format=b,file=heap.hprof <pid>`
  - Analyze the dump with MAT or VisualVM.

## 2. CPU Hotspots

### Key Ideas
- A *CPU hotspot* is a method or code path that consumes a large portion of CPU time.
- Causes:
  - Inefficient algorithms (e.g., O(n²) where O(n log n) is possible).
  - Busy-wait loops.
  - Excessive synchronization or contention.
  - Unnecessary work in tight loops.

### Example of a CPU Hotspot

```java
public class CpuHotspotExample {

    // Inefficient prime check (very slow for large numbers)
    public static boolean isPrime(long n) {
        if (n < 2) return false;
        for (long i = 2; i < n; i++) { // O(n)
            if (n % i == 0) return false;
        }
        return true;
    }

    public static void main(String[] args) {
        long count = 0;
        long limit = 200_000; // large number

        long start = System.currentTimeMillis();
        for (long i = 2; i < limit; i++) {
            if (isPrime(i)) {
                count++;
            }
        }
        long end = System.currentTimeMillis();

        System.out.println("Prime count: " + count);
        System.out.println("Time (ms): " + (end - start));
    }
}
```

**Example output (depends on machine):**
```text
Prime count: 17984
Time (ms): 4500
```

**Why this is a hotspot:**  
`isPrime` is called many times and does too much work per call.

### Optimized Version

```java
public static boolean isPrimeOptimized(long n) {
    if (n < 2) return false;
    if (n % 2 == 0) return n == 2;
    long limit = (long) Math.sqrt(n);
    for (long i = 3; i <= limit; i += 2) {
        if (n % i == 0) return false;
    }
    return true;
}
```

Replacing `isPrime` with `isPrimeOptimized` will significantly reduce CPU time.

## 3. How to Find CPU Hotspots

Use profilers and tools:
- **Java Flight Recorder / Mission Control**
- **VisualVM** (Sampler/Profiler tab)
- **YourKit**, **JProfiler** (commercial)
- **jcmd** and `-Xprof` (basic profiling)

Typical steps:
1. Run the application under a profiler.
2. Look at:
   - Methods with highest *self time* or *CPU time*.
   - Call trees to see which callers trigger heavy work.
3. Optimize the worst offenders first (top 5–10 methods).

---

## 4. General Performance Tuning Tips

- Measure first: use profiling and metrics, don’t guess.
- Fix the biggest problems first (memory or CPU).
- Re-measure after each change.
- Use appropriate data structures and algorithms.
- Avoid unnecessary object creation in hot paths.
- Use caching carefully (with eviction) to avoid memory leaks.
