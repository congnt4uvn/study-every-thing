
Here’s a concise summary and examples about common Java performance anti‑patterns:

---

## 1. Excessive Object Creation

**Idea:** Creating too many short‑lived objects increases GC pressure and CPU usage.

### Bad pattern
```java
public class StringJoinerBad {
    public static String joinWithComma(List<String> items) {
        String result = "";
        for (String item : items) {
            // Creates a new String object every iteration
            result = result + item + ",";
        }
        return result;
    }

    public static void main(String[] args) {
        List<String> list = List.of("a", "b", "c");
        System.out.println(joinWithComma(list));
    }
}
```

**Output:**
```
a,b,c,
```

**Problem:** Each `result = result + item` creates a new `String` object.

### Better pattern
Use mutable objects or reuse instances.

```java
public class StringJoinerGood {
    public static String joinWithComma(List<String> items) {
        StringBuilder sb = new StringBuilder();
        for (String item : items) {
            sb.append(item).append(",");
        }
        return sb.toString();
    }

    public static void main(String[] args) {
        List<String> list = List.of("a", "b", "c");
        System.out.println(joinWithComma(list));
    }
}
```

**Output:**
```
a,b,c,
```

**Key points:**
- Avoid creating objects in tight loops when possible.
- Prefer `StringBuilder`/`StringBuffer` over repeated string concatenation in loops.
- Reuse expensive objects (e.g., `DateFormat`, `Pattern`) when safe.

---

## 2. Excessive Synchronization

**Idea:** Overusing `synchronized` or locking large code blocks can cause contention and reduce throughput.

### Bad pattern
```java
public class CounterBad {
    private int count = 0;

    // Entire method is synchronized, even for read
    public synchronized int incrementAndGet() {
        count++;
        return count;
    }

    public synchronized int get() {
        return count;
    }

    public static void main(String[] args) throws InterruptedException {
        CounterBad counter = new CounterBad();

        Runnable task = () -> {
            for (int i = 0; i < 1000; i++) {
                counter.incrementAndGet();
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println(counter.get());
    }
}
```

**Output (example):**
```
2000
```

**Problem:** All threads serialize on the same lock, even for reads.

### Better pattern
Use concurrent utilities or finer‑grained locking.

```java
import java.util.concurrent.atomic.AtomicInteger;

public class CounterGood {
    private final AtomicInteger count = new AtomicInteger(0);

    public int incrementAndGet() {
        return count.incrementAndGet();
    }

    public int get() {
        return count.get();
    }

    public static void main(String[] args) throws InterruptedException {
        CounterGood counter = new CounterGood();

        Runnable task = () -> {
            for (int i = 0; i < 1000; i++) {
                counter.incrementAndGet();
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println(counter.get());
    }
}
```

**Output (example):**
```
2000
```

**Key points:**
- Avoid synchronizing entire methods if only a small part needs protection.
- Prefer `java.util.concurrent` (e.g., `Atomic*`, `ConcurrentHashMap`, `ReentrantLock`) when appropriate.
- Minimize lock scope and avoid nested locks to reduce contention and deadlock risk.

## 3. I/O Blocking

**Idea:** Blocking I/O (disk, network) on main or shared threads can stall the application and reduce throughput.

### Bad pattern
```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;

public class BlockingIOBad {
    public static void main(String[] args) throws IOException {
        // Suppose this is called on a main server thread
        String filePath = "large-file.txt";
        String line;

        try (BufferedReader br = new BufferedReader(new FileReader(filePath))) {
            // This loop blocks until the entire file is read
            while ((line = br.readLine()) != null) {
                processLine(line);
            }
        }
        System.out.println("Done");
    }

    private static void processLine(String line) {
        // Some processing
    }
}
```

**Output (example):**
```
Done
```
…but only after the whole file is read, blocking the calling thread.

### Better pattern (offload to worker thread / async style)

```java
import java.io.BufferedReader;
import java.io.FileReader;
import java.io.IOException;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class BlockingIOBetter {
    private static final ExecutorService IO_POOL = Executors.newFixedThreadPool(4);

    public static void main(String[] args) {
        String filePath = "large-file.txt";

        IO_POOL.submit(() -> {
            try (BufferedReader br = new BufferedReader(new FileReader(filePath))) {
                String line;
                while ((line = br.readLine()) != null) {
                    processLine(line);
                }
                System.out.println("Done reading file");
            } catch (IOException e) {
                e.printStackTrace();
            }
        });

        System.out.println("Main thread continues...");
        IO_POOL.shutdown();
    }

    private static void processLine(String line) {
        // Some processing
    }
}
```

**Possible output:**
```
Main thread continues...
Done reading file
```

**Key points:**
- Don’t perform long‑running I/O on UI threads or main server threads.
- Use thread pools, asynchronous APIs (e.g., NIO, reactive frameworks) to handle many I/O operations efficiently.
- Batch I/O operations and avoid frequent small reads/writes when possible.

---

**Overall main ideas:**

1. **Excessive object creation** → more GC, slower performance. Reuse and use efficient data structures.
2. **Excessive synchronization** → thread contention, low throughput. Use concurrent utilities and minimize lock scope.
3. **I/O blocking** → threads stuck waiting on disk/network. Offload to worker threads or use non‑blocking/async I/O.
