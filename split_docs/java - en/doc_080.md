
Here’s a concise summary of the key ideas around SQL & RDBMS: Transactions, ACID, isolation levels, and locking, with Java examples.

---

## 1. Transactions

A **transaction** is a unit of work that is executed as a single logical operation on the database. It can contain multiple SQL statements and must either:

- **Commit**: all changes are saved.
- **Rollback**: all changes are undone.

### Java Example (JDBC)

```java
import java.sql.*;

public class TransactionExample {
    public static void main(String[] args) throws Exception {
        String url = "jdbc:postgresql://localhost:5432/testdb";
        String user = "test";
        String pass = "test";

        try (Connection conn = DriverManager.getConnection(url, user, pass)) {
            // Turn off auto-commit to manage transaction manually
            conn.setAutoCommit(false);

            try (PreparedStatement ps1 = conn.prepareStatement(
                     "UPDATE accounts SET balance = balance - ? WHERE id = ?");
                 PreparedStatement ps2 = conn.prepareStatement(
                     "UPDATE accounts SET balance = balance + ? WHERE id = ?")) {

                // Transfer 100 from account 1 to account 2
                ps1.setBigDecimal(1, new java.math.BigDecimal("100.00"));
                ps1.setInt(2, 1);
                ps1.executeUpdate();

                ps2.setBigDecimal(1, new java.math.BigDecimal("100.00"));
                ps2.setInt(2, 2);
                ps2.executeUpdate();

                // If both succeed
                conn.commit();
                System.out.println("Transaction committed");
            } catch (Exception e) {
                // If any error occurs, rollback
                conn.rollback();
                System.out.println("Transaction rolled back: " + e.getMessage());
            }
        }
    }
}
```

**Input (conceptual):**  
- Account 1 balance: 500  
- Account 2 balance: 200  

**Output after successful commit:**  
- Account 1 balance: 400  
- Account 2 balance: 300  

If an error occurs in any update, both balances remain unchanged.

## 2. ACID Properties

ACID describes the guarantees of a transaction:

1. **Atomicity**  
   All operations in a transaction succeed or all fail. No partial updates.

2. **Consistency**  
   A transaction moves the database from one valid state to another, respecting all constraints (foreign keys, unique constraints, etc.).

3. **Isolation**  
   Concurrent transactions do not interfere in a way that breaks correctness. Each transaction behaves as if it’s running alone (to some degree, depending on isolation level).

4. **Durability**  
   Once a transaction commits, its changes are permanent, even if the system crashes.

---

## 3. Isolation Levels

Isolation levels control how much one transaction can see the intermediate results of others. Common levels (from weakest to strongest):

1. **READ UNCOMMITTED**  
   - Can see uncommitted changes from other transactions (dirty reads).  
   - Rarely used.

2. **READ COMMITTED**  
   - Only sees committed data.  
   - Prevents dirty reads, but allows non-repeatable reads and phantom reads.

3. **REPEATABLE READ**  
   - Same row read multiple times in a transaction will not change.  
   - Prevents dirty and non-repeatable reads; phantoms may still occur (depends on DB).

4. **SERIALIZABLE**  
   - Highest isolation; transactions behave as if executed one after another.  
   - Prevents dirty reads, non-repeatable reads, and phantom reads.  
   - More locking, less concurrency.

### Java Example: Setting Isolation Level (JDBC)

```java
Connection conn = DriverManager.getConnection(url, user, pass);

// Example: set isolation to REPEATABLE_READ
conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);
conn.setAutoCommit(false);

try (PreparedStatement ps = conn.prepareStatement(
        "SELECT balance FROM accounts WHERE id = ?")) {
    ps.setInt(1, 1);
    ResultSet rs1 = ps.executeQuery();
    rs1.next();
    System.out.println("First read balance: " + rs1.getBigDecimal(1));

    // ... another transaction might try to update this row here ...

    ResultSet rs2 = ps.executeQuery();
    rs2.next();
    System.out.println("Second read balance: " + rs2.getBigDecimal(1));

    conn.commit();
}
```

At **REPEATABLE_READ**, the two reads of the same row will return the same value within this transaction, even if another transaction commits an update in between (depending on DB implementation).

## 4. Locking

**Locks** are mechanisms used by the database to control concurrent access to data and enforce isolation.

Common types:

- **Shared lock (read lock)**  
  - Multiple transactions can read the same data.  
  - Prevents others from writing to it while the lock is held.

- **Exclusive lock (write lock)**  
  - Only one transaction can hold it.  
  - Prevents others from reading or writing the locked data (depending on DB).

- **Row-level locks**  
  - Lock individual rows; allow high concurrency.

- **Table-level locks**  
  - Lock entire tables; simpler but less concurrent.

### Example Scenario (Conceptual)

Transaction A:
```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- holds an exclusive lock on row with id = 1
```

Transaction B:
```sql
BEGIN;
SELECT balance FROM accounts WHERE id = 1;
-- may block until Transaction A commits/rolls back, depending on isolation level
```

### Java Example: Explicit Locking (SELECT ... FOR UPDATE)

```java
try (Connection conn = DriverManager.getConnection(url, user, pass)) {
    conn.setAutoCommit(false);

    // Lock the row for update
    try (PreparedStatement ps = conn.prepareStatement(
            "SELECT balance FROM accounts WHERE id = ? FOR UPDATE")) {
        ps.setInt(1, 1);
        ResultSet rs = ps.executeQuery();
        if (rs.next()) {
            java.math.BigDecimal balance = rs.getBigDecimal(1);
            System.out.println("Locked balance: " + balance);

            // Perform some logic, then update
            try (PreparedStatement update = conn.prepareStatement(
                    "UPDATE accounts SET balance = ? WHERE id = ?")) {
                update.setBigDecimal(1, balance.add(new java.math.BigDecimal("50.00")));
                update.setInt(2, 1);
                update.executeUpdate();
            }
        }
        conn.commit();
    } catch (Exception e) {
        conn.rollback();
    }
}
```

**Behavior:**  
- While this transaction holds the `FOR UPDATE` lock on account 1, other transactions trying to update the same row will block until commit/rollback.

---

### Summary

- **Transactions** group multiple operations into an all-or-nothing unit.
- **ACID** ensures reliability: Atomicity, Consistency, Isolation, Durability.
- **Isolation levels** balance correctness vs. concurrency (READ UNCOMMITTED → SERIALIZABLE).
- **Locking** is how the database enforces isolation and prevents conflicting concurrent operations.

These concepts are fundamental when writing Java applications that interact with relational databases via JDBC, JPA/Hibernate, or other frameworks.
