
Here’s a concise summary and some practical examples around JPA/Hibernate fetch strategies and batch fetching.

---

## 1. Fetch Strategies: EAGER vs LAZY

### Definitions
- **LAZY**: Data is loaded **on demand** (when the association is accessed).
- **EAGER**: Data is loaded **immediately** with the owning entity (via join or additional queries).

### Example: Entity Mapping

```java
@Entity
public class Order {

    @Id
    private Long id;

    // Many orders belong to one customer
    @ManyToOne(fetch = FetchType.LAZY) // or EAGER
    private Customer customer;

    // One order has many order items
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items = new ArrayList<>();
}
```

**Key points:**
- Default for `@ManyToOne` and `@OneToOne` is **EAGER** (JPA spec).
- Default for `@OneToMany` and `@ManyToMany` is **LAZY**.
- In practice, prefer **LAZY** and control fetching via queries (JOIN FETCH) or entity graphs.

---

## 2. N+1 Select Problem

Occurs when:
1. You load a list of entities (1 query).
2. For each entity, a lazy association triggers another query (N queries).

**Example scenario:**

```java
List<Order> orders = em.createQuery(
    "SELECT o FROM Order o", Order.class
).getResultList();

// Accessing lazy association in a loop
for (Order o : orders) {
    System.out.println(o.getCustomer().getName());
}
```

**SQL pattern (simplified):**
- 1 query: `SELECT * FROM orders`
- N queries: `SELECT * FROM customers WHERE id = ?` (for each order)

Total: **N+1 queries** → performance issue.

---

## 3. Solving N+1 with JOIN FETCH

Use `JOIN FETCH` to load associations in a single query.

```java
List<Order> orders = em.createQuery(
    "SELECT o FROM Order o " +
    "JOIN FETCH o.customer " +        // fetch customer
    "LEFT JOIN FETCH o.items",        // fetch items
    Order.class
).getResultList();
```

**Result:**
- Hibernate generates a single SQL query with joins.
- `customer` and `items` are initialized without extra queries.

**Note:** For collections, join fetch can cause duplicates in the result set; use `DISTINCT` in JPQL if needed:

```java
"SELECT DISTINCT o FROM Order o JOIN FETCH o.items"
```

## 4. Batch Fetching (Hibernate-specific)

Batch fetching reduces the number of queries when loading multiple lazy associations of the same type.

### 4.1. Batch fetching for collections

```java
@Entity
public class Order {

    @Id
    private Long id;

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    @BatchSize(size = 10) // Hibernate annotation
    private List<OrderItem> items = new ArrayList<>();
}
```

**Behavior:**
- When Hibernate needs to initialize `items` for multiple `Order` entities, it loads them in batches of 10 instead of one query per order.

**Example flow:**
```java
List<Order> orders = em.createQuery(
    "SELECT o FROM Order o", Order.class
).getResultList();

// Later, accessing items in a loop
for (Order o : orders) {
    o.getItems().size(); // triggers batch fetch
}
```

**SQL pattern (simplified):**
- Instead of:
  - `SELECT * FROM order_item WHERE order_id = ?` (for each order)
- Hibernate does:
  - `SELECT * FROM order_item WHERE order_id IN (?, ?, ..., ?)` (up to batch size)

---

### 4.2. Batch fetching for `@ManyToOne` / `@OneToOne`

```java
@Entity
public class Order {

    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @BatchSize(size = 20)
    private Customer customer;
}
```

When multiple `Order` entities need their `customer` initialized, Hibernate will load customers in batches:

**SQL pattern:**
- `SELECT * FROM customer WHERE id IN (?, ?, ..., ?)` (up to 20 IDs)

---

## 5. Global Batch Fetching (Hibernate config)

Instead of per-association `@BatchSize`, you can configure globally:

```properties
# application.properties
spring.jpa.properties.hibernate.default_batch_fetch_size=16
```

Hibernate will then batch lazy loading of collections and proxies up to 16 at a time.

---

## 6. Example: Comparing Queries

### Without batch fetching (N+1):

**Code:**
```java
List<Order> orders = em.createQuery(
    "SELECT o FROM Order o", Order.class
).getResultList();

for (Order o : orders) {
    System.out.println(o.getCustomer().getName());
}
```

**Queries (if 5 orders):**
1. `SELECT * FROM orders`
2. `SELECT * FROM customers WHERE id = ?` (5 times)

Total: 6 queries.

---

### With batch fetching (batch size = 3):

**Same code**, but `@BatchSize(size = 3)` on `customer` or global config.

**Queries (if 5 orders):**
1. `SELECT * FROM orders`
2. `SELECT * FROM customers WHERE id IN (?, ?, ?)`  // first 3
3. `SELECT * FROM customers WHERE id IN (?, ?)`     // remaining 2

Total: 3 queries instead of 6.

---

## 7. Practical Guidelines

- Default to **LAZY** for associations.
- Use **JOIN FETCH** in queries for known use cases where you need related data immediately.
- Use **batch fetching** to mitigate N+1 when:
  - You can’t or don’t want to use join fetch (e.g., multiple collections).
  - You load many entities and later access the same association.
- Avoid excessive **EAGER** fetching; it can cause:
  - Large, complex SQL queries.
  - Unnecessary data loading.
  - Circular fetch graphs and performance issues.
