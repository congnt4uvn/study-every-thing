
Here’s a concise summary of the key ideas about `hashCode()` and `equals()` in Java, plus examples.

---

## 1. Purpose of `equals()` and `hashCode()`

- `equals(Object o)` checks **logical equality** between two objects.
- `hashCode()` returns an `int` used by **hash-based collections** (e.g., `HashMap`, `HashSet`) to place objects into buckets.

They must be consistent so collections work correctly.

---

## 2. The Contract

If you override `equals()`, you **must** also override `hashCode()` so that:

1. **Consistency between equals and hashCode**
   - If `a.equals(b)` is `true`, then `a.hashCode() == b.hashCode()` must also be `true`.
   - If `a.equals(b)` is `false`, their hash codes *may* be equal (collisions are allowed, just not ideal).

2. **equals() contract**
   - Reflexive: `x.equals(x)` is `true`.
   - Symmetric: `x.equals(y)` is `true` ⇒ `y.equals(x)` is `true`.
   - Transitive: `x.equals(y)` and `y.equals(z)` ⇒ `x.equals(z)`.
   - Consistent: repeated calls give the same result as long as objects don’t change.
   - `x.equals(null)` is always `false`.

3. **hashCode() contract**
   - Consistent: multiple calls on the same object (while it doesn’t change) return the same value.
   - Equal objects must have equal hash codes.

---

## 3. Why It Matters (Collections)

Hash-based collections use:

- `hashCode()` to find the **bucket**.
- `equals()` to find the **exact element** inside that bucket.

If the contract is broken:

- Objects may “disappear” from `HashSet` / `HashMap` (cannot be found even though they are there).
- Lookups, `contains`, `remove`, etc. may fail unexpectedly.

---

## 4. Correct Implementation Example

```java
import java.util.Objects;

public class Person {
    private final String name;
    private final int age;

    public Person(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;              // same reference
        if (o == null || getClass() != o.getClass()) return false;
        Person person = (Person) o;
        return age == person.age &&
               Objects.equals(name, person.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(name, age);          // consistent with equals
    }
}
```

### Example usage (input/output)

```java
import java.util.HashSet;
import java.util.Set;

public class Test {
    public static void main(String[] args) {
        Person p1 = new Person("Alice", 30);
        Person p2 = new Person("Alice", 30);

        System.out.println(p1.equals(p2));       // Output: true
        System.out.println(p1.hashCode() == p2.hashCode()); // Output: true

        Set<Person> set = new HashSet<>();
        set.add(p1);
        System.out.println(set.contains(p2));    // Output: true
    }
}
```

## 5. Common Pitfalls

### 5.1 Overriding equals() but not hashCode()

```java
class BadPerson {
    private String name;

    public BadPerson(String name) {
        this.name = name;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof BadPerson)) return false;
        BadPerson that = (BadPerson) o;
        return name.equals(that.name);
    }

    // hashCode() NOT overridden → uses Object's identity hashCode()
}
```

Usage:

```java
import java.util.HashSet;
import java.util.Set;

public class BadTest {
    public static void main(String[] args) {
        BadPerson p1 = new BadPerson("Bob");
        BadPerson p2 = new BadPerson("Bob");

        System.out.println(p1.equals(p2));       // Output: true

        Set<BadPerson> set = new HashSet<>();
        set.add(p1);
        System.out.println(set.contains(p2));    // Output: false (BUG!)
    }
}
```

Reason: `p1` and `p2` are equal by `equals()`, but have different hash codes, so they go into different buckets.

---

### 5.2 Using mutable fields in equals/hashCode

If fields used in `equals()` / `hashCode()` change **after** insertion into a `HashSet` or as a key in `HashMap`, the object may become “unfindable”.

```java
class MutableKey {
    String key;

    MutableKey(String key) {
        this.key = key;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof MutableKey)) return false;
        MutableKey that = (MutableKey) o;
        return key.equals(that.key);
    }

    @Override
    public int hashCode() {
        return key.hashCode();
    }
}
```

Usage:

```java
import java.util.HashMap;
import java.util.Map;

public class MutableTest {
    public static void main(String[] args) {
        MutableKey k = new MutableKey("A");
        Map<MutableKey, String> map = new HashMap<>();
        map.put(k, "value");

        System.out.println(map.get(k)); // Output: value

        k.key = "B";                    // key changed!

        System.out.println(map.get(k)); // Output: null (cannot find it)
    }
}
```

---

## 6. Practical Guidelines

- Always override `hashCode()` when you override `equals()`.
- Use `Objects.equals()` and `Objects.hash()` to simplify implementations.
- Prefer **immutable** fields for equality and hash code.
- Don’t rely on `hashCode()` for uniqueness; collisions are allowed.
- For IDEs: let IntelliJ/Eclipse generate `equals()` and `hashCode()` for you to avoid mistakes.
