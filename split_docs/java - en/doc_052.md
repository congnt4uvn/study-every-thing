
Here’s a concise summary of Java Garbage Collection (GC) collectors, with simple examples and notes on behavior.  

---

## 1. Serial GC
**Idea:**  
Single-threaded collector; stops the application (Stop-The-World) and uses one thread for GC.  
Best for small applications or single-core environments.

**How to enable:**
```bash
java -XX:+UseSerialGC Main
```

**Example (Java):**
```java
public class SerialGCExample {
    public static void main(String[] args) {
        // Allocate many short-lived objects
        for (int i = 0; i < 1_000_000; i++) {
            String s = new String("test" + i);
        }
        System.out.println("Done");
    }
}
```

**Typical output (simplified, if you enable GC logging):**
```text
[GC (Allocation Failure) ...]  // Stop-the-world pause
[Full GC ...]                  // Another stop-the-world pause
Done
```

---

## 2. Parallel GC
**Idea:**  
Multiple threads perform GC work in parallel; still Stop-The-World, but faster on multi-core CPUs.  
Good for throughput-oriented applications.

**How to enable (default in some older Java versions):**
```bash
java -XX:+UseParallelGC Main
```

**Behavior:**  
Same code as above, but GC phases use multiple threads, reducing pause time compared to Serial GC on multi-core machines.

---

## 3. CMS (Concurrent Mark-Sweep) – Legacy
**Idea:**  
Old, mostly deprecated collector. Tries to reduce pause times by doing most of the marking concurrently with the application.  
Replaced by G1 and newer collectors.

**How to enable (Java 8, legacy):**
```bash
java -XX:+UseConcMarkSweepGC Main
```

**Behavior:**  
- Shorter pauses than Serial/Parallel in many cases.  
- More CPU overhead and fragmentation issues.  
- Removed/disabled in newer Java versions.

---

## 4. G1 (Garbage-First) GC
**Idea:**  
Region-based collector; aims for predictable, low pause times.  
Performs concurrent marking and evacuates “garbage-first” regions.  
Default in modern Java (e.g., Java 11+).

**How to enable (often default):**
```bash
java -XX:+UseG1GC Main
```

**Example with pause target:**
```bash
java -XX:+UseG1GC -XX:MaxGCPauseMillis=200 Main
```

**Behavior:**  
- Heap split into regions.  
- Tries to meet pause-time goals (e.g., 200 ms).  
- Good general-purpose choice.

---

## 5. ZGC
**Idea:**  
Very low pause times (typically < 10 ms), even on very large heaps (multi-GB to TB).  
Highly concurrent; uses colored pointers and load barriers.

**How to enable (Java 11+ with support):**
```bash
java -XX:+UseZGC Main
```

**Behavior:**  
- Most GC work is concurrent.  
- Pauses are very short and mostly independent of heap size.  
- Good for latency-sensitive, large-memory services.

---

## 6. Shenandoah
**Idea:**  
Another low-pause, concurrent collector (similar goal to ZGC).  
Developed by Red Hat; integrated into OpenJDK.

**How to enable (if available in your JDK):**
```bash
java -XX:+UseShenandoahGC Main
```

**Behavior:**  
- Concurrent compaction to avoid long pauses.  
- Designed for low-latency workloads.

---

## Simple Comparison (Conceptual)

| Collector   | Parallel? | Concurrent? | Typical Use Case                         |
|------------|-----------|------------|------------------------------------------|
| Serial     | No        | No         | Small apps, single-core, testing         |
| Parallel   | Yes       | No         | Throughput-focused, multi-core           |
| CMS        | Partial   | Yes        | Legacy low-pause (Java 8 era)            |
| G1         | Yes       | Yes        | Default general-purpose, balanced        |
| ZGC        | Yes       | Yes        | Very low pause, very large heaps         |
| Shenandoah | Yes       | Yes        | Low-latency, concurrent compaction       |

---

## Minimal Demo to Observe GC (any collector)

```java
public class GCDemo {
    public static void main(String[] args) {
        for (int i = 0; i < 5_000_000; i++) {
            byte[] data = new byte[1024]; // 1 KB
        }
        System.out.println("Finished allocations");
    }
}
```

Run with different collectors and GC logging:

```bash
# Serial
java -XX:+UseSerialGC -Xms256m -Xmx256m -Xlog:gc GCDemo

# G1
java -XX:+UseG1GC -Xms256m -Xmx256m -Xlog:gc GCDemo

# ZGC (if supported)
java -XX:+UseZGC -Xms256m -Xmx256m -Xlog:gc GCDemo
```

**Output (simplified):**
```text
[0.123s][info][gc] GC(0) Pause Young (Allocation Failure) ...
[0.456s][info][gc] GC(1) Pause Young (Normal) ...
Finished allocations
```

You can compare pause times and frequency between collectors using these logs.
