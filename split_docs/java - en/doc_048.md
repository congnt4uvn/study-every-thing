
Here’s a concise summary of the Java object lifecycle in the context of the Java Memory Model, plus simple code examples.

---

## Key Points

1. **Object Creation**
   - Objects are created with `new`.
   - Memory is allocated on the **heap**.
   - Constructors initialize the object’s state.

   ```java
   class Person {
       String name;

       Person(String name) {
           this.name = name;
       }
   }

   public class Main {
       public static void main(String[] args) {
           Person p = new Person("Alice"); // object creation
       }
   }
   ```

2. **Object Usage**
   - You access objects through **references**.
   - Multiple references can point to the same object.
   - The object stays alive as long as it is reachable from a root (e.g., local variables, static fields, active threads).

   ```java
   public class Main {
       public static void main(String[] args) {
           Person p1 = new Person("Alice");
           Person p2 = p1; // p2 references the same object as p1

           System.out.println(p1.name); // Output: Alice
           System.out.println(p2.name); // Output: Alice
       }
   }
   ```

3. **Reachability and Lifetime**
   - An object’s lifetime ends when it becomes **unreachable** (no live references from any root).
   - The Java Memory Model defines when writes to fields become visible to other threads, but the GC decides when to reclaim memory.

   ```java
   public class Main {
       static Person global;

       public static void main(String[] args) {
           Person local = new Person("Bob");
           global = local; // still reachable via global

           local = null;   // object still alive (reachable via global)
           global = null;  // now object may become unreachable
       }
   }
   ```

4. **Garbage Collection (GC)**
   - GC automatically reclaims memory of unreachable objects.
   - You cannot precisely control when GC runs.
   - `System.gc()` is only a **request**, not a guarantee.

   ```java
   public class Main {
       public static void main(String[] args) {
           createObjects();
           System.gc(); // may or may not run GC immediately
       }

       private static void createObjects() {
           for (int i = 0; i < 1_000_000; i++) {
               new Object(); // many short-lived objects
           }
       }
   }
   ```

5. **Finalization (Deprecated / Avoid)**
   - `finalize()` was used to run code before GC, but it is deprecated and unreliable.
   - Use `try-with-resources` or explicit `close()` instead.

   ```java
   class Resource implements AutoCloseable {
       @Override
       public void close() {
           System.out.println("Resource closed");
       }
   }

   public class Main {
       public static void main(String[] args) {
           try (Resource r = new Resource()) {
               // use resource
           } // Output: Resource closed
       }
   }
   ```

6. **Escape and Thread Visibility**
   - If an object reference “escapes” to another thread, you must ensure **proper synchronization** (e.g., `synchronized`, `volatile`, `final` fields, or concurrent utilities).
   - The Java Memory Model guarantees visibility only when you use these mechanisms correctly.

   ```java
   public class Shared {
       private int value;
       public synchronized void set(int v) { value = v; }
       public synchronized int get() { return value; }
   }

   // Thread A:
   shared.set(42);

   // Thread B:
   int x = shared.get(); // guaranteed to see 42 due to synchronization
   ```

### Simple Input/Output Example

```java
class Counter {
    private int count = 0;

    public void increment() {
        count++;
    }

    public int get() {
        return count;
    }
}

public class Main {
    public static void main(String[] args) {
        Counter c = new Counter(); // object created

        c.increment();
        c.increment();

        System.out.println(c.get());
    }
}
```

**Output:**
```text
2
```

- Object lifecycle here:
  - `new Counter()` → creation on heap.
  - `c` reference used to call methods.
  - After `main` ends and no references remain, `Counter` object becomes unreachable and is eligible for GC.
