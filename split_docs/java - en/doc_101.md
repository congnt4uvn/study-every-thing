
Here’s a concise summary of the key ideas, with simple Java examples.

---

## 1. In-Memory Caching

**Idea:** Store frequently used data in the application’s memory (e.g., using a `Map`) to avoid repeated expensive operations (DB calls, API calls, etc.).

**Pros:**
- Very fast (no network).
- Simple to implement.

**Cons:**
- Limited by JVM memory.
- Not shared across multiple application instances.
- Need a strategy for eviction (LRU, TTL, etc.).

### Simple In-Memory Cache Example

```java
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

public class InMemoryCache<K, V> {
    private final Map<K, V> cache = new ConcurrentHashMap<>();

    public V get(K key) {
        return cache.get(key);
    }

    public void put(K key, V value) {
        cache.put(key, value);
    }

    public void invalidate(K key) {
        cache.remove(key);
    }

    public void clear() {
        cache.clear();
    }

    public static void main(String[] args) {
        InMemoryCache<String, String> cache = new InMemoryCache<>();

        // Input
        cache.put("user:1", "Alice");
        cache.put("user:2", "Bob");

        // Usage
        System.out.println(cache.get("user:1")); // Output: Alice
        System.out.println(cache.get("user:2")); // Output: Bob

        cache.invalidate("user:1");
        System.out.println(cache.get("user:1")); // Output: null
    }
}
```

---

## 2. Redis Caching

**Idea:** Use Redis (an in-memory data store) as a cache. It’s external to your app, so multiple instances can share it.

**Pros:**
- Shared across services/instances.
- Supports TTL, eviction policies, data structures.
- Very fast network access.

**Cons:**
- Requires network and external service.
- Operational complexity (deployment, monitoring).

### Simple Java + Redis Example (using Jedis)

```xml
<!-- pom.xml dependency -->
<dependency>
    <groupId>redis.clients</groupId>
    <artifactId>jedis</artifactId>
    <version>5.1.0</version>
</dependency>
```

```java
import redis.clients.jedis.Jedis;

public class RedisCacheExample {
    public static void main(String[] args) {
        try (Jedis jedis = new Jedis("localhost", 6379)) {
            // Input: set key with value and TTL (10 seconds)
            jedis.setex("user:1", 10, "Alice");

            // Read from cache
            String value = jedis.get("user:1");
            System.out.println("Cached value: " + value); // Output: Cached value: Alice

            // Invalidate (delete) key
            jedis.del("user:1");
            System.out.println("After delete: " + jedis.get("user:1")); // Output: null
        }
    }
}
```

## 3. Cache Invalidation

**Idea:** Keeping cache consistent with the source of truth (usually a database). When data changes, the cache must be updated or cleared.

Common strategies:

1. **Write-Through**
   - Write to cache and database at the same time.
   - Cache is always up-to-date, but writes are slower.

2. **Write-Back (Write-Behind)**
   - Write to cache first, then asynchronously write to DB.
   - Faster writes, but risk of data loss if cache fails before DB update.

3. **Cache-Aside (Lazy Loading)**
   - Application checks cache first:
     - If **hit**: return cached value.
     - If **miss**: load from DB, put into cache, return.
   - On update: write to DB and then invalidate or update cache.

### Cache-Aside Pattern Example (In-Memory or Redis)

```java
import java.util.HashMap;
import java.util.Map;

class User {
    public final int id;
    public String name;
    public User(int id, String name) { this.id = id; this.name = name; }
}

// Simulated DB
class UserRepository {
    private final Map<Integer, User> db = new HashMap<>();

    public UserRepository() {
        db.put(1, new User(1, "Alice"));
        db.put(2, new User(2, "Bob"));
    }

    public User findById(int id) {
        System.out.println("DB hit for id " + id);
        return db.get(id);
    }

    public void updateUserName(int id, String newName) {
        db.get(id).name = newName;
    }
}

class UserCacheService {
    private final Map<Integer, User> cache = new HashMap<>();
    private final UserRepository repository;

    public UserCacheService(UserRepository repository) {
        this.repository = repository;
    }

    // Cache-aside read
    public User getUser(int id) {
        if (cache.containsKey(id)) {
            System.out.println("Cache hit for id " + id);
            return cache.get(id);
        }
        System.out.println("Cache miss for id " + id);
        User user = repository.findById(id);
        if (user != null) {
            cache.put(id, user);
        }
        return user;
    }

    // Update DB and invalidate cache
    public void updateUserName(int id, String newName) {
        repository.updateUserName(id, newName);
        cache.remove(id); // cache invalidation
    }
}

public class CacheAsideExample {
    public static void main(String[] args) {
        UserRepository repo = new UserRepository();
        UserCacheService service = new UserCacheService(repo);

        // First read: cache miss, DB hit
        User u1 = service.getUser(1);
        System.out.println("User 1 name: " + u1.name);
        // Output:
        // Cache miss for id 1
        // DB hit for id 1
        // User 1 name: Alice

        // Second read: cache hit
        User u2 = service.getUser(1);
        System.out.println("User 1 name: " + u2.name);
        // Output:
        // Cache hit for id 1
        // User 1 name: Alice

        // Update user name
        service.updateUserName(1, "AliceUpdated");

        // Next read: cache miss again (invalidated), DB hit with new value
        User u3 = service.getUser(1);
        System.out.println("User 1 name: " + u3.name);
        // Output:
        // Cache miss for id 1
        // DB hit for id 1
        // User 1 name: AliceUpdated
    }
}
```

---

### Core Takeaways

- **In-memory cache**: simple, fast, per-instance.
- **Redis cache**: shared, scalable, feature-rich.
- **Cache invalidation** is critical to keep data correct; common patterns are **cache-aside**, **write-through**, and **write-back**.
