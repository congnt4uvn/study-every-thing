
Here’s a concise summary of method-level security in Spring Security using `@PreAuthorize` and `@Secured`, with code examples and sample inputs/outputs.

---

## 1. Enabling Method-Level Security

In Spring Security 6+ (Spring Boot 3+):

```java
@Configuration
@EnableMethodSecurity(securedEnabled = true, prePostEnabled = true)
public class SecurityConfig {
}
```

- `securedEnabled = true` → enables `@Secured`
- `prePostEnabled = true` → enables `@PreAuthorize` / `@PostAuthorize`

---

## 2. `@PreAuthorize`

- Evaluates a SpEL (Spring Expression Language) expression **before** the method is executed.
- Can use:
  - `authentication`, `principal`
  - `#paramName` (method parameters)
  - `hasRole(...)`, `hasAuthority(...)`, etc.

### Example: Role-based access

```java
@Service
public class ProductService {

    // Only users with role ADMIN can call this
    @PreAuthorize("hasRole('ADMIN')")
    public String deleteProduct(Long id) {
        // delete logic...
        return "Product " + id + " deleted";
    }
}
```

**Request (input):**

```http
DELETE /products/10
Authorization: Bearer <token-of-user-with-role-ADMIN>
```

**Output (success):**

```json
"Product 10 deleted"
```

**If user does NOT have role ADMIN:**

- HTTP 403 Forbidden (by default)

---

### Example: Using method parameters

```java
@Service
public class OrderService {

    // Only the owner of the order or an ADMIN can view it
    @PreAuthorize("#username == authentication.name or hasRole('ADMIN')")
    public String getOrder(String username, Long orderId) {
        return "Order " + orderId + " for user " + username;
    }
}
```

**Input 1 (owner user):**

- Authenticated user: `john`
- Method call: `getOrder("john", 5L)`

**Output 1:**

```text
"Order 5 for user john"
```

**Input 2 (different user, not admin):**

- Authenticated user: `alice`
- Method call: `getOrder("john", 5L)`

**Output 2:**

- Access denied → 403 Forbidden

---

## 3. `@Secured`

- Simpler, role-based annotation.
- Does **not** use SpEL.
- Uses role names directly (often with `ROLE_` prefix).

### Example

```java
@Service
public class UserService {

    // Only ROLE_ADMIN can call this
    @Secured("ROLE_ADMIN")
    public String createUser(String username) {
        return "User " + username + " created";
    }

    // Multiple roles allowed
    @Secured({"ROLE_ADMIN", "ROLE_MANAGER"})
    public String getAllUsers() {
        return "List of users";
    }
}
```

**Input 1:**

- Authenticated user roles: `ROLE_ADMIN`
- Call: `createUser("bob")`

**Output 1:**

```text
"User bob created"
```

**Input 2:**

- Authenticated user roles: `ROLE_USER`
- Call: `createUser("bob")`

**Output 2:**

- Access denied → 403 Forbidden

---

## 4. When to Use Which

- Use `@PreAuthorize` when you need:
  - Complex conditions
  - Access based on method parameters or current user
- Use `@Secured` when you need:
  - Simple, role-only checks

Both are checked **before** the method runs and will throw an access denied exception (mapped to HTTP 403) if the condition fails.
