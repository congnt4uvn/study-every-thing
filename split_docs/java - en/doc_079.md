## Key Ideas: Indexes in SQL & RDBMS

### 1. What is an Index?

- An index is a data structure (like a sorted lookup table) that helps the database find rows faster.
- It speeds up `SELECT` queries but can slow down `INSERT`, `UPDATE`, and `DELETE` because the index must be updated.

---

## 2. Clustered Index

- Defines the **physical order** of rows in the table.
- A table can usually have **only one** clustered index.
- The table data itself is stored in the order of the clustered index key.
- Often created on the **primary key**.

### When to use a clustered index

- On columns that:
  - Are frequently used in `WHERE`, `JOIN`, `ORDER BY`, or `GROUP BY`.
  - Are **range queries** (e.g., dates, IDs).
  - Are unique or mostly unique.

### Example (SQL Server / MySQL InnoDB)

```sql
-- Create a table
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,      -- often becomes clustered index
    CustomerID INT,
    OrderDate DATE,
    TotalAmount DECIMAL(10, 2)
);

-- Explicit clustered index (SQL Server syntax)
CREATE CLUSTERED INDEX IX_Orders_OrderDate
ON Orders (OrderDate);
```

**Effect:**

- Queries like:

```sql
SELECT * FROM Orders
WHERE OrderDate BETWEEN '2024-01-01' AND '2024-01-31'
ORDER BY OrderDate;
```

will be fast because the data is physically ordered by `OrderDate`.

---

## 3. Non-Clustered Index

- A separate structure that stores:
  - The indexed column(s)
  - A pointer (reference) to the actual row in the table.
- A table can have **many** non-clustered indexes.
- Does **not** change the physical order of the table.

### When to use a non-clustered index

- On columns frequently used in:
  - `WHERE` conditions
  - `JOIN` conditions
  - `ORDER BY` / `GROUP BY`
- On columns that are searched often but are **not** good candidates for clustered index.

### Example

```sql
-- Non-clustered index on CustomerID
CREATE NONCLUSTERED INDEX IX_Orders_CustomerID
ON Orders (CustomerID);

-- Non-clustered index on multiple columns
CREATE NONCLUSTERED INDEX IX_Orders_CustomerID_OrderDate
ON Orders (CustomerID, OrderDate);
```

**Effect:**

```sql
SELECT * FROM Orders
WHERE CustomerID = 123;
```

- The database uses `IX_Orders_CustomerID` to quickly find all rows for `CustomerID = 123`.

---

## 4. Performance Impact

### Benefits

- **Faster reads**:
  - `SELECT` queries that use indexed columns can be much faster.
- **Efficient range scans**:
  - Clustered indexes are very good for ranges (e.g., date ranges, ID ranges).

### Costs

- **Slower writes**:
  - `INSERT`: must insert into the index structure.
  - `UPDATE`: if indexed column changes, index must be updated.
  - `DELETE`: must remove from index.
- **Storage**:
  - Non-clustered indexes take extra disk space.

---

## 5. Simple Input/Output Example

Assume this table and indexes:

```sql
CREATE TABLE Users (
    UserID INT PRIMARY KEY,          -- clustered index
    Email VARCHAR(255),
    CreatedAt DATETIME
);

CREATE NONCLUSTERED INDEX IX_Users_Email
ON Users (Email);
```

### Input: Insert data

```sql
INSERT INTO Users (UserID, Email, CreatedAt) VALUES
(1, 'alice@example.com', '2024-01-01 10:00:00'),
(2, 'bob@example.com',   '2024-01-02 11:00:00'),
(3, 'alice@example.com', '2024-01-03 12:00:00');
```

### Input: Query using non-clustered index

```sql
SELECT UserID, CreatedAt
FROM Users
WHERE Email = 'alice@example.com'
ORDER BY CreatedAt;
```

### Output (result set)

| UserID | CreatedAt           |
|--------|---------------------|
| 1      | 2024-01-01 10:00:00 |
| 3      | 2024-01-03 12:00:00 |

- The database uses `IX_Users_Email` to quickly find rows with `Email = 'alice@example.com'`.

---

## 6. Quick Comparison

| Feature              | Clustered Index                         | Non-Clustered Index                      |
|----------------------|-----------------------------------------|------------------------------------------|
| Physical row order   | Yes                                     | No                                       |
| Number per table     | Usually 1                               | Many allowed                             |
| Storage              | Table data itself                       | Separate structure                       |
| Best for             | Range queries, primary key, sorting     | Frequent filters/joins on other columns  |
| Impact on writes     | Medium                                  | Medium to high (more indexes = more cost)|

This is the core idea:  
- Use **clustered index** to define how data is stored and to speed up range and ordered queries.  
- Use **non-clustered indexes** to speed up lookups and joins on frequently queried columns.
