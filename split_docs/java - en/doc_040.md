
Here’s a concise summary of “Advanced Concurrency: Designing thread-safe classes” with key ideas and code examples.

---

## 1. What is a Thread-Safe Class?

A thread-safe class behaves correctly when accessed from multiple threads simultaneously, without additional synchronization by the caller.

Key properties:
- No data races
- No inconsistent state visible to other threads
- Operations behave as specified, regardless of interleaving

---

## 2. Strategies for Thread Safety

### 2.1 Immutable Objects

Immutable objects are inherently thread-safe:
- All fields are `final`
- No setters / no state changes after construction
- `this` reference does not escape during construction

```java
public final class Money {
    private final int amount;
    private final String currency;

    public Money(int amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }

    public int getAmount() { return amount; }
    public String getCurrency() { return currency; }

    // returns a new object instead of modifying this
    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Different currencies");
        }
        return new Money(this.amount + other.amount, this.currency);
    }
}
```

**Input/Output example:**

```java
Money m1 = new Money(10, "USD");
Money m2 = new Money(5, "USD");
Money m3 = m1.add(m2);

System.out.println(m1.getAmount()); // 10
System.out.println(m2.getAmount()); // 5
System.out.println(m3.getAmount()); // 15
```

Multiple threads can share `Money` instances safely.

---

### 2.2 Confinement

Keep data confined to:
- A single thread (thread confinement)
- A single object (encapsulation)

Example: local variables are automatically thread-confined.

```java
public void process() {
    // local variable, not shared
    int localCounter = 0;
    localCounter++;
}
```

---

### 2.3 Using Existing Thread-Safe Classes

Prefer using:
- `java.util.concurrent` collections (`ConcurrentHashMap`, `CopyOnWriteArrayList`, etc.)
- Atomic variables (`AtomicInteger`, `AtomicReference`, …)

```java
import java.util.concurrent.ConcurrentHashMap;

public class UserCache {
    private final ConcurrentHashMap<String, String> cache = new ConcurrentHashMap<>();

    public String getOrLoad(String key) {
        return cache.computeIfAbsent(key, k -> loadFromDb(k));
    }

    private String loadFromDb(String key) {
        // simulate DB
        return "value-for-" + key;
    }
}
```

**Input/Output example:**

```java
UserCache c = new UserCache();
System.out.println(c.getOrLoad("user1")); // value-for-user1
System.out.println(c.getOrLoad("user1")); // same value, from cache
```

## 3. Synchronization and State

### 3.1 Intrinsic Locks (`synchronized`)

Use `synchronized` to protect shared mutable state.

```java
public class Counter {
    private int value = 0;

    public synchronized void increment() {
        value++;
    }

    public synchronized int getValue() {
        return value;
    }
}
```

**Input/Output example (single thread):**

```java
Counter c = new Counter();
c.increment();
c.increment();
System.out.println(c.getValue()); // 2
```

Multiple threads calling `increment()` and `getValue()` will see consistent results.

---

### 3.2 Atomic Variables

For simple counters, use `AtomicInteger`:

```java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private final AtomicInteger value = new AtomicInteger(0);

    public void increment() {
        value.incrementAndGet();
    }

    public int getValue() {
        return value.get();
    }
}
```

## 4. Invariants and Compound Actions

Thread-safe classes must preserve invariants (rules that must always hold).

Example invariant: `lower <= upper`

```java
public class Range {
    private int lower;
    private int upper;

    public synchronized void setLower(int value) {
        if (value > upper) throw new IllegalArgumentException();
        lower = value;
    }

    public synchronized void setUpper(int value) {
        if (value < lower) throw new IllegalArgumentException();
        upper = value;
    }

    public synchronized int getLower() { return lower; }
    public synchronized int getUpper() { return upper; }
}
```

Both `setLower` and `setUpper` are synchronized to ensure the invariant is never broken when accessed concurrently.

## 5. Publishing and Visibility

### 5.1 Safe Publication

Make sure other threads see a fully constructed object:
- Use `final` fields
- Publish via:
  - `volatile` reference
  - `synchronized` block
  - thread-safe collections
  - static initializers

```java
public class Holder {
    private final int n;

    public Holder(int n) {
        this.n = n;
    }

    public int get() { return n; }
}

public class HolderPublisher {
    private static volatile Holder holder;

    public static void init() {
        holder = new Holder(42); // safely published
    }

    public static int read() {
        return holder.get();
    }
}
```

## 6. Designing the Class API

- Decide the **thread-safety level** and document it:
  - Fully thread-safe
  - Conditionally thread-safe (some methods require external locking)
  - Not thread-safe
- Minimize mutable state
- Keep invariants simple and well-encapsulated
- Avoid exposing internal mutable objects directly

Example of defensive copying:

```java
public class SafePoint {
    private int x, y;

    public SafePoint(int x, int y) {
        this.x = x; this.y = y;
    }

    public synchronized int[] get() {
        return new int[]{x, y}; // return copy
    }

    public synchronized void set(int x, int y) {
        this.x = x; this.y = y;
    }
}
```

---

## 7. Summary of Main Ideas

- Prefer immutability; it simplifies thread safety.
- If mutable, protect all shared state with a consistent locking strategy.
- Use high-level concurrency utilities instead of low-level `wait/notify` when possible.
- Preserve invariants across all methods that modify state.
- Ensure safe publication so other threads see a consistent object.
- Clearly document the thread-safety guarantees of your class.
