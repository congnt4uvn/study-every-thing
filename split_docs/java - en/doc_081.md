
Here’s a concise summary of **Normalization vs Denormalization** in the context of SQL & RDBMS, with examples.

---

## 1. Normalization

**Goal:**  
Reduce data redundancy and improve data integrity by splitting data into multiple related tables.

**Key ideas:**
- Data is organized into logical tables.
- Each piece of information is stored in exactly one place.
- Uses **primary keys** and **foreign keys** to relate tables.
- Common normal forms: 1NF, 2NF, 3NF (and beyond).

**Pros:**
- Less redundant data.
- Easier to maintain consistency (update in one place).
- Smaller storage footprint.
- Better for OLTP (transactional) systems.

**Cons:**
- More joins needed in queries.
- Can be slower for read-heavy, complex queries.

### Normalized Example

Suppose we store orders and customers.

**Table: `customers`**

| customer_id | name    | email              |
|-------------|---------|--------------------|
| 1           | Alice   | alice@example.com  |
| 2           | Bob     | bob@example.com    |

**Table: `orders`**

| order_id | customer_id | order_date  | total_amount |
|----------|-------------|------------|--------------|
| 101      | 1           | 2024-05-01 | 100.00       |
| 102      | 2           | 2024-05-02 | 50.00        |

**SQL: Get all orders with customer name**

```sql
SELECT o.order_id,
       c.name AS customer_name,
       o.order_date,
       o.total_amount
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;
```

**Sample Output:**

| order_id | customer_name | order_date  | total_amount |
|----------|---------------|------------|--------------|
| 101      | Alice         | 2024-05-01 | 100.00       |
| 102      | Bob           | 2024-05-02 | 50.00        |

---

## 2. Denormalization

**Goal:**  
Improve read performance by intentionally duplicating data and reducing joins.

**Key ideas:**
- Combine related data into fewer tables.
- Store redundant data (e.g., customer name inside orders).
- Often used in reporting, analytics, or high-read systems.

**Pros:**
- Fewer joins → faster reads in many cases.
- Simpler queries for reporting.

**Cons:**
- Data redundancy.
- Harder to keep data consistent (must update in multiple places).
- Larger storage.

### Denormalized Example

Combine customer info into the `orders` table:

**Table: `orders` (denormalized)**

| order_id | customer_id | customer_name | customer_email      | order_date  | total_amount |
|----------|-------------|---------------|---------------------|------------|--------------|
| 101      | 1           | Alice         | alice@example.com   | 2024-05-01 | 100.00       |
| 102      | 2           | Bob           | bob@example.com     | 2024-05-02 | 50.00        |

**SQL: Get all orders with customer name**

```sql
SELECT order_id,
       customer_name,
       order_date,
       total_amount
FROM orders;
```

**Sample Output:**

| order_id | customer_name | order_date  | total_amount |
|----------|---------------|------------|--------------|
| 101      | Alice         | 2024-05-01 | 100.00       |
| 102      | Bob           | 2024-05-02 | 50.00        |

No join is needed, but if Alice changes her email, you must update it in every row where she appears.

---

## 3. When to Use Which

- **Normalization**:  
  - OLTP systems (banking, e-commerce transactions).  
  - When data integrity and consistency are critical.  
  - When writes/updates are frequent.

- **Denormalization**:  
  - Reporting, analytics, dashboards.  
  - Read-heavy workloads where performance is more important than strict normalization.  
  - When you can tolerate some redundancy and manage it carefully.

## Simple Java + SQL Example (Conceptual)

Java code querying a normalized schema:

```java
String sql = """
    SELECT o.order_id, c.name, o.order_date, o.total_amount
    FROM orders o
    JOIN customers c ON o.customer_id = c.customer_id
    WHERE c.customer_id = ?
""";

try (PreparedStatement ps = connection.prepareStatement(sql)) {
    ps.setInt(1, 1); // Input: customer_id = 1
    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) {
            int orderId = rs.getInt("order_id");
            String name = rs.getString("name");
            Date date = rs.getDate("order_date");
            double total = rs.getDouble("total_amount");
            System.out.printf("Order %d by %s on %s: %.2f%n",
                              orderId, name, date, total);
        }
    }
}
```

**Possible Console Output:**

```
Order 101 by Alice on 2024-05-01: 100.00
```

This shows how normalized data requires a join but keeps customer data in one place.
