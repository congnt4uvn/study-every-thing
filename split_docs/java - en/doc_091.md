
Here’s a concise summary of the three principles, with simple Java-style examples.

---

## 1. DRY – Don’t Repeat Yourself

**Idea:** Avoid duplicating logic or knowledge in multiple places. Put shared logic in one place and reuse it.

### Bad (repetition)

```java
public class DiscountService {

    public double applyBlackFridayDiscount(double price) {
        double discounted = price - (price * 0.2);
        if (discounted < 0) {
            discounted = 0;
        }
        return discounted;
    }

    public double applyChristmasDiscount(double price) {
        double discounted = price - (price * 0.2);
        if (discounted < 0) {
            discounted = 0;
        }
        return discounted;
    }
}
```

Same discount logic is repeated.

### Good (DRY)

```java
public class DiscountService {

    private double applyPercentageDiscount(double price, double percent) {
        double discounted = price - (price * percent);
        return Math.max(discounted, 0);
    }

    public double applyBlackFridayDiscount(double price) {
        return applyPercentageDiscount(price, 0.2);
    }

    public double applyChristmasDiscount(double price) {
        return applyPercentageDiscount(price, 0.2);
    }
}
```

**Input/Output example:**

```java
public static void main(String[] args) {
    DiscountService service = new DiscountService();
    System.out.println(service.applyBlackFridayDiscount(100)); // 80.0
    System.out.println(service.applyChristmasDiscount(50));    // 40.0
}
```

---

## 2. KISS – Keep It Simple, Stupid

**Idea:** Prefer simple, clear solutions over clever or overly complex ones. Easier to read, test, and maintain.

### Overly complex

```java
public class EvenChecker {
    public boolean isEven(int n) {
        return ((n & 1) ^ 1) == 1;
    }
}
```

This works, but is hard to read.

### Simple (KISS)

```java
public class EvenChecker {
    public boolean isEven(int n) {
        return n % 2 == 0;
    }
}
```

**Input/Output example:**

```java
public static void main(String[] args) {
    EvenChecker checker = new EvenChecker();
    System.out.println(checker.isEven(4)); // true
    System.out.println(checker.isEven(7)); // false
}
```

---

## 3. YAGNI – You Aren’t Gonna Need It

**Idea:** Don’t implement features or abstractions “just in case.” Build only what you need now; add more when there is a real requirement.

### Bad (over‑engineering)

```java
public interface PaymentProcessor {
    void processCreditCard(String cardNumber, double amount);
    void processCrypto(String wallet, double amount);
    void processGiftCard(String code, double amount);
    // many more methods for future payment types...
}

public class MyPaymentProcessor implements PaymentProcessor {
    // All methods implemented, but only credit card is actually used today
}
```

You only need credit card payments now, but you designed for many future types.

### Good (YAGNI)

```java
public class PaymentService {

    public void processCreditCard(String cardNumber, double amount) {
        // current real requirement
        System.out.println("Charging card " + cardNumber + " amount: " + amount);
    }
}
```

Later, when you truly need crypto or gift cards, you can refactor.

**Input/Output example:**

```java
public static void main(String[] args) {
    PaymentService service = new PaymentService();
    service.processCreditCard("4111-1111-1111-1111", 99.99);
    // Output: Charging card 4111-1111-1111-1111 amount: 99.99
}
```

---

### Summary

- **DRY:** Centralize shared logic; avoid copy-paste code.
- **KISS:** Choose the simplest solution that works and is easy to understand.
- **YAGNI:** Don’t build features or abstractions until you actually need them.

These principles guide cleaner, more maintainable Java system design and architecture.
