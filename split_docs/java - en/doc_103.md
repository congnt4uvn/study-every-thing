
Here’s a concise summary of the three topics, with simple examples.

---

## 1. Database Sharding

**Idea:**  
Split one large logical database into multiple smaller physical databases (“shards”), usually by a key (e.g., user_id). Each shard holds a subset of the data.

**Why:**
- Scale horizontally (more machines instead of one big one)
- Reduce load and improve performance
- Avoid single-node storage limits

**Common sharding keys:**
- User ID
- Region / tenant
- Hash of a primary key

**Example (conceptual Java):**

```java
class ShardManager {
    private DataSource shard1;
    private DataSource shard2;

    public ShardManager(DataSource shard1, DataSource shard2) {
        this.shard1 = shard1;
        this.shard2 = shard2;
    }

    private DataSource getShardForUser(long userId) {
        // Simple modulo-based sharding
        return (userId % 2 == 0) ? shard1 : shard2;
    }

    public User getUser(long userId) throws SQLException {
        DataSource ds = getShardForUser(userId);
        try (Connection conn = ds.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "SELECT id, name FROM users WHERE id = ?")) {
            ps.setLong(1, userId);
            ResultSet rs = ps.executeQuery();
            if (rs.next()) {
                return new User(rs.getLong("id"), rs.getString("name"));
            }
            return null;
        }
    }
}
```

**Example input/output:**

- Input: `getUser(10)`  
  - 10 % 2 == 0 → goes to `shard1`
- Input: `getUser(11)`  
  - 11 % 2 == 1 → goes to `shard2`

## 2. Replication

**Idea:**  
Copy data from one database (primary/master) to one or more others (replicas/slaves). Writes go to primary; replicas receive changes asynchronously or synchronously.

**Why:**
- High availability (if primary fails, promote a replica)
- Read scalability (many replicas can serve reads)
- Data redundancy

**Types:**
- **Synchronous:** replicas confirm write before success → strong consistency, higher latency.
- **Asynchronous:** primary returns quickly, replicas catch up later → eventual consistency.

**Example (conceptual Java):**

```java
class ReplicatedDbClient {
    private DataSource primary;
    private List<DataSource> replicas;
    private AtomicInteger counter = new AtomicInteger(0);

    public ReplicatedDbClient(DataSource primary, List<DataSource> replicas) {
        this.primary = primary;
        this.replicas = replicas;
    }

    public void writeUser(User user) throws SQLException {
        try (Connection conn = primary.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "INSERT INTO users (id, name) VALUES (?, ?)")) {
            ps.setLong(1, user.getId());
            ps.setString(2, user.getName());
            ps.executeUpdate();
        }
        // Replication to replicas is handled by the DB, not by this code
    }

    public User readUserFromReplica(long id) throws SQLException {
        DataSource replica = chooseReplica();
        try (Connection conn = replica.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "SELECT id, name FROM users WHERE id = ?")) {
            ps.setLong(1, id);
            ResultSet rs = ps.executeQuery();
            if (rs.next()) {
                return new User(rs.getLong("id"), rs.getString("name"));
            }
            return null;
        }
    }

    private DataSource chooseReplica() {
        int index = counter.getAndIncrement() % replicas.size();
        return replicas.get(index);
    }
}
```

**Example input/output:**

- Input: `writeUser(new User(1, "Alice"))`  
  - Writes to primary; replicas get the data via DB replication.
- Input: `readUserFromReplica(1)`  
  - Reads from one of the replicas (e.g., replica #1).

## 3. Read/Write Splitting

**Idea:**  
Use different database nodes for reads and writes:
- Writes → primary
- Reads → replicas

Often built on top of replication.

**Why:**
- Offload read traffic from primary
- Improve read throughput and latency

**Trade-offs:**
- Replication lag: recent writes may not be visible immediately on replicas.
- Need routing logic in application or middleware.

**Example (Java):**

```java
class ReadWriteSplittingClient {
    private DataSource primary;          // for writes
    private List<DataSource> replicas;   // for reads
    private AtomicInteger counter = new AtomicInteger(0);

    public ReadWriteSplittingClient(DataSource primary, List<DataSource> replicas) {
        this.primary = primary;
        this.replicas = replicas;
    }

    public void createOrder(Order order) throws SQLException {
        try (Connection conn = primary.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "INSERT INTO orders (id, user_id, amount) VALUES (?, ?, ?)")) {
            ps.setLong(1, order.getId());
            ps.setLong(2, order.getUserId());
            ps.setBigDecimal(3, order.getAmount());
            ps.executeUpdate();
        }
    }

    public Order getOrder(long orderId) throws SQLException {
        DataSource replica = chooseReplica();
        try (Connection conn = replica.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "SELECT id, user_id, amount FROM orders WHERE id = ?")) {
            ps.setLong(1, orderId);
            ResultSet rs = ps.executeQuery();
            if (rs.next()) {
                return new Order(
                    rs.getLong("id"),
                    rs.getLong("user_id"),
                    rs.getBigDecimal("amount")
                );
            }
            return null;
        }
    }

    private DataSource chooseReplica() {
        int index = counter.getAndIncrement() % replicas.size();
        return replicas.get(index);
    }
}
```

**Example input/output:**

- Input: `createOrder(new Order(100, 1, new BigDecimal("9.99")))`  
  - Insert goes to primary.
- Input: `getOrder(100)`  
  - Select goes to one of the replicas (may not see the order immediately if replication is delayed).

---

### Key Takeaways

- **Sharding:** Split data across multiple databases to scale horizontally.
- **Replication:** Copy data from primary to replicas for availability and read scaling.
- **Read/Write Splitting:** Route writes to primary and reads to replicas to use replication effectively.

These techniques are often combined in large systems: sharded primaries, each with replicas, plus read/write splitting at the application or middleware layer.
