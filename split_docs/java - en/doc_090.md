
Here’s a concise summary of the SOLID principles in Java, with simple code examples and sample input/output where it makes sense.

---

## 1. S — Single Responsibility Principle (SRP)

**Idea:**  
A class should have only one reason to change (one responsibility).

**Bad example (too many responsibilities):**

```java
class UserService {
    public void registerUser(String username, String email) {
        // 1. Business logic
        System.out.println("Saving user to database: " + username);

        // 2. Email sending logic (another responsibility)
        System.out.println("Sending welcome email to: " + email);
    }
}
```

**Better example (split responsibilities):**

```java
class UserService {
    private final EmailService emailService;

    public UserService(EmailService emailService) {
        this.emailService = emailService;
    }

    public void registerUser(String username, String email) {
        // Business logic
        System.out.println("Saving user to database: " + username);
        emailService.sendWelcomeEmail(email);
    }
}

class EmailService {
    public void sendWelcomeEmail(String email) {
        System.out.println("Sending welcome email to: " + email);
    }
}
```

**Sample usage / output:**

```java
public class Main {
    public static void main(String[] args) {
        EmailService emailService = new EmailService();
        UserService userService = new UserService(emailService);

        userService.registerUser("alice", "alice@example.com");
    }
}
```

**Output:**
```
Saving user to database: alice
Sending welcome email to: alice@example.com
```

## 2. O — Open/Closed Principle (OCP)

**Idea:**  
Software entities should be **open for extension** but **closed for modification**.  
You should be able to add new behavior without changing existing code.

**Example: discount calculation**

```java
interface DiscountPolicy {
    double applyDiscount(double price);
}

class NoDiscount implements DiscountPolicy {
    public double applyDiscount(double price) {
        return price;
    }
}

class ChristmasDiscount implements DiscountPolicy {
    public double applyDiscount(double price) {
        return price * 0.8; // 20% off
    }
}

class Order {
    private final DiscountPolicy discountPolicy;

    public Order(DiscountPolicy discountPolicy) {
        this.discountPolicy = discountPolicy;
    }

    public double finalPrice(double basePrice) {
        return discountPolicy.applyDiscount(basePrice);
    }
}
```

To add a new discount (e.g., `BlackFridayDiscount`), you create a new class implementing `DiscountPolicy` without modifying `Order`.

**Sample usage / output:**

```java
public class Main {
    public static void main(String[] args) {
        Order order1 = new Order(new NoDiscount());
        Order order2 = new Order(new ChristmasDiscount());

        System.out.println(order1.finalPrice(100)); // 100.0
        System.out.println(order2.finalPrice(100)); // 80.0
    }
}
```

**Output:**
```
100.0
80.0
```

## 3. L — Liskov Substitution Principle (LSP)

**Idea:**  
Subclasses should be substitutable for their base classes without breaking behavior.  
If `B` extends `A`, you should be able to use `B` wherever `A` is expected.

**Good example:**

```java
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }

    public int area() { return width * height; }
}

class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width;
    }

    @Override
    public void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}
```

This classic example actually shows a **violation** of LSP in real life (because `Square` changes expected behavior of `Rectangle` setters).  
A better design would avoid this inheritance and use composition or separate interfaces.

**LSP-friendly example:**

```java
interface Shape {
    int area();
}

class RectangleShape implements Shape {
    private final int width;
    private final int height;

    public RectangleShape(int width, int height) {
        this.width = width;
        this.height = height;
    }

    public int area() { return width * height; }
}

class SquareShape implements Shape {
    private final int side;

    public SquareShape(int side) {
        this.side = side;
    }

    public int area() { return side * side; }
}
```

Now any `Shape` can be used interchangeably:

```java
public class Main {
    public static void main(String[] args) {
        Shape s1 = new RectangleShape(2, 3);
        Shape s2 = new SquareShape(4);

        System.out.println(s1.area()); // 6
        System.out.println(s2.area()); // 16
    }
}
```

**Output:**
```
6
16
```

## 4. I — Interface Segregation Principle (ISP)

**Idea:**  
Clients should not be forced to depend on methods they do not use.  
Prefer many small, specific interfaces over one large, “fat” interface.

**Bad example (fat interface):**

```java
interface Worker {
    void work();
    void eat();
}

class Robot implements Worker {
    public void work() {
        System.out.println("Robot working");
    }

    public void eat() {
        // Robot doesn't eat, but must implement this
        throw new UnsupportedOperationException("Robots don't eat");
    }
}
```

**Better example (segregated interfaces):**

```java
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

class Human implements Workable, Eatable {
    public void work() {
        System.out.println("Human working");
    }

    public void eat() {
        System.out.println("Human eating");
    }
}

class Robot implements Workable {
    public void work() {
        System.out.println("Robot working");
    }
}
```

**Sample usage / output:**

```java
public class Main {
    public static void main(String[] args) {
        Workable worker1 = new Human();
        Workable worker2 = new Robot();

        worker1.work();
        worker2.work();
    }
}
```

**Output:**
```
Human working
Robot working
```

## 5. D — Dependency Inversion Principle (DIP)

**Idea:**  
- High-level modules should not depend on low-level modules. Both should depend on abstractions.  
- Depend on interfaces/abstractions, not concrete implementations.

**Bad example (high-level depends on concrete class):**

```java
class MySQLDatabase {
    public void save(String data) {
        System.out.println("Saving to MySQL: " + data);
    }
}

class ReportService {
    private final MySQLDatabase database = new MySQLDatabase();

    public void generateReport(String data) {
        // ...
        database.save(data);
    }
}
```

**Better example (depend on abstraction):**

```java
interface Database {
    void save(String data);
}

class MySQLDatabase implements Database {
    public void save(String data) {
        System.out.println("Saving to MySQL: " + data);
    }
}

class InMemoryDatabase implements Database {
    public void save(String data) {
        System.out.println("Saving to in-memory DB: " + data);
    }
}

class ReportService {
    private final Database database;

    public ReportService(Database database) {
        this.database = database;
    }

    public void generateReport(String data) {
        // ...
        database.save(data);
    }
}
```

**Sample usage / output:**

```java
public class Main {
    public static void main(String[] args) {
        Database db1 = new MySQLDatabase();
        Database db2 = new InMemoryDatabase();

        ReportService report1 = new ReportService(db1);
        ReportService report2 = new ReportService(db2);

        report1.generateReport("Sales data");
        report2.generateReport("Test data");
    }
}
```

**Output:**
```
Saving to MySQL: Sales data
Saving to in-memory DB: Test data
```

---

### Overall Summary

- **S**: Each class has one responsibility.  
- **O**: Add new behavior by extending, not modifying existing code.  
- **L**: Subtypes must be usable wherever their base type is expected.  
- **I**: Prefer small, focused interfaces.  
- **D**: High-level code depends on abstractions, not concrete implementations.

These principles help make Java systems easier to maintain, test, and extend.
