
Here’s a concise summary and some illustrative Java examples around:

**“Advanced Concurrency: Immutability as a concurrency strategy”**

---

## Key Ideas

1. **Immutability**  
   - An immutable object’s state cannot change after construction.  
   - All fields are `final` and set only in the constructor.  
   - No setters or methods that modify internal state.

2. **Why it helps with concurrency**  
   - Immutable objects are **inherently thread-safe**:  
     - No race conditions on internal state.  
     - No need for synchronization (`synchronized`, locks) to read them.  
   - They can be **freely shared** between threads without defensive copying.

3. **Typical pattern**  
   - Use immutable objects for:
     - Value objects (e.g., money, coordinates, configuration, messages).  
     - Data passed between threads (e.g., tasks, results, events).  
   - When “changing” state, create a **new immutable instance** instead of mutating the old one.

4. **Trade-offs**  
   - Pros:
     - Simpler reasoning about correctness.  
     - Fewer concurrency bugs (no data races, no visibility issues).  
   - Cons:
     - More object allocations (can affect performance in very hot paths).  
   - Often worth it for clarity and safety; JVM optimizations (escape analysis, etc.) can mitigate costs.

## Basic Immutable Class Example

```java
public final class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    // No setters

    public int getX() {
        return x;
    }

    public int getY() {
        return y;
    }

    // "Modify" by creating a new instance
    public Point move(int dx, int dy) {
        return new Point(this.x + dx, this.y + dy);
    }

    @Override
    public String toString() {
        return "Point(" + x + ", " + y + ")";
    }
}
```

### Example usage (single-threaded, but thread-safe by design)

```java
public class Main {
    public static void main(String[] args) {
        Point p1 = new Point(1, 2);
        Point p2 = p1.move(3, 4); // returns new Point

        System.out.println("p1 = " + p1);
        System.out.println("p2 = " + p2);
    }
}
```

**Output:**
```
p1 = Point(1, 2)
p2 = Point(4, 6)
```

`p1` never changes; `p2` is a new object.

## Immutability in a Concurrent Context

Imagine multiple threads reading a shared configuration object.

### Mutable (problematic) version

```java
public class MutableConfig {
    private String mode;

    public MutableConfig(String mode) {
        this.mode = mode;
    }

    public String getMode() {
        return mode;
    }

    public void setMode(String mode) {
        this.mode = mode; // can change at any time
    }
}
```

- If multiple threads read and write `mode`, you need synchronization or `volatile` to avoid visibility issues and race conditions.

### Immutable version

```java
public final class ImmutableConfig {
    private final String mode;

    public ImmutableConfig(String mode) {
        this.mode = mode;
    }

    public String getMode() {
        return mode;
    }

    // No setter
}
```

Now you can safely share `ImmutableConfig` across threads:

```java
import java.util.concurrent.*;

public class ConfigDemo {
    private static volatile ImmutableConfig config = new ImmutableConfig("DEFAULT");

    public static void main(String[] args) throws InterruptedException {
        ExecutorService pool = Executors.newFixedThreadPool(4);

        // Reader tasks
        Runnable reader = () -> {
            for (int i = 0; i < 3; i++) {
                ImmutableConfig c = config; // safe read
                System.out.println(Thread.currentThread().getName() +
                        " sees mode = " + c.getMode());
                try { Thread.sleep(100); } catch (InterruptedException ignored) {}
            }
        };

        // Writer task: "update" config by replacing the reference
        Runnable writer = () -> {
            try { Thread.sleep(150); } catch (InterruptedException ignored) {}
            config = new ImmutableConfig("UPDATED");
            System.out.println("Writer updated config to UPDATED");
        };

        pool.submit(reader);
        pool.submit(reader);
        pool.submit(writer);

        pool.shutdown();
        pool.awaitTermination(5, TimeUnit.SECONDS);
    }
}
```

**Typical output (order may vary):**
```
pool-1-thread-1 sees mode = DEFAULT
pool-1-thread-2 sees mode = DEFAULT
pool-1-thread-1 sees mode = DEFAULT
Writer updated config to UPDATED
pool-1-thread-2 sees mode = UPDATED
pool-1-thread-1 sees mode = UPDATED
pool-1-thread-2 sees mode = UPDATED
```

Key points:
- The **object itself** is immutable.  
- Concurrency is handled by **replacing the reference** (`config = new ImmutableConfig(...)`), not by mutating the object.  
- Only the reference needs to be safely published (here via `volatile`).

## Functional-Style Updates with Immutability

For more complex state, you can use a “copy with changes” pattern:

```java
public final class UserProfile {
    private final String name;
    private final int age;
    private final String city;

    public UserProfile(String name, int age, String city) {
        this.name = name;
        this.age = age;
        this.city = city;
    }

    public UserProfile withAge(int newAge) {
        return new UserProfile(this.name, newAge, this.city);
    }

    public UserProfile withCity(String newCity) {
        return new UserProfile(this.name, this.age, newCity);
    }

    @Override
    public String toString() {
        return name + " (" + age + ", " + city + ")";
    }
}
```

### Example usage

```java
public class UserDemo {
    public static void main(String[] args) {
        UserProfile u1 = new UserProfile("Alice", 30, "Seoul");
        UserProfile u2 = u1.withAge(31);
        UserProfile u3 = u2.withCity("Busan");

        System.out.println("u1 = " + u1);
        System.out.println("u2 = " + u2);
        System.out.println("u3 = " + u3);
    }
}
```

**Output:**
```
u1 = Alice (30, Seoul)
u2 = Alice (31, Seoul)
u3 = Alice (31, Busan)
```

Each “change” creates a new immutable object, which is safe to share across threads.

---

## Summary

- Immutability is a powerful concurrency strategy in Java:
  - Make objects immutable (final fields, no setters, no internal mutation).
  - Share immutable objects freely between threads without locks.
  - “Update” state by creating new instances and safely publishing references.
- This reduces the need for explicit synchronization and makes concurrent code easier to reason about and less error-prone.
