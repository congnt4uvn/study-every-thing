
Here’s a concise summary of the core ideas behind **latency** and **throughput** in distributed systems, with Java-style examples.

---

## Key Concepts

### Latency
- **Definition**: How long a single operation takes from start to finish.
- **Unit**: Usually milliseconds (ms) or microseconds (µs).
- **User perspective**: “How long does this request take?”

### Throughput
- **Definition**: How many operations can be completed per unit of time.
- **Unit**: Requests per second (RPS), transactions per second (TPS), messages per second, etc.
- **System perspective**: “How many requests can we handle per second?”

### Relationship
- Lower latency often helps throughput, but:
  - You can have **low latency but low throughput** (fast per request, but few parallel requests).
  - You can have **high throughput but higher latency** (many requests in parallel, each waits in a queue).
- Tuning systems usually involves **trade-offs** between latency and throughput.

## Simple Java Example: Measuring Latency

```java
public class LatencyExample {
    public static void main(String[] args) {
        long start = System.nanoTime();
        
        // Simulate a single request (e.g., DB call, remote service)
        simulateRequest();
        
        long end = System.nanoTime();
        long latencyMicros = (end - start) / 1_000; // convert ns to µs
        
        System.out.println("Latency: " + latencyMicros + " µs");
    }

    private static void simulateRequest() {
        try {
            // Simulate 50 ms latency
            Thread.sleep(50);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

**Sample output:**
```text
Latency: 50000 µs
```

## Simple Java Example: Measuring Throughput

```java
import java.util.concurrent.*;

public class ThroughputExample {
    public static void main(String[] args) throws InterruptedException {
        int totalRequests = 100;
        int threads = 10;
        
        ExecutorService executor = Executors.newFixedThreadPool(threads);
        long start = System.nanoTime();
        
        CountDownLatch latch = new CountDownLatch(totalRequests);
        
        for (int i = 0; i < totalRequests; i++) {
            executor.submit(() -> {
                simulateRequest();
                latch.countDown();
            });
        }
        
        latch.await(); // wait for all tasks to finish
        long end = System.nanoTime();
        executor.shutdown();
        
        double seconds = (end - start) / 1_000_000_000.0;
        double throughput = totalRequests / seconds;
        
        System.out.println("Total time: " + seconds + " s");
        System.out.println("Throughput: " + throughput + " requests/sec");
    }

    private static void simulateRequest() {
        try {
            // Simulate 50 ms work per request
            Thread.sleep(50);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

**Sample output (will vary):**
```text
Total time: 0.55 s
Throughput: 181.8 requests/sec
```

---

## Practical Takeaways

- **Latency** matters for user experience (page load time, API response time).
- **Throughput** matters for system capacity (how many users/requests you can handle).
- In distributed systems, network delays, serialization, and remote calls affect both.
- Design choices (batching, concurrency, queueing, caching) often trade latency for throughput or vice versa.
