
Here’s a concise summary of the three structural patterns, with Java-style examples and sample I/O where it makes sense.

---

## 1. Adapter Pattern

**Goal:**  
Make incompatible interfaces work together by wrapping one object with another that exposes the expected interface.

**When to use:**
- You have existing code (legacy or third‑party) that doesn’t match the interface your system expects.
- You want to reuse a class without modifying it.

### Example

Suppose your system expects a `MediaPlayer` interface, but you have an existing `AdvancedPlayer` class with a different API.

```java
// Target interface (what the client expects)
interface MediaPlayer {
    void play(String fileName);
}

// Existing class with incompatible interface
class AdvancedPlayer {
    public void playFile(String filePath) {
        System.out.println("Playing file: " + filePath);
    }
}

// Adapter: makes AdvancedPlayer look like a MediaPlayer
class MediaPlayerAdapter implements MediaPlayer {
    private final AdvancedPlayer advancedPlayer;

    public MediaPlayerAdapter(AdvancedPlayer advancedPlayer) {
        this.advancedPlayer = advancedPlayer;
    }

    @Override
    public void play(String fileName) {
        // Translate call
        advancedPlayer.playFile(fileName);
    }
}

// Client code
public class AdapterDemo {
    public static void main(String[] args) {
        MediaPlayer player = new MediaPlayerAdapter(new AdvancedPlayer());
        player.play("song.mp3");
    }
}
```

**Output:**
```text
Playing file: song.mp3
```

## 2. Facade Pattern

**Goal:**  
Provide a simple, unified interface to a complex subsystem.

**When to use:**
- You have many classes with complex interactions.
- You want to offer a “simple API” for common use cases.
- You want to decouple client code from subsystem details.

### Example

Imagine a video conversion library with many steps. The Facade hides the complexity.

```java
// Complex subsystem (simplified)
class VideoFile {
    String name;
    public VideoFile(String name) { this.name = name; }
}

class Codec {}
class MPEG4Codec extends Codec {}
class OggCodec extends Codec {}

class CodecFactory {
    public static Codec extract(VideoFile file) {
        System.out.println("Extracting codec from " + file.name);
        return new MPEG4Codec();
    }
}

class BitrateReader {
    public static VideoFile read(VideoFile file, Codec codec) {
        System.out.println("Reading file with codec");
        return file;
    }
    public static VideoFile convert(VideoFile buffer, Codec codec) {
        System.out.println("Converting file to target codec");
        return buffer;
    }
}

class AudioMixer {
    public VideoFile fix(VideoFile result) {
        System.out.println("Fixing audio...");
        return result;
    }
}

// Facade
class VideoConverterFacade {
    public VideoFile convertVideo(String fileName, String format) {
        VideoFile file = new VideoFile(fileName);
        Codec sourceCodec = CodecFactory.extract(file);
        Codec destinationCodec = "ogg".equalsIgnoreCase(format)
                ? new OggCodec()
                : new MPEG4Codec();

        VideoFile buffer = BitrateReader.read(file, sourceCodec);
        VideoFile intermediateResult = BitrateReader.convert(buffer, destinationCodec);
        VideoFile finalResult = new AudioMixer().fix(intermediateResult);

        System.out.println("Conversion finished.");
        return finalResult;
    }
}

// Client code
public class FacadeDemo {
    public static void main(String[] args) {
        VideoConverterFacade converter = new VideoConverterFacade();
        VideoFile result = converter.convertVideo("movie.avi", "ogg");
        System.out.println("Result file: " + result.name);
    }
}
```

**Possible output:**
```text
Extracting codec from movie.avi
Reading file with codec
Converting file to target codec
Fixing audio...
Conversion finished.
Result file: movie.avi
```

## 3. Decorator Pattern

**Goal:**  
Add responsibilities/behavior to objects dynamically, without changing their class, by wrapping them in decorator objects.

**When to use:**
- You want to add features at runtime (e.g., logging, compression, encryption).
- You want to avoid a large inheritance hierarchy of subclasses.

### Example

Base `DataSource` can be decorated with compression and encryption.

```java
// Component
interface DataSource {
    void writeData(String data);
    String readData();
}

// Concrete component
class FileDataSource implements DataSource {
    private StringBuilder storage = new StringBuilder();

    @Override
    public void writeData(String data) {
        storage.append(data);
    }

    @Override
    public String readData() {
        return storage.toString();
    }
}

// Base decorator
abstract class DataSourceDecorator implements DataSource {
    protected DataSource wrappee;

    public DataSourceDecorator(DataSource source) {
        this.wrappee = source;
    }

    @Override
    public void writeData(String data) {
        wrappee.writeData(data);
    }

    @Override
    public String readData() {
        return wrappee.readData();
    }
}

// Concrete decorator: adds "encryption"
class EncryptionDecorator extends DataSourceDecorator {
    public EncryptionDecorator(DataSource source) {
        super(source);
    }

    @Override
    public void writeData(String data) {
        String encrypted = "ENC(" + data + ")";
        super.writeData(encrypted);
    }

    @Override
    public String readData() {
        String data = super.readData();
        // Very fake "decryption"
        if (data.startsWith("ENC(") && data.endsWith(")")) {
            return data.substring(4, data.length() - 1);
        }
        return data;
    }
}

// Concrete decorator: adds "compression"
class CompressionDecorator extends DataSourceDecorator {
    public CompressionDecorator(DataSource source) {
        super(source);
    }

    @Override
    public void writeData(String data) {
        String compressed = "ZIP[" + data + "]";
        super.writeData(compressed);
    }

    @Override
    public String readData() {
        String data = super.readData();
        // Fake "decompression"
        if (data.startsWith("ZIP[") && data.endsWith("]")) {
            return data.substring(4, data.length() - 1);
        }
        return data;
    }
}

// Client code
public class DecoratorDemo {
    public static void main(String[] args) {
        DataSource source = new FileDataSource();

        // Wrap with decorators: first encryption, then compression
        DataSource decorated = new CompressionDecorator(
                                   new EncryptionDecorator(source));

        decorated.writeData("Hello World");

        // Reading through the same decorator chain
        String result = decorated.readData();
        System.out.println("Final read: " + result);

        // Reading raw stored data (bypassing decorators)
        System.out.println("Raw stored: " + source.readData());
    }
}
```

**Possible output:**
```text
Final read: Hello World
Raw stored: ZIP[ENC(Hello World)]
```

---

### Core Ideas Summary

- **Adapter**:  
  - Converts one interface to another.  
  - Focus: compatibility between existing classes and new interfaces.

- **Facade**:  
  - Provides a simple interface to a complex subsystem.  
  - Focus: simplification and hiding complexity.

- **Decorator**:  
  - Wraps objects to add behavior dynamically.  
  - Focus: flexible extension of functionality without subclass explosion.
