
Here’s a concise summary of the key ideas about Java exceptions and error handling, with examples.

---

## 1. Wrapping Exceptions

**Idea:** When you catch a low-level exception (e.g., from JDBC, IO, etc.), wrap it in a higher-level, domain-specific exception that makes sense to your application.

**Why:**
- Preserves the original cause (stack trace).
- Gives more meaningful context to callers.
- Decouples your code from low-level exception types.

**Example:**

```java
// Domain-specific exception
public class UserServiceException extends RuntimeException {
    public UserServiceException(String message, Throwable cause) {
        super(message, cause);
    }
}

// Service code that wraps a low-level exception
public class UserService {
    private final UserRepository repo;

    public UserService(UserRepository repo) {
        this.repo = repo;
    }

    public User getUserById(long id) {
        try {
            return repo.findById(id);
        } catch (SQLException e) {
            // Wrap low-level exception
            throw new UserServiceException("Failed to load user with id " + id, e);
        }
    }
}
```

**Input/Output behavior:**

- Input: `getUserById(42)`  
- If DB is fine → returns `User` object.  
- If DB throws `SQLException` → caller sees `UserServiceException` with message `"Failed to load user with id 42"` and cause `SQLException`.

## 2. Custom Exceptions

**Idea:** Define your own exception types to represent domain or application-specific error conditions.

**Why:**
- Makes code self-documenting.
- Allows callers to catch specific problems.
- Can carry extra data (e.g., error codes, fields).

**Example:**

```java
// Custom checked exception
public class InsufficientBalanceException extends Exception {
    private final double currentBalance;
    private final double requiredAmount;

    public InsufficientBalanceException(double currentBalance, double requiredAmount) {
        super("Insufficient balance: have " + currentBalance + ", need " + requiredAmount);
        this.currentBalance = currentBalance;
        this.requiredAmount = requiredAmount;
    }

    public double getCurrentBalance() {
        return currentBalance;
    }

    public double getRequiredAmount() {
        return requiredAmount;
    }
}

// Usage in a domain method
public class Account {
    private double balance;

    public Account(double initialBalance) {
        this.balance = initialBalance;
    }

    public void withdraw(double amount) throws InsufficientBalanceException {
        if (amount > balance) {
            throw new InsufficientBalanceException(balance, amount);
        }
        balance -= amount;
    }

    public double getBalance() {
        return balance;
    }
}
```

**Input/Output behavior:**

```java
public static void main(String[] args) {
    Account account = new Account(100.0);

    try {
        account.withdraw(150.0);
    } catch (InsufficientBalanceException e) {
        System.out.println(e.getMessage());
        System.out.println("Current: " + e.getCurrentBalance());
        System.out.println("Required: " + e.getRequiredAmount());
    }
}
```

**Console output:**

```
Insufficient balance: have 100.0, need 150.0
Current: 100.0
Required: 150.0
```

## 3. Avoid Swallowing Exceptions

**Idea:** Don’t catch an exception and then ignore it (or just print a stack trace) without handling it properly or rethrowing.

**Why:**
- Hides bugs and makes debugging very hard.
- Application may continue in an inconsistent state.
- Logs may be missing or misleading.

**Bad example (swallowing):**

```java
public void processFile(String path) {
    try {
        List<String> lines = Files.readAllLines(Path.of(path));
        // ... process lines
    } catch (IOException e) {
        // BAD: exception is swallowed, no log, no rethrow
    }
}
```

**Better example (log and rethrow or handle):**

```java
private static final Logger logger = Logger.getLogger(MyService.class.getName());

public void processFile(String path) {
    try {
        List<String> lines = Files.readAllLines(Path.of(path));
        // ... process lines
    } catch (IOException e) {
        logger.severe("Failed to read file: " + path + " - " + e.getMessage());
        // Option 1: wrap and rethrow
        throw new RuntimeException("Could not process file: " + path, e);
        // Option 2: return a fallback value or notify caller in some defined way
    }
}
```

**Input/Output behavior:**

- Input: `processFile("missing.txt")`  
- Output:  
  - Logs: `Failed to read file: missing.txt - <IO error message>`  
  - Throws `RuntimeException` with cause `IOException`.

---

## Summary of Best Practices

1. **Wrap low-level exceptions** in higher-level, meaningful exceptions while preserving the cause.
2. **Use custom exceptions** to represent domain-specific errors and make APIs clearer.
3. **Never swallow exceptions** silently; at minimum, log them and usually rethrow or handle them in a defined way.
