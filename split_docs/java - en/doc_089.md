
Here’s a concise summary of CAP theorem basics in the context of NoSQL and distributed systems.

---

## 1. What is the CAP Theorem?

In a distributed system, you can’t fully guarantee all three of these at the same time:

- **C – Consistency**  
  Every read receives the most recent write (or an error).  
  All nodes see the same data at the same time.

- **A – Availability**  
  Every request receives a non-error response, without guaranteeing it’s the latest data.

- **P – Partition Tolerance**  
  The system continues to operate even if there are network failures that split the cluster into parts (partitions).

**CAP theorem:**  
Under a network partition, a distributed system must choose between **Consistency** and **Availability**.

---

## 2. What does this mean in practice?

When a partition happens (nodes can’t talk to each other):

- **CP systems (Consistency + Partition tolerance)**  
  Prefer correct, up-to-date data over availability.  
  Some requests may fail or block.

- **AP systems (Availability + Partition tolerance)**  
  Prefer always responding over strict consistency.  
  Some reads may see stale or conflicting data.

You can’t “turn off” partition tolerance in a real distributed system; network failures are inevitable. So the real trade-off is **C vs A when P happens**.

---

## 3. NoSQL Databases and CAP

Typical tendencies (simplified):

- **CP-oriented**: HBase, MongoDB (in some configurations), etc.  
- **AP-oriented**: Cassandra, Riak, Dynamo-style systems, etc.

Many modern systems offer **tunable consistency**: you can choose per operation how strongly consistent you want it to be.

---

## 4. Simple Code Example (Simulated)

Below is a very simplified Java-like example to illustrate **AP behavior** (eventual consistency) vs **CP behavior** (strong consistency) in a key-value store.

### Example: Eventual Consistency (AP-style)

```java
import java.util.*;

class Node {
    private Map<String, String> data = new HashMap<>();

    public void write(String key, String value) {
        data.put(key, value);
    }

    public String read(String key) {
        return data.get(key);
    }

    public void syncFrom(Node other) {
        // simple "last write wins" merge
        data.putAll(other.data);
    }
}

public class EventualConsistencyDemo {
    public static void main(String[] args) {
        Node nodeA = new Node();
        Node nodeB = new Node();

        // Network partition: A and B cannot sync
        nodeA.write("user:1", "Alice_v1");
        nodeB.write("user:1", "Alice_v2");

        // Reads during partition
        System.out.println("NodeA read: " + nodeA.read("user:1"));
        System.out.println("NodeB read: " + nodeB.read("user:1"));

        // Later, partition heals and nodes sync
        nodeA.syncFrom(nodeB);
        nodeB.syncFrom(nodeA);

        // After sync, both see the same value
        System.out.println("NodeA read after sync: " + nodeA.read("user:1"));
        System.out.println("NodeB read after sync: " + nodeB.read("user:1"));
    }
}
```

**Possible Output:**

```
NodeA read: Alice_v1
NodeB read: Alice_v2
NodeA read after sync: Alice_v2
NodeB read after sync: Alice_v2
```

- During the partition, **availability** is preserved (both nodes answer), but **consistency** is broken.
- After synchronization, the system becomes consistent again (eventual consistency).

---

### Example: Strong Consistency (CP-style, simplified)

```java
import java.util.*;

class StrongStore {
    private Map<String, String> data = new HashMap<>();
    private boolean partitioned = false;

    public void setPartitioned(boolean p) {
        this.partitioned = p;
    }

    public void write(String key, String value) {
        if (partitioned) {
            throw new RuntimeException("Write failed: partition, cannot ensure consistency");
        }
        data.put(key, value);
    }

    public String read(String key) {
        if (partitioned) {
            throw new RuntimeException("Read failed: partition, cannot ensure consistency");
        }
        return data.get(key);
    }
}

public class StrongConsistencyDemo {
    public static void main(String[] args) {
        StrongStore store = new StrongStore();

        store.write("user:1", "Alice_v1");
        System.out.println("Read before partition: " + store.read("user:1"));

        // Simulate partition
        store.setPartitioned(true);

        try {
            store.write("user:1", "Alice_v2");
        } catch (RuntimeException e) {
            System.out.println(e.getMessage());
        }

        try {
            System.out.println("Read during partition: " + store.read("user:1"));
        } catch (RuntimeException e) {
            System.out.println(e.getMessage());
        }
    }
}
```

**Possible Output:**

```
Read before partition: Alice_v1
Write failed: partition, cannot ensure consistency
Read failed: partition, cannot ensure consistency
```

- The system **refuses** to serve reads/writes during partition to keep data consistent.
- This is **CP**: it sacrifices availability when partitions occur.

---

## 5. Key Takeaways

- CAP theorem applies to distributed systems, including NoSQL databases.
- Under network partitions, you must choose between:
  - **Consistency (C)**: always correct, up-to-date data
  - **Availability (A)**: system always responds
- Real systems are **CP** or **AP** under partition; many offer **tunable consistency**.
- NoSQL designs often explicitly choose a point on this CAP spectrum depending on use case.
