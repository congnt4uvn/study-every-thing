
Here’s a concise summary of the topic “Exceptions & Error Handling: When to use exceptions vs return types” with Java-focused examples.

---

## Key Ideas

1. **Use exceptions for truly exceptional situations**
   - Situations that are *unexpected*, *rare*, or *cannot be handled locally*.
   - Examples: I/O failures, database connection issues, invalid state, programming errors.

2. **Use return types for expected, normal outcomes**
   - Situations that are part of normal control flow.
   - Examples: “not found” results, empty lists, validation failures, optional values.

3. **Exceptions are for control flow across layers**
   - Good when an error must be propagated up the call stack to a higher-level handler.
   - Bad when used for routine branching (e.g., using exceptions instead of `if`).

4. **Return types make success/failure explicit**
   - Using `Optional<T>`, custom result types, or status codes can make the API clearer.
   - Callers are forced to handle the “no result” or “failure” case.

5. **Performance considerations**
   - Throwing and catching exceptions is more expensive than simple return checks.
   - Don’t use exceptions in tight loops or for very frequent, expected conditions.

---

## When to Use Exceptions

- The operation *should* normally succeed; failure is unusual.
- The caller cannot reasonably recover at the point of failure.
- You want to centralize error handling (e.g., at a higher layer).
- The error represents a violation of preconditions or invariants.

### Example: Using Exceptions

```java
public int parsePort(String value) {
    try {
        return Integer.parseInt(value);
    } catch (NumberFormatException e) {
        throw new IllegalArgumentException("Invalid port: " + value, e);
    }
}

// Input/Output example:
public static void main(String[] args) {
    System.out.println(parsePort("8080"));  // Output: 8080
    System.out.println(parsePort("abc"));   // Throws IllegalArgumentException
}
```

---

## When to Use Return Types

- “Failure” is a normal, expected outcome.
- You want to avoid exceptions in hot paths.
- You want the API to clearly show that a value may be absent or invalid.

### Example 1: Using `Optional` Instead of Exception

```java
import java.util.Optional;

public Optional<User> findUserById(String id) {
    // Suppose this queries a database
    User user = database.get(id);
    return Optional.ofNullable(user);
}

// Input/Output example:
public static void main(String[] args) {
    Optional<User> user1 = findUserById("123");
    user1.ifPresent(u -> System.out.println("Found: " + u.getName()));

    Optional<User> user2 = findUserById("999");
    System.out.println(user2.isPresent());  // Output: false (no exception)
}
```

### Example 2: Custom Result Type

```java
public class ValidationResult {
    private final boolean valid;
    private final String message;

    private ValidationResult(boolean valid, String message) {
        this.valid = valid;
        this.message = message;
    }

    public static ValidationResult ok() {
        return new ValidationResult(true, "OK");
    }

    public static ValidationResult error(String message) {
        return new ValidationResult(false, message);
    }

    public boolean isValid() { return valid; }
    public String getMessage() { return message; }
}

public ValidationResult validateAge(int age) {
    if (age < 0) {
        return ValidationResult.error("Age cannot be negative");
    }
    if (age > 150) {
        return ValidationResult.error("Age is unrealistically high");
    }
    return ValidationResult.ok();
}

// Input/Output example:
public static void main(String[] args) {
    ValidationResult r1 = validateAge(25);
    System.out.println(r1.isValid());      // true
    System.out.println(r1.getMessage());   // "OK"

    ValidationResult r2 = validateAge(-5);
    System.out.println(r2.isValid());      // false
    System.out.println(r2.getMessage());   // "Age cannot be negative"
}
```

---

## Practical Guidelines

- **Expected condition?** Use return types (`Optional`, custom result, empty collections).
- **Unexpected or unrecoverable?** Use exceptions.
- **Public API design:** Prefer clear return types for common “no result” cases; use exceptions for invalid usage or system failures.
- **Don’t use exceptions for normal branching** (e.g., parsing a format you expect to fail often—check first if possible).

This balance keeps your Java code clearer, more efficient, and easier to maintain.
