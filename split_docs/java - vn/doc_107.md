
Dưới đây là tóm tắt ngắn gọn về **mô hình nhất quán (consistency models)** trong hệ thống phân tán, kèm ví dụ Java minh họa.

---

## 1. Nhất quán là gì trong hệ thống phân tán?

Trong hệ thống phân tán (nhiều node / server), **nhất quán** mô tả cách mà các node “nhìn thấy” dữ liệu sau khi có các thao tác đọc/ghi:

- Khi một client ghi dữ liệu, **khi nào** các client khác sẽ thấy giá trị mới?
- Các lần đọc từ các node khác nhau có **thấy cùng một giá trị** hay không?

---

## 2. Một số mô hình nhất quán phổ biến

### 2.1. Strong Consistency (Nhất quán mạnh)

- Sau khi một thao tác ghi hoàn tất, **mọi lần đọc sau đó (từ bất kỳ node nào)** đều thấy giá trị mới.
- Trông giống như hệ thống chỉ có **một bản sao dữ liệu**.

**Ưu điểm**: Dễ suy luận, logic đơn giản.  
**Nhược điểm**: Độ trễ cao, khó mở rộng, kém chịu lỗi hơn.

**Ví dụ Java (mô phỏng)**

```java
import java.util.concurrent.atomic.AtomicInteger;

public class StrongConsistencyExample {
    private static final AtomicInteger value = new AtomicInteger(0);

    public static void main(String[] args) throws InterruptedException {
        Thread writer = new Thread(() -> {
            value.set(42); // ghi giá trị mới
            System.out.println("Writer: set value = 42");
        });

        Thread reader = new Thread(() -> {
            // giả sử chỉ chạy sau khi writer hoàn tất
            int v = value.get();
            System.out.println("Reader: read value = " + v);
        });

        writer.start();
        writer.join(); // đảm bảo writer xong trước
        reader.start();
    }
}
```

**Output (mô phỏng):**
```
Writer: set value = 42
Reader: read value = 42
```

Ở đây, `AtomicInteger` + `join()` đảm bảo reader luôn thấy giá trị mới → giống strong consistency.

### 2.2. Eventual Consistency (Nhất quán cuối cùng)

- Sau khi ghi, các node **có thể tạm thời thấy giá trị khác nhau**.
- Nhưng nếu **không có thêm ghi mới**, cuối cùng tất cả node sẽ hội tụ về **cùng một giá trị**.

**Ưu điểm**: Hiệu năng cao, dễ mở rộng, phù hợp hệ thống phân tán lớn (NoSQL, cache phân tán).  
**Nhược điểm**: Ứng dụng phải chấp nhận đọc dữ liệu “cũ” trong một khoảng thời gian.

**Ví dụ Java (mô phỏng nhiều node cache)**

```java
import java.util.HashMap;
import java.util.Map;

class Node {
    private final String name;
    private final Map<String, String> store = new HashMap<>();

    public Node(String name) {
        this.name = name;
    }

    public void write(String key, String value) {
        store.put(key, value);
        System.out.println(name + " write: " + key + " = " + value);
    }

    public String read(String key) {
        return store.get(key);
    }

    // đồng bộ dữ liệu từ node khác (giả lập eventual consistency)
    public void syncFrom(Node other, String key) {
        String v = other.read(key);
        store.put(key, v);
        System.out.println(name + " sync from " + other.name + ": " + key + " = " + v);
    }
}

public class EventualConsistencyExample {
    public static void main(String[] args) throws InterruptedException {
        Node nodeA = new Node("NodeA");
        Node nodeB = new Node("NodeB");

        // ban đầu ghi vào NodeA
        nodeA.write("x", "1");

        // NodeB chưa sync, đọc ra null (dữ liệu cũ / chưa có)
        System.out.println("NodeB read x (before sync): " + nodeB.read("x"));

        // một lúc sau, NodeB sync từ NodeA
        nodeB.syncFrom(nodeA, "x");

        // giờ NodeB đọc ra giá trị mới
        System.out.println("NodeB read x (after sync): " + nodeB.read("x"));
    }
}
```

**Output (mô phỏng):**
```
NodeA write: x = 1
NodeB read x (before sync): null
NodeB sync from NodeA: x = 1
NodeB read x (after sync): 1
```

Ban đầu NodeB không thấy giá trị mới, nhưng “cuối cùng” sau sync thì thấy → eventual consistency.

---

### 2.3. Causal Consistency (Nhất quán nhân quả)

- Nếu thao tác B **phụ thuộc nhân quả** vào thao tác A (B xảy ra sau khi thấy kết quả A), thì **mọi node** phải thấy A **trước** B.
- Các thao tác **không liên quan** có thể thấy theo thứ tự khác nhau.

Ví dụ:  
User1: “Xin chào”  
User2: trả lời “Chào bạn” sau khi thấy “Xin chào”.  
Mọi người phải thấy “Xin chào” trước “Chào bạn”.

---

### 2.4. Read-Your-Writes Consistency

- Một client sau khi ghi dữ liệu, **chính client đó** luôn đọc thấy giá trị mới (dù node khác có thể chưa thấy).
- Đảm bảo “tôi thấy những gì tôi vừa ghi”.

---

### 2.5. Monotonic Reads / Writes

- **Monotonic Reads**: Một client sẽ không bao giờ đọc được giá trị **cũ hơn** so với lần đọc trước.
- **Monotonic Writes**: Các ghi của một client được áp dụng theo **đúng thứ tự**.

---

## 3. Liên hệ với Java và hệ thống thực tế

- Trong Java, các khái niệm như `volatile`, `synchronized`, `Atomic*` cung cấp **nhất quán bộ nhớ** trong một JVM, nhưng không tự động giải quyết **nhất quán giữa nhiều node**.
- Trong hệ thống phân tán (microservices, database cluster, cache phân tán), bạn phải chọn mô hình nhất quán phù hợp:
  - Giao dịch ngân hàng → ưu tiên **strong consistency**.
  - Mạng xã hội, news feed → chấp nhận **eventual consistency** để tăng hiệu năng.
