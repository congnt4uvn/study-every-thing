
Dưới đây là tóm tắt ngắn gọn về chủ đề System Design: Caching (bộ nhớ đệm) trong Java, kèm ví dụ code và input/output.

---

## 1. Khái niệm chung về Caching

- **Mục tiêu**: Giảm tải cho database / service chậm, tăng tốc độ phản hồi bằng cách lưu tạm dữ liệu đã truy vấn.
- **Ý tưởng**: Khi client yêu cầu dữ liệu:
  - Nếu có trong cache → trả về ngay (nhanh).
  - Nếu không có → lấy từ nguồn gốc (DB, API), sau đó lưu vào cache.

---

## 2. In-memory Cache (Cache trong bộ nhớ JVM)

- Lưu dữ liệu trực tiếp trong bộ nhớ của ứng dụng (JVM).
- Rất nhanh, nhưng:
  - Mất dữ liệu khi ứng dụng restart.
  - Không chia sẻ được giữa nhiều instance (nhiều server).

### Ví dụ đơn giản với `ConcurrentHashMap`

```java
import java.util.concurrent.ConcurrentHashMap;

public class InMemoryCache {
    private final ConcurrentHashMap<String, String> cache = new ConcurrentHashMap<>();

    // Lấy dữ liệu từ cache, nếu không có thì load từ "DB"
    public String getData(String key) {
        // computeIfAbsent: nếu key chưa có thì gọi hàm để tạo value
        return cache.computeIfAbsent(key, this::loadFromDb);
    }

    // Giả lập truy vấn DB
    private String loadFromDb(String key) {
        System.out.println("Query DB for key = " + key);
        // Ví dụ: trả về "value-of-" + key
        return "value-of-" + key;
    }

    public static void main(String[] args) {
        InMemoryCache cache = new InMemoryCache();

        // Input
        String key = "user:1";

        // Lần 1: chưa có trong cache → gọi DB
        String v1 = cache.getData(key);
        System.out.println("First call: " + v1);

        // Lần 2: đã có trong cache → không gọi DB
        String v2 = cache.getData(key);
        System.out.println("Second call: " + v2);
    }
}
```

**Output (ví dụ):**
```text
Query DB for key = user:1
First call: value-of-user:1
Second call: value-of-user:1
```

## 3. Redis Cache (Cache phân tán, ngoài JVM)

- Redis là in-memory data store chạy riêng (server khác).
- Ưu điểm:
  - Chia sẻ được giữa nhiều instance của ứng dụng.
  - Có thể cấu hình persistence, replication, cluster.
- Dùng khi hệ thống nhiều server, cần cache chung.

### Ví dụ Java dùng Jedis (client Redis)

```xml
<!-- pom.xml -->
<dependencies>
    <dependency>
        <groupId>redis.clients</groupId>
        <artifactId>jedis</artifactId>
        <version>5.1.0</version>
    </dependency>
</dependencies>
```

```java
import redis.clients.jedis.Jedis;

public class RedisCacheExample {

    public static void main(String[] args) {
        // Kết nối Redis (mặc định localhost:6379)
        try (Jedis jedis = new Jedis("localhost", 6379)) {

            String key = "user:1:name";

            // Kiểm tra cache
            String cached = jedis.get(key);
            if (cached != null) {
                System.out.println("From Redis cache: " + cached);
            } else {
                // Giả lập lấy từ DB
                String fromDb = "Nguyen Van A";
                System.out.println("From DB: " + fromDb);

                // Lưu vào Redis với TTL 60 giây
                jedis.setex(key, 60, fromDb);
                System.out.println("Saved to Redis cache");
            }

            // Lấy lại lần nữa để thấy cache hoạt động
            String cachedAgain = jedis.get(key);
            System.out.println("Second read (should be from cache): " + cachedAgain);
        }
    }
}
```

**Output (lần chạy đầu):**
```text
From DB: Nguyen Van A
Saved to Redis cache
Second read (should be from cache): Nguyen Van A
```

**Output (chạy lại trong vòng 60s):**
```text
From Redis cache: Nguyen Van A
Second read (should be from cache): Nguyen Van A
```

## 4. Cache Invalidation (Xóa / làm mới cache)

Vấn đề khó: **khi dữ liệu gốc thay đổi, cache phải được cập nhật hoặc xóa** để tránh trả dữ liệu cũ.

### Các chiến lược phổ biến

1. **TTL (Time-To-Live)**  
   - Mỗi key có thời gian sống (ví dụ 60s).
   - Hết thời gian → tự động hết hạn.
   - Dễ dùng, nhưng có thể vẫn trả dữ liệu cũ trong khoảng TTL.

   ```java
   // Redis: set key với TTL 60 giây
   jedis.setex("user:1:name", 60, "Nguyen Van A");
   ```

2. **Write-through / Write-back / Write-around**  
   - **Write-through**: khi ghi vào DB → ghi luôn vào cache.
   - **Write-back**: ghi vào cache trước, sau đó async ghi DB.
   - **Write-around**: ghi thẳng DB, cache sẽ được cập nhật khi đọc lại.

3. **Explicit Invalidation (xóa chủ động)**  
   - Khi update/delete DB → xóa key tương ứng trong cache.

   ```java
   public void updateUserName(String userId, String newName) {
       // 1. Update DB
       updateUserNameInDb(userId, newName);

       // 2. Xóa cache
       String key = "user:" + userId + ":name";
       jedis.del(key);
   }
   ```

## 5. Tóm tắt ý chính

- **In-memory cache**: nhanh, đơn giản, phù hợp app nhỏ / single instance.
- **Redis cache**: phân tán, chia sẻ giữa nhiều server, phù hợp hệ thống lớn.
- **Cache invalidation**:
  - Cần chiến lược rõ ràng (TTL, xóa chủ động, write-through, …).
  - Mục tiêu: cân bằng giữa hiệu năng và tính nhất quán dữ liệu.

Nếu bạn muốn, tôi có thể viết một ví dụ hoàn chỉnh hơn (service Java có cả DB giả lập + in-memory cache + Redis cache) để so sánh.
