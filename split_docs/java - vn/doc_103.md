## 1. Database Sharding (Phân mảnh cơ sở dữ liệu)

**Ý tưởng chính**  
- Chia dữ liệu của **một bảng lớn** thành nhiều phần nhỏ (shard) và lưu trên **nhiều database/server khác nhau**.  
- Mỗi shard chứa **một tập con** của dữ liệu (theo userId, theo range, theo hash, theo khu vực địa lý…).  
- Mục tiêu: **scale ngang**, giảm tải cho một DB duy nhất, tăng hiệu năng.

**Ưu điểm**  
- Tăng khả năng mở rộng (horizontal scaling).  
- Giảm tải, tăng tốc độ truy vấn.  
- Có thể đặt shard gần người dùng (geo-sharding).

**Nhược điểm**  
- Thiết kế phức tạp (chọn key sharding, routing).  
- Join giữa shard khó, thường phải xử lý ở tầng ứng dụng.  
- Rebalancing (di chuyển dữ liệu giữa shard) phức tạp.

### Ví dụ: Sharding theo `userId` (Java pseudo-code)

Giả sử có 2 shard: `dbShard0`, `dbShard1`.  
Quy tắc: `shardId = userId % 2`.

```java
public class ShardRouter {
    private DataSource shard0;
    private DataSource shard1;

    public ShardRouter(DataSource shard0, DataSource shard1) {
        this.shard0 = shard0;
        this.shard1 = shard1;
    }

    private DataSource getDataSourceForUser(long userId) {
        int shardId = (int) (userId % 2);
        return shardId == 0 ? shard0 : shard1;
    }

    public User getUserById(long userId) throws SQLException {
        DataSource ds = getDataSourceForUser(userId);
        try (Connection conn = ds.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "SELECT id, name, email FROM users WHERE id = ?")) {
            ps.setLong(1, userId);
            ResultSet rs = ps.executeQuery();
            if (rs.next()) {
                return new User(
                        rs.getLong("id"),
                        rs.getString("name"),
                        rs.getString("email")
                );
            }
            return null;
        }
    }
}
```

**Input**: `userId = 5`  
- `5 % 2 = 1` → route đến `dbShard1`.

**Output**: Trả về bản ghi user từ `dbShard1`.

## 2. Replication (Nhân bản cơ sở dữ liệu)

**Ý tưởng chính**  
- Sao chép dữ liệu từ **một DB chính (primary/master)** sang **một hoặc nhiều DB phụ (replica/slave)**.  
- Primary xử lý **ghi** (write), replica chủ yếu xử lý **đọc** (read).  
- Dùng để: tăng khả năng đọc, tăng độ sẵn sàng (high availability), backup.

**Kiểu replication phổ biến**  
- **Master–Slave (Primary–Replica)**: 1 master, nhiều replica.  
- **Master–Master**: nhiều master, phức tạp hơn (conflict resolution).  
- **Synchronous** vs **Asynchronous**:  
  - Sync: đảm bảo replica cập nhật ngay, nhưng chậm hơn.  
  - Async: nhanh hơn, nhưng có độ trễ (replication lag).

**Ưu điểm**  
- Tăng throughput đọc.  
- Failover: nếu primary chết, có thể promote replica thành primary.  

**Nhược điểm**  
- Độ trễ dữ liệu (replica có thể chưa kịp cập nhật).  
- Cần cơ chế failover, monitoring.

## 3. Read/Write Splitting (Tách đọc/ghi)

**Ý tưởng chính**  
- Kết hợp với replication:  
  - **Write** → gửi đến **primary**.  
  - **Read** → gửi đến **replica**.  
- Mục tiêu: tận dụng replica để **scale đọc**, giảm tải cho primary.

**Vấn đề cần chú ý**  
- **Read-after-write**: vừa ghi xong mà đọc ngay từ replica có thể chưa thấy dữ liệu (do replication lag).  
  - Giải pháp:  
    - Đọc các request “ngay sau khi ghi” từ primary.  
    - Hoặc dùng “session stickiness” (trong 1 session, sau khi ghi thì đọc từ primary một thời gian).

### Ví dụ Java: Read/Write Splitting

```java
public class ReadWriteDataSourceRouter {
    private DataSource writeDataSource;      // primary
    private List<DataSource> readDataSources; // replicas
    private AtomicInteger counter = new AtomicInteger(0);

    public ReadWriteDataSourceRouter(DataSource writeDataSource,
                                     List<DataSource> readDataSources) {
        this.writeDataSource = writeDataSource;
        this.readDataSources = readDataSources;
    }

    // Chọn replica theo round-robin
    private DataSource getReadDataSource() {
        int index = counter.getAndIncrement() % readDataSources.size();
        return readDataSources.get(index);
    }

    public void insertUser(User user) throws SQLException {
        try (Connection conn = writeDataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "INSERT INTO users (id, name, email) VALUES (?, ?, ?)")) {
            ps.setLong(1, user.getId());
            ps.setString(2, user.getName());
            ps.setString(3, user.getEmail());
            ps.executeUpdate();
        }
    }

    public User getUserById(long userId, boolean strongConsistency) throws SQLException {
        DataSource ds;
        if (strongConsistency) {
            // Đọc từ primary để đảm bảo thấy dữ liệu mới ghi
            ds = writeDataSource;
        } else {
            // Đọc từ replica để scale đọc
            ds = getReadDataSource();
        }

        try (Connection conn = ds.getConnection();
             PreparedStatement ps = conn.prepareStatement(
                     "SELECT id, name, email FROM users WHERE id = ?")) {
            ps.setLong(1, userId);
            ResultSet rs = ps.executeQuery();
            if (rs.next()) {
                return new User(
                        rs.getLong("id"),
                        rs.getString("name"),
                        rs.getString("email")
                );
            }
            return null;
        }
    }
}
```

**Input ví dụ**  
1. Ghi:  
   - `insertUser(new User(10, "An", "an@example.com"))` → gửi đến primary.  
2. Đọc ngay sau khi ghi:  
   - `getUserById(10, true)` → đọc từ primary (strongConsistency = true).  
3. Đọc bình thường (không yêu cầu nhất quán mạnh):  
   - `getUserById(10, false)` → đọc từ một replica (có thể bị trễ).

**Output**  
- Trả về đối tượng `User` tương ứng, hoặc `null` nếu không tìm thấy.

---

## Tóm tắt ngắn

- **Sharding**: chia dữ liệu theo key (vd: userId) sang nhiều DB → scale ngang, nhưng phức tạp join & rebalancing.  
- **Replication**: nhân bản dữ liệu từ primary sang replica → tăng khả năng đọc, tăng HA, nhưng có replication lag.  
- **Read/Write Splitting**: write → primary, read → replica → tận dụng replication để scale đọc; cần xử lý vấn đề nhất quán (read-after-write).
