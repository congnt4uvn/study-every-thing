
Dưới đây là tóm tắt ngắn gọn về: **Transactions, ACID, Isolation Levels, Locking** trong SQL & RDBMS, kèm ví dụ Java.

---

## 1. Transaction (Giao dịch)

- Là một **nhóm các câu lệnh SQL** được thực thi như một đơn vị logic.
- Mục tiêu: hoặc **thành công toàn bộ**, hoặc **hủy toàn bộ** (không để dữ liệu ở trạng thái dở dang).

### Ví dụ Java (JDBC)

```java
Connection conn = null;
try {
    conn = DriverManager.getConnection(url, user, pass);
    conn.setAutoCommit(false); // Bắt đầu transaction

    // Ví dụ chuyển tiền: trừ tài khoản A, cộng tài khoản B
    PreparedStatement stmt1 = conn.prepareStatement(
        "UPDATE account SET balance = balance - ? WHERE id = ?");
    stmt1.setBigDecimal(1, new BigDecimal("100.00"));
    stmt1.setInt(2, 1); // account A
    stmt1.executeUpdate();

    PreparedStatement stmt2 = conn.prepareStatement(
        "UPDATE account SET balance = balance + ? WHERE id = ?");
    stmt2.setBigDecimal(1, new BigDecimal("100.00"));
    stmt2.setInt(2, 2); // account B
    stmt2.executeUpdate();

    conn.commit(); // Thành công -> ghi nhận toàn bộ
} catch (Exception e) {
    if (conn != null) conn.rollback(); // Lỗi -> rollback toàn bộ
} finally {
    if (conn != null) conn.close();
}
```

**Ý nghĩa:** Nếu một trong hai câu `UPDATE` lỗi, toàn bộ giao dịch bị hủy, dữ liệu không bị “lệch”.

## 2. ACID

ACID là 4 tính chất quan trọng của transaction:

1. **Atomicity (Tính nguyên tử)**  
   - Giao dịch là “tất cả hoặc không gì cả”.
   - Ví dụ: chuyển tiền, không thể chỉ trừ tiền ở A mà không cộng cho B.

2. **Consistency (Tính nhất quán)**  
   - Sau mỗi transaction, dữ liệu phải ở trạng thái **hợp lệ** theo các ràng buộc (constraints, khóa ngoại, v.v.).
   - Ví dụ: tổng tiền trong hệ thống không tự nhiên “biến mất”.

3. **Isolation (Tính cô lập)**  
   - Các transaction đồng thời **không làm sai lệch** lẫn nhau.
   - Mỗi transaction như đang chạy “một mình” (tùy mức độ isolation).

4. **Durability (Tính bền vững)**  
   - Khi `COMMIT`, dữ liệu được ghi bền vững (thường là xuống đĩa).
   - Mất điện, restart DB vẫn giữ được dữ liệu đã commit.

## 3. Isolation Levels (Mức độ cô lập)

Các mức isolation giải quyết các vấn đề sau:

- **Dirty Read**: đọc dữ liệu chưa commit của transaction khác.
- **Non-repeatable Read**: đọc cùng một dòng 2 lần trong cùng transaction nhưng giá trị khác nhau (do transaction khác update).
- **Phantom Read**: đọc tập bản ghi 2 lần, lần sau có thêm/bớt dòng (do transaction khác insert/delete).

### Các mức chuẩn (SQL standard)

1. **READ UNCOMMITTED**
   - Cho phép dirty read.
   - Ít khóa, hiệu năng cao nhưng rủi ro dữ liệu “bẩn”.

2. **READ COMMITTED**
   - Không cho dirty read.
   - Mỗi lần đọc có thể thấy dữ liệu mới commit từ transaction khác → có thể non-repeatable read, phantom.

3. **REPEATABLE READ**
   - Đảm bảo cùng một dòng đọc nhiều lần trong transaction sẽ không đổi (không non-repeatable read).
   - Có thể vẫn bị phantom (thêm dòng mới).

4. **SERIALIZABLE**
   - Mức cao nhất, như thể các transaction chạy tuần tự.
   - Tránh được dirty, non-repeatable, phantom.
   - Tốn tài nguyên, dễ gây lock nhiều.

### Ví dụ set isolation level trong Java

```java
Connection conn = DriverManager.getConnection(url, user, pass);
conn.setAutoCommit(false);

// Ví dụ: đặt mức REPEATABLE_READ
conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);

// Thực hiện các câu lệnh SQL trong transaction này
// ...

conn.commit();
```

## 4. Locking (Cơ chế khóa)

Lock dùng để **bảo vệ dữ liệu** khi nhiều transaction truy cập đồng thời.

### Các loại khóa phổ biến

1. **Shared Lock (S-lock)**  
   - Dùng khi **đọc** dữ liệu.
   - Nhiều transaction có thể cùng giữ S-lock trên một dòng/bản ghi.
   - Nhưng S-lock chặn X-lock (khóa ghi).

2. **Exclusive Lock (X-lock)**  
   - Dùng khi **ghi** (INSERT/UPDATE/DELETE).
   - Chỉ một transaction được giữ X-lock trên một bản ghi.
   - Chặn cả S-lock và X-lock khác.

3. **Lock granularity (độ chi tiết khóa)**  
   - Khóa có thể ở mức: **row**, **page**, **table**, **database**.
   - Khóa càng “thô” (table-level) thì càng dễ gây chặn (blocking), nhưng quản lý đơn giản.

### Vấn đề thường gặp

- **Deadlock (khoá chết)**:  
  - T1 giữ khóa A, chờ khóa B.  
  - T2 giữ khóa B, chờ khóa A.  
  - Cả hai chờ nhau mãi → DB phải phát hiện và hủy một transaction.

## 5. Ví dụ minh họa lỗi do isolation thấp

Giả sử mức **READ UNCOMMITTED**:

### Transaction 1

```sql
BEGIN;
UPDATE account SET balance = balance - 100 WHERE id = 1;
-- chưa COMMIT
```

### Transaction 2 (đọc dirty data)

```sql
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
SELECT balance FROM account WHERE id = 1;
-- Có thể thấy số dư đã bị trừ 100, dù T1 chưa commit
```

Nếu T1 sau đó `ROLLBACK`, T2 đã đọc một giá trị **không bao giờ tồn tại chính thức** trong DB → dirty read.

---

## Tóm tắt ngắn

- **Transaction**: nhóm câu lệnh SQL thực thi như một đơn vị, dùng `COMMIT`/`ROLLBACK`.
- **ACID**: đảm bảo giao dịch an toàn, nhất quán, cô lập, bền vững.
- **Isolation Levels**: cân bằng giữa **độ chính xác** và **hiệu năng** (READ UNCOMMITTED → SERIALIZABLE).
- **Locking**: dùng shared/exclusive lock để bảo vệ dữ liệu khi truy cập đồng thời, nhưng có thể gây blocking, deadlock.

Nếu bạn muốn, tôi có thể viết thêm ví dụ cụ thể với một bảng (VD: `account`) và log input/output chi tiết cho từng mức isolation.
