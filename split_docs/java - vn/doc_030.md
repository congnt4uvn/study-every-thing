
Dưới đây là tóm tắt ngắn gọn về chủ đề **Exceptions & Error Handling: Khi nào dùng exception, khi nào dùng kiểu trả về** trong Java, kèm ví dụ.

---

## 1. Khi nào nên dùng Exception?

Dùng **exception** khi:

1. **Lỗi là bất thường (exceptional)**  
   - Không xảy ra thường xuyên.  
   - Khó xử lý bằng logic bình thường.  
   - Ví dụ: lỗi I/O, lỗi kết nối DB, file không tồn tại, dữ liệu không hợp lệ nghiêm trọng.

2. **Bạn muốn tách luồng xử lý lỗi khỏi luồng xử lý chính**  
   - Code chính tập trung vào “happy path”.  
   - Xử lý lỗi được gom vào `catch` hoặc tầng xử lý riêng.

3. **Cần truyền thông tin lỗi chi tiết lên các tầng cao hơn**  
   - Dùng `throw` để đẩy lỗi lên caller.  
   - Có thể kèm message, cause, stack trace.

4. **Lỗi không thể (hoặc không nên) được xử lý tại chỗ**  
   - Ví dụ: service tầng dưới không thể tự quyết định cách xử lý, nên ném exception cho tầng trên.

### Ví dụ dùng Exception

```java
public class UserNotFoundException extends Exception {
    public UserNotFoundException(String message) {
        super(message);
    }
}

public class UserService {
    public User findUserById(String id) throws UserNotFoundException {
        // Giả sử không tìm thấy user
        boolean found = false;
        if (!found) {
            throw new UserNotFoundException("User with id " + id + " not found");
        }
        // return user nếu có
        return null;
    }
}

public class Main {
    public static void main(String[] args) {
        UserService service = new UserService();
        try {
            User user = service.findUserById("123");
            System.out.println("Found user: " + user);
        } catch (UserNotFoundException e) {
            System.out.println("Error: " + e.getMessage());
        }
    }
}
```

**Output (giả sử không tìm thấy user):**
```text
Error: User with id 123 not found
```

## 2. Khi nào nên dùng kiểu trả về (return types) để báo lỗi?

Dùng **return type** (ví dụ: `null`, `Optional`, mã lỗi, object kết quả) khi:

1. **Lỗi là một phần bình thường của luồng xử lý**  
   - Ví dụ: tìm kiếm có thể không thấy kết quả.  
   - Đây không phải “exceptional”, mà là trạng thái hợp lệ.

2. **Bạn muốn tránh chi phí và độ phức tạp của exception**  
   - Exception có chi phí runtime cao hơn.  
   - Dùng return type giúp code đơn giản hơn trong các trường hợp lỗi thường xuyên.

3. **API rõ ràng về các trạng thái có thể xảy ra**  
   - Ví dụ: dùng `Optional<T>` để thể hiện “có hoặc không có kết quả”.  
   - Dùng class kết quả: `Result<T>` chứa `value` hoặc `errorCode`.

### Ví dụ dùng `Optional` thay vì Exception

```java
import java.util.Optional;

public class UserService {
    public Optional<User> findUserById(String id) {
        boolean found = false;
        if (!found) {
            return Optional.empty();
        }
        // return Optional.of(user) nếu có
        return Optional.empty();
    }
}

public class Main {
    public static void main(String[] args) {
        UserService service = new UserService();
        Optional<User> userOpt = service.findUserById("123");

        if (userOpt.isPresent()) {
            System.out.println("Found user: " + userOpt.get());
        } else {
            System.out.println("User not found");
        }
    }
}
```

**Output:**
```text
User not found
```

## 3. So sánh nhanh

| Tiêu chí                          | Exception                                  | Return type (Optional, mã lỗi, v.v.)          |
|----------------------------------|--------------------------------------------|-----------------------------------------------|
| Tần suất xảy ra                  | Hiếm, bất thường                           | Thường xuyên, bình thường                     |
| Độ rõ ràng luồng xử lý chính     | Rõ (happy path sạch)                       | Có thêm if/else kiểm tra kết quả              |
| Chi phí runtime                  | Cao hơn (stack trace, throw/catch)         | Thấp hơn                                      |
| Truyền thông tin lỗi chi tiết    | Tốt (message, cause, stack trace)          | Hạn chế, trừ khi tự thiết kế object kết quả   |
| Phù hợp cho                      | Lỗi nghiêm trọng, không mong đợi           | Trạng thái hợp lệ: không tìm thấy, rỗng, v.v. |

## 4. Ví dụ kết hợp: Result object

Khi cần nhiều thông tin nhưng không muốn dùng exception cho lỗi thường xuyên:

```java
public class Result<T> {
    private final T value;
    private final String error;

    private Result(T value, String error) {
        this.value = value;
        this.error = error;
    }

    public static <T> Result<T> ok(T value) {
        return new Result<>(value, null);
    }

    public static <T> Result<T> error(String error) {
        return new Result<>(null, error);
    }

    public boolean isOk() { return error == null; }
    public T getValue() { return value; }
    public String getError() { return error; }
}

public class Calculator {
    public Result<Integer> divide(int a, int b) {
        if (b == 0) {
            return Result.error("Division by zero");
        }
        return Result.ok(a / b);
    }
}

public class Main {
    public static void main(String[] args) {
        Calculator calc = new Calculator();

        Result<Integer> r1 = calc.divide(10, 2);
        Result<Integer> r2 = calc.divide(10, 0);

        if (r1.isOk()) {
            System.out.println("10 / 2 = " + r1.getValue());
        } else {
            System.out.println("Error: " + r1.getError());
        }

        if (r2.isOk()) {
            System.out.println("10 / 0 = " + r2.getValue());
        } else {
            System.out.println("Error: " + r2.getError());
        }
    }
}
```

**Output:**
```text
10 / 2 = 5
Error: Division by zero
```

---

## 5. Tóm tắt ý chính

- **Dùng exception** cho lỗi bất thường, nghiêm trọng, khó xử lý bằng logic bình thường, cần truyền thông tin lỗi chi tiết lên trên.
- **Dùng return type (Optional, Result, mã lỗi, null)** cho các trạng thái lỗi/bất thành công **thường xuyên và hợp lệ** trong luồng xử lý.
- Thiết kế API:  
  - “Không tìm thấy” → thường dùng `Optional` hoặc `Result`.  
  - “Lỗi hệ thống, dữ liệu hỏng, không thể tiếp tục” → dùng exception.
