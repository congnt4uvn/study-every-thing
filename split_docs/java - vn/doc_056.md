
Dưới đây là tóm tắt ngắn gọn về các “anti-pattern” (cách làm sai) thường gặp trong tối ưu hiệu năng Java, kèm ví dụ code và input/output minh họa.

---

## 1. Tạo quá nhiều đối tượng (Excessive Object Creation)

**Ý chính:**
- Tạo đối tượng mới liên tục trong vòng lặp hoặc cho các giá trị bất biến (immutable) gây:
  - Tốn bộ nhớ
  - Tăng áp lực GC (Garbage Collection)
  - Giảm hiệu năng

**Ví dụ xấu:**

```java
public class BadObjectCreation {
    public static void main(String[] args) {
        long sum = 0;
        for (int i = 0; i < 1_000_000; i++) {
            // Mỗi vòng lặp tạo một Integer mới
            Integer value = new Integer(i); // anti-pattern
            sum += value;
        }
        System.out.println(sum);
    }
}
```

**Input:** không có (chạy trực tiếp)  
**Output (kết quả logic):**  
`499999500000`  
Nhưng chương trình chạy chậm và tốn bộ nhớ hơn cần thiết.

**Cách tốt hơn:**

- Dùng kiểu nguyên thủy (`int`, `long`) khi có thể.
- Tái sử dụng đối tượng, dùng cache, pool, hoặc các factory method như `Integer.valueOf()`.

```java
public class GoodObjectCreation {
    public static void main(String[] args) {
        long sum = 0;
        for (int i = 0; i < 1_000_000; i++) {
            // Dùng kiểu nguyên thủy, không tạo object thừa
            int value = i;
            sum += value;
        }
        System.out.println(sum);
    }
}
```

## 2. Đồng bộ hóa quá mức (Excessive Synchronization)

**Ý chính:**
- Dùng `synchronized` không cần thiết hoặc khóa (lock) phạm vi quá rộng:
  - Gây tắc nghẽn (contention)
  - Giảm khả năng chạy song song (concurrency)
  - Làm chương trình chậm dù logic đơn giản

**Ví dụ xấu:**

```java
public class BadSynchronization {
    private int counter = 0;

    // Đồng bộ hóa cả method, dù chỉ có một thao tác đơn giản
    public synchronized void increment() {
        counter++;
    }

    public static void main(String[] args) throws InterruptedException {
        BadSynchronization obj = new BadSynchronization();
        Runnable task = () -> {
            for (int i = 0; i < 1_000_000; i++) {
                obj.increment();
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        long start = System.currentTimeMillis();
        t1.start(); t2.start();
        t1.join(); t2.join();
        long end = System.currentTimeMillis();

        System.out.println("Counter = " + obj.counter);
        System.out.println("Time(ms) = " + (end - start));
    }
}
```

**Output (ví dụ):**
- `Counter = 2000000` (đúng)
- `Time(ms) = 300` (ví dụ, nhưng có thể cao do lock nhiều)

**Cách tốt hơn:**

- Dùng `AtomicInteger` hoặc thu hẹp phạm vi lock.
- Chỉ đồng bộ phần thực sự cần thiết.

```java
import java.util.concurrent.atomic.AtomicInteger;

public class GoodSynchronization {
    private AtomicInteger counter = new AtomicInteger(0);

    public void increment() {
        counter.incrementAndGet(); // lock-free, tối ưu hơn
    }

    public static void main(String[] args) throws InterruptedException {
        GoodSynchronization obj = new GoodSynchronization();
        Runnable task = () -> {
            for (int i = 0; i < 1_000_000; i++) {
                obj.increment();
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        long start = System.currentTimeMillis();
        t1.start(); t2.start();
        t1.join(); t2.join();
        long end = System.currentTimeMillis();

        System.out.println("Counter = " + obj.counter.get());
        System.out.println("Time(ms) = " + (end - start));
    }
}
```

## 3. I/O Blocking (Chặn I/O)

**Ý chính:**
- Gọi I/O (đọc/ghi file, network) theo kiểu blocking trong luồng chính hoặc ít luồng:
  - Làm treo (block) luồng
  - Giảm throughput khi xử lý nhiều request
- Đặc biệt tệ trong server xử lý nhiều kết nối.

**Ví dụ xấu (đọc file theo kiểu blocking, từng dòng một):**

```java
import java.io.*;

public class BadIOBlocking {
    public static void main(String[] args) {
        String filePath = "large_file.txt";
        long start = System.currentTimeMillis();
        try (BufferedReader reader = new BufferedReader(new FileReader(filePath))) {
            String line;
            // Đọc tuần tự, nếu file rất lớn sẽ mất nhiều thời gian
            while ((line = reader.readLine()) != null) {
                // Xử lý chậm, ví dụ: in ra console
                System.out.println(line);
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
        long end = System.currentTimeMillis();
        System.out.println("Time(ms) = " + (end - start));
    }
}
```

**Input:** file `large_file.txt` lớn  
**Output:** in toàn bộ nội dung file + thời gian chạy lớn

**Cách tốt hơn (gợi ý):**

- Dùng I/O bất đồng bộ (NIO, `AsynchronousFileChannel`, `Selector`) cho server nhiều kết nối.
- Hoặc dùng thread pool để không chặn luồng chính.
- Hạn chế in ra console trong vòng lặp lớn (I/O console cũng chậm).

Ví dụ đơn giản dùng NIO để đọc nhanh hơn (vẫn blocking nhưng hiệu quả hơn):

```java
import java.nio.file.*;
import java.io.IOException;
import java.util.List;

public class BetterIO {
    public static void main(String[] args) {
        String filePath = "large_file.txt";
        long start = System.currentTimeMillis();
        try {
            // Đọc toàn bộ file vào memory (phù hợp file vừa phải)
            List<String> lines = Files.readAllLines(Paths.get(filePath));
            // Xử lý nhanh, hạn chế I/O console
            long count = lines.size();
            System.out.println("Total lines = " + count);
        } catch (IOException e) {
            e.printStackTrace();
        }
        long end = System.currentTimeMillis();
        System.out.println("Time(ms) = " + (end - start));
    }
}
```

**Output (ví dụ):**
- `Total lines = 1000000`
- `Time(ms) = 50` (nhanh hơn, tùy kích thước file và máy)

## Tóm tắt ngắn gọn

1. **Tạo quá nhiều đối tượng**
   - Tránh tạo object trong vòng lặp nếu không cần.
   - Ưu tiên kiểu nguyên thủy, tái sử dụng object, dùng cache/pool.

2. **Đồng bộ hóa quá mức**
   - Không dùng `synchronized` rộng rãi nếu không cần.
   - Dùng `Atomic*`, `Lock`, hoặc thu hẹp phạm vi lock.

3. **I/O Blocking**
   - Tránh I/O blocking trong luồng chính hoặc ít luồng.
   - Dùng NIO, async I/O, hoặc thread pool để tăng throughput.
