
Dưới đây là tóm tắt ngắn gọn về **Java Memory Model (JMM)**, tập trung vào:  
- happens-before  
- visibility (tính nhìn thấy)  
- reordering (tái sắp xếp lệnh)  

---

## 1. Java Memory Model (JMM) là gì?

JMM định nghĩa cách **thread** đọc/ghi dữ liệu trong bộ nhớ, và quy định khi nào một thread **nhìn thấy** (thấy được) các thay đổi do thread khác ghi ra.  
Nó giải quyết 3 vấn đề chính:

1. **Happens-before**: quan hệ thứ tự logic giữa các thao tác.
2. **Visibility**: một thread có thấy được giá trị mới do thread khác ghi không.
3. **Reordering**: CPU/JVM có thể sắp xếp lại lệnh, nhưng vẫn phải tuân thủ JMM.

---

## 2. Happens-before

Quan hệ **happens-before** nói rằng:  
Nếu A *happens-before* B, thì **mọi thay đổi bộ nhớ** của A sẽ **nhìn thấy được** bởi B.

Một số quy tắc quan trọng:

1. **Trong cùng một thread**:  
   Thứ tự lệnh trong code (theo chương trình) tạo ra quan hệ happens-before.
   ```java
   int x = 0;      // A
   x = 10;         // B
   // A happens-before B
   ```

2. **Khóa (synchronized)**:
   - `unlock` trên một monitor happens-before `lock` tiếp theo trên cùng monitor.
   ```java
   class Counter {
       private int count = 0;

       public synchronized void increment() { // lock
           count++;
       } // unlock

       public synchronized int get() { // lock
           return count;
       } // unlock
   }
   ```
   Ở đây:
   - Mọi ghi trong `increment()` happens-before mọi đọc trong `get()` (khi gọi từ các thread khác nhau).

3. **Biến volatile**:
   - Ghi vào một biến `volatile` happens-before mọi đọc sau đó của biến đó.
   ```java
   class FlagExample {
       private volatile boolean flag = false;
       private int data = 0;

       public void writer() {
           data = 42;      // 1
           flag = true;    // 2 (volatile write)
       }

       public void reader() {
           if (flag) {     // 3 (volatile read)
               System.out.println(data); // 4
           }
       }
   }
   ```
   - Ghi `flag = true` (2) happens-before đọc `flag` thấy `true` (3).  
   - Do đó, nếu `reader()` in ra, thì chắc chắn `data` đã là 42.

4. **Thread start / join**:
   ```java
   int x = 0;

   Thread t = new Thread(() -> {
       System.out.println(x); // B
   });

   x = 10;    // A
   t.start(); // A happens-before B
   ```

   ```java
   class Example {
       private int result = 0;

       public void compute() {
           Thread t = new Thread(() -> {
               result = 100; // A
           });
           t.start();
           try {
               t.join();     // B
           } catch (InterruptedException e) {}
           System.out.println(result); // C
       }
   }
   ```
   - A happens-before B (kết thúc thread t).  
   - B happens-before C.  
   → C chắc chắn thấy `result = 100`.

## 3. Visibility (tính nhìn thấy)

Không có đồng bộ (synchronized, volatile, lock, …), các thread có thể **không thấy** giá trị mới.

### Ví dụ lỗi visibility

```java
class VisibilityProblem {
    private static boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (running) {
                // do something
            }
            System.out.println("Stopped");
        });
        t.start();

        Thread.sleep(1000);
        running = false; // main thread thay đổi
    }
}
```

- Có thể xảy ra: thread `t` **không bao giờ dừng**, vì nó đọc `running` từ cache, không thấy `false`.

### Cách sửa: dùng volatile

```java
class VisibilityFixed {
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (running) {
                // do something
            }
            System.out.println("Stopped");
        });
        t.start();

        Thread.sleep(1000);
        running = false; // đảm bảo thread t sẽ thấy false
    }
}
```

**Output (có thể):**
```
Stopped
```

## 4. Reordering (tái sắp xếp lệnh)

CPU và JVM có thể **tự động sắp xếp lại** lệnh để tối ưu, miễn là **không phá vỡ JMM** trong một thread đơn.  
Nhưng giữa nhiều thread, nếu không có happens-before, ta có thể thấy hành vi “kỳ lạ”.

### Ví dụ reordering kinh điển

```java
class ReorderingExample {
    static int x = 0, y = 0;
    static int a = 0, b = 0;

    public static void main(String[] args) throws InterruptedException {
        for (int i = 0; i < 1_000_000; i++) {
            x = y = a = b = 0;

            Thread t1 = new Thread(() -> {
                a = 1;          // 1
                x = b;          // 2
            });

            Thread t2 = new Thread(() -> {
                b = 1;          // 3
                y = a;          // 4
            });

            t1.start();
            t2.start();
            t1.join();
            t2.join();

            if (x == 0 && y == 0) {
                System.out.println("x = " + x + ", y = " + y);
                break;
            }
        }
    }
}
```

- Về mặt trực giác, ta nghĩ ít nhất một trong `x` hoặc `y` phải là 1.  
- Nhưng do **reordering + thiếu happens-before**, trường hợp `x = 0, y = 0` **có thể xảy ra**.

### Cách ngăn reordering

- Dùng `volatile`, `synchronized`, `Lock`, `Atomic*`, hoặc các cấu trúc đồng bộ khác.  
- Chúng tạo ra **memory barrier** và quan hệ happens-before, ngăn reordering “nguy hiểm”.

---

## 5. Tóm tắt ngắn

- **JMM** định nghĩa cách các thread chia sẻ bộ nhớ và khi nào thấy được thay đổi của nhau.
- **Happens-before**: nếu A happens-before B, thì B thấy mọi thay đổi bộ nhớ của A.
- **Visibility**: không có đồng bộ → thread có thể không thấy giá trị mới.
- **Reordering**: CPU/JVM có thể sắp xếp lại lệnh; dùng `volatile`, `synchronized`, `join`, `start`, v.v. để đảm bảo thứ tự logic giữa các thread.

Nếu bạn muốn, tôi có thể viết thêm ví dụ đầy đủ hơn với input/output mô phỏng (ví dụ dùng `ExecutorService`, `AtomicInteger`, v.v.).
