
Dưới đây là tóm tắt ngắn gọn về chủ đề “Performance Tuning: Identifying memory leaks, CPU hotspots” trong Java, kèm ví dụ mã và input/output minh họa.

---

## 1. Khái niệm chính

### a. Memory Leak (rò rỉ bộ nhớ)
- Xảy ra khi đối tượng **không còn cần dùng** nhưng vẫn còn **được tham chiếu**, nên GC (Garbage Collector) không thu hồi được.
- Hậu quả: tăng dần sử dụng heap, dẫn đến `OutOfMemoryError`, ứng dụng chậm dần hoặc treo.

### b. CPU Hotspot (điểm nóng CPU)
- Là những đoạn code **tiêu tốn nhiều CPU** nhất (vòng lặp nặng, thuật toán kém tối ưu, lock contention…).
- Hậu quả: CPU luôn cao, request xử lý chậm, throughput thấp.

---

## 2. Cách nhận diện Memory Leak

### Công cụ thường dùng
- **Java Flight Recorder (JFR)**, **Java Mission Control (JMC)**
- **VisualVM**, **Eclipse MAT (Memory Analyzer Tool)**
- **jmap**, **jstat**, **jcmd**

### Dấu hiệu
- Heap usage tăng dần theo thời gian, không giảm sau GC.
- Thường xuyên `Full GC` nhưng bộ nhớ không được giải phóng nhiều.
- Cuối cùng có thể gặp `java.lang.OutOfMemoryError: Java heap space`.

### Ví dụ code gây memory leak

```java
import java.util.ArrayList;
import java.util.List;

public class MemoryLeakExample {
    private static final List<byte[]> cache = new ArrayList<>();

    public static void main(String[] args) throws InterruptedException {
        while (true) {
            // Mỗi lần thêm 1MB vào list static, không bao giờ xóa
            cache.add(new byte[1024 * 1024]); // 1MB
            System.out.println("Allocated 1MB, total: " + cache.size() + " MB");
            Thread.sleep(200);
        }
    }
}
```

**Mô tả input/output:**
- Input: không có input từ người dùng, chương trình chỉ chạy vô hạn.
- Output (console, ví dụ):
  ```text
  Allocated 1MB, total: 1 MB
  Allocated 1MB, total: 2 MB
  ...
  Exception in thread "main" java.lang.OutOfMemoryError: Java heap space
  ```
- Quan sát bằng VisualVM/JFR: heap tăng đều, không giảm.

### Cách khắc phục (ý tưởng)
- Giải phóng reference khi không cần:
  - Xóa khỏi collection.
  - Dùng `WeakReference`/`WeakHashMap` nếu phù hợp.
- Tránh dùng biến `static` giữ dữ liệu tạm.
- Đảm bảo listener, callback được unregister khi không dùng.

## 3. Cách nhận diện CPU Hotspots

### Công cụ thường dùng
- **Java Flight Recorder**, **JMC**
- **VisualVM CPU Profiler**
- **YourKit**, **JProfiler**
- `jstack` để xem thread dump (deadlock, busy loop).

### Dấu hiệu
- CPU sử dụng cao (gần 100%) trong thời gian dài.
- Một hoặc vài thread luôn ở trạng thái `RUNNABLE`.
- Thời gian response tăng, throughput giảm.

### Ví dụ code gây CPU hotspot

```java
public class CpuHotspotExample {
    public static void main(String[] args) {
        long count = 0;
        while (true) {
            // Vòng lặp vô hạn, không sleep, không I/O
            count++;
            if (count % 1_000_000_000L == 0) {
                System.out.println("Count = " + count);
            }
        }
    }
}
```

**Mô tả input/output:**
- Input: không có.
- Output (console, ví dụ):
  ```text
  Count = 1000000000
  Count = 2000000000
  ...
  ```
- Quan sát: CPU của process Java ~100% (1 core).

### Cách khắc phục (ý tưởng)
- Tối ưu thuật toán (giảm độ phức tạp, dùng cấu trúc dữ liệu phù hợp).
- Thêm cơ chế chờ/sleep nếu là polling:
  ```java
  Thread.sleep(10);
  ```
- Giảm lock contention (dùng lock tinh vi hơn, concurrent collections).
- Cache kết quả tính toán nặng (memoization).
- Tránh tạo object quá nhiều trong vòng lặp (giảm GC pressure).

## 4. Ví dụ đơn giản: đo thời gian để tìm hotspot

```java
public class PerformanceTest {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();
        long sum = 0;

        for (int i = 0; i < 100_000_000; i++) {
            sum += heavyOperation(i);
        }

        long end = System.currentTimeMillis();
        System.out.println("Sum = " + sum);
        System.out.println("Time = " + (end - start) + " ms");
    }

    private static int heavyOperation(int x) {
        // Giả lập xử lý nặng
        int result = 0;
        for (int i = 0; i < 100; i++) {
            result += (x * i) % 7;
        }
        return result;
    }
}
```

**Input/Output:**
- Input: không có.
- Output (ví dụ):
  ```text
  Sum = 1234567890
  Time = 3500 ms
  ```
- Dùng profiler sẽ thấy `heavyOperation` chiếm phần lớn thời gian → đây là CPU hotspot cần tối ưu.

---

## 5. Tóm tắt ngắn gọn

- **Memory leak**: đối tượng không còn dùng nhưng vẫn được tham chiếu → heap tăng, GC không giải phóng được → có thể dẫn đến `OutOfMemoryError`.  
  - Dùng profiler/bộ phân tích heap (JFR, VisualVM, MAT) để tìm class/collection giữ nhiều object bất thường.
- **CPU hotspot**: đoạn code tiêu tốn nhiều CPU nhất (vòng lặp nặng, thuật toán kém, lock…) → CPU cao, app chậm.  
  - Dùng CPU profiler (JFR, VisualVM, YourKit…) để tìm method chiếm nhiều thời gian nhất.

Nếu bạn muốn, tôi có thể viết thêm ví dụ “trước và sau tối ưu” cho một memory leak hoặc CPU hotspot cụ thể.
