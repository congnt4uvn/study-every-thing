
Dưới đây là tóm tắt ngắn gọn về **Performance Testing** với **JMeter** và **Gatling**, kèm ví dụ code và input/output minh họa.

---

## 1. Performance Testing là gì?

- Kiểm tra **tốc độ, độ ổn định, khả năng chịu tải** của hệ thống.
- Mục tiêu:
  - Hệ thống chịu được bao nhiêu **người dùng đồng thời**?
  - Thời gian phản hồi (**response time**) có đạt yêu cầu không?
  - Có bị **timeout, lỗi 5xx, nghẽn cổ chai** (bottleneck) không?

Các loại phổ biến:
- **Load Testing**: kiểm tra hệ thống dưới tải mong đợi.
- **Stress Testing**: đẩy tải vượt mức để xem hệ thống “vỡ” ở đâu.
- **Soak/Endurance Testing**: chạy lâu dài để phát hiện rò rỉ bộ nhớ, giảm hiệu năng theo thời gian.

## 2. JMeter – Cơ bản

**Apache JMeter** là công cụ Java-based, thường dùng để:
- Mô phỏng nhiều user gửi request HTTP, JDBC, gRPC, v.v.
- Đo: response time, throughput, error rate.

### 2.1. Cấu trúc test plan cơ bản

- **Test Plan**
  - **Thread Group** (số user ảo, thời gian ramp-up, số vòng lặp)
    - **Sampler** (HTTP Request, JDBC Request, …)
    - **Listener** (View Results Tree, Summary Report, Aggregate Report)

### 2.2. Ví dụ: Test API HTTP đơn giản

Giả sử bạn test API:
- URL: `https://api.example.com/users`
- Method: `GET`

**Cấu hình Thread Group:**
- Number of Threads (users): `10`
- Ramp-Up Period: `5` giây (tăng dần lên 10 user trong 5 giây)
- Loop Count: `5` (mỗi user gọi 5 lần)

**Kết quả (ví dụ trong Summary Report):**

| Label        | #Samples | Average (ms) | Min | Max  | Error % | Throughput (req/s) |
|-------------|----------|-------------|-----|------|---------|---------------------|
| /users      | 50       | 120         | 80  | 300  | 0.00%   | 9.5                 |

- **Input**: 10 user ảo, mỗi user gọi 5 lần → 50 request.
- **Output**:
  - Thời gian phản hồi trung bình: 120 ms
  - Không có lỗi (Error % = 0)
  - Throughput ~ 9.5 request/giây

## 3. Gatling – Cơ bản

**Gatling** là công cụ performance testing dựa trên **Scala**, thường dùng cho:
- Test HTTP, WebSocket.
- Tích hợp CI/CD dễ dàng (chạy bằng CLI, script).

### 3.1. Khái niệm chính

- **Simulation**: lớp mô tả kịch bản test.
- **Scenario**: chuỗi hành động của user (gửi request, pause, lặp lại).
- **Injection profile**: cách “bơm” user vào hệ thống (bao nhiêu user, trong bao lâu).

### 3.2. Ví dụ code Gatling (Scala)

Giả sử test API `GET /users`:

```scala
import io.gatling.core.Predef._
import io.gatling.http.Predef._
import scala.concurrent.duration._

class UsersSimulation extends Simulation {

  val httpProtocol = http
    .baseUrl("https://api.example.com") // base URL
    .acceptHeader("application/json")

  val scn = scenario("Get Users Scenario")
    .exec(
      http("Get Users Request")
        .get("/users")
        .check(status.is(200)) // kiểm tra HTTP status = 200
    )

  setUp(
    scn.inject(
      rampUsers(10) during (5.seconds) // tăng từ 0 lên 10 user trong 5 giây
    )
  ).protocols(httpProtocol)
}
```

**Input:**
- 10 user ảo, ramp-up trong 5 giây.
- Mỗi user thực hiện 1 request `GET /users`.

**Output (ví dụ report HTML của Gatling):**
- `Requests`:
  - Total: 10
  - OK: 10
  - KO: 0
- `Response Time`:
  - Min: 90 ms
  - Max: 250 ms
  - Mean: 130 ms
- `Percentiles`:
  - 95th percentile: 220 ms
- `Throughput`: ~ 2 request/giây (tùy thời gian thực tế)

## 4. So sánh nhanh JMeter vs Gatling

- **JMeter**
  - Giao diện GUI, dễ làm quen.
  - Hỗ trợ nhiều protocol (HTTP, JDBC, JMS, FTP, …).
  - Phù hợp cho người mới, QA không chuyên code.

- **Gatling**
  - Viết kịch bản bằng code (Scala/Java), dễ version control.
  - Báo cáo HTML đẹp, chi tiết.
  - Tích hợp CI/CD tốt, phù hợp team dev.

---

## 5. Ý chính cần nhớ

- Performance testing giúp đảm bảo hệ thống:
  - Đáp ứng nhanh, ổn định, chịu được tải thực tế.
- JMeter và Gatling là hai công cụ phổ biến:
  - JMeter: GUI, đa giao thức.
  - Gatling: code-based, mạnh cho automation.
- Luôn quan tâm đến:
  - Response time, throughput, error rate, percentiles (p95, p99).
  - Cách mô phỏng user thật (số user, ramp-up, thời gian test).
