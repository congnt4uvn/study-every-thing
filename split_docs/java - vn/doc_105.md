
Dưới đây là tóm tắt ngắn gọn về **Rate Limiting** và **Throttling** trong hệ thống (gắn với Java khi có thể).

---

## 1. Khái niệm chính

### Rate Limiting (Giới hạn tần suất)
- Mục tiêu: Giới hạn **số lượng request** trong một khoảng thời gian (vd: 100 request / phút / user).
- Dùng để:
  - Bảo vệ hệ thống khỏi bị spam, DDoS, lạm dụng API.
  - Đảm bảo công bằng giữa các client.
  - Kiểm soát chi phí (đặc biệt với API trả phí).

### Throttling (Tiết lưu / Giảm tốc)
- Mục tiêu: **Làm chậm** hoặc **giảm tốc độ xử lý** request khi hệ thống quá tải.
- Dùng để:
  - Ngăn hệ thống bị quá tải (overload).
  - Giữ hệ thống ở mức hoạt động ổn định (graceful degradation).
  - Có thể delay, queue, hoặc từ chối bớt request.

> Tóm lại:  
> - **Rate limiting**: “Không cho vượt quá X request trong Y thời gian.”  
> - **Throttling**: “Khi quá tải thì xử lý chậm lại hoặc giảm bớt.”

---

## 2. Một số thuật toán phổ biến (Rate Limiting)

### 2.1. Fixed Window Counter
- Đếm số request trong một “cửa sổ thời gian” cố định (vd: mỗi phút).
- Đơn giản nhưng có thể bị “burst” ở ranh giới thời gian.

### 2.2. Sliding Window
- Cửa sổ trượt theo thời gian thực, chính xác hơn Fixed Window.

### 2.3. Token Bucket
- Có “xô” chứa token, mỗi request cần 1 token.
- Token được nạp đều theo thời gian.
- Cho phép burst ngắn hạn nhưng vẫn giữ tốc độ trung bình.

### 2.4. Leaky Bucket
- Request vào “xô”, xô rò ra với tốc độ cố định.
- Làm mượt (smooth) lưu lượng, tránh burst.

## 3. Ví dụ Java: Rate Limiting đơn giản (Token Bucket)

### Yêu cầu:  
- Tối đa 5 request / 10 giây cho mỗi user.

```java
import java.time.Instant;
import java.util.concurrent.atomic.AtomicInteger;

public class TokenBucketRateLimiter {
    private final int capacity;          // Số token tối đa
    private final int refillTokens;      // Số token nạp mỗi chu kỳ
    private final long refillIntervalMs; // Chu kỳ nạp (ms)

    private AtomicInteger tokens;
    private long lastRefillTimestamp;

    public TokenBucketRateLimiter(int capacity, int refillTokens, long refillIntervalMs) {
        this.capacity = capacity;
        this.refillTokens = refillTokens;
        this.refillIntervalMs = refillIntervalMs;
        this.tokens = new AtomicInteger(capacity);
        this.lastRefillTimestamp = Instant.now().toEpochMilli();
    }

    private void refill() {
        long now = Instant.now().toEpochMilli();
        long elapsed = now - lastRefillTimestamp;
        if (elapsed > refillIntervalMs) {
            int newTokens = (int) (elapsed / refillIntervalMs) * refillTokens;
            int current = tokens.get();
            int updated = Math.min(capacity, current + newTokens);
            tokens.set(updated);
            lastRefillTimestamp = now;
        }
    }

    public synchronized boolean allowRequest() {
        refill();
        if (tokens.get() > 0) {
            tokens.decrementAndGet();
            return true;
        }
        return false;
    }

    public static void main(String[] args) throws InterruptedException {
        // 5 token, nạp 5 token mỗi 10 giây
        TokenBucketRateLimiter limiter = new TokenBucketRateLimiter(5, 5, 10_000);

        for (int i = 1; i <= 10; i++) {
            boolean allowed = limiter.allowRequest();
            System.out.println("Request " + i + " -> " + (allowed ? "ALLOWED" : "BLOCKED"));
            Thread.sleep(1000); // mỗi 1 giây gửi 1 request
        }
    }
}
```

### Kết quả (ví dụ output)
- 5 request đầu: `ALLOWED`  
- 5 request sau (trong cùng 10 giây): `BLOCKED`  
- Sau 10 giây, token được nạp lại, request lại được `ALLOWED`.

## 4. Ví dụ Java: Throttling đơn giản (Giới hạn số request đồng thời)

Giả sử chỉ cho phép tối đa 3 request xử lý cùng lúc, request dư sẽ phải chờ.

```java
import java.util.concurrent.Semaphore;

public class ThrottlingExample {
    private final Semaphore semaphore;

    public ThrottlingExample(int maxConcurrentRequests) {
        this.semaphore = new Semaphore(maxConcurrentRequests);
    }

    public void handleRequest(String name) {
        try {
            semaphore.acquire(); // chờ nếu đang quá số lượng cho phép
            System.out.println("Handling request: " + name + " at " + System.currentTimeMillis());
            Thread.sleep(2000); // giả lập xử lý mất 2 giây
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            semaphore.release();
        }
    }

    public static void main(String[] args) {
        ThrottlingExample throttler = new ThrottlingExample(3);

        for (int i = 1; i <= 6; i++) {
            final String reqName = "Req-" + i;
            new Thread(() -> throttler.handleRequest(reqName)).start();
        }
    }
}
```

### Hành vi (output mô tả)
- Tối đa 3 request được xử lý cùng lúc.
- 3 request còn lại phải chờ đến khi có slot trống (semaphore được release).

## 5. Khi nào dùng cái gì?

- **Rate limiting**:
  - API public, API trả phí.
  - Bảo vệ backend khỏi spam, brute-force, abuse.
- **Throttling**:
  - Khi tài nguyên backend có giới hạn (DB, service downstream).
  - Muốn hệ thống “chậm nhưng sống” thay vì “chết hẳn” khi quá tải.

Nếu bạn muốn, tôi có thể viết thêm ví dụ với Spring Boot (filter/interceptor) để áp dụng rate limiting/throttling cho REST API.
