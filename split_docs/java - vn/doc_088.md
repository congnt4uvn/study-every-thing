
Dưới đây là tóm tắt ngắn gọn về các loại NoSQL: key-value, document, column-family, graph, kèm ví dụ minh họa (giả lập bằng Java / JSON).  

---

## 1. Key-Value Store

**Ý tưởng chính**  
- Lưu dữ liệu dưới dạng cặp: `key` → `value`.  
- Rất đơn giản, hiệu năng cao, dễ scale.  
- Không có cấu trúc phức tạp, không query theo field bên trong value (trừ khi hệ thống hỗ trợ thêm).  
- Ví dụ: Redis, Amazon DynamoDB (chế độ key-value), Riak.

**Ví dụ (giả lập bằng Java Map)**

```java
import java.util.HashMap;
import java.util.Map;

public class KeyValueExample {
    public static void main(String[] args) {
        Map<String, String> store = new HashMap<>();

        // Insert
        store.put("user:1001", "{\"name\":\"An\",\"age\":25}");
        store.put("user:1002", "{\"name\":\"Bình\",\"age\":30}");

        // Get
        String user1001 = store.get("user:1001");
        System.out.println("user:1001 = " + user1001);

        // Output:
        // user:1001 = {"name":"An","age":25}
    }
}
```

## 2. Document Store

**Ý tưởng chính**  
- Lưu trữ tài liệu (document) dạng JSON, BSON, XML…  
- Mỗi document có cấu trúc linh hoạt, không cần schema cố định.  
- Có thể query theo field bên trong document (ví dụ: `name`, `age`).  
- Phù hợp cho ứng dụng web, API, dữ liệu bán cấu trúc.  
- Ví dụ: MongoDB, CouchDB.

**Ví dụ document (JSON)**

```json
{
  "_id": "user_1001",
  "name": "An",
  "age": 25,
  "address": {
    "city": "Hà Nội",
    "street": "Trần Duy Hưng"
  },
  "skills": ["Java", "Spring", "MongoDB"]
}
```

**Ví dụ query (MongoDB shell)**

```js
// Insert
db.users.insertOne({
  _id: "user_1001",
  name: "An",
  age: 25,
  city: "Hà Nội"
});

// Query theo field
db.users.find({ age: { $gt: 20 }, city: "Hà Nội" });
```

**Output (kết quả query)**

```json
[
  {
    "_id": "user_1001",
    "name": "An",
    "age": 25,
    "city": "Hà Nội"
  }
]
```

## 3. Column-Family Store (Wide-Column)

**Ý tưởng chính**  
- Dữ liệu được tổ chức theo: **key hàng (row key)** → **các cột (columns)**, nhóm thành **column family**.  
- Mỗi hàng có thể có các cột khác nhau (rất linh hoạt).  
- Tối ưu cho dữ liệu lớn, truy vấn theo key và range, phân tán trên nhiều node.  
- Ví dụ: Apache Cassandra, HBase.

**Mô hình đơn giản**

- Row key: `user_1001`  
- Column family: `profile`  
- Columns:
  - `name` = "An"
  - `age` = 25
  - `city` = "Hà Nội"

**Ví dụ (CQL – Cassandra Query Language)**

```sql
CREATE TABLE users (
    user_id text PRIMARY KEY,
    name text,
    age int,
    city text
);

// Insert
INSERT INTO users (user_id, name, age, city)
VALUES ('user_1001', 'An', 25, 'Hà Nội');

// Query
SELECT * FROM users WHERE user_id = 'user_1001';
```

**Output**

```text
 user_id   | name | age | city
-----------+------+-----+--------
 user_1001 | An   |  25 | Hà Nội
```

## 4. Graph Database

**Ý tưởng chính**  
- Lưu trữ dữ liệu dưới dạng **đồ thị**:  
  - **Node (đỉnh)**: thực thể (User, Product, City, …)  
  - **Edge (cạnh)**: quan hệ giữa các node (FRIEND_OF, LIVES_IN, BOUGHT, …)  
- Tối ưu cho dữ liệu có quan hệ phức tạp, nhiều cấp (mạng xã hội, recommendation, đường đi ngắn nhất…).  
- Ví dụ: Neo4j, JanusGraph.

**Ví dụ mô hình**

- Node: `User {name: "An"}`  
- Node: `User {name: "Bình"}`  
- Edge: `(An)-[:FRIEND_OF]->(Bình)`

**Ví dụ query (Cypher – Neo4j)**

```cypher
// Tạo node
CREATE (a:User {name: 'An'})
CREATE (b:User {name: 'Bình'})
CREATE (a)-[:FRIEND_OF]->(b);

// Tìm bạn của An
MATCH (a:User {name: 'An'})-[:FRIEND_OF]->(friend)
RETURN friend.name;
```

**Output**

```text
 friend.name
-------------
 Bình
```

## So sánh nhanh

- **Key-Value**  
  - Ưu: Đơn giản, rất nhanh, dễ scale.  
  - Nhược: Khó query phức tạp, không query sâu vào value (trừ khi hỗ trợ thêm).

- **Document**  
  - Ưu: Linh hoạt, hỗ trợ query theo field, phù hợp API/web.  
  - Nhược: Join phức tạp, không mạnh bằng SQL cho quan hệ nhiều bảng.

- **Column-Family**  
  - Ưu: Tối ưu cho big data, ghi/đọc nhanh, phân tán tốt.  
  - Nhược: Mô hình dữ liệu phức tạp hơn, cần thiết kế theo pattern truy vấn.

- **Graph**  
  - Ưu: Rất mạnh cho dữ liệu quan hệ phức tạp, truy vấn theo đường đi, kết nối.  
  - Nhược: Không phù hợp cho dữ liệu dạng bảng đơn giản, có thể khó scale ngang trong một số trường hợp.
