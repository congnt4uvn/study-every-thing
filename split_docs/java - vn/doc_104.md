
Dưới đây là tóm tắt ngắn gọn về 2 chủ đề trong System Design: **Idempotency** và **Eventual Consistency**, kèm ví dụ code (Java) và input/output minh họa.

---

## 1. Idempotency (Tính bất biến khi gọi lặp lại)

**Ý tưởng chính**  
- Một **operation idempotent**: gọi 1 lần hay nhiều lần với cùng dữ liệu **kết quả cuối cùng như nhau**.  
- Rất quan trọng trong:
  - API thanh toán (tránh trừ tiền 2 lần)
  - Hệ thống retry (gọi lại khi timeout, network lỗi)
  - Message queue (tránh xử lý trùng)

**Cách làm phổ biến**

1. **Idempotency Key**  
   - Client gửi một `idempotencyKey` duy nhất cho mỗi request “logic” (ví dụ: 1 giao dịch thanh toán).
   - Server lưu kết quả theo key này.  
   - Nếu nhận lại request với cùng key → trả lại kết quả cũ, không xử lý lại.

### Ví dụ Java: API thanh toán idempotent

```java
class PaymentRequest {
    String idempotencyKey;
    String userId;
    int amount;
    // getters/setters
}

class PaymentResponse {
    String status;   // "SUCCESS", "FAILED"
    String transactionId;
    String message;
}

class PaymentService {
    // Giả sử là storage in-memory, thực tế dùng DB/Redis
    private final Map<String, PaymentResponse> store = new ConcurrentHashMap<>();

    public PaymentResponse processPayment(PaymentRequest request) {
        // 1. Nếu đã xử lý key này → trả lại kết quả cũ
        if (store.containsKey(request.getIdempotencyKey())) {
            return store.get(request.getIdempotencyKey());
        }

        // 2. Xử lý thanh toán (giả lập)
        String transactionId = UUID.randomUUID().toString();
        PaymentResponse response = new PaymentResponse();
        response.setStatus("SUCCESS");
        response.setTransactionId(transactionId);
        response.setMessage("Payment processed");

        // 3. Lưu kết quả theo idempotencyKey
        store.put(request.getIdempotencyKey(), response);

        return response;
    }
}
```

**Input/Output minh họa**

- Request 1:

```json
POST /payments
{
  "idempotencyKey": "order-123",
  "userId": "u1",
  "amount": 100000
}
```

- Response 1:

```json
{
  "status": "SUCCESS",
  "transactionId": "tx-abc-1",
  "message": "Payment processed"
}
```

- Request 2 (gửi lại y hệt, do client retry):

```json
POST /payments
{
  "idempotencyKey": "order-123",
  "userId": "u1",
  "amount": 100000
}
```

- Response 2 (trả lại kết quả cũ, không trừ tiền lần nữa):

```json
{
  "status": "SUCCESS",
  "transactionId": "tx-abc-1",
  "message": "Payment processed"
}
```

**Kết luận**: Idempotency giúp hệ thống an toàn khi retry, tránh side-effect lặp lại.

## 2. Eventual Consistency (Nhất quán dần)

**Ý tưởng chính**  
- Trong hệ thống phân tán (microservices, nhiều DB), rất khó đảm bảo **strong consistency** (tất cả nơi luôn giống nhau ngay lập tức).
- **Eventual consistency**: dữ liệu có thể **tạm thời không đồng bộ**, nhưng **sau một thời gian** (khi các update được propagate) → các bản sao sẽ **nhất quán**.

**Khi dùng**  
- Hệ thống lớn, phân tán, cần:
  - High availability
  - Partition tolerance
- Ví dụ:
  - User service + Order service + Notification service
  - CQRS, Event Sourcing
  - Hệ thống dùng message queue (Kafka, RabbitMQ)

### Ví dụ Java: Cập nhật số dư tài khoản với eventual consistency

Giả sử có 2 service:

- `OrderService`: tạo order, gửi event “OrderCreated”
- `WalletService`: lắng nghe event, trừ tiền user sau đó

#### OrderService (tạo order và publish event)

```java
class OrderCreatedEvent {
    String orderId;
    String userId;
    int amount;
    // getters/setters
}

class OrderService {
    private final MessageBroker broker; // Kafka/RabbitMQ giả lập

    public String createOrder(String userId, int amount) {
        String orderId = UUID.randomUUID().toString();
        // Lưu order vào DB (giả lập)
        System.out.println("Order created: " + orderId);

        // Gửi event
        OrderCreatedEvent event = new OrderCreatedEvent();
        event.setOrderId(orderId);
        event.setUserId(userId);
        event.setAmount(amount);

        broker.publish("order-created", event);
        return orderId;
    }
}
```

#### WalletService (consume event và trừ tiền sau)

```java
class WalletService {
    private final Map<String, Integer> balance = new ConcurrentHashMap<>();

    public WalletService() {
        balance.put("u1", 200000); // user u1 có 200k
    }

    public void onOrderCreated(OrderCreatedEvent event) {
        String userId = event.getUserId();
        int amount = event.getAmount();
        int current = balance.getOrDefault(userId, 0);

        if (current >= amount) {
            balance.put(userId, current - amount);
            System.out.println("Wallet updated for user " + userId +
                               ", new balance = " + balance.get(userId));
        } else {
            System.out.println("Not enough balance for user " + userId);
        }
    }

    public int getBalance(String userId) {
        return balance.getOrDefault(userId, 0);
    }
}
```

#### MessageBroker giả lập

```java
class MessageBroker {
    private Consumer<OrderCreatedEvent> orderCreatedHandler;

    public void subscribe(String topic, Consumer<OrderCreatedEvent> handler) {
        if ("order-created".equals(topic)) {
            this.orderCreatedHandler = handler;
        }
    }

    public void publish(String topic, OrderCreatedEvent event) {
        // Giả lập delay mạng
        new Thread(() -> {
            try {
                Thread.sleep(2000); // delay 2s
            } catch (InterruptedException ignored) {}
            if (orderCreatedHandler != null) {
                orderCreatedHandler.accept(event);
            }
        }).start();
    }
}
```

#### Chạy thử (Input/Output)

```java
public class Demo {
    public static void main(String[] args) throws Exception {
        MessageBroker broker = new MessageBroker();
        WalletService walletService = new WalletService();
        broker.subscribe("order-created", walletService::onOrderCreated);

        OrderService orderService = new OrderService(broker);

        System.out.println("Balance ban đầu: " + walletService.getBalance("u1"));
        String orderId = orderService.createOrder("u1", 50000);
        System.out.println("Đã tạo order: " + orderId);

        // Ngay sau khi tạo order, số dư chưa trừ (chưa nhất quán)
        System.out.println("Balance ngay sau khi tạo order: " + walletService.getBalance("u1"));

        // Chờ 3s cho event xử lý
        Thread.sleep(3000);
        System.out.println("Balance sau 3s: " + walletService.getBalance("u1"));
    }
}
```

**Output (minh họa)**

```text
Balance ban đầu: 200000
Order created: 3f1c... (orderId)
Đã tạo order: 3f1c...
Balance ngay sau khi tạo order: 200000
Wallet updated for user u1, new balance = 150000
Balance sau 3s: 150000
```

- Ngay sau khi tạo order: số dư **chưa trừ** → dữ liệu **tạm thời không nhất quán**.
- Sau vài giây: event được xử lý, số dư **được trừ** → hệ thống **nhất quán dần (eventually consistent)**.

## Tóm tắt ngắn

- **Idempotency**:  
  - Gọi 1 hay nhiều lần → kết quả cuối cùng như nhau.  
  - Dùng `idempotencyKey`, lưu kết quả để tránh xử lý trùng.  
  - Quan trọng trong retry, thanh toán, message processing.

- **Eventual Consistency**:  
  - Dữ liệu có thể lệch tạm thời giữa các service/DB.  
  - Sau một thời gian, khi các event/update được propagate → hệ thống trở nên nhất quán.  
  - Thường dùng trong microservices, message queue, hệ thống phân tán lớn.
