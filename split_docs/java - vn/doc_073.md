
Dưới đây là tóm tắt ngắn gọn về **Spring Data & Persistence – Transactions** với **@Transactional, propagation, isolation levels**, kèm ví dụ code và input/output.

---

## 1. @Transactional là gì?

- Annotation dùng để đánh dấu **phương thức** hoặc **class** sẽ chạy trong **transaction**.
- Spring sẽ:
  - Mở transaction trước khi chạy method
  - Commit nếu không lỗi
  - Rollback nếu có exception (mặc định: RuntimeException)

### Ví dụ cơ bản

```java
@Service
public class UserService {

    @Autowired
    private UserRepository userRepository;

    @Transactional
    public void createUser(String name) {
        User user = new User();
        user.setName(name);
        userRepository.save(user);
        // nếu có RuntimeException xảy ra ở đây -> rollback
    }
}
```

**Input (gọi service):**

```java
userService.createUser("An");
```

**Output (trong DB):**

- Nếu không lỗi: có 1 record user tên "An".
- Nếu trong method ném RuntimeException: không có user nào được lưu (rollback).

## 2. Propagation (Mức lan truyền transaction)

Xác định **transaction hiện tại sẽ được dùng như thế nào** khi method được gọi bên trong một method khác.

Các kiểu phổ biến:

1. `REQUIRED` (mặc định)
   - Nếu đã có transaction → dùng chung
   - Nếu chưa có → tạo mới

2. `REQUIRES_NEW`
   - Luôn tạo **transaction mới**
   - Tạm dừng transaction hiện tại (nếu có)

3. `MANDATORY`
   - Bắt buộc phải có transaction sẵn
   - Nếu không có → ném exception

4. `SUPPORTS`
   - Có transaction thì dùng, không có thì chạy **không transaction**

5. `NOT_SUPPORTED`
   - Luôn chạy **ngoài transaction**
   - Tạm dừng transaction hiện tại

6. `NEVER`
   - Không được phép có transaction
   - Nếu đang có → ném exception

### Ví dụ: REQUIRED vs REQUIRES_NEW

```java
@Service
public class OrderService {

    @Autowired
    private PaymentService paymentService;

    @Autowired
    private OrderRepository orderRepository;

    @Transactional // propagation mặc định: REQUIRED
    public void placeOrder() {
        Order order = new Order();
        order.setStatus("CREATED");
        orderRepository.save(order);

        paymentService.processPayment(order);

        // nếu processPayment ném RuntimeException -> toàn bộ placeOrder rollback
    }
}

@Service
public class PaymentService {

    @Transactional(propagation = Propagation.REQUIRED)
    public void processPayment(Order order) {
        // dùng chung transaction với placeOrder
        // nếu lỗi -> rollback cả order + payment
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void processPaymentInNewTx(Order order) {
        // transaction mới, độc lập với placeOrder
        // nếu lỗi -> rollback payment, nhưng order có thể vẫn commit
    }
}
```

**Tình huống:**

```java
orderService.placeOrder();
```

- Nếu `processPayment()` dùng `REQUIRED` và bị lỗi:
  - Order không được lưu (rollback cùng transaction).
- Nếu `processPaymentInNewTx()` dùng `REQUIRES_NEW` và bị lỗi:
  - Payment rollback, nhưng `placeOrder()` có thể vẫn commit (tùy cách xử lý exception).

## 3. Isolation Levels (Mức cô lập transaction)

Quy định **mức độ “nhìn thấy” dữ liệu** giữa các transaction đồng thời, để tránh các vấn đề như:

- Dirty read (đọc dữ liệu chưa commit)
- Non-repeatable read (đọc cùng 1 record 2 lần cho kết quả khác nhau)
- Phantom read (số lượng record thay đổi giữa 2 lần query)

Các mức phổ biến:

1. `READ_UNCOMMITTED`
   - Cho phép dirty read
   - Ít an toàn nhất

2. `READ_COMMITTED`
   - Chỉ đọc dữ liệu đã commit
   - Tránh dirty read
   - Có thể bị non-repeatable read, phantom read

3. `REPEATABLE_READ`
   - Cùng 1 record, đọc nhiều lần trong 1 transaction → kết quả giống nhau
   - Tránh dirty read, non-repeatable read
   - Có thể vẫn bị phantom read (tùy DB)

4. `SERIALIZABLE`
   - Mức an toàn cao nhất
   - Gần như chạy tuần tự
   - Tránh cả phantom read
   - Hiệu năng thấp hơn

### Ví dụ dùng isolation

```java
@Service
public class AccountService {

    @Autowired
    private AccountRepository accountRepository;

    @Transactional(isolation = Isolation.READ_COMMITTED)
    public BigDecimal getBalance(Long accountId) {
        Account acc = accountRepository.findById(accountId).orElseThrow();
        return acc.getBalance();
    }

    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void transfer(Long fromId, Long toId, BigDecimal amount) {
        Account from = accountRepository.findById(fromId).orElseThrow();
        Account to = accountRepository.findById(toId).orElseThrow();

        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));

        accountRepository.save(from);
        accountRepository.save(to);
    }
}
```

**Input:**

```java
accountService.transfer(1L, 2L, new BigDecimal("100.00"));
```

**Output (trong DB):**

- Tài khoản 1: trừ 100
- Tài khoản 2: cộng 100
- Nếu có lỗi giữa chừng → cả hai thay đổi bị rollback.

## 4. Một số lưu ý quan trọng

- `@Transactional` chỉ hoạt động với:
  - Method **public** (theo mặc định proxy)
  - Gọi từ **bean khác** (không phải self-invocation)
- Mặc định rollback cho `RuntimeException` và `Error`.
  - Có thể cấu hình: `rollbackFor = Exception.class`, `noRollbackFor = ...`

### Ví dụ cấu hình rollback

```java
@Transactional(rollbackFor = Exception.class)
public void doSomething() throws Exception {
    // checked exception cũng sẽ rollback
}
```

---

**Tóm tắt ngắn:**

- `@Transactional` quản lý mở/đóng transaction, commit/rollback tự động.
- `propagation` quyết định cách method lồng nhau dùng chung hay tách transaction.
- `isolation` quyết định mức độ an toàn khi nhiều transaction truy cập dữ liệu đồng thời.
