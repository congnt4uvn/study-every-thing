
Dưới đây là tóm tắt ngắn gọn về **Parallel Streams trong Java** (khi nên dùng, rủi ro, ví dụ code):

---

## 1. Parallel Stream là gì?

- Là cách chạy stream theo **nhiều luồng (threads)** để tận dụng **đa nhân CPU**.
- Thay vì `.stream()`, dùng `.parallelStream()` hoặc `.stream().parallel()`.

```java
List<Integer> list = List.of(1, 2, 3, 4, 5);

// Stream tuần tự
int sum1 = list.stream()
               .mapToInt(i -> i)
               .sum();

// Stream song song
int sum2 = list.parallelStream()
               .mapToInt(i -> i)
               .sum();

System.out.println(sum1); // Output: 15
System.out.println(sum2); // Output: 15
```

---

## 2. Khi nên dùng Parallel Stream

Nên dùng khi:

1. **Dữ liệu lớn** (hàng chục nghìn, trăm nghìn phần tử trở lên).
2. **Tác vụ nặng CPU** (tính toán phức tạp, xử lý số lớn, mã hóa, xử lý ảnh…).
3. Máy có **nhiều core** (4 core trở lên).
4. Không yêu cầu **thứ tự kết quả** nghiêm ngặt (hoặc dùng các toán tử hỗ trợ).

Ví dụ: tính tổng số lượng lớn với xử lý nặng:

```java
import java.util.stream.LongStream;

public class HeavyCalc {
    private static long heavyOperation(long n) {
        // Giả lập tác vụ nặng
        for (int i = 0; i < 1_000; i++) {
            Math.sqrt(n + i);
        }
        return n * 2;
    }

    public static void main(String[] args) {
        long start, end;

        // Tuần tự
        start = System.currentTimeMillis();
        long sumSequential = LongStream.rangeClosed(1, 100_000)
                .map(HeavyCalc::heavyOperation)
                .sum();
        end = System.currentTimeMillis();
        System.out.println("Sequential sum = " + sumSequential);
        System.out.println("Sequential time = " + (end - start) + " ms");

        // Song song
        start = System.currentTimeMillis();
        long sumParallel = LongStream.rangeClosed(1, 100_000)
                .parallel()
                .map(HeavyCalc::heavyOperation)
                .sum();
        end = System.currentTimeMillis();
        System.out.println("Parallel sum = " + sumParallel);
        System.out.println("Parallel time = " + (end - start) + " ms");
    }
}
```

**Kỳ vọng**: `sumSequential == sumParallel`, nhưng thời gian `Parallel` thường nhanh hơn (tùy máy).

## 3. Khi không nên dùng Parallel Stream

Tránh dùng khi:

1. **Dữ liệu nhỏ** → overhead tạo task > lợi ích.
2. Tác vụ **I/O-bound** (gọi API, đọc file, DB) → thread chờ I/O, không tận dụng CPU tốt.
3. Cần **thứ tự chính xác** (ví dụ xử lý log theo thứ tự).
4. Môi trường **server** đã có thread pool riêng (ví dụ trong web app) → dễ gây tranh chấp tài nguyên.
5. Dùng trong **UI (JavaFX, Swing)** → không được cập nhật UI từ nhiều thread.

Ví dụ: xử lý danh sách nhỏ, parallel có thể chậm hơn:

```java
List<Integer> smallList = List.of(1, 2, 3, 4, 5);

// Tuần tự
smallList.stream()
         .map(i -> i * 2)
         .forEach(System.out::println);

// Song song (overkill)
smallList.parallelStream()
         .map(i -> i * 2)
         .forEach(System.out::println);
```

## 4. Các “pitfall” (rủi ro) chính

### 4.1. Không thread-safe

Không được cập nhật biến dùng chung mà không đồng bộ.

```java
// Ví dụ SAI: race condition
List<Integer> list = List.of(1, 2, 3, 4, 5);
int[] sum = {0};

list.parallelStream().forEach(i -> sum[0] += i);

System.out.println(sum[0]); // Output: có thể KHÔNG phải 15
```

Cách đúng: dùng reduce hoặc collector thread-safe:

```java
int sumCorrect = list.parallelStream()
                     .mapToInt(i -> i)
                     .sum();
System.out.println(sumCorrect); // Output: 15
```

---

### 4.2. Thứ tự không đảm bảo

`forEach` trên parallel stream **không đảm bảo thứ tự**.

```java
List<Integer> list = List.of(1, 2, 3, 4, 5);

System.out.println("forEach (parallel):");
list.parallelStream()
    .forEach(System.out::println); // Thứ tự có thể: 3 1 5 2 4 ...

System.out.println("forEachOrdered (parallel):");
list.parallelStream()
    .forEachOrdered(System.out::println); // Đảm bảo: 1 2 3 4 5
```

---

### 4.3. Deadlock / Starvation khi kết hợp với API khác

- Parallel stream dùng **ForkJoinPool.commonPool()**.
- Nếu bên trong lại dùng các API cũng dùng common pool hoặc chờ kết quả đồng bộ → có thể deadlock/starvation.

---

### 4.4. Chi phí chia nhỏ (splitting) & kết hợp (merging)

- Một số cấu trúc dữ liệu **không chia nhỏ tốt** (ví dụ `LinkedList`, `Iterator` tùy chỉnh).
- Khi đó parallel stream có thể **chậm hơn** stream thường.

---

## 5. Ví dụ: so sánh sequential vs parallel

```java
import java.util.List;
import java.util.stream.IntStream;

public class CompareParallel {
    public static void main(String[] args) {
        List<Integer> data = IntStream.range(0, 1_000_000)
                                      .boxed()
                                      .toList();

        long start, end;

        // Tuần tự
        start = System.currentTimeMillis();
        long countSeq = data.stream()
                .filter(i -> i % 2 == 0)
                .count();
        end = System.currentTimeMillis();
        System.out.println("Sequential count = " + countSeq);
        System.out.println("Sequential time = " + (end - start) + " ms");

        // Song song
        start = System.currentTimeMillis();
        long countPar = data.parallelStream()
                .filter(i -> i % 2 == 0)
                .count();
        end = System.currentTimeMillis();
        System.out.println("Parallel count = " + countPar);
        System.out.println("Parallel time = " + (end - start) + " ms");
    }
}
```

**Input**: danh sách 0..999999  
**Output (ví dụ)**:
- `Sequential count = 500000`
- `Parallel count = 500000`
- Thời gian: tùy máy, parallel thường nhanh hơn nếu đủ core.

---

## 6. Tóm tắt ý chính

- Parallel stream giúp **tận dụng đa nhân CPU** cho tác vụ **CPU-bound, dữ liệu lớn**.
- Không nên dùng cho **dữ liệu nhỏ, I/O-bound, yêu cầu thứ tự chặt chẽ**.
- Tránh **trạng thái chia sẻ (shared mutable state)**; dùng `map`, `reduce`, `collect` thuần túy.
- Cẩn thận với **thứ tự**, **thread-safety**, và **môi trường server/UI**.
- Luôn **đo đạc (benchmark)** trước khi quyết định dùng parallel stream.
