
Dưới đây là tóm tắt ngắn gọn về SOLID trong thiết kế hệ thống & kiến trúc (Java), kèm ví dụ code và input/output đơn giản.

---

## 1. S – Single Responsibility Principle (SRP)  
**Mỗi class chỉ nên có một lý do để thay đổi** → chỉ làm *một* nhiệm vụ chính.

**Ví dụ sai:**

```java
class Invoice {
    private double amount;

    public Invoice(double amount) {
        this.amount = amount;
    }

    // Tính tổng tiền
    public double calculateTotal() {
        return amount * 1.1; // +10% tax
    }

    // In hóa đơn ra console
    public void printInvoice() {
        System.out.println("Invoice total: " + calculateTotal());
    }

    // Lưu hóa đơn vào DB
    public void saveToDatabase() {
        // code lưu DB
    }
}
```

Class `Invoice` vừa tính toán, vừa in, vừa lưu DB → nhiều trách nhiệm.

**Ví dụ đúng (tách trách nhiệm):**

```java
class Invoice {
    private double amount;

    public Invoice(double amount) {
        this.amount = amount;
    }

    public double calculateTotal() {
        return amount * 1.1;
    }
}

class InvoicePrinter {
    public void print(Invoice invoice) {
        System.out.println("Invoice total: " + invoice.calculateTotal());
    }
}

class InvoiceRepository {
    public void save(Invoice invoice) {
        // code lưu DB
        System.out.println("Saved invoice with total: " + invoice.calculateTotal());
    }
}
```

**Input/Output demo:**

```java
public class Main {
    public static void main(String[] args) {
        Invoice invoice = new Invoice(100);
        InvoicePrinter printer = new InvoicePrinter();
        InvoiceRepository repo = new InvoiceRepository();

        printer.print(invoice);
        repo.save(invoice);
    }
}
```

Output:

```
Invoice total: 110.0
Saved invoice with total: 110.0
```

## 2. O – Open/Closed Principle (OCP)  
**Mở rộng được nhưng hạn chế sửa đổi** → thêm chức năng bằng cách *mở rộng* class, không sửa code cũ.

**Ví dụ: tính chiết khấu cho nhiều loại khách hàng**

```java
interface DiscountPolicy {
    double applyDiscount(double amount);
}

class RegularDiscount implements DiscountPolicy {
    public double applyDiscount(double amount) {
        return amount * 0.95; // giảm 5%
    }
}

class VipDiscount implements DiscountPolicy {
    public double applyDiscount(double amount) {
        return amount * 0.90; // giảm 10%
    }
}

class CheckoutService {
    private final DiscountPolicy discountPolicy;

    public CheckoutService(DiscountPolicy discountPolicy) {
        this.discountPolicy = discountPolicy;
    }

    public double checkout(double amount) {
        return discountPolicy.applyDiscount(amount);
    }
}
```

**Input/Output demo:**

```java
public class Main {
    public static void main(String[] args) {
        CheckoutService regularService = new CheckoutService(new RegularDiscount());
        CheckoutService vipService = new CheckoutService(new VipDiscount());

        System.out.println(regularService.checkout(100)); // 95.0
        System.out.println(vipService.checkout(100));     // 90.0
    }
}
```

Output:

```
95.0
90.0
```

Muốn thêm loại chiết khấu mới → tạo class mới implement `DiscountPolicy`, không sửa `CheckoutService`.

## 3. L – Liskov Substitution Principle (LSP)  
**Class con phải thay thế được class cha mà không làm sai hành vi**.

**Ví dụ đúng:**

```java
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }

    public int area() { return width * height; }
}

class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width;
    }

    @Override
    public void setHeight(int height) {
        this.height = height;
        this.width = height;
    }
}
```

Ví dụ này thường được dùng để minh họa *vi phạm* LSP (vì code dùng `Rectangle` có thể bị sai khi truyền `Square`).  
Cách tốt hơn là tách `Rectangle` và `Square` thành hai loại riêng, hoặc dùng interface hình học chung.

Ý chính: khi thay `Rectangle r = new Rectangle()` bằng `Rectangle r = new Square()`, chương trình không được “hành xử bất ngờ”.

## 4. I – Interface Segregation Principle (ISP)  
**Không ép client phụ thuộc vào interface mà nó không dùng** → chia nhỏ interface.

**Ví dụ sai:**

```java
interface Worker {
    void work();
    void eat();
}

class Robot implements Worker {
    public void work() {
        System.out.println("Robot working");
    }

    public void eat() {
        // Robot không ăn, nhưng vẫn phải implement
        throw new UnsupportedOperationException("Robot doesn't eat");
    }
}
```

**Ví dụ đúng (tách interface):**

```java
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

class Human implements Workable, Eatable {
    public void work() {
        System.out.println("Human working");
    }

    public void eat() {
        System.out.println("Human eating");
    }
}

class Robot implements Workable {
    public void work() {
        System.out.println("Robot working");
    }
}
```

**Input/Output demo:**

```java
public class Main {
    public static void main(String[] args) {
        Workable human = new Human();
        Workable robot = new Robot();

        human.work();
        robot.work();
    }
}
```

Output:

```
Human working
Robot working
```

## 5. D – Dependency Inversion Principle (DIP)  
**Phụ thuộc vào abstraction (interface), không phụ thuộc vào implementation cụ thể.**

**Ví dụ:**

```java
interface MessageSender {
    void send(String message);
}

class EmailSender implements MessageSender {
    public void send(String message) {
        System.out.println("Sending email: " + message);
    }
}

class SmsSender implements MessageSender {
    public void send(String message) {
        System.out.println("Sending SMS: " + message);
    }
}

class NotificationService {
    private final MessageSender sender;

    // Inject qua constructor
    public NotificationService(MessageSender sender) {
        this.sender = sender;
    }

    public void notifyUser(String message) {
        sender.send(message);
    }
}
```

**Input/Output demo:**

```java
public class Main {
    public static void main(String[] args) {
        NotificationService emailService = new NotificationService(new EmailSender());
        NotificationService smsService = new NotificationService(new SmsSender());

        emailService.notifyUser("Hello via Email");
        smsService.notifyUser("Hello via SMS");
    }
}
```

Output:

```
Sending email: Hello via Email
Sending SMS: Hello via SMS
```

`NotificationService` chỉ biết `MessageSender` (abstraction), không phụ thuộc trực tiếp `EmailSender` hay `SmsSender`.

### Tóm tắt ngắn gọn SOLID

- **S (Single Responsibility)**: Mỗi class làm một việc.
- **O (Open/Closed)**: Dễ mở rộng, hạn chế sửa code cũ.
- **L (Liskov Substitution)**: Class con thay thế được class cha mà không phá vỡ logic.
- **I (Interface Segregation)**: Nhiều interface nhỏ, chuyên biệt; không ép implement phương thức không dùng.
- **D (Dependency Inversion)**: Phụ thuộc vào interface/abstraction, không phụ thuộc vào class cụ thể.

Nếu bạn muốn, tôi có thể viết thêm một ví dụ nhỏ kết hợp nhiều nguyên tắc SOLID trong một mini-project (ví dụ hệ thống đặt hàng đơn giản).
