
Dưới đây là tóm tắt ngắn gọn về các khái niệm: race condition, deadlock, livelock, starvation trong Java, kèm ví dụ mã và input/output minh họa.

## 1. Race Condition (Điều kiện tranh chấp)

**Ý chính**  
- Xảy ra khi nhiều thread cùng truy cập và ghi lên **dữ liệu dùng chung** mà không có đồng bộ (synchronization) đúng cách.  
- Kết quả cuối cùng **không đoán trước được**, phụ thuộc vào thứ tự chạy của thread.

**Ví dụ Java**

```java
public class RaceConditionDemo {
    private static int counter = 0;

    public static void main(String[] args) throws InterruptedException {
        Runnable task = () -> {
            for (int i = 0; i < 1000; i++) {
                counter++; // không đồng bộ
            }
        };

        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println("Counter = " + counter);
    }
}
```

**Kỳ vọng (nếu không có race condition)**  
- Input: 2 thread, mỗi thread tăng 1000 lần  
- Output mong muốn: `Counter = 2000`

**Thực tế**  
- Output có thể là: `Counter = 1783`, `Counter = 1950`, `Counter = 2000`, … (không ổn định).

**Cách khắc phục (ví dụ)**

```java
private static int counter = 0;

private static synchronized void increment() {
    counter++;
}
```

## 2. Deadlock (Kẹt khóa)

**Ý chính**  
- Hai (hoặc nhiều) thread **chờ nhau** giải phóng tài nguyên, dẫn đến **tất cả đều đứng yên**.  
- Thường xảy ra khi thread A giữ lock1 và chờ lock2, trong khi thread B giữ lock2 và chờ lock1.

**Ví dụ Java**

```java
public class DeadlockDemo {
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (LOCK_A) {
                System.out.println("T1: Đã giữ LOCK_A");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                System.out.println("T1: Đang chờ LOCK_B");
                synchronized (LOCK_B) {
                    System.out.println("T1: Đã giữ LOCK_B");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (LOCK_B) {
                System.out.println("T2: Đã giữ LOCK_B");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                System.out.println("T2: Đang chờ LOCK_A");
                synchronized (LOCK_A) {
                    System.out.println("T2: Đã giữ LOCK_A");
                }
            }
        });

        t1.start();
        t2.start();
    }
}
```

**Output điển hình**

```text
T1: Đã giữ LOCK_A
T2: Đã giữ LOCK_B
T1: Đang chờ LOCK_B
T2: Đang chờ LOCK_A
// Sau đó chương trình treo, không in thêm gì nữa (deadlock)
```

**Cách tránh (ý tưởng)**  
- Luôn khóa theo **cùng một thứ tự** (lock ordering).  
- Dùng `tryLock` với timeout (trong `ReentrantLock`).

## 3. Livelock (Kẹt sống)

**Ý chính**  
- Các thread **không bị đứng yên** (vẫn chạy, vẫn đổi trạng thái) nhưng **không tiến triển** được công việc.  
- Thường do các thread liên tục “nhường nhau” hoặc thay đổi chiến lược để tránh xung đột, nhưng lại gây vòng lặp vô tận.

**Ví dụ minh họa đơn giản**

```java
public class LivelockDemo {
    static class Resource {
        private volatile boolean taken = false;

        public synchronized boolean tryUse(String name) {
            if (!taken) {
                taken = true;
                System.out.println(name + " đang dùng resource");
                return true;
            }
            return false;
        }

        public synchronized void release(String name) {
            taken = false;
            System.out.println(name + " nhả resource");
        }
    }

    public static void main(String[] args) {
        Resource resource = new Resource();

        Runnable worker = () -> {
            String name = Thread.currentThread().getName();
            while (true) {
                if (resource.tryUse(name)) {
                    try { Thread.sleep(50); } catch (InterruptedException e) {}
                    resource.release(name);
                    // “Lịch sự” nhường cho thread khác, nhưng cả hai làm giống nhau
                    try { Thread.sleep(50); } catch (InterruptedException e) {}
                } else {
                    // Không dùng được thì lại nhường tiếp
                    try { Thread.sleep(10); } catch (InterruptedException e) {}
                }
            }
        };

        new Thread(worker, "T1").start();
        new Thread(worker, "T2").start();
    }
}
```

**Đặc điểm output**  
- Hai thread liên tục in log “đang dùng resource” / “nhả resource”, nhưng **không bao giờ hoàn thành** một mục tiêu cụ thể (ví dụ: xử lý xong N tác vụ).  
- Chương trình chạy mãi mà không tiến triển thực sự.

## 4. Starvation (Đói tài nguyên)

**Ý chính**  
- Một thread **hiếm khi hoặc không bao giờ** được chạy vì tài nguyên (CPU, lock) luôn bị các thread khác chiếm.  
- Thường xảy ra khi có **ưu tiên không công bằng** (priority, lock không fair).

**Ví dụ Java (minh họa)**

```java
public class StarvationDemo {
    private static final Object LOCK = new Object();
    private static boolean running = true;

    public static void main(String[] args) {
        // Thread "tham lam" luôn giữ lock lâu
        Thread greedy = new Thread(() -> {
            while (running) {
                synchronized (LOCK) {
                    System.out.println("Greedy: giữ lock");
                    try { Thread.sleep(100); } catch (InterruptedException e) {}
                }
                try { Thread.sleep(10); } catch (InterruptedException e) {}
            }
        });

        // Thread "yếu thế" khó có cơ hội
        Thread poor = new Thread(() -> {
            while (running) {
                synchronized (LOCK) {
                    System.out.println("Poor: cuối cùng cũng giữ được lock");
                }
                try { Thread.sleep(200); } catch (InterruptedException e) {}
            }
        });

        greedy.setPriority(Thread.MAX_PRIORITY);
        poor.setPriority(Thread.MIN_PRIORITY);

        greedy.start();
        poor.start();

        try { Thread.sleep(3000); } catch (InterruptedException e) {}
        running = false;
    }
}
```

**Output điển hình**  
- Rất nhiều dòng: `Greedy: giữ lock`  
- Rất ít hoặc gần như không có: `Poor: cuối cùng cũng giữ được lock`  
→ Thread “poor” bị **starvation**.

**Cách giảm starvation**  
- Dùng lock **công bằng**: `new ReentrantLock(true)`  
- Hạn chế dùng `Thread.setPriority` để ưu tiên quá chênh lệch.

---

## Tóm tắt nhanh

- **Race condition**: Nhiều thread truy cập/ghi dữ liệu chung không đồng bộ → kết quả sai, không ổn định.  
- **Deadlock**: Các thread chờ nhau giải phóng lock → tất cả đứng yên.  
- **Livelock**: Các thread vẫn chạy, liên tục thay đổi trạng thái, nhưng không tiến triển công việc.  
- **Starvation**: Một số thread hiếm khi được chạy vì bị “lấn át” tài nguyên bởi thread khác.

Nếu bạn muốn, tôi có thể viết thêm ví dụ với `ReentrantLock`, `synchronized`, hoặc cách debug các vấn đề này trong thực tế.
