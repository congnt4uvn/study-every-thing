
Dưới đây là tóm tắt ngắn gọn về **vòng đời đối tượng (object lifecycle) trong Java**, dựa trên chủ đề *Memory Model: Object lifecycle*.

---

## 1. Các giai đoạn chính của vòng đời đối tượng

1. **Cấp phát (Allocation)**  
   - Đối tượng được tạo trong **heap** bằng từ khóa `new`.
   - Tham chiếu (reference) đến đối tượng thường nằm trong **stack** (biến cục bộ) hoặc trong field của object khác.

2. **Sử dụng (Use / Live)**  
   - Đối tượng “sống” khi còn ít nhất một tham chiếu trỏ tới nó.
   - Trong giai đoạn này, ta gọi phương thức, đọc/ghi field, truyền đối tượng vào hàm,…

3. **Mất tham chiếu (Becoming unreachable)**  
   - Khi không còn biến nào trỏ tới đối tượng (hoặc chuỗi tham chiếu dẫn tới nó), đối tượng trở thành **unreachable**.
   - Lúc này, đối tượng **không còn được dùng nữa** và đủ điều kiện để **garbage collection**.

4. **Thu gom rác (Garbage Collection)**  
   - JVM tự động giải phóng bộ nhớ của các đối tượng unreachable.
   - Lập trình viên **không cần (và không thể) gọi `free()`** như C/C++.
   - Có thể gợi ý bằng `System.gc()`, nhưng JVM **không đảm bảo** sẽ chạy GC ngay lập tức.

---

## 2. Ví dụ đơn giản về vòng đời đối tượng

```java
public class Person {
    String name;

    public Person(String name) {
        this.name = name;
    }

    public void sayHello() {
        System.out.println("Hello, I'm " + name);
    }

    public static void main(String[] args) {
        // 1. Cấp phát: tạo đối tượng Person trong heap
        Person p = new Person("An");

        // 2. Sử dụng: gọi phương thức trên đối tượng
        p.sayHello();  // Output: Hello, I'm An

        // 3. Mất tham chiếu: gán p = null
        p = null;

        // Từ đây, đối tượng Person("An") không còn tham chiếu nào trỏ tới
        // => đủ điều kiện để GC thu gom trong tương lai
        System.gc(); // chỉ là gợi ý, không đảm bảo GC chạy ngay
    }
}
```

**Input (chạy chương trình):**  
Không có input từ người dùng.

**Output:**
```text
Hello, I'm An
```

## 3. Đối tượng local vs. static vs. field

```java
public class Demo {
    // 1. Field (instance field) – sống cùng với object Demo
    private Person member = new Person("Member");

    // 2. Static field – sống gần như suốt vòng đời chương trình
    private static Person staticPerson = new Person("Static");

    public void method() {
        // 3. Local variable – tham chiếu nằm trong stack frame của method
        Person local = new Person("Local");
        local.sayHello();
        // Sau khi method kết thúc, biến local biến mất,
        // nếu không còn tham chiếu nào khác tới đối tượng "Local"
        // thì nó đủ điều kiện GC.
    }

    public static void main(String[] args) {
        Demo d = new Demo();
        d.method();
    }
}
```

**Ý chính:**

- **Local reference** (biến cục bộ) hết hiệu lực khi method kết thúc → có thể làm đối tượng trở thành unreachable.
- **Field / static field** giữ tham chiếu lâu hơn → đối tượng sống lâu hơn, dễ gây **memory leak logic** nếu không bỏ tham chiếu khi không cần nữa.

## 4. Vòng đời và phạm vi (scope) không giống nhau

- **Scope**: vùng code mà biến có thể được truy cập (ví dụ: trong một method, trong một block `{}`).
- **Lifetime của object**: phụ thuộc vào việc còn tham chiếu tới nó hay không, **không chỉ** phụ thuộc vào scope của biến.

Ví dụ:

```java
public Person createPerson() {
    Person p = new Person("Binh"); // p là local
    return p; // trả về tham chiếu
}

public void test() {
    Person x = createPerson(); // x trỏ tới cùng object "Binh"
    x.sayHello();
}
```

- Biến `p` hết scope khi `createPerson()` kết thúc, nhưng object `"Binh"` vẫn sống vì còn tham chiếu `x`.

---

## 5. Một số lưu ý quan trọng

- Không có `delete` trong Java; quản lý bộ nhớ là **tự động**.
- Đối tượng chỉ bị thu gom khi **unreachable**, không phải ngay khi ra khỏi scope.
- Giữ tham chiếu không cần thiết (đặc biệt trong **static collections**, cache, singleton, listener không gỡ bỏ,…) có thể gây **memory leak logic**.
- `finalize()` đã bị **deprecate**; nên dùng `try-with-resources` hoặc explicit `close()` cho tài nguyên (file, socket, DB connection).
