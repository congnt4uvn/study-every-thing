**“Advanced Concurrency: Immutability as a concurrency strategy”**  
(Tiến trình nâng cao: Bất biến như một chiến lược đồng thời)

---

## 1. Ý tưởng chính

1. **Đối tượng bất biến (Immutable Object)**  
   - Trạng thái không bao giờ thay đổi sau khi được tạo.  
   - Không có setter, không cho phép sửa đổi field sau khi khởi tạo.  
   - Tất cả field bên trong cũng phải là bất biến (hoặc không thể bị thay đổi từ bên ngoài).

2. **Lợi ích trong lập trình đồng thời (concurrency)**  
   - **Không cần đồng bộ (no synchronization)**:  
     Nhiều thread có thể đọc cùng một đối tượng bất biến mà không cần `synchronized`, `lock`, v.v.  
   - **Tránh race condition**:  
     Vì không có thay đổi trạng thái, không có tranh chấp ghi (write-write).  
   - **Dễ suy luận, dễ debug**:  
     Trạng thái không đổi → ít bug khó đoán trong môi trường đa luồng.

3. **Chiến lược sử dụng bất biến thay cho mutable + lock**  
   - Thay vì chia sẻ một đối tượng có thể thay đổi giữa các thread, ta:
     - Tạo đối tượng mới mỗi khi cần “thay đổi”.  
     - Chia sẻ đối tượng bất biến cho nhiều thread đọc.  
   - Có thể kết hợp với:
     - `final` fields  
     - `Collections.unmodifiableList/Set/Map`  
     - `record` (Java 16+)  
     - `AtomicReference` để thay thế “root” bất biến.

4. **Trade-off (đánh đổi)**  
   - Tốn thêm bộ nhớ và chi phí tạo object mới khi thay đổi nhiều.  
   - Nhưng đổi lại: code đơn giản hơn, an toàn hơn trong môi trường đa luồng.

## 2. Ví dụ cơ bản: Lớp bất biến

### Ví dụ 1: Lớp `Point` bất biến

```java
public final class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    // Không có setter

    public int getX() {
        return x;
    }

    public int getY() {
        return y;
    }

    // "Thay đổi" bằng cách tạo đối tượng mới
    public Point move(int dx, int dy) {
        return new Point(this.x + dx, this.y + dy);
    }
}
```

**Đặc điểm:**

- `final class`: không cho kế thừa (tránh bị override làm mất tính bất biến).
- Các field `final` và không có setter.
- Mọi “thay đổi” trả về **đối tượng mới**.

## 3. Ví dụ đa luồng: Nhiều thread dùng chung đối tượng bất biến

```java
public class ImmutableDemo {
    public static void main(String[] args) throws InterruptedException {
        Point sharedPoint = new Point(0, 0); // Đối tượng bất biến dùng chung

        Runnable task = () -> {
            // Mỗi thread "tính toán" ra một Point mới, nhưng không làm thay đổi sharedPoint
            Point newPoint = sharedPoint.move(1, 1);
            System.out.println(Thread.currentThread().getName() +
                    " - sharedPoint: (" + sharedPoint.getX() + ", " + sharedPoint.getY() + ")" +
                    " - newPoint: (" + newPoint.getX() + ", " + newPoint.getY() + ")");
        };

        Thread t1 = new Thread(task, "T1");
        Thread t2 = new Thread(task, "T2");
        Thread t3 = new Thread(task, "T3");

        t1.start();
        t2.start();
        t3.start();

        t1.join();
        t2.join();
        t3.join();
    }
}
```

### Kết quả (output) ví dụ:

```text
T1 - sharedPoint: (0, 0) - newPoint: (1, 1)
T2 - sharedPoint: (0, 0) - newPoint: (1, 1)
T3 - sharedPoint: (0, 0) - newPoint: (1, 1)
```

- `sharedPoint` luôn là `(0, 0)` trong mọi thread.  
- Không cần `synchronized`, không có race condition.

## 4. Ví dụ nâng cao: Trạng thái hệ thống bất biến + AtomicReference

Giả sử ta có một trạng thái hệ thống phức tạp, nhiều thread đọc, thỉnh thoảng có thread cập nhật.

### Bước 1: Định nghĩa trạng thái bất biến

```java
import java.util.List;
import java.util.Collections;

public final class SystemState {
    private final List<String> users;

    public SystemState(List<String> users) {
        // Sao chép và bọc thành unmodifiable để đảm bảo bất biến
        this.users = Collections.unmodifiableList(List.copyOf(users));
    }

    public List<String> getUsers() {
        return users;
    }

    public SystemState addUser(String user) {
        List<String> newList = new java.util.ArrayList<>(users);
        newList.add(user);
        return new SystemState(newList);
    }
}
```

### Bước 2: Dùng `AtomicReference<SystemState>` để cập nhật an toàn

```java
import java.util.concurrent.atomic.AtomicReference;
import java.util.List;

public class StateManager {
    private final AtomicReference<SystemState> stateRef =
            new AtomicReference<>(new SystemState(List.of()));

    public void addUser(String user) {
        while (true) {
            SystemState current = stateRef.get();
            SystemState updated = current.addUser(user);
            // CAS: chỉ set nếu stateRef vẫn là current
            if (stateRef.compareAndSet(current, updated)) {
                break;
            }
        }
    }

    public SystemState getState() {
        return stateRef.get();
    }

    public static void main(String[] args) throws InterruptedException {
        StateManager manager = new StateManager();

        Runnable task = () -> {
            for (int i = 0; i < 3; i++) {
                manager.addUser(Thread.currentThread().getName() + "-user" + i);
            }
        };

        Thread t1 = new Thread(task, "T1");
        Thread t2 = new Thread(task, "T2");

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        SystemState finalState = manager.getState();
        System.out.println("Users: " + finalState.getUsers());
    }
}
```

### Kết quả (output) ví dụ (thứ tự có thể khác):

```text
Users: [T1-user0, T1-user1, T1-user2, T2-user0, T2-user1, T2-user2]
```

- `SystemState` là bất biến → mọi thread đọc `getState()` đều an toàn.  
- `AtomicReference` đảm bảo cập nhật “root” trạng thái một cách thread-safe mà không cần lock.

---

## 5. Tóm tắt ngắn

- **Bất biến** là một chiến lược mạnh mẽ cho lập trình đồng thời trong Java.  
- Đối tượng bất biến:
  - Không thay đổi sau khi tạo.
  - Cho phép nhiều thread đọc mà không cần đồng bộ.  
- “Thay đổi” được thực hiện bằng cách tạo **đối tượng mới**.  
- Có thể kết hợp với `final`, `Collections.unmodifiable*`, `record`, `AtomicReference` để xây dựng hệ thống đồng thời an toàn, dễ hiểu, ít bug.
