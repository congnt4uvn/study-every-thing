
Dưới đây là tóm tắt ngắn gọn về 3 pattern quan trọng trong hệ thống phân tán: **Circuit Breaker**, **Retry**, **Timeout** (kèm ví dụ Java, input/output minh họa).

---

## 1. Circuit Breaker (Cầu dao)

**Ý tưởng chính**

- Bảo vệ hệ thống khỏi việc gọi liên tục đến một service đang lỗi hoặc quá tải.  
- Có 3 trạng thái:
  - **CLOSED**: hoạt động bình thường, cho phép gọi.
  - **OPEN**: ngắt ngay lập tức, không gọi sang service (trả lỗi nhanh).
  - **HALF_OPEN**: thử cho một số ít request đi qua để kiểm tra service đã hồi phục chưa.

**Lợi ích**

- Tránh “đổ vỡ dây chuyền” (cascading failure).
- Giảm thời gian chờ vô ích.
- Tăng độ ổn định toàn hệ thống.

**Ví dụ Java với Resilience4j**

```xml
<!-- pom.xml -->
<dependencies>
    <dependency>
        <groupId>io.github.resilience4j</groupId>
        <artifactId>resilience4j-circuitbreaker</artifactId>
        <version>2.2.0</version>
    </dependency>
</dependencies>
```

```java
import io.github.resilience4j.circuitbreaker.*;
import java.time.Duration;
import java.util.function.Supplier;

public class CircuitBreakerExample {
    public static void main(String[] args) {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
                .failureRateThreshold(50)                 // >50% lỗi thì mở cầu dao
                .waitDurationInOpenState(Duration.ofSeconds(5)) // mở 5s rồi chuyển HALF_OPEN
                .slidingWindowSize(10)                    // tính lỗi trên 10 request gần nhất
                .build();

        CircuitBreaker circuitBreaker =
                CircuitBreaker.of("myService", config);

        Supplier<String> remoteCall = CircuitBreaker
                .decorateSupplier(circuitBreaker, CircuitBreakerExample::callRemoteService);

        for (int i = 1; i <= 15; i++) {
            try {
                String result = remoteCall.get();
                System.out.println("Request " + i + " -> " + result);
            } catch (Exception e) {
                System.out.println("Request " + i + " -> ERROR: " + e.getMessage()
                        + " | state=" + circuitBreaker.getState());
            }
        }
    }

    // Giả lập service hay bị lỗi
    private static String callRemoteService() {
        if (Math.random() < 0.7) { // 70% lỗi
            throw new RuntimeException("Remote service failed");
        }
        return "OK from remote service";
    }
}
```

**Ví dụ output (minh họa)**

- Lúc đầu (CLOSED):  
  - Một số request: `OK from remote service`  
  - Nhiều request: `ERROR: Remote service failed | state=CLOSED`
- Khi tỷ lệ lỗi > 50%:  
  - `ERROR: Call not permitted | state=OPEN` (cầu dao mở, chặn luôn)
- Sau 5s: chuyển HALF_OPEN, thử lại vài request:
  - Nếu OK: quay về CLOSED  
  - Nếu vẫn lỗi: quay lại OPEN

## 2. Retry Pattern (Thử lại)

**Ý tưởng chính**

- Khi gọi service bị lỗi tạm thời (network glitch, timeout nhẹ, lỗi 5xx), tự động **thử lại** vài lần trước khi bỏ cuộc.
- Thường kết hợp với:
  - **Backoff**: mỗi lần retry chờ lâu hơn (exponential backoff).
  - **Jitter**: thêm chút ngẫu nhiên để tránh “dồn sóng” (thundering herd).

**Lợi ích**

- Tăng khả năng thành công với lỗi tạm thời.
- Giảm lỗi do sự cố mạng ngắn hạn.

**Ví dụ Java với Resilience4j Retry**

```xml
<!-- pom.xml -->
<dependencies>
    <dependency>
        <groupId>io.github.resilience4j</groupId>
        <artifactId>resilience4j-retry</artifactId>
        <version>2.2.0</version>
    </dependency>
</dependencies>
```

```java
import io.github.resilience4j.retry.*;
import java.time.Duration;
import java.util.function.Supplier;

public class RetryExample {
    public static void main(String[] args) {
        RetryConfig config = RetryConfig.custom()
                .maxAttempts(3)                           // tối đa 3 lần (1 lần chính + 2 lần retry)
                .waitDuration(Duration.ofMillis(500))     // chờ 500ms giữa các lần
                .retryExceptions(RuntimeException.class)  // chỉ retry khi gặp RuntimeException
                .build();

        Retry retry = Retry.of("myRetry", config);

        Supplier<String> remoteCall = Retry
                .decorateSupplier(retry, RetryExample::unstableService);

        try {
            String result = remoteCall.get();
            System.out.println("Final result: " + result);
        } catch (Exception e) {
            System.out.println("All attempts failed: " + e.getMessage());
        }
    }

    // Giả lập service: lần 1,2 lỗi; lần 3 thành công
    static int counter = 0;
    private static String unstableService() {
        counter++;
        System.out.println("Call attempt #" + counter);
        if (counter < 3) {
            throw new RuntimeException("Temporary error");
        }
        return "Success on attempt " + counter;
    }
}
```

**Ví dụ output**

```
Call attempt #1
Call attempt #2
Call attempt #3
Final result: Success on attempt 3
```

Nếu service lỗi cả 3 lần:

```
Call attempt #1
Call attempt #2
Call attempt #3
All attempts failed: Temporary error
```

## 3. Timeout Pattern (Giới hạn thời gian)

**Ý tưởng chính**

- Không chờ vô thời hạn khi gọi service bên ngoài.  
- Nếu quá thời gian cho phép (timeout), hủy request và xử lý fallback (trả lỗi, giá trị mặc định, v.v.).

**Lợi ích**

- Tránh treo thread, tránh “kẹt” tài nguyên.
- Giảm độ trễ tổng thể của hệ thống.

**Ví dụ Java dùng `CompletableFuture`**

```java
import java.util.concurrent.*;

public class TimeoutExample {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newSingleThreadExecutor();

        Callable<String> slowService = () -> {
            Thread.sleep(3000); // giả lập xử lý 3s
            return "Done from slow service";
        };

        Future<String> future = executor.submit(slowService);

        try {
            // chỉ chờ tối đa 1s
            String result = future.get(1, TimeUnit.SECONDS);
            System.out.println("Result: " + result);
        } catch (TimeoutException e) {
            System.out.println("Timeout! Canceling task...");
            future.cancel(true); // hủy task
        } catch (Exception e) {
            System.out.println("Error: " + e.getMessage());
        } finally {
            executor.shutdown();
        }
    }
}
```

**Ví dụ output**

```
Timeout! Canceling task...
```

Nếu giảm `Thread.sleep(3000)` xuống `500` ms:

```
Result: Done from slow service
```

## 4. Kết hợp 3 pattern

Trong hệ thống thực tế, thường **kết hợp**:

- **Timeout**: giới hạn thời gian mỗi lần gọi.
- **Retry**: nếu lỗi tạm thời, thử lại vài lần.
- **Circuit Breaker**: nếu lỗi kéo dài, ngắt hẳn trong một thời gian.

Ví dụ luồng xử lý:

1. Client gọi service A.
2. A gọi service B với **timeout 1s**.
3. Nếu B lỗi/timeout → **retry** tối đa 3 lần.
4. Nếu tỷ lệ lỗi cao → **circuit breaker** mở, các request sau trả lỗi nhanh mà không gọi B nữa.

---

## Tóm tắt ngắn

- **Circuit Breaker**: Ngắt gọi đến service khi lỗi nhiều, tránh đổ vỡ dây chuyền.
- **Retry**: Tự động thử lại khi gặp lỗi tạm thời, thường kèm backoff.
- **Timeout**: Giới hạn thời gian chờ, tránh treo hệ thống.

Ba pattern này là nền tảng để xây dựng hệ thống phân tán **ổn định, chịu lỗi tốt** trong Java (thường dùng cùng Resilience4j, Spring Cloud, v.v.).
