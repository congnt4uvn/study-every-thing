## 1. CAP theorem là gì?

CAP theorem (Brewer’s theorem) nói rằng trong một hệ thống phân tán, ta **không thể đồng thời tối ưu cả 3** thuộc tính sau, mà chỉ có thể chọn **tối đa 2**:

1. **C – Consistency (Nhất quán)**  
   Mọi node nhìn thấy cùng một dữ liệu tại cùng một thời điểm.  
   - Sau khi ghi xong, mọi client đọc đều thấy giá trị mới.

2. **A – Availability (Sẵn sàng)**  
   Mỗi yêu cầu từ client đều nhận được phản hồi (thành công hoặc lỗi), không bị “treo” vô thời hạn.  
   - Hệ thống luôn trả lời, không im lặng.

3. **P – Partition Tolerance (Chịu phân hoạch)**  
   Hệ thống vẫn hoạt động được dù mạng bị chia cắt (một số node không thể liên lạc với nhau).  
   - Trong hệ phân tán thực tế, P gần như luôn phải chấp nhận.

---

## 2. Ý nghĩa với NoSQL

Vì **P gần như luôn bắt buộc**, nên hệ thống thường phải chọn giữa:

- **CP (Consistency + Partition tolerance)**  
  - Ưu tiên dữ liệu nhất quán, chấp nhận có lúc không trả lời (giảm Availability).  
  - Ví dụ: Hệ thống tài chính, ngân hàng.

- **AP (Availability + Partition tolerance)**  
  - Ưu tiên luôn trả lời, chấp nhận dữ liệu có thể tạm thời không nhất quán (eventual consistency).  
  - Ví dụ: Mạng xã hội, hệ thống log, analytics.

Một số NoSQL tiêu biểu (tùy cấu hình, cách triển khai thực tế có thể linh hoạt):

- **CP thiên hướng**: HBase, MongoDB (khi cấu hình ưu tiên strong consistency).
- **AP thiên hướng**: Cassandra, Riak, DynamoDB (mặc định ưu tiên availability, eventual consistency).

## 3. Ví dụ Java minh họa ý tưởng Consistency vs Availability

### 3.1. Mô phỏng “strong consistency” (CP style)

Giả sử ta có một “cluster” đơn giản với 2 node, luôn yêu cầu cả 2 node ghi thành công mới trả về OK (ưu tiên Consistency, có thể làm giảm Availability).

```java
import java.util.HashMap;
import java.util.Map;

class Node {
    private final Map<String, String> store = new HashMap<>();

    public void write(String key, String value) {
        store.put(key, value);
    }

    public String read(String key) {
        return store.get(key);
    }
}

class StrongConsistentCluster {
    private final Node node1 = new Node();
    private final Node node2 = new Node();
    private boolean partitioned = false; // mô phỏng mạng bị chia cắt

    public void setPartitioned(boolean partitioned) {
        this.partitioned = partitioned;
    }

    // Ghi: chỉ thành công nếu cả 2 node đều ghi được
    public boolean write(String key, String value) {
        if (partitioned) {
            // để giữ Consistency, ta từ chối ghi khi bị partition
            System.out.println("WRITE FAILED: network partition, giữ Consistency -> giảm Availability");
            return false;
        }
        node1.write(key, value);
        node2.write(key, value);
        return true;
    }

    // Đọc: đọc từ node1 (giả sử luôn đồng bộ với node2 nếu không partition)
    public String read(String key) {
        return node1.read(key);
    }
}

public class StrongConsistencyDemo {
    public static void main(String[] args) {
        StrongConsistentCluster cluster = new StrongConsistentCluster();

        // Trường hợp bình thường
        boolean ok = cluster.write("balance", "100");
        System.out.println("Write OK? " + ok);
        System.out.println("Read balance: " + cluster.read("balance"));

        // Mô phỏng network partition
        cluster.setPartitioned(true);
        ok = cluster.write("balance", "200");
        System.out.println("Write OK? " + ok);
        System.out.println("Read balance (sau partition): " + cluster.read("balance"));
    }
}
```

**Output (ví dụ):**
```
Write OK? true
Read balance: 100
WRITE FAILED: network partition, giữ Consistency -> giảm Availability
Write OK? false
Read balance (sau partition): 100
```

- Hệ thống **từ chối ghi** khi bị partition để giữ dữ liệu nhất quán → **CP**.

### 3.2. Mô phỏng “eventual consistency” (AP style)

Giờ ta ưu tiên Availability: vẫn cho ghi trên từng node dù bị partition, chấp nhận dữ liệu tạm thời lệch nhau.

```java
import java.util.HashMap;
import java.util.Map;

class APNode {
    private final Map<String, String> store = new HashMap<>();

    public void write(String key, String value) {
        store.put(key, value);
    }

    public String read(String key) {
        return store.get(key);
    }

    public Map<String, String> getStore() {
        return store;
    }
}

class APCluster {
    private final APNode node1 = new APNode();
    private final APNode node2 = new APNode();
    private boolean partitioned = false;

    public void setPartitioned(boolean partitioned) {
        this.partitioned = partitioned;
    }

    // Ghi: luôn ghi vào node local (giả sử client1 nói chuyện với node1)
    public void writeToNode1(String key, String value) {
        node1.write(key, value);
    }

    // Ghi: client2 nói chuyện với node2
    public void writeToNode2(String key, String value) {
        node2.write(key, value);
    }

    public String readFromNode1(String key) {
        return node1.read(key);
    }

    public String readFromNode2(String key) {
        return node2.read(key);
    }

    // Đồng bộ lại sau khi hết partition (eventual consistency)
    public void reconcile() {
        if (!partitioned) {
            // đơn giản: ưu tiên giá trị từ node2 nếu tồn tại
            for (Map.Entry<String, String> e : node2.getStore().entrySet()) {
                node1.write(e.getKey(), e.getValue());
            }
        }
    }
}

public class APDemo {
    public static void main(String[] args) {
        APCluster cluster = new APCluster();

        // Bình thường: ghi vào node1
        cluster.writeToNode1("status", "OK");
        System.out.println("Node1 read: " + cluster.readFromNode1("status"));
        System.out.println("Node2 read (chưa sync): " + cluster.readFromNode2("status"));

        // Mô phỏng partition
        cluster.setPartitioned(true);
        // Client1 ghi vào node1
        cluster.writeToNode1("status", "DEGRADED");
        // Client2 ghi vào node2
        cluster.writeToNode2("status", "MAINTENANCE");

        System.out.println("Trong partition:");
        System.out.println("Node1 read: " + cluster.readFromNode1("status"));
        System.out.println("Node2 read: " + cluster.readFromNode2("status"));

        // Hết partition, đồng bộ lại
        cluster.setPartitioned(false);
        cluster.reconcile();

        System.out.println("Sau reconcile:");
        System.out.println("Node1 read: " + cluster.readFromNode1("status"));
        System.out.println("Node2 read: " + cluster.readFromNode2("status"));
    }
}
```

**Output (ví dụ):**
```
Node1 read: OK
Node2 read (chưa sync): null
Trong partition:
Node1 read: DEGRADED
Node2 read: MAINTENANCE
Sau reconcile:
Node1 read: MAINTENANCE
Node2 read: MAINTENANCE
```

- Trong lúc partition, 2 node có giá trị khác nhau → **không strong consistency**, nhưng hệ thống vẫn **available**.
- Sau khi hết partition, ta đồng bộ lại → **eventual consistency** → kiểu **AP**.

---

## 4. Tóm tắt ngắn

- CAP theorem: không thể tối ưu đồng thời C, A, P; trong hệ phân tán thực tế, luôn phải chấp nhận P.
- Thường phải chọn:
  - **CP**: ưu tiên dữ liệu chính xác, chấp nhận có lúc không trả lời.
  - **AP**: ưu tiên luôn trả lời, chấp nhận dữ liệu tạm thời không đồng nhất.
- NoSQL database thường được thiết kế xoay quanh lựa chọn CP/AP tùy use case.
