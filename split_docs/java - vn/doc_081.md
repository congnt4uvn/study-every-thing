
Dưới đây là tóm tắt ngắn gọn về **Normalization vs Denormalization** trong SQL & RDBMS, kèm ví dụ minh họa (dùng SQL, không phải Java vì nội dung là về CSDL).

---

## 1. Normalization (Chuẩn hóa)

**Mục tiêu:**
- Giảm trùng lặp dữ liệu
- Đảm bảo tính toàn vẹn (data integrity)
- Dễ bảo trì, cập nhật

**Ý tưởng chính:**
- Tách bảng lớn thành nhiều bảng nhỏ hơn
- Mỗi bảng lưu một “thực thể” rõ ràng
- Dùng khóa ngoại (FOREIGN KEY) để liên kết

### Ví dụ: Bảng chưa chuẩn hóa

```sql
-- Bảng đơn lưu cả thông tin khách hàng và đơn hàng
CREATE TABLE OrdersRaw (
    order_id    INT,
    customer_id INT,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    product_name VARCHAR(100),
    quantity INT
);
```

Vấn đề:
- `customer_name`, `customer_email` bị lặp lại cho mỗi đơn hàng
- Nếu khách đổi email → phải cập nhật nhiều dòng

### Chuẩn hóa (ví dụ đến 3NF)

```sql
-- Bảng khách hàng
CREATE TABLE Customers (
    customer_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100)
);

-- Bảng đơn hàng
CREATE TABLE Orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES Customers(customer_id)
);

-- Bảng chi tiết đơn hàng
CREATE TABLE OrderItems (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES Orders(order_id)
);
```

**Truy vấn ví dụ (input):**

```sql
SELECT o.order_id, c.customer_name, c.customer_email
FROM Orders o
JOIN Customers c ON o.customer_id = c.customer_id
WHERE o.order_id = 101;
```

**Kết quả (output):**

| order_id | customer_name | customer_email      |
|----------|---------------|---------------------|
| 101      | Nguyen Van A  | a.nguyen@example.com|

**Ưu điểm:**
- Ít trùng lặp dữ liệu
- Cập nhật dễ, ít lỗi
- Dữ liệu nhất quán

**Nhược điểm:**
- Nhiều JOIN → query có thể chậm hơn trong hệ thống đọc nhiều (read-heavy)

---

## 2. Denormalization (Phi chuẩn hóa)

**Mục tiêu:**
- Tăng tốc độ đọc (read performance)
- Giảm số lượng JOIN trong truy vấn

**Ý tưởng chính:**
- Cố ý chấp nhận trùng lặp dữ liệu
- Gộp một số bảng lại, lưu sẵn dữ liệu đã “join”
- Thường dùng cho báo cáo, dashboard, hệ thống đọc nhiều

### Ví dụ: Bảng phi chuẩn hóa

```sql
CREATE TABLE OrdersDenorm (
    order_id        INT PRIMARY KEY,
    order_date      DATE,
    customer_id     INT,
    customer_name   VARCHAR(100),
    customer_email  VARCHAR(100),
    total_amount    DECIMAL(10,2)
);
```

Ở đây:
- Thông tin khách hàng được lưu trực tiếp trong bảng đơn hàng
- `customer_name`, `customer_email` có thể lặp lại ở nhiều dòng

**Truy vấn ví dụ (input):**

```sql
SELECT order_id, customer_name, customer_email, total_amount
FROM OrdersDenorm
WHERE order_id = 101;
```

**Kết quả (output):**

| order_id | customer_name | customer_email      | total_amount |
|----------|---------------|---------------------|--------------|
| 101      | Nguyen Van A  | a.nguyen@example.com| 1500000.00   |

**Ưu điểm:**
- Truy vấn nhanh hơn (ít JOIN)
- Phù hợp cho báo cáo, analytics, hệ thống đọc nhiều

**Nhược điểm:**
- Dữ liệu trùng lặp
- Cập nhật khó, dễ không nhất quán (update 1 nơi quên update nơi khác)
- Tăng dung lượng lưu trữ

---

## 3. Khi nào dùng Normalization vs Denormalization?

**Nên ưu tiên chuẩn hóa khi:**
- Hệ thống OLTP (giao dịch online): nhiều INSERT/UPDATE/DELETE
- Cần tính chính xác, nhất quán cao
- Dữ liệu thay đổi thường xuyên

**Có thể phi chuẩn hóa khi:**
- Hệ thống đọc nhiều (reporting, BI, dashboard)
- Chấp nhận trùng lặp để đổi lấy tốc độ
- Có cơ chế đồng bộ/ETL để cập nhật dữ liệu

---

## 4. Tóm tắt ngắn

- **Normalization**: tách bảng, giảm trùng lặp, tăng tính toàn vẹn → tốt cho cập nhật, nhưng query có thể cần nhiều JOIN.
- **Denormalization**: gộp dữ liệu, chấp nhận trùng lặp để tăng tốc độ đọc → tốt cho báo cáo, nhưng khó cập nhật và dễ không nhất quán.

Nếu bạn muốn, tôi có thể viết thêm ví dụ Java (JDBC/JPA) minh họa truy vấn trên các bảng đã chuẩn hóa và phi chuẩn hóa.
