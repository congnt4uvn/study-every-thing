---

## 1. Lazy Loading vs Eager Loading

### a) Lazy Loading (`FetchType.LAZY`)
- Dữ liệu liên quan **chỉ được tải khi cần dùng** (khi truy cập vào thuộc tính).
- Giúp **giảm truy vấn không cần thiết**, tối ưu hiệu năng.
- Nhưng nếu dùng sai (nhất là ngoài transaction) dễ bị:
  - `LazyInitializationException`
  - N+1 query problem

**Ví dụ:**

```java
@Entity
public class User {
    @Id
    private Long id;

    private String name;

    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;
}
```

**Khi truy vấn:**

```java
// Input: gọi service lấy user
User user = userRepository.findById(1L).get();

// Lúc này chỉ query bảng user
// SQL: select * from user where id = 1;

// Khi gọi:
List<Order> orders = user.getOrders();
// Lúc này mới query bảng order
// SQL: select * from orders where user_id = 1;
```

---

### b) Eager Loading (`FetchType.EAGER`)
- Dữ liệu liên quan được **tải ngay lập tức** cùng với entity chính.
- Tiện khi **luôn luôn cần** dữ liệu liên quan.
- Nhưng có thể gây:
  - Truy vấn nặng, join nhiều bảng
  - Tải dư dữ liệu không dùng

**Ví dụ:**

```java
@Entity
public class User {
    @Id
    private Long id;

    private String name;

    @OneToMany(mappedBy = "user", fetch = FetchType.EAGER)
    private List<Order> orders;
}
```

**Khi truy vấn:**

```java
User user = userRepository.findById(1L).get();

// Thường sẽ có join hoặc nhiều query để lấy luôn orders
// SQL (ví dụ): select u.*, o.* from user u left join orders o on u.id = o.user_id where u.id = 1;
```

## 2. N+1 Query Problem

### Khái niệm
- Xảy ra khi:
  1. Chạy **1 query** để lấy danh sách entity (N bản ghi).
  2. Sau đó, với mỗi entity, lại chạy **1 query riêng** để lấy dữ liệu liên quan.
- Tổng cộng: **N + 1 query** → rất tốn kém.

### Ví dụ N+1 với Lazy Loading

```java
// Repository
List<User> users = userRepository.findAll(); 
// SQL 1: select * from user;

// Trong vòng lặp:
for (User u : users) {
    List<Order> orders = u.getOrders();
    // Mỗi lần gọi getOrders() -> 1 query
    // SQL 2: select * from orders where user_id = ?
    // SQL 3: select * from orders where user_id = ?
    // ...
}
```

**Kết quả:**
- Nếu có 100 users → 1 (lấy users) + 100 (lấy orders) = **101 query**.

---

## 3. Cách tránh N+1

### a) Dùng `JOIN FETCH` trong JPQL

```java
@Query("SELECT u FROM User u LEFT JOIN FETCH u.orders")
List<User> findAllWithOrders();
```

**Khi gọi:**

```java
List<User> users = userRepository.findAllWithOrders();

// SQL (ví dụ):
// select u.*, o.* from user u
// left join orders o on u.id = o.user_id;
```

- Lúc này chỉ **1 query** lấy cả `User` và `Order`.

---

### b) Dùng `@EntityGraph`

```java
@EntityGraph(attributePaths = "orders")
@Query("SELECT u FROM User u")
List<User> findAllWithOrders();
```

- Tác dụng tương tự `JOIN FETCH`, nhưng code gọn hơn, dễ tái sử dụng.

---

### c) Chọn FetchType hợp lý

- Quan hệ `@ManyToOne`, `@OneToOne`: thường **EAGER** mặc định → cân nhắc chuyển sang `LAZY`.
- Quan hệ `@OneToMany`, `@ManyToMany`: nên để **LAZY**, và dùng `JOIN FETCH`/`EntityGraph` khi cần.

---

## 4. Tóm tắt ý chính

- **Lazy Loading**: chỉ tải dữ liệu khi truy cập; tốt cho hiệu năng nhưng dễ gây N+1 nếu không kiểm soát.
- **Eager Loading**: tải ngay dữ liệu liên quan; tiện nhưng có thể tải dư, query nặng.
- **N+1 Problem**: 1 query lấy danh sách + N query cho từng phần tử → rất tốn tài nguyên.
- **Giải pháp**:
  - Dùng `JOIN FETCH` hoặc `@EntityGraph` để load dữ liệu liên quan trong **1 query**.
  - Thiết kế `FetchType` hợp lý, tránh EAGER mặc định ở quan hệ phức tạp.
  - Kiểm tra log SQL để phát hiện N+1.
