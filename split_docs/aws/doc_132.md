


Người hướng dẫn giải thích cách các yêu cầu HTTP tới AWS được “ký” để AWS biết bạn là ai và cho phép/ từ chối yêu cầu:

- Khi gọi bất kỳ API HTTP nào của AWS (S3, EC2, …), yêu cầu phải được ký bằng thông tin xác thực AWS (Access Key, Secret Key).
- Việc ký này dùng cơ chế **Signature Version 4 (SigV4)**. Quy trình nội bộ khá phức tạp (4 bước), nhưng thường bạn không phải tự làm vì:
  - **AWS SDK** và **AWS CLI** tự động ký tất cả yêu cầu.
- Một số trường hợp đặc biệt (ví dụ đọc đối tượng S3 công khai) không cần ký, nhưng đa số API khác thì bắt buộc.

Có **hai cách** để gửi chữ ký SigV4 tới AWS:

1. **Trong header Authorization của HTTP**  
   - Chữ ký được đặt trong header `Authorization`.  
   - Đây là cách mà AWS CLI/SDK sử dụng mặc định.

2. **Trong query string của URL (Presigned URL)**  
   - Chữ ký được nhúng trực tiếp vào URL dưới dạng tham số truy vấn.  
   - Tham số quan trọng là `X-Amz-Signature`.  
   - Ngoài ra còn có các tham số khác như:
     - `X-Amz-Algorithm` (ví dụ: `AWS4-HMAC-SHA256`)
     - `X-Amz-Date`
     - `X-Amz-Expires` (thời gian hết hạn URL)
     - `X-Amz-Credential` (thông tin credential)
     - `X-Amz-Security-Token` (nếu dùng tạm thời)

Ví dụ minh họa:

- Trong S3 console, khi mở một file (ví dụ `coffee.jpg`), trình duyệt dùng một URL đã được ký sẵn (presigned URL).
- URL này chứa đầy đủ các tham số SigV4 ở dạng query string, cho phép truy cập đối tượng trong một khoảng thời gian nhất định.
- Khi hết hạn (`X-Amz-Expires`), URL không còn dùng được.

Tóm lại:

- **SigV4** là cơ chế chuẩn để ký các yêu cầu tới AWS.
- Chữ ký có thể được gửi:
  - Trong **HTTP header** (`Authorization`), hoặc
  - Trong **query string** với khóa `X-Amz-Signature`.
- SDK/CLI tự động xử lý việc ký, nên lập trình viên thường chỉ cần hiểu khái niệm và cách chữ ký được truyền đi.
