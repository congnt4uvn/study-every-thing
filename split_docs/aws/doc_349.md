


Dưới đây là tóm tắt ngắn gọn nội dung bạn cung cấp về giám sát API Gateway:

1. **Ghi nhật ký với CloudWatch Logs**  
   - Bật tích hợp CloudWatch Logs ở **mức Stage** (có thể override theo API).  
   - Chọn **mức log**: ERROR, INFO, DEBUG (DEBUG chi tiết nhất).  
   - API Gateway ghi lại **request** và **response** (từ client → API Gateway → backend → API Gateway → client).  
   - Cần chú ý vì có thể ghi cả **thông tin nhạy cảm** vào log.  

2. **Theo dõi với AWS X-Ray**  
   - Bật X-Ray cho **API Gateway** và **Lambda** để có **bức tranh end-to-end** về luồng request.  
   - X-Ray dùng để **trace** request, không phải để log nội dung chi tiết như CloudWatch Logs.  

3. **CloudWatch Metrics cho API Gateway**  
   Các metric quan trọng:  
   - **CacheHitCount / CacheMissCount**: đánh giá hiệu quả cache.  
   - **Count**: số lượng request trong một khoảng thời gian.  
   - **IntegrationLatency**: thời gian từ khi API Gateway gửi request đến backend đến khi nhận response (thời gian xử lý của backend).  
   - **Latency**: thời gian từ khi API Gateway nhận request từ client đến khi trả response (bao gồm IntegrationLatency + thời gian xử lý trong API Gateway: auth, validation, cache, mapping…).  
   - **Giới hạn thời gian**: API Gateway chỉ xử lý tối đa **29 giây** cho một request. Nếu Latency/IntegrationLatency > 29s → timeout.  

4. **Lỗi 4XX và 5XX**  
   - **4XX (Client errors)**: lỗi phía client sử dụng API Gateway.  
     - 400: Bad Request  
     - 403: Access Denied (hoặc bị WAF chặn)  
     - 429: Too Many Requests (bị throttling hoặc vượt quota)  
   - **5XX (Server errors)**: lỗi phía server/backend.  
     - 502: Lỗi tích hợp (ví dụ Lambda proxy integration trả về không đúng)  
     - 503: Backend không khả dụng  
     - 504: Integration Failure do **timeout** (sau 29 giây không nhận được phản hồi từ backend).  

5. **Giới hạn tài khoản & Throttling**  
   - Mặc định, API Gateway có **account-level limit**: ~**10.000 requests/second** trên **tất cả API** (soft limit, có thể tăng).  
   - Nếu một API dùng quá nhiều, các API khác có thể bị **throttling**.  
   - Khi bị throttling, client nhận **HTTP 429**. Nên dùng **exponential backoff** để retry.  
   - Có thể:  
     - Đặt **Stage-level** và **Method-level throttling** để bảo vệ từng stage/method.  
     - Dùng **Usage Plans** + API Keys để giới hạn/throttling theo từng khách hàng.  
   - Tương tự Lambda concurrency: nếu không giới hạn, một API có thể “ăn hết” tài nguyên, làm API khác bị throttling.  

Tóm lại:  
- **CloudWatch Logs**: ghi chi tiết request/response (cẩn trọng dữ liệu nhạy cảm).  
- **X-Ray**: trace luồng request end-to-end.  
- **CloudWatch Metrics**: theo dõi cache, latency, lỗi 4XX/5XX, timeout 29s.  
- **Throttling & Limits**: quản lý lưu lượng bằng account limit, stage/method throttling, usage plans; lỗi 429 khi vượt giới hạn.
