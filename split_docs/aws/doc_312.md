


Dưới đây là tóm tắt nội dung:

1. Tạo bảng Users trong DynamoDB  
   - DynamoDB là dịch vụ NoSQL, serverless, bạn không tạo “database” mà tạo trực tiếp các bảng.  
   - Tạo bảng tên `Users` với:
     - Partition key: `user_ID` (kiểu String).
     - Không dùng sort key ở ví dụ đầu.  
   - Chọn “Standard DynamoDB table class”.  
   - Chế độ dung lượng: chọn “Provisioned” (nằm trong free tier).  
   - Tắt auto scaling, đặt 2 read capacity units và 2 write capacity units (trong free tier).  
   - Bật mã hóa bằng khóa mặc định của DynamoDB.  
   - Tạo bảng xong có thể xem các thông tin: partition key, sort key, chế độ dung lượng, ngày tạo, v.v.

2. Thêm dữ liệu (items) vào bảng Users  
   - Vào tab Items → Create item.  
   - Nhập `user_ID` (partition key), ví dụ: `John123`.  
   - Thêm các thuộc tính:
     - `first_name = "John"`
     - `last_name = "Doe"`  
   - Tạo item thành công, hiển thị trong bảng.  
   - Thử tạo item thứ hai với cùng `user_ID = John123` sẽ bị lỗi vì partition key phải duy nhất (khi không có sort key).  
   - Tạo item khác:
     - `user_ID = "Alice456"`
     - `first_name = "Alice"`
     - `age = 41`  
   - DynamoDB cho phép mỗi item có tập thuộc tính khác nhau:
     - John có `first_name`, `last_name` nhưng không có `age`.
     - Alice có `first_name`, `age` nhưng không có `last_name`.  
   - Các thuộc tính không tồn tại coi như null; DynamoDB không có ràng buộc kiểu “NOT NULL” như SQL.  
   - Đây là ưu điểm (linh hoạt) nhưng cũng là rủi ro (dễ thiếu nhất quán dữ liệu).

3. Tạo bảng thứ hai: UserPosts  
   - Tạo bảng `UserPosts` với:
     - Partition key: `user_ID`
     - Sort key: `TS` (timestamp)  
   - Dùng lại chế độ provisioned, 2 read / 2 write capacity units.  
   - Bảng này dùng để lưu các bài đăng của người dùng.

4. Thêm dữ liệu vào bảng UserPosts  
   - Vào Items → Create item.  
   - Ví dụ item 1:
     - `user_ID = "John123"`
     - `TS = "2021-10-09T12:34:09Z"` (chuỗi timestamp)
     - `content = "Hello world, đây là blog đầu tiên của tôi."`  
   - Item 2:
     - `user_ID = "John123"`
     - `TS = "2021-11-04T..."` (timestamp khác)
     - `content = "Bài đăng thứ hai tuyệt vời!"`  
   - Cả hai item đều hợp lệ vì cặp (partition key, sort key) là duy nhất:
     - Tính duy nhất = `user_ID` + `TS`.  
   - Có thể thêm item cho `Alice456` với timestamp và content riêng.  

5. Ý nghĩa partition key và sort key  
   - Partition key (`user_ID`) dùng để phân vùng dữ liệu.  
   - Sort key (`TS`) dùng để sắp xếp và truy vấn theo thứ tự (ví dụ: các bài đăng của một user theo thời gian).  
   - Khi truy vấn theo `user_ID`, có thể sắp xếp/giới hạn theo `TS`.  
   - Việc chọn partition key rất quan trọng:
     - Nếu chỉ một user (ví dụ `John123`) có 10.000 bài đăng, toàn bộ dữ liệu sẽ dồn vào một partition → mất cân bằng (hot partition).  

6. Kết luận  
   - DynamoDB:
     - Không cần quản lý server, database đã có sẵn, chỉ tạo bảng.  
     - Linh hoạt về schema: mỗi item có thể có thuộc tính khác nhau.  
     - Khóa chính có thể là:
       - Chỉ partition key (unique).
       - Hoặc partition key + sort key (unique theo cặp).  
   - Bài học: hiểu cách thiết kế bảng, chọn partition key/sort key, và cách thêm item với schema linh hoạt.
