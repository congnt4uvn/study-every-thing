


Dưới đây là tóm tắt nội dung:

1. Bối cảnh  
   - Đã học Lambda và DynamoDB, dùng Lambda + DynamoDB làm backend (CRUD).  
   - Cần cách để khách hàng gọi được Lambda mà không phải cấp quyền IAM trực tiếp.

2. Các cách expose Lambda  
   - Cho client gọi trực tiếp Lambda → phải cấp quyền IAM (không phù hợp cho khách ngoài).  
   - Dùng Application Load Balancer (ALB) để tạo endpoint HTTP cho Lambda.  
   - Dùng API Gateway (trọng tâm bài): dịch vụ serverless để tạo REST API công khai.

3. API Gateway và Lambda  
   - Client gọi API Gateway → API Gateway ủy quyền (proxy) request đến Lambda.  
   - Kết hợp API Gateway + Lambda + DynamoDB → ứng dụng serverless hoàn chỉnh, không quản lý server.

4. Tính năng chính của API Gateway  
   - Hỗ trợ REST API, WebSocket (real-time, 2 chiều).  
   - Quản lý version API (v1, v2, v3…) mà không làm hỏng client cũ.  
   - Hỗ trợ nhiều môi trường (dev, test, prod).  
   - Bảo mật: xác thực, ủy quyền, API keys.  
   - Usage plans, throttling (giới hạn/tốc độ request).  
   - Caching response.  
   - Hỗ trợ import/export chuẩn Swagger / OpenAPI 3.0.  
   - Chuyển đổi, validate request/response ở tầng API.  
   - Tạo SDK và tài liệu API.

5. Các kiểu tích hợp của API Gateway  
   - Lambda integration: phổ biến nhất, dễ tạo REST API serverless.  
   - HTTP integration: proxy đến bất kỳ HTTP endpoint nào (on-prem, ALB, v.v.) để tận dụng rate limiting, auth, caching, API keys…  
   - AWS service integration: expose trực tiếp các dịch vụ AWS (Step Functions, SQS, Kinesis, …) ra ngoài qua API Gateway.

6. Ví dụ với Kinesis Data Streams  
   - Client gửi HTTP request đến API Gateway.  
   - API Gateway gửi dữ liệu vào Kinesis Data Streams.  
   - Từ Kinesis có thể đẩy sang Kinesis Firehose → lưu vào S3 (JSON).  
   - Không cần cấp credential AWS cho client, vẫn cho phép gửi dữ liệu an toàn.

7. Ba kiểu endpoint (triển khai) của API Gateway  
   - Edge-Optimized (mặc định):  
     - Dành cho khách hàng toàn cầu.  
     - Sử dụng CloudFront Edge để giảm độ trễ.  
     - API thực sự nằm trong một region, nhưng được phân phối qua edge locations.  
   - Regional:  
     - Không dùng CloudFront Edge mặc định.  
     - Phù hợp khi user chủ yếu ở cùng region.  
     - Có thể tự tạo CloudFront distribution riêng nếu muốn kiểm soát chi tiết.  
   - Private:  
     - Không công khai Internet.  
     - Chỉ truy cập từ trong VPC qua VPC endpoint (interface).  
     - Dùng resource policy để kiểm soát truy cập.

8. Bảo mật trên API Gateway  
   - IAM roles:  
     - Dùng cho ứng dụng nội bộ (ví dụ EC2) gọi API Gateway.  
   - Amazon Cognito:  
     - Dùng cho user bên ngoài (mobile, web app) để xác thực.  
   - Custom Authorizer:  
     - Lambda function tự viết logic auth/authorization.  
   - HTTPS + custom domain:  
     - Dùng AWS Certificate Manager (ACM) để cấp chứng chỉ.  
     - Với Edge-Optimized: chứng chỉ phải ở us-east-1.  
     - Với Regional: chứng chỉ ở cùng region với API.  
     - Dùng Route 53 (CNAME hoặc A-alias) trỏ domain đến API Gateway.

Tóm lại: API Gateway là lớp front-door mạnh mẽ cho các dịch vụ serverless (Lambda, DynamoDB, Kinesis, v.v.), cung cấp endpoint HTTP/REST/WebSocket kèm theo bảo mật, throttling, caching, versioning và quản lý môi trường, giúp xây dựng ứng dụng serverless hoàn chỉnh mà không cần quản lý hạ tầng.
