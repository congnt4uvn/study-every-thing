


Dưới đây là tóm tắt nội dung:

1. Có 2 cách triển khai (upload) hàm Lambda bằng CloudFormation:  
   - **Inline (nội tuyến)**  
     - Viết trực tiếp mã Lambda trong template CloudFormation.  
     - Dùng thuộc tính `ZipFile`.  
     - Chỉ phù hợp cho hàm đơn giản, **không có dependency** (thư viện ngoài).  
     - Mã Lambda nằm ngay trong file template.

   - **Qua file zip trên S3**  
     - Đóng gói mã Lambda (kèm dependency) thành file `.zip`.  
     - Upload file zip lên **Amazon S3**.  
     - Trong CloudFormation, tham chiếu đến:  
       - `S3Bucket`: tên bucket chứa file zip.  
       - `S3Key`: đường dẫn/ tên file zip trong bucket.  
       - `S3ObjectVersion` (tùy chọn nhưng **được khuyến nghị** nếu bucket bật versioning).  
     - Nếu chỉ thay mã trong S3 mà không đổi `S3Key` hoặc `S3ObjectVersion` trong template, CloudFormation sẽ **không** cập nhật hàm.  
     - Do đó nên bật versioning và cập nhật `S3ObjectVersion` mỗi lần đổi mã.

2. Triển khai Lambda qua CloudFormation **đa tài khoản** (multi-account)  
   - Ví dụ:  
     - **Account 1**: chứa S3 bucket với mã Lambda.  
     - **Account 2, 3**: muốn triển khai Lambda dùng mã từ bucket ở Account 1.  
   - Cách làm:  
     - Trong Account 2 (hoặc 3), chạy CloudFormation stack tham chiếu đến bucket S3 ở Account 1.  
     - Cần đảm bảo quyền truy cập:  
       1. **S3 bucket policy** ở Account 1 cho phép CloudFormation (từ Account 2/3) truy cập (get/list) object.  
       2. **Execution role** cho CloudFormation trong Account 2/3 có quyền `s3:GetObject`, `s3:ListBucket` trên bucket ở Account 1.  
     - Kết hợp cả bucket policy và execution role sẽ cho phép CloudFormation đọc mã zip và tạo/ cập nhật hàm Lambda ở các tài khoản khác.

3. Ý chính  
   - Inline code: nhanh, đơn giản, không hỗ trợ dependency.  
   - Zip trên S3: dùng cho thực tế, hỗ trợ dependency, nên bật versioning.  
   - Multi-account: cần cấu hình quyền (bucket policy + execution role) để CloudFormation ở tài khoản khác đọc được mã Lambda từ S3.
