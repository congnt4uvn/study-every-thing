


Dưới đây là tóm tắt ngắn gọn nội dung bạn cung cấp:

1. So sánh kiến trúc truyền thống (SQL/RDS) và NoSQL  
   - Kiến trúc truyền thống: Client → lớp ứng dụng (ELB + EC2 + Auto Scaling) → lớp CSDL (RDS: MySQL, PostgreSQL…).  
   - RDBMS dùng SQL, có schema rõ ràng, hỗ trợ join, aggregate (SUM, AVG…), truy vấn phức tạp.  
   - Về mở rộng:
     - Chủ yếu là scale vertical (tăng CPU, RAM, I/O cho 1 instance DB).  
     - Horizontal chủ yếu cho đọc: thêm EC2 ở lớp app, thêm Read Replica ở RDS.  
     - Bị giới hạn số lượng replica, không có scale ngang thực sự cho ghi.

2. Giới thiệu NoSQL  
   - NoSQL = cơ sở dữ liệu phi quan hệ, phân tán, hỗ trợ scale ngang tốt.  
   - Ví dụ: MongoDB, DynamoDB.  
   - Hạn chế (giả định đơn giản trong bài):  
     - Không (hoặc rất hạn chế) join.  
     - Không (hoặc hạn chế) aggregate như SUM, AVG…  
   - Bù lại: nhờ thiết kế phân tán, có thể scale ngang mạnh cho cả đọc và ghi.  
   - Không có đúng/sai tuyệt đối giữa SQL và NoSQL; phụ thuộc:
     - Cách mô hình hóa dữ liệu.  
     - Kiểu truy vấn.  
     - Nhu cầu mở rộng.

3. Tổng quan DynamoDB  
   - DynamoDB là CSDL NoSQL được quản lý hoàn toàn, phân tán, multi-AZ, highly available.  
   - Khả năng mở rộng:
     - Hàng triệu request/giây.  
     - Hàng nghìn tỷ dòng, hàng trăm TB dữ liệu.  
   - Hiệu năng: độ trễ thấp, hiệu suất ổn định.  
   - Tích hợp IAM cho bảo mật, phân quyền.  
   - Hỗ trợ event-driven qua DynamoDB Streams.  
   - Chi phí thấp, tự động scale.  
   - Có các lớp lưu trữ: Standard, Infrequent Access (IA).

4. Cấu trúc dữ liệu trong DynamoDB  
   - Gồm: Table → Item (row) → Attribute (cột).  
   - Không cần schema cứng:
     - Thuộc tính có thể thêm dần theo thời gian.  
     - Một số item có thể thiếu thuộc tính (null/không tồn tại) là bình thường.  
   - Mỗi item tối đa 400 KB.  
   - Kiểu dữ liệu:
     - Vô hướng: String, Number, Binary, Boolean, Null.  
     - Document: List, Map (cho phép lồng dữ liệu).  
     - Set: String Set, Number Set, Binary Set.

5. Khóa chính (Primary Key) trong DynamoDB  
   Có 2 kiểu:

   a) Partition Key (Hash Key)  
   - Chỉ có 1 khóa: Partition Key.  
   - Phải duy nhất cho mỗi item.  
   - Partition Key cần “đa dạng” để dữ liệu phân tán đều.  
   - Ví dụ: bảng Users với partition key = user_id, các thuộc tính khác: first_name, last_name, age.  
   - Các item có thể thiếu một số thuộc tính (VD: thiếu last_name).

   b) Partition Key + Sort Key (Hash + Range)  
   - Khóa chính là tổ hợp (Partition Key, Sort Key).  
   - Tổ hợp này phải duy nhất cho mỗi item.  
   - Dữ liệu được nhóm theo Partition Key, Sort Key dùng để sắp xếp/định danh trong nhóm.  
   - Ví dụ: bảng user-game:
     - Partition Key: user_id  
     - Sort Key: game_id  
     - Các thuộc tính khác: score, result…  
   - Một user có thể có nhiều game (cùng partition key, sort key khác nhau).  
   - Việc chọn partition key tốt rất quan trọng để tránh lệch dữ liệu (hot partition).

6. Bài tập chọn Partition Key  
   - Bài toán: CSDL phim, chọn partition key để phân phối dữ liệu tốt nhất:  
     - movie_id  
     - producer_name  
     - lead_actor_name  
     - movie_language  
   - Đáp án: movie_id  
     - Vì movie_id là duy nhất cho mỗi phim → phân phối đều.  
     - movie_language (VD: English) có ít giá trị, dễ lệch dữ liệu (nhiều phim cùng 1 ngôn ngữ).  
   - Nguyên tắc: chọn khóa có:
     - Số lượng giá trị khác nhau lớn.  
     - Phân phối đều, tránh tập trung vào một vài giá trị.

7. Kết luận  
   - DynamoDB là NoSQL, phân tán, scale ngang mạnh, phù hợp khối lượng lớn, độ trễ thấp.  
   - Thiết kế khóa chính (đặc biệt là partition key) là yếu tố cốt lõi.  
   - Kỳ thi (AWS) thường hỏi về cách chọn partition key tối ưu và mô hình dữ liệu phù hợp.
