
Here’s a concise summary of the **Outbox Pattern (DB → Kafka)** in the context of Kafka and Spring Boot, with code examples.

---

## Core Idea

When a service needs to:

1. **Update its database** (e.g., create an order), and  
2. **Publish an event to Kafka** (e.g., `OrderCreated`),

doing both separately can cause inconsistencies (DB updated but event not sent, or vice versa).

**Outbox pattern** solves this by:

- Writing both:
  - the **business data** (e.g., `orders` table) and  
  - the **event data** (e.g., `outbox` table)  
- in the **same local database transaction**.
- A separate process (or thread) then reads the `outbox` table and publishes events to Kafka.

This ensures **atomicity**: either both DB change and event are recorded, or neither is.

---

## Typical Flow

1. **Service handles request** (e.g., create order).
2. In a single DB transaction:
   - Insert into `orders` table.
   - Insert corresponding event into `outbox` table.
3. A **background publisher**:
   - Polls `outbox` table for new events.
   - Publishes them to Kafka.
   - Marks them as sent (or deletes them).

---

## Example Schema

```sql
CREATE TABLE orders (
    id           BIGSERIAL PRIMARY KEY,
    customer_id  BIGINT      NOT NULL,
    status       VARCHAR(50) NOT NULL,
    created_at   TIMESTAMP   NOT NULL DEFAULT now()
);

CREATE TABLE outbox_event (
    id           BIGSERIAL PRIMARY KEY,
    aggregate_id BIGINT      NOT NULL,      -- e.g., order id
    aggregate_type VARCHAR(100) NOT NULL,   -- e.g., 'Order'
    event_type   VARCHAR(100) NOT NULL,     -- e.g., 'OrderCreated'
    payload      JSONB        NOT NULL,     -- event data
    status       VARCHAR(20)  NOT NULL DEFAULT 'NEW', -- NEW, SENT, FAILED
    created_at   TIMESTAMP    NOT NULL DEFAULT now(),
    sent_at      TIMESTAMP
);
```

## Spring Boot: Writing to DB + Outbox (Same Transaction)

```java
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final OutboxEventRepository outboxEventRepository;

    @Transactional
    public Order createOrder(CreateOrderRequest request) {
        // 1. Save business entity
        Order order = new Order();
        order.setCustomerId(request.getCustomerId());
        order.setStatus("CREATED");
        order = orderRepository.save(order);

        // 2. Create outbox event
        OrderCreatedEvent event = new OrderCreatedEvent(
                order.getId(),
                order.getCustomerId(),
                order.getStatus()
        );

        OutboxEvent outboxEvent = new OutboxEvent();
        outboxEvent.setAggregateId(order.getId());
        outboxEvent.setAggregateType("Order");
        outboxEvent.setEventType("OrderCreated");
        outboxEvent.setPayload(toJson(event)); // serialize to JSON
        outboxEvent.setStatus("NEW");

        outboxEventRepository.save(outboxEvent);

        // If anything fails here, the whole transaction rolls back
        return order;
    }

    private String toJson(Object obj) {
        try {
            return new ObjectMapper().writeValueAsString(obj);
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
```

## Spring Boot: Outbox Publisher → Kafka

```java
@Service
public class OutboxPublisher {

    private final OutboxEventRepository outboxEventRepository;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 5000) // every 5 seconds
    public void publishOutboxEvents() {
        List<OutboxEvent> events = outboxEventRepository.findTop100ByStatusOrderByIdAsc("NEW");

        for (OutboxEvent event : events) {
            try {
                // topic could be derived from aggregateType or eventType
                String topic = "order-events";

                kafkaTemplate.send(topic, event.getAggregateId().toString(), event.getPayload())
                             .get(); // wait for send result (optional)

                event.setStatus("SENT");
                event.setSentAt(Instant.now());
                outboxEventRepository.save(event);
            } catch (Exception e) {
                // mark as FAILED or keep as NEW for retry
                event.setStatus("FAILED");
                outboxEventRepository.save(event);
            }
        }
    }
}
```

- **Key**: `"1"`  
- **Value (payload)**:

  "status": "CREATED"

## Key Benefits

- **Atomicity**: DB change and event recording are in one transaction.
- **Reliability**: If Kafka is down, events are still stored in DB and retried later.
- **Replayability**: Outbox table can be used to replay or audit events.
- **Decoupling**: Business logic doesn’t depend on Kafka availability at request time.

---

## Key Trade-offs

- Requires **extra table** and **background process**.
- Need to handle **idempotency** on consumers (events may be retried).
- Outbox table can grow; you need **cleanup/archiving** strategy.
