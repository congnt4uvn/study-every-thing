
Here’s a concise summary of Kafka log retention & compaction, with Spring Boot examples.

---

## 1. Retention vs Log Compaction

**Retention**  
- Deletes old messages based on **time** or **size**.  
- Keeps a full history until it expires.  
- Good for: event history, audits, replaying all events.

**Log Compaction**  
- Keeps **only the latest value per key** (plus some older records for safety).  
- Deletes obsolete records for the same key.  
- Good for:  
  - **CDC (Change Data Capture)**: latest state of DB rows.  
  - **State topics**: user profile, account balance, configuration, etc.

---

## 2. How Log Compaction Works

- Topic must be configured with `cleanup.policy=compact` (or `compact,delete`).
- For each **key**, Kafka keeps the **most recent record** and eventually removes older ones.
- Compaction runs in the background; it’s **not immediate**.
- Consumers can reconstruct the **current state** by reading the compacted topic from the beginning.

---

## 3. Tombstones

- A **tombstone** is a record with:
  - `key = someKey`
  - `value = null`
- Meaning: “delete this key’s state.”
- During compaction, Kafka removes:
  - The tombstone itself (after some time).
  - All previous records with that key.
- Used in CDC/state topics to represent **deletes**.

**Example (conceptual):**

Input records in topic (key → value):
1. `user-1 → {name: "Alice", city: "NY"}`
2. `user-1 → {name: "Alice", city: "LA"}`
3. `user-2 → {name: "Bob", city: "SF"}`
4. `user-1 → null`  (tombstone: delete user-1)

After compaction, topic might contain:
- `user-2 → {name: "Bob", city: "SF"}`  
- (user-1 records removed)

---

## 4. Kafka Topic Configuration (Compacted Topic)

Example `server.properties` or topic-level config:

```properties
# Enable compaction
cleanup.policy=compact

# Optional: control compaction behavior
min.cleanable.dirty.ratio=0.5
segment.ms=3600000
```

Using `kafka-topics`:

```bash
kafka-topics.sh --create \
  --topic user-state \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.5
```

## 5. Spring Boot Producer Example (CDC / Latest State)

### 5.1. Producer configuration (application.yml)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
```

### 5.2. DTO

```java
public class UserEvent {
    private String userId;
    private String name;
    private String city;
    // getters/setters/constructors
}
```

### 5.3. Producer service

```java
@Service
public class UserStateProducer {

    private final KafkaTemplate<String, UserEvent> kafkaTemplate;
    private static final String TOPIC = "user-state";

    public UserStateProducer(KafkaTemplate<String, UserEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    // Upsert (create/update) user state
    public void sendUserState(UserEvent event) {
        kafkaTemplate.send(TOPIC, event.getUserId(), event);
    }

    // Delete user state (tombstone)
    public void deleteUser(String userId) {
        kafkaTemplate.send(TOPIC, userId, null); // tombstone
    }
}
```

### 5.4. Example inputs/outputs

**Input (upsert):**

```java
userStateProducer.sendUserState(
    new UserEvent("user-1", "Alice", "NY")
);
userStateProducer.sendUserState(
    new UserEvent("user-1", "Alice", "LA")
);
userStateProducer.sendUserState(
    new UserEvent("user-2", "Bob", "SF")
);
```

Topic (before compaction):

1. `key=user-1, value={name:"Alice", city:"NY"}`
2. `key=user-1, value={name:"Alice", city:"LA"}`
3. `key=user-2, value={name:"Bob", city:"SF"}`

**Input (delete):**

```java
userStateProducer.deleteUser("user-1");
```

Topic (before compaction):

4. `key=user-1, value=null` (tombstone)

After compaction, effective state:

- `user-1` → deleted (no record)
- `user-2` → `{name:"Bob", city:"SF"}`

## 6. Spring Boot Consumer Example (Rebuild Latest State)

```java
@Service
public class UserStateConsumer {

    // In-memory state store (for demo)
    private final Map<String, UserEvent> userState = new ConcurrentHashMap<>();

    @KafkaListener(topics = "user-state", groupId = "user-state-consumer")
    public void consume(ConsumerRecord<String, UserEvent> record) {
        String key = record.key();
        UserEvent value = record.value();

        if (value == null) {
            // tombstone: delete from local state
            userState.remove(key);
        } else {
            // upsert latest state
            userState.put(key, value);
        }
    }

    public UserEvent getUser(String userId) {
        return userState.get(userId);
    }
}
```

**Behavior:**

- When consumer reads `user-1 → {Alice, NY}` then `user-1 → {Alice, LA}`, the map ends with LA.
- When it reads `user-1 → null`, `user-1` is removed from the map.
- After reading the entire compacted topic from the beginning, `userState` holds the **current state** of all users.

---

## 7. Typical Use Cases

- **CDC from DB**:  
  - Insert/update → send key/value.  
  - Delete → send tombstone (key + null).
- **Stateful microservices**:  
  - Store latest configuration, user settings, balances.
- **Kafka Streams state stores**:  
  - Internal changelog topics are compacted to keep only latest state.
