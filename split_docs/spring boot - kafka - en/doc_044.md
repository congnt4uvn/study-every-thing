
Here’s a concise summary of the key Kafka consumer configs and how they relate, plus a Spring Boot example.

---

## Key Consumer Configs

### 1. `max.poll.records`
- **What it is**: Maximum number of records returned in a single `poll()` call.
- **Why it matters**: Controls batch size → affects processing latency and throughput.
- **Trade‑off**:
  - Higher value: better throughput, but each poll may take longer to process.
  - Lower value: more frequent polls, lower per-batch latency, but more overhead.

### 2. `max.poll.interval.ms`
- **What it is**: Maximum delay between two `poll()` calls before the consumer is considered dead.
- **Default**: 300000 ms (5 minutes).
- **Why it matters**: If your processing of a batch takes longer than this, Kafka will think the consumer is stuck and trigger a rebalance.
- **Rule of thumb**: Must be **greater than** the worst-case processing time of one `poll()` batch.

### 3. `session.timeout.ms`
- **What it is**: Time the group coordinator waits for heartbeats before marking the consumer as dead.
- **Default**: ~10 seconds (varies by version).
- **Why it matters**: If heartbeats stop for longer than this, the consumer is removed from the group and a rebalance occurs.
- **Relation**: Must be **greater than** `heartbeat.interval.ms`.

### 4. `heartbeat.interval.ms`
- **What it is**: How often the consumer sends heartbeats to the group coordinator.
- **Default**: ~3 seconds (varies by version).
- **Why it matters**: More frequent heartbeats detect failures faster but add overhead.
- **Rule**: Typically `session.timeout.ms` is 3× `heartbeat.interval.ms`.

---

## How They Work Together

- The consumer must:
  - Call `poll()` frequently enough to:
    - Send heartbeats (directly or indirectly).
    - Not exceed `max.poll.interval.ms`.
- If:
  - Processing a batch (size controlled by `max.poll.records`) takes too long → risk exceeding `max.poll.interval.ms`.
  - Heartbeats are not sent within `session.timeout.ms` → consumer is considered dead.

So you tune them together:

- Large `max.poll.records` → increase `max.poll.interval.ms`.
- Long processing per record → increase `max.poll.interval.ms` or reduce `max.poll.records`.
- Want faster failure detection → reduce `session.timeout.ms` and `heartbeat.interval.ms` (keeping the required relationship).

## Spring Boot / Spring Kafka Example

### `application.yml` configuration

```yaml
spring:
  kafka:
    consumer:
      bootstrap-servers: localhost:9092
      group-id: my-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer

      # Core tuning parameters
      properties:
        max.poll.records: 100          # batch size per poll
        max.poll.interval.ms: 600000   # 10 minutes
        session.timeout.ms: 15000      # 15 seconds
        heartbeat.interval.ms: 5000    # 5 seconds
```

### Listener example

```java
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class MyKafkaConsumer {

    @KafkaListener(topics = "my-topic", groupId = "my-group")
    public void listen(ConsumerRecord<String, String> record) {
        // Input: message from Kafka
        String key = record.key();
        String value = record.value();

        // Example processing
        System.out.printf("Received message: key=%s, value=%s, offset=%d%n",
                key, value, record.offset());

        // Output: side effects (e.g., DB write, HTTP call, log, etc.)
    }
}
```

### Example Input / Output

- **Kafka message (input)**:
  - Topic: `my-topic`
  - Key: `"user-123"`
  - Value: `"{"event":"LOGIN","timestamp":"2026-05-20T10:00:00Z"}"`

- **Application log (output)**:
```text
Received message: key=user-123, value={"event":"LOGIN","timestamp":"2026-05-20T10:00:00Z"}, offset=42
```

## Simple Tuning Scenario

- You expect each record to take up to **200 ms** to process.
- You set `max.poll.records = 100`.

Worst-case processing time per batch ≈ 100 × 200 ms = 20,000 ms (20 s).

So you should set:

```properties
max.poll.interval.ms = 60000   # e.g., 60 seconds (must be > 20 seconds)
session.timeout.ms = 15000
heartbeat.interval.ms = 5000
```

This way:
- The consumer has enough time to process a batch before Kafka considers it stuck.
- Heartbeats are sent often enough to keep the session alive.
