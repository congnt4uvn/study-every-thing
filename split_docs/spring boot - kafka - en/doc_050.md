
Here’s a concise summary and some practical Spring Boot + Kafka examples around **consumer reliability** and **poison message handling** (DLQ + retries).

---

## Key Ideas

1. **Poison messages**  
   - Messages that always fail processing (e.g., invalid format, missing required fields, business rule violations).  
   - If not handled, they can cause:
     - Infinite retry loops
     - Consumer lag
     - Service instability

2. **Retry patterns**
   - **Immediate retries**: Retry within the same consumer call (e.g., with `RetryTemplate` or manual loops).
   - **Backoff retries**: Wait between retries (fixed or exponential backoff).
   - **Topic-based retries**: Send failed messages to a *retry topic* with delayed consumption.

3. **Dead Letter Queue (DLQ)**
   - A dedicated topic (e.g., `my-topic.DLT`) where messages go after exceeding retry attempts.
   - Allows:
     - Manual inspection
     - Offline reprocessing
     - Avoiding blocking the main consumer

4. **Goals**
   - Ensure the consumer keeps processing other messages.
   - Preserve failed messages for later analysis.
   - Make failures observable and traceable (headers, error info, stack traces).

## Basic Spring Kafka Setup with Error Handling

### 1. Consumer Configuration with Error Handler + DLQ

```java
@Configuration
@EnableKafka
public class KafkaConsumerConfig {

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "my-group");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        return new DefaultKafkaConsumerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate(ProducerFactory<String, String> pf) {
        return new KafkaTemplate<>(pf);
    }

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory,
            KafkaTemplate<String, String> kafkaTemplate) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);

        // Retry 3 times with 1 second backoff, then send to DLT
        DeadLetterPublishingRecoverer recoverer =
                new DeadLetterPublishingRecoverer(kafkaTemplate,
                        (record, ex) -> new TopicPartition(record.topic() + ".DLT", record.partition()));

        DefaultErrorHandler errorHandler =
                new DefaultErrorHandler(recoverer, new FixedBackOff(1000L, 3L));

        // Optionally ignore some exceptions (no retry, no DLQ)
        // errorHandler.addNotRetryableExceptions(IllegalArgumentException.class);

        factory.setCommonErrorHandler(errorHandler);
        return factory;
    }
}
```

### 2. Consumer with Potential Poison Messages

```java
@Service
public class MyKafkaConsumer {

    @KafkaListener(topics = "my-topic", groupId = "my-group")
    public void consume(String message) {
        // Example: poison message if not valid JSON
        if (!message.startsWith("{")) {
            throw new IllegalArgumentException("Invalid JSON format: " + message);
        }

        // Normal processing
        System.out.println("Processed message: " + message);
    }
}
```

## What Happens at Runtime (Example I/O)

### Input Messages

Topic: `my-topic`

1. Valid message:
   ```text
   {"orderId": 123, "status": "NEW"}
   ```
2. Poison message:
   ```text
   INVALID_JSON
   ```

### Processing Flow

1. **Valid message**
   - Consumer receives `{"orderId": 123, "status": "NEW"}`
   - `consume()` runs successfully
   - Output (log):
     ```text
     Processed message: {"orderId": 123, "status": "NEW"}
     ```

2. **Poison message**
   - Consumer receives `INVALID_JSON`
   - `consume()` throws `IllegalArgumentException`
   - `DefaultErrorHandler`:
     - Retries up to 3 times with 1 second delay
     - After 3 failures, sends the record to `my-topic.DLT`
   - Output (logs):
     ```text
     Error processing message, attempt 1/3
     Error processing message, attempt 2/3
     Error processing message, attempt 3/3
     Sent to DLT: topic=my-topic.DLT, value=INVALID_JSON
     ```

### DLQ Consumer (Optional)

```java
@Service
public class DlqConsumer {

    @KafkaListener(topics = "my-topic.DLT", groupId = "my-dlq-group")
    public void consumeDlq(String message,
                           @Header(KafkaHeaders.DLT_EXCEPTION_MESSAGE) String exceptionMessage,
                           @Header(KafkaHeaders.DLT_ORIGINAL_TOPIC) String originalTopic) {

        System.out.printf("DLQ message from %s: %s (error: %s)%n",
                originalTopic, message, exceptionMessage);

        // Here you can:
        // - store in DB
        // - send alert
        // - attempt manual fix/reprocess
    }
}
```

**DLQ Input:**  
Topic: `my-topic.DLT`  
Message: `INVALID_JSON`

**DLQ Output (log):**
```text
DLQ message from my-topic: INVALID_JSON (error: Invalid JSON format: INVALID_JSON)
```

## Variations

- **No retries, direct DLQ**: set `FixedBackOff(0L, 0L)` in `DefaultErrorHandler`.
- **Different DLQ per topic**: use a custom `BiFunction<ConsumerRecord<?,?>, Exception, TopicPartition>` in `DeadLetterPublishingRecoverer`.
- **Retry topics**: use a pattern like `my-topic.retry` with delayed consumption before DLQ.

---

**In summary:**  
Use Spring Kafka’s `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` to implement robust consumer reliability: retry transient failures a limited number of times, and route poison messages to a DLQ topic for later inspection and recovery.
