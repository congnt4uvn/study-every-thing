
Here’s a concise summary and practical examples around producer reliability in Kafka with Spring Boot, focusing on these exceptions:

- `TimeoutException`
- `RecordTooLargeException`
- `NotEnoughReplicasException` (or `NotEnoughReplicasAfterAppendException`)

---

## 1. General Producer Reliability Settings

In `application.yml`:

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      acks: all              # wait for all in-sync replicas
      retries: 5             # retry on transient errors
      batch-size: 16384
      buffer-memory: 33554432
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      properties:
        delivery.timeout.ms: 120000
        request.timeout.ms: 30000
        max.block.ms: 60000
```

---

## 2. Handling `TimeoutException`

**Meaning:**  
The producer did not receive an acknowledgment from the broker within the configured timeout (e.g., network issues, broker slow/unavailable).

**Key ideas:**

- Tune `delivery.timeout.ms`, `request.timeout.ms`, and `retries`.
- Implement retry with backoff in your application logic.
- Log and possibly send to a dead-letter topic or store for later retry.

**Spring Boot example (async send with callback):**

```java
@Service
public class ReliableProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public ReliableProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendReliable(String topic, String key, String value) {
        ListenableFuture<SendResult<String, String>> future =
                kafkaTemplate.send(topic, key, value);

        future.addCallback(
            result -> {
                // Success
                RecordMetadata meta = result.getRecordMetadata();
                System.out.printf("Sent to %s-%d@%d%n",
                        meta.topic(), meta.partition(), meta.offset());
            },
            ex -> {
                if (ex.getCause() instanceof org.apache.kafka.common.errors.TimeoutException) {
                    // Handle timeout: log, custom retry, alert, etc.
                    System.err.println("Timeout sending message, will retry later: " + ex.getMessage());
                    // customRetry(topic, key, value);
                } else {
                    System.err.println("Send failed: " + ex.getMessage());
                }
            }
        );
    }
}
```

**Example input/output:**

- Input: `sendReliable("orders", "order-123", "{...json...}")`
- Output (success): `Sent to orders-0@42`
- Output (timeout): `Timeout sending message, will retry later: ...`

## 3. Handling `RecordTooLargeException`

**Meaning:**  
The message size exceeds `max.request.size` (producer) or `message.max.bytes` / `max.message.bytes` (broker/topic).

**Key ideas:**

- Check and adjust:
  - Producer: `max.request.size`
  - Broker: `message.max.bytes`
  - Topic: `max.message.bytes`
- Compress payload or split large messages into chunks.
- Validate size before sending.

**Size check example:**

```java
@Service
public class SizeCheckingProducer {

    private static final int MAX_BYTES = 900_000; // example limit

    private final KafkaTemplate<String, byte[]> kafkaTemplate;

    public SizeCheckingProducer(KafkaTemplate<String, byte[]> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendWithSizeCheck(String topic, String key, byte[] payload) {
        if (payload.length > MAX_BYTES) {
            throw new IllegalArgumentException("Payload too large, size=" + payload.length);
        }

        kafkaTemplate.send(topic, key, payload).addCallback(
            result -> System.out.println("Sent large message OK"),
            ex -> {
                if (ex.getCause() instanceof org.apache.kafka.common.errors.RecordTooLargeException) {
                    System.err.println("Record too large, consider splitting/compressing");
                } else {
                    System.err.println("Send failed: " + ex.getMessage());
                }
            }
        );
    }
}
```

**Example input/output:**

- Input: payload size 500 KB → send OK.
- Input: payload size 2 MB → throws `IllegalArgumentException` or `RecordTooLargeException` if not pre-checked.

## 4. Handling `NotEnoughReplicas` / `NotEnoughReplicasAfterAppendException`

**Meaning:**  
The producer requested a certain level of durability (`acks=all`), but not enough in-sync replicas (ISR) were available to satisfy it.

**Key ideas:**

- Caused by broker failures, ISR shrink, or under-replicated partitions.
- You can:
  - Lower `acks` (less durable) or
  - Keep `acks=all` and implement retry/backoff and alerting.
- Monitor cluster health and ISR.

**Example with specific handling:**

```java
import org.apache.kafka.common.errors.NotEnoughReplicasException;
import org.apache.kafka.common.errors.NotEnoughReplicasAfterAppendException;

@Service
public class ReplicaAwareProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public ReplicaAwareProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendWithReplicaCheck(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value).addCallback(
            result -> System.out.println("Message sent with full replication"),
            ex -> {
                Throwable cause = ex.getCause();
                if (cause instanceof NotEnoughReplicasException ||
                    cause instanceof NotEnoughReplicasAfterAppendException) {
                    System.err.println("Not enough replicas available, will retry later / alert ops");
                    // e.g. push to retry queue, send alert, etc.
                } else {
                    System.err.println("Send failed: " + ex.getMessage());
                }
            }
        );
    }
}
```

**Example input/output:**

- Input: `sendWithReplicaCheck("payments", "pay-1", "{...}")`
- Output (healthy cluster): `Message sent with full replication`
- Output (replica issue): `Not enough replicas available, will retry later / alert ops`

---

## 5. Summary of Main Ideas

- Use `acks=all`, `retries`, and proper timeouts for reliability.
- `TimeoutException`: network/latency issues; handle with retries and monitoring.
- `RecordTooLargeException`: message exceeds size limits; compress, split, or increase limits.
- `NotEnoughReplicas*`: cluster replication problem; keep `acks=all`, retry, and alert operations.
- Always add callbacks or error handlers around `KafkaTemplate.send(...)` to react to these failures.
