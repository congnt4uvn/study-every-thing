
**Topic:** Consumer performance – monitoring consumer lag in Kafka (with Spring Boot)

---

## 1. What is Consumer Lag?

- **Consumer lag** = `latest offset in partition` − `current committed offset of consumer`
- It shows **how far behind** a consumer is from the head of the topic.
- High lag = consumer cannot keep up with incoming messages → risk of delays or data backlog.

---

## 2. Why Monitor Consumer Lag?

- Detect **slow consumers** or **performance bottlenecks**.
- Ensure **near real-time processing**.
- Plan **scaling** (more consumer instances, more partitions).
- Detect **consumer failures** (stuck or dead consumers).

---

## 3. How to Monitor Consumer Lag

### A. Kafka Built-in Tools

1. **kafka-consumer-groups.sh** (CLI)

```bash
# Example: check lag for a consumer group
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group my-consumer-group \
  --describe
```

**Sample output:**

| GROUP             | TOPIC   | PARTITION | CURRENT-OFFSET | LOG-END-OFFSET | LAG | CONSUMER-ID | HOST       | CLIENT-ID |
|------------------|---------|-----------|----------------|----------------|-----|-------------|------------|----------|
| my-consumer-group| myTopic | 0         | 1500           | 1600           | 100 | ...         | ...        | ...      |

- `LAG` column is the consumer lag per partition.

2. **Kafka metrics via JMX**

- Kafka brokers and consumers expose metrics like:
  - `records-lag-max`
  - `records-lag`
  - `records-lag-avg`
- These can be scraped by Prometheus, visualized in Grafana, etc.

## 4. Monitoring Lag in Spring Boot (Spring Kafka)

### A. Enable Kafka Metrics in Spring Boot

Spring Boot auto-configures Micrometer metrics for Kafka if Micrometer is on the classpath.

**Gradle example:**

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-actuator'
    implementation 'io.micrometer:micrometer-registry-prometheus'
    implementation 'org.springframework.kafka:spring-kafka'
}
```

**application.yml:**

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
```

Then you can scrape `/actuator/prometheus` and look for metrics like:

- `kafka_consumer_records_lag_max`
- `kafka_consumer_records_lag`
- `kafka_consumer_records_lag_avg`

**Example metric output (Prometheus format):**

```text
kafka_consumer_records_lag_max{client_id="consumer-1",topic="myTopic",partition="0"} 100.0
```

This means the **max lag** for that consumer on `myTopic-0` is 100 records.

---

### B. Programmatic Lag Check in Spring (AdminClient)

You can compute lag by comparing **end offsets** with **consumer group offsets**.

**Code example (Spring Boot component):**

```java
import org.apache.kafka.clients.admin.*;
import org.apache.kafka.common.TopicPartition;
import org.springframework.stereotype.Component;

import java.time.Duration;
import java.util.*;
import java.util.concurrent.ExecutionException;

@Component
public class ConsumerLagMonitor {

    private final AdminClient adminClient;

    public ConsumerLagMonitor() {
        Properties props = new Properties();
        props.put(AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        this.adminClient = AdminClient.create(props);
    }

    public Map<TopicPartition, Long> getConsumerLag(String groupId, String topic) throws ExecutionException, InterruptedException {
        // 1. Get partitions for topic
        List<TopicPartition> partitions = new ArrayList<>();
        DescribeTopicsResult describeTopicsResult = adminClient.describeTopics(Collections.singletonList(topic));
        TopicDescription description = describeTopicsResult.values().get(topic).get();
        description.partitions().forEach(p ->
                partitions.add(new TopicPartition(topic, p.partition()))
        );

        // 2. Get end offsets
        Map<TopicPartition, ListOffsetsResult.ListOffsetsResultInfo> endOffsets =
                adminClient.listOffsets(
                        partitions.stream().collect(
                                HashMap::new,
                                (m, tp) -> m.put(tp, OffsetSpec.latest()),
                                HashMap::putAll
                        )
                ).all().get();

        // 3. Get committed offsets for consumer group
        ListConsumerGroupOffsetsResult groupOffsetsResult =
                adminClient.listConsumerGroupOffsets(groupId);
        Map<TopicPartition, OffsetAndMetadata> groupOffsets =
                groupOffsetsResult.partitionsToOffsetAndMetadata().get();

        // 4. Compute lag
        Map<TopicPartition, Long> lagPerPartition = new HashMap<>();
        for (TopicPartition tp : partitions) {
            long logEndOffset = endOffsets.get(tp).offset();
            long committedOffset = Optional.ofNullable(groupOffsets.get(tp))
                    .map(OffsetAndMetadata::offset)
                    .orElse(0L);
            long lag = logEndOffset - committedOffset;
            lagPerPartition.put(tp, lag);
        }

        return lagPerPartition;
    }
}
```

**Example usage:**

```java
@Autowired
private ConsumerLagMonitor lagMonitor;

public void printLag() throws Exception {
    Map<TopicPartition, Long> lag = lagMonitor.getConsumerLag("my-consumer-group", "myTopic");
    lag.forEach((tp, value) ->
            System.out.println("Partition " + tp.partition() + " lag = " + value)
    );
}
```

**Sample output:**

```text
Partition 0 lag = 100
Partition 1 lag = 0
Partition 2 lag = 50
```

## 5. What to Do When Lag is High

- Increase **consumer concurrency** (more consumer instances or threads).
- Increase **partitions** for the topic (for more parallelism).
- Optimize consumer processing:
  - Batch processing
  - Reduce heavy I/O or external calls
- Tune consumer configs:
  - `max.poll.records`
  - `fetch.min.bytes`, `fetch.max.wait.ms`
  - `max.poll.interval.ms`
