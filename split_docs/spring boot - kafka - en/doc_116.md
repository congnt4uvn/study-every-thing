
Here’s a concise summary and some practical code around **multi-environment schema evolution with Kafka + Spring Boot**.

---

## Key Ideas

1. **Multiple environments**  
   - Typical setup: `dev` → `test` → `staging` → `prod`.  
   - Each environment may have:
     - Its own Kafka cluster
     - Its own Schema Registry
     - Different versions of schemas deployed at different times

2. **Schema evolution**  
   - Producers and consumers evolve over time (new fields, removed fields, type changes).
   - Schemas must evolve in a **compatible** way so that:
     - Old consumers can read messages produced by new producers.
     - New consumers can read messages produced by old producers.

3. **Compatibility modes** (Schema Registry)  
   Common modes:
   - **BACKWARD**: New schema can read data written with the previous schema.
   - **FORWARD**: Old schema can read data written with the new schema.
   - **FULL**: Both backward and forward compatible.
   - **NONE**: No compatibility checks.
   - For multi-environment setups, you typically:
     - Use **FULL** or **BACKWARD** in higher environments (staging/prod).
     - Possibly more relaxed in dev, but keep it close to prod to avoid surprises.

4. **Promotion across environments**  
   - Flow:  
     1. Change schema in dev → test with dev apps.  
     2. Promote schema to test → verify compatibility with test apps.  
     3. Promote to staging → run integration tests.  
     4. Promote to prod → ensure no breaking changes.
   - Use **the same subject name and versioning strategy** across environments, but separate registries/clusters.

5. **Safe schema changes**  
   Typically safe (with BACKWARD or FULL):
   - Adding a field with a default value.
   - Making a field optional (nullable).
   Typically unsafe:
   - Removing a required field.
   - Changing field type incompatibly (e.g., `string` → `int`).
   - Renaming fields without aliases (in Avro).

## Example: Avro Schema Evolution

### v1 schema (User)

```json
{
  "type": "record",
  "name": "User",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "email", "type": "string" }
  ]
}
```

### v2 schema (add optional field)

```json
{
  "type": "record",
  "name": "User",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "email", "type": "string" },
    {
      "name": "phone",
      "type": ["null", "string"],
      "default": null
    }
  ]
}
```

- This is **backward compatible**: old messages (without `phone`) can still be read; new messages have `phone` (or null).

## Spring Boot + Kafka + Schema Registry Example

Assume Confluent Schema Registry and Avro.

### Maven dependencies

```xml
<dependencies>
    <!-- Spring Kafka -->
    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>

    <!-- Confluent Avro serializer/deserializer -->
    <dependency>
        <groupId>io.confluent</groupId>
        <artifactId>kafka-avro-serializer</artifactId>
        <version>7.6.0</version>
    </dependency>
</dependencies>
```

### application.yml (per environment)

**dev:**

```yaml
spring:
  kafka:
    bootstrap-servers: dev-kafka:9092
    properties:
      schema.registry.url: http://dev-schema-registry:8081
      specific.avro.reader: true
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
```

**prod:**

```yaml
spring:
  kafka:
    bootstrap-servers: prod-kafka:9092
    properties:
      schema.registry.url: http://prod-schema-registry:8081
      specific.avro.reader: true
```

Each environment points to its own **Kafka cluster** and **Schema Registry**.

## Producer Example (Spring Boot)

Assume Avro class `com.example.avro.User` generated from the schema.

```java
@Service
public class UserProducer {

    private final KafkaTemplate<String, User> kafkaTemplate;
    private static final String TOPIC = "users";

    public UserProducer(KafkaTemplate<String, User> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUser(String id, String email, String phone) {
        User user = User.newBuilder()
                .setId(id)
                .setEmail(email)
                .setPhone(phone) // in v1 this field doesn’t exist; in v2 it does
                .build();

        kafkaTemplate.send(TOPIC, id, user);
    }
}
```

### Example input → output

**Input (method call):**

```java
userProducer.sendUser("u-123", "user@example.com", "123-456-7890");
```

**Output (Kafka message on topic `users`):**

- Key: `"u-123"`
- Value (logical Avro record):

```json
{
  "id": "u-123",
  "email": "user@example.com",
  "phone": "123-456-7890"
}
```

In an environment still using v1 schema, the consumer will ignore `phone` (if using a generic reader or older specific class), or you must deploy the new schema first.

## Consumer Example (Spring Boot)

```java
@Service
public class UserConsumer {

    @KafkaListener(topics = "users", groupId = "user-service")
    public void consume(User user) {
        // user.getPhone() may be null if produced by old schema
        System.out.printf("Received user: id=%s, email=%s, phone=%s%n",
                user.getId(), user.getEmail(), user.getPhone());
    }
}
```

**Example consumed output (console):**

```text
Received user: id=u-123, email=user@example.com, phone=123-456-7890
```

or, for old messages:

```text
Received user: id=u-001, email=old@example.com, phone=null
```

## Environment-Specific Schema Compatibility

You typically configure compatibility per environment in Schema Registry (e.g., via REST API):

```bash
# Set FULL compatibility for prod
curl -X PUT \
  -H "Content-Type: application/json" \
  --data '{"compatibility": "FULL"}' \
  http://prod-schema-registry:8081/config
```

- **dev**: maybe `BACKWARD` or `FULL` but can be more flexible.
- **prod**: strict `FULL` or at least `BACKWARD` to avoid breaking running consumers.

---

## Practical Pattern

1. **Define schema v1** → deploy to dev, test producer/consumer.
2. **Evolve schema to v2** (only compatible changes).
3. **Register v2 in dev Schema Registry** → run tests.
4. **Promote v2 to test/staging** → run integration tests.
5. **Promote v2 to prod**:
   - First deploy consumers that can handle both v1 and v2.
   - Then deploy producers that start sending v2 messages.

This pattern ensures safe schema evolution across multiple environments without breaking existing Kafka clients.
