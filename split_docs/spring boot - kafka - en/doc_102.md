
Here’s a concise, self-contained summary focused on **data contracts**, **breaking changes**, and **deprecation** in a Kafka + Spring Boot context, with code examples.

---

## 1. Core Idea: Data Contracts in Kafka

A **data contract** is an agreement between producers and consumers about:
- Topic name
- Message schema (fields, types, semantics)
- Compatibility rules (what changes are allowed)

In Kafka, this is usually enforced via:
- Schema Registry (e.g., Confluent Schema Registry)
- Avro / Protobuf / JSON Schema
- Versioning and compatibility settings (BACKWARD, FORWARD, FULL, etc.)

Goal: Allow services to evolve **without breaking existing consumers**.

---

## 2. What Is a Breaking Change?

A change is **breaking** if existing consumers or producers can no longer:
- Deserialize messages
- Understand required fields
- Rely on previous semantics

Typical **breaking changes**:
- Removing a required field
- Renaming a field without backward-compatible alias
- Changing field type incompatibly (e.g., `string` → `int`)
- Changing meaning of a field in a way that old consumers misinterpret

**Non-breaking (usually safe) changes**:
- Adding a new optional field with a default
- Adding new enum values (if consumers ignore unknown values)
- Deprecating a field but keeping it in the schema

---

## 3. Strategy: Evolve Schemas Safely

### 3.1 Use Schema Registry & Compatibility

Example: Confluent Schema Registry with Avro

```properties
# application.properties
spring.kafka.properties.schema.registry.url=http://localhost:8081
spring.kafka.properties.specific.avro.reader=true
spring.kafka.producer.value-serializer=io.confluent.kafka.serializers.KafkaAvroSerializer
spring.kafka.consumer.value-deserializer=io.confluent.kafka.serializers.KafkaAvroDeserializer
```

Set compatibility mode (e.g., BACKWARD) in Schema Registry:

```bash
curl -X PUT \
  -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  --data '{"compatibility": "BACKWARD"}' \
  http://localhost:8081/config/my-topic-value
```

This prevents registering schemas that would break existing consumers.

## 4. Example: Evolving a Message Schema

### 4.1 Initial Schema (v1)

```avro
{
  "type": "record",
  "name": "OrderEvent",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "amount",  "type": "double" }
  ]
}
```

Java class (generated or manual):

```java
public class OrderEventV1 {
    private String orderId;
    private double amount;
    // getters/setters
}
```

Producer (Spring Boot):

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, OrderEventV1> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, OrderEventV1> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, double amount) {
        OrderEventV1 event = new OrderEventV1();
        event.setOrderId(orderId);
        event.setAmount(amount);

        kafkaTemplate.send("orders", orderId, event);
    }
}
```

Consumer:

```java
@KafkaListener(topics = "orders", groupId = "billing-service")
public void handleOrder(OrderEventV1 event) {
    // INPUT example:
    // { "orderId": "ORD-123", "amount": 99.99 }

    System.out.println("Received order " + event.getOrderId()
        + " amount=" + event.getAmount());

    // OUTPUT (log):
    // Received order ORD-123 amount=99.99
}
```

---

### 4.2 Non-Breaking Change: Add Optional Field (v2)

Add a new field `currency` with default:

```avro
{
  "type": "record",
  "name": "OrderEvent",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "amount",  "type": "double" },
    { "name": "currency", "type": "string", "default": "USD" }
  ]
}
```

Generated Java:

```java
public class OrderEventV2 {
    private String orderId;
    private double amount;
    private String currency = "USD";
    // getters/setters
}
```

Producer using v2:

```java
public void sendOrder(String orderId, double amount, String currency) {
    OrderEventV2 event = new OrderEventV2();
    event.setOrderId(orderId);
    event.setAmount(amount);
    event.setCurrency(currency);

    kafkaTemplate.send("orders", orderId, event);
}
```

**Compatibility behavior**:
- Old consumers (v1) still work: they ignore `currency`.
- New consumers (v2) reading old messages get `currency = "USD"` (default).

Example message on the wire (v2 producer):

```json
{
  "orderId": "ORD-456",
  "amount": 150.0,
  "currency": "EUR"
}
```

Consumer log (v2):

```java
@KafkaListener(topics = "orders", groupId = "reporting-service")
public void handleOrder(OrderEventV2 event) {
    System.out.printf("Order %s: %.2f %s%n",
        event.getOrderId(), event.getAmount(), event.getCurrency());
}
```

Output:

```text
Order ORD-456: 150.00 EUR
```

## 5. Deprecation Strategy

When you need to remove or change a field:

1. **Deprecate first** (keep field, mark as deprecated in code and docs).
2. **Stop using it** in producers and consumers.
3. Wait until all consumers are updated.
4. Only then consider removing it in a new major version / new topic.

Example: Deprecate `amount` in favor of `grossAmount` and `netAmount`.

Schema v3 (non-breaking, but deprecating):

```avro
{
  "type": "record",
  "name": "OrderEvent",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "orderId", "type": "string" },
    { "name": "amount",  "type": ["null", "double"], "default": null },  // deprecated
    { "name": "currency", "type": "string", "default": "USD" },
    { "name": "grossAmount", "type": "double", "default": 0.0 },
    { "name": "netAmount",   "type": "double", "default": 0.0 }
  ]
}
```

Java:

```java
public class OrderEventV3 {
    private String orderId;

    @Deprecated
    private Double amount; // nullable

    private String currency = "USD";
    private double grossAmount;
    private double netAmount;
    // getters/setters
}
```

Producer (new logic, not using deprecated field):

```java
public void sendOrder(String orderId, double gross, double net, String currency) {
    OrderEventV3 event = new OrderEventV3();
    event.setOrderId(orderId);
    event.setGrossAmount(gross);
    event.setNetAmount(net);
    event.setCurrency(currency);
    event.setAmount(null); // explicitly not used

    kafkaTemplate.send("orders", orderId, event);
}
```

## 6. When You Truly Need a Breaking Change

If you must:
- Remove fields
- Change types incompatibly
- Change semantics drastically

Use one of these patterns:

1. **New topic** (recommended):
   - `orders.v1` → `orders.v2`
   - Run both topics in parallel during migration.
   - Consumers gradually switch to `orders.v2`.

2. **New schema subject**:
   - `orders-value` → `orders-v2-value` in Schema Registry.

Example: New topic configuration in Spring Boot:

```yaml
spring:
  kafka:
    template:
      default-topic: orders.v2
```

Producer:

```java
kafkaTemplate.send("orders.v2", orderId, newOrderEventV2);
```

Consumers subscribe to `orders.v2` once they are ready.

---

## 7. Practical Guidelines

- Always use a **Schema Registry** and enforce **compatibility**.
- Prefer **additive changes** (new optional fields) over destructive ones.
- Use **deprecation** instead of immediate removal.
- For hard breaking changes, use **new topics** and run both versions during migration.
- Document your **data contract**: fields, types, meaning, and evolution rules.
