
Here’s a concise summary and some practical examples around **Kafka producer performance tuning** in **Spring Boot** focusing on **batch size**, **linger.ms**, and **compression**.

---

## Key Ideas

1. **batch.size**
   - Maximum size (in bytes) of a batch of records sent to a partition.
   - Larger batch → better throughput (more data per request) but higher memory usage.
   - If too small → many small requests → lower throughput.
   - If too large → may increase latency and memory pressure.

2. **linger.ms**
   - How long the producer waits before sending a batch, even if it’s not full.
   - Higher linger.ms → more time to accumulate records → larger batches → better compression and throughput.
   - But also increases **end-to-end latency**.

3. **compression.type**
   - Compresses the data before sending to Kafka.
   - Common values: `none`, `gzip`, `snappy`, `lz4`, `zstd`.
   - Compression reduces network usage and can increase throughput, but uses CPU.
   - `lz4` and `zstd` are often good choices for performance.

---

## Typical Tuning Strategy

- Start with:
  - `batch.size`: 32 KB – 128 KB (e.g., `32768` or `65536`)
  - `linger.ms`: 5–20 ms
  - `compression.type`: `lz4` or `zstd`
- Measure:
  - Throughput (records/sec, MB/sec)
  - Latency (producer send time, end-to-end)
  - CPU usage (producer and brokers)
- Adjust:
  - Increase `batch.size` and `linger.ms` if you want more throughput and can tolerate more latency.
  - Decrease them if you need lower latency.

## Spring Boot Kafka Producer Configuration (Java)

**application.yml**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer

      # Performance tuning
      batch-size: 65536          # 64 KB
      linger-ms: 10              # wait up to 10 ms to fill batch
      compression-type: lz4      # or zstd, snappy, gzip
      acks: 1                    # trade-off between durability and latency
      retries: 3
      buffer-memory: 33554432    # 32 MB
```

**Producer code example**

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        kafkaTemplate.send("orders-topic", orderId, payload)
                     .whenComplete((result, ex) -> {
                         if (ex != null) {
                             System.err.println("Send failed: " + ex.getMessage());
                         } else {
                             System.out.println("Sent to partition " +
                                 result.getRecordMetadata().partition() +
                                 " offset " + result.getRecordMetadata().offset());
                         }
                     });
    }
}
```

## Example: Effect of Settings (Conceptual)

Assume you send **10,000 messages**, each **1 KB**:

### Case A: Small batch, no linger, no compression
- `batch.size = 16384` (16 KB)
- `linger.ms = 0`
- `compression.type = none`

**Behavior:**
- Many small requests (about 10,000 / ~16 = ~625 requests).
- Higher network overhead.
- Lower throughput, lower latency per message.

### Case B: Larger batch, small linger, compression
- `batch.size = 65536` (64 KB)
- `linger.ms = 10`
- `compression.type = lz4`

**Behavior:**
- Fewer, larger requests (about 10,000 / ~64 = ~156 requests).
- Better compression ratio.
- Higher throughput, slightly higher latency (up to ~10 ms extra).

**Example output logs (simplified)**

Case A:
```text
Sent to partition 0 offset 100
Sent to partition 0 offset 101
...
Total time: 2.5 seconds
Throughput: ~4,000 msg/s
```

Case B:
```text
Sent to partition 0 offset 100
Sent to partition 0 offset 101
...
Total time: 1.2 seconds
Throughput: ~8,300 msg/s
```

(Exact numbers depend on hardware and environment, but pattern is typical.)
