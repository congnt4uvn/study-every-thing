
Here’s a concise summary of offset management patterns in Kafka with Spring Boot, plus code examples.

---

## 1. At-least-once

**Idea:**  
Each message is processed **one or more times**. You might get duplicates, but you don’t lose messages.

**How it works:**

- Consume record
- Process it
- Commit offset **after** successful processing
- If consumer crashes after processing but before commit → message is reprocessed

**Spring Kafka example (manual commit):**

```java
@EnableKafka
@Configuration
public class KafkaConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
        return factory;
    }
}

@Service
public class AtLeastOnceConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(String message, Acknowledgment ack) {
        // 1. Process message
        processOrder(message);

        // 2. Commit offset after successful processing
        ack.acknowledge();
    }

    private void processOrder(String message) {
        // business logic
    }
}
```

**Example I/O:**

- Input message in topic `orders`:
  ```json
  {"orderId": 101, "status": "NEW"}
  ```
- If processing succeeds and ack is called → offset committed.
- If app crashes after `processOrder` but before `ack.acknowledge()` → message is read again on restart (duplicate processing possible).

---

## 2. At-most-once

**Idea:**  
Each message is processed **zero or one time**. You never process a message twice, but you might lose some messages.

**How it works:**

- Commit offset **before** processing
- If consumer crashes after commit but before processing → message is lost

**Spring Kafka example (auto commit / early commit):**

```yaml
spring:
  kafka:
    consumer:
      enable-auto-commit: true
      auto-commit-interval: 1000
      group-id: order-service
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

```java
@Service
public class AtMostOnceConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(String message) {
        // Offsets are committed automatically as messages are polled
        // 1. Process message (may be skipped if crash happens before this)
        processOrder(message);
    }

    private void processOrder(String message) {
        // business logic
    }
}
```

**Example I/O:**

- Input message:
  ```json
  {"orderId": 102, "status": "NEW"}
  ```
- Offset may be committed as soon as message is fetched.
- If app crashes before `processOrder` → message is not reprocessed (lost).

---

## 3. Effectively-once (a.k.a. “exactly-once semantics in practice”)

**Idea:**  
Each message’s **effect** is applied **exactly once**, even if the message is processed multiple times. Achieved via **idempotent processing** and/or **transactions**.

**How it works:**

- Use Kafka transactions (producer + consumer) **or**
- Make processing idempotent (e.g., deduplicate by key, check if already processed)
- Offsets and output are committed atomically

### 3.1. Idempotent consumer (common pattern)

**Key idea:**  
Store a “processed” flag or use a unique key so reprocessing doesn’t change the result.

```java
@Service
public class EffectivelyOnceConsumer {

    private final ProcessedEventRepository repository; // e.g., JPA or Redis

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(String message) {
        OrderEvent event = parse(message);

        // Check if already processed
        if (repository.existsById(event.getOrderId())) {
            return; // ignore duplicate
        }

        // Process business logic
        processOrder(event);

        // Mark as processed
        repository.save(new ProcessedEvent(event.getOrderId()));
    }

    private OrderEvent parse(String message) {
        // parse JSON to object
        return new OrderEvent(/* ... */);
    }

    private void processOrder(OrderEvent event) {
        // business logic
    }
}
```

**Example I/O:**

- Input message (possibly delivered twice):
  ```json
  {"orderId": 103, "status": "NEW"}
  ```
- First time:
  - `existsById(103)` → false → process + save
- Second time:
  - `existsById(103)` → true → skip
- Effect: order 103 is created/updated only once.

### 3.2. Kafka transactions (read-process-write in one transaction)

Use when you consume from one topic and produce to another and want them to be atomic.

**Producer config (idempotent + transactional):**

```yaml
spring:
  kafka:
    producer:
      enable-idempotence: true
      acks: all
      retries: 10
      transaction-id-prefix: tx-order-service-
```

**Transactional processing:**

```java
@Service
public class TransactionalProcessor {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public TransactionalProcessor(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(String message) {
        kafkaTemplate.executeInTransaction(operations -> {
            // 1. Process input
            String result = process(message);

            // 2. Produce result to another topic
            operations.send("processed-orders", result);

            // Offsets are committed as part of the transaction
            return null;
        });
    }

    private String process(String message) {
        // transform message
        return message.toUpperCase();
    }
}
```

**Example I/O:**

- Input topic: `orders`
  ```json
  {"orderId": 104, "status": "NEW"}
  ```
- Output topic: `processed-orders`
  ```json
  {"orderId": 104, "status": "NEW", "processed": true}
  ```
- If crash happens inside transaction → no output message and no offset commit; message will be reprocessed, but the transaction ensures the output is not duplicated.

---

## When to use which

- **At-least-once**: default, safe for most systems; handle duplicates in your logic.
- **At-most-once**: only when losing some messages is acceptable (e.g., metrics, logs).
- **Effectively-once**: for financial/critical operations; use idempotent design and/or Kafka transactions.
