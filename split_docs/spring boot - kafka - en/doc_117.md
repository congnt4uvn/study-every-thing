
Here’s a concise summary and some practical patterns/code you can use for **multi-environment Kafka + Spring Boot** with **backward-compatible deployments** and **rolling upgrades**.

---

## Key Ideas

1. **Multi-environment setup**
   - Separate environments: `dev`, `staging`, `prod`.
   - Each environment has its own Kafka cluster (or at least its own topics/namespace).
   - Spring Boot uses profiles and external config to switch between environments.

2. **Backward-compatible deployments**
   - New versions of services must work with:
     - Old messages already in Kafka.
     - Other services that are still on the old version.
   - Use **schema evolution** rules (e.g., Avro/JSON Schema):
     - Only add optional fields with defaults.
     - Do not remove or rename fields in a breaking way.
   - Consumers should be tolerant of unknown/missing fields.

3. **Rolling upgrades**
   - Deploy new version gradually (e.g., Kubernetes rolling update).
   - During rollout, both old and new versions run at the same time.
   - Kafka topics and message formats must be compatible with both versions.
   - Use feature flags or versioned topics if you need to introduce breaking changes.

4. **Versioning strategies**
   - Prefer **schema evolution** over topic proliferation.
   - If a breaking change is unavoidable:
     - Introduce a new topic: `orders.v2`.
     - Run both consumers/producers until migration is complete.
   - Use headers or payload fields to carry a `schemaVersion` or `eventType`.

## Spring Boot + Kafka: Multi-environment Configuration

**application.yml** (common):

```yaml
spring:
  kafka:
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      group-id: order-service
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

**application-dev.yml**:

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
topic:
  orders: orders-dev
```

**application-prod.yml**:

```yaml
spring:
  kafka:
    bootstrap-servers: kafka-prod:9092
    properties:
      schema.registry.url: http://schema-registry:8081
topic:
  orders: orders
```

Use profiles:

```bash
# Dev
SPRING_PROFILES_ACTIVE=dev java -jar app.jar

# Prod
SPRING_PROFILES_ACTIVE=prod java -jar app.jar
```

## Backward-compatible Message Schema (Example)

Assume we start with:

```json
// v1 OrderCreated event
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.0
}
```

We evolve to v2:

```json
// v2 OrderCreated event (backward compatible)
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.0,
  "currency": "USD",          // new optional field with default
  "metadata": {               // new optional object
    "source": "web"
  }
}
```

Rules:
- Old consumers ignore `currency` and `metadata`.
- New consumers treat missing `currency` as default (e.g., `"USD"`).

## Spring Boot Producer (Backward-compatible)

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private final String topicName;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate,
                         @Value("${topic.orders}") String topicName) {
        this.kafkaTemplate = kafkaTemplate;
        this.topicName = topicName;
    }

    public void sendOrderCreatedV2(Order order) throws JsonProcessingException {
        ObjectMapper mapper = new ObjectMapper();

        Map<String, Object> payload = new HashMap<>();
        payload.put("orderId", order.getId());
        payload.put("customerId", order.getCustomerId());
        payload.put("totalAmount", order.getTotalAmount());
        // new fields (optional)
        payload.put("currency", order.getCurrency() != null ? order.getCurrency() : "USD");
        payload.put("metadata", Map.of("source", "web"));

        String json = mapper.writeValueAsString(payload);

        kafkaTemplate.send(topicName, order.getId(), json);
    }
}
```

**Input (Java object):**

```java
Order order = new Order("123", "C-1", 100.0, null);
orderProducer.sendOrderCreatedV2(order);
```

**Output (Kafka message value):**

```json
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.0,
  "currency": "USD",
  "metadata": {
    "source": "web"
  }
}
```

## Spring Boot Consumer (Tolerant to Old/New Messages)

```java
@Component
public class OrderConsumer {

    private final ObjectMapper mapper = new ObjectMapper();

    @KafkaListener(topics = "${topic.orders}", groupId = "order-service")
    public void consume(String message) throws JsonProcessingException {
        JsonNode node = mapper.readTree(message);

        String orderId = node.get("orderId").asText();
        String customerId = node.get("customerId").asText();
        double totalAmount = node.get("totalAmount").asDouble();

        // Handle optional fields safely
        String currency = node.has("currency") ? node.get("currency").asText() : "USD";
        String source = node.path("metadata").path("source").asText("unknown");

        // Process
        System.out.printf(
            "OrderId=%s, customerId=%s, amount=%.2f %s, source=%s%n",
            orderId, customerId, totalAmount, currency, source
        );
    }
}
```

**Input (old v1 message):**

```json
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.0
}
```

**Consumer behavior:**

- `currency` → `"USD"` (default)
- `source` → `"unknown"` (default)

**Input (new v2 message):**

```json
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.0,
  "currency": "EUR",
  "metadata": { "source": "mobile" }
}
```

**Consumer behavior:**

- `currency` → `"EUR"`
- `source` → `"mobile"`

This allows **rolling upgrades**: old and new messages are both handled correctly while you deploy new versions.

## Handling Breaking Changes with Versioned Topics

If you must break compatibility:

1. Introduce a new topic:

```yaml
topic:
  ordersV1: orders
  ordersV2: orders.v2
```

2. New producer writes to `orders.v2`, old consumers keep reading `orders`, new consumers read both or only `orders.v2` depending on migration strategy.

3. After all consumers are upgraded and data migrated, decommission `orders`.
