
Here’s a concise summary and some practical examples around **tuning Kafka producers/consumers in Spring Boot for low latency vs high throughput**.

---

## 1. Core Idea

You usually trade off between:

- **Low latency**: messages are sent/processed as fast as possible, even if that means more network calls and lower overall throughput.
- **High throughput**: messages are batched and compressed to maximize data per request, at the cost of higher end-to-end latency.

Tuning is mostly about adjusting **batching**, **buffering**, **acks**, and **concurrency**.

---

## 2. Producer Tuning

### Key Producer Settings

- `acks`
  - `0` or `1`: lower latency, less durability.
  - `all`: higher latency, stronger durability.
- `linger.ms`
  - Small (0–5 ms): low latency, less batching.
  - Larger (10–50+ ms): higher throughput, more batching.
- `batch.size`
  - Small: low latency, less batching.
  - Large: high throughput, more batching.
- `compression.type`
  - `none`: lower CPU, higher bandwidth usage.
  - `snappy`/`lz4`/`zstd`: higher throughput over network, more CPU, slightly higher latency.
- `max.in.flight.requests.per.connection`
  - Higher: more throughput, but can cause reordering on retries.
- `buffer.memory`
  - Larger: can hold more records for batching.

### Spring Boot Producer Config – Low Latency

```java
@Bean
public Map<String, Object> producerConfigs() {
    Map<String, Object> props = new HashMap<>();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
    props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
    props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

    // Low latency focus
    props.put(ProducerConfig.ACKS_CONFIG, "1");          // faster than "all"
    props.put(ProducerConfig.LINGER_MS_CONFIG, 0);       // send immediately
    props.put(ProducerConfig.BATCH_SIZE_CONFIG, 16_384); // small batch
    props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "none");
    props.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 1); // strict ordering

    return props;
}
```

**Example usage (input/output):**

```java
@Autowired
private KafkaTemplate<String, String> kafkaTemplate;

public void sendLowLatency(String userId, String event) {
    // Input: userId="u123", event="login"
    kafkaTemplate.send("user-events", userId, event);
    // Output: message is sent almost immediately, small batch, more requests per second
}
```

### Spring Boot Producer Config – High Throughput

```java
@Bean
public Map<String, Object> producerConfigsHighThroughput() {
    Map<String, Object> props = new HashMap<>();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
    props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
    props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

    // High throughput focus
    props.put(ProducerConfig.ACKS_CONFIG, "all");          // safer, but slower
    props.put(ProducerConfig.LINGER_MS_CONFIG, 20);        // wait to batch
    props.put(ProducerConfig.BATCH_SIZE_CONFIG, 64_000);   // larger batch
    props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");
    props.put(ProducerConfig.BUFFER_MEMORY_CONFIG, 64_000_000L);

    return props;
}
```

**Example behavior:**

- Input: 10,000 events in a burst.
- Output: fewer, larger requests to Kafka; higher MB/s throughput, but each individual event waits up to `linger.ms` before being sent.

## 3. Consumer Tuning

### Key Consumer Settings

- `max.poll.records`
  - Small: lower latency per record, less batch processing.
  - Large: higher throughput, more records per poll.
- `fetch.min.bytes` / `fetch.max.wait.ms`
  - Small/low: low latency, broker returns data quickly even if little data.
  - Large/high: high throughput, broker waits to accumulate more data.
- `enable.auto.commit` / `max.poll.interval.ms`
  - Affects how often offsets are committed and how long processing can take.

### Spring Boot Consumer Config – Low Latency

```java
@Bean
public Map<String, Object> consumerConfigsLowLatency() {
    Map<String, Object> props = new HashMap<>();
    props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
    props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
    props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
    props.put(ConsumerConfig.GROUP_ID_CONFIG, "low-latency-group");

    // Low latency focus
    props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, 10);   // small batch
    props.put(ConsumerConfig.FETCH_MIN_BYTES_CONFIG, 1);      // return as soon as possible
    props.put(ConsumerConfig.FETCH_MAX_WAIT_MS_CONFIG, 50);   // don't wait long
    props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, true);

    return props;
}
```

**Listener example (input/output):**

```java
@KafkaListener(topics = "user-events", groupId = "low-latency-group")
public void handleEvent(String message) {
    // Input: message="login"
    // Output: processed almost immediately after it arrives
    System.out.println("Received: " + message);
}
```

### Spring Boot Consumer Config – High Throughput

```java
@Bean
public Map<String, Object> consumerConfigsHighThroughput() {
    Map<String, Object> props = new HashMap<>();
    props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
    props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
    props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
    props.put(ConsumerConfig.GROUP_ID_CONFIG, "high-throughput-group");

    // High throughput focus
    props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, 1000);   // large batch
    props.put(ConsumerConfig.FETCH_MIN_BYTES_CONFIG, 50_000);  // wait for more data
    props.put(ConsumerConfig.FETCH_MAX_WAIT_MS_CONFIG, 500);   // allow batching
    props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, false);

    return props;
}
```

**Batch listener example:**

```java
@KafkaListener(
    topics = "user-events",
    groupId = "high-throughput-group",
    containerFactory = "batchFactory"
)
public void handleBatch(List<String> messages) {
    // Input: batch of up to 1000 messages
    // Output: process them in bulk (e.g., single DB batch insert)
    System.out.println("Processing batch size: " + messages.size());
}
```

## 4. Spring Kafka Container Tuning (Concurrency)

For consumers, you can also tune **concurrency**:

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
        ConsumerFactory<String, String> consumerFactory) {

    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);

    // Low latency: more threads to process messages in parallel
    factory.setConcurrency(4); // up to number of partitions

    return factory;
}
```

- Higher `concurrency` → more parallel processing → higher throughput and lower processing latency, but more load on downstream systems.

---

## 5. Summary of Tuning Directions

- **For low latency**:
  - Producer: `linger.ms` small, `batch.size` small, `compression.type=none`, `acks=1`.
  - Consumer: `max.poll.records` small, `fetch.min.bytes` small, `fetch.max.wait.ms` small.
  - Possibly higher concurrency on consumers.

- **For high throughput**:
  - Producer: larger `batch.size`, higher `linger.ms`, compression enabled, `acks=all`.
  - Consumer: larger `max.poll.records`, larger `fetch.min.bytes`, higher `fetch.max.wait.ms`, batch listeners.
  - Use concurrency and batch processing (DB bulk writes, etc.).
