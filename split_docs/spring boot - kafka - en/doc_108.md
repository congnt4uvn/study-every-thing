
Here’s a concise summary of the “multi-tenant topics vs per-tenant topics” design pattern in Kafka (with Spring Boot–style examples).

---

## Core Idea

When building a multi-tenant system on Kafka, you typically choose between:

1. **Multi-tenant topics**  
   - One (or a few) shared topics for all tenants.  
   - Tenant is identified in the message (e.g., header or payload field).

2. **Per-tenant topics**  
   - Each tenant gets its own topic (or topic set).  
   - Tenant isolation is done at the Kafka resource level.

---

## Multi-tenant Topics

### Characteristics

- **Single topic** (e.g., `events`) for all tenants.
- Messages carry a **tenant identifier**:
  - In the key, headers, or payload.
- Consumers filter/route based on tenant ID.

### Pros

- Fewer topics → easier cluster management.
- Better partition utilization.
- Easier to add new tenants (no topic creation).

### Cons

- Weaker isolation (no per-tenant quotas at topic level).
- Harder to do per-tenant retention or ACLs.
- Consumers must filter messages themselves.

### Example: Producer (Spring Boot / Spring Kafka)

```java
@Service
public class TenantEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public TenantEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendEvent(String tenantId, String payload) {
        Message<String> message = MessageBuilder
                .withPayload(payload)
                .setHeader(KafkaHeaders.TOPIC, "events")          // shared topic
                .setHeader("tenantId", tenantId)                  // tenant marker
                .build();

        kafkaTemplate.send(message);
    }
}
```

**Input example:**

```json
{
  "tenantId": "tenantA",
  "payload": "{\"orderId\":123,\"status\":\"CREATED\"}"
}
```

**Kafka record (conceptual):**

- Topic: `events`
- Headers: `tenantId=tenantA`
- Value: `{"orderId":123,"status":"CREATED"}`

### Example: Consumer Filtering by Tenant

```java
@KafkaListener(topics = "events", groupId = "tenantA-service")
public void listen(ConsumerRecord<String, String> record) {
    String tenantId = record.headers().lastHeader("tenantId") != null
            ? new String(record.headers().lastHeader("tenantId").value())
            : null;

    if (!"tenantA".equals(tenantId)) {
        // ignore or route elsewhere
        return;
    }

    // process only tenantA messages
    System.out.println("Processing for tenantA: " + record.value());
}
```

**Output example (for tenantA consumer):**

```text
Processing for tenantA: {"orderId":123,"status":"CREATED"}
```

---

## Per-tenant Topics

### Characteristics

- Each tenant has its own topic, e.g.:
  - `tenantA.events`, `tenantB.events`, ...
- Tenant isolation is done by topic naming and ACLs.

### Pros

- Stronger isolation (ACLs, quotas, retention per tenant).
- Easier per-tenant monitoring and debugging.
- Can scale tenants independently (partitions per tenant).

### Cons

- Many topics → higher metadata overhead.
- Operational complexity (topic creation, lifecycle).
- Harder to manage thousands of tenants.

### Example: Producer with Per-tenant Topic

```java
@Service
public class TenantEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public TenantEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendEvent(String tenantId, String payload) {
        String topic = tenantId + ".events";  // per-tenant topic
        kafkaTemplate.send(topic, payload);
    }
}
```

**Input example:**

```json
{
  "tenantId": "tenantB",
  "payload": "{\"orderId\":456,\"status\":\"PAID\"}"
}
```

**Kafka record (conceptual):**

- Topic: `tenantB.events`
- Value: `{"orderId":456,"status":"PAID"}`

### Example: Consumer per Tenant

```java
@KafkaListener(topics = "tenantB.events", groupId = "tenantB-service")
public void listen(String message) {
    // Only tenantB messages arrive here
    System.out.println("Processing tenantB message: " + message);
}
```

**Output example:**

```text
Processing tenantB message: {"orderId":456,"status":"PAID"}
```

---

## When to Choose Which

**Use multi-tenant topics when:**

- You have many tenants (hundreds/thousands).
- You want simpler topic management.
- You don’t need strict per-tenant isolation at Kafka level.

**Use per-tenant topics when:**

- You have a manageable number of tenants.
- You need strong isolation (security, quotas, retention).
- You want clear per-tenant monitoring and SLAs.

---

## Hybrid Approach (Common in Practice)

- Use **multi-tenant topics** for small/low-traffic tenants.
- Use **per-tenant topics** for large or premium tenants.
- Implement a naming convention and a topic management service.
