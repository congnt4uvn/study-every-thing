
Here’s a concise summary and some practical examples around Kafka architecture, focusing on ZooKeeper vs KRaft, and how this relates to Spring Boot.

---

## 1. Kafka Architecture: Key Components

Core components (common to both ZooKeeper and KRaft):

- **Broker**: Kafka server that stores and serves messages.
- **Topic**: Named stream of records.
- **Partition**: Ordered, immutable sequence of records within a topic.
- **Producer**: Sends messages to topics.
- **Consumer**: Reads messages from topics.
- **Controller**: Special broker responsible for cluster metadata (leader election, partition assignments, etc.).

---

## 2. ZooKeeper-based Kafka (Legacy)

- **ZooKeeper** is an external service used by Kafka to:
  - Store cluster metadata (brokers, topics, ACLs).
  - Perform leader election for partitions.
  - Track broker liveness.
- Kafka brokers connect to ZooKeeper to:
  - Register themselves.
  - Get metadata updates.
- This was the default architecture up to Kafka 2.x and early 3.x.

**High-level flow:**
1. Broker starts → registers in ZooKeeper.
2. One broker becomes the **controller** via ZooKeeper.
3. Controller uses ZooKeeper to manage partition leaders and replicas.
4. Producers/consumers talk only to brokers (not ZooKeeper).

---

## 3. KRaft (Kafka Raft) – New Metadata Quorum

- **KRaft** removes the dependency on ZooKeeper.
- Kafka uses its own **Raft-based consensus** implementation to manage metadata.
- A set of brokers form a **KRaft quorum**:
  - One is elected as **controller**.
  - Metadata is stored in an internal topic and replicated via Raft.
- Benefits:
  - Simpler deployment (no ZooKeeper cluster).
  - Better scalability and performance for metadata operations.
  - Unified architecture (everything inside Kafka).

**High-level flow:**
1. KRaft controllers form a Raft quorum.
2. One controller is elected via Raft.
3. Metadata changes (create topic, broker join/leave) are written to the Raft log.
4. Brokers read metadata from the controller quorum.

## 4. Configuration Examples

### 4.1 ZooKeeper-based Kafka (server.properties)

```properties
# Broker ID
broker.id=1

# Listeners
listeners=PLAINTEXT://:9092

# ZooKeeper connection
zookeeper.connect=localhost:2181

# Log dirs
log.dirs=/tmp/kafka-logs
```

### 4.2 KRaft-based Kafka (server.properties)

```properties
# Unique node ID
node.id=1

# Process roles: broker and controller
process.roles=broker,controller

# KRaft controller quorum
controller.quorum.voters=1@localhost:9093

# Listeners
listeners=PLAINTEXT://:9092,CONTROLLER://:9093
listener.security.protocol.map=PLAINTEXT:PLAINTEXT,CONTROLLER:PLAINTEXT
inter.broker.listener.name=PLAINTEXT
controller.listener.names=CONTROLLER

# Log dirs (includes metadata log)
log.dirs=/tmp/kraft-combined-logs
```

## 5. Spring Boot + Kafka: Basic Examples

Spring Boot doesn’t care whether Kafka uses ZooKeeper or KRaft; it just connects to brokers via `bootstrap.servers`.

### 5.1 Maven Dependency

```xml
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>
```

### 5.2 Application Properties

```properties
spring.kafka.bootstrap-servers=localhost:9092

# Producer
spring.kafka.producer.key-serializer=org.apache.kafka.common.serialization.StringSerializer
spring.kafka.producer.value-serializer=org.apache.kafka.common.serialization.StringSerializer

# Consumer
spring.kafka.consumer.group-id=my-group
spring.kafka.consumer.key-deserializer=org.apache.kafka.common.serialization.StringDeserializer
spring.kafka.consumer.value-deserializer=org.apache.kafka.common.serialization.StringDeserializer
```

### 5.3 Producer Example (Spring Boot)

```java
@Service
public class MyKafkaProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public MyKafkaProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value)
                     .whenComplete((result, ex) -> {
                         if (ex == null) {
                             System.out.println("Sent: " + value +
                                 " to partition " + result.getRecordMetadata().partition() +
                                 " offset " + result.getRecordMetadata().offset());
                         } else {
                             ex.printStackTrace();
                         }
                     });
    }
}
```

**Example input:**

```java
myKafkaProducer.sendMessage("orders", "order-1", "{ \"id\": 1, \"amount\": 100 }");
```

**Example output (console):**

```text
Sent: { "id": 1, "amount": 100 } to partition 0 offset 5
```

### 5.4 Consumer Example (Spring Boot)

```java
@Service
public class MyKafkaConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(String message) {
        System.out.println("Received message: " + message);
        // process message...
    }
}
```

**Example input (message in Kafka topic `orders`):**

```json
{ "id": 1, "amount": 100 }
```

**Example output (consumer log):**

```text
Received message: { "id": 1, "amount": 100 }
```

---

## 6. Summary of ZooKeeper vs KRaft

- **ZooKeeper-based**:
  - External ZooKeeper cluster required.
  - Metadata and leader election handled by ZooKeeper.
  - Older, widely used, but being phased out.

- **KRaft-based**:
  - No ZooKeeper; Kafka manages metadata internally via Raft.
  - Simpler architecture, better scalability.
  - New standard for Kafka 3.x+ and future versions.

From a Spring Boot application’s perspective, the main difference is in how Kafka is deployed and configured, not in how the app produces/consumes messages.
