
Here’s a concise summary of the topic “Failure scenarios: What happens during leader election and rebalancing” in a Kafka + Spring Boot context, with code examples.

---

## 1. Core Concepts

### Leader Election (Kafka Brokers / Partitions)
- Each partition has **one leader** and zero or more **followers**.
- Producers and consumers read/write to the **leader**.
- If the leader broker fails:
  - Kafka’s **controller** detects the failure (via ZooKeeper or KRaft).
  - A new leader is elected from the **in-sync replicas (ISR)**.
  - Clients (producers/consumers) get errors briefly, then automatically reconnect to the new leader.

### Rebalancing (Consumer Groups)
- A **consumer group** shares partitions of a topic.
- When:
  - a consumer joins,
  - a consumer leaves/crashes,
  - or topic partitions change,
  Kafka triggers a **rebalance**.
- During rebalance:
  - Partition ownership is redistributed among consumers.
  - Consumers temporarily stop processing and re-assign partitions.
  - After rebalance, each partition is assigned to exactly one consumer in the group.

---

## 2. Typical Failure Scenarios

### Scenario A: Broker (Leader) Failure
- **What happens:**
  - The controller marks the broker as dead.
  - For each affected partition, a new leader is chosen from ISR.
  - Producers/consumers may see `LEADER_NOT_AVAILABLE` or similar errors briefly.
- **Impact on Spring Boot apps:**
  - Spring Kafka clients will retry automatically (depending on config).
  - Short pause in message production/consumption.

### Scenario B: Consumer Failure (Rebalance)
- **What happens:**
  - Kafka detects the consumer is gone (session timeout).
  - Remaining consumers in the group rebalance and take over its partitions.
- **Impact on Spring Boot apps:**
  - Other instances start consuming the partitions of the failed instance.
  - In-flight messages on the failed instance may be reprocessed if offsets weren’t committed.

### Scenario C: New Consumer Instance Starts
- **What happens:**
  - New consumer joins the group.
  - Rebalance occurs; partitions are redistributed.
- **Impact:**
  - Each consumer gets fewer partitions (more parallelism overall).
  - Short pause during rebalance.

## 3. Spring Boot / Spring Kafka Configuration Examples

### 3.1 Basic Consumer Configuration (Java)

```java
@Configuration
@EnableKafka
public class KafkaConsumerConfig {

    @Bean
    public Map<String, Object> consumerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "my-consumer-group");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
                  StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
                  StringDeserializer.class);

        // Rebalance-related settings
        props.put(ConsumerConfig.SESSION_TIMEOUT_MS_CONFIG, 10000);
        props.put(ConsumerConfig.HEARTBEAT_INTERVAL_MS_CONFIG, 3000);
        props.put(ConsumerConfig.MAX_POLL_INTERVAL_MS_CONFIG, 300000);

        // Enable manual commit if you want more control
        props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, false);

        return props;
    }

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        return new DefaultKafkaConsumerFactory<>(consumerConfigs());
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
        return factory;
    }
}
```

### 3.2 Listener with Rebalance Callbacks

```java
@Component
public class MyKafkaListener {

    @KafkaListener(topics = "orders", groupId = "order-group")
    public void listen(String message,
                       Acknowledgment ack,
                       Consumer<?, ?> consumer) {
        // Process message
        System.out.println("Received: " + message);
        // Commit offset after successful processing
        ack.acknowledge();
    }

    @KafkaListener(
        topics = "orders",
        groupId = "order-group",
        containerFactory = "kafkaListenerContainerFactory"
    )
    public void listenWithRebalance(String message) {
        // normal processing
    }

    @EventListener
    public void handleRebalance(ConsumerPartitionAssignedEvent event) {
        System.out.println("Partitions assigned: " + event.getPartitions());
    }

    @EventListener
    public void handleRebalanceRevoked(ConsumerPartitionRevokedEvent event) {
        System.out.println("Partitions revoked: " + event.getPartitions());
    }
}
```

## 4. Example: Input/Output Behavior During Failure

### Setup
- Topic: `orders` with 3 partitions: `orders-0`, `orders-1`, `orders-2`
- Consumer group: `order-group`
- Two Spring Boot instances: `consumer-A`, `consumer-B`

### Normal State (before failure)
- Partition assignment:
  - `consumer-A`: `orders-0`, `orders-1`
  - `consumer-B`: `orders-2`

**Input (messages produced):**
```text
orders-0: "order-1"
orders-1: "order-2"
orders-2: "order-3"
```

**Output (logs):**
```text
[consumer-A] Received from orders-0: order-1
[consumer-A] Received from orders-1: order-2
[consumer-B] Received from orders-2: order-3
```

### Failure: `consumer-A` crashes

**What happens:**
1. Kafka detects `consumer-A` is dead (session timeout).
2. Rebalance:
   - `consumer-B` now gets all partitions: `orders-0`, `orders-1`, `orders-2`.

**New Input:**
```text
orders-0: "order-4"
orders-1: "order-5"
orders-2: "order-6"
```

**New Output (after rebalance):**
```text
[consumer-B] Partitions assigned: [orders-0, orders-1, orders-2]
[consumer-B] Received from orders-0: order-4
[consumer-B] Received from orders-1: order-5
[consumer-B] Received from orders-2: order-6
```

If `consumer-A` had uncommitted messages when it crashed, those messages may be reprocessed by `consumer-B` after rebalance.

## 5. Handling Leader Failure in Spring Boot

You don’t usually handle leader election directly in Spring Boot; Kafka does it. But you can configure retries and error handling.

### Producer with Retries

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public Map<String, Object> producerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
                  StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
                  StringSerializer.class);

        // Retry on transient errors (e.g., leader not available)
        props.put(ProducerConfig.RETRIES_CONFIG, 5);
        props.put(ProducerConfig.ACKS_CONFIG, "all");
        props.put(ProducerConfig.RETRY_BACKOFF_MS_CONFIG, 1000);

        return props;
    }

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        return new DefaultKafkaProducerFactory<>(producerConfigs());
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

**Behavior:**
- If a leader fails while sending:
  - Producer gets an error, retries automatically.
  - After new leader election, send succeeds (if within retry limits).
