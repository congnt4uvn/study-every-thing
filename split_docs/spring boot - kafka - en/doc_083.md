
Here’s a concise summary of Spring Kafka error handling with `RetryTemplate` and `RecoveringBatchErrorHandler`, plus code examples.

---

## 1. Context: Error Handling in Spring Kafka

When consuming messages with Spring Kafka, you often need:

- Retries: try processing again if something fails (e.g., transient errors).
- Recovery: if retries are exhausted, decide what to do with the failed record(s) (log, send to dead-letter topic, etc.).
- Batch handling: when using batch listeners (`@KafkaListener` with `List<ConsumerRecord<...>>`), you need batch-aware error handlers.

Two key tools:

- `RetryTemplate` – for retrying operations in the listener container.
- `RecoveringBatchErrorHandler` – for handling failures in batch listeners with recovery logic.

---

## 2. Using `RetryTemplate` with Spring Kafka

`RetryTemplate` lets you configure retry behavior (max attempts, backoff, which exceptions to retry, etc.) for your listener container.

### Basic configuration

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.config.ConcurrentKafkaListenerContainerFactory;
import org.springframework.kafka.core.ConsumerFactory;
import org.springframework.retry.backoff.FixedBackOffPolicy;
import org.springframework.retry.policy.SimpleRetryPolicy;
import org.springframework.retry.support.RetryTemplate;

@Configuration
public class KafkaConfig {

    @Bean
    public RetryTemplate kafkaRetryTemplate() {
        RetryTemplate retryTemplate = new RetryTemplate();

        // Retry policy: 3 attempts total (1 initial + 2 retries)
        SimpleRetryPolicy retryPolicy = new SimpleRetryPolicy();
        retryPolicy.setMaxAttempts(3);

        // Backoff policy: wait 1 second between retries
        FixedBackOffPolicy backOffPolicy = new FixedBackOffPolicy();
        backOffPolicy.setBackOffPeriod(1000L);

        retryTemplate.setRetryPolicy(retryPolicy);
        retryTemplate.setBackOffPolicy(backOffPolicy);

        return retryTemplate;
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory,
            RetryTemplate kafkaRetryTemplate) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);

        // Enable retry for the listener
        factory.setRetryTemplate(kafkaRetryTemplate);

        // Optional: what to do when retries are exhausted
        factory.setRecoveryCallback(context -> {
            Object record = context.getAttribute("record");
            // custom recovery logic (e.g., log, send to DLT)
            System.err.println("Recovery after retries exhausted for record: " + record);
            return null;
        });

        return factory;
    }
}
```

### Example listener

```java
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class MyKafkaConsumer {

    @KafkaListener(topics = "input-topic", groupId = "my-group")
    public void listen(ConsumerRecord<String, String> record) {
        String value = record.value();
        System.out.println("Processing: " + value);

        // Simulate transient error
        if (value.contains("fail")) {
            throw new RuntimeException("Simulated processing error");
        }

        // Normal processing...
    }
}
```

### Example behavior (input/output)

- Input messages on topic `input-topic`:
  - `"ok-1"`
  - `"fail-1"`
  - `"ok-2"`

- Processing:
  - `"ok-1"` → processed once, no retry.
  - `"fail-1"` → listener throws exception → retried up to 3 times with 1s delay → if still failing, `recoveryCallback` is invoked.
  - `"ok-2"` → processed normally.

Console output (simplified):

```text
Processing: ok-1
Processing: fail-1
Processing: fail-1
Processing: fail-1
Recovery after retries exhausted for record: ConsumerRecord(... value=fail-1 ...)
Processing: ok-2
```

## 3. `RecoveringBatchErrorHandler` for Batch Listeners

When using batch listeners (`List<ConsumerRecord<...>>`), you need a batch error handler. `RecoveringBatchErrorHandler`:

- Handles exceptions thrown while processing a batch.
- Can recover individual records or the whole batch.
- Often used with a `BiConsumer` or `Consumer` to define recovery logic (e.g., send failed records to a dead-letter topic).

### Batch listener configuration

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.config.ConcurrentKafkaListenerContainerFactory;
import org.springframework.kafka.core.ConsumerFactory;
import org.springframework.kafka.listener.BatchErrorHandler;
import org.springframework.kafka.listener.RecoveringBatchErrorHandler;

@Configuration
public class BatchKafkaConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> batchKafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);
        factory.setBatchListener(true); // enable batch mode

        // Simple recovering batch error handler
        BatchErrorHandler errorHandler = new RecoveringBatchErrorHandler((record, ex) -> {
            // record: the failed ConsumerRecord
            // ex: the exception thrown during processing
            System.err.println("Recovering failed record: " + record.value() +
                    " due to: " + ex.getMessage());
            // Here you could send to a dead-letter topic instead
        });

        factory.setBatchErrorHandler(errorHandler);

        return factory;
    }
}
```

### Batch listener example

```java
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

import java.util.List;

@Service
public class BatchConsumer {

    @KafkaListener(
            topics = "batch-topic",
            groupId = "batch-group",
            containerFactory = "batchKafkaListenerContainerFactory"
    )
    public void listenBatch(List<ConsumerRecord<String, String>> records) {
        for (ConsumerRecord<String, String> record : records) {
            String value = record.value();
            System.out.println("Processing batch record: " + value);

            if (value.contains("bad")) {
                throw new IllegalStateException("Bad record: " + value);
            }
        }
    }
}
```

### Example behavior (input/output)

- Batch of messages on `batch-topic`:
  - `"good-1"`, `"bad-1"`, `"good-2"`

Processing:

- Listener processes `"good-1"` → OK.
- When it hits `"bad-1"`, it throws `IllegalStateException`.
- `RecoveringBatchErrorHandler` is invoked for the failed record(s).

Console output (simplified):

```text
Processing batch record: good-1
Processing batch record: bad-1
Recovering failed record: bad-1 due to: Bad record: bad-1
```

Depending on configuration, the handler may:

- Skip the failed record and continue with others.
- Or stop the batch and recover only the failed record(s).
- Or send failed records to a dead-letter topic.

---

## 4. Combining Retries and Recovery in Batch

You can also combine retry behavior with batch error handling, but typically:

- `RetryTemplate` is used at the container level (per batch or per record).
- `RecoveringBatchErrorHandler` is used to decide what to do when processing fails (after retries, if configured).

The exact behavior depends on your container factory configuration and the Spring Kafka version.
