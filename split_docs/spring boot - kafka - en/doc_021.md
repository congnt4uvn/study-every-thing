
Here’s a concise summary of the topic **“How partitioning affects: Hot partitions, skew, and mitigation strategies”** in Kafka with Spring Boot, plus code examples.

---

## Key Ideas

1. **Partitions in Kafka**
   - A topic is split into multiple partitions.
   - Each partition is an ordered, append-only log.
   - Parallelism and throughput are largely determined by the number of partitions.

2. **Hot Partitions**
   - A **hot partition** is a partition that receives disproportionately more messages than others.
   - Causes:
     - Many messages share the same key.
     - Custom partitioner routes too many records to one partition.
   - Effects:
     - One broker/partition becomes a bottleneck.
     - Higher latency and potential throttling.
     - Uneven consumer load (one consumer in a group does most of the work).

3. **Skew (Partition Skew)**
   - **Skew** = uneven distribution of messages across partitions.
   - Even if not extreme, skew reduces:
     - Effective parallelism.
     - Resource utilization (some consumers idle, others overloaded).

4. **Mitigation Strategies**
   - **Key design**
     - Choose keys that distribute well (high cardinality, not dominated by a few values).
     - Avoid using a single “hot” key (e.g., “ALL”, “GLOBAL”).
   - **Custom partitioner**
     - Implement a partitioner that spreads load more evenly.
   - **Increase partitions**
     - More partitions → more parallelism, but also more overhead.
     - Be careful: changing partition count can change key→partition mapping.
   - **Rebalancing / rekeying**
     - Re-publish data with better keys or a different partitioning strategy.
   - **Consumer-side scaling**
     - Increase consumer instances to match partition count.
   - **Batching and backpressure**
     - Use producer configs (batch.size, linger.ms) and consumer configs to handle spikes.

## Spring Boot + Kafka: Code Examples

### 1. Basic Producer with Keyed Messages

```java
// build.gradle or pom.xml: include spring-kafka

@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String customerId, String orderJson) {
        // customerId is the key → affects partition selection
        kafkaTemplate.send("orders", customerId, orderJson)
                     .whenComplete((result, ex) -> {
                         if (ex == null) {
                             System.out.println("Sent to partition: " +
                                 result.getRecordMetadata().partition());
                         } else {
                             ex.printStackTrace();
                         }
                     });
    }
}
```

**Example input:**

```java
orderProducer.sendOrder("customer-123", "{\"orderId\":1,\"amount\":100}");
orderProducer.sendOrder("customer-123", "{\"orderId\":2,\"amount\":50}");
orderProducer.sendOrder("customer-999", "{\"orderId\":3,\"amount\":200}");
```

**Example output (conceptual):**

- Messages with key `"customer-123"` → always same partition (e.g., partition 1).
- Message with key `"customer-999"` → maybe partition 3.
- If most traffic is `"customer-123"`, partition 1 becomes hot.

### 2. Custom Partitioner to Reduce Hot Partitions

You can implement a custom partitioner to spread load more evenly, e.g., by:

- Hashing only part of the key.
- Adding some randomization for extremely hot keys.

**Custom partitioner:**

```java
public class CustomOrderPartitioner implements org.apache.kafka.clients.producer.Partitioner {

    @Override
    public void configure(Map<String, ?> configs) {}

    @Override
    public int partition(String topic,
                         Object key,
                         byte[] keyBytes,
                         Object value,
                         byte[] valueBytes,
                         Cluster cluster) {

        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();

        if (key == null) {
            // No key → round-robin
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }

        String keyStr = key.toString();

        // Example: if key is extremely hot, add some randomization
        if ("GLOBAL".equals(keyStr)) {
            // Spread "GLOBAL" key across all partitions
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }

        // Default: hash-based partitioning
        return (keyStr.hashCode() & 0x7fffffff) % numPartitions;
    }

    @Override
    public void close() {}
}
```

**Producer configuration in Spring Boot:**

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public Map<String, Object> producerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
                  org.apache.kafka.common.serialization.StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
                  org.apache.kafka.common.serialization.StringSerializer.class);
        props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG,
                  CustomOrderPartitioner.class.getName());
        return props;
    }

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        return new DefaultKafkaProducerFactory<>(producerConfigs());
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

**Example input:**

```java
kafkaTemplate.send("orders", "GLOBAL", "{\"orderId\":10}");
kafkaTemplate.send("orders", "GLOBAL", "{\"orderId\":11}");
kafkaTemplate.send("orders", "GLOBAL", "{\"orderId\":12}");
```

**Example output (conceptual):**

- All messages have key `"GLOBAL"`, but the custom partitioner spreads them across all partitions instead of a single one, reducing the hot partition problem.

### 3. Consumer Group and Skew

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void consume(ConsumerRecord<String, String> record) {
        System.out.printf("Consumer %s got key=%s, partition=%d, value=%s%n",
                Thread.currentThread().getName(),
                record.key(),
                record.partition(),
                record.value());
    }
}
```

**Behavior:**

- If one partition has 80% of messages, the consumer assigned to that partition will be overloaded.
- Adding more consumers only helps if there are enough partitions and the data is evenly distributed.

## Practical Checklist

- Design keys to avoid a few dominant values.
- Monitor per-partition metrics (bytes in, records in, latency).
- Use a custom partitioner for known hot keys.
- Scale partitions and consumers together.
- Consider rekeying or repartitioning if skew is severe and persistent.
