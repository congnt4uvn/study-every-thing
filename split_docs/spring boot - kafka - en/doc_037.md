**“Producer performance: Impact of partitions on producer throughput”**

---

## Key Ideas

1. **More partitions ⇒ higher potential throughput**  
   - Kafka writes to partitions in parallel.  
   - A topic with more partitions can be handled by more brokers/threads, increasing total write throughput.

2. **Producer throughput is limited by:**
   - Number of partitions (parallelism).
   - Number of brokers and their hardware (CPU, disk, network).
   - Producer configuration (batching, compression, acks, linger.ms, etc.).

3. **Diminishing returns with too many partitions**
   - Each partition has overhead (file handles, memory, metadata).  
   - Very high partition counts can:
     - Increase controller and broker load.
     - Slow down leader elections and metadata updates.
     - Increase end-to-end latency.

4. **Partitioning and key distribution**
   - If you use a key, records with the same key go to the same partition.  
   - Skewed keys (hot keys) can overload a single partition and limit throughput, even if the topic has many partitions.

5. **Producer configuration matters**
   - `batch.size` and `linger.ms` help aggregate records into fewer, larger requests → higher throughput.
   - `acks=all` is safer but slower; `acks=1` or `acks=0` can increase throughput at the cost of durability.
   - Compression (`snappy`, `lz4`, `zstd`) can increase throughput by reducing network I/O.

6. **Spring Boot / Spring Kafka specifics**
   - You configure partitions at topic creation (e.g., via `NewTopic` bean).
   - Producer properties (batching, acks, compression) are set in `application.yml` or via `ProducerFactory`.

## Simple Kafka Producer Example (Plain Java)

**Producer config:**

```java
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());

// Throughput-related configs
props.put(ProducerConfig.ACKS_CONFIG, "1");          // or "all"
props.put(ProducerConfig.BATCH_SIZE_CONFIG, 32768);  // 32 KB
props.put(ProducerConfig.LINGER_MS_CONFIG, 10);      // wait up to 10 ms to batch
props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
```

**Sending messages:**

```java
String topic = "orders"; // assume this topic has N partitions

for (int i = 0; i < 1000; i++) {
    String key = "user-" + (i % 10); // 10 users
    String value = "order-" + i;

    ProducerRecord<String, String> record = new ProducerRecord<>(topic, key, value);

    producer.send(record, (metadata, exception) -> {
        if (exception == null) {
            System.out.printf("Sent to partition=%d, offset=%d%n",
                    metadata.partition(), metadata.offset());
        } else {
            exception.printStackTrace();
        }
    });
}

producer.flush();
producer.close();
```

**Example output (simplified):**

```
Sent to partition=0, offset=0
Sent to partition=1, offset=0
Sent to partition=2, offset=0
...
```

If the topic has more partitions, you’ll see records distributed across more partitions, allowing more parallel writes and higher potential throughput.

## Spring Boot + Spring Kafka Example

### 1. Topic with configurable partitions

```java
@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic ordersTopic() {
        // 12 partitions, replication factor 3
        return TopicBuilder.name("orders")
                .partitions(12)
                .replicas(3)
                .build();
    }
}
```

### 2. Producer configuration in `application.yml`

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      acks: 1
      properties:
        linger.ms: 10
        batch.size: 32768
        compression.type: lz4
```

### 3. Producer service

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String userId, String orderJson) {
        kafkaTemplate.send("orders", userId, orderJson)
                .whenComplete((result, ex) -> {
                    if (ex == null) {
                        System.out.printf("Sent order for user=%s to partition=%d, offset=%d%n",
                                userId,
                                result.getRecordMetadata().partition(),
                                result.getRecordMetadata().offset());
                    } else {
                        ex.printStackTrace();
                    }
                });
    }
}
```

**Example input:**

```java
orderProducer.sendOrder("user-1", "{\"orderId\": 123, \"amount\": 50.0}");
orderProducer.sendOrder("user-2", "{\"orderId\": 124, \"amount\": 75.0}");
```

**Example output (logs):**

```
Sent order for user=user-1 to partition=3, offset=10
Sent order for user=user-2 to partition=7, offset=11
```

## Practical Guidelines

- Start with a moderate number of partitions (e.g., 6–12 per topic) and measure throughput.
- Increase partitions if:
  - Producer CPU/network is underutilized.
  - Brokers are not saturated.
- Watch for:
  - Controller/broker CPU and memory.
  - Long metadata refresh times.
  - Skewed partitions (one partition much hotter than others).
