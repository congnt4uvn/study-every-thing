
Here’s a concise overview of Kafka Streams processing patterns: aggregations, joins, and windowing, with Spring Boot–style examples.

---

## 1. Aggregations

**Idea:** Group records by key and compute a running result (count, sum, etc.).

Common operations:
- `count()` – number of records per key  
- `reduce()` – combine values into one  
- `aggregate()` – custom aggregation logic

### Example: Count events per user

**Input topic:** `user-events`  
Key: `userId` (String)  
Value: JSON event

```java
@Bean
public KStream<String, String> kStream(StreamsBuilder builder) {
    KStream<String, String> stream = builder.stream("user-events");

    KTable<String, Long> userCounts = stream
            .groupByKey()              // group by userId
            .count(Materialized.as("user-counts-store"));

    // Optionally write counts to a topic
    userCounts.toStream().to("user-event-counts", Produced.with(Serdes.String(), Serdes.Long()));

    return stream;
}
```

**Example input:**
- Key: `"user1"`, Value: `{"action":"login"}`
- Key: `"user1"`, Value: `{"action":"click"}`
- Key: `"user2"`, Value: `{"action":"login"}`

**Output (to `user-event-counts`):**
- `"user1" -> 1`
- `"user1" -> 2`
- `"user2" -> 1`

## 2. Joins

**Idea:** Combine data from two streams or a stream and a table based on key.

Types:
- **KStream–KStream join**: join two event streams within a time window  
- **KStream–KTable join**: enrich events with latest state  
- **KTable–KTable join**: join two state tables

### 2.1 KStream–KTable Join (Enrichment)

**Use case:** Enrich user events with user profile data.

- Topic `user-events` → `KStream<String, UserEvent>`
- Topic `user-profiles` → `KTable<String, UserProfile>`

```java
@Bean
public KStream<String, EnrichedUserEvent> userEventStream(StreamsBuilder builder) {
    KStream<String, UserEvent> events = builder.stream("user-events",
            Consumed.with(Serdes.String(), userEventSerde()));

    KTable<String, UserProfile> profiles = builder.table("user-profiles",
            Consumed.with(Serdes.String(), userProfileSerde()));

    KStream<String, EnrichedUserEvent> enriched = events.join(
            profiles,
            (event, profile) -> new EnrichedUserEvent(event, profile)
    );

    enriched.to("enriched-user-events",
            Produced.with(Serdes.String(), enrichedUserEventSerde()));

    return enriched;
}
```

**Example input:**

`user-profiles`:
- `"user1" -> { "name": "Alice", "tier": "GOLD" }`

`user-events`:
- `"user1" -> { "action": "purchase", "amount": 100 }`

**Output (`enriched-user-events`):**
- `"user1" -> { "action": "purchase", "amount": 100, "name": "Alice", "tier": "GOLD" }`

### 2.2 KStream–KStream Join (Time-based)

**Use case:** Join orders with payments that happen within a time window.

```java
@Bean
public KStream<String, EnrichedOrder> orderStream(StreamsBuilder builder) {
    KStream<String, Order> orders = builder.stream("orders",
            Consumed.with(Serdes.String(), orderSerde()));

    KStream<String, Payment> payments = builder.stream("payments",
            Consumed.with(Serdes.String(), paymentSerde()));

    KStream<String, EnrichedOrder> joined = orders.join(
            payments,
            (order, payment) -> new EnrichedOrder(order, payment),
            JoinWindows.ofTimeDifferenceWithNoGrace(Duration.ofMinutes(5)),
            StreamJoined.with(Serdes.String(), orderSerde(), paymentSerde())
    );

    joined.to("enriched-orders",
            Produced.with(Serdes.String(), enrichedOrderSerde()));

    return joined;
}
```

**Example input (same key = orderId):**

`orders`:
- `"order-1" -> { "amount": 50 }` at 10:00:00

`payments`:
- `"order-1" -> { "status": "PAID" }` at 10:03:00

**Output (`enriched-orders`):**
- `"order-1" -> { "amount": 50, "status": "PAID" }`  
(Joined because payment is within 5 minutes of order)

## 3. Windowing

**Idea:** Group records by time windows (e.g., per minute, per hour) for time-based aggregations.

Common window types:
- **Tumbling window**: fixed, non-overlapping (e.g., every 1 minute)  
- **Hopping window**: fixed size, overlapping (e.g., size 5 min, advance 1 min)  
- **Sliding window**: defined by distance between events  
- **Session window**: based on periods of activity separated by inactivity

### Example: Count events per user per 1-minute window

```java
@Bean
public KStream<String, String> windowedCounts(StreamsBuilder builder) {
    KStream<String, String> events = builder.stream("user-events");

    TimeWindows windows = TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1));

    KTable<Windowed<String>, Long> counts = events
            .groupByKey()
            .windowedBy(windows)
            .count(Materialized.as("user-counts-per-minute"));

    counts.toStream()
          .map((windowedKey, count) -> {
              String userId = windowedKey.key();
              String windowStart = windowedKey.window().startTime().toString();
              String windowEnd = windowedKey.window().endTime().toString();
              String value = String.format(
                      "{\"userId\":\"%s\",\"windowStart\":\"%s\",\"windowEnd\":\"%s\",\"count\":%d}",
                      userId, windowStart, windowEnd, count
              );
              return KeyValue.pair(userId, value);
          })
          .to("user-counts-per-minute-topic", Produced.with(Serdes.String(), Serdes.String()));

    return events;
}
```

**Example input (user1 events):**
- 10:00:10
- 10:00:30
- 10:01:05

**Output (per 1-minute window):**
- Window [10:00:00–10:01:00): `user1 -> count = 2`
- Window [10:01:00–10:02:00): `user1 -> count = 1`

---

## Summary

- **Aggregations**: group by key and compute counts/sums/custom state.
- **Joins**: combine streams/tables by key; can be time-based (stream–stream) or state-based (stream–table).
- **Windowing**: apply time windows to aggregations or joins to get time-bounded results.

All of these are typically wired in Spring Boot via `StreamsBuilder` beans and configured with SerDes and topics.
