
Here’s a concise summary of the key ideas around **Kafka Streams state**, focusing on:

- State stores  
- Changelog topics  
- Fault tolerance  

with small code examples.

---

## 1. State in Kafka Streams

Kafka Streams applications can be **stateful**: they keep data in memory/disk to support operations like:

- `aggregate`, `count`, `reduce`
- `join` (KStream-KTable, KTable-KTable, etc.)
- windowed operations

This state is stored in **state stores**.

---

## 2. State Stores

A **state store** is a local store (in-memory or on-disk) managed by Kafka Streams.  
Types:

- **Key-value store** (most common)
- **Window store**
- **Session store**

They are:

- Local to each Kafka Streams instance
- Backed up to Kafka via **changelog topics** for durability

### Example: Using a state store in a processor

```java
import org.apache.kafka.streams.processor.api.*;
import org.apache.kafka.streams.state.*;

public class CountingProcessor extends ContextualProcessor<String, String, String, Long> {

    private KeyValueStore<String, Long> store;

    @Override
    public void init(ProcessorContext<String, Long> context) {
        super.init(context);
        store = context.getStateStore("counts-store");
    }

    @Override
    public void process(Record<String, String> record) {
        Long current = store.get(record.key());
        if (current == null) current = 0L;
        current++;
        store.put(record.key(), current);

        // forward updated count
        context().forward(record.withValue(current));
    }
}
```

### Registering the store and processor in the topology

```java
import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.state.Stores;

StreamsBuilder builder = new StreamsBuilder();

// define a key-value state store
var storeBuilder = Stores.keyValueStoreBuilder(
        Stores.persistentKeyValueStore("counts-store"),
        org.apache.kafka.common.serialization.Serdes.String(),
        org.apache.kafka.common.serialization.Serdes.Long()
);

builder.addStateStore(storeBuilder);

builder.stream("input-topic")
       .process(() -> new CountingProcessor(), "counts-store")
       .to("output-topic");
```

**Input example** (input-topic):

- Key: `"user1"`, Value: `"click"`
- Key: `"user1"`, Value: `"click"`
- Key: `"user2"`, Value: `"click"`

**Output example** (output-topic):

- Key: `"user1"`, Value: `1`
- Key: `"user1"`, Value: `2`
- Key: `"user2"`, Value: `1`

## 3. Changelog Topics

Each state store can be backed by a **changelog topic**:

- Every update to the store is written as an event to a Kafka topic.
- This topic is **partitioned** and **replicated** like any other Kafka topic.
- On restart or rebalance, the store is **rebuilt** by replaying the changelog.

Key points:

- Name is usually derived from the store name (e.g., `appId-counts-store-changelog`).
- Provides **durability** and **recovery** for local state.

You don’t usually write to the changelog topic directly; Kafka Streams does it automatically.

---

## 4. Fault Tolerance

Kafka Streams achieves fault tolerance for stateful processing via:

1. **Local state stores** (fast access)
2. **Changelog topics** (durable backup)
3. **Standby replicas** (optional, faster failover)
4. **Rebalancing** (tasks move to other instances)

### Failure scenario

- Instance A with task T and state store S crashes.
- Another instance B takes over task T.
- B restores S by:
  - Loading any local state (if present)
  - Replaying the changelog topic from the last committed offset

Processing continues from the last committed position, ensuring **exactly-once** or **at-least-once** semantics (depending on configuration).

## 5. Simple Spring Boot + Kafka Streams Example

Using Spring Boot with `spring-kafka` and Kafka Streams:

```java
import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.kstream.KStream;
import org.apache.kafka.streams.kstream.KTable;
import org.apache.kafka.streams.kstream.Materialized;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.apache.kafka.common.serialization.Serdes;

@Configuration
public class WordCountTopology {

    @Bean
    public KStream<String, String> kStream(StreamsBuilder builder) {
        KStream<String, String> textLines = builder.stream("text-input");

        KTable<String, Long> wordCounts = textLines
                .flatMapValues(value -> java.util.Arrays.asList(value.toLowerCase().split("\\W+")))
                .groupBy((key, word) -> word)
                .count(Materialized.as("word-counts-store")); // state store

        wordCounts.toStream().to("word-counts-output");

        return textLines;
    }
}
```

- `word-counts-store` is a **state store**.
- Kafka Streams creates a **changelog topic** for it.
- On failure, the word counts are restored from that changelog.

**Input** (`text-input`):

- `"Hello Kafka"`
- `"Hello Streams"`

**Output** (`word-counts-output`):

- Key: `"hello"`, Value: `1`
- Key: `"kafka"`, Value: `1`
- Key: `"hello"`, Value: `2`
- Key: `"streams"`, Value: `1`

---

### Core Takeaways

- **State stores** hold local state for stateful operations.
- **Changelog topics** back up state stores to Kafka for durability.
- **Fault tolerance** is achieved by restoring state from changelog topics when tasks move or instances restart.
