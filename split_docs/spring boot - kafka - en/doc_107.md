
**Key idea**

For Kafka with Spring Boot, use the **Single Writer Principle per key**:  
all messages for the **same key** must be written by **exactly one producer instance / thread** to guarantee **strict ordering**.

Kafka guarantees ordering **only within a partition**. If multiple writers send messages with the same key concurrently (even to the same partition), you can get race conditions and unexpected order. A single writer per key avoids this.

---

## Why this matters

- **Ordering requirement**: e.g., account balance updates, order status changes, inventory changes.
- **Kafka guarantee**: messages are ordered per partition, not globally.
- **Risk**: multiple producers writing the same key can interleave messages in unexpected ways.
- **Solution**: ensure that for a given key, only one logical writer sends messages.

---

## Typical patterns to implement this

1. **Partitioning by key + single writer per key group**

   - Use a key-based partitioner so all messages for a key go to the same partition.
   - Ensure that for each key (or key group), only one producer thread writes.

2. **Single producer with internal serialization**

   - Use one `KafkaTemplate` and one sending component.
   - Serialize access per key (e.g., via queues or locks).

3. **Sharded writers**

   - Map keys to a fixed set of writer threads (e.g., by hash).
   - Each writer thread is the only one allowed to send for its assigned keys.

## Spring Boot + Kafka: basic producer example

**Producer configuration**

```java
@Configuration
@EnableKafka
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> configProps = new HashMap<>();
        configProps.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        configProps.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        configProps.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        // Default partitioner uses key hash -> partition
        return new DefaultKafkaProducerFactory<>(configProps);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

**Single writer service (per key)**

```java
@Service
public class OrderEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private final String topic = "order-events";

    public OrderEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    // Single entry point for all writes for a given orderId
    public ListenableFuture<SendResult<String, String>> sendOrderEvent(String orderId, String payload) {
        // orderId is the key -> all events for this order go to same partition
        return kafkaTemplate.send(topic, orderId, payload);
    }
}
```

**Example input/output**

- Input call:

```java
orderEventProducer.sendOrderEvent("order-123", "CREATED");
orderEventProducer.sendOrderEvent("order-123", "PAID");
orderEventProducer.sendOrderEvent("order-123", "SHIPPED");
```

- Kafka topic `order-events` (for partition that holds `order-123`):

```text
Key        | Value
-----------+---------
order-123  | CREATED
order-123  | PAID
order-123  | SHIPPED
```

Because there is a **single writer** for `order-123`, the order is preserved.

## Sharded single-writer pattern (per key group)

If you need concurrency but still want single writer per key, you can shard keys across a fixed set of writer threads:

```java
@Service
public class ShardedOrderEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private final String topic = "order-events";
    private final int shardCount = 8;
    private final ExecutorService[] executors;

    public ShardedOrderEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
        this.executors = new ExecutorService[shardCount];
        for (int i = 0; i < shardCount; i++) {
            executors[i] = Executors.newSingleThreadExecutor();
        }
    }

    public void sendOrderEvent(String orderId, String payload) {
        int shard = Math.abs(orderId.hashCode()) % shardCount;
        executors[shard].submit(() -> kafkaTemplate.send(topic, orderId, payload));
    }
}
```

- All events for the same `orderId` go to the **same single-threaded executor**, so they are written in order.
- Different keys can be processed in parallel by different executors.

## Consumer side

Ordering is preserved **per partition** and **per consumer instance**. To keep ordering:

- Use **one consumer thread per partition**.
- Do not process messages for the same key in parallel within a partition.

Example:

```java
@KafkaListener(topics = "order-events", groupId = "order-service")
public void handleOrderEvent(ConsumerRecord<String, String> record) {
    String orderId = record.key();
    String event = record.value();
    // Process events for this order sequentially
}
```

---

**Summary**

- Kafka orders messages **per partition**, not globally.
- To guarantee ordering for a key, ensure:
  - All messages for that key go to the **same partition** (key-based partitioning).
  - There is a **single logical writer per key** (or per key group).
- In Spring Boot, centralize writes through a single producer component or use sharded single-threaded writers to maintain ordering while allowing concurrency.
