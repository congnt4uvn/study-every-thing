
Here’s a concise summary and some practical examples around:

**“Consumer performance: Tuning fetch sizes, concurrency, and partition assignment”**  
(using **Kafka + Spring Boot**)

---

## 1. Fetch Size Tuning

Goal: Balance **throughput** vs **latency** and **memory usage**.

Key ideas:
- `fetch.min.bytes`: Minimum data (in bytes) the broker should return.  
  - Higher → better throughput, but higher latency (waits for more data).
- `fetch.max.bytes`: Maximum data (in bytes) per fetch request.  
  - Higher → more data per request, but more memory and possible GC pressure.
- `max.partition.fetch.bytes`: Max bytes per partition per fetch.  
  - Controls how much data is fetched from each partition.

### Spring Boot (Spring Kafka) config example

```yaml
spring:
  kafka:
    consumer:
      bootstrap-servers: localhost:9092
      group-id: my-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        fetch.min.bytes: 50000          # 50 KB
        fetch.max.bytes: 5242880        # 5 MB
        max.partition.fetch.bytes: 1048576  # 1 MB per partition
        fetch.max.wait.ms: 100          # wait up to 100ms to fill fetch.min.bytes
```

**Effect (example):**
- If traffic is low, consumer may wait up to 100 ms to accumulate 50 KB before returning data.
- If traffic is high, consumer can get up to 5 MB per fetch, improving throughput.

## 2. Concurrency Tuning

Goal: Use **multiple threads** to process messages in parallel.

Key ideas:
- In Spring Kafka, **concurrency** is usually set on `ConcurrentKafkaListenerContainerFactory`.
- Max effective concurrency per topic = **number of partitions**.
- Too high concurrency with too few partitions → idle threads, no benefit.
- Too low concurrency with many partitions → underutilized CPU, slower processing.

### Spring Boot Java config

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.config.ConcurrentKafkaListenerContainerFactory;
import org.springframework.kafka.core.ConsumerFactory;

@Configuration
public class KafkaConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);
        factory.setConcurrency(4); // 4 concurrent consumer threads
        factory.getContainerProperties().setPollTimeout(3000);

        return factory;
    }
}
```

### Listener example

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class MyConsumer {

    @KafkaListener(topics = "orders", groupId = "order-group")
    public void consume(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                + " received: " + message);
    }
}
```

**Example behavior (inputs/outputs):**

- Topic `orders` has **4 partitions**: `orders-0`, `orders-1`, `orders-2`, `orders-3`.
- Concurrency = 4 → Spring creates 4 consumer threads.

Input messages:
- `orders-0`: `"order-1"`, `"order-5"`
- `orders-1`: `"order-2"`
- `orders-2`: `"order-3"`
- `orders-3`: `"order-4"`

Possible console output:
```text
Thread: consumer-0-C-1 received: order-1
Thread: consumer-1-C-1 received: order-2
Thread: consumer-2-C-1 received: order-3
Thread: consumer-3-C-1 received: order-4
Thread: consumer-0-C-1 received: order-5
```

## 3. Partition Assignment Strategy

Goal: Control **how partitions are distributed** among consumers in a group.

Key ideas:
- Common strategies:
  - `RangeAssignor`: Assigns contiguous ranges of partitions per topic.
  - `RoundRobinAssignor`: Distributes partitions more evenly across consumers.
  - `StickyAssignor`: Tries to keep assignments stable between rebalances.
- Affects **load balancing** and **rebalance behavior**.

### Configuring assignment strategy

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

Or multiple strategies (Kafka will use the first common one across consumers):

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor,
          org.apache.kafka.clients.consumer.RangeAssignor
```

**Example effect:**

- Topic `orders` with 6 partitions, 3 consumers in the same group.
- With `RoundRobinAssignor`:
  - C1: partitions 0, 3
  - C2: partitions 1, 4
  - C3: partitions 2, 5
- With `RangeAssignor`:
  - C1: partitions 0, 1
  - C2: partitions 2, 3
  - C3: partitions 4, 5

This can impact:
- How evenly load is distributed.
- How much data moves when a consumer joins/leaves (rebalance cost).

## 4. Putting It Together: Example Consumer Setup

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: high-throughput-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        fetch.min.bytes: 100000          # 100 KB
        fetch.max.bytes: 10485760        # 10 MB
        max.partition.fetch.bytes: 2097152  # 2 MB per partition
        fetch.max.wait.ms: 200
        partition.assignment.strategy: org.apache.kafka.clients.consumer.StickyAssignor
```

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
        ConsumerFactory<String, String> consumerFactory) {

    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();

    factory.setConsumerFactory(consumerFactory);
    factory.setConcurrency(6); // match or slightly below total partitions
    factory.getContainerProperties().setPollTimeout(3000);

    return factory;
}
```

## Summary of Main Ideas

- **Fetch sizes** (`fetch.min.bytes`, `fetch.max.bytes`, `max.partition.fetch.bytes`)  
  → Tune for throughput vs latency and memory.
- **Concurrency** (Spring Kafka `setConcurrency`)  
  → Use multiple threads; effective max ≈ number of partitions.
- **Partition assignment strategy**  
  → Controls how partitions are distributed; affects load balance and rebalance stability.
