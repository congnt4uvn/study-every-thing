
Here’s a concise summary and some practical examples around:

**“Consumer performance: Scaling consumers horizontally vs increasing partitions”**

---

## Key Ideas

1. **Kafka parallelism is driven by partitions, not consumer count**
   - Each partition can be consumed by **only one consumer** in a consumer group at a time.
   - Maximum parallelism per consumer group = **number of partitions**.
   - If you have 3 partitions and 10 consumers in the same group, only 3 consumers will be active; 7 will be idle.

2. **Scaling by increasing partitions**
   - Pros:
     - Increases maximum parallelism.
     - Allows more consumers in the same group to work in parallel.
   - Cons:
     - Rebalancing cost when you change partition count.
     - More partitions → more overhead on brokers and producers.
     - Ordering is guaranteed only **within a partition**, not across all partitions.

3. **Scaling by adding more consumers (horizontally)**
   - Pros:
     - Easy to scale application instances (e.g., more Spring Boot pods).
     - Good when you already have enough partitions.
   - Cons:
     - If consumer count > partition count, extra consumers are idle.
     - Rebalancing happens when consumers join/leave, which can cause short pauses.

4. **When to increase partitions vs consumers**
   - If **consumers are fully utilized** and you still need more throughput → **increase partitions**.
   - If you have **more partitions than active consumers** and CPU is not saturated → **add more consumers** (scale app instances).
   - For new topics, plan partition count based on:
     - Expected throughput
     - Number of consumer instances you want to run
     - Ordering requirements

## Spring Boot + Kafka: Code Examples

### 1. Basic Consumer with Concurrency (Horizontal Scaling)

`application.yml`:

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: my-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
    listener:
      concurrency: 3   # up to 3 concurrent consumer threads
```

Consumer:

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class MyKafkaConsumer {

    @KafkaListener(topics = "my-topic", groupId = "my-consumer-group")
    public void listen(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                + " | Received: " + message);
    }
}
```

**Behavior:**

- If `my-topic` has **3 partitions** and `concurrency = 3`:
  - Each thread gets one partition → **3-way parallelism**.
- If `my-topic` has **1 partition** and `concurrency = 3`:
  - Only **1 thread** will actually receive messages; others are idle.

**Example Input (messages in topic):**

Partition 0:
- `msg-1`
- `msg-2`

Partition 1:
- `msg-3`
- `msg-4`

Partition 2:
- `msg-5`

**Example Output (console):**

```text
Thread: consumer-0-C-1 | Received: msg-1
Thread: consumer-0-C-1 | Received: msg-2
Thread: consumer-0-C-2 | Received: msg-3
Thread: consumer-0-C-2 | Received: msg-4
Thread: consumer-0-C-3 | Received: msg-5
```

Parallelism is per partition.

---

### 2. Increasing Partitions (Broker / Admin)

Using Kafka CLI:

```bash
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --alter \
  --topic my-topic \
  --partitions 6
```

Now, if you set:

```yaml
spring:
  kafka:
    listener:
      concurrency: 6
```

You can have up to **6 parallel consumer threads** in the same group.

---

### 3. Multiple Consumer Instances (Horizontal Scaling via Instances)

Instead of only using `concurrency`, you can run **multiple instances** of the same Spring Boot app (e.g., in Kubernetes):

- Topic: `my-topic` with **6 partitions**
- Each app instance: `concurrency = 2`
- Run **3 instances** of the app

Total potential consumers in the group = 3 instances × 2 threads = 6  
→ All 6 partitions can be consumed in parallel.

**Key point:** If you only have 3 partitions, then at most 3 of those 6 consumers will be active.

---

### 4. Ordering and Partitions

If you need ordering per key (e.g., per userId):

Producer:

```java
kafkaTemplate.send("my-topic", userId, eventJson);
```

- All messages with the same `userId` go to the **same partition**.
- They will be processed in order by **one consumer**.
- Increasing partitions increases parallelism across **different keys**, not within the same key.

---

## Practical Rules of Thumb

- **Don’t just add consumers** if you don’t have enough partitions; they’ll sit idle.
- **Don’t create too many partitions** without reason; it increases broker and producer overhead.
- For a new topic, choose partitions based on:
  - Target number of consumer instances (e.g., 10 instances → at least 10 partitions).
  - Expected throughput and latency requirements.
- Use **Spring Kafka `concurrency`** for intra-instance parallelism and **more app instances** for horizontal scaling.
- Monitor:
  - Consumer lag
  - CPU usage
  - Rebalance frequency  
  to decide whether to add partitions or consumers.
