
Here’s a concise summary of Kafka Streams state and Exactly-Once Semantics (EOS v2), with code examples.

---

## Key Ideas

1. **Stateful processing in Kafka Streams**
   - Kafka Streams applications can keep local state (e.g., aggregations, joins, windowed counts).
   - This state is stored in **state stores** (e.g., RocksDB) and backed up (changelogged) to Kafka topics for fault tolerance.

2. **Exactly-Once Semantics (EOS)**
   - EOS ensures each input record affects the output **exactly once**, even in the presence of failures and retries.
   - Without EOS, you typically get **at-least-once** (possible duplicates) or **at-most-once** (possible data loss).

3. **EOS v2 (improved exactly-once)**
   - EOS v2 simplifies configuration and improves performance over EOS v1.
   - Uses **idempotent producers** and **transactions** to atomically:
     - Read input records
     - Update state stores
     - Write output records
   - Reduces the number of internal producers/consumers and transaction overhead.

4. **How EOS works conceptually**
   - Each processing step is wrapped in a transaction:
     1. Read records from input topic.
     2. Process and update local state store.
     3. Produce results to output topic.
     4. Commit transaction (or abort on failure).
   - On restart, Kafka Streams restores state from changelog topics and resumes processing without double-applying records.

## Basic Configuration (Spring Boot + Kafka Streams, EOS v2)

**application.yml:**

```yaml
spring:
  kafka:
    streams:
      application-id: my-eos-app
      bootstrap-servers: localhost:9092
      properties:
        processing.guarantee: exactly_once_v2
        # Optional tuning
        commit.interval.ms: 100
        replication.factor: 3
```

Key property:
- `processing.guarantee=exactly_once_v2` → enables EOS v2.

## Simple Kafka Streams Topology with State (Spring Boot)

Example: count messages per key with exactly-once semantics.

**Kafka Streams configuration bean:**

```java
import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.kstream.KStream;
import org.apache.kafka.streams.kstream.KTable;
import org.apache.kafka.streams.kstream.Materialized;
import org.apache.kafka.streams.kstream.Produced;
import org.apache.kafka.common.serialization.Serdes;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class KafkaStreamsConfig {

    @Bean
    public KStream<String, String> kStream(StreamsBuilder builder) {
        KStream<String, String> input = builder.stream("input-topic");

        // Stateful aggregation: count messages per key
        KTable<String, Long> counts = input
                .groupByKey()
                .count(Materialized.as("counts-store")); // state store name

        // Write counts to output topic
        counts.toStream()
              .to("output-topic", Produced.with(Serdes.String(), Serdes.Long()));

        return input;
    }
}
```

- `counts-store` is a local state store (backed by a changelog topic).
- With EOS v2, updates to `counts-store` and writes to `output-topic` are transactional.

## Example Input / Output

**Input topic: `input-topic`**

| Key   | Value   |
|-------|---------|
| user1 | click   |
| user1 | click   |
| user2 | view    |
| user1 | click   |

**Internal state (counts-store) after processing:**

- `user1` → 3  
- `user2` → 1  

**Output topic: `output-topic` (KTable changelog as stream):**

| Key   | Value (count) |
|-------|----------------|
| user1 | 1              |
| user1 | 2              |
| user2 | 1              |
| user1 | 3              |

If the application crashes and restarts:
- State is restored from the changelog.
- Records are not double-counted due to EOS v2.

## Enabling EOS v2 in Plain Kafka Streams (non-Spring)

```java
import org.apache.kafka.streams.KafkaStreams;
import org.apache.kafka.streams.StreamsConfig;
import java.util.Properties;

Properties props = new Properties();
props.put(StreamsConfig.APPLICATION_ID_CONFIG, "my-eos-app");
props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG, StreamsConfig.EXACTLY_ONCE_V2);

KafkaStreams streams = new KafkaStreams(builder.build(), props);
streams.start();
```

---

**Summary:**  
Kafka Streams uses local state stores plus changelog topics for stateful processing. EOS v2 provides exactly-once processing by combining idempotent producers and transactions, ensuring that state updates and output writes are applied exactly once, even under failures. In Spring Boot, you enable this with `processing.guarantee=exactly_once_v2` and then build your stateful topologies as usual.
