
Here’s a concise summary and some practical examples around **capacity planning for Kafka** in a **Spring Boot** context.

---

## 1. Key Concepts in Capacity Planning

### a. Number of Partitions
- **Partitions determine parallelism and throughput**.
- More partitions ⇒ higher potential throughput and consumer parallelism, but:
  - More open files and memory usage on brokers.
  - Longer leader election and rebalance times.

**Rules of thumb:**
- At least as many partitions as the **maximum number of consumer instances** you want to run in a consumer group.
- Estimate based on required throughput per partition.

---

### b. Throughput Estimation

You need to estimate:
- **Messages per second** (or per minute).
- **Average message size** (bytes).
- **Total throughput** = messages/sec × size.

Example:
- 10,000 messages/sec
- 1 KB per message
- Total throughput ≈ 10,000 × 1 KB = 10 MB/sec

If a single partition can safely handle ~5 MB/sec:
- Required partitions ≈ 10 MB/sec ÷ 5 MB/sec = 2 partitions (round up, add safety margin, e.g., 4).

---

### c. Retention and Storage

Kafka storage needed is driven by:
- **Throughput** (bytes/sec)
- **Retention time** (how long you keep data)
- **Replication factor** (for durability)

Formula (approx):

```text
Storage_per_topic = throughput_bytes_per_sec 
                    × retention_seconds 
                    × replication_factor
```

Example:
- Throughput: 10 MB/sec
- Retention: 7 days = 7 × 24 × 3600 = 604,800 sec
- Replication factor: 3

```text
Storage_per_topic = 10 MB/s × 604,800 s × 3
                  = 18,144,000 MB
                  ≈ 18,144 GB
                  ≈ 18 TB
```

You’d then divide this across brokers and disks.

## 2. Simple Calculation Example (Inputs/Outputs)

**Input assumptions:**
- Messages: 5,000 msg/sec
- Avg size: 2 KB
- Retention: 3 days
- Replication factor: 3
- Target throughput per partition: 4 MB/sec

**Step 1 – Throughput:**

```text
Throughput = 5,000 msg/s × 2 KB = 10,000 KB/s = 10 MB/s
```

**Step 2 – Partitions:**

```text
Partitions = 10 MB/s ÷ 4 MB/s ≈ 2.5 → round up to 3–4
```

**Step 3 – Storage:**

Retention: 3 days = 259,200 sec

```text
Storage = 10 MB/s × 259,200 s × 3
        = 7,776,000 MB
        ≈ 7.8 TB
```

## 3. Spring Boot + Kafka: Configuration Examples

### a. application.yml – Basic Producer/Consumer Tuning

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092

    producer:
      acks: all
      batch-size: 16384
      linger-ms: 5
      buffer-memory: 33554432
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer

    consumer:
      group-id: my-consumer-group
      auto-offset-reset: earliest
      max-poll-records: 500
      fetch-min-size: 1048576   # 1 MB
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer

    listener:
      concurrency: 4  # match or be <= number of partitions
```

- `listener.concurrency` should not exceed the number of partitions for that topic in a single consumer group.

---

### b. Topic Creation with Partitions & Retention (Spring Boot)

Use a `NewTopic` bean to define partitions and retention:

```java
import org.apache.kafka.clients.admin.NewTopic;
import org.apache.kafka.common.config.TopicConfig;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic ordersTopic() {
        int partitions = 4;          // from your capacity planning
        short replicationFactor = 3; // cluster size permitting

        Map<String, String> configs = new HashMap<>();
        // Retention: 3 days in ms
        configs.put(TopicConfig.RETENTION_MS_CONFIG, String.valueOf(3L * 24 * 60 * 60 * 1000));
        // Optional: size-based retention
        // configs.put(TopicConfig.RETENTION_BYTES_CONFIG, String.valueOf(500L * 1024 * 1024 * 1024)); // 500 GB

        return new NewTopic("orders", partitions, replicationFactor)
                .configs(configs);
    }
}
```

### c. Simple Producer & Consumer Example

**Producer (Spring Boot service):**

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        kafkaTemplate.send("orders", orderId, payload);
    }
}
```

**Consumer (Spring Kafka listener):**

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service-group")
    public void consume(String message) {
        // process message
        System.out.println("Received order: " + message);
    }
}
```

## 4. Example Input/Output Flow

**Input:**
```http
POST /orders
{
  "orderId": "123",
  "item": "Book",
  "quantity": 2
}
```

Your controller calls `OrderProducer.sendOrder("123", "{...json...}")`.

**Kafka:**
- Message goes to topic `orders` with key `"123"`.
- Partition is chosen based on key hash (ensures same key → same partition).

**Output (consumer log):**
```text
Received order: {"orderId":"123","item":"Book","quantity":2}
```

---

**Summary:**  
Capacity planning in Kafka revolves around:
- Choosing **enough partitions** for required throughput and consumer parallelism.
- Calculating **throughput** (msg/sec × size).
- Estimating **storage** from throughput, retention, and replication.
Spring Boot integrates with Kafka by:
- Configuring producer/consumer performance settings.
- Defining topics with specific partitions and retention.
- Implementing producers/consumers that respect your capacity plan (e.g., concurrency vs partitions).
