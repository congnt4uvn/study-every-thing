
Here’s a concise summary and some practical examples around:

**“Spring integration: Monitoring Kafka Streams apps (thread states, lag, store sizes)”**

---

## Key Ideas

1. **Kafka Streams + Spring Boot**
   - You can run Kafka Streams applications inside a Spring Boot app using `spring-kafka`.
   - Spring provides integration points to **monitor**:
     - Thread states (e.g., RUNNING, DEAD, CREATED)
     - Consumer lag
     - State store sizes

2. **Thread State Monitoring**
   - Kafka Streams exposes thread metadata and states.
   - You can access this via `KafkaStreams` bean in Spring and expose it via REST or metrics.

3. **Lag Monitoring**
   - Lag = difference between the latest offset in a partition and the current consumer position.
   - You can:
     - Use Kafka consumer metrics (via Micrometer / Actuator).
     - Or manually query offsets using an `AdminClient` and a `KafkaConsumer`.

4. **State Store Size Monitoring**
   - Kafka Streams state stores (RocksDB, in-memory) can be queried for approximate sizes.
   - You can use `QueryableStoreTypes` and `ReadOnlyKeyValueStore` to inspect store contents and estimate size.

## Example: Basic Kafka Streams + Spring Boot Configuration

**application.yml**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    streams:
      application-id: my-streams-app
      properties:
        commit.interval.ms: 1000
        num.stream.threads: 3
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,beans
```

**Stream topology configuration**

```java
@Configuration
@EnableKafkaStreams
public class KafkaStreamsConfig {

    @Bean(name = KafkaStreamsDefaultConfiguration.DEFAULT_STREAMS_CONFIG_BEAN_NAME)
    public KafkaStreamsConfiguration kStreamsConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "my-streams-app");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        return new KafkaStreamsConfiguration(props);
    }

    @Bean
    public KStream<String, String> kStream(StreamsBuilder builder) {
        KStream<String, String> input = builder.stream("input-topic");
        input
            .mapValues(v -> v.toUpperCase())
            .to("output-topic");
        return input;
    }
}
```

## Monitoring Thread States

**Expose thread states via a REST controller**

```java
@RestController
@RequestMapping("/streams")
public class StreamsMonitoringController {

    private final KafkaStreams kafkaStreams;

    public StreamsMonitoringController(KafkaStreams kafkaStreams) {
        this.kafkaStreams = kafkaStreams;
    }

    @GetMapping("/threads")
    public List<Map<String, Object>> getThreadStates() {
        return kafkaStreams
                .localThreadsMetadata()
                .stream()
                .map(meta -> {
                    Map<String, Object> m = new HashMap<>();
                    m.put("threadName", meta.threadName());
                    m.put("threadState", meta.threadState().name());
                    m.put("activeTasks", meta.activeTasks().size());
                    m.put("standbyTasks", meta.standbyTasks().size());
                    return m;
                })
                .toList();
    }
}
```

**Example output (JSON)**

```json
[
  {
    "threadName": "my-streams-app-StreamThread-1",
    "threadState": "RUNNING",
    "activeTasks": 2,
    "standbyTasks": 1
  },
  {
    "threadName": "my-streams-app-StreamThread-2",
    "threadState": "RUNNING",
    "activeTasks": 1,
    "standbyTasks": 2
  }
]
```

## Monitoring Lag (Conceptual Example)

You can compute lag by comparing **end offsets** vs **current positions**.

```java
@Component
public class LagMonitor {

    private final AdminClient adminClient;
    private final KafkaConsumer<String, String> consumer;

    public LagMonitor() {
        Properties props = new Properties();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "my-streams-app");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false");
        this.consumer = new KafkaConsumer<>(props);
        this.adminClient = AdminClient.create(props);
    }

    public Map<TopicPartition, Long> getLag(String topic) {
        List<TopicPartition> partitions = consumer.partitionsFor(topic).stream()
                .map(info -> new TopicPartition(info.topic(), info.partition()))
                .toList();

        consumer.assign(partitions);

        Map<TopicPartition, Long> endOffsets = consumer.endOffsets(partitions);
        Map<TopicPartition, Long> currentPositions = new HashMap<>();
        for (TopicPartition tp : partitions) {
            currentPositions.put(tp, consumer.position(tp));
        }

        Map<TopicPartition, Long> lag = new HashMap<>();
        for (TopicPartition tp : partitions) {
            long end = endOffsets.get(tp);
            long pos = currentPositions.get(tp);
            lag.put(tp, end - pos);
        }
        return lag;
    }
}
```

**Example output (lag per partition)**

```text
TopicPartition{topic='input-topic', partition=0} -> lag: 120
TopicPartition{topic='input-topic', partition=1} -> lag: 0
```

In production, you’d usually rely on **Micrometer + Actuator + Kafka metrics** instead of a manual consumer.

## Monitoring State Store Sizes

Assume you have a state store named `"my-store"`.

**Defining a state store in the topology**

```java
@Bean
public KStream<String, String> kStreamWithStore(StreamsBuilder builder) {
    StoreBuilder<KeyValueStore<String, String>> storeBuilder =
            Stores.keyValueStoreBuilder(
                    Stores.persistentKeyValueStore("my-store"),
                    Serdes.String(),
                    Serdes.String()
            );
    builder.addStateStore(storeBuilder);

    KStream<String, String> input = builder.stream("input-topic");

    input.transformValues(() -> new ValueTransformerWithKey<>() {
        private KeyValueStore<String, String> store;

        @Override
        public void init(ProcessorContext context) {
            this.store = (KeyValueStore<String, String>) context.getStateStore("my-store");
        }

        @Override
        public String transform(String readOnlyKey, String value) {
            store.put(readOnlyKey, value);
            return value;
        }

        @Override
        public void close() {}
    }, "my-store").to("output-topic");

    return input;
}
```

**Querying store size via interactive queries**

```java
@RestController
@RequestMapping("/stores")
public class StoreMonitoringController {

    private final KafkaStreams kafkaStreams;

    public StoreMonitoringController(KafkaStreams kafkaStreams) {
        this.kafkaStreams = kafkaStreams;
    }

    @GetMapping("/my-store/size")
    public Map<String, Object> getStoreSize() {
        ReadOnlyKeyValueStore<String, String> store =
                kafkaStreams.store(
                        StoreQueryParameters.fromNameAndType(
                                "my-store",
                                QueryableStoreTypes.keyValueStore()
                        )
                );

        long count = 0;
        try (KeyValueIterator<String, String> all = store.all()) {
            while (all.hasNext()) {
                all.next();
                count++;
            }
        }

        Map<String, Object> result = new HashMap<>();
        result.put("storeName", "my-store");
        result.put("approximateEntryCount", count);
        return result;
    }
}
```

**Example output**

```json
{
  "storeName": "my-store",
  "approximateEntryCount": 5234
}
```

## Summary

- Use `@EnableKafkaStreams` and `KafkaStreamsConfiguration` to integrate Kafka Streams with Spring Boot.
- Monitor **thread states** via `KafkaStreams.localThreadsMetadata()` and expose via REST or metrics.
- Monitor **lag** using Kafka consumer metrics (Micrometer/Actuator) or manual offset comparison.
- Monitor **state store sizes** using interactive queries (`QueryableStoreTypes`) and expose counts or other metrics via REST/Actuator.
