
**Summary: Partitioning Strategies in Kafka (Key-Based Partitioning)**

### 1. What is Key-Based Partitioning?

In Kafka, messages are written to *topics* that are split into *partitions*.  
With **key-based partitioning**, the partition is chosen based on the message key:

```text
partition = hash(key) % number_of_partitions
```

This means:
- Same key → always goes to the same partition (as long as partition count and partitioner stay the same).
- Different keys → distributed across partitions.

---

### 2. Impact on Data Locality

**Data locality** = related data ends up in the same partition.

Benefits:
- All events for a given key (e.g., `userId`, `orderId`) are in one partition.
- Kafka guarantees **ordering per partition**, so you get **per-key ordering**.
- Consumers can process all events for a key in order, which is useful for:
  - Aggregations (e.g., total orders per user)
  - Stateful processing (e.g., user session state)
  - Exactly-once or idempotent updates per key

Example:
- Key: `user-123`
- All messages with key `user-123` go to partition 2.
- A consumer reading partition 2 sees all `user-123` events in order.

---

### 3. Hot Partitions (Skew)

**Hot partition** = one partition receives much more traffic than others.

This happens when:
- A few keys are extremely popular (e.g., `key = "ALL"`, or a few “celebrity” users).
- Hash distribution is skewed because of key distribution.

Consequences:
- One broker/partition becomes a bottleneck.
- That partition’s consumer is overloaded.
- Overall throughput and latency suffer, even if other partitions are idle.

Mitigation strategies:
- Choose keys with good cardinality and distribution (e.g., `userId` instead of `country`).
- Use **composite keys** (e.g., `country#userId`).
- Use a **custom partitioner** to spread hot keys across multiple partitions.
- Increase number of partitions (helps only if keys are reasonably distributed).

---

### 4. Basic Spring Boot + Kafka Example (Key-Based Partitioning)

**Producer configuration (application.yml):**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

**Producer code:**

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class UserEventProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private static final String TOPIC = "user-events";

    public UserEventProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(String userId, String event) {
        // userId is the key → determines partition
        kafkaTemplate.send(TOPIC, userId, event);
    }
}
```

**Example inputs and resulting partition behavior:**

Assume topic `user-events` has 3 partitions (0, 1, 2).

```java
sendUserEvent("user-1", "login");
sendUserEvent("user-1", "click");
sendUserEvent("user-2", "login");
sendUserEvent("user-3", "logout");
```

Possible mapping (implementation-dependent, but consistent):

- `user-1` → hash("user-1") % 3 = 1 → **partition 1**
- `user-2` → hash("user-2") % 3 = 0 → **partition 0**
- `user-3` → hash("user-3") % 3 = 2 → **partition 2**

**Output (conceptual):**

- Partition 1: `("user-1", "login")`, `("user-1", "click")` (ordered)
- Partition 0: `("user-2", "login")`
- Partition 2: `("user-3", "logout")`

All events for `user-1` are in the same partition and in order.

### 5. Custom Partitioner Example (to handle hot keys)

You can define a custom partitioner to spread a hot key across partitions.

**Custom partitioner:**

```java
import org.apache.kafka.clients.producer.Partitioner;
import org.apache.kafka.common.Cluster;

import java.util.Map;

public class CustomUserPartitioner implements Partitioner {

    @Override
    public int partition(String topic, Object keyObj, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {

        int numPartitions = cluster.partitionsForTopic(topic).size();
        String key = (String) keyObj;

        // Example: if key is "HOT_USER", spread across all partitions
        if ("HOT_USER".equals(key)) {
            // simple round-robin based on value hash
            return Math.abs(value.hashCode()) % numPartitions;
        }

        // default: normal key-based partitioning
        return Math.abs(key.hashCode()) % numPartitions;
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}
```

**Producer config using custom partitioner:**

```yaml
spring:
  kafka:
    producer:
      properties:
        partitioner.class: com.example.kafka.CustomUserPartitioner
```

**Input / Output behavior:**

Inputs:

```java
sendUserEvent("HOT_USER", "event-1");
sendUserEvent("HOT_USER", "event-2");
sendUserEvent("HOT_USER", "event-3");
```

Outputs (conceptual):

- `HOT_USER` events are spread across partitions (0, 1, 2) instead of all going to one partition, reducing the hot-partition problem, but losing strict per-key ordering for that special key.

---

### 6. Key Takeaways

- Key-based partitioning:
  - Ensures **per-key ordering** and **data locality**.
  - Is essential for stateful and ordered processing.
- Risk:
  - Poor key choice → **hot partitions** and performance bottlenecks.
- Design keys and partitioning strategy carefully:
  - Use well-distributed keys.
  - Consider composite keys or custom partitioners for skewed workloads.
