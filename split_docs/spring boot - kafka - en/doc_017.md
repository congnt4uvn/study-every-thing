
Here’s a concise summary and some practical examples around:

**“Topics, Partitions, Ordering: Rebalancing partitions across brokers”**

---

## Key Concepts

1. **Topic**
   - A named stream of records (e.g., `"orders"`, `"payments"`).
   - Logically groups messages of the same type.

2. **Partition**
   - A topic is split into one or more partitions.
   - Each partition is an ordered, append-only log.
   - Ordering is guaranteed **only within a single partition**, not across the whole topic.

3. **Ordering**
   - If you need strict ordering for related messages, they must go to the **same partition**.
   - This is usually done by using a **key** (e.g., `orderId`) so all messages with the same key are routed to the same partition.

4. **Rebalancing Partitions Across Brokers**
   - Kafka clusters have multiple **brokers** (servers).
   - Partitions are distributed across brokers for:
     - Load balancing
     - Fault tolerance
   - When:
     - A broker is added/removed
     - A broker fails
     - Topic partition count or replication changes  
     Kafka will **rebalance** partitions across brokers.
   - Rebalancing moves partition leadership and/or replicas to different brokers.
   - **Ordering is preserved within each partition**, but:
     - Consumers may be reassigned to different partitions.
     - There can be a short pause in consumption during rebalance.

---

## Practical Implications

- Use **keys** to ensure related messages go to the same partition → preserves ordering for that key.
- Be aware that **consumer groups** will rebalance when:
  - A consumer joins/leaves the group
  - Topic partitions change
- During rebalancing:
  - Some consumers stop consuming and then resume with new partition assignments.
  - Your application should handle `onPartitionsRevoked` / `onPartitionsAssigned` events (in Spring Kafka: `ConsumerRebalanceListener`).

## Spring Boot / Spring Kafka Examples

### 1. Producer: Send messages with a key (to keep ordering per key)

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        // orderId is the key -> all messages for same orderId go to same partition
        kafkaTemplate.send("orders", orderId, payload);
    }
}
```

**Input example:**

```java
orderProducer.sendOrder("order-123", "CREATED");
orderProducer.sendOrder("order-123", "PAID");
orderProducer.sendOrder("order-123", "SHIPPED");
```

**Resulting behavior:**

- All three messages with key `"order-123"` go to the **same partition** of topic `"orders"`.
- A consumer reading that partition will see them in this order:
  1. `CREATED`
  2. `PAID`
  3. `SHIPPED`

---

### 2. Consumer: Basic Spring Kafka listener

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf(
            "Partition=%d, Offset=%d, Key=%s, Value=%s%n",
            record.partition(),
            record.offset(),
            record.key(),
            record.value()
        );
    }
}
```

**Example output (console):**

```
Partition=1, Offset=10, Key=order-123, Value=CREATED
Partition=1, Offset=11, Key=order-123, Value=PAID
Partition=1, Offset=12, Key=order-123, Value=SHIPPED
```

Ordering is guaranteed for `order-123` within partition 1.

### 3. Handling Rebalance Events (Spring Kafka)

You can customize behavior during rebalancing (e.g., commit offsets, clean up resources):

```java
@Configuration
public class KafkaConsumerConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);

        factory.getContainerProperties().setConsumerRebalanceListener(
            new ConsumerRebalanceListener() {
                @Override
                public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
                    System.out.println("Partitions revoked: " + partitions);
                    // e.g., commit offsets or cleanup
                }

                @Override
                public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
                    System.out.println("Partitions assigned: " + partitions);
                    // e.g., seek to specific offsets if needed
                }
            }
        );

        return factory;
    }
}
```

During a rebalance (e.g., when a new consumer joins the group), you’ll see logs like:

**Example output:**

```
Partitions revoked: [orders-0, orders-1]
Partitions assigned: [orders-1]
```

This indicates that the consumer’s partition assignments changed due to rebalancing.

---

### 4. Changing Partition Count (Triggers Rebalance)

Using Kafka CLI (outside Spring) to increase partitions:

```bash
kafka-topics.sh --alter \
  --topic orders \
  --partitions 6 \
  --bootstrap-server localhost:9092
```

- This will trigger a **rebalance** in all consumer groups reading from `orders`.
- New partitions will be distributed across brokers.
- Consumers in the same group will be reassigned partitions.

---

## Summary

- **Topic**: logical stream.
- **Partition**: ordered log; ordering guaranteed only within a partition.
- **Keyed messages**: ensure related messages go to the same partition → preserves ordering per key.
- **Rebalancing**: happens when brokers/consumers/partitions change; partitions are redistributed across brokers and consumers.
- Spring Kafka provides:
  - `KafkaTemplate` for keyed sends (ordering per key).
  - `@KafkaListener` for consumption.
  - `ConsumerRebalanceListener` hooks to handle rebalancing events safely.
