
Here’s a concise summary and some practical examples around **data contracts** and **backward/forward compatibility** in a Kafka + Spring Boot context.

---

## 1. Core Idea: Data Contracts in Kafka

A **data contract** defines the structure and meaning of messages on a Kafka topic (e.g., fields, types, semantics).  
The main goal: allow **independent evolution** of producers and consumers without breaking each other.

Key points:
- Producers and consumers may run different versions at the same time.
- Schema changes must be done in a way that:
  - Old consumers can still read new messages (**backward compatibility**).
  - New consumers can still read old messages (**forward compatibility**).
- Typically enforced via a **schema registry** (e.g., Confluent Schema Registry) and a serialization format (Avro, Protobuf, JSON Schema).

---

## 2. Compatibility Types

### Backward Compatible
New schema can read data written with the **previous** schema.

Common rules:
- You can **add optional fields with defaults**.
- You can **remove fields** that consumers don’t rely on.
- You **cannot** change field types in incompatible ways.

Use when:
- You want to deploy **new consumers** first, then **new producers**.

### Forward Compatible
Old schema can read data written with the **new** schema.

Common rules:
- You can **add fields** only if old consumers can ignore them or have defaults.
- You **cannot** remove fields that old consumers require.

Use when:
- You want to deploy **new producers** first, then **new consumers**.

### Full Compatibility
Both backward and forward compatible:
- New and old versions can read each other’s data.
- Most restrictive, but safest for long-lived topics.

## 3. Example: Avro Schema Evolution

### v1 Schema

```json
{
  "type": "record",
  "name": "UserEvent",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "userId", "type": "string" },
    { "name": "action", "type": "string" }
  ]
}
```

### v2 Schema (Backward compatible)

- Add a new field with a default value:

```json
{
  "type": "record",
  "name": "UserEvent",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "userId", "type": "string" },
    { "name": "action", "type": "string" },
    { "name": "timestamp", "type": "long", "default": 0 }
  ]
}
```

Why backward compatible?
- New consumers (v2) can read old messages (v1).  
  For old messages, `timestamp` will be set to `0` (the default).

## 4. Spring Boot + Kafka + Avro Example

Assume:
- Using Confluent Schema Registry.
- Using Avro-generated class `UserEvent`.

### Maven Dependencies

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>

    <dependency>
        <groupId>io.confluent</groupId>
        <artifactId>kafka-avro-serializer</artifactId>
        <version>7.6.0</version>
    </dependency>
</dependencies>
```

### application.yml

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
      specific.avro.reader: true
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      group-id: user-events-group
```

### Producer Example

```java
@Service
public class UserEventProducer {

    private final KafkaTemplate<String, UserEvent> kafkaTemplate;

    public UserEventProducer(KafkaTemplate<String, UserEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUserEvent(String userId, String action, long timestamp) {
        UserEvent event = UserEvent.newBuilder()
                .setUserId(userId)
                .setAction(action)
                .setTimestamp(timestamp) // in v1 this field didn’t exist
                .build();

        kafkaTemplate.send("user-events", userId, event);
    }
}
```

**Example input (method call):**

```java
sendUserEvent("u-123", "LOGIN", 1716200000000L);
```

**Resulting message (conceptual JSON view):**

```json
{
  "userId": "u-123",
  "action": "LOGIN",
  "timestamp": 1716200000000
}
```

Old consumers using v1 schema:
- Will ignore `timestamp` or treat it as unknown (depending on their schema).
- If they use v2 schema with default, old messages will have `timestamp = 0`.

### Consumer Example

```java
@Service
public class UserEventConsumer {

    @KafkaListener(topics = "user-events", groupId = "user-events-group")
    public void consume(UserEvent event) {
        System.out.printf(
            "Received event: userId=%s, action=%s, timestamp=%d%n",
            event.getUserId(), event.getAction(), event.getTimestamp()
        );
    }
}
```

**Example output (console):**

```text
Received event: userId=u-123, action=LOGIN, timestamp=1716200000000
```

## 5. Practical Strategy

1. **Define schemas** and store them in a schema registry.
2. **Set compatibility mode** (e.g., BACKWARD, FORWARD, FULL) per subject/topic.
3. When changing schemas:
   - Add fields with defaults (for backward compatibility).
   - Avoid changing field types or removing required fields.
4. Deploy order:
   - For backward compatibility: deploy **consumers** first, then **producers**.
   - For forward compatibility: deploy **producers** first, then **consumers**.

This approach keeps Kafka topics stable while allowing your Spring Boot services to evolve independently.
