
Here’s a concise summary of Kafka cluster metadata and how it’s handled by brokers and clients, with simple examples.

---

## 1. What is Kafka Cluster Metadata?

Cluster metadata is information about the Kafka cluster, including:

- **Brokers**: their IDs and addresses
- **Topics**: names, partitions, replication factor
- **Partitions**: which broker is leader, which are replicas, ISR (in-sync replicas)
- **Controller**: which broker is the controller
- **Configs**: topic-level and broker-level configs

Clients (producers/consumers) need this metadata to know:
- Where to send messages (which broker/partition)
- Where to read messages from

---

## 2. How Metadata is Stored

- **On the brokers**:
  - In **ZooKeeper** (older Kafka) or **KRaft metadata quorum** (newer Kafka without ZooKeeper).
  - The **controller broker** manages the authoritative metadata and propagates changes to other brokers.
- **On each broker**:
  - Brokers keep an in-memory view of the cluster metadata.
  - They update it when the controller sends metadata updates (e.g., new topic, leader change).

---

## 3. How Metadata is Propagated

1. **Controller detects change**  
   Example: a broker goes down, or a new topic is created.
2. **Controller updates metadata store**  
   - ZooKeeper or KRaft log.
3. **Controller sends updates to brokers**  
   - Brokers update their in-memory metadata.
4. **Clients refresh metadata**  
   - Clients periodically or on error send a `Metadata` request to a broker.
   - The broker responds with the latest cluster metadata.

---

## 4. How Clients Cache Metadata

- **Clients (producers/consumers)** keep a **local cache** of:
  - Topic → partitions
  - Partition → leader broker
  - Broker → host:port
- They **don’t query metadata on every request**; they:
  - Use cached metadata for routing.
  - Refresh metadata:
    - On a schedule (e.g., every few minutes).
    - On errors (e.g., `LEADER_NOT_AVAILABLE`, `UNKNOWN_TOPIC_OR_PARTITION`).
- This reduces load on the cluster and improves performance.

---

## 5. Example: Producer Metadata Flow

1. Producer starts with **bootstrap servers** (a few broker addresses).
2. Producer sends a **MetadataRequest** to one of them.
3. Broker returns:
   - List of brokers
   - Topics, partitions, leaders
4. Producer caches this and uses it to:
   - Choose partition (based on key or partitioner)
   - Send records directly to the leader broker for that partition
5. If leader changes or topic is unknown:
   - Producer gets an error
   - Triggers metadata refresh
   - Updates cache and retries

---

## 6. Spring Boot + Spring Kafka Example

### 6.1. Basic Configuration (application.yml)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
    consumer:
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

Spring Kafka uses the underlying Kafka client, which handles metadata caching and refresh automatically.

---

### 6.2. Producer Example (Spring Boot)

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class DemoProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public DemoProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value)
                     .whenComplete((result, ex) -> {
                         if (ex != null) {
                             System.out.println("Send failed: " + ex.getMessage());
                         } else {
                             System.out.println("Sent to partition " +
                                 result.getRecordMetadata().partition() +
                                 " offset " + result.getRecordMetadata().offset());
                         }
                     });
    }
}
```

**Input** (call from a REST controller):

```java
@RestController
@RequestMapping("/api")
public class DemoController {

    private final DemoProducer producer;

    public DemoController(DemoProducer producer) {
        this.producer = producer;
    }

    @PostMapping("/send")
    public String send(@RequestParam String topic,
                       @RequestParam String key,
                       @RequestParam String value) {
        producer.sendMessage(topic, key, value);
        return "OK";
    }
}
```

**Example HTTP Request (input):**

```http
POST /api/send?topic=test-topic&key=user1&value=hello
```

**Example Output (console log):**

```text
Sent to partition 0 offset 15
```

Behind the scenes:
- The producer uses **metadata cache** to know which broker is leader for `test-topic` partition 0.
- If metadata is stale (e.g., leader changed), it gets an error, refreshes metadata, and retries.

---

### 6.3. Consumer Example (Spring Boot)

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class DemoConsumer {

    @KafkaListener(topics = "test-topic", groupId = "demo-group")
    public void listen(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Input**: message produced to `test-topic`  
**Output (console)**:

```text
Received: hello
```

The consumer:
- Uses metadata to find which broker/partition to read from.
- Automatically rebalances and refreshes metadata when partitions or leaders change.

---

## 7. Key Takeaways

- Metadata is **centralized** (controller + ZooKeeper/KRaft) and **cached** on brokers and clients.
- Clients:
  - Use **bootstrap servers** only to discover the cluster.
  - Maintain a **local metadata cache** for routing.
  - **Refresh metadata** periodically and on errors.
- Spring Boot + Spring Kafka rely on the Kafka client’s built-in metadata handling; you usually don’t manage metadata manually.
