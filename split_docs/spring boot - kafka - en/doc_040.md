
Here’s a concise summary of the Kafka consumer group model and partition assignment strategies, with Spring Boot / Spring Kafka examples.

---

## 1. Consumer Group Model

- A **consumer group** is a set of consumers that work together to consume data from one or more topics.
- Each **partition** of a topic is consumed by **only one consumer** in a group at a time.
- If you have:
  - `P` partitions
  - `C` consumers in the same group  
  Then:
  - If `C <= P`: some consumers may handle multiple partitions.
  - If `C > P`: some consumers will be idle.

### Rebalancing

- **Rebalancing** happens when:
  - A consumer joins the group
  - A consumer leaves the group (crash, shutdown)
  - Topic partitions change (added/removed)
- During rebalance:
  - Partitions are reassigned among consumers.
  - Consumers temporarily stop consuming while the rebalance occurs.

---

## 2. Partition Assignment Strategies

Kafka decides which consumer gets which partition using an **assignment strategy**. Common ones:

### a) RangeAssignor (Range)

- Partitions are divided into **contiguous ranges** per topic.
- Example:
  - Topic `T` with partitions: `T-0, T-1, T-2, T-3`
  - Consumers: `C1, C2`
  - Assignment:
    - `C1`: `T-0, T-1`
    - `C2`: `T-2, T-3`
- Can lead to **uneven load** if number of partitions is not divisible by number of consumers.

### b) RoundRobinAssignor (Round-robin)

- Partitions are assigned in a **round-robin** fashion across all consumers.
- Example:
  - Topic `T` with partitions: `T-0, T-1, T-2, T-3`
  - Consumers: `C1, C2`
  - Assignment:
    - `C1`: `T-0, T-2`
    - `C2`: `T-1, T-3`
- Tends to give a more **balanced distribution** across consumers.

### c) StickyAssignor (Sticky)

- Tries to:
  1. **Keep existing assignments** as stable as possible (minimize movement).
  2. **Balance** partitions across consumers.
- On rebalance, it moves the **fewest possible partitions** to achieve balance.
- Good for reducing **rebalance overhead** and improving **cache locality**.

## 3. Spring Boot / Spring Kafka Configuration Examples

### 3.1 Basic Consumer Group Configuration

`application.yml`:

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: my-consumer-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

Consumer:

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

@Service
public class MyConsumer {

    @KafkaListener(topics = "my-topic", groupId = "my-consumer-group")
    public void listen(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Input (producer sends):**

```bash
# Using kafka-console-producer
> kafka-console-producer --broker-list localhost:9092 --topic my-topic
> Hello
> World
```

**Output (consumer logs):**

```text
Received: Hello
Received: World
```

If you run **3 instances** of this Spring Boot app with the same `groupId` and the topic has **3 partitions**, each instance will consume from (roughly) one partition.

---

### 3.2 Setting Partition Assignment Strategy in Spring Boot

You can configure the assignment strategy via consumer properties.

`application.yml`:

```yaml
spring:
  kafka:
    consumer:
      group-id: my-consumer-group
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor
```

Or multiple strategies (Kafka will use the first common one between client and broker):

```yaml
spring:
  kafka:
    consumer:
      group-id: my-consumer-group
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.StickyAssignor,
          org.apache.kafka.clients.consumer.RoundRobinAssignor
```

Other options:

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.RangeAssignor
```

### 3.3 Example: Observing Rebalance

Consumer with logging:

```java
import org.apache.kafka.clients.consumer.ConsumerRebalanceListener;
import org.apache.kafka.common.TopicPartition;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.kafka.listener.ConsumerAwareRebalanceListener;
import org.springframework.stereotype.Component;

import java.util.Collection;

@Component
public class LoggingRebalanceListener implements ConsumerAwareRebalanceListener {

    @Override
    public void onPartitionsRevokedBeforeCommit(org.apache.kafka.clients.consumer.Consumer<?, ?> consumer,
                                                Collection<TopicPartition> partitions) {
        System.out.println("Partitions revoked: " + partitions);
    }

    @Override
    public void onPartitionsAssigned(org.apache.kafka.clients.consumer.Consumer<?, ?> consumer,
                                     Collection<TopicPartition> partitions) {
        System.out.println("Partitions assigned: " + partitions);
    }
}
```

Consumer:

```java
@Service
public class MyConsumer {

    @KafkaListener(topics = "my-topic", groupId = "my-consumer-group")
    public void listen(String message) {
        System.out.println("Received: " + message);
    }
}
```

**Behavior:**

- Start 1 instance → it gets all partitions.
- Start a 2nd instance → rebalance occurs:
  - Logs show partitions revoked/assigned.
  - Partitions are redistributed according to the configured strategy (range, round-robin, or sticky).

---

## 4. Summary of When to Use Which Strategy

- **RangeAssignor**: Simple, per-topic contiguous ranges; may be uneven.
- **RoundRobinAssignor**: Better balance across consumers; good default for many cases.
- **StickyAssignor**: Balanced and minimizes movement on rebalance; often preferred in modern setups to reduce churn and improve stability.
