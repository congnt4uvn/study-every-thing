
Here’s a concise summary and some practical examples around **Kafka replication**, **rack awareness**, and **cross-AZ replication patterns** in a Spring Boot context.

---

## Key Ideas

### 1. Kafka Replication Basics
- **Replication factor**: Each topic partition is stored on multiple brokers.
- Purpose:
  - High availability (if one broker fails, others have the data).
  - Durability (data not lost when a broker goes down).
- One replica is the **leader**, others are **followers**. Producers/consumers normally talk to the leader.

---

### 2. Rack Awareness

**Goal**: Avoid all replicas of a partition being placed in the same failure domain (rack / AZ).

- Kafka brokers can be assigned a **rack** (or AZ) label.
- When rack awareness is enabled, Kafka tries to place replicas across different racks.
- This improves resilience against:
  - Rack failures
  - Availability Zone (AZ) failures in the cloud

**Broker config example** (e.g., `server.properties`):

```properties
# Broker 1
broker.id=1
broker.rack=az-a

# Broker 2
broker.id=2
broker.rack=az-b

# Broker 3
broker.id=3
broker.rack=az-c
```

**Topic creation with replication factor 3**:

```bash
kafka-topics.sh \
  --create \
  --topic orders \
  --partitions 6 \
  --replication-factor 3 \
  --bootstrap-server broker1:9092
```

With rack awareness, Kafka will try to place the 3 replicas of each partition across `az-a`, `az-b`, and `az-c`.

---

### 3. Cross-AZ Replication Patterns

**Goal**: Survive an entire AZ outage and/or serve traffic from multiple AZs.

Common patterns:

1. **Single Kafka cluster spanning multiple AZs**
   - Brokers distributed across AZs.
   - Use rack awareness to spread replicas.
   - Pros: simpler to manage (one cluster).
   - Cons: cross-AZ network traffic, possible higher latency.

2. **Multiple Kafka clusters (one per region/AZ group) with replication**
   - Use **MirrorMaker 2**, **Confluent Replicator**, or **Kafka Connect** to replicate topics between clusters.
   - Patterns:
     - Active–passive: one primary cluster, one DR cluster.
     - Active–active: both clusters serve traffic and replicate to each other (more complex conflict handling).

**MirrorMaker 2 example (conceptual)**:

```properties
# mm2.properties
clusters = primary, secondary

primary.bootstrap.servers=primary-kafka:9092
secondary.bootstrap.servers=secondary-kafka:9092

# Replicate topics from primary to secondary
primary->secondary.enabled=true
primary->secondary.topics=orders, payments
```

## Spring Boot + Kafka: Code Examples

### 1. Producer Example (Spring Boot)

**application.yml**:

```yaml
spring:
  kafka:
    bootstrap-servers: broker1:9092,broker2:9092,broker3:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      acks: all          # wait for all in-sync replicas
      retries: 5
      properties:
        enable.idempotence: true
```

**Producer service**:

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        kafkaTemplate.send("orders", orderId, payload)
                     .whenComplete((result, ex) -> {
                         if (ex != null) {
                             // handle failure
                         } else {
                             // success: partition/offset info
                             var metadata = result.getRecordMetadata();
                             System.out.printf("Sent to partition=%d, offset=%d%n",
                                     metadata.partition(), metadata.offset());
                         }
                     });
    }
}
```

**Example input**:

```java
orderProducer.sendOrder("order-123", "{\"id\":\"order-123\",\"amount\":100.0}");
```

**Example output (log)**:

```text
Sent to partition=2, offset=45
```

---

### 2. Consumer Example (Spring Boot)

**application.yml**:

```yaml
spring:
  kafka:
    consumer:
      group-id: order-service
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

**Consumer listener**:

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf("Received key=%s, value=%s, partition=%d, offset=%d%n",
                record.key(), record.value(), record.partition(), record.offset());
        // process order...
    }
}
```

**Example output (log)**:

```text
Received key=order-123, value={"id":"order-123","amount":100.0}, partition=2, offset=45
```

---

### 3. Cross-AZ Considerations in Spring Boot

- Use **multiple bootstrap servers** from different AZs:
  ```yaml
  spring:
    kafka:
      bootstrap-servers: broker-az-a:9092,broker-az-b:9092,broker-az-c:9092
  ```
- Set `acks=all` and `min.insync.replicas` (broker/topic config) to ensure writes are replicated across AZs before acknowledging.

**Broker/topic config example**:

```properties
# server.properties or topic-level override
min.insync.replicas=2
```

With `replication.factor=3` and `min.insync.replicas=2`, a write is only acknowledged if at least 2 replicas (possibly in different AZs) have the data.

---

**In summary**:  
- Use **replication** and **rack awareness** to spread data across AZs.  
- For **cross-AZ** or **cross-region** setups, choose between a single multi-AZ cluster or multiple clusters with replication (MirrorMaker 2, etc.).  
- In Spring Boot, configure producers/consumers to leverage these replication guarantees (`acks`, `min.insync.replicas`, multiple bootstrap servers).
