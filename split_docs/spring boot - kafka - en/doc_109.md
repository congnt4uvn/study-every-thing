**“Processing patterns: Stream processing vs simple consumers”** in Kafka with Spring Boot.

## 1. Simple Consumers

**Idea:**  
Read messages from Kafka topics, process them one-by-one (or in small batches), and optionally write results somewhere (DB, another topic, etc.). Logic is usually stateless or manually manages state.

**When to use:**

- Simple event handling (e.g., send email, update DB).
- Low to moderate throughput.
- No complex event correlations or windowing.
- You don’t need Kafka Streams features (state stores, joins, windowing).

        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
                  StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
                  StringDeserializer.class);
        return props;

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        return new DefaultKafkaConsumerFactory<>(consumerConfigs());

    public ConcurrentKafkaListenerContainerFactory<String, String>
    kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        return factory;
    }
}

## 2. Stream Processing (Kafka Streams)

**Idea:**  
Treat topics as continuous streams of events. You define a topology (pipeline) of transformations: filter, map, aggregate, join, window, etc. Kafka Streams manages state, fault-tolerance, and scaling.

**When to use:**

- Real-time analytics (aggregations, counts, metrics).
- Stateful processing (running totals, sessionization).
- Joining multiple streams or stream-table joins.
- Windowed operations (e.g., “orders per user in last 5 minutes”).

### Spring Boot Example – Kafka Streams

**Dependencies (Gradle):**
```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.kafka:spring-kafka'
    implementation 'org.apache.kafka:kafka-streams:3.7.0'
}
```

**Streams configuration:**
```java
@Configuration
@EnableKafkaStreams
public class KafkaStreamsConfig {

    @Bean(name = KafkaStreamsDefaultConfiguration.DEFAULT_STREAMS_CONFIG_BEAN_NAME)
    public KafkaStreamsConfiguration kStreamsConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "order-analytics-app");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG,
                  Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG,
                  Serdes.String().getClass());
        return new KafkaStreamsConfiguration(props);
    }
}
```

## 3. Key Differences

**Simple Consumers:**

- Pull messages, process, commit offsets.
- Stateless by default; state handled externally (DB, cache).
- Good for:
  - Event-driven microservices.
  - Side effects (emails, notifications, DB updates).
  - Straightforward logic.

**Stream Processing (Kafka Streams):**

- Define a **topology** of transformations.
- Built-in support for:
  - Stateful operations (aggregations, joins).
  - Windowing (time-based operations).
  - Local state stores with changelog topics.
- Good for:
  - Real-time analytics and dashboards.
  - Complex event processing.
  - Continuous transformations from input topics to output topics.

---

## 4. How to Choose

- Use **simple consumers** if:
  - You just need to react to events and perform side effects.
  - Logic is simple and mostly stateless.
  - You don’t need advanced stream features.

- Use **Kafka Streams** if:
  - You need aggregations, joins, or windowing.
  - You want a continuous processing pipeline.
  - You want to keep state in Kafka (state stores + changelog).
