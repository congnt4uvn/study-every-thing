
Here’s a concise summary of the key ideas behind **static membership** and **cooperative rebalancing** in Kafka consumer groups (with Spring Boot examples).

---

## 1. Consumer Group Model (Context)

- A **consumer group** is a set of consumers that share the work of reading from a set of Kafka partitions.
- Kafka assigns partitions to consumers in the group.
- When consumers join/leave or subscriptions change, Kafka performs a **rebalance** to redistribute partitions.

Rebalances are necessary but can be disruptive (pauses in consumption, duplicate processing, etc.).  
**Static membership** and **cooperative rebalancing** are mechanisms to make this smoother and less disruptive.

---

## 2. Static Membership

### Idea

- Normally, each consumer instance gets a **new member ID** when it joins.
- If a consumer restarts, Kafka treats it as a **new** member → triggers a full rebalance.
- **Static membership** lets you assign a **stable, user-defined ID** to each consumer instance.
- When the instance restarts quickly, Kafka recognizes it as the same member and can **avoid or minimize rebalancing**.

### Benefits

- Fewer full rebalances on restarts.
- More stable partition assignments.
- Better for microservices that restart often (deployments, rolling updates).

### Key Config

- `group.instance.id` (Kafka client property): a unique, stable ID per instance within the group.

### Spring Boot / Spring Kafka Example

**application.yml**

```yaml
spring:
  kafka:
    consumer:
      bootstrap-servers: localhost:9092
      group-id: my-consumer-group
      # Static membership: must be unique per instance
      properties:
        group.instance.id: consumer-instance-1
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
```

For another instance of the same service:

```yaml
spring:
  kafka:
    consumer:
      group-id: my-consumer-group
      properties:
        group.instance.id: consumer-instance-2
```

**Listener example**

```java
@KafkaListener(topics = "orders", groupId = "my-consumer-group")
public void listen(String message) {
    System.out.println("Received: " + message);
}
```

**Input/Output behavior**

- Input: Two instances with `group.instance.id` = `consumer-instance-1` and `consumer-instance-2`.
- Output:  
  - Partitions are split between the two instances.  
  - If `consumer-instance-1` restarts quickly, Kafka keeps its partition assignments (or minimizes changes) instead of doing a full rebalance.

---

## 3. Cooperative Rebalancing

### Idea

- Traditional rebalancing is **eager**: all consumers give up their partitions, then reassign from scratch.
- **Cooperative rebalancing** (a.k.a. incremental rebalancing) lets consumers **gradually** give up and acquire partitions.
- Consumers only revoke partitions that need to move, and can keep others.
- This reduces downtime and message processing gaps.

### Benefits

- Less disruptive rebalances.
- Better throughput during membership changes.
- Especially useful in large consumer groups.

### Key Config

- `partition.assignment.strategy` includes a **cooperative** strategy, e.g.:
  - `org.apache.kafka.clients.consumer.CooperativeStickyAssignor`

### Spring Boot / Spring Kafka Example

**application.yml**

```yaml
spring:
  kafka:
    consumer:
      bootstrap-servers: localhost:9092
      group-id: my-consumer-group
      properties:
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

You can combine with static membership:

```yaml
spring:
  kafka:
    consumer:
      group-id: my-consumer-group
      properties:
        group.instance.id: consumer-instance-1
        partition.assignment.strategy: >
          org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

**Listener with rebalance callbacks (optional)**

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
        ConsumerFactory<String, String> consumerFactory) {

    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);

    factory.getContainerProperties().setConsumerRebalanceListener(
        new ConsumerRebalanceListener() {
            @Override
            public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
                System.out.println("Partitions revoked: " + partitions);
            }

            @Override
            public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
                System.out.println("Partitions assigned: " + partitions);
            }
        }
    );

    return factory;
}
```

**Input/Output behavior**

- Input: Add a new consumer instance to the group.
- Output with cooperative rebalancing:
  - Only some partitions are moved to the new instance.
  - Existing consumers keep most of their partitions and continue processing.
  - Rebalance is shorter and less disruptive than eager rebalancing.

---

## 4. How They Work Together

- **Static membership**: stabilizes *who* is in the group (same logical member across restarts).
- **Cooperative rebalancing**: stabilizes *how* partitions move when the group changes.

Using both:

- Fewer rebalances (static membership).
- When rebalances happen, they are incremental and less disruptive (cooperative).
