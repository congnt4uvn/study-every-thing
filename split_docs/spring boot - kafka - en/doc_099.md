## Key Ideas

1. **Schema Registry Role**
   - Stores versions of message schemas (e.g., Avro, Protobuf, JSON Schema).
   - Ensures producers and consumers agree on the structure of Kafka messages.
   - Enforces compatibility rules (BACKWARD, FORWARD, FULL, etc.) when schemas evolve.

2. **Why Schema Evolution Matters**
   - Services change over time: new fields, renamed fields, removed fields.
   - You want to deploy new producers/consumers without breaking existing ones.
   - Schema Registry + compatible schema evolution allows:
     - Old consumers to read messages produced with new schemas.
     - New consumers to read messages produced with old schemas.

3. **Typical Evolution Rules (Avro example)**
   - **Adding a field with a default value** ŌåÆ usually backward compatible.
   - **Removing a field** that consumers still use ŌåÆ not backward compatible.
   - **Changing field type** (e.g., `int` ŌåÆ `string`) ŌåÆ usually incompatible.
   - **Renaming fields** ŌåÆ incompatible unless handled via aliases or custom logic.

4. **Spring Boot + Kafka + Schema Registry**
   - Use Confluent serializers/deserializers (SerDes) with Spring Kafka.
   - Configure:
     - `key.serializer` / `value.serializer` for producers.
     - `key.deserializer` / `value.deserializer` for consumers.
     - `schema.registry.url` to point to Schema Registry.
   - Spring Boot auto-configures Kafka clients; you mainly set properties.

## Basic Setup (Spring Boot + Kafka + Avro + Schema Registry)

**Gradle dependencies (example):**

```groovy
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.kafka:spring-kafka'

    // Confluent Avro SerDes
    implementation 'io.confluent:kafka-avro-serializer:7.6.0'
    implementation 'org.apache.avro:avro:1.11.3'
}
```

**application.yml:**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
      specific.avro.reader: true
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      group-id: my-service-group
      auto-offset-reset: earliest
```

## Example: Initial Schema and Message Flow

**Avro schema v1 (user-value-v1.avsc):**

```json
{
  "type": "record",
  "name": "User",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "email", "type": "string" }
  ]
}
```

Generate Java classes (e.g., via Avro plugin) ŌåÆ `com.example.avro.User`.

**Producer code (Spring service):**

```java
@Service
public class UserProducer {

    private final KafkaTemplate<String, User> kafkaTemplate;

    public UserProducer(KafkaTemplate<String, User> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUser(String topic, String id, String email) {
        User user = User.newBuilder()
                .setId(id)
                .setEmail(email)
                .build();

        kafkaTemplate.send(topic, id, user);
    }
}
```

**Consumer code:**

```java
@Component
public class UserConsumer {

    @KafkaListener(topics = "users", groupId = "my-service-group")
    public void listen(User user) {
        System.out.println("Received user: " + user);
    }
}
```

**Example Input (producer call):**

```java
userProducer.sendUser("users", "u-123", "user@example.com");
```

**Example Output (consumer log):**

```text
Received user: {"id": "u-123", "email": "user@example.com"}
```

Schema Registry will register this schema under a subject like `users-value` (depending on configuration).

## Schema Evolution Example

### Step 1: Add a New Field (Backward Compatible)

**New schema v2 (user-value-v2.avsc):**

```json
{
  "type": "record",
  "name": "User",
  "namespace": "com.example.avro",
  "fields": [
    { "name": "id", "type": "string" },
    { "name": "email", "type": "string" },
    { "name": "phone", "type": ["null", "string"], "default": null }
  ]
}
```

- Added `phone` as optional (`["null", "string"]`) with default `null`.
- With **BACKWARD** compatibility, old consumers (v1) can still read messages produced with v2.

**Updated producer:**

```java
public void sendUser(String topic, String id, String email, String phone) {
    User user = User.newBuilder()
            .setId(id)
            .setEmail(email)
            .setPhone(phone) // new field
            .build();

    kafkaTemplate.send(topic, id, user);
}
```

**Input (new producer):**

```java
userProducer.sendUser("users", "u-456", "new@example.com", "+1-555-1234");
```

**Outputs:**

- **Old consumer (v1 schema)**: sees only `id` and `email`, ignores `phone`.
  ```text
  Received user: {"id": "u-456", "email": "new@example.com"}
  ```
- **New consumer (v2 schema)**: sees `id`, `email`, `phone`.
  ```text
  Received user: {"id": "u-456", "email": "new@example.com", "phone": "+1-555-1234"}
  ```

### Step 2: Removing or Changing Fields (Potentially Incompatible)

- Removing `email` or changing its type from `string` to `int` will likely break compatibility.
- Schema Registry will reject the new schema if compatibility is enforced (e.g., BACKWARD).

## Configuring Compatibility in Schema Registry

Using Confluent CLI (example):

```bash
# Set global compatibility to BACKWARD
curl -X PUT \
  -H "Content-Type: application/json" \
  --data '{"compatibility": "BACKWARD"}' \
  http://localhost:8081/config
```

Or per subject:

```bash
curl -X PUT \
  -H "Content-Type: application/json" \
  --data '{"compatibility": "BACKWARD"}' \
  http://localhost:8081/config/users-value
```

If you try to register an incompatible schema, youŌĆÖll get an error from the serializer at runtime or from the Schema Registry API.

## Summary

- Use **Schema Registry** with Spring Kafka to manage message schemas and their evolution.
- Configure **Confluent Avro SerDes** in Spring Boot to automatically register and fetch schemas.
- Follow **compatibility rules** (e.g., add optional fields with defaults) to evolve schemas safely.
- Old and new Spring services can coexist as long as schema evolution respects the configured compatibility mode.
