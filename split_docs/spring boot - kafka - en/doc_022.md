
Here’s a concise summary of Kafka partitioning strategies in a Spring Boot context, with code examples.

---

## 1. Default Partitioner

**Idea:**  
Kafka’s default partitioner decides which partition a message goes to based on the **key**:

- If a key is provided → hash(key) % number_of_partitions  
- If no key is provided → round-robin across partitions

This ensures:
- Same key → always goes to the same partition (ordering per key)
- Load is spread across partitions

### Spring Boot Producer Configuration (using default partitioner)

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        config.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        config.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        // No custom partitioner set → default partitioner is used
        return new DefaultKafkaProducerFactory<>(config);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

### Sending Messages (default partitioner)

```java
@Service
public class DefaultPartitionerProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public DefaultPartitionerProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    // With key → partition determined by hash(key)
    public void sendWithKey() {
        kafkaTemplate.send("orders-topic", "user-123", "Order created");
    }

    // Without key → round-robin across partitions
    public void sendWithoutKey() {
        kafkaTemplate.send("orders-topic", "Order created without key");
    }
}
```

**Example behavior (assuming 3 partitions: 0,1,2):**

- Input: key = `"user-123"`, value = `"Order created"`  
  - Output: goes to partition = `hash("user-123") % 3` (e.g., partition 1)
- Input: no key, value = `"Order created without key"`  
  - Output: messages distributed round-robin: partition 0 → 1 → 2 → 0 → …

## 2. Custom Partitioner

**Idea:**  
You define your own logic to choose the partition. Useful when you need:

- Domain-based routing (e.g., VIP users to specific partitions)
- Region-based partitioning
- Special load balancing rules

### Step 1: Implement a Custom Partitioner

```java
public class VipUserPartitioner implements Partitioner {

    @Override
    public void configure(Map<String, ?> configs) {
        // Optional: read custom configs
    }

    @Override
    public int partition(
            String topic,
            Object key,
            byte[] keyBytes,
            Object value,
            byte[] valueBytes,
            Cluster cluster) {

        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();

        // Example: VIP users go to partition 0, others distributed by hash
        if (key != null && key.toString().startsWith("VIP-")) {
            return 0; // always partition 0 for VIP users
        }

        // Fallback: hash-based
        if (keyBytes == null) {
            // no key → simple round-robin-like fallback
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }

        return Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
    }

    @Override
    public void close() {
        // cleanup if needed
    }
}
```

### Step 2: Configure Spring Boot to Use the Custom Partitioner

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        config.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        config.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        // Use custom partitioner
        config.put(ProducerConfig.PARTITIONER_CLASS_CONFIG, VipUserPartitioner.class.getName());

        return new DefaultKafkaProducerFactory<>(config);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

### Step 3: Sending Messages (custom partitioner in action)

```java
@Service
public class CustomPartitionerProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public CustomPartitionerProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessages() {
        // VIP user
        kafkaTemplate.send("users-topic", "VIP-001", "VIP user event");

        // Normal user
        kafkaTemplate.send("users-topic", "USER-123", "Normal user event");
    }
}
```

**Example behavior (assuming 3 partitions: 0,1,2):**

- Input: key = `"VIP-001"`, value = `"VIP user event"`  
  - Output: partition = `0` (by custom rule)
- Input: key = `"USER-123"`, value = `"Normal user event"`  
  - Output: partition = `hash("USER-123") % 3` (e.g., partition 2)

---

## Key Takeaways

- **Default partitioner**:
  - Simple, automatic.
  - Same key → same partition → ordered per key.
  - No key → round-robin.

- **Custom partitioner**:
  - You control partition selection logic.
  - Good for domain-specific routing (VIP, region, tenant, etc.).
  - Configured via `ProducerConfig.PARTITIONER_CLASS_CONFIG` in Spring Boot.
