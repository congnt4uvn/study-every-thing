
Here’s a concise summary and some practical patterns around **Spring Kafka transactions**, `@Transactional`, and **exactly-once processing**.

---

## 1. Core Ideas

1. **Kafka transactions** allow producing messages atomically:
   - Either all records in a transaction are written and visible, or none are.
   - Prevents partial writes when errors occur.

2. **Spring Kafka + Spring Transactions**:
   - You can use `@Transactional` to wrap:
     - Kafka consumption
     - Database operations (e.g., JPA)
     - Kafka production
   - With proper configuration, this can give you **exactly-once processing semantics** (E2E: consume → process → DB → produce).

3. **Exactly-once processing pattern**:
   - Use a **transactional Kafka producer** and **idempotent consumer logic** or **transactional DB writes**.
   - Common pattern:
     - Read from Kafka
     - Process and write to DB in a transaction
     - Produce result/event to another Kafka topic in the same transaction
   - If anything fails, the whole transaction rolls back.

## 2. Basic Configuration for Kafka Transactions

```java
// application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      transaction-id-prefix: tx-  # enables transactional producer
      properties:
        enable.idempotence: true
    consumer:
      group-id: my-group
      enable-auto-commit: false
    listener:
      ack-mode: RECORD  # or BATCH, MANUAL, etc.
```

```java
// Producer factory with transactions
@Bean
public ProducerFactory<String, String> producerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
    props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
    props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
    props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
    props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "tx-producer-1");
    return new DefaultKafkaProducerFactory<>(props);
}

@Bean
public KafkaTemplate<String, String> kafkaTemplate() {
    return new KafkaTemplate<>(producerFactory());
}
```

## 3. Using `@Transactional` with Kafka + DB

### Example: Consume → Process → Save to DB → Produce

```java
@Service
public class OrderService {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private final OrderRepository orderRepository;

    public OrderService(KafkaTemplate<String, String> kafkaTemplate,
                        OrderRepository orderRepository) {
        this.kafkaTemplate = kafkaTemplate;
        this.orderRepository = orderRepository;
    }

    @Transactional  // Spring transaction (DB + Kafka)
    @KafkaListener(topics = "orders", groupId = "order-service")
    public void handleOrder(String message) {
        // 1. Parse input
        OrderRequest request = parse(message);

        // 2. Save to DB
        Order order = new Order();
        order.setCustomerId(request.getCustomerId());
        order.setAmount(request.getAmount());
        orderRepository.save(order);

        // 3. Produce event to another topic
        String event = "OrderCreated:" + order.getId();
        kafkaTemplate.send("order-events", event);

        // If any exception is thrown here, DB save and Kafka send are rolled back
    }

    private OrderRequest parse(String message) {
        // simple example
        String[] parts = message.split(",");
        OrderRequest req = new OrderRequest();
        req.setCustomerId(parts[0]);
        req.setAmount(Double.parseDouble(parts[1]));
        return req;
    }
}
```

### Example Input / Output

- **Input Kafka message** on topic `orders`:
  - Key: `null`
  - Value: `"cust-123,99.99"`

- **DB result** (if success):
  - `orders` table:
    - `id = 1`
    - `customer_id = "cust-123"`
    - `amount = 99.99`

- **Output Kafka message** on topic `order-events`:
  - Key: `null`
  - Value: `"OrderCreated:1"`

If an exception occurs after DB save but before commit, both:
- DB insert
- Kafka `order-events` message  
are rolled back.

## 4. Kafka Transaction Manager (Optional Explicit Use)

You can explicitly use a `KafkaTransactionManager` and a chained transaction manager with DB:

```java
@Bean
public KafkaTransactionManager<String, String> kafkaTransactionManager(
        ProducerFactory<String, String> producerFactory) {
    return new KafkaTransactionManager<>(producerFactory);
}

@Bean
public PlatformTransactionManager chainedTxManager(
        KafkaTransactionManager<String, String> ktm,
        JpaTransactionManager jpaTxManager) {
    return new ChainedKafkaTransactionManager<>(ktm, jpaTxManager);
}
```

Then:

```java
@Service
public class PaymentService {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private final PaymentRepository paymentRepository;

    public PaymentService(KafkaTemplate<String, String> kafkaTemplate,
                          PaymentRepository paymentRepository) {
        this.kafkaTemplate = kafkaTemplate;
        this.paymentRepository = paymentRepository;
    }

    @Transactional("chainedTxManager")
    @KafkaListener(topics = "payments", groupId = "payment-service")
    public void handlePayment(String message) {
        // parse, save to DB, send Kafka, all in one chained transaction
    }
}
```

## 5. Exactly-Once Semantics (EOS) Notes

- **Producer side**:
  - `enable.idempotence=true`
  - `transactional.id` set
- **Consumer side**:
  - `enable.auto.commit=false`
  - Use Spring’s container-managed commits with transactions.
- **Pattern**:
  - Consume records
  - Process + DB write + produce new records in one transaction
  - Commit offsets only when the transaction commits  
  → Each input message is processed **once**, and side effects are not duplicated.
