
Here’s a concise summary of the key ideas around **Spring Kafka consumer concurrency** and **partition assignment**, with code examples.

---

## 1. Concurrency in Spring Kafka

In Spring Kafka, you can run multiple consumer threads for a single `@KafkaListener` by using the `concurrency` attribute. This is **thread-level parallelism** within the same consumer group.

### Key points

- `concurrency` = number of consumer threads created for that listener.
- Each thread is a separate Kafka consumer instance in the **same consumer group**.
- Kafka partitions are distributed across these consumer instances.
- **Upper bound**: Effective concurrency is limited by the number of partitions in the topic.
  - If topic has 3 partitions and `concurrency = 5`, only 3 threads will actually receive data (2 will be idle).

### Example: Basic concurrent listener

```java
@EnableKafka
@Configuration
public class KafkaConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);
        // Default concurrency can be set here if you want
        factory.setConcurrency(3);
        return factory;
    }
}
```

```java
@Component
public class MyKafkaConsumer {

    // Override factory-level concurrency here
    @KafkaListener(
        topics = "my-topic",
        groupId = "my-group",
        concurrency = "3"
    )
    public void listen(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                + " | Received: " + message);
    }
}
```

**Input**: Messages on topic `my-topic` with 3 partitions.  
**Output (example logs)**:

```text
Thread: my-topic-0-C-1 | Received: msg-1
Thread: my-topic-1-C-1 | Received: msg-2
Thread: my-topic-2-C-1 | Received: msg-3
Thread: my-topic-0-C-1 | Received: msg-4
...
```

Each thread handles messages from one or more partitions.

---

## 2. Partition Assignment

Kafka assigns partitions to consumer instances in a consumer group. With Spring Kafka:

- Each consumer thread (from `concurrency`) is a **member of the same group**.
- Kafka’s **partition assignment strategy** (e.g., range, round-robin, cooperative-sticky) decides which thread gets which partition.
- **One partition is consumed by at most one consumer in a group** at a time.
- If you increase/decrease `concurrency` or add/remove instances of the application, Kafka will **rebalance** partition assignments.

### Example: Topic with 4 partitions

- Topic: `my-topic` with partitions: `0, 1, 2, 3`
- `concurrency = 2`

Possible assignment:

- Thread 1 → partitions 0, 1  
- Thread 2 → partitions 2, 3  

If you change `concurrency = 4`, then:

- Thread 1 → partition 0  
- Thread 2 → partition 1  
- Thread 3 → partition 2  
- Thread 4 → partition 3  

---

## 3. Concurrency vs Partitions

Rules of thumb:

- **Max parallelism per consumer group = number of partitions**.
- Setting `concurrency` higher than partition count doesn’t increase throughput; extra threads will be idle.
- For **strict ordering per key**, ensure that:
  - Messages with the same key go to the same partition (Kafka default).
  - You don’t process a single partition in multiple threads (Spring Kafka respects this: one partition → one consumer thread).

## 4. Example with Partition Info in Listener

You can access partition and offset to see how assignments work:

```java
@KafkaListener(
    topics = "my-topic",
    groupId = "my-group",
    concurrency = "3"
)
public void listen(
        String message,
        @Header(KafkaHeaders.RECEIVED_PARTITION_ID) int partition,
        @Header(KafkaHeaders.OFFSET) long offset) {

    System.out.printf("Thread=%s | Partition=%d | Offset=%d | Message=%s%n",
            Thread.currentThread().getName(), partition, offset, message);
}
```

**Input**:  
- `my-topic` has 3 partitions.  
- Messages: `msg-1` to `msg-6`.

**Possible output**:

```text
Thread=my-topic-0-C-1 | Partition=0 | Offset=0 | Message=msg-1
Thread=my-topic-1-C-1 | Partition=1 | Offset=0 | Message=msg-2
Thread=my-topic-2-C-1 | Partition=2 | Offset=0 | Message=msg-3
Thread=my-topic-0-C-1 | Partition=0 | Offset=1 | Message=msg-4
Thread=my-topic-1-C-1 | Partition=1 | Offset=1 | Message=msg-5
Thread=my-topic-2-C-1 | Partition=2 | Offset=1 | Message=msg-6
```

You can see each thread is tied to specific partitions.

---

## 5. When to Use Concurrency

Use `concurrency` when:

- You want **parallel processing** of messages from different partitions.
- You have enough partitions to support the desired level of parallelism.
- You’re okay with **per-partition ordering** (not global ordering).

Avoid overusing `concurrency` when:

- Topic has few partitions.
- You need strict global ordering across all messages.
- You’re constrained by external resources (DB, downstream services) rather than CPU.
