**“Topics, Partitions, Ordering: Impact of partition count on throughput, latency, and consumer scaling”**

---

## 1. Topics & Partitions: Core Ideas

- **Topic**: A named stream of messages (e.g., `orders`, `payments`).
- **Partition**: A topic is split into multiple partitions. Each partition is:
  - An **append-only log**.
  - **Ordered** within itself.
  - Processed by **only one consumer in a consumer group** at a time.

So:  
**Ordering is guaranteed per partition, not per topic.**

---

## 2. Impact of Partition Count

### a) Throughput

- **More partitions → higher potential throughput**:
  - Producers can write in parallel to multiple partitions.
  - Consumers can read in parallel from multiple partitions.
- But:
  - Too many partitions can increase **broker overhead** (more files, more metadata).
  - There is a practical limit where adding partitions no longer helps and may hurt performance.

**Rule of thumb**: Increase partitions to match or slightly exceed the number of consumer instances you expect to run.

---

### b) Latency

- **More partitions can reduce latency**:
  - Work is spread across more consumers → each consumer has less data to process.
- But:
  - Too many partitions can increase **coordination overhead** (rebalance time, metadata updates).
  - This can increase **end-to-end latency** during rebalances or under heavy load.

---

### c) Consumer Scaling

- **Max parallelism = number of partitions**:
  - In a consumer group, **each partition is consumed by at most one consumer instance**.
  - If you have 3 partitions and 5 consumers in the same group:
    - Only 3 consumers will be active; 2 will be idle.
- To scale consumers horizontally:
  - You must have **at least as many partitions as consumer instances** you want to run in parallel.

---

### d) Ordering Guarantees

- **Within a partition**: Kafka guarantees message order.
- **Across partitions**: No global ordering.
- If you need ordering per key (e.g., per `userId` or `orderId`):
  - Use a **partition key** so all messages for that key go to the same partition.

## 3. Spring Boot + Kafka: Code Examples

### 3.1 Producer Example (Spring Boot)

**Producer configuration & template**

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        config.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        config.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        return new DefaultKafkaProducerFactory<>(config);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

**Sending messages with a key (for ordering per key)**

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        // orderId is the key → all messages for the same orderId go to the same partition
        kafkaTemplate.send("orders-topic", orderId, payload);
    }
}
```

**Example input → output behavior**

- Input calls:

```java
sendOrder("order-1", "created");
sendOrder("order-1", "paid");
sendOrder("order-1", "shipped");
```

- Output behavior:
  - All three messages go to the **same partition** (because key = `"order-1"`).
  - A single consumer instance will see them in this order:
    1. `created`
    2. `paid`
    3. `shipped`

Ordering is preserved **for that key**.

### 3.2 Consumer Example (Spring Boot)

```java
@EnableKafka
@Configuration
public class KafkaConsumerConfig {

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        Map<String, Object> config = new HashMap<>();
        config.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        config.put(ConsumerConfig.GROUP_ID_CONFIG, "order-service-group");
        config.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        config.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        return new DefaultKafkaConsumerFactory<>(config);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        // concurrency should not exceed partition count
        factory.setConcurrency(3);
        return factory;
    }
}
```

**Listener**

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders-topic", groupId = "order-service-group")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf(
            "Partition=%d, Key=%s, Value=%s, Offset=%d%n",
            record.partition(),
            record.key(),
            record.value(),
            record.offset()
        );
    }
}
```

**Example behavior with partitions and concurrency**

- Suppose `orders-topic` has **3 partitions**.
- `factory.setConcurrency(3)` → up to 3 consumer threads.
- Kafka will assign:
  - 1 partition per thread (in a balanced way).
- If you increase concurrency to 5:
  - Still only 3 active threads (one per partition); 2 will be idle.

## 4. Creating a Topic with Partitions

Using Kafka CLI:

```bash
kafka-topics.sh \
  --create \
  --bootstrap-server localhost:9092 \
  --topic orders-topic \
  --partitions 6 \
  --replication-factor 3
```

- `--partitions 6`: up to 6 consumers in the same group can process in parallel.
- Ordering is guaranteed **per partition**, not across all 6.

---

## 5. Practical Guidelines

1. **Need more throughput / parallelism?**  
   Increase partition count and consumer instances (up to partition count).

2. **Need strict ordering per key (e.g., per user/order)?**  
   Use a **keyed producer** so all events for that key go to the same partition.

3. **Don’t overshoot partitions**:  
   Too many partitions can increase broker and rebalance overhead, hurting latency.

4. **Consumer scaling rule**:  
   `max_parallel_consumers_in_group ≈ number_of_partitions`.

If you share your expected traffic (messages/sec) and number of consumers, I can suggest a concrete partition count and Spring Boot configuration.
