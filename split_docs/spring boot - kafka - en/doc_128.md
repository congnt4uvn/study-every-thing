**“Capacity planning: Impact of message size, compression, and retention on hardware”**

## 1. Key Concepts

### Message Size
- **Larger messages**:
  - Increase **network bandwidth** usage.
  - Increase **disk I/O** and **storage** requirements.
  - Increase **broker memory** usage (page cache, heap for batching, etc.).
  - Can **reduce throughput** and increase **latency**.
- **Smaller messages**:
  - Better throughput, but more **per-message overhead** (more records, more metadata).

**Implication for hardware**:
- Large messages → need **faster disks** (NVMe/SSD), **more disk capacity**, and **more network bandwidth**.
- Consider **max.message.bytes** and **fetch.max.bytes** to avoid OOM or timeouts.

---

### Compression
- Kafka supports **gzip, snappy, lz4, zstd**.
- **Compression reduces disk and network usage**, but costs **CPU**.
- Higher compression ratio (e.g., gzip, zstd) → **more CPU**, less disk/network.
- Lower compression (e.g., lz4, snappy) → **less CPU**, more disk/network.

**Implication for hardware**:
- If CPU is cheap and disk/network is expensive → use **stronger compression**.
- If CPU is constrained → use **lighter compression** or none.

---

### Retention
- Retention is how long Kafka keeps data:
  - **time-based**: `retention.ms`
  - **size-based**: `retention.bytes`
- Longer retention → more **disk capacity** needed.
- Shorter retention → less disk, but less historical data for replays/analytics.

**Implication for hardware**:
- Long retention + high throughput + large messages → **very large disks** or more brokers.
- Plan disk capacity using:
  - `required_disk ≈ throughput (bytes/sec) × retention_time (sec) × replication_factor`

---

## 2. Capacity Planning – Putting It Together

When planning hardware, consider:

1. **Message size** (avg and max)
2. **Throughput** (messages/sec or MB/sec per topic)
3. **Compression type** and expected compression ratio
4. **Retention** (days or total bytes)
5. **Replication factor** (multiplies storage and network)

Example rough calculation:

- Avg message size: **10 KB**
- Throughput: **10,000 messages/sec**
- Compression ratio: **2:1** (so effective 5 KB/message on disk)
- Retention: **7 days**
- Replication factor: **3**

Disk per topic (approx):

```text
raw_rate = 10,000 msg/s * 10 KB = 100,000 KB/s ≈ 100 MB/s
compressed_rate = 100 MB/s / 2 = 50 MB/s
per_replica_per_day = 50 MB/s * 86,400 s ≈ 4,320,000 MB ≈ 4.32 TB
per_replica_for_7_days ≈ 4.32 TB * 7 ≈ 30.24 TB
cluster_total (RF=3) ≈ 30.24 TB * 3 ≈ 90.72 TB
```

So you’d need ~30 TB per replica (spread across brokers) for that topic.

## 3. Spring Boot + Kafka: Configuration Examples

### Producer configuration (message size & compression)

`application.yml`:

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      properties:
        # Compression: gzip, snappy, lz4, zstd
        compression.type: lz4
        # Control batch size (bytes) – affects memory and throughput
        batch.size: 65536
        # Max request size (protects broker and network)
        max.request.size: 1048576  # 1 MB
```

### Consumer configuration (fetch size, memory impact)

```yaml
spring:
  kafka:
    consumer:
      group-id: example-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        # Max bytes per fetch – affects memory and latency
        fetch.max.bytes: 5242880  # 5 MB
        max.partition.fetch.bytes: 1048576  # 1 MB per partition
```

## 4. Topic-Level Settings (Retention & Size)

You can set retention and max message size per topic.

### Using `kafka-topics.sh`

```bash
# Create topic with retention and size limits
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --partitions 6 \
  --replication-factor 3 \
  --config retention.ms=604800000 \        # 7 days
  --config retention.bytes=107374182400 \  # 100 GB per partition
  --config max.message.bytes=1048576      # 1 MB
```

### Using AdminClient in Spring Boot (Java)

```java
@Bean
public NewTopic ordersTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("retention.ms", "604800000");       // 7 days
    configs.put("retention.bytes", "107374182400"); // 100 GB
    configs.put("max.message.bytes", "1048576");    // 1 MB

    return TopicBuilder.name("orders")
            .partitions(6)
            .replicas(3)
            .configs(configs)
            .build();
}
```

## 5. Simple Producer/Consumer Example (with I/O)

### Producer (Spring Boot)

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        kafkaTemplate.send("orders", orderId, payload)
                     .whenComplete((result, ex) -> {
                         if (ex == null) {
                             System.out.println("Sent: key=" + orderId +
                                     ", partition=" + result.getRecordMetadata().partition() +
                                     ", offset=" + result.getRecordMetadata().offset());
                         } else {
                             ex.printStackTrace();
                         }
                     });
    }
}
```

**Example input**:

```java
orderProducer.sendOrder("order-123", "{\"id\":\"order-123\",\"amount\":99.99}");
```

**Example output (log)**:

```text
Sent: key=order-123, partition=2, offset=45
```

---

### Consumer (Spring Boot)

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf(
            "Received: key=%s, value=%s, partition=%d, offset=%d%n",
            record.key(), record.value(), record.partition(), record.offset()
        );
    }
}
```

**Example output (log)**:

```text
Received: key=order-123, value={"id":"order-123","amount":99.99}, partition=2, offset=45
```

## 6. Practical Guidelines

- **Message size**:
  - Prefer **< 1 MB**; if larger, consider splitting or using external storage (S3, etc.).
- **Compression**:
  - Start with **lz4** or **snappy**; monitor CPU and disk usage.
- **Retention**:
  - Use **time-based** retention for most use cases.
  - Use **size-based** as a safety cap to avoid filling disks.
- **Hardware**:
  - For high throughput + long retention:
    - Use **SSD/NVMe**.
    - Ensure **enough disk** for retention × replication.
    - Ensure **network** can handle peak compressed throughput.
    - Monitor **CPU** if using heavy compression.
