
Key idea:  
Kafka producers can compress message batches before sending them to brokers using `compression.type`. This affects throughput, latency, CPU usage, and network/storage usage.

## `compression.type` options

- `none` (default if not set)
  - No compression.
  - Lowest CPU usage.
  - Highest network and disk usage.
  - Good for very low-latency, low-throughput scenarios.

- `gzip`
  - High compression ratio (smallest message size).
  - Highest CPU cost and latency.
  - Good when network bandwidth is expensive and throughput is moderate.

- `snappy`
  - Balanced: decent compression, low CPU cost.
  - Often good for general-purpose, high-throughput workloads.

- `lz4`
  - Very fast compression/decompression.
  - Good compression ratio.
  - Often better than snappy in both speed and ratio.
  - Common choice for high-throughput, low-latency systems.

- `zstd`
  - Newer algorithm with very good compression ratio and tunable levels.
  - Can achieve better ratio than gzip with similar or lower CPU, depending on level.
  - Good for large-scale systems where both bandwidth and storage matter.

## Performance trade-offs

- Higher compression ratio → less network and disk usage, but more CPU and potentially higher latency.
- Faster algorithms (snappy, lz4) → better throughput and lower latency, but slightly larger messages than gzip/zstd.
- Choice depends on:
  - Network bandwidth constraints
  - Broker disk usage
  - CPU availability on producers/brokers
  - Latency requirements

## Spring Boot / Spring Kafka example

### Maven dependency (if not already present)

```xml
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>
```

### Producer configuration (Java)

```java
import org.apache.kafka.clients.producer.ProducerConfig;
import org.apache.kafka.common.serialization.StringSerializer;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.DefaultKafkaProducerFactory;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.core.ProducerFactory;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        // Set compression type: gzip, snappy, lz4, zstd, or none
        props.put(ProducerConfig.COMPRESSION_TYPE_CONFIG, "lz4");

        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

### Sending a message (Producer)

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class MyProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public MyProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value);
    }
}
```

**Input example:**

```java
myProducer.sendMessage("orders", "order-123", "{\"id\":123,\"amount\":99.99}");
```

**What happens:**

- The producer batches messages for topic `orders`.
- The batch is compressed using `lz4` (as configured).
- The compressed batch is sent to the Kafka broker.
- On the consumer side, Kafka automatically decompresses the batch; consumers see the original message:

**Consumer output (value):**

```json
{"id":123,"amount":99.99}
```

## When to choose which

- `none`: Very low latency, low volume, CPU-constrained.
- `gzip`: Maximize compression when bandwidth is expensive and throughput is moderate.
- `snappy`: Good general-purpose choice for high throughput.
- `lz4`: Often best default for high throughput + low latency.
- `zstd`: When you want strong compression and can tune levels; good for large-scale systems.
