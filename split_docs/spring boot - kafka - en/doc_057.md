
Here’s a concise summary of the topic “Replication: Unclean leader election, data loss scenarios” in Kafka, with Spring Boot–style examples.

---

## Key Ideas

1. **Replication in Kafka**
   - Each topic partition has:
     - **Leader**: handles all reads/writes.
     - **Followers**: replicate data from the leader.
   - Replication factor (e.g., 3) determines how many copies of data exist.

2. **In-Sync Replicas (ISR)**
   - ISR = set of replicas that are fully caught up with the leader.
   - Only ISR members are considered “safe” candidates for leader election.

3. **Unclean Leader Election**
   - Happens when **no ISR** is available, and Kafka allows a **non-ISR replica** to become leader.
   - This can **prevent downtime** but may cause **data loss**.
   - Controlled by broker config:
     ```properties
     unclean.leader.election.enable=true  # allow unclean election (risk data loss)
     unclean.leader.election.enable=false # disallow unclean election (prefer availability loss)
     ```

4. **Data Loss Scenarios**
   - **Scenario 1: Leader has unreplicated data, then dies**
     - Producer writes messages to leader.
     - Followers are lagging (not in ISR).
     - Leader crashes.
     - If unclean leader election is **enabled**, a stale follower becomes leader:
       - New leader does not have the latest messages → **data loss**.
   - **Scenario 2: Producer uses acks=0 or acks=1**
     - `acks=0`: producer doesn’t wait for any acknowledgment → messages may be lost if leader fails immediately.
     - `acks=1`: waits only for leader → if leader fails before followers replicate, data can be lost.
   - **Scenario 3: Min in-sync replicas not enforced**
     - If `min.insync.replicas` is low and some replicas are down, Kafka may still accept writes with fewer replicas → higher risk of loss on failure.

5. **How to Reduce Data Loss**
   - Disable unclean leader election:
     ```properties
     unclean.leader.election.enable=false
     ```
   - Use **acks=all** and set **min.insync.replicas**:
     ```properties
     min.insync.replicas=2
     ```
   - Ensure enough brokers and stable network so ISR stays healthy.

## Spring Boot / Spring Kafka Example

### Producer Configuration (Safe Settings)

```java
import org.apache.kafka.clients.producer.ProducerConfig;
import org.apache.kafka.common.serialization.StringSerializer;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.DefaultKafkaProducerFactory;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.core.ProducerFactory;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        // Safer settings
        props.put(ProducerConfig.ACKS_CONFIG, "all");        // wait for all ISR replicas
        props.put(ProducerConfig.RETRIES_CONFIG, 5);
        props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true); // avoid duplicates

        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

### Producer Usage Example (Input/Output)

**Input (application code):**

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        kafkaTemplate.send("orders", orderId, payload)
                     .whenComplete((result, ex) -> {
                         if (ex != null) {
                             System.err.println("Failed to send: " + ex.getMessage());
                         } else {
                             System.out.println("Sent to partition " +
                                 result.getRecordMetadata().partition() +
                                 " offset " + result.getRecordMetadata().offset());
                         }
                     });
    }
}
```

**Example call:**

```java
orderProducer.sendOrder("order-123", "{ \"item\": \"book\", \"qty\": 1 }");
```

**Possible output (logs):**

- Success:
  ```
  Sent to partition 0 offset 42
  ```
- Failure (e.g., not enough in-sync replicas):
  ```
  Failed to send: NotEnoughReplicasException: ...
  ```

## Broker-Side Topic Configuration Example

Create a topic with safer replication settings:

```bash
kafka-topics.sh \
  --create \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --partitions 3 \
  --replication-factor 3 \
  --config min.insync.replicas=2
```

Broker config (server.properties):

```properties
unclean.leader.election.enable=false
```

This combination:
- Requires at least 2 in-sync replicas for writes.
- Disallows unclean leader election.
- Reduces data loss risk at the cost of possible temporary unavailability.
