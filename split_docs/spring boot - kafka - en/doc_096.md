
Here’s a concise summary of Schema Registry compatibility modes, with examples.

---

## 1. What is Compatibility?

Compatibility defines whether a **new schema** can safely replace an **old schema** without breaking producers or consumers that use older/newer versions.

---

## 2. Compatibility Modes

### BACKWARD
- New schema must be readable by **consumers using the previous schema**.
- Old consumers can read data produced with the new schema.
- Typical for **evolving producers** while keeping old consumers working.

**Example (Avro)**

Old schema (v1):
```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "int" },
    { "name": "name", "type": "string" }
  ]
}
```

New schema (v2) – BACKWARD compatible:
```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "int" },
    { "name": "name", "type": "string" },
    { "name": "email", "type": ["null", "string"], "default": null }
  ]
}
```

- Old consumers (v1) ignore `email` → still work.

---

### FORWARD
- New schema must be readable by **consumers using the new schema** when data was produced with the **old schema**.
- New consumers can read old data.
- Typical for **evolving consumers** first.

New schema (v2) – FORWARD compatible:
```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "int" },
    { "name": "name", "type": "string", "default": "" }
  ]
}
```

If old schema had fewer fields, new fields must have **defaults** so new consumers can read old messages.

---

### FULL
- New schema must be both:
  - BACKWARD compatible
  - FORWARD compatible
- Ensures **old and new producers/consumers** can all interoperate.

Example FULL-compatible change:
- Add a field with:
  - type: union with `"null"` (e.g. `["null", "string"]`)
  - `default: null`

```json
{
  "type": "record",
  "name": "User",
  "fields": [
    { "name": "id", "type": "int" },
    { "name": "name", "type": "string" },
    { "name": "email", "type": ["null", "string"], "default": null }
  ]
}
```

---

### NONE
- No compatibility checks.
- Any schema change is allowed.
- Risky: can easily break consumers/producers.

---

## 3. Setting Compatibility (Schema Registry REST API)

**Set global compatibility:**
```bash
curl -X PUT \
  -H "Content-Type: application/json" \
  --data '{"compatibility": "BACKWARD"}' \
  http://localhost:8081/config
```

**Set compatibility for a specific subject:**
```bash
curl -X PUT \
  -H "Content-Type: application/json" \
  --data '{"compatibility": "FULL"}' \
  http://localhost:8081/config/my-topic-value
```

**Get current compatibility:**
```bash
curl http://localhost:8081/config
curl http://localhost:8081/config/my-topic-value
```

## 4. Spring Boot + Kafka + Schema Registry Example

**Producer configuration (application.yml):**
```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      schema.registry.url: http://localhost:8081
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
```

**Consumer configuration:**
```yaml
spring:
  kafka:
    consumer:
      group-id: user-service
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
      properties:
        specific.avro.reader: true
```

**Producer code (Java):**
```java
@Service
public class UserProducer {

    private final KafkaTemplate<String, User> kafkaTemplate;

    public UserProducer(KafkaTemplate<String, User> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendUser(User user) {
        kafkaTemplate.send("user-topic", user.getId().toString(), user);
    }
}
```

**Consumer code (Java):**
```java
@Service
public class UserConsumer {

    @KafkaListener(topics = "user-topic", groupId = "user-service")
    public void listen(User user) {
        System.out.println("Received user: " + user);
    }
}
```

---

## 5. Example Input/Output Flow

1. **Input (Producer app):**
   ```java
   userProducer.sendUser(new User(1, "Alice", "alice@example.com"));
   ```

2. **Kafka message (conceptual):**
   - Key: `"1"`
   - Value: Avro-encoded `User` with schema ID from Schema Registry.

3. **Output (Consumer log):**
   ```
   Received user: User{id=1, name='Alice', email='alice@example.com'}
   ```

Compatibility mode (e.g., BACKWARD or FULL) determines whether future schema changes will be accepted by Schema Registry when you evolve the `User` schema.
