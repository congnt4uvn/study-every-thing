
**Key idea**

In Kafka 2.4+, the **sticky partitioner** is the default partitioner for the Java producer.  
Its goal: **improve batching efficiency** by sending multiple records in a row to the same partition, which increases throughput and reduces overhead.

---

## What is the Sticky Partitioner?

- For records **without a key**:
  - The producer picks a **random partition** for the first record.
  - It then **keeps using that same partition** for subsequent records until:
    - The current batch is full, or
    - A certain time/condition triggers a new batch.
  - Then it picks another partition and repeats.
- For records **with a key**:
  - Behavior is unchanged: partition is chosen by **hash(key) % numPartitions** (so same key → same partition).

**Benefit:**  
By sending many key-less records to the same partition consecutively, the producer can create **larger batches**, which:
- Reduces network overhead
- Increases throughput
- Improves overall performance

---

## Basic Producer Example (Spring Boot + Kafka)

**Producer configuration (application.yml):**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      # Sticky partitioner is default in Kafka 2.4+; explicit for clarity:
      properties:
        partitioner.class: org.apache.kafka.clients.producer.internals.DefaultPartitioner
      batch-size: 16384        # bytes
      linger-ms: 5             # wait up to 5ms to form a batch
```

**Producer code (no key → sticky partitioner in action):**

```java
@Service
public class MessageProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;
    private static final String TOPIC = "demo-topic";

    public MessageProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessagesWithoutKey() {
        for (int i = 0; i < 10; i++) {
            String value = "message-" + i;
            kafkaTemplate.send(TOPIC, value); // no key
        }
    }

    public void sendMessagesWithKey() {
        for (int i = 0; i < 10; i++) {
            String key = "user-1";           // same key
            String value = "event-" + i;
            kafkaTemplate.send(TOPIC, key, value);
        }
    }
}
```

### Expected behavior (conceptual)

Assume `demo-topic` has 3 partitions: `0, 1, 2`.

**Case 1: No key (sticky partitioner)**

Input (producer calls):

```text
send("demo-topic", "message-0")
send("demo-topic", "message-1")
...
send("demo-topic", "message-9")
```

Possible partition assignment (example):

```text
Partition 1: message-0, message-1, message-2, message-3, message-4
Partition 0: message-5, message-6, message-7, message-8, message-9
Partition 2: (none in this small example)
```

- Messages are **grouped** into partitions in chunks, improving batching.

**Case 2: With key (hash-based)**

Input:

```text
send("demo-topic", "user-1", "event-0")
send("demo-topic", "user-1", "event-1")
...
send("demo-topic", "user-1", "event-9")
```

Partition assignment (example):

```text
Partition 2: event-0, event-1, ..., event-9  (all with key "user-1")
```

- All messages with the same key go to the **same partition** (ordering guarantee per key).

## Simple Consumer Example (Spring Boot)

```java
@Service
public class MessageConsumer {

    @KafkaListener(topics = "demo-topic", groupId = "demo-group")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf(
            "Consumed from partition=%d, offset=%d, key=%s, value=%s%n",
            record.partition(),
            record.offset(),
            record.key(),
            record.value()
        );
    }
}
```

**Sample output (illustrative):**

```text
Consumed from partition=1, offset=0, key=null, value=message-0
Consumed from partition=1, offset=1, key=null, value=message-1
Consumed from partition=1, offset=2, key=null, value=message-2
Consumed from partition=1, offset=3, key=null, value=message-3
Consumed from partition=1, offset=4, key=null, value=message-4
Consumed from partition=0, offset=0, key=null, value=message-5
Consumed from partition=0, offset=1, key=null, value=message-6
...
```

You can see how messages are **clustered per partition**, which reflects the sticky partitioner’s batching behavior.

---

## Summary

- **Sticky partitioner** (Kafka 2.4+ default) optimizes **batching** for records without keys.
- It sends multiple consecutive records to the **same partition** before switching.
- Records **with keys** still use **hash-based partitioning** for key-based ordering.
- Result: better throughput and more efficient use of network and broker resources.
