## Key Ideas: Cloud & Managed Kafka – Observability and Limits

1. **Managed Kafka = Less Operational Burden, More Abstractions**  
   - Cloud/managed Kafka services (e.g., Confluent Cloud, AWS MSK, Azure Event Hubs for Kafka, etc.) remove the need to manage brokers, ZooKeeper/Kraft, OS, and hardware.  
   - In exchange, you lose some low‑level control and visibility you’d have with self‑managed Kafka.

2. **Observability Differences vs. Self‑Managed Kafka**
   - **Metrics**:  
     - You typically get a curated set of metrics (through UI, APIs, or Prometheus endpoints).  
     - You may not see all broker‑level or OS‑level metrics (e.g., detailed disk I/O, page cache behavior).
   - **Logs**:  
     - Access to broker logs is often limited or abstracted away.  
     - You rely more on platform dashboards and alerts instead of tailing broker logs directly.
   - **Tracing**:  
     - End‑to‑end tracing (producer → Kafka → consumer) is still your responsibility, usually via OpenTelemetry or similar.  
     - The managed provider rarely gives full distributed tracing out of the box.

3. **Common Limits in Managed Offerings**
   - **Resource limits**:  
     - Max throughput (MB/s), max partitions per cluster, max connections, message size limits.  
   - **Configuration limits**:  
     - Restricted broker configs (e.g., log.retention.*, num.partitions, compression settings) may be fixed or only partially configurable.  
   - **Multi‑tenancy constraints**:  
     - No direct access to underlying machines, disks, or network.  
     - Some noisy‑neighbor protections and quotas are enforced.

4. **Implications for Application Design**
   - You must design producers/consumers to respect quotas and limits (e.g., backoff, batching, compression).  
   - You rely heavily on **client‑side metrics** and **application logs** for troubleshooting.  
   - Capacity planning is done via provider sizing options (e.g., “Basic/Standard/Premium” tiers, or specific throughput/partition SKUs).

5. **Spring Boot + Kafka: How to Work with Managed Kafka**
   - Use Spring Kafka with the managed Kafka bootstrap servers and security settings.  
   - Expose and monitor **client metrics** (producer/consumer) via Micrometer/Prometheus.  
   - Implement retries, backoff, and error handling to cope with throttling or transient errors.

## Example: Spring Boot Producer/Consumer with Managed Kafka

### Maven Dependencies

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>
    <dependency>
        <groupId>io.micrometer</groupId>
        <artifactId>micrometer-registry-prometheus</artifactId>
    </dependency>
</dependencies>
```

### `application.yml` (Example for a Managed Kafka Cluster)

```yaml
spring:
  kafka:
    bootstrap-servers: my-managed-kafka:9092
    properties:
      security.protocol: SASL_SSL
      sasl.mechanism: PLAIN
      sasl.jaas.config: >
        org.apache.kafka.common.security.plain.PlainLoginModule required
        username="my-user"
        password="my-password";
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      retries: 5
      properties:
        linger.ms: 10
        batch.size: 16384
    consumer:
      group-id: demo-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

### Producer Example

```java
@Service
public class DemoProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public DemoProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value)
                     .whenComplete((result, ex) -> {
                         if (ex != null) {
                             // log and handle error (e.g., throttling, quota exceeded)
                             System.err.println("Send failed: " + ex.getMessage());
                         } else {
                             System.out.println("Sent to partition " +
                                 result.getRecordMetadata().partition() +
                                 " offset " + result.getRecordMetadata().offset());
                         }
                     });
    }
}
```

**Input example:**

```java
demoProducer.sendMessage("orders", "order-123", "{\"id\":123,\"status\":\"NEW\"}");
```

**Output (log):**

```text
Sent to partition 1 offset 42
```

### Consumer Example

```java
@Service
public class DemoConsumer {

    @KafkaListener(topics = "orders", groupId = "demo-group")
    public void listen(String message) {
        // Process message
        System.out.println("Received: " + message);
    }
}
```

**Input (message in Kafka topic `orders`):**

```json
{"id":123,"status":"NEW"}
```

**Output (log):**

```text
Received: {"id":123,"status":"NEW"}
```

## Observability with Spring Boot (Client‑Side)

Expose metrics to Prometheus (for example):

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
  metrics:
    tags:
      application: kafka-demo
```

Then you can scrape `/actuator/prometheus` and monitor:

- `kafka_producer_record_send_total`
- `kafka_consumer_records_consumed_total`
- Latency, error counts, etc.

These client metrics are crucial in managed Kafka, where broker internals are partially hidden.
