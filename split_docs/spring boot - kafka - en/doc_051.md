
Here’s a concise summary and some practical examples around **consumer reliability** in Kafka with **idempotent consumers**, **deduplication**, and **transactional reads** using Spring Boot.

---

## Key Ideas

1. **Idempotent Consumers**
   - An idempotent consumer can process the same message multiple times but produce the same final result as if it were processed once.
   - This is crucial when producers or brokers may cause duplicate messages (e.g., retries, failover).

2. **Deduplication**
   - The consumer keeps track of which messages have already been processed (e.g., by message key, unique ID, or offset).
   - Before processing, it checks if the message was already handled; if yes, it skips it.
   - Common storage: database table, cache (Redis), or in-memory (for small/simple cases).

3. **Transactional Reads**
   - Combine **reading from Kafka** and **writing to a database** (or another Kafka topic) in a single atomic operation.
   - With Kafka transactions and Spring Kafka, you can ensure:
     - If processing fails, offsets are not committed and DB changes are rolled back.
     - If processing succeeds, both DB changes and offset commits are committed together.

---

## Example 1: Idempotent Consumer with Deduplication (Spring Boot + Spring Kafka)

### Assumptions
- Each message has a unique `eventId`.
- We store processed `eventId`s in a database table `processed_events`.

### Entity for Deduplication

```java
@Entity
@Table(name = "processed_events")
public class ProcessedEvent {

    @Id
    private String eventId;

    private Instant processedAt;

    // getters/setters
}
```

### Repository

```java
public interface ProcessedEventRepository extends JpaRepository<ProcessedEvent, String> {
}
```

### Kafka Listener (Idempotent Consumer)

```java
@Service
public class OrderEventConsumer {

    private final ProcessedEventRepository processedEventRepository;
    private final OrderService orderService;

    public OrderEventConsumer(ProcessedEventRepository processedEventRepository,
                              OrderService orderService) {
        this.processedEventRepository = processedEventRepository;
        this.orderService = orderService;
    }

    @KafkaListener(topics = "orders", groupId = "order-service")
    @Transactional  // DB transaction
    public void consumeOrderEvent(String messageJson) {
        // 1. Parse message
        OrderEvent event = parse(messageJson);

        // 2. Deduplication check
        boolean alreadyProcessed = processedEventRepository.existsById(event.getEventId());
        if (alreadyProcessed) {
            // Duplicate message -> skip processing
            return;
        }

        // 3. Business logic
        orderService.processOrder(event);

        // 4. Mark as processed
        ProcessedEvent processed = new ProcessedEvent();
        processed.setEventId(event.getEventId());
        processed.setProcessedAt(Instant.now());
        processedEventRepository.save(processed);
    }

    private OrderEvent parse(String json) {
        // use ObjectMapper or similar
        // ...
        return null;
    }
}
```

### Example Input / Output

**Kafka message (input):**

```json
{
  "eventId": "order-123",
  "orderId": 42,
  "status": "CREATED",
  "amount": 100.50
}
```

**First time processing:**
- `processed_events` does not contain `order-123`.
- `orderService.processOrder(...)` is executed.
- Row inserted into `processed_events` with `eventId = "order-123"`.

**Second time (duplicate message):**
- `processed_events` already has `order-123`.
- Consumer returns early; no business logic executed.
- Final state is the same as if the message was processed once.

## Example 2: Transactional Reads (Kafka + DB in One Transaction)

Goal: Ensure that **DB update** and **offset commit** are atomic.

### Configuration (Spring Kafka)

```java
@Configuration
@EnableKafka
public class KafkaConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
        props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "order-tx-producer-1");
        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTransactionManager<String, String> kafkaTransactionManager(
            ProducerFactory<String, String> producerFactory) {
        return new KafkaTransactionManager<>(producerFactory);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory,
            KafkaTransactionManager<String, String> kafkaTransactionManager) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);

        // Enable Kafka transactions for the listener
        factory.getContainerProperties().setTransactionManager(kafkaTransactionManager);

        // Use record-level acknowledgment within the transaction
        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.RECORD);

        return factory;
    }
}
```

### Transactional Listener

```java
@Service
public class PaymentConsumer {

    private final PaymentService paymentService;

    public PaymentConsumer(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

    @KafkaListener(topics = "payments", containerFactory = "kafkaListenerContainerFactory")
    @Transactional  // participates in Kafka + DB transaction
    public void consumePayment(String messageJson) {
        PaymentEvent event = parse(messageJson);

        // 1. Update DB (e.g., mark invoice as paid)
        paymentService.applyPayment(event);

        // 2. Optionally produce another Kafka message within same transaction
        // kafkaTemplate.send("payment-confirmations", event.getInvoiceId(), "PAID");

        // If an exception is thrown here:
        // - DB changes are rolled back
        // - Kafka offsets are not committed
    }

    private PaymentEvent parse(String json) {
        // ...
        return null;
    }
}
```

### Behavior (Input/Output)

**Input message:**

```json
{
  "paymentId": "pay-777",
  "invoiceId": "inv-1001",
  "amount": 250.00
}
```

- If `paymentService.applyPayment` succeeds:
  - DB row for `inv-1001` is updated (e.g., status = PAID).
  - Kafka offset for this message is committed.
- If an exception occurs:
  - DB update is rolled back.
  - Offset is not committed; message will be re-consumed later.

---

## Summary

- **Idempotent consumers** ensure that duplicate messages do not cause incorrect state changes.
- **Deduplication** is typically implemented by storing processed message IDs and checking before processing.
- **Transactional reads** (Kafka + DB) ensure that message consumption and side effects (DB updates, new messages) are atomic, improving reliability in distributed systems.
