
Here’s a concise summary and some practical examples around:

**“Data contracts: Versioning topics vs versioning schemas”**  
(in the context of Kafka + Spring Boot)

---

## Core Idea

When evolving messages in Kafka, you have two main strategies:

1. **Versioning topics**  
   - Create a new topic when you change the message structure.
   - Example: `orders.v1`, `orders.v2`.

2. **Versioning schemas (data contracts)**  
   - Keep the same topic, but evolve the schema in a compatible way.
   - Use schema registry (e.g., Confluent Schema Registry) and schema versions.
   - Example: Topic `orders` with schema v1, v2, v3…

The “data contract” is the agreed structure of messages between producers and consumers. The main question is:  
**Do you express versioning in the topic name or in the schema definition?**

---

## Versioning Topics

**Pros:**
- Very explicit: each topic has a fixed schema.
- Easy to reason about: `orders.v1` always has schema v1.
- Safe for breaking changes: old consumers stay on old topic.

**Cons:**
- Topic explosion: many topics over time.
- Operational overhead: ACLs, monitoring, retention per version.
- Consumers may need to subscribe to multiple topics if they want all versions.

**When to use:**
- You need **breaking changes** that cannot be made backward compatible.
- You want **hard isolation** between versions.
- Different teams or services own different versions.

### Example: Topic Versioning

**Topics:**
- `orders.v1`
- `orders.v2`

**Schema v1 (JSON):**
```json
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.5
}
```

**Schema v2 (JSON):**
```json
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.5,
  "currency": "USD"
}
```

**Spring Boot Producer for `orders.v2`:**

```java
// build.gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.kafka:spring-kafka'
    implementation 'com.fasterxml.jackson.core:jackson-databind'
}
```

```java
// OrderV2.java
public class OrderV2 {
    private String orderId;
    private String customerId;
    private double totalAmount;
    private String currency;

    // getters/setters
}
```

```java
// KafkaConfig.java
@Configuration
@EnableKafka
public class KafkaConfig {

    @Bean
    public ProducerFactory<String, OrderV2> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
                  org.springframework.kafka.support.serializer.JsonSerializer.class);
        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, OrderV2> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

```java
// ProducerService.java
@Service
public class ProducerService {

    private final KafkaTemplate<String, OrderV2> kafkaTemplate;

    public ProducerService(KafkaTemplate<String, OrderV2> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(OrderV2 order) {
        kafkaTemplate.send("orders.v2", order.getOrderId(), order);
    }
}
```

**Example Input (Java call):**
```java
OrderV2 order = new OrderV2();
order.setOrderId("123");
order.setCustomerId("C-1");
order.setTotalAmount(100.5);
order.setCurrency("USD");

producerService.sendOrder(order);
```

**Example Output (message on `orders.v2` topic, JSON):**
```json
{
  "orderId": "123",
  "customerId": "C-1",
  "totalAmount": 100.5,
  "currency": "USD"
}
```

## Versioning Schemas (Same Topic, Evolving Contract)

**Pros:**
- Fewer topics, simpler operations.
- Schema registry can enforce compatibility (backward/forward).
- Consumers can gradually upgrade.

**Cons:**
- Requires careful schema evolution rules.
- Old consumers must tolerate new fields (or ignore them).
- Harder to handle truly breaking changes.

**When to use:**
- You can evolve schemas **backward/forward compatibly**.
- You want a **stable topic name** as an API.
- You use Avro/Protobuf/JSON Schema with a registry.

### Example: Schema Versioning with Avro

**Topic:** `orders`

**Avro schema v1:**
```avro
{
  "type": "record",
  "name": "Order",
  "namespace": "com.example",
  "fields": [
    {"name": "orderId", "type": "string"},
    {"name": "customerId", "type": "string"},
    {"name": "totalAmount", "type": "double"}
  ]
}
```

**Avro schema v2 (backward compatible: new field with default):**
```avro
{
  "type": "record",
  "name": "Order",
  "namespace": "com.example",
  "fields": [
    {"name": "orderId", "type": "string"},
    {"name": "customerId", "type": "string"},
    {"name": "totalAmount", "type": "double"},
    {"name": "currency", "type": "string", "default": "USD"}
  ]
}
```

**Spring Boot Producer with Schema Registry (Confluent):**

```java
// build.gradle
dependencies {
    implementation 'org.springframework.boot:spring-boot-starter'
    implementation 'org.springframework.kafka:spring-kafka'
    implementation 'io.confluent:kafka-avro-serializer:7.6.0' // version example
}
```

```java
// KafkaConfig.java
@Configuration
public class KafkaConfig {

    @Bean
    public ProducerFactory<String, com.example.Order> avroProducerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put("schema.registry.url", "http://localhost:8081");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
                  io.confluent.kafka.serializers.KafkaAvroSerializer.class);
        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, com.example.Order> avroKafkaTemplate() {
        return new KafkaTemplate<>(avroProducerFactory());
    }
}
```

```java
// ProducerService.java
@Service
public class ProducerService {

    private final KafkaTemplate<String, com.example.Order> kafkaTemplate;

    public ProducerService(KafkaTemplate<String, com.example.Order> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(com.example.Order order) {
        kafkaTemplate.send("orders", order.getOrderId(), order);
    }
}
```

**Example Input (Java call using v2 schema):**
```java
Order order = Order.newBuilder()
    .setOrderId("123")
    .setCustomerId("C-1")
    .setTotalAmount(100.5)
    .setCurrency("EUR") // new field in v2
    .build();

producerService.sendOrder(order);
```

**Example Output (logical message on `orders` topic):**
- Schema registry stores this as **schema version 2** for subject `orders-value`.
- Old consumers using v1 schema will see `currency` as `"USD"` (default) if configured for backward compatibility.

---

## How to Choose

**Prefer schema versioning (same topic) when:**
- You can maintain **backward/forward compatibility**.
- You want a **stable topic as an API**.
- You have a **schema registry** and governance.

**Prefer topic versioning when:**
- You need **breaking changes** (remove fields, change types, semantics).
- You want **clear separation** between old and new consumers.
- You can handle **more topics** operationally.

Often, teams use **both**:
- Minor, compatible changes → new schema version on same topic.
- Major, breaking changes → new topic (`orders.v2`).
