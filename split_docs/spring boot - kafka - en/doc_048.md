
Here’s a concise summary of the key ideas around Kafka offset management in Spring Boot, focusing on:

- Manual commit
- Sync vs async commit
- Commit per record vs per batch  

---

## 1. Manual Offset Commit in Spring Kafka

To manage offsets manually, you typically:

- Disable auto-commit
- Use `AckMode` that allows manual control
- Call `Acknowledgment.acknowledge()` or use `KafkaTemplate`/`KafkaConsumer` commit APIs

Example configuration (Spring Boot, `ConcurrentKafkaListenerContainerFactory`):

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
        ConsumerFactory<String, String> consumerFactory) {

    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);

    // Disable auto-commit in consumer properties:
    // spring.kafka.consumer.enable-auto-commit=false

    // Use manual ack mode
    factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);

    return factory;
}
```

Listener with manual acknowledgment:

```java
@KafkaListener(topics = "my-topic", groupId = "my-group")
public void listen(String message, Acknowledgment ack) {
    // Process message
    System.out.println("Received: " + message);

    // Manually commit offset for this record
    ack.acknowledge();
}
```

**Input (Kafka message):**
```text
Topic: my-topic
Partition: 0
Offset: 15
Value: "order-123"
```

**Output (application log):**
```text
Received: order-123
# After processing, offset 15 is committed manually
```

## 2. Sync vs Async Commit (Low-level Kafka Consumer)

If you use the raw Kafka client (`KafkaConsumer`) instead of Spring’s listener abstraction, you can commit offsets manually in two ways:

### 2.1 Synchronous Commit

- Blocks until the broker confirms the commit
- Safer (you know the commit succeeded)
- Higher latency

```java
try (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {
    consumer.subscribe(Collections.singletonList("my-topic"));

    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf("Consumed: key=%s value=%s offset=%d%n",
                    record.key(), record.value(), record.offset());
            // process(record);
        }

        // Commit offsets synchronously for all processed records
        consumer.commitSync();
    }
}
```

**Behavior:**  
- If `commitSync()` throws an exception, you know the commit failed and can retry or handle it.

### 2.2 Asynchronous Commit

- Non-blocking; returns immediately
- Better throughput
- You may need a callback to handle failures

```java
consumer.commitAsync((offsets, exception) -> {
    if (exception != null) {
        System.err.println("Commit failed for offsets " + offsets);
        exception.printStackTrace();
    } else {
        System.out.println("Commit succeeded for offsets " + offsets);
    }
});
```

**Behavior:**  
- If commit fails, you see it in the callback, but your main loop doesn’t block.

## 3. Commit per Record vs Commit per Batch

### 3.1 Commit per Record

- Commit after each message is processed
- Maximum safety (minimal reprocessing on failure)
- Lower throughput (many small commits)

Spring Kafka example:

```java
@KafkaListener(topics = "my-topic", groupId = "my-group")
public void listenPerRecord(String message, Acknowledgment ack) {
    // Process single record
    System.out.println("Processing: " + message);

    // Commit offset for this single record
    ack.acknowledge();
}
```

**Pros:**
- If the app crashes, at most one record is reprocessed.

**Cons:**
- High commit overhead under high load.

---

### 3.2 Commit per Batch

- Process a batch of records, then commit once
- Better performance
- More records may be reprocessed on failure

Spring Kafka batch listener:

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> batchFactory(
        ConsumerFactory<String, String> consumerFactory) {

    ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);
    factory.setBatchListener(true);
    factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);

    return factory;
}

@KafkaListener(
    topics = "my-topic",
    groupId = "my-group",
    containerFactory = "batchFactory"
)
public void listenBatch(List<String> messages, Acknowledgment ack) {
    for (String msg : messages) {
        System.out.println("Processing: " + msg);
        // process(msg);
    }

    // Commit offsets for the whole batch
    ack.acknowledge();
}
```

**Input (batch from Kafka):**
```text
Offsets: 100, 101, 102
Values: "msg-100", "msg-101", "msg-102"
```

**Output (application log):**
```text
Processing: msg-100
Processing: msg-101
Processing: msg-102
# After all processed, offsets up to 102 are committed
```

**Pros:**
- Fewer commits → better throughput.

**Cons:**
- If the app crashes after processing but before commit, the whole batch may be reprocessed.

---

## 4. Summary of Trade-offs

- **Manual commit**: You control exactly when offsets are marked as processed.
- **Sync commit**: Reliable, but slower (waits for broker).
- **Async commit**: Faster, but commit failures must be handled via callback.
- **Per record commit**: Highest safety, lowest performance.
- **Per batch commit**: Higher performance, more potential reprocessing on failure.

Choose the strategy based on your requirements for:
- At-least-once vs at-most-once semantics
- Throughput vs safety
- Acceptable level of duplicate processing.
