
Here’s a concise summary and some practical examples around **reliability patterns in Kafka with Spring Boot**, focusing on **idempotent consumers** and a **deduplication store**.

---

## 1. Idempotent Consumers – Core Idea

**Goal:** Ensure that if the same message is processed multiple times (due to retries, rebalances, etc.), the **business effect happens only once**.

**Key points:**
- Consumers may receive the same message more than once.
- Make your processing logic **idempotent**: applying it multiple times yields the same result as applying it once.
- Common strategies:
  - Use a **unique business key** (e.g., orderId, transactionId).
  - Check if the operation has already been applied before performing it.
  - Use a **deduplication store** (DB, cache, etc.) to track processed messages.

---

## 2. Deduplication Store – Core Idea

**Goal:** Detect and skip duplicate messages.

**Key points:**
- Store a **message ID** (or business key) in a persistent store:
  - Relational DB (e.g., PostgreSQL)
  - NoSQL (e.g., Redis)
- Before processing:
  1. Check if the ID exists in the store.
  2. If it exists → **skip** processing.
  3. If not → process and then **insert** the ID.

---

## 3. Example: Spring Boot Kafka Consumer with Deduplication (JPA + DB)

### 3.1. Entity for Deduplication

```java
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.Table;

@Entity
@Table(name = "processed_messages")
public class ProcessedMessage {

    @Id
    private String messageId;

    protected ProcessedMessage() {}

    public ProcessedMessage(String messageId) {
        this.messageId = messageId;
    }

    public String getMessageId() {
        return messageId;
    }
}
```

### 3.2. Repository

```java
import org.springframework.data.jpa.repository.JpaRepository;

public interface ProcessedMessageRepository extends JpaRepository<ProcessedMessage, String> {
}
```

### 3.3. Kafka Consumer with Idempotent Logic

Assume each message has a unique `eventId`.

```java
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class OrderEventConsumer {

    private final ProcessedMessageRepository processedMessageRepository;
    private final OrderService orderService;

    public OrderEventConsumer(ProcessedMessageRepository processedMessageRepository,
                              OrderService orderService) {
        this.processedMessageRepository = processedMessageRepository;
        this.orderService = orderService;
    }

    @KafkaListener(topics = "order-events", groupId = "order-service")
    @Transactional
    public void consume(OrderEvent event) {
        String eventId = event.getEventId();

        // 1. Check deduplication store
        boolean alreadyProcessed = processedMessageRepository.existsById(eventId);
        if (alreadyProcessed) {
            // Duplicate message → skip
            return;
        }

        // 2. Business logic (idempotent effect)
        orderService.applyOrderEvent(event);

        // 3. Mark as processed
        processedMessageRepository.save(new ProcessedMessage(eventId));
    }
}
```

### 3.4. Example Input / Output

**Kafka message (JSON):**
```json
{
  "eventId": "evt-123",
  "orderId": "order-999",
  "type": "CREATE",
  "amount": 100.0
}
```

**First time consumption:**
- `existsById("evt-123")` → false  
- `applyOrderEvent` is executed → order created  
- `processed_messages` table now contains `"evt-123"`  

**Second time (duplicate) consumption:**
- `existsById("evt-123")` → true  
- Consumer returns immediately → **no duplicate order** created

## 4. Example: Deduplication with Redis (Alternative)

```java
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Service;

import java.time.Duration;

@Service
public class PaymentEventConsumer {

    private final StringRedisTemplate redisTemplate;
    private final PaymentService paymentService;

    public PaymentEventConsumer(StringRedisTemplate redisTemplate,
                                PaymentService paymentService) {
        this.redisTemplate = redisTemplate;
        this.paymentService = paymentService;
    }

    @KafkaListener(topics = "payment-events", groupId = "payment-service")
    public void consume(PaymentEvent event) {
        String eventId = event.getEventId();
        String key = "processed:" + eventId;

        // SETNX pattern: set if not exists
        Boolean success = redisTemplate.opsForValue()
                .setIfAbsent(key, "1", Duration.ofHours(24));

        if (Boolean.FALSE.equals(success)) {
            // Key already exists → duplicate
            return;
        }

        // Process payment (idempotent effect)
        paymentService.processPayment(event);
    }
}
```

---

## 5. Summary of Main Ideas

- **Idempotent consumers** are essential in Kafka because messages can be delivered more than once.
- Use a **deduplication store** (DB, Redis, etc.) keyed by a unique message ID or business key.
- Pattern:
  1. Check if message ID is already processed.
  2. If not, process and record it.
  3. If yes, skip to avoid duplicate side effects.
- Combine with **transactional** processing (DB + Kafka) for stronger guarantees when needed.
