
Key idea: The `acks` setting in a Kafka producer controls how many broker acknowledgments are required before a send is considered successful. It directly affects durability, latency, and throughput.

### `acks` options and trade-offs

1. **acks=0**
   - Producer does **not wait** for any acknowledgment.
   - **Pros**: Lowest latency, highest throughput.
   - **Cons**: Messages can be lost if the broker is down or fails right after receiving data.
   - Use when: You can tolerate data loss and need maximum speed (e.g., non-critical logs).

2. **acks=1**
   - Producer waits for **leader broker** to acknowledge.
   - **Pros**: Balanced: decent durability and performance.
   - **Cons**: If the leader crashes before followers replicate, data can be lost.
   - Use when: Moderate durability is acceptable.

3. **acks=all** (or `-1`)
   - Producer waits until **all in-sync replicas (ISR)** acknowledge.
   - **Pros**: Highest durability; message is stored on multiple brokers.
   - **Cons**: Higher latency, lower throughput; depends on slowest ISR.
   - Use when: Data loss is unacceptable (financial transactions, critical events).

### Basic Kafka Producer config (Java)

```java
Properties props = new Properties();
props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG,
          "org.apache.kafka.common.serialization.StringSerializer");
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG,
          "org.apache.kafka.common.serialization.StringSerializer");

// Change this to 0, 1, or "all"
props.put(ProducerConfig.ACKS_CONFIG, "all");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);

ProducerRecord<String, String> record =
        new ProducerRecord<>("my-topic", "key1", "hello world");

producer.send(record, (metadata, exception) -> {
    if (exception == null) {
        System.out.println("Sent to partition " + metadata.partition()
                + " offset " + metadata.offset());
    } else {
        exception.printStackTrace();
    }
});

producer.flush();
producer.close();
```

**Example behavior (conceptual):**

- Input: send `"hello world"` with `acks=0`
  - Output: `send()` returns immediately; no guarantee message reached broker.
- Input: send `"hello world"` with `acks=1`
  - Output: callback prints partition/offset once leader writes it.
- Input: send `"hello world"` with `acks=all`
  - Output: callback fires only after all in-sync replicas confirm write.

### Spring Boot + Spring Kafka example

**application.yml**

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      acks: all   # 0, 1, or all
```

**Producer service**

```java
@Service
public class MyKafkaProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public MyKafkaProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public ListenableFuture<SendResult<String, String>> sendMessage(String topic, String key, String value) {
        return kafkaTemplate.send(topic, key, value);
    }
}
```

**Example call and output**

```java
ListenableFuture<SendResult<String, String>> future =
        myKafkaProducer.sendMessage("my-topic", "key1", "hello world");

future.addCallback(
    result -> {
        RecordMetadata m = result.getRecordMetadata();
        System.out.println("Sent to topic=" + m.topic()
                + ", partition=" + m.partition()
                + ", offset=" + m.offset());
    },
    ex -> System.err.println("Send failed: " + ex.getMessage())
);
```

### Summary

- `acks=0`: fastest, lowest durability.
- `acks=1`: middle ground; leader-only durability.
- `acks=all`: slowest, highest durability (all in-sync replicas).
- Choose `acks` based on how much data loss you can tolerate vs. how much latency/throughput you need.
