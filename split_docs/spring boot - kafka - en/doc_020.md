
Here’s a concise summary of how **partitioning** in Kafka affects **ordering guarantees** and **key affinity**, with Spring Boot code examples.

---

## 1. Ordering Guarantees

**Key idea:**  
Kafka guarantees **message order only within a single partition**.  
There is **no global ordering** across partitions in a topic.

- If all messages for a given key go to the **same partition**, their order is preserved.
- If messages for the same logical entity are spread across multiple partitions, you **lose ordering guarantees** for that entity.

### Example: Producer with ordering per key

```java
// build.gradle (or pom.xml) should include spring-kafka

// application.yml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer

// Producer service
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderEvents(String orderId, List<String> events) {
        for (String event : events) {
            // Using orderId as key ensures all events for this order
            // go to the same partition -> ordering guaranteed per orderId
            kafkaTemplate.send("orders-topic", orderId, event);
        }
    }
}
```

**Input example:**

```java
orderProducer.sendOrderEvents("ORDER-123", List.of(
    "CREATED",
    "PAID",
    "SHIPPED"
));
```

**Kafka behavior:**

- All three messages have key `"ORDER-123"`.
- They are routed to the **same partition** (e.g., partition 2).
- Consumer reading partition 2 will see: `CREATED -> PAID -> SHIPPED` in order.

## 2. Key Affinity

**Key idea:**  
**Key affinity** means that all messages with the same key are consistently sent to the **same partition**. This is crucial for:

- Per-entity ordering (e.g., per user, per order).
- Local state in stream processing (e.g., Kafka Streams, state stores).
- Load distribution based on keys.

Kafka uses a **partitioner** (by default, a hash of the key) to map keys to partitions:

```text
partition = hash(key) % number_of_partitions
```

So:

- Same key → same partition → key affinity.
- Different keys → likely different partitions → parallelism.

### Example: Custom partitioner (optional)

If you want custom key affinity logic:

```java
public class CustomKeyPartitioner implements Partitioner {

    @Override
    public int partition(String topic,
                         Object key,
                         byte[] keyBytes,
                         Object value,
                         byte[] valueBytes,
                         Cluster cluster) {

        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();

        String keyStr = (String) key;

        // Example: route VIP users to partition 0, others by hash
        if (keyStr.startsWith("VIP-")) {
            return 0;
        }

        return Math.abs(keyStr.hashCode()) % numPartitions;
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}
```

Configure in Spring Boot:

```yaml
spring:
  kafka:
    producer:
      properties:
        partitioner.class: com.example.kafka.CustomKeyPartitioner
```

## 3. Consumer Side: Ordering per Partition

Consumers read messages **in order per partition**. If you have multiple consumer instances in the same group:

- Each partition is assigned to **only one consumer instance**.
- Parallelism = number of partitions (max number of active consumers in a group).

### Example: Spring Kafka listener

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders-topic", groupId = "order-service")
    public void listen(ConsumerRecord<String, String> record) {
        String key = record.key();      // e.g., "ORDER-123"
        String value = record.value();  // e.g., "PAID"
        int partition = record.partition();

        System.out.printf("Partition=%d, Key=%s, Value=%s%n",
                partition, key, value);
    }
}
```

**Sample output:**

```text
Partition=2, Key=ORDER-123, Value=CREATED
Partition=2, Key=ORDER-123, Value=PAID
Partition=2, Key=ORDER-123, Value=SHIPPED
```

Order is preserved for `"ORDER-123"` because all its messages are in partition 2.

---

## 4. Design Implications

- To **preserve ordering** for an entity (user, order, account), always use a **stable key** for that entity.
- To **scale** consumers, increase the **number of partitions**; each partition can be processed in parallel by a different consumer instance.
- Do **not** rely on ordering across different keys or across the whole topic.

---

**In summary:**

- **Ordering guarantee**: only within a partition.
- **Key affinity**: same key → same partition → per-key ordering and localized processing.
- Use keys intentionally in your Spring Boot Kafka producers to control both ordering and distribution.
