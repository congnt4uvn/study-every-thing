
Here’s a concise summary of how **Kafka partitioning** affects **parallelism** and **throughput**, with Spring Boot code examples.

---

## Key Ideas

1. **Partition = Unit of Parallelism**
   - A Kafka topic is split into **partitions**.
   - Each partition can be consumed by **only one consumer in a consumer group** at a time.
   - More partitions ⇒ more consumers can work in parallel ⇒ higher **parallelism**.

2. **Throughput Scales with Partitions**
   - Producers can send data to multiple partitions in parallel.
   - Brokers store and serve partitions independently.
   - More partitions (up to a point) ⇒ higher **write and read throughput**.

3. **Consumer Group Behavior**
   - If you have **more consumers than partitions**, some consumers will be idle.
   - If you have **fewer consumers than partitions**, some consumers will handle multiple partitions.

4. **Ordering Guarantees**
   - Kafka guarantees **message order only within a partition**, not across the whole topic.
   - If you need ordering per key (e.g., per user), you must send all messages for that key to the **same partition**.

5. **Trade-offs**
   - Too few partitions: low parallelism, limited throughput.
   - Too many partitions: more overhead (metadata, open files, rebalancing time).
   - Choose partition count based on expected **throughput** and **number of consumers**.

---

## Simple Example: Topic with 3 Partitions

- Topic: `orders`
- Partitions: `orders-0`, `orders-1`, `orders-2`
- Consumer group: `order-service-group`
- Up to **3 consumers** in that group can process messages in parallel (one per partition).

---

## Spring Boot: Producer Example

**Producer config (application.yml):**
```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

**Producer code (partitioning by key):**
```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        // orderId is the key -> all messages for same orderId go to same partition
        kafkaTemplate.send("orders", orderId, payload);
    }
}
```

**Example input (method call):**
```java
orderProducer.sendOrder("order-123", "{ \"status\": \"CREATED\" }");
orderProducer.sendOrder("order-123", "{ \"status\": \"PAID\" }");
orderProducer.sendOrder("order-999", "{ \"status\": \"CREATED\" }");
```

**Effect on partitions (conceptual):**
- `"order-123"` → always hashed to the same partition (e.g., `orders-1`)
- `"order-999"` → may go to another partition (e.g., `orders-0`)

So:
- All events for `order-123` are ordered within `orders-1`.
- `order-123` and `order-999` can be processed in parallel (different partitions).

## Spring Boot: Consumer Example (Parallelism via Concurrency)

**Consumer config (application.yml):**
```yaml
spring:
  kafka:
    consumer:
      bootstrap-servers: localhost:9092
      group-id: order-service-group
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

**Consumer code with concurrency:**
```java
@Service
public class OrderConsumer {

    @KafkaListener(
        topics = "orders",
        groupId = "order-service-group",
        concurrency = "3" // up to 3 threads -> up to 3 partitions in parallel
    )
    public void consume(String message) {
        System.out.println("Thread: " + Thread.currentThread().getName()
                           + " | Message: " + message);
        // process message...
    }
}
```

**Example runtime behavior:**
- Topic `orders` has 3 partitions.
- `concurrency = 3` ⇒ Spring creates 3 consumer threads in the same group.
- Each thread gets assigned one partition:
  - Thread-1 → `orders-0`
  - Thread-2 → `orders-1`
  - Thread-3 → `orders-2`
- Messages from different partitions are processed **in parallel**, increasing throughput.

---

## Summary in One Line

Kafka partitioning controls **how many consumers can work in parallel** and thus how much **throughput** you can achieve; more partitions generally mean more parallelism and higher throughput, while ordering is guaranteed only within each partition.
