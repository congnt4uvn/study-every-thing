
Here’s a concise summary of Kafka retention & compaction, focusing on time-based and size-based retention, with Spring Boot examples.

---

## 1. Kafka Retention: Core Idea

Kafka **retention** controls **how long** or **how much** data a topic keeps before old records are deleted.

Two main policies:
1. **Time-based retention** – keep messages for a certain duration.
2. **Size-based retention** – keep messages until the log reaches a certain size.

These can be combined: Kafka deletes data when **either** the time limit or size limit is exceeded.

---

## 2. Time-based Retention

- Controlled by `retention.ms` (per topic) or `log.retention.hours` (broker default).
- Example: keep messages for 7 days.

### Topic-level config (CLI)

```bash
# Create topic with 7 days retention
kafka-topics.sh --create \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=604800000   # 7 * 24 * 60 * 60 * 1000
```

### Spring Boot (application.yml)

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    properties:
      # default for topics auto-created by broker (if allowed)
      log.retention.hours: 168  # 7 days
```

### Programmatic topic creation in Spring Boot

```java
@Configuration
public class KafkaTopicConfig {

    @Bean
    public NewTopic ordersTopic() {
        Map<String, String> configs = new HashMap<>();
        configs.put("retention.ms", String.valueOf(7L * 24 * 60 * 60 * 1000)); // 7 days

        return TopicBuilder.name("orders")
                .partitions(3)
                .replicas(1)
                .configs(configs)
                .build();
    }
}
```

**Behavior example**

- Input: produce messages to `orders` topic continuously.
- After 7 days: messages older than 7 days are eligible for deletion.
- Output: consumers cannot read messages older than 7 days; they are gone from the log.

---

## 3. Size-based Retention

- Controlled by `retention.bytes` (per topic) or `log.retention.bytes` (broker default).
- Example: keep up to 10 GB per topic partition.

### Topic-level config (CLI)

```bash
kafka-topics.sh --create \
  --topic metrics \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.bytes=10737418240  # 10 GB
```

### Programmatic in Spring Boot

```java
@Bean
public NewTopic metricsTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("retention.bytes", String.valueOf(10L * 1024 * 1024 * 1024)); // 10 GB

    return TopicBuilder.name("metrics")
            .partitions(3)
            .replicas(1)
            .configs(configs)
            .build();
}
```

**Behavior example**

- Input: produce messages until the topic reaches ~10 GB per partition.
- When size > 10 GB: Kafka starts deleting oldest segments.
- Output: only the most recent ~10 GB of data is retained.

---

## 4. Combining Time & Size

You can set **both**:

```bash
kafka-configs.sh --alter \
  --bootstrap-server localhost:9092 \
  --entity-type topics \
  --entity-name events \
  --add-config retention.ms=259200000,retention.bytes=21474836480
# 3 days, 20 GB
```

Kafka deletes data when **either**:
- data is older than 3 days, OR
- total size exceeds 20 GB.

## 5. Log Compaction (vs Retention)

Retention deletes old data based on **time/size**.

**Log compaction** keeps **the latest record per key**, regardless of age, and removes older versions of the same key.

- Enabled with `cleanup.policy=compact` or `compact,delete`.
- Good for **state-like** topics (e.g., user profiles, account balances).

### Compacted topic example

```bash
kafka-topics.sh --create \
  --topic user-profiles \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.5
```

### Spring Boot topic config

```java
@Bean
public NewTopic userProfilesTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("cleanup.policy", "compact");
    configs.put("min.cleanable.dirty.ratio", "0.5");

    return TopicBuilder.name("user-profiles")
            .partitions(3)
            .replicas(1)
            .configs(configs)
            .build();
}
```

**Input / Output example (compaction)**

Input messages (keyed by `userId`):

1. `key=user-1, value={name: "Alice", age: 25}`
2. `key=user-2, value={name: "Bob", age: 30}`
3. `key=user-1, value={name: "Alice", age: 26}`  (update)
4. `key=user-2, value={name: "Bob", age: 31}`   (update)

After compaction, log keeps only the **latest per key**:

- `key=user-1, value={name: "Alice", age: 26}`
- `key=user-2, value={name: "Bob", age: 31}`

Consumers that read from the beginning reconstruct the **current state**.

## 6. Spring Boot Producer/Consumer Example

### Producer (sending messages)

```java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(String orderId, String payload) {
        kafkaTemplate.send("orders", orderId, payload);
    }
}
```

**Example input**

```java
orderProducer.sendOrder("order-1", "{\"status\":\"CREATED\"}");
orderProducer.sendOrder("order-2", "{\"status\":\"CREATED\"}");
```

### Consumer

```java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void listen(ConsumerRecord<String, String> record) {
        System.out.printf("Received: key=%s, value=%s, offset=%d%n",
                record.key(), record.value(), record.offset());
    }
}
```

**Example output (console)**

```
Received: key=order-1, value={"status":"CREATED"}, offset=0
Received: key=order-2, value={"status":"CREATED"}, offset=1
```

Old messages will **disappear from the log** according to your retention settings, but consumers that already processed them are unaffected.

---

## 7. When to Use What

- **Time-based retention**: event history, logs, audit trails (e.g., keep 7 days).
- **Size-based retention**: control disk usage strictly (e.g., max 100 GB).
- **Compaction**: topics representing **current state** (user profiles, configs).
- **Compact + delete**: keep latest state but also drop very old tombstones or data.
