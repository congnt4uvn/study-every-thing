
Here’s a concise summary of the key ideas around Kafka retention & compaction, plus Spring Boot config and code examples.

---

## 1. Segment Rolling

Kafka logs are split into **segments** (files). As data is appended, Kafka **rolls** (closes current segment and opens a new one) based on:

- Size (`log.segment.bytes`)
- Time (`log.roll.ms` / `log.roll.hours`)

This affects when old data becomes eligible for deletion or compaction.

**Broker-level example (server.properties):**
```properties
log.segment.bytes=1073741824   # 1 GB
log.roll.ms=604800000          # 7 days
```

---

## 2. Retention (Deletion Policy)

Retention controls **how long** or **how much** data Kafka keeps before deleting old segments.

Key configs (per topic or broker):

- `retention.ms` – keep data for this many milliseconds
- `retention.bytes` – keep up to this many bytes per partition
- `cleanup.policy=delete` – use time/size-based deletion

When a segment is older than `retention.ms` (or total size exceeds `retention.bytes`), and it’s closed, Kafka can delete it.

**Topic-level example (Spring Boot application.yml):**
```yaml
spring:
  kafka:
    admin:
      properties:
        bootstrap.servers: localhost:9092

    # Example: create topic with retention
    topic:
      name: my-delete-topic
      partitions: 3
      replication-factor: 1
```

You can create/configure the topic via `KafkaAdmin`:

```java
@Bean
public NewTopic deleteTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("cleanup.policy", "delete");
    configs.put("retention.ms", String.valueOf(24 * 60 * 60 * 1000)); // 1 day
    configs.put("retention.bytes", String.valueOf(1073741824));       // 1 GB

    return TopicBuilder.name("my-delete-topic")
            .partitions(3)
            .replicas(1)
            .configs(configs)
            .build();
}
```

**Effect (conceptual input/output):**

- Input: produce messages continuously to `my-delete-topic`.
- After 1 day (or when size > 1 GB per partition), **old segments** are deleted.
- Consumers cannot read messages that were deleted.

---

## 3. Log Compaction

Compaction keeps **only the latest value per key** and removes older versions, but **never deletes keys that have no newer value** (unless tombstones are used).

Key configs:

- `cleanup.policy=compact` or `compact,delete`
- `min.cleanable.dirty.ratio` – how much of the log must be “dirty” before compaction
- `segment.ms` / `segment.bytes` – still control segment rolling

**Topic-level example (Spring Boot):**
```java
@Bean
public NewTopic compactTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("cleanup.policy", "compact");
    configs.put("min.cleanable.dirty.ratio", "0.5"); // start compaction when 50% dirty

    return TopicBuilder.name("my-compact-topic")
            .partitions(3)
            .replicas(1)
            .configs(configs)
            .build();
}
```

**Conceptual input/output:**

- Input messages (keyed):
  - `key=user1, value="v1"`
  - `key=user1, value="v2"`
  - `key=user2, value="a1"`
  - `key=user1, value="v3"`

- After compaction, log effectively contains:
  - `key=user1, value="v3"` (latest for user1)
  - `key=user2, value="a1"` (latest for user2)

Old versions (`v1`, `v2`) are removed from compacted segments.

---

## 4. Compaction vs Deletion Policies

You can choose:

1. **Deletion only**  
   `cleanup.policy=delete`  
   - Retention based on time/size.
   - Entire segments are deleted.
   - Good for event logs, audit logs, etc.

2. **Compaction only**  
   `cleanup.policy=compact`  
   - Keeps latest value per key.
   - Log acts like a durable key-value store.
   - Good for state, configuration, user profiles.

3. **Both**  
   `cleanup.policy=compact,delete`  
   - Compaction keeps latest per key.
   - Deletion still applies based on retention.
   - Good when you want latest state but don’t want to keep it forever.

**Example: compact + delete topic:**
```java
@Bean
public NewTopic compactAndDeleteTopic() {
    Map<String, String> configs = new HashMap<>();
    configs.put("cleanup.policy", "compact,delete");
    configs.put("retention.ms", String.valueOf(7L * 24 * 60 * 60 * 1000)); // 7 days

    return TopicBuilder.name("my-compact-delete-topic")
            .partitions(3)
            .replicas(1)
            .configs(configs)
            .build();
}
```

## 5. Simple Spring Boot Producer/Consumer Example

**Producer (keyed messages for compaction):**
```java
@Service
public class UserStateProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public UserStateProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void updateUser(String userId, String stateJson) {
        kafkaTemplate.send("my-compact-topic", userId, stateJson);
    }
}
```

**Consumer:**
```java
@KafkaListener(topics = "my-compact-topic", groupId = "user-state-consumer")
public void consume(ConsumerRecord<String, String> record) {
    String userId = record.key();
    String state = record.value();
    // process latest user state
}
```

**Behavior:**

- Input: multiple updates for the same `userId`.
- Output (over time): consumer may see all updates, but **storage** on the broker will eventually keep only the latest value per `userId` after compaction.

---

**In summary:**

- **Segment rolling**: splits log into files; enables retention/compaction to act on closed segments.
- **Retention (deletion)**: removes old segments based on time/size.
- **Compaction**: removes old records per key, keeping only the latest value.
- **Policies**: `delete`, `compact`, or `compact,delete` depending on whether you want a pure event log, a key-value store, or a mix.
