
Here’s a concise summary of the key producer configs and how they work together, plus code examples.

---

## 1. `retries`

**What it is:**  
Number of times the producer will retry sending a record if a transient error occurs (e.g., network glitch, leader not available).

**Key points:**
- Default is often `Integer.MAX_VALUE` in newer clients (via `delivery.timeout.ms`), but check your Kafka version.
- Retries are only attempted within the time window defined by `delivery.timeout.ms`.
- Works closely with `max.in.flight.requests.per.connection` to avoid message reordering.

**Example behavior:**
- If `retries = 5`, the producer will attempt to send the same record up to 6 times total (1 initial + 5 retries), as long as it doesn’t exceed `delivery.timeout.ms`.

---

## 2. `delivery.timeout.ms`

**What it is:**  
Upper bound (in milliseconds) on the total time a record can spend in the producer (including retries, backoff, batching, etc.) before failing.

**Key points:**
- If a record cannot be successfully acknowledged by the broker within this time, the send fails with a timeout.
- It effectively caps how long `retries` can keep trying.
- Default is often 120000 ms (2 minutes), but verify your client version.

**Example behavior:**
- `delivery.timeout.ms = 60000` (60 seconds):  
  Even if `retries` is large, the record will fail if not acknowledged within 60 seconds.

---

## 3. `max.in.flight.requests.per.connection`

**What it is:**  
Maximum number of unacknowledged requests the producer can send on a single connection.

**Key points:**
- Higher values improve throughput (more parallel requests).
- If `retries > 0` and `max.in.flight.requests.per.connection > 1`, message reordering can occur when a batch fails and is retried while later batches have already succeeded.
- To guarantee ordering with retries, set `max.in.flight.requests.per.connection = 1` (at the cost of throughput).

**Example behavior:**
- `max.in.flight.requests.per.connection = 5`:  
  Producer can send 5 batches without waiting for responses. If batch 2 fails and is retried after batch 3 succeeds, the final order on the broker might be 1,3,2,4,5.

---

## How they work together

- `retries` controls how many times to retry.
- `delivery.timeout.ms` limits the total time spent trying.
- `max.in.flight.requests.per.connection` affects throughput and ordering guarantees.

Typical patterns:
- **High throughput, possible reordering:**
  - `retries` large (or default)
  - `delivery.timeout.ms` moderate/high
  - `max.in.flight.requests.per.connection` > 1
- **Strict ordering, lower throughput:**
  - `retries` > 0
  - `delivery.timeout.ms` tuned
  - `max.in.flight.requests.per.connection = 1`

---

## Spring Boot / Spring Kafka Example

### application.yml

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      retries: 5
      properties:
        delivery.timeout.ms: 60000
        max.in.flight.requests.per.connection: 1
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

### Producer configuration in Java (Spring Kafka)

```java
import org.apache.kafka.clients.producer.ProducerConfig;
import org.apache.kafka.common.serialization.StringSerializer;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.DefaultKafkaProducerFactory;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.core.ProducerFactory;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");

        // Key configs
        props.put(ProducerConfig.RETRIES_CONFIG, 5);
        props.put(ProducerConfig.DELIVERY_TIMEOUT_MS_CONFIG, 60000);
        props.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, 1);

        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

### Sending a message and observing behavior

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;

@Service
public class MyProducerService {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public MyProducerService(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendMessage(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value)
            .addCallback(
                result -> {
                    if (result != null) {
                        System.out.println("SUCCESS: " +
                            "topic=" + result.getRecordMetadata().topic() +
                            ", partition=" + result.getRecordMetadata().partition() +
                            ", offset=" + result.getRecordMetadata().offset());
                    }
                },
                ex -> {
                    System.out.println("FAILURE: " + ex.getClass().getSimpleName() +
                        " - " + ex.getMessage());
                }
            );
    }
}
```

### Example input/output

**Input call:**
```java
sendMessage("orders", "order-123", "Order created");
```

**Possible outputs:**

- Successful send:
  ```
  SUCCESS: topic=orders, partition=0, offset=42
  ```

- Failure due to timeout (after retries within `delivery.timeout.ms`):
  ```
  FAILURE: TimeoutException - Expiring 1 record(s) for orders-0 due to delivery timeout
  ```

This shows how `retries`, `delivery.timeout.ms`, and `max.in.flight.requests.per.connection` influence whether the message eventually succeeds, how long the producer waits, and whether ordering is preserved.
