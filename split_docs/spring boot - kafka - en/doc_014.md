
Here’s a concise summary of Kafka retention & compaction and how they affect storage, performance, and consumers, with Spring Boot examples.

---

## 1. Retention (Time/Size-Based)

**What it is**  
Kafka deletes old messages based on:
- **Time**: `retention.ms` (e.g., keep data for 7 days)
- **Size**: `retention.bytes` (e.g., keep last 10GB per partition)

**Impact on storage**
- Limits how much data is stored.
- Old segments are deleted once retention is exceeded.
- Helps control disk usage.

**Impact on performance**
- Smaller log = faster log cleanup and less disk I/O.
- But if retention is too small, consumers that are far behind may lose data they haven’t read yet.

**Impact on consumers**
- Consumers can only read messages that are still retained.
- If a consumer is too slow and data is deleted, it will miss messages (data loss from consumer perspective).

**Topic config example (CLI)**:
```bash
kafka-topics.sh --create \
  --topic orders \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config retention.ms=604800000 \   # 7 days
  --config retention.bytes=10737418240  # 10 GB
```

---

## 2. Log Compaction

**What it is**  
Compaction keeps **only the latest value per key** in a topic. Old records with the same key are removed eventually.

- Enabled with: `cleanup.policy=compact` or `compact,delete`
- Topic must have keys for compaction to be meaningful.

**Impact on storage**
- Keeps a “compressed” view of the latest state per key.
- Storage grows with number of unique keys, not with all historical updates.

**Impact on performance**
- Compaction is a background process; heavy compaction can increase disk I/O.
- But reading the latest state is efficient because old versions are removed.

**Impact on consumers**
- Consumers can reconstruct the latest state by reading the full compacted log.
- Deleted keys are represented by **tombstone records** (key with null value); after compaction, those keys disappear.

**Topic config example (CLI)**:
```bash
kafka-topics.sh --create \
  --topic user-profile \
  --bootstrap-server localhost:9092 \
  --partitions 3 \
  --replication-factor 1 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.5 \
  --config segment.ms=600000
```

---

## 3. Retention vs Compaction

You can use:
- **Retention only** (`cleanup.policy=delete`):  
  Good for event streams, logs, analytics.
- **Compaction only** (`cleanup.policy=compact`):  
  Good for stateful data (user profiles, account balances, configs).
- **Both** (`cleanup.policy=compact,delete`):  
  Keep recent history + compact older segments.

## 4. Spring Boot Producer Example

**application.yml**
```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

**Producer code (compacted topic: user-profile)**

```java
@Service
public class UserProfileProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public UserProfileProducer(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void updateUserProfile(String userId, String profileJson) {
        // key = userId, value = latest profile
        kafkaTemplate.send("user-profile", userId, profileJson);
    }

    public void deleteUserProfile(String userId) {
        // tombstone record: key with null value
        kafkaTemplate.send("user-profile", userId, null);
    }
}
```

**Example input/output**

- Input call:
  ```java
  updateUserProfile("user-123", "{\"name\":\"Alice\",\"age\":30}");
  updateUserProfile("user-123", "{\"name\":\"Alice\",\"age\":31}");
  deleteUserProfile("user-123");
  ```

- Log content over time (before compaction):
  1. key=user-123, value={"name":"Alice","age":30}
  2. key=user-123, value={"name":"Alice","age":31}
  3. key=user-123, value=null (tombstone)

- After compaction:
  - All records for `user-123` may be removed (because last record is tombstone).

## 5. Spring Boot Consumer Example

**application.yml**
```yaml
spring:
  kafka:
    consumer:
      group-id: user-profile-service
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
```

**Consumer code**

```java
@Service
public class UserProfileConsumer {

    private final Map<String, String> cache = new ConcurrentHashMap<>();

    @KafkaListener(topics = "user-profile", groupId = "user-profile-service")
    public void consume(ConsumerRecord<String, String> record) {
        String key = record.key();
        String value = record.value();

        if (value == null) {
            // tombstone: delete from local state
            cache.remove(key);
        } else {
            // latest state for this key
            cache.put(key, value);
        }
    }

    public String getUserProfile(String userId) {
        return cache.get(userId);
    }
}
```

**Behavior**
- On startup with `auto-offset-reset=earliest`, the consumer reads the entire compacted log and reconstructs the latest state in `cache`.
- After compaction, the consumer still sees the same final state; only disk storage is reduced.

---

## 6. Practical Guidelines

- Use **retention** to control disk usage and how long history is available.
- Use **compaction** for topics that represent **current state per key**.
- For critical consumers, ensure:
  - Retention is long enough, or
  - They process in near real-time, or
  - You use compacted topics so state can be rebuilt from the log.
