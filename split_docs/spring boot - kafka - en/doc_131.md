
Here’s a concise summary and some practical examples around **GC and JVM tuning for Kafka brokers and Spring Boot Kafka clients**.

---

## 1. Key Ideas: GC & JVM Tuning for Kafka Brokers

1. **Goal**:  
   - Minimize GC pauses (stop-the-world) to avoid broker latency spikes and timeouts.  
   - Keep heap usage stable and predictable.

2. **GC Choice (modern JDKs)**  
   - Prefer **G1GC** (JDK 11+) or **ZGC** (JDK 17+) for low pause times.  
   - For most Kafka clusters, **G1GC** is the standard choice.

3. **Heap Size**  
   - Typical broker heap: **4–8 GB** (sometimes up to 16 GB, but avoid very large heaps).  
   - Too small → frequent GC; too large → long GC pauses.  
   - Kafka relies heavily on the OS page cache, so don’t give all memory to the JVM.

4. **GC Tuning Goals**  
   - Keep **young generation** large enough to handle short-lived objects.  
   - Avoid frequent full GCs.  
   - Monitor GC logs and adjust heap / GC parameters based on real data.

5. **Monitoring**  
   - Enable GC logging.  
   - Track: GC pause times, frequency, heap usage, allocation rate.  
   - Use tools: `jstat`, `jcmd`, Prometheus + Grafana, or APM tools.

## 2. Example: JVM Options for Kafka Broker (G1GC)

**kafka-server-start.sh** (or systemd unit) JVM options:

```bash
KAFKA_HEAP_OPTS="-Xms6g -Xmx6g"

KAFKA_JVM_PERFORMANCE_OPTS="
  -server
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=50
  -XX:InitiatingHeapOccupancyPercent=35
  -XX:+ParallelRefProcEnabled
  -XX:+DisableExplicitGC
  -XX:+AlwaysPreTouch
  -XX:+PerfDisableSharedMem
  -XX:+ExitOnOutOfMemoryError
  -XX:+UnlockDiagnosticVMOptions
  -XX:+G1SummarizeRSetStats
  -XX:+PrintGCDetails
  -XX:+PrintGCDateStamps
  -Xloggc:/var/log/kafka/gc.log
"
```

**Input**: High-throughput producer traffic (e.g., 100k msgs/s).  
**Expected Output/Effect**:
- GC pauses mostly under **50 ms**.  
- Heap usage stable around 50–70%.  
- No frequent full GCs under normal load.

## 3. JVM & GC Tuning for Spring Boot Kafka Clients

1. **Clients are usually lighter** than brokers:
   - Smaller heap (e.g., **512m–2g**).  
   - Still benefit from G1GC or ZGC for low latency.

2. **Spring Boot JVM Options (example)**

`JAVA_TOOL_OPTIONS` or container `JAVA_OPTS`:

```bash
JAVA_OPTS="
  -Xms512m
  -Xmx512m
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=100
  -XX:+UseStringDeduplication
  -XX:+ExitOnOutOfMemoryError
  -XX:+PrintGCDetails
  -XX:+PrintGCDateStamps
  -Xloggc:/var/log/app-gc.log
"
```

**Input**: Spring Boot app consuming from Kafka at 5k msgs/s.  
**Expected Output/Effect**:
- Consumer latency stable (e.g., p99 < 200 ms).  
- GC pauses short and infrequent.  
- No OOM under normal load.

## 4. Spring Boot Kafka Client Code Example

### Producer Configuration

```java
@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "broker1:9092,broker2:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        // GC-related impact: batching & buffering affect allocation rate
        props.put(ProducerConfig.BATCH_SIZE_CONFIG, 32_768);        // 32 KB
        props.put(ProducerConfig.LINGER_MS_CONFIG, 10);             // 10 ms
        props.put(ProducerConfig.BUFFER_MEMORY_CONFIG, 67_108_864); // 64 MB

        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
        return new KafkaTemplate<>(producerFactory());
    }
}
```

**Input**:

```java
@Autowired
private KafkaTemplate<String, String> kafkaTemplate;

public void sendMessage() {
    kafkaTemplate.send("orders", "order-1", "{ \"id\": 1, \"amount\": 100 }");
}
```

**Output**:
- Message written to topic `orders` with key `order-1`.  
- GC impact: batching reduces per-message allocations and GC pressure.

### Consumer Configuration

```java
@Configuration
@EnableKafka
public class KafkaConsumerConfig {

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, "broker1:9092,broker2:9092");
        props.put(ConsumerConfig.GROUP_ID_CONFIG, "order-service");
        props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
        props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);

        // GC-related impact: max.poll.records affects batch size and memory usage
        props.put(ConsumerConfig.MAX_POLL_RECORDS_CONFIG, 500);
        props.put(ConsumerConfig.FETCH_MIN_BYTES_CONFIG, 1_048_576); // 1 MB

        return new DefaultKafkaConsumerFactory<>(props);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        factory.setConcurrency(3); // more threads, more heap usage
        return factory;
    }
}
```

**Input**:

```java
@KafkaListener(topics = "orders", groupId = "order-service")
public void handleOrder(String message) {
    // process order
}
```

**Output**:
- Messages from `orders` consumed and processed.  
- GC impact: `MAX_POLL_RECORDS` and `concurrency` control memory footprint and allocation rate.

## 5. Practical Checklist

- **Brokers**:
  - Use **G1GC** (or ZGC on newer JDKs).
  - Heap: **4–8 GB**; leave RAM for OS page cache.
  - Enable GC logs and monitor pause times.
  - Adjust `MaxGCPauseMillis` and heap size based on real metrics.

- **Spring Boot Clients**:
  - Use **G1GC** with **512m–2g** heap.
  - Tune producer batching and consumer `max.poll.records` to balance throughput vs memory.
  - Monitor GC and latency under realistic load tests.
