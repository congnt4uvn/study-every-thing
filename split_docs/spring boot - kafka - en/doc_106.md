
Here’s a concise summary and some practical examples around **Kafka topic design patterns** in a **Spring Boot** context.

---

## 1. Topic Design by Entity

**Idea:**  
Create topics that map directly to core domain entities.

- Examples: `user-events`, `order-events`, `product-events`
- Each topic carries all events related to that entity (created, updated, deleted, etc.)

**Pros:**
- Simple mental model: “one entity → one topic”
- Easy to find all events for a given entity type
- Good for event sourcing and audit logs

**Cons:**
- Topic can become a mix of many event types (create/update/delete)
- Consumers must filter by event type

**Example:**

**Topic:** `order-events`  
**Message key:** `orderId`  
**Message value (JSON):**
```json
{
  "eventType": "ORDER_CREATED",
  "orderId": "ORD-123",
  "customerId": "CUST-10",
  "totalAmount": 99.50,
  "createdAt": "2026-05-20T10:15:00Z"
}
```

### Spring Boot Producer (by entity)

```java
@Service
public class OrderEventProducer {

    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    @Value("${app.kafka.topics.order-events}")
    private String orderEventsTopic;

    public OrderEventProducer(KafkaTemplate<String, OrderEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderCreated(OrderEvent event) {
        kafkaTemplate.send(orderEventsTopic, event.getOrderId(), event);
    }
}
```

### Spring Boot Consumer (by entity)

```java
@Service
public class OrderEventConsumer {

    @KafkaListener(topics = "${app.kafka.topics.order-events}", groupId = "order-service")
    public void handleOrderEvent(OrderEvent event) {
        switch (event.getEventType()) {
            case "ORDER_CREATED" -> handleCreated(event);
            case "ORDER_UPDATED" -> handleUpdated(event);
            case "ORDER_CANCELLED" -> handleCancelled(event);
        }
    }

    private void handleCreated(OrderEvent event) {
        // process new order
    }

    private void handleUpdated(OrderEvent event) {
        // process order update
    }

    private void handleCancelled(OrderEvent event) {
        // process cancellation
    }
}
```

**Input → Output example:**

- Input to producer: `OrderEvent(eventType="ORDER_CREATED", orderId="ORD-123", ...)`
- Output on topic `order-events`: JSON message with that event
- Consumer receives the event and calls `handleCreated`.

## 2. Topic Design by Use Case

**Idea:**  
Create topics around specific workflows or business processes, not raw entities.

- Examples:  
  - `payment-authorization-requests`  
  - `payment-authorization-results`  
  - `email-notification-commands`  
  - `shipping-fulfillment-commands`

**Pros:**
- Very focused topics; each has a clear purpose
- Consumers are simpler (they handle one use case)
- Good for command/event separation

**Cons:**
- More topics to manage
- Same entity may appear in multiple topics (duplication)

**Example:**

**Topic:** `payment-authorization-requests`  
**Message value:**
```json
{
  "requestId": "REQ-1",
  "orderId": "ORD-123",
  "amount": 99.50,
  "paymentMethodId": "PM-55"
}
```

**Topic:** `payment-authorization-results`  
**Message value:**
```json
{
  "requestId": "REQ-1",
  "orderId": "ORD-123",
  "status": "APPROVED",
  "authorizedAmount": 99.50,
  "approvedAt": "2026-05-20T10:16:00Z"
}
```

### Spring Boot Producer (use-case: send authorization request)

```java
@Service
public class PaymentRequestProducer {

    private final KafkaTemplate<String, PaymentAuthorizationRequest> kafkaTemplate;

    @Value("${app.kafka.topics.payment-authorization-requests}")
    private String requestTopic;

    public PaymentRequestProducer(KafkaTemplate<String, PaymentAuthorizationRequest> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendAuthorizationRequest(PaymentAuthorizationRequest request) {
        kafkaTemplate.send(requestTopic, request.getRequestId(), request);
    }
}
```

### Spring Boot Consumer (use-case: handle result)

```java
@Service
public class PaymentResultConsumer {

    @KafkaListener(
        topics = "${app.kafka.topics.payment-authorization-results}",
        groupId = "order-orchestration-service"
    )
    public void handlePaymentResult(PaymentAuthorizationResult result) {
        if ("APPROVED".equals(result.getStatus())) {
            // continue order workflow
        } else {
            // mark order as payment_failed
        }
    }
}
```

## 3. Topic Design by Bounded Context

**Idea:**  
Align topics with **bounded contexts** from Domain-Driven Design (DDD).  
Each context owns its topics and publishes events about its internal state.

- Examples:
  - `billing.order-billed`, `billing.invoice-issued`
  - `shipping.shipment-created`, `shipping.shipment-delivered`
  - `catalog.product-updated`, `catalog.price-changed`

**Pros:**
- Strong ownership and clear boundaries
- Good for microservices and DDD
- Topics reflect domain language

**Cons:**
- Requires clear domain modeling
- Cross-context communication may need translation/anti-corruption layers

**Example:**

**Topic:** `shipping.shipment-created`  
**Message value:**
```json
{
  "shipmentId": "SHIP-100",
  "orderId": "ORD-123",
  "address": "123 Main St",
  "createdAt": "2026-05-20T10:20:00Z"
}
```

### Spring Boot Producer (bounded context: Shipping)

```java
@Service
public class ShipmentEventProducer {

    private final KafkaTemplate<String, ShipmentCreatedEvent> kafkaTemplate;

    @Value("${app.kafka.topics.shipping-shipment-created}")
    private String shipmentCreatedTopic;

    public ShipmentEventProducer(KafkaTemplate<String, ShipmentCreatedEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void publishShipmentCreated(ShipmentCreatedEvent event) {
        kafkaTemplate.send(shipmentCreatedTopic, event.getShipmentId(), event);
    }
}
```

### Spring Boot Consumer (another context listening)

```java
@Service
public class BillingShipmentConsumer {

    @KafkaListener(
        topics = "${app.kafka.topics.shipping-shipment-created}",
        groupId = "billing-service"
    )
    public void onShipmentCreated(ShipmentCreatedEvent event) {
        // update billing records, maybe trigger invoice
    }
}
```

---

## How to Choose a Pattern

- **By Entity**  
  - When you want a central event log per entity type  
  - Good for audit, event sourcing, and generic consumers

- **By Use Case**  
  - When you have clear workflows/commands  
  - Good for orchestration and process-driven systems

- **By Bounded Context**  
  - When using DDD and microservices  
  - Good for clear ownership and domain boundaries

In real systems, you often **combine** these:
- Inside a bounded context, you might use **entity-based** topics.
- For cross-service workflows, you might use **use-case** topics.
- Topic naming often reflects both: `billing.order-events`, `shipping.shipment-commands`, etc.
