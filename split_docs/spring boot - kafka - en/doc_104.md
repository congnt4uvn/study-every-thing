
Here’s a concise summary of **Event Sourcing** and **CQRS** using **Kafka** with **Spring Boot**, plus simple code-style examples.

## 1. Event Sourcing with Kafka

**Idea:**  
Instead of storing only the current state (e.g., a row in a DB), you store **all events** that happened to an entity.  
The current state is rebuilt by **replaying events**.

**Kafka’s role:**

- Each event is a **message** in a Kafka topic.
- The topic is the **source of truth** (event log).
- Services can **replay** events from the beginning to rebuild state.

**Typical flow:**

1. Command comes in (e.g., “CreateOrder”).
2. Service validates and produces an **OrderCreatedEvent** to Kafka.
3. Other services consume this event and update their own state (DB, cache, projections).

### Example: Event Sourcing with Spring Boot + Kafka

**Event class:**

```java
public class OrderCreatedEvent {
    private String orderId;
    private String customerId;
    private BigDecimal amount;
    private Instant createdAt;
    // getters/setters
}
```

**Producer (write side):**

```java
@Service
public class OrderCommandService {

    private final KafkaTemplate<String, OrderCreatedEvent> kafkaTemplate;

    public OrderCommandService(KafkaTemplate<String, OrderCreatedEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void createOrder(String orderId, String customerId, BigDecimal amount) {
        OrderCreatedEvent event = new OrderCreatedEvent();
        event.setOrderId(orderId);
        event.setCustomerId(customerId);
        event.setAmount(amount);
        event.setCreatedAt(Instant.now());

        kafkaTemplate.send("order-events", orderId, event);
    }
}
```

**Consumer building state (projection):**

```java
@Service
public class OrderProjectionService {

    private final OrderRepository orderRepository;

    public OrderProjectionService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @KafkaListener(topics = "order-events", groupId = "order-projections")
    public void handleOrderCreated(OrderCreatedEvent event) {
        OrderEntity entity = new OrderEntity();
        entity.setId(event.getOrderId());
        entity.setCustomerId(event.getCustomerId());
        entity.setAmount(event.getAmount());
        entity.setCreatedAt(event.getCreatedAt());
        orderRepository.save(entity);
    }
}
```

**Input/Output example:**

- **Input (HTTP command):**

  ```http
  POST /orders
  Content-Type: application/json

  {
    "orderId": "o-123",
    "customerId": "c-10",
    "amount": 99.99
  }
  ```

- **Kafka event produced (logical):**

  ```json
  {
    "orderId": "o-123",
    "customerId": "c-10",
    "amount": 99.99,
    "createdAt": "2026-05-20T10:00:00Z"
  }
  ```

- **Output (query later):**

  ```http
  GET /orders/o-123
  ```

  ```json
  {
    "orderId": "o-123",
    "customerId": "c-10",
    "amount": 99.99,
    "status": "CREATED"
  }
  ```

## 2. CQRS (Command Query Responsibility Segregation) with Kafka

**Idea:**  
Split the system into:

- **Command side (write model)**: handles commands, validates business rules, emits events.
- **Query side (read model)**: listens to events, builds optimized read models (e.g., denormalized tables, caches).

**Kafka’s role:**

- Commands → processed by write service → **events** published to Kafka.
- Read services subscribe to Kafka topics and maintain their own **read-optimized** data stores.

### Simple CQRS structure

**Command API (write side):**

```java
@RestController
@RequestMapping("/orders")
public class OrderCommandController {

    private final OrderCommandService commandService;

    public OrderCommandController(OrderCommandService commandService) {
        this.commandService = commandService;
    }

    @PostMapping
    public ResponseEntity<Void> createOrder(@RequestBody CreateOrderRequest request) {
        commandService.createOrder(request.getOrderId(), request.getCustomerId(), request.getAmount());
        return ResponseEntity.accepted().build(); // async
    }
}
```

**Query API (read side):**

```java
@RestController
@RequestMapping("/orders")
public class OrderQueryController {

    private final OrderRepository orderRepository;

    public OrderQueryController(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @GetMapping("/{id}")
    public ResponseEntity<OrderDto> getOrder(@PathVariable String id) {
        return orderRepository.findById(id)
                .map(entity -> ResponseEntity.ok(new OrderDto(entity)))
                .orElse(ResponseEntity.notFound().build());
    }
}
```

**Kafka listener on read side** (as above) keeps the read DB in sync.

---

## 3. How Event Sourcing + CQRS fit together with Kafka

- **Event Sourcing**: Kafka topic = event store.
- **CQRS**:  
  - Write side: handles commands, emits events to Kafka.  
  - Read side: consumes events, builds query models.
- Benefits:
  - Scalability (read and write scale independently).
  - Audit log (full history of changes).
  - Easy to add new read models by replaying events.
- Trade-offs:
  - Eventual consistency between write and read.
  - More complexity (multiple models, versioning events).
