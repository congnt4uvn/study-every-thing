
Here’s a concise summary and some concrete examples around **producer reliability** in Kafka with **transactional producers**, **fencing**, and **exactly-once semantics (EOS)** in Spring Boot / Kafka Streams.

---

## Key Ideas

1. **Transactional Producer**
   - Allows grouping multiple send operations into a single atomic transaction.
   - Either all messages in the transaction are visible to consumers, or none are.
   - Used to achieve **exactly-once semantics** when combined with idempotence and proper consumer configuration.

2. **Fencing**
   - Prevents multiple producers with the same `transactional.id` from corrupting data.
   - If an old producer instance is still alive but a new one starts with the same `transactional.id`, the old one is **fenced off** (its transactions will fail).
   - This is crucial in distributed environments where producers may be restarted or scaled.

3. **Exactly-Once Semantics (EOS)**
   - Guarantees that each record is processed and written to output topics **exactly once**, even in the presence of retries, failures, or restarts.
   - Achieved by:
     - Enabling idempotent and transactional producers.
     - Using read-process-write patterns with transactions.
     - In Kafka Streams, EOS is built-in with the right configuration.

---

## Spring Boot + Kafka: Transactional Producer Example

### Maven Dependencies (Spring Boot)

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.kafka</groupId>
        <artifactId>spring-kafka</artifactId>
    </dependency>
</dependencies>
```

### Producer Configuration (Java)

```java
import org.apache.kafka.clients.producer.ProducerConfig;
import org.apache.kafka.common.serialization.StringSerializer;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.DefaultKafkaProducerFactory;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.core.ProducerFactory;
import org.springframework.kafka.transaction.KafkaTransactionManager;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaProducerConfig {

    @Bean
    public ProducerFactory<String, String> producerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);

        // Enable idempotence
        props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);

        // Transactional ID (used for fencing)
        props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "my-tx-producer-1");

        return new DefaultKafkaProducerFactory<>(props);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate(
            ProducerFactory<String, String> producerFactory) {
        KafkaTemplate<String, String> template = new KafkaTemplate<>(producerFactory);
        // Initialize transactions
        producerFactory.transactionCapable();
        return template;
    }

    @Bean
    public KafkaTransactionManager<String, String> kafkaTransactionManager(
            ProducerFactory<String, String> producerFactory) {
        return new KafkaTransactionManager<>(producerFactory);
    }
}
```

### Using Transactions in a Service

```java
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class OrderService {

    private final KafkaTemplate<String, String> kafkaTemplate;

    public OrderService(KafkaTemplate<String, String> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    @Transactional("kafkaTransactionManager")
    public void processOrder(String orderId) {
        // Example: write multiple messages atomically
        kafkaTemplate.send("orders", orderId, "ORDER_CREATED:" + orderId);
        kafkaTemplate.send("order-events", orderId, "EVENT_CREATED:" + orderId);

        // If an exception occurs here, both sends are rolled back
        // throw new RuntimeException("Something went wrong");
    }
}
```

### Example Input / Output

**Input (application call):**

```java
orderService.processOrder("order-123");
```

**Output in Kafka (if transaction commits successfully):**

- Topic `orders`:
  - Key: `order-123`
  - Value: `ORDER_CREATED:order-123`
- Topic `order-events`:
  - Key: `order-123`
  - Value: `EVENT_CREATED:order-123`

If an exception is thrown inside `processOrder`, **no messages** appear in either topic.

## Fencing in Practice

- You run two instances of the same Spring Boot app with the same `transactional.id` (`my-tx-producer-1`).
- Kafka will **fence** the older producer when the new one starts.
- The old producer’s attempts to commit transactions will fail with `ProducerFencedException`.
- This ensures only one active producer instance can write with that `transactional.id`, preventing mixed or corrupted transactional state.

## Kafka Streams: Exactly-Once Semantics

Kafka Streams can handle EOS with configuration only.

### Streams Configuration (Java)

```java
import org.apache.kafka.streams.StreamsConfig;
import org.apache.kafka.common.serialization.Serdes;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.config.KafkaStreamsConfiguration;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaStreamsConfig {

    @Bean(name = "kafkaStreamsConfig")
    public KafkaStreamsConfiguration kStreamsConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "orders-streams-app");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, Serdes.String().getClass());

        // Enable exactly-once semantics
        props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG,
                  StreamsConfig.EXACTLY_ONCE_V2);

        return new KafkaStreamsConfiguration(props);
    }
}
```

### Simple Topology Example

```java
import org.apache.kafka.streams.StreamsBuilder;
import org.apache.kafka.streams.kstream.KStream;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class OrdersTopology {

    @Bean
    public KStream<String, String> kStream(StreamsBuilder builder) {
        KStream<String, String> orders = builder.stream("orders");

        // Example: transform and write to another topic
        orders
            .mapValues(value -> "PROCESSED_" + value)
            .to("orders-processed");

        return orders;
    }
}
```

### Example Input / Output (Kafka Streams)

**Input topic `orders`:**

- Key: `order-123`, Value: `ORDER_CREATED:order-123`

**Output topic `orders-processed` (with EOS):**

- Key: `order-123`, Value: `PROCESSED_ORDER_CREATED:order-123`

Even if the Streams app crashes and restarts during processing, each input record will be written to `orders-processed` **exactly once**.

---

## Summary

- **Transactional producers** in Kafka (and Spring Boot) allow atomic writes across multiple records and topics.
- **Fencing** ensures only one active producer instance per `transactional.id`, preventing data corruption.
- **Exactly-once semantics** are achieved by combining transactional producers with proper configuration, and are built-in for Kafka Streams via `PROCESSING_GUARANTEE_CONFIG = EXACTLY_ONCE_V2`.
