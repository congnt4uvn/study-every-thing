


Dưới đây là tóm tắt nội dung:

1. Xem khóa KMS do AWS quản lý  
   - Vào KMS, bên trái là danh sách các khóa do AWS quản lý (AWS managed keys) cho các dịch vụ như EBS, SQS…  
   - Mỗi khóa có một “key policy” (chính sách khóa) quy định ai/ dịch vụ nào được phép dùng khóa.  
   - Ví dụ:
     - Khóa EBS: điều kiện “Via service” là EC2 (dịch vụ cao hơn EBS), caller phải thuộc đúng tài khoản.  
     - Khóa SQS: “Via service” là SQS, chỉ cho phép SQS dùng khóa đó.  
   - Có thể xem cấu hình mật mã: khóa đối xứng, KMS origin, dùng để mã hóa/giải mã.

2. Phân loại khóa trong KMS  
   - AWS managed keys: AWS tạo và quản lý cho từng dịch vụ.  
   - Customer managed keys (CMK): khóa do khách hàng tự tạo và quản lý.  
   - Custom key store (CloudHSM): ngoài phạm vi bài giảng.  
   → Bài giảng tập trung vào customer managed keys.

3. Tạo khóa KMS do khách hàng quản lý  
   - Lưu ý: mỗi khóa tốn khoảng 1 USD/tháng.  
   - Chọn loại khóa:
     - Symmetric (đối xứng): dùng để encrypt/decrypt (được dùng trong ví dụ).  
     - Asymmetric (bất đối xứng): dùng cho encrypt/decrypt hoặc sign/verify (ngoài phạm vi bài).  
   - Advanced options:
     - Key origin: chọn KMS (KMS tự tạo khóa).  
     - Có thể chọn external key hoặc custom key store (CloudHSM) nhưng không đề cập sâu.  
   - Key type theo vùng:
     - Single-region key (khóa vùng đơn): loại truyền thống, phổ biến.  
     - Multi-region key: không dùng trong ví dụ, chỉ chọn single-region.  
   - Đặt bí danh (alias), ví dụ: “huongdan”.

4. Thiết lập quyền và chính sách khóa  
   - Chọn key administrators (ai quản trị khóa).  
   - Nếu không cấu hình chi tiết, dùng “default key policy”:
     - Chính sách này chủ yếu “bật” quyền cho IAM; ai có IAM permission phù hợp thì dùng được.  
   - Chọn “key users” (ai được dùng khóa):
     - Có thể cho “mọi người” trong tài khoản (miễn có IAM phù hợp).  
     - Hoặc chỉ định cụ thể một user (ví dụ chỉ Stefan).  
   - Có thể cho phép tài khoản AWS khác dùng khóa:
     - Dùng trong trường hợp chia sẻ snapshot EBS được mã hóa, v.v.  
   - Sau khi tạo xong, có thể:
     - Xem key policy (dạng JSON giống IAM policy).  
     - Xem “default view” để biết ai là admin, ai là user, tài khoản nào khác được truy cập.

5. Cấu hình bổ sung cho khóa  
   - Cryptographic configuration: loại khóa, mục đích sử dụng (encrypt/decrypt).  
   - Key rotation (xoay vòng khóa):
     - Bật “automatic rotation”, chọn chu kỳ từ 90 đến 2560 ngày (mặc định ~1 năm).  
     - Có thể xem “next rotation date” và “rotation history”.  
     - Có thể “rotate on-demand” (xoay theo yêu cầu).  
   - Alias: có thể dùng alias ARN để tham chiếu khóa cho tiện.  
   - Key actions:
     - Disable key (tạm dừng).  
     - Schedule key deletion (lên lịch xóa).

6. Demo dùng AWS CLI để mã hóa/giải mã với KMS  
   - Tạo file plaintext:
     - Tên: `exampleSecretFile.txt`.  
     - Nội dung: ví dụ “SuperSecretPassword”.  

   a) Mã hóa (encrypt)  
   - Lệnh `aws kms encrypt` với các tham số:
     - `--key-id`: dùng alias (vd: `alias/huongdan`), hoặc key ID, hoặc full ARN.  
     - `--plaintext`: file plaintext (`exampleSecretFile.txt`).  
     - `--output text` + `--query CiphertextBlob`: lấy phần ciphertext.  
     - `--region`: vùng chứa khóa (vd: `eu-west-2`).  
   - Kết quả: file `exampleSecretFileEncrypted` chứa ciphertext dạng Base64 (chỉ là chuỗi chữ/số).  
   - Sau đó:
     - Giải Base64 để lấy file nhị phân (Linux/Mac/Windows lệnh khác nhau).  
     - File nhị phân này không mở được bằng text editor → đây là dữ liệu mã hóa thực sự, có thể chia sẻ cho người khác.

   b) Giải mã (decrypt)  
   - Dùng `aws kms decrypt`:
     - Input: file nhị phân đã mã hóa.  
     - `--query Plaintext` + `--output text`: lấy plaintext (dạng Base64).  
     - Ghi ra file mới, ví dụ `exampleSecretFileDecryptedBase64`.  
   - KMS tự biết phải dùng khóa nào vì thông tin khóa nằm trong ciphertext.  
   - Sau đó:
     - Giải Base64 để lấy file plaintext cuối cùng, ví dụ `exampleSecretFileDecrypted.txt`.  
     - Mở file sẽ thấy lại “SuperSecretPassword”.  

7. Ý nghĩa  
   - Ví dụ minh họa đầy đủ quy trình:
     - Tạo customer managed key trong KMS.  
     - Thiết lập policy mặc định để IAM kiểm soát quyền.  
     - Dùng CLI để encrypt/decrypt file bằng KMS.  
   - Trong thực tế, SDK sẽ trừu tượng hóa bớt các bước, nhưng ví dụ này cho thấy rõ cách hoạt động của lệnh `encrypt` và `decrypt` trong KMS.
