


Dưới đây là tóm tắt ngắn gọn nội dung bài giảng:

1. **Bảo mật API Gateway bằng IAM**
   - Xác thực: dùng IAM (user, role, EC2, Lambda… trong cùng tài khoản AWS).
   - Ủy quyền: dùng **IAM Policy** gắn cho user/role.
   - Client ký request bằng **Signature Version 4 (SigV4)**, thông tin xác thực được đưa vào header.
   - API Gateway giải mã SigV4, kiểm tra với IAM; nếu được phép thì gọi Lambda backend và trả kết quả.

2. **Kết hợp với Resource Policy**
   - Resource Policy cho phép gắn chính sách trực tiếp lên API Gateway (giống Lambda resource policy).
   - Dùng để:
     - Cho phép **cross-account access** (user/role ở tài khoản AWS khác truy cập API).
     - Lọc theo **IP** (whitelist/blacklist).
     - Chỉ cho phép truy cập từ **VPC endpoint**.
   - Thường kết hợp với IAM để kiểm soát truy cập chi tiết hơn.

3. **Bảo mật bằng Cognito User Pool**
   - Cognito User Pool là “database người dùng” do AWS quản lý.
   - Cognito xử lý:
     - Đăng ký, đăng nhập, vòng đời user.
     - Phát hành và hết hạn token (JWT).
   - Luồng:
     - Client xác thực với Cognito User Pool → nhận token.
     - Client gửi token trong request đến API Gateway.
     - API Gateway tích hợp trực tiếp với Cognito, xác minh token.
     - Nếu hợp lệ, API Gateway cho phép truy cập Lambda backend.
   - Không cần viết code custom cho phần xác thực/ủy quyền ở layer API Gateway.

4. **Lambda Authorizer (Custom Authorizer)**
   - Dùng khi có **hệ thống xác thực bên thứ ba** (Auth0, hệ thống nội bộ, OAuth, JWT…).
   - Luồng:
     - Client xác thực với hệ thống bên thứ ba → nhận token.
     - Client gửi token (qua header hoặc query string) đến API Gateway.
     - API Gateway gọi **Lambda Authorizer**.
     - Lambda Authorizer:
       - Nhận token + context request.
       - Tự kiểm tra token (có thể gọi lại hệ thống auth bên thứ ba).
       - Nếu hợp lệ, tạo và trả về **IAM Policy** (Allow/Deny).
     - Policy được **cache** để tăng hiệu năng.
     - Sau đó API Gateway mới gọi Lambda backend.
   - Ưu điểm: rất linh hoạt, tự quyết định logic ủy quyền.
   - Nhược điểm: phải tự viết code, mỗi lần ủy quyền là một lần gọi Lambda (tốn chi phí và thời gian).

5. **So sánh & ghi nhớ cho thi**
   - **IAM + Resource Policy**:
     - Phù hợp khi tất cả user/role nằm trong AWS account (hoặc cross-account).
     - Dùng SigV4.
     - Resource Policy để cross-account, lọc IP, VPC endpoint.
   - **Cognito User Pool**:
     - Dùng khi bạn quản lý user trong Cognito.
     - Không cần code custom ở API Gateway.
     - API Gateway xác minh token Cognito trực tiếp.
   - **Lambda Authorizer**:
     - Dùng khi có **bên thứ ba** hoặc hệ thống auth riêng.
     - Rất linh hoạt, tự tạo IAM Policy.
     - Phải tự xử lý xác thực + ủy quyền trong Lambda Authorizer.
     - Có cache nhưng vẫn tốn chi phí gọi Lambda.

6. **Minh họa trên Console**
   - Trong **Method Request** của API:
     - Có thể chọn **IAM** làm Authorizer.
   - **Resource Policy**:
     - Mẫu sẵn: cross-account access, IP blacklist, source VPC whitelist.
   - **Authorizers**:
     - Tạo **Lambda Authorizer**: chọn hàm Lambda, cấu hình caching.
     - Tạo **Cognito Authorizer**: chỉ cần ARN của Cognito User Pool.

Nếu bạn muốn, tôi có thể vẽ lại một sơ đồ so sánh 3 cách (IAM, Cognito, Lambda Authorizer) theo dạng bảng cho dễ học thuộc.
