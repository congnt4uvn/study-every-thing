


Dưới đây là tóm tắt ngắn gọn nội dung bạn đưa ra:

1. Khái niệm cân bằng tải  
- Load Balancer (bộ cân bằng tải) là một (nhóm) máy chủ đứng phía trước, nhận toàn bộ lưu lượng và phân phối đến nhiều máy chủ backend (EC2, server hạ lưu).  
- Người dùng chỉ thấy một điểm truy cập duy nhất (DNS/URL của Load Balancer), không biết cụ thể đang vào instance nào.

2. Lợi ích khi dùng Load Balancer  
- Phân phối đều tải giữa nhiều EC2, giúp hệ thống chịu tải tốt hơn.  
- Xử lý lỗi mượt mà: nhờ cơ chế health check, LB chỉ gửi traffic đến các instance “healthy”.  
- Hỗ trợ SSL termination (giải mã HTTPS tại LB).  
- Hỗ trợ sticky session (gắn kết bằng cookie).  
- Tăng tính sẵn sàng cao (multi-AZ).  
- Tách biệt traffic public và internal.  
- Được AWS quản lý hoàn toàn: nâng cấp, bảo trì, đảm bảo HA, chi phí thường rẻ và dễ mở rộng hơn tự dựng LB.  
- Tích hợp sâu với nhiều dịch vụ AWS: EC2, Auto Scaling Group, ECS, ACM, CloudWatch, Route 53, WAF, Global Accelerator,…

3. Health Check (kiểm tra tình trạng hoạt động)  
- LB định kỳ gọi đến một endpoint trên instance (ví dụ: HTTP, port 4567, path “/health”).  
- Nếu không trả về mã thành công (thường HTTP 200), instance bị đánh dấu “unhealthy” và không nhận traffic.  
- Health check là thành phần cốt lõi để đảm bảo chỉ instance khỏe mới nhận lưu lượng.

4. Các loại Load Balancer trên AWS (Managed ELB)  
- Classic Load Balancer (CLB – V1, từ 2009): hỗ trợ HTTP, HTTPS, TCP, SSL. Đã bị “khai tử” trong UI, nhưng vẫn dùng được; AWS không khuyến khích dùng mới.  
- Application Load Balancer (ALB – từ 2016): hỗ trợ HTTP, HTTPS, WebSocket; dùng cho layer 7, nhiều tính năng nâng cao.  
- Network Load Balancer (NLB – từ 2017): hỗ trợ TCP, TLS, UDP; hoạt động ở layer 4, hiệu năng cao, độ trễ thấp.  
- Gateway Load Balancer (GWLB – từ 2020): hoạt động ở layer 3 (IP), dùng cho các thiết bị ảo như firewall, IDS/IPS,…

- Các LB có thể cấu hình dạng:  
  - Public (internet-facing): cho website/app công khai.  
  - Internal: chỉ dùng trong mạng nội bộ (VPC).

5. Bảo mật với Security Group  
- Security Group của Load Balancer:  
  - Cho phép HTTP (80) và/hoặc HTTPS (443) từ mọi nơi: source = 0.0.0.0/0.  
- Security Group của EC2 backend:  
  - Chỉ cho phép HTTP (port 80) từ Security Group của Load Balancer (không phải từ IP bất kỳ).  
  - Điều này đảm bảo EC2 chỉ nhận traffic đi qua LB, tăng mức độ bảo mật.

6. Kết luận  
- Sử dụng Elastic Load Balancer trên AWS là lựa chọn “mặc định” vì: dễ dùng, chi phí hợp lý, mở rộng tốt, được quản lý hoàn toàn, tích hợp sâu với các dịch vụ AWS.  
- Các phần tiếp theo sẽ đi sâu hơn vào ALB và NLB.
