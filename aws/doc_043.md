


Nội dung trên là một bài thực hành về cách dùng **IAM Role cho EC2** thay vì cấu hình Access Key trực tiếp trên máy.

Tóm tắt ý chính:

1. **Kết nối vào EC2**
   - Giáo viên dùng EC2 Instance Connect (hoặc SSH đều được) để vào máy.
   - Đây chỉ là một terminal Linux trên cloud, có thể chạy lệnh bình thường (ping, clear, v.v.).

2. **AWS CLI đã cài sẵn**
   - Amazon Linux AMI đã có sẵn AWS CLI.
   - Khi chạy lệnh `aws iam list-users` lần đầu, CLI báo không có credentials và gợi ý dùng `aws configure`.

3. **Tại sao không dùng `aws configure` trên EC2**
   - Không được nhập Access Key ID và Secret Access Key cá nhân vào EC2.
   - Lý do: bất kỳ ai có thể truy cập vào instance (ví dụ qua EC2 Instance Connect) đều có thể lấy cắp các credentials này.
   - Nguyên tắc: **không bao giờ** lưu IAM access key trực tiếp trên EC2.

4. **Giải pháp đúng: dùng IAM Role cho EC2**
   - Trong IAM đã tạo sẵn một role, ví dụ: `DemoRoleForEC2` với policy `IAMReadOnlyAccess`.
   - Gắn role này vào EC2:
     - Vào EC2 → chọn instance → tab Security → Actions → Security → Modify IAM role.
     - Chọn `DemoRoleForEC2` → Save.
   - Sau khi gắn role:
     - Chạy lại `aws iam list-users` → lệnh hoạt động, dù không hề chạy `aws configure`.
     - Chứng tỏ EC2 đang dùng **tạm thời** credentials lấy từ IAM Role.

5. **Kiểm chứng quyền của role**
   - Nếu tách (remove) role khỏi instance rồi chạy lại lệnh → bị `AccessDenied`.
   - Gắn lại role, gắn policy `IAMReadOnlyAccess` cho role:
     - Ban đầu có thể vẫn `AccessDenied` do IAM cần thời gian propagate.
     - Chạy lại sau vài giây → lệnh hoạt động, trả về danh sách user IAM.

6. **Kết luận**
   - Cách đúng để cấp quyền AWS cho EC2 là **chỉ dùng IAM Role**, không dùng access key thủ công.
   - IAM Role cung cấp credentials tạm thời, an toàn hơn và dễ quản lý hơn.
   - Đây là khái niệm rất quan trọng khi làm việc với EC2 và IAM.
