


Dưới đây là tóm tắt nội dung bài giảng về CloudFront cache:

1. Cách CloudFront cache hoạt động  
- Mỗi Edge Location của CloudFront có một bộ nhớ đệm riêng.  
- Khi có request:  
  - Edge Location kiểm tra xem đối tượng đã có trong cache chưa.  
  - Nếu có và chưa hết hạn (dựa trên TTL) → trả về từ cache (cache hit).  
  - Nếu chưa có hoặc đã hết hạn → chuyển tiếp request về Origin, nhận response rồi lưu vào cache cho các request sau.  
- Mục tiêu: tăng Cache Hit Ratio để giảm số request về Origin.

2. Cache Key là gì  
- Cache Key là định danh duy nhất cho mỗi đối tượng trong cache.  
- Mặc định, Cache Key = Hostname + Resource path của URL.  
  - Ví dụ:  
    - Host: `mywebsite.com`  
    - Resource: `/content/stories/example-story.html`  
- Các request có cùng Host và Resource sẽ dùng chung đối tượng cache.

3. Mở rộng Cache Key (Cache Policy)  
- Khi nội dung thay đổi theo user, device, language, location… ta cần Cache Key “giàu” hơn.  
- Có thể thêm vào Cache Key:  
  - HTTP headers  
  - Cookies  
  - Query strings  
- Cách cấu hình thông qua CloudFront Cache Policy:  
  - Headers: không dùng / whitelist / tất cả.  
  - Cookies: không / whitelist / all / all except.  
  - Query strings: không / whitelist / all / all except.  
- Trong Cache Policy cũng cấu hình TTL (0 giây đến 1 năm) và có thể dùng các header Cache-Control, Expires.  
- Có thể dùng Cache Policy tự tạo hoặc policy được AWS quản lý.  
- Quan trọng: mọi header, cookie, query string được đưa vào Cache Key sẽ tự động được forward đến Origin.

4. Ví dụ với HTTP headers (Accept-Language)  
- Ví dụ header `Accept-Language: fr-fr` để yêu cầu blog tiếng Pháp.  
- Nếu Cache Policy không cache theo header (none):  
  - Header không nằm trong Cache Key, cũng không được forward (mặc định).  
  - Hiệu suất cache tốt nhất nhưng không phân biệt ngôn ngữ.  
- Nếu whitelist header `Accept-Language`:  
  - Header này được đưa vào Cache Key → cache theo ngôn ngữ.  
  - Đồng thời header được forward đến Origin để Origin trả đúng nội dung (ví dụ blog tiếng Pháp).

5. Query string  
- Query string là phần sau dấu `?` trong URL, ví dụ: `?border=red&size=large`.  
- Tương tự headers:  
  - None: không dùng query string cho Cache Key, không forward.  
  - Whitelist: chỉ một số query string được dùng.  
  - All except: dùng tất cả trừ một số.  
  - All: tất cả query string đều vào Cache Key và forward.  
- Càng nhiều yếu tố trong Cache Key → càng nhiều biến thể cache → hiệu suất cache giảm.

6. Origin Request Policy (chính sách yêu cầu gốc)  
- Dùng khi muốn forward thêm thông tin đến Origin nhưng không muốn chúng ảnh hưởng đến Cache Key.  
- Origin Request Policy cho phép:  
  - Forward thêm headers, cookies, query strings đến Origin.  
  - Nhưng những thứ này không được dùng để tạo Cache Key.  
- Có thể thêm custom headers hoặc CloudFront headers vào request gửi đến Origin (ví dụ API key, secret header), ngay cả khi viewer request không có.  
- Có thể dùng policy tự tạo hoặc policy managed của AWS.

7. Phân biệt Cache Policy vs Origin Request Policy  
- Cache Policy:  
  - Quyết định những gì được dùng để tạo Cache Key (host, path, headers, cookies, query strings).  
  - Ảnh hưởng trực tiếp đến cách CloudFront cache nội dung.  
- Origin Request Policy:  
  - Quyết định thêm những gì sẽ được forward đến Origin (headers, cookies, query strings) mà không ảnh hưởng đến Cache Key.  
- Quy trình:  
  - Viewer request có headers, cookies, query strings.  
  - Cache Policy chọn một phần trong đó để tạo Cache Key.  
  - Origin Request Policy có thể thêm/bổ sung thêm dữ liệu (user-agent, session-id, referrer…) để gửi đến Origin.  
  - Cache vẫn chỉ dựa trên những gì Cache Policy quy định.

Tóm lại:  
- Cache Policy = định nghĩa Cache Key + TTL.  
- Origin Request Policy = định nghĩa dữ liệu nào được gửi đến Origin mà không làm thay đổi Cache Key.  
- Kết hợp hai policy giúp tối ưu vừa hiệu suất cache, vừa đáp ứng đúng nội dung động/phân loại theo user, ngôn ngữ, thiết bị…
