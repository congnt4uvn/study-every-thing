


Người tường thuật đang demo cách xử lý lỗi trong AWS Step Functions kết hợp với AWS Lambda:

1. Tạo Lambda cố tình gây lỗi  
   - Dùng blueprint `step-functions-error`.  
   - Đặt tên hàm: `MyLambdaFunctionThatFails`.  
   - Mã Lambda ném ra `CustomError` với thông báo “This is a custom error!”.  
   - Kiểm tra hàm bằng một test event (Foobar) → Lambda luôn thất bại và trả về `CustomError`.

2. Tạo State Machine trong Step Functions  
   - Tạo máy trạng thái mới “from scratch”.  
   - Dán file định nghĩa `state-machine.json` vào phần Definition.  
   - Máy trạng thái có một Task state gọi Lambda vừa tạo.  
   - Nếu thành công → chuyển sang trạng thái kết thúc.  
   - Nếu lỗi → có nhiều nhánh xử lý:
     - `CustomErrorFallback`
     - `ReservedTypeFallback`
     - `CatchAllFallback` (States.ALL)

3. Cấu hình Retry & Catch theo loại lỗi  
   - Với `CustomError`:
     - Định nghĩa `Retry`:  
       - `ErrorEquals`: ["CustomError"]  
       - `IntervalSeconds`: 1  
       - `MaxAttempts`: 2  
       - `BackoffRate`: 2  
     - Nếu sau khi retry vẫn lỗi → chuyển sang `Catch`:
       - `ErrorEquals`: ["CustomError"] → `CustomErrorFallback`.
   - Với lỗi khác (ví dụ Reserved type hoặc States.ALL):
     - Có retry riêng, thời gian chờ lâu hơn (30s, 60s, …) với `BackoffRate` lớn hơn.  
     - Sau khi hết retry:
       - Lỗi thuộc reserved type → `ReservedTypeFallback`.
       - Lỗi khác → `CatchAllFallback`.

4. Thực thi lần 1 (với `CustomError`)  
   - Start execution của State Machine.  
   - Task gọi Lambda → Lambda ném `CustomError`.  
   - Step Functions tự động retry theo cấu hình (1s, rồi 2s, …).  
   - Sau khi hết số lần retry:
     - Task state thoát với lỗi.  
     - Bị `Catch` bởi `CustomErrorFallback`.  
     - Workflow kết thúc thành công ở nhánh `CustomErrorFallback`.

5. Thay đổi loại lỗi trong Lambda  
   - Sửa Lambda để ném `NotCustomError` (một lỗi khác, không phải `CustomError`).  
   - Triển khai lại và test → lỗi mới không còn là `CustomError`.

6. Thực thi lần 2 (với lỗi khác)  
   - Start execution mới.  
   - Task gọi Lambda → ném `NotCustomError`.  
   - Step Functions áp dụng logic retry khác (dành cho reserved type hoặc States.ALL):
     - Lần 1 thất bại → đợi 30s.  
     - Lần 2 thất bại → đợi 60s.  
     - Sau số lần retry cấu hình → Task state thoát.  
   - Lỗi được `Catch` bởi `ReservedTypeFallback` (hoặc nhánh tương ứng).  
   - Workflow kết thúc thành công ở nhánh này.

7. Ý nghĩa chính  
   - Có thể cấu hình Step Functions để:
     - Retry khác nhau tùy loại lỗi (thời gian chờ, số lần thử, backoff rate).  
     - Bắt (Catch) lỗi khác nhau và chuyển sang các nhánh xử lý khác nhau.  
   - Ví dụ:
     - `CustomError` → retry nhanh, fallback riêng.  
     - Lỗi khác → retry lâu hơn, fallback khác.  
   - Điều này giúp xây dựng workflow linh hoạt, kiểm soát tốt cách xử lý lỗi.

8. Dọn dẹp  
   - Cuối cùng, người tường thuật nhắc có thể xóa State Machine trong Step Functions để dọn dẹp, dù giữ lại cũng không tốn nhiều chi phí.
