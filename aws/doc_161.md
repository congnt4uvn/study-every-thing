


Dưới đây là tóm tắt ngắn gọn nội dung:

1. Mục tiêu  
- Bạn có một CloudFront Distribution ở chế độ riêng tư.  
- Bạn muốn phân phối nội dung trả phí (premium) cho người dùng toàn cầu, nhưng vẫn kiểm soát và biết ai được phép truy cập.

2. Cách cấp quyền truy cập  
- Dùng **CloudFront Signed URL** hoặc **CloudFront Signed Cookie**.  
- Khi tạo Signed URL / Signed Cookie, phải gắn **policy** gồm:
  - Thời gian hết hạn (URL/cookie có hiệu lực đến khi nào).  
  - Phạm vi IP được phép truy cập (nếu biết IP khách hàng).  
  - Trusted signer (tài khoản/địa chỉ nào được phép tạo Signed URL).

3. Thời gian hiệu lực  
- Nội dung như phim/nhạc: thường đặt thời gian rất ngắn (vài phút).  
- Nội dung riêng tư dài hạn (ví dụ tài liệu cá nhân): có thể đặt hiệu lực nhiều năm.

4. Khác nhau giữa Signed URL và Signed Cookie  
- **Signed URL**:
  - Cấp quyền truy cập cho **từng file riêng lẻ**.  
  - Có bao nhiêu file thì cần bấy nhiêu URL (100 file → 100 URL).  
- **Signed Cookie**:
  - Cấp quyền truy cập cho **nhiều file cùng lúc**.  
  - Một cookie có thể dùng lại cho nhiều file.

5. Cách hoạt động của CloudFront Signed URL  
- S3 bucket được bảo vệ bằng OAC/OAI → chỉ CloudFront mới truy cập được.  
- Ứng dụng backend (do bạn viết) sẽ:
  - Xác thực/ủy quyền người dùng.  
  - Dùng AWS SDK để tạo Signed URL từ CloudFront.  
  - Trả Signed URL cho client.  
- Client dùng Signed URL để tải file trực tiếp từ CloudFront.  
- Cách này cũng áp dụng tương tự cho Signed Cookie.

6. So sánh CloudFront Signed URL vs S3 Pre-signed URL  

**CloudFront Signed URL**  
- Dùng để cấp quyền truy cập vào **đường dẫn qua CloudFront**, không phụ thuộc origin (S3, HTTP backend, EC2, v.v.).  
- Dùng key pair ở cấp tài khoản (chỉ root hoặc quản trị mới quản lý).  
- Có thể lọc theo IP, path, ngày, thời gian hết hạn.  
- Tận dụng được toàn bộ cơ chế cache của CloudFront.  
- Luồng: Client → CloudFront (Signed URL) → Origin (ví dụ EC2).

**S3 Pre-signed URL**  
- URL đại diện cho **yêu cầu trực tiếp đến S3** với quyền của IAM principal đã ký.  
- Ai có URL sẽ có quyền giống như IAM principal đó (trong thời gian hiệu lực).  
- Cho phép client truy cập trực tiếp S3 mà không cần CloudFront.  
- Thích hợp khi:
  - Người dùng truy cập trực tiếp S3.  
  - Bạn không dùng CloudFront, chỉ muốn chia sẻ tạm thời object S3.

7. Khi nào dùng cái nào?  
- CloudFront đứng trước S3 và bucket bị khóa chỉ cho CloudFront (OAI/OAC):  
  → **Phải dùng CloudFront Signed URL / Signed Cookie**.  
- Người dùng truy cập trực tiếp S3, không qua CloudFront:  
  → **Dùng S3 Pre-signed URL** là phù hợp.
