


Dưới đây là tóm tắt nội dung:

1. So sánh Secrets Manager và SSM Parameter Store  
   - **AWS Secrets Manager**  
     - Đắt hơn.  
     - Hỗ trợ **tự động xoay vòng (rotation)** bí mật bằng Lambda.  
     - Có sẵn các hàm Lambda mẫu do AWS cung cấp cho RDS, Redshift, DocumentDB…  
     - Tích hợp chặt với các dịch vụ này nên tiết kiệm thời gian.  
     - **Bắt buộc** dùng mã hóa KMS cho bí mật.  
     - Hỗ trợ tích hợp với CloudFormation.  
     - Có thể lấy bí mật thông qua API riêng của Secrets Manager.

   - **SSM Parameter Store**  
     - Rẻ hơn, phạm vi sử dụng rộng hơn (không chỉ bí mật mà còn nhiều loại tham số cấu hình).  
     - API đơn giản.  
     - **Không có tính năng xoay vòng bí mật gốc**.  
     - Mã hóa KMS là **tùy chọn** (có thể lưu tham số thường hoặc bí mật).  
     - Hỗ trợ CloudFormation.  
     - Có thể truy xuất bí mật từ Secrets Manager thông qua API của Parameter Store.

2. Cách xoay vòng bí mật  
   - **Với Secrets Manager**  
     - Ví dụ: xoay mật khẩu DB Amazon RDS.  
     - Cấu hình Secrets Manager gọi Lambda tự động mỗi 30 ngày.  
     - Lambda (do AWS cung cấp cho RDS) sẽ:  
       - Đổi mật khẩu trong RDS.  
       - Cập nhật lại bí mật trong Secrets Manager.  
     - Nếu là bí mật tùy chỉnh không có tích hợp sẵn, bạn phải tự viết Lambda, nhưng AWS có tài liệu hướng dẫn.

   - **Với SSM Parameter Store**  
     - Không có cơ chế xoay vòng sẵn.  
     - Ví dụ: bạn lưu mật khẩu RDS trong Parameter Store.  
     - Cách làm:  
       - Tạo **Amazon EventBridge Rule** chạy 30 ngày một lần.  
       - Rule gọi một Lambda **do bạn tự viết** để:  
         - Đổi mật khẩu trong RDS.  
         - Cập nhật giá trị mới trong SSM Parameter Store.

3. Kết luận  
   - Secrets Manager: phù hợp cho quản lý bí mật, có xoay vòng tự động, tích hợp sâu với các dịch vụ DB, nhưng chi phí cao hơn.  
   - SSM Parameter Store: rẻ, linh hoạt cho nhiều loại tham số, nhưng muốn xoay vòng phải tự xây dựng cơ chế bằng EventBridge + Lambda.
