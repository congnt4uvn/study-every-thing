


Người hướng dẫn đang demo cách sử dụng AWS X-Ray (tích hợp trong CloudWatch):

1. Mở X-Ray trong CloudWatch  
   - Vào CloudWatch, X-Ray console hiện nằm trong đó, dưới mục Service map.  
   - Ban đầu chưa có dữ liệu, cần có ứng dụng gửi trace.

2. Triển khai ứng dụng demo bằng CloudFormation  
   - Vào CloudFormation, tạo stack mới.  
   - Upload template trong thư mục X-Ray: “EB Javas Scorekeep X-Ray simplified” (bản đơn giản hóa từ AWS).  
   - Đặt tên stack: “Scorekeep X-Ray”.  
   - Giữ nguyên mặc định, chỉ chọn:
     - Subnet 1: subnet đầu tiên  
     - Subnet 2: subnet thứ hai  
     - VPCID: VPC tương ứng  
   - Next → Next → Confirm → Submit.  
   - Stack này triển khai một cụm ECS, UI, backend API, tất cả đều tích hợp X-Ray, chạy trên T2/T3.

3. Sử dụng ứng dụng web  
   - Khi stack hoàn tất, vào tab Resources để xem tài nguyên (ECS role, DynamoDB tables, v.v.).  
   - Vào tab Outputs, refresh để lấy URL Load Balancer.  
   - Mở URL, giao diện web Scorekeep xuất hiện.  
   - Tạo session/game:
     - Đặt tên game (ví dụ “Sample Game”).  
     - Chọn luật (ví dụ Tic Tac Toe/Cờ caro).  
   - Nhấn Create, sau đó Play để chơi Tic Tac Toe.  
   - Mỗi lần chơi, ứng dụng gửi trace lên X-Ray.

4. Xem Service Map trong X-Ray  
   - Vào X-Ray → Service map.  
   - Thu nhỏ để xem toàn bộ sơ đồ:  
     - ECS container (service chính).  
     - Nhiều bảng DynamoDB (game table, session table, state table, v.v.).  
     - SNS topic.  
   - Service map thể hiện:
     - Các thành phần (microservices, DB, SNS…)  
     - Quan hệ phụ thuộc và các API call giữa chúng.  
     - Lỗi được tô màu (ví dụ SNS đang có 100% lỗi, hiển thị màu cam).

5. Phân tích chi tiết lỗi và hiệu năng  
   - Nhấp vào một node (ví dụ SNS) để xem:
     - Độ trễ trung bình theo thời gian.  
     - Số lượng request, số lỗi.  
     - Phân phối thời gian phản hồi.  
   - Nếu thấy 100% lỗi tới SNS, có thể nhấn “View traces” để phân tích.  

6. Làm việc với Trace (Traces console)  
   - Vào phần Traces để chạy truy vấn.  
   - Có thể:
     - Xem tất cả trace (ví dụ 21 trace).  
     - Lọc theo annotation/thuộc tính (chỉ trace liên quan ECS container, game score table, v.v.).  
   - Ví dụ: lọc chỉ “game score” → còn 10 trace.  
   - Xem biểu đồ latency:
     - Hầu hết request < 200 ms.  
     - Một số request > 500 ms → có thể điều tra sâu.

7. Xem chi tiết một trace  
   - Chọn trace có thời lượng cao → xem “trace details”:  
     - Hiển thị bản đồ trace cho request đó (ECS + các bảng DynamoDB).  
     - Bên dưới là timeline chi tiết các sự kiện:
       - Các API call tới DynamoDB: GetItem trên game table, state table, v.v.  
       - Thời gian từng call, độ trễ.  
   - Nhấp vào từng segment để xem:
     - Lỗi (nếu có).  
     - Tài nguyên liên quan.  
     - Annotation, metadata.  
   - Nếu bật tích hợp log (CloudWatch Logs), có thể xem log tương ứng với trace → giúp debug rất mạnh.

8. Hạn chế của X-Ray console mới  
   - Console X-Ray mới trong CloudWatch hiện chỉ có:
     - Service map  
     - Traces  
   - Các tính năng khác (sampling, encryption, groups) vẫn phải dùng X-Ray console cũ.

9. Dọn dẹp tài nguyên  
   - Sau khi demo xong, cần xóa stack CloudFormation “Scorekeep X-Ray” để tránh tốn chi phí.  
   - Vào CloudFormation → Delete stack → chờ xóa xong.

Ý chính: Bài giảng hướng dẫn triển khai một ứng dụng demo trên ECS qua CloudFormation, chơi game Tic Tac Toe để tạo traffic, rồi dùng AWS X-Ray (tích hợp trong CloudWatch) để xem service map, phân tích trace, lỗi và độ trễ giữa các dịch vụ (ECS, DynamoDB, SNS), đồng thời nhấn mạnh sức mạnh của X-Ray khi kết hợp với CloudWatch Logs và lưu ý các tính năng hiện chưa có trong console mới.
