


Người hướng dẫn đang demo tính năng Canary Deployment trong Amazon API Gateway:

1. Tạo tài nguyên mới tên `canary-demo`, thêm phương thức GET tích hợp với Lambda (kiểu proxy), chọn hàm Lambda `stage-variables-get` có 2 phiên bản:
   - v1: trả về "Hello from Lambda v1"
   - v2: trả về "Hello from Lambda v2"

2. Ở bước đầu, cấu hình tích hợp để gọi phiên bản 1 của Lambda (thêm `:1` vào sau tên hàm), triển khai API lên một stage mới tên `canary`, kiểm tra URL `/canary/canary-demo` thấy trả về "Hello from Lambda v1".

3. Thiết lập Canary cho stage `canary`:
   - Tạo Canary, cấu hình phân phối lưu lượng 50% cho Canary, 50% cho stage hiện tại (thực tế thường dùng tỷ lệ nhỏ hơn, ví dụ 10%, nhưng ở đây dùng 50/50 để dễ quan sát).

4. Cập nhật tích hợp của phương thức GET:
   - Sửa Integration Request để gọi phiên bản 2 của Lambda (đổi từ `:1` sang `:2`).
   - Test trong API Gateway thấy trả về "Hello from Lambda v2".
   - Triển khai API lên stage `canary` (đây là bản mới dành cho Canary).

5. Kiểm tra lại URL:
   - Khi refresh nhiều lần, lúc thì thấy "Hello from Lambda v1", lúc thì "Hello from Lambda v2" → chứng tỏ Canary đang chia lưu lượng giữa v1 và v2 theo tỷ lệ đã cấu hình.

6. Khi xác nhận phiên bản mới (v2) hoạt động tốt:
   - “Promote” Canary, nghĩa là cập nhật stage để 100% lưu lượng chuyển sang phiên bản mới.
   - Sau một thời gian, refresh lại chỉ còn thấy "Hello from Lambda v2" → Canary đã được thăng cấp, v2 trở thành phiên bản chính.

Kết luận: Quy trình dùng Canary trong API Gateway là:
- Sửa tài nguyên / phương thức để trỏ tới phiên bản mới.
- Tạo Canary trong stage, cấu hình tỷ lệ lưu lượng.
- Triển khai API lên Canary.
- Kiểm thử, sau đó promote Canary để toàn bộ lưu lượng dùng phiên bản mới.
