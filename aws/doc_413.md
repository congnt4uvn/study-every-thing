


Dưới đây là tóm tắt ngắn gọn nội dung bạn đưa:

1. Khái quát về AWS KMS  
- KMS là dịch vụ quản lý khóa mã hóa của AWS, được tích hợp sâu với IAM.  
- Hầu hết các dịch vụ AWS khi nói “mã hóa” đều dùng KMS (EBS, S3, RDS, SSM, DynamoDB…).  
- Ưu điểm: AWS quản lý vòng đời khóa, bạn chỉ dùng API; mọi lệnh gọi KMS đều được log trong CloudTrail.

2. Cách sử dụng KMS  
- Có thể dùng KMS trực tiếp qua AWS CLI/SDK để mã hóa dữ liệu bí mật (secret).  
- Lưu trữ secret dưới dạng đã mã hóa (trong code, biến môi trường…) thay vì plaintext.

3. Các loại KMS key (về mặt mật mã)  
- Khóa đối xứng: 1 khóa duy nhất dùng cho cả mã hóa và giải mã; hầu hết dịch vụ AWS tích hợp KMS dùng loại này. Người dùng không bao giờ thấy khóa thật, chỉ gọi API KMS.  
- Khóa bất đối xứng: có cặp public/private key.  
  - Public key dùng để mã hóa hoặc verify, có thể tải xuống.  
  - Private key dùng để giải mã hoặc sign, chỉ dùng qua API KMS.  
  - Dùng khi cần mã hóa bên ngoài AWS (client không có quyền gọi KMS), họ dùng public key, gửi dữ liệu đã mã hóa về, bạn giải mã trong AWS bằng private key.

4. Các loại KMS key (về mặt sở hữu/quản lý)  
- AWS-owned keys:  
  - “Ẩn” phía sau các dịch vụ như SSE-S3, SSE-DynamoDB (chọn “AWS owned key”).  
  - Bạn không thấy/không quản lý trực tiếp, miễn phí.  
- AWS-managed keys:  
  - Tên dạng `aws/<service>` (vd: `aws/rds`, `aws/ebs`, `aws/dynamodb`).  
  - Miễn phí, dùng được trong phạm vi dịch vụ tương ứng.  
- Customer-managed keys (CMK – giờ gọi là KMS key do khách hàng quản lý):  
  - Bạn tạo, cấu hình, gắn policy, alias…  
  - Tính phí ~1 USD/tháng mỗi key; nhập key (import) cũng tương tự.  
- Ngoài phí key, KMS tính phí theo số lượng API call (~0.03 USD cho 10.000 request).

5. Key rotation (xoay vòng khóa)  
- AWS-managed KMS key: AWS tự động xoay mỗi năm.  
- Customer-managed key:  
  - Có thể bật auto-rotation (chu kỳ do bạn cấu hình) hoặc xoay thủ công.  
- Imported key: chỉ xoay thủ công.  
- Alias được dùng để trỏ đến phiên bản key hiện tại, giúp ứng dụng không phải đổi key ID.

6. Phạm vi theo vùng (region)  
- KMS key là resource theo region.  
- Không thể có “cùng một” KMS key ở hai region.  
- Khi sao chép EBS snapshot sang region khác:  
  1) Tạo snapshot từ volume mã hóa (snapshot cũng mã hóa bằng key A ở region nguồn).  
  2) Copy snapshot sang region đích, re-encrypt bằng key B ở region đích (AWS thực hiện).  
  3) Từ snapshot đã mã hóa bằng key B, tạo EBS volume ở region đích.

7. KMS key policy  
- Dùng để kiểm soát truy cập vào KMS key, tương tự S3 bucket policy.  
- Nếu không có key policy, không ai truy cập được key.  
- Hai kiểu chính:  
  - Default key policy: tạo tự động nếu bạn không cung cấp policy tùy chỉnh; thường cho phép mọi principal trong tài khoản (kết hợp với IAM policy) dùng key.  
  - Custom key policy:  
    - Chỉ rõ user/role nào được dùng key, ai được quản trị key.  
    - Hữu ích cho kịch bản cross-account (nhiều tài khoản cùng dùng một key).

8. Ví dụ cross-account với snapshot mã hóa  
- Tài khoản nguồn:  
  1) Tạo snapshot EBS mã hóa bằng customer-managed KMS key.  
  2) Gắn custom key policy cho phép tài khoản khác (target) sử dụng key (multi-account access).  
  3) Chia sẻ snapshot mã hóa cho tài khoản target.  
- Tài khoản đích:  
  4) Tạo bản copy snapshot, re-encrypt bằng customer-managed key của tài khoản đích.  
  5) Tạo EBS volume từ snapshot này và sử dụng bình thường.

Nếu bạn muốn, tôi có thể vẽ lại sơ đồ luồng (flow) cho một trong các kịch bản trên (ví dụ: copy EBS snapshot cross-region hoặc cross-account) để dễ nhớ cho kỳ thi.
