


Người hướng dẫn đang demo cách tạo Metric Filter và Alarm trong AWS CloudWatch dựa trên log Nginx:

1. Vào CloudWatch Logs, mở log access của Nginx, tìm các dòng có mã lỗi HTTP 400 (ví dụ `"HTTP/1.1" 400`).

2. Tạo Metric Filter:
   - Chọn “Create metric filter” từ log group.
   - Nhập pattern đơn giản, ví dụ chỉ cần chứa “400”.
   - Test pattern với log mẫu, thấy 14/50 sự kiện khớp → pattern hoạt động.
   - Đặt tên filter: `MetricFilter400Code`.
   - Chọn Metric namespace: `MetricFilters`.
   - Đặt Metric name: `MyDemoFilter`.
   - Metric value: 1 (mỗi lần match sẽ ghi giá trị 1).
   - Default value khi không có match: 0.
   - Tạo filter. Lưu ý: Metric filter chỉ áp dụng cho log mới, không hồi tố log cũ.

3. Tạo log mới để filter bắt đầu có dữ liệu:
   - Vào môi trường Elastic Beanstalk (`MyFirstBeanstalk`).
   - Restart application server để sinh thêm log mới lên CloudWatch.
   - Gửi request (ví dụ `/test`) để tạo thêm log.
   - Chờ vài phút rồi quay lại CloudWatch Metrics.

4. Kiểm tra Metric:
   - Trong CloudWatch Metrics, thấy namespace `MetricFilters` và metric `MyDemoFilter`.
   - Hiện tại giá trị là 0 (chưa có lỗi 400 mới), nhưng mục đích là minh họa rằng metric chỉ bắt đầu ghi từ lúc filter được tạo.

5. Tạo CloudWatch Alarm trên Metric:
   - Chọn metric `MyDemoFilter`, nhấn “Create alarm”.
   - Đặt ngưỡng, ví dụ: nếu giá trị > 50 thì coi là có vấn đề với ứng dụng web.
   - Chọn hành động khi alarm (gửi thông báo đến SNS topic).
   - Đặt tên alarm: `DemoMetricFilterAlarm`.
   - Sau khi tạo, metric filter được liên kết với alarm này.

Kết luận: Ví dụ minh họa cách:
- Lọc log theo mã lỗi (400) bằng Metric Filter.
- Xuất metric tùy chỉnh từ CloudWatch Logs.
- Tạo CloudWatch Alarm dựa trên metric đó để tự động gửi thông báo khi số lỗi vượt ngưỡng.
