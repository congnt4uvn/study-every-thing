


Dưới đây là tóm tắt nội dung:

1. Long polling (thăm dò dài) trong SQS  
   - Khi client gọi ReceiveMessage, có thể yêu cầu SQS “chờ” (1–20 giây) nếu hàng đợi đang trống.  
   - Nếu trong thời gian chờ có message mới vào hàng đợi, SQS sẽ trả message đó ngay cho client.  
   - Lợi ích:
     - Giảm số lượng cuộc gọi API → giảm chi phí, giảm sử dụng CPU.  
     - Giảm độ trễ: message đến hàng đợi là gần như lập tức đến client.  
   - Long polling có thể cấu hình:
     - Ở mức hàng đợi (ReceiveMessageWaitTimeSeconds).  
     - Ở mức từng lệnh gọi API (tham số ReceiveMessageWaitTimeSeconds trong ReceiveMessage).  
   - Khuyến nghị: dùng long polling (thường đặt 20 giây) thay vì short polling.

2. SQS Extended Client (xử lý message lớn hơn 256 KB / 1.024 KB)  
   - SQS giới hạn kích thước message tối đa ~1 MB.  
   - Để gửi message lớn (ví dụ 1 GB), dùng thư viện Java SQS Extended Client (hoặc tự triển khai logic tương tự):  
     - Dữ liệu lớn được lưu vào Amazon S3.  
     - Hàng đợi SQS chỉ chứa một message nhỏ (metadata) có “con trỏ” (thông tin tham chiếu) đến object lớn trong S3.  
     - Consumer đọc message nhỏ từ SQS, sau đó dùng thông tin trong đó để tải dữ liệu lớn từ S3.  
   - Use case điển hình: xử lý file video, file lớn → upload file lên S3, gửi vào SQS một message nhỏ chứa link/ID của file.

3. Các API quan trọng của SQS  
   - CreateQueue: tạo hàng đợi; có thể thiết lập MessageRetentionPeriod (thời gian giữ message trước khi bị xóa).  
   - DeleteQueue: xóa hàng đợi và toàn bộ message trong đó.  
   - PurgeQueue: xóa tất cả message trong hàng đợi (hàng đợi vẫn tồn tại).  
   - SendMessage: gửi message; có thể dùng DelaySeconds để trì hoãn message.  
   - ReceiveMessage: nhận/thăm dò message;  
     - MaxNumberOfMessages mặc định = 1, có thể tăng tối đa 10 để nhận batch.  
     - ReceiveMessageWaitTimeSeconds: thời gian chờ → bật long polling.  
   - DeleteMessage: xóa message sau khi xử lý xong.  
   - ChangeMessageVisibility: thay đổi visibility timeout nếu cần thêm thời gian xử lý.  
   - Có các phiên bản batch cho: SendMessage, DeleteMessage, ChangeMessageVisibility → giảm số lượng API call, giảm chi phí.

4. Demo long polling trong console  
   - Vào hàng đợi demo, chỉnh Receive message wait time từ 0 (short polling) lên 20 giây (long polling).  
   - Khi bật long polling:
     - Consumer thực hiện một cuộc gọi ReceiveMessage và “chờ” tối đa 20 giây nếu hàng đợi trống.  
     - Khi producer gửi message (“hello world”), message gần như lập tức được consumer nhận vì consumer đang chờ sẵn.  
   - Kết luận: long polling giúp giảm số API call, giảm chi phí, giảm độ trễ nhận message.
